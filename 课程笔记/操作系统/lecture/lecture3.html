进程线程模型

Summary

这一章讲的是操作系统里最基本的运行单位——进程和线程。我们从”程序怎么动起来”开始，聊到进程的生命周期、PCB、状态转换，再到 fork/exec 是怎么工作的，最后把线程和协程也串起来。看完这部分，你应该能讲清楚：进程和程序到底啥区别？fork 之后父子怎么共享内存？用户级线程和内核级线程差在哪？协程又是来凑什么热闹的？

进程模型

Summary

先搞定进程模型的地基：程序是怎么从静态代码变成动态执行流的？操作系统为什么要搞出”进程”这个东西？进程在内存里长什么样、状态怎么变来变去？这些是理解后面一切的基础。

基本概念：顺序程序、多道程序与并发环境

程序就是一堆指令和数据的静态集合，躺在硬盘上的时候啥也没干。

在早期的顺序环境里，计算机一次只能跑一个程序，这个程序独占地用着 CPU、内存、外设。因为没人跟它抢资源，它的执行是顺序的、封闭的，而且结果可预测——给定同样的输入，跑多少遍结果都一样，跟跑得快慢没关系。这就是程序的结果可再现性。

但后来人们发现，CPU 跑计算的时候 I/O 设备常常闲着，I/O 干活的时候 CPU 又没事做，太浪费了。于是就有了多道程序设计：让多个程序同时待在内存里，谁需要 CPU 就给谁，谁去等 I/O 就先把 CPU 让出来给别人。

Intuition

你可以把单核 CPU 想象成厨房里的一个灶眼。多道程序不是说你突然有了两个灶眼，而是你学会了一边煮面、一边等水开的间隙去切菜，让灶眼尽量不闲着。

所以单处理器上的”同时”并不是真正的硬件并行，而是在一段时间内多个程序都处于”开始执行但还没结束”的状态。它们的执行顺序不预先确定，这种现象就叫做并发。并发带来了很多新麻烦：程序执行可能断断续续，资源要共享，执行结果跟速度有关了，程序和一次具体的运行过程也不再是一一对应的关系。

进程的定义、作用与进程映像

操作系统为了解决这些麻烦，抽象出了**进程（Process）**这个概念。

进程是具有独立功能的程序在某个数据集合上的一次运行活动，是系统进行资源分配和处理器调度的独立单位。简单来说：

  • 程序是静态的代码，进程是动态的执行实体。
  • 操作系统通过让 CPU 时间复用、在多个进程之间来回切换，让每个进程都有一种”CPU 独属于我”的幻觉。
  • 每个进程有自己独立的地址空间，运行时以进程映像的形式存在。

进程映像包括：程序代码、数据、堆、栈，以及内核里的**进程控制块（PCB）**等。运行期间还关联着页表、打开的文件、信号处理状态、调度信息等等内核对象。用户态的部分主要在内存里（挂起或交换时可能部分移到磁盘），内核态部分由操作系统直接维护。

同一个程序可以对应多个进程（比如同时打开两个微信窗口），进程有自己的生命周期，还能创建新的子进程。

进程的分类

  • 系统进程 vs 用户进程：一个跑操作系统服务，一个跑你的应用程序。
  • 前台进程 vs 后台进程：前台跟你交互，后台默默干活（比如守护进程）。
  • CPU 密集型 vs I/O 密集型：前者爱算数（比如视频编码），后者爱读写（比如文件下载器）。
  • UNIX 里进程常以家族树组织，init 或它的后继进程是树根。
  • Windows 里进程关系更平等，父子关系主要用于创建和管理，不像 UNIX 那样是整个系统的骨架。

进程状态与状态转换

一个进程这辈子会在几个状态之间跳来跳去。最经典的是三态模型：

  • 运行态：进程正在 CPU 上执行。
  • 就绪态：进程一切准备就绪，只等 CPU 空出来。
  • 等待态（阻塞态）：进程在等某个事件发生，比如磁盘 I/O 完成、键盘输入、或者其他进程发消息，暂时没法运行。

Trap

注意：“阻塞”和”就绪”的区别不是”想不想跑”，而是”能不能立刻跑”。阻塞是因为条件不满足（比如数据没准备好），就绪是因为条件满足但没抢到 CPU。

一个进程在任一时刻只能处于且仅处于一种状态。

flowchart TD
    New[创建] --> Ready[就绪]
    Ready -->|调度| Running[运行]
    Running -->|时间片到/被抢占| Ready
    Running -->|等待事件| Blocked[等待/阻塞]
    Blocked -->|事件发生| Ready
    Running -->|执行结束| Terminated[终止]

在三状态模型之上，实际系统还会加入更多状态：

  • 创建态：PCB 和 PID 已经分配好了，但系统还没正式接纳它进可执行队伍（可能资源不够）。
  • 终止态：进程执行完了，但还留着一些信息没清干净，等着父进程来”收尸”或者系统做统计。
  • 挂起态：进程映像被换出内存、放到磁盘上，用来释放内存压力。挂起可以和就绪/阻塞结合，形成就绪挂起和阻塞挂起。

Note

Android 中与挂起态最接近的机制，是应用进入后台后的 stopped/cached/frozen 状态：系统停止其前台活动，限制或冻结其执行，尽量保留其进程与内存以支持快速恢复；在内存压力下则可能直接终止该进程，并在用户返回时依据保存状态重新创建。

iOS 中与挂起态最接近的是应用进入后台后的 suspended 状态。应用在该状态下通常仍保留在内存中，但不再执行代码；当用户重新进入应用时，系统可直接恢复其运行。若系统发生内存压力或其他后台终止条件，应用也可能在 suspended 或 background 状态下被直接终止，之后通过状态恢复机制重新创建。

状态转换并不是严格成对发生的。一个进程从运行态变成阻塞态，只是它自己放弃了 CPU，不意味着立刻就有另一个进程从就绪态变成运行态——这得看调度器的心情。反过来，一个阻塞进程因为事件发生变成就绪态，也不代表它立刻就能上 CPU。这种”局部状态变化”和”全局调度决策”的分离，是进程模型里特别重要的一个特征。

Linux 与 xv6 的状态表示

Linux 的进程状态并不严格对应教科书的三态或五态，而是围绕内核自己的调度和睡眠机制来组织的。历史上常见的状态有：

  • TASK_RUNNING：既包括”正在 CPU 上跑”，也包括”已经准备好可以跑”。
  • TASK_INTERRUPTIBLE：可中断睡眠，可以被信号唤醒。
  • TASK_UNINTERRUPTIBLE：不可中断睡眠，只能等目标事件完成（比如等磁盘 I/O）。
  • TASK_STOPPED：被暂停了（比如收到 SIGSTOP）。
  • TASK_ZOMBIE：进程已经退出，但父进程还没回收它的退出信息。

Linux 把”就绪”和”运行”统一放进可运行集合，这是实现层面对调度队列操作做的优化。

xv6 的状态更简洁，非常适合教学：

  • UNUSED：进程槽空着。
  • USED：已经分配了，但还没正式进入生命周期。
  • SLEEPING：在睡觉等事件。
  • RUNNABLE：可以被调度。
  • RUNNING：正在执行。
  • ZOMBIE：退出了，等着被回收。

你可以通过 allocproc、fork、sleep、wakeup、yield、exit、wait 这些函数，清楚地看到 xv6 进程状态机的控制关系。

进程控制块与进程表

**进程控制块（PCB, Process Control Block）**是操作系统里用来描述进程的数据结构，也叫进程描述符。进程和 PCB 是一一对应的，PCB 是操作系统”感知”进程存在、管理进程、调度进程的唯一依据。

Note

我还以为是电路板呢。

所有 PCB 的集合就构成了进程表。Linux 里的核心对象是 task_struct，Windows 里对应 EPROCESS、KPROCESS、PEB 等结构。

PCB 里至少要存四类信息：

  1. 标识与关系信息：PID、进程名、用户标识、父子关系、进程组、会话关系。
  2. 调度与管理信息：当前状态、优先级、调度参数、运行时间统计、上下文切换次数、队列链接、阻塞原因等。
  3. 资源与地址空间信息：虚拟内存描述、页表或内存映射、打开文件表、当前工作目录、IPC 状态、已占有的资源等。
  4. 处理器现场信息：通用寄存器、程序计数器、程序状态字、栈指针等。发生进程或线程切换时，这些信息需要被保存下来。

Note

IPC 是操作系统为相互隔离的进程提供的通信与同步机制。

它的目标是支持进程之间的数据交换、事件通知和协作执行。常见 IPC 机制包括管道、命名管道、消息队列、共享内存、信号量、信号和套接字等。其中，共享内存效率最高，但必须配合同步机制使用；消息传递机制实现简单、隔离性好，但通常开销较大。

IPC 的实现与阻塞唤醒、调度、文件描述符、信号处理以及资源管理密切相关，是操作系统并发与协作机制的重要组成部分。

Solaris 的 proc_t 很清晰地展示了”进程是资源容器”这个设计思想：它直接关联地址空间 p_as、认证信息 p_cred、会话信息、PID/PGID、信号信息、打开文件表 u_finfo、可执行文件 p_exec、线程链表 p_tlist 等等。

Linux 的 task_struct 则包含状态、优先级、运行队列链表、允许执行的 CPU 集合、mm_struct/active_mm、父子与线程组关系、文件系统状态、信号处理对象、定时器、上下文切换统计和体系结构相关的 thread_struct。

xv6 的 struct proc 只保留最关键字段：锁、状态、睡眠通道、终止标志、退出码、PID、父进程、内核栈、用户地址空间大小、页表、陷入帧、上下文、打开文件数组、当前目录和进程名。真实系统远比教学系统复杂，因为权限管理、信号处理、多处理器调度、命名空间、NUMA、文件系统等等都会往 PCB 里塞额外状态。

进程地址空间：对内存的抽象

操作系统分配给进程的不是一段固定的物理内存，而是虚拟地址空间。虚拟地址空间主要有三个作用：

  1. 统一地址视图：程序可以在”看似连续”的地址空间里运行，不用关心物理内存到底怎么布局。
  2. 保护与隔离：不同进程的地址互不冲突，A 进程不能随便访问 B 进程的内存。
  3. 支持高级机制：共享内存、内存映射文件、按需装入（demand paging）、换页等都需要虚拟地址空间做基础。

Example

你可以把虚拟地址空间理解为”每个人的微信昵称”——两个人可以都叫”小明”（虚拟地址相同），但系统后台知道他们对应两个完全不同的用户 ID（物理页不同）。所以两个同时运行的相同程序能看到相同的变量地址，却有着不同的变量值。

现实系统通常会开启地址空间布局随机化（ASLR），所以同一程序每次运行时的虚拟地址位置可能都不一样；关了 ASLR 后，布局才更可能保持一致。

一个典型的进程地址空间至少包含：代码段、数据段、堆、栈、共享库区域、内存映射区域。

在 Linux 里，你可以通过 /proc/PID/maps 看到一个进程所有的内存映射：可执行文件映射、堆区 [heap]、栈区 [stack]、共享库、[vdso]、[vsyscall] 等等。内核里，一段连续且属性一致的虚拟地址区域通常由 vm_area_struct 描述，所有区域通过 task->mm->mmap 等结构组织起来。映射了文件的区域叫有名映射，纯内存分配的区域叫匿名映射。

进程队列与调度关系

进程状态在操作系统内部通常是用队列来组织的。系统为每类状态维护一个或多个队列，队列里的元素就是 PCB。就绪态对应就绪队列，等待态按等待事件划分成多个等待队列；单处理器系统里，运行态本质上只对应”当前正在 CPU 上跑的那个进程”。进程状态一变，它的 PCB 就会从一个队列挪到另一个队列。

flowchart TD
    ReadyQ[就绪队列]
    Wait1[等待队列1]
    Wait2[等待队列2]
    CPU[处理器]
    CPU -->|等待事件1| Wait1
    CPU -->|等待事件2| Wait2
    Wait1 -->|事件1发生| ReadyQ
    Wait2 -->|事件2发生| ReadyQ
    ReadyQ -->|调度| CPU
    CPU -->|时间片到/抢占| ReadyQ

Intuition

队列模型其实体现了操作系统设计里很经典的一个思想：机制和策略分离。怎么把 PCB 从一个队列移到另一个队列、怎么保存上下文、怎么阻塞唤醒，这些是”机制”，是底层固定不变的通路；而就绪队列是按 FIFO 还是按优先级排序、时间片给多长、什么时候抢占，这些是”策略”，可以根据不同系统、不同应用灵活调整。

进程控制原语

进程的各种控制操作需要通过原语来实现。

Note

原语（primitive）是不可分割、不可中断的操作序列，执行时必须保持原子性。否则在修改 PCB、迁移队列或回收资源的过程中就可能出现竞态条件。（你可以把它理解成操作系统层面的”原子操作”）

常见的进程控制原语包括：创建、撤销、阻塞、唤醒、挂起、激活（解挂）、调整优先级等。

进程创建一般要做这些事：

  1. 分配唯一的 PID 和 PCB；
  2. 建立地址空间；
  3. 初始化 PCB 的各个字段；
  4. 建立或扩展内核相关的数据结构；
  5. 设置队列指针，把新进程插入到适当的队列中。

进程在什么时候会被创建？系统初始化、用户登录、提交程序执行、请求操作系统服务、或者现有进程派生子进程的时候都可能。

进程撤销则要结束它的线程或子任务，回收占用的资源，最后删掉 PCB。资源回收包括关闭打开的文件、断开网络连接、释放内存、解除内核对象引用等。

进程终止可能是自愿的（正常退出、检测到错误后主动结束），也可能是非自愿的（非法指令、越界访问、保护错误、I/O 失败、超时、被其他进程 kill）。终止不一定意味着 PCB 立刻消失，很多系统会保留一个”僵尸”阶段来记录退出码和统计信息。

阻塞和唤醒是进程控制中最典型的状态切换。运行中的进程发现自己要等某个事件（比如读磁盘），就主动执行阻塞原语，把自己从运行态变成等待态；等目标事件发生了（比如磁盘读完了），中断处理程序或其他进程执行唤醒原语，把它移回就绪队列。阻塞是说”我现在推进不了”，唤醒是说”条件满足了，你可以重新排队等 CPU 了”。

UNIX 进程 API：fork、exec、wait、exit

UNIX 进程控制有四剑客：fork()、exec()、wait()、exit()。

  • fork()：复制当前进程，创建子进程。
  • exec()：用新程序映像替换当前进程的用户地址空间。
  • wait()：父进程等待子进程结束，实现最基本的父子同步。
  • exit()：终止进程，给父进程留下可回收的退出信息。

Windows 对应的接口是 CreateProcess、ExitProcess、WaitForSingleObject 等。

Intuition

UNIX 把”创建子进程”和”让子进程执行新程序”拆成两步，这是个特别聪明的设计。想象一下，如果这两步合二为一，Shell 在创建进程的同时就换了新程序，那它还怎么帮你在 exec 之前重定向标准输入输出、建立管道、修改环境变量？正是 fork + exec 的分拆，让 Shell 有了”夹带私货”的机会。

    Child -->|继承文件描述符/工作目录/环境| Prepare[执行前定制]
    Prepare -->|exec| Program[目标程序映像]
    Program -->|exit| Zombie[退出待回收]

fork 的实现与写时复制

传统 fork() 的语义是为子进程构造父进程地址空间的副本，并继承共享资源（比如打开的文件、当前工作目录）。典型步骤是：给子进程分配进程描述符和 PID；复制父进程地址空间；继承共享内核资源；把子进程设为就绪并插入就绪队列；对子进程返回 0，对父进程返回子进程的 PID。

但直接逐页复制整个地址空间代价很高，尤其在 fork() 之后紧接 exec() 的常见场景下，复制过来的页面几乎马上就被扔掉了，纯属浪费。

COW技术来源于存储管理模块。Linux使用该技术对fork()进行了优化。

Linux 用**写时复制（Copy-on-Write, COW）**优化了 fork()：

  1. 为子进程创建新的 mm_struct、vm_area_struct 和页表副本，让父子拥有相同的虚拟地址布局；
  2. 把父子双方对应页面都标记为只读，并把相应区域标记为私有 COW；
  3. 当父进程或子进程第一次尝试写某个共享页面时，硬件因为写只读页触发异常，内核再分配新的物理页、复制原内容，让写入发生在私有副本上；
  4. 只有真正被修改的页面才会发生实际复制。
flowchart TD
    A[fork] --> B[复制 mm_struct / VMA / 页表布局]
    B --> C[父子页表都改为只读]
    C --> D[区域标记为私有 COW]
    D --> E[父子继续并发执行]
    E -->|读页面| F[直接共享物理页]
    E -->|写页面| G[触发缺页/保护异常]
    G --> H[分配新页并复制内容]
    H --> I[写入私有页]

Intuition

COW 就像兄弟俩共用一本教科书，一开始谁都不抄，只是约定”谁也不许在上面乱画”。某天哥哥想记笔记了，才偷偷复印了一本属于自己的。如果弟弟从来不想记笔记，那这本书就永远只存在一份，省纸又省力。

写时复制也说明，进程管理和内存管理并不是彼此孤立的模块。fork() 的高效实现依赖页表权限位、异常处理路径和虚拟内存子系统协同工作。进程抽象对 CPU 的复用，地址空间抽象对内存的复用，最终在 fork() 的实现中交汇。

中断、异常、陷入与进程线程模型的关系

进程和线程的运行并不是纯粹的软件约定，真正的控制权转移依赖于中断、异常和陷入机制。

  • 时钟中断：提供时间片边界，让操作系统可以抢占当前运行实体并执行调度器。
  • 设备中断：I/O 完成时唤醒等待队列里的进程或线程。
  • 系统调用（陷入）：让用户态程序进入内核态，获取操作系统服务。
  • 异常（页故障、保护错误、非法指令等）：可能触发缺页处理、信号投递、阻塞恢复或进程终止。

Intuition

如果把进程线程模型比作交通规则，那中断/异常/陷入就是红绿灯和交警。规则再完善，也得靠红绿灯切换和交警指挥才能让车流（执行流）有序运转。

所以进程状态转换背后的真正驱动源可以分成三类：

  1. 当前运行实体主动请求（系统调用、显式阻塞）；
  2. 外部硬件事件到达（时钟中断、设备中断）；
  3. 执行过程中检测到的异常事件（访问无效地址、写只读页等）。

进程特性与可重入程序

进程有五个基本特性：

  • 并发性：多个进程可以同时向前推进；
  • 动态性：进程会诞生、运行、阻塞、唤醒、消亡；
  • 独立性：每个进程有隔离的地址空间和资源边界；
  • 制约性：进程会因为共享资源或同步需求而相互影响；
  • 异步性：各进程推进速度相对独立、不可预知。

可重入程序是指能够被多个进程或执行流同时调用的程序。它的关键条件是：代码本身在执行过程中不修改自己，调用方为它提供独立的数据区。可重入性强调”代码可共享、状态不共享”，这跟进程地址空间隔离和线程共享地址空间中的同步问题都密切相关。

线程模型

Summary

进程虽然是好抽象，但把所有事都绑在一个执行流上效率不高。线程把”资源拥有者”和”调度执行者”这两个角色拆开，让同一个地址空间内可以跑多个执行流。我们会聊清楚：为什么要引入线程？用户级线程和核心级线程各有什么优缺点？混合模型又是怎么折中的？

引入线程的原因

在线程出现之前，进程同时承担着两个角色：资源拥有者和调度单位。随着应用变复杂，这种绑定开始暴露出问题：

  1. 应用需要多个执行流协同推进：比如图形编辑器或字处理软件，需要一边处理用户输入、一边后台排版、一边自动保存磁盘。放在一个进程里用多个执行流做，响应更快。
  2. 进程创建、撤销、通信和切换开销太大：限制了细粒度并发。
  3. 多核/多处理器架构要求单个地址空间内部也能并行执行。

Example

Web 服务器是线程价值的典型例子。服务器需要接收请求、检索网页、利用缓存、返回响应。如果不使用线程，要么每次只处理一个请求（吞吐量极差），要么基于非阻塞 I/O 和有限状态机搞事件驱动（编程模型复杂）。多线程服务器则可以让一个线程收连接、多个工作线程处理请求，既保持了阻塞式编程的简单自然，又获得了不错的并发性。

从并发服务器的构造方式来看，常见模型如下：

模型并发来源地址空间关系I/O 风格主要特征
多进程内核在多个进程间切换各进程独立可阻塞隔离性强，创建与通信开销大
事件驱动/有限状态机单线程手工交错多个逻辑流共享同一空间非阻塞高效但控制流复杂
多线程内核或线程库在多个线程间切换共享同一进程空间可阻塞或混合编程自然，适合通用服务器

线程的基本概念与属性

线程是进程中的一个运行实体，是处理器调度的基本单位。引入线程后，进程主要作为资源拥有者存在，线程承担执行路径和调度单位的角色。

线程有时也被称为”轻量级进程”，但它本质上不是”更小的进程”，而是把原先绑定在进程上的两个属性拆开了：

  • 进程拥有：地址空间、文件、I/O 设备等资源；
  • 线程拥有：程序计数器、寄存器状态、栈和可运行状态。

线程至少具有以下属性：

  • 有独立的执行状态和状态转换；
  • 不运行时要保存寄存器上下文，尤其是程序计数器、栈指针；
  • 有自己的栈和栈指针；
  • 共享所属进程的地址空间、打开文件和其他资源；
  • 能创建和撤销其他线程。

一个程序最初通常以单线程进程启动，然后才能在内部派生更多线程。

Trap

线程栈虽然在语义上属于各自线程私有，但因为多线程进程共享同一个地址空间，其他线程如果拿到了某个线程栈的地址，理论上是可以访问那片内存的。但这不代表它应该被共享——从程序正确性角度，乱访问别的线程栈是非常危险的行为。内核栈则由内核为每个线程维护，用户态代码不能直接碰。

用户级线程

用户级线程完全在用户空间实现，线程库和运行时系统负责线程创建、销毁、线程表维护和线程切换。

内核根本不知道一个进程内部有线程，从内核视角看，一个多线程进程仍然是单一的调度对象。因此用户级线程切换不需要进入内核态，不涉及特权转换，速度快、开销小，而且可以在不支持线程的操作系统上通过库机制实现。

用户级线程的优点：

  • 创建和撤销代价低，切换快；
  • 调度算法可以由应用自己控制，更贴近应用语义；
  • 线程库可移植性好。

但缺点也很明显：

  • 只要某个线程发起阻塞式系统调用，内核就会阻塞整个进程，连带阻塞该进程内的全部用户线程；
  • 内核只把处理器分配给进程，所以同一进程中的多个用户级线程无法真正同时运行在多个处理器上。

Example

想象用户级线程就像一家公司（进程）里的一群员工（线程），公司对外只有一个法人身份。如果其中一个员工去银行办贷款（阻塞系统调用），银行（内核）会把整个公司冻结，其他员工即使没事做也全都被卡住了。更惨的是，税务局（内核调度器）只认公司法人，不认具体员工，所以这群员工没法同时出现在不同的办事窗口（多处理器）上。

用户级线程里，“进程状态”和”线程状态”可能出现分离。比如线程 2 执行了阻塞系统调用，内核看到的是整个进程 B 阻塞；但当进程 B 再次被调度时，用户态线程库可以选择让线程 1 继续跑。如果线程 1 只是调用用户级原语主动让出处理器，线程库可以直接在进程内部切到线程 2，内核甚至察觉不到。

也正因为内核不参与，用户级线程的抢占式调度难以完全依赖内核完成，库实现经常需要线程自愿让出处理器，或者借助定时信号模拟抢占。

为了规避阻塞系统调用的问题，常见处理方式是：

  • 用包装技术把阻塞系统调用改写为非阻塞形式；
  • 重写 I/O 库函数，采用轮询、事件通知等异步手段。

这些方法本质上都是在缩小”内核只识别进程”这个前提带来的负面影响。

POSIX 线程接口

Pthreads 提供的基本接口包括：

  • pthread_create：创建新线程
  • pthread_exit：结束当前线程
  • pthread_join：等待指定线程结束
  • pthread_yield：主动让出处理器
  • pthread_attr_init / pthread_attr_destroy：初始化和释放线程属性对象

Intuition

课件里会问：“线程为什么要自愿让出 CPU？“答案就在用户级线程的实现约束里——内核只把整个进程当成可调度实体，线程库除非得到线程的”主动配合”（调用 yield），或者靠额外信号打断，否则没法在用户空间重新拿到调度权。

核心级线程

核心级线程由内核直接管理。内核维护进程和线程两层上下文，向应用程序提供线程 API。线程切换需要内核支持，调度器也以线程为基本对象进行处理。

因此核心级线程有两个关键优势：

  • 同一进程中的不同线程可以在多个处理器上并行执行；
  • 某个线程阻塞，通常只会阻塞该线程，不会连累整个进程。

Windows 是典型的核心级线程系统。

但代价也是明显的：

  • 线程创建、销毁、切换都要进入内核，系统调用频繁时开销高于纯用户级线程；
  • 内核必须维护更多线程级状态，调度器和同步机制也更复杂。

Intuition

核心级线程相当于给每个员工都发了独立的法人执照。好处是大家都能各自去不同窗口办事（多核并行），一个人被卡住不影响其他人。坏处是办执照和维护档案的成本变高了（内核开销大）。

混合模型

混合模型试图结合用户级线程和核心级线程的优点：

  • 线程创建在用户空间完成，用户级运行时维护大量用户线程；
  • 真正和处理器绑定的是较少数量的核心级线程，用户线程被多路复用到这些内核线程之上。

Solaris 曾经采用过这种思路。混合模型的目标是同时获得低开销的线程管理和多处理器并行能力，但它也带来了映射关系维护、阻塞传播和调度协同的复杂度。

线程实现方式的边界

线程实现方式的选择体现了操作系统设计中的边界权衡：

  • 用户级线程：把灵活性和低开销放首位，但依赖应用或运行时避免阻塞式内核接口；
  • 核心级线程：把通用性和并行性放首位，但增加了系统态切换与内核维护负担；
  • 混合模型：把复杂性转移到了运行时和内核之间的协作层。

Intuition

没有银弹。用户级线程快但不能真并行，核心级线程能并行但开销大，混合模型想两头占便宜但实现复杂。实际系统选哪种，取决于你最在意什么。

协程

Summary

线程虽然比进程轻，但数量上去了照样扛不住。协程是比线程更轻量的用户态执行流，通过主动挂起和恢复来协作运行。这一节我们会搞清楚：协程是为了解决什么问题？它和线程的边界在哪？以及进程、线程、协程三者怎么分工。

协程的引入动机

协程主要针对高并发和异步任务处理场景，尤其适用于大量连接、I/O 密集型服务与细粒度异步流程。线程虽然比进程轻量，但当线程数量进一步膨胀时，栈空间占用、切换开销、调度开销和同步复杂度都会迅速上升。“线程越多越好”并不成立。

面向 C10K（一台服务器如何同时处理 10,000 并发连接）一类高并发问题，系统往往需要把线程池、非阻塞 I/O、I/O 多路复用和更轻量的执行流组织方式结合起来。

协程的概念与边界

协程是在一个线程或进程内部实现多个执行流，并让这些执行流通过显式挂起与恢复进行协作的一种机制。

协程不是内核调度实体，通常运行在用户态，更接近语言或运行时提供的控制流抽象。它强调协作式切换——执行流在合适的时机主动让出控制权，而不是依赖内核随时抢占。

Example

你可以把协程想象成一场接力赛，每个选手（协程）跑到自己那一棒的终点后，主动把接力棒交给下一个人。没有裁判（内核）吹哨强制换人，全靠选手自己交接。所以协程切换非常轻量，不需要陷入内核。

协程与线程的边界：

  • 线程是操作系统可见的调度单位，有内核可识别的上下文和阻塞语义；
  • 协程通常只由语言运行时管理，本身不能直接在多个处理器上并行，除非底层仍然建立在多个线程之上；
  • 协程与用户级线程相似，都属于用户态执行流组织机制，但协程更强调异步编程模型中的挂起点、恢复点与状态机转化。

协程的使用与实现

协程通常依赖编程语言或库支持，例如 Python asyncio、Rust async_std、C++20 coroutine，以及 Go 的 goroutine/coroutine 风格支持。

它们常与事件循环、异步 I/O 或运行时调度器配合，把原本需要大量线程维护的等待状态，压缩成可恢复的轻量任务对象。典型收益是：减少线程数量和切换代价，使等待 I/O 的大量任务能够以极低成本挂起与恢复。

Windows 中的 **fiber（纤程）**可以视为与协程相近的用户态执行流机制：一个线程内部可以包含多个纤程，由用户态显式切换。纤程进一步说明，执行流组织并不必然等同于内核线程，语言、库和运行时可以在同一线程之上构造更轻量的并发抽象。

进程、线程与协程的层次关系

三者可以概括为如下层次：

  • 进程：解决”资源隔离与地址空间边界”问题，是资源分配单位。
  • 线程：解决”同一资源容器内部的并发执行与调度”问题，是经典的 CPU 调度单位。
  • 协程：解决”单线程内部大量异步任务的低成本组织”问题，是语言或运行时层面的协作执行流。
flowchart TD
    P[进程<br/>资源边界 / 地址空间 / 文件]
    T[线程<br/>调度单位 / 执行上下文 / 栈]
    C[协程<br/>用户态任务 / 挂起恢复 / 低切换开销]
    P --> T
    T --> C

Intuition

这三者不是谁替代谁的关系，而是分层的。一个服务器可以是这样构成的：多个进程做隔离，每个进程里多个线程利用多核，每个线程里再跑大量协程处理海量连接。 Engineering is all about trade-offs，关键是找到隔离性、并行性、编程复杂度、阻塞语义和切换成本之间的平衡点。

Summary

进程把 CPU 时间复用和资源边界组织起来，是操作系统最基础的运行抽象；PCB、进程状态机、地址空间、进程控制原语构成了它的实现骨架。线程把”资源拥有者”和”执行路径”分离，让同一地址空间内的多个执行流成为可能；用户级线程、核心级线程和混合模型分别对应不同的内核可见性、阻塞语义和并行能力。协程则进一步把并发组织推进到语言与运行时层，让大量异步任务能以更低成本运行。从底层控制机制到上层编程抽象，整个模型逐层展开，这就是现代操作系统和并发软件系统的基本结构。
