xv6 OS Lab Part 1：系统启动、自举过程与首批系统调用

本部分的核心任务不是编写一组新的用户程序，而是在现有 xv6-k210 内核上建立一套能够兼容预编译测试程序的系统调用接口，使系统可以正确启动、装载测试程序、输出可比对结果，并在测试结束后稳定退出。Part 1 的测试点集中在 getcwd、write、getpid、times 与 uname，因此这一部分同时牵涉启动链路、自举机制、评测流程、系统调用 ABI 对齐、首进程构造以及若干系统调用的实现或适配。

一、整体任务与系统主线

从工程角度看，Part 1 的工作不是零散地补几个系统调用，而是打通一条完整链路：

  1. 使用户态测试程序所使用的系统调用号与内核分发表完全一致；
  2. 使首个用户进程能够在当前平台上正确启动；
  3. 使 /init 能够按预期顺序调度测试程序；
  4. 使测试输出经过 write 稳定写到 QEMU 标准输出，供评测脚本逐字比对；
  5. 使测试结束后系统能够通过 shutdown 正常退出，而不是悬停或触发 panic: init exiting。

这一主线可概括为：先对齐 ABI，再重建 initcode，随后补齐关机与测试所需系统调用，最后将本地运行路径与平台评测路径分别处理。

    A[统一系统调用号] --> B[重建 initcode]

    B --> C[补全 shutdown 调用链]

    C --> D[修正 getcwd 语义]

    D --> E[补齐 times]

    E --> F[补齐 uname]

    F --> G[确认 write 与 getpid ABI 已对齐]

    G --> H[区分本地与平台构建路径]

    H --> I[按固定顺序运行测试并正常关机]

二、启动链路与首进程的形成

xv6 在 QEMU 中的启动过程由虚拟硬件、引导固件、内核初始化与首个用户进程启动四个层面组成。QEMU 提供 CPU、内存和块设备等虚拟硬件；RustSBI 作为引导固件首先运行，完成底层初始化并装载内核；内核接管控制权后完成进程管理、内存管理与文件系统挂载；随后内核创建第一个用户进程，并让它执行一段极小的用户态启动代码，由这段代码继续将控制权交给真正的 /init 程序。


graph TD

    A[QEMU 提供虚拟硬件] --> B[RustSBI 启动并装载内核]

    B --> C[内核初始化]

    C --> D[挂载文件系统镜像]

    D --> E[userinit 创建 PID=1]

    E --> F[将 initcode 复制到首进程地址空间]

    F --> G[首进程执行 initcode]

    G --> H[执行 /init 或直接进入完整 init 逻辑]

    H --> I["/init 调度测试程序或启动 shell"]

这一链路决定了 Part 1 中所有测试都以 /init 为用户态入口，而不是由 shell 手工启动。首进程的构造方式直接影响后续所有测试程序是否能够被正确装载和执行。

三、initcode 的作用与自举意义

initcode 的作用是完成系统自举。首个用户进程出现时，完整的用户态运行环境尚未建立，因而不能依赖动态链接器、标准库或常规 ELF 装载路径去“再启动一个加载器”。因此，内核必须人工创建一个最小用户进程，将一段极小的机器码直接复制到它的用户地址空间中，并把程序计数器指向这段代码的起始位置。随后，这段代码以用户态身份发起 exec("/init")，从而将真正的 /init 程序装入进程映像，完成从“尚无用户态环境”到“已有正式用户程序运行”的过渡。


flowchart TD

    A[想运行 /init] --> B[必须先有用户进程]

    B --> C[但首个用户进程尚未建立]

    C --> D[内核人工创建最小用户进程]

    D --> E[将 initcode 拷入其用户内存]

    E --> F[CPU 从该段用户代码开始执行]

    F --> G["initcode 发起 exec('/init')"]

    G --> H["/init 成为真正运行的首个正式用户程序"]

因此，initcode 不是长期运行的用户程序，而只是完成自举所需的最小桥梁。

四、initcode 的生成方式与构建位置

在实现上，initcode.S 是汇编源文件。它先经过汇编器生成目标文件，再经过链接与提取得到一段原始机器码字节，最后这段字节被转成可以被 C 源文件包含的数组文本，编入内核镜像。于是，userinit() 在创建 PID 1 时无需从磁盘读取程序，而是直接把这些字节复制到新进程地址空间中。调度器首次运行该进程时，它执行的第一组用户态指令就是这段被嵌入内核的 initcode。

这一生成路径如下：


flowchart TD

    A[initcode.S 汇编源文件]

    --> B[汇编得到 initcode.o]

    --> C[链接得到可执行映像]

    --> D[objcopy 提取纯二进制]

    --> E[脚本转成十六进制字节列表]

    --> F[生成 initcode.h]

    --> G[proc.c 中 include]

    --> H["编译进内核中的 initcode[]"]

需要特别区分两件事：

  • initcode.h 参与的是内核的编译，不是运行时加载；

  • CPU 最先执行的是被复制进首进程用户地址空间的那段机器码，而不是 initcode.h 这个头文件本身。

从构建顺序看，过程如下：

    A[执行 make] --> B[Makefile 解析依赖]

    C --> D[编译 proc.c]

    D --> E[预处理阶段 include initcode.h]

    E --> F["得到带有 initcode[] 的 proc.o"]

    F --> G[与其他目标文件一起链接为内核]

    G --> H[启动 QEMU]

    H --> I[内核创建首进程]

    I --> J["将 initcode[] 复制到首进程用户内存"]

    J --> K[首进程执行这段最小机器码]

五、为什么 initcode 不能沿用旧版本

initcode 不能被视作一段长期固定不变的裸字节，原因主要来自两方面。

1. 执行环境变化

旧框架中的 initcode.S 往往面向 32 位 RISC-V，而当前实验环境是 64 位 RISC-V。initcode 最终以机器码字节的形式嵌入内核，只要指令集位宽、地址宽度或调用约定发生变化，原先生成好的字节序列就可能不再适用。因此，在从 RV32 迁移到 RV64 时，必须重新生成 initcode，而不能继续沿用旧产物。

2. 系统调用 ABI 变化

initcode.S 会通过 SYS_exec、SYS_exit 等宏，把系统调用号直接编码进机器码，再通过 ecall 发起系统调用。这意味着，initcode 不仅依赖于“系统中存在 exec 和 exit”，更依赖于“这些系统调用在当前 ABI 中对应哪些具体编号、如何传参”。只要系统调用号发生变化，initcode 就必须同步重建，否则首进程在最早阶段就会提交错误的调用号，直接触发 unknown sys call。

这一关系可以表示为：


flowchart TD

    A[修改运行环境或接口] --> B{变化类型}

    B --> C[32 位 RISC-V -> 64 位 RISC-V]

    B --> D[sysnum.h 中系统调用号变化]

    C --> E[旧 initcode 字节序列失配]

    D --> F[旧 initcode 中的调用号失效]

    E --> G[必须重新生成 initcode]

    F --> G

    G --> H[首进程才能正确执行]

因此，initcode 同时受 ISA 位宽和系统调用 ABI 约束，其维护方式必须从“直接手改裸字节”转变为“修改源程序，再自动生成头文件”。

六、本地测试与平台评测的两条启动路径

理想情况下，initcode 应当只承担“发起第一个 exec("/init")”这一最小职责。本地测试环境通常可以保证文件系统镜像中存在可供装载的 /init，因此可以采用标准自举路径：首进程先执行短小的 initcode，再通过 exec("/init") 装载真正的 /init。

但平台评测环境与本地环境不同。平台运行时使用预先提供的 sdcard.img，其中并没有可供标准自举链装载的 /init，而且编译阶段也无法改写该镜像。于是，若平台仍沿用“短 initcode → exec("/init")”这条路径，自举过程会在查找 /init 时中断。

为了适配平台环境，必须采用另一条工程化路径：不再让首进程从磁盘中装载 /init，而是将完整的 init 用户程序预先编译为机器码，再转换为字节数组，直接作为 initcode[] 的内容嵌入内核。这样首进程一开始执行的就不是“只负责跳转的最小跳板”，而是完整的 init 逻辑本体，从而绕过“从平台镜像中查找 /init”这一步。

两条路径的区别如下：


flowchart TD

    A[内核创建首进程] --> B{运行环境}

  

    B --> C[本地测试]

    B --> D[平台评测]

  

    C --> C1[使用短 initcode.S]

    C1 --> C2[生成 initcode.h]

    C2 --> C3[首进程执行最小自举片段]

    C3 --> C4["exec('/init')"]

    C4 --> C5[从镜像装载 /init]

    C5 --> C6[进入正式用户态初始化]

  

    D --> D1[平台镜像中没有可装载的 /init]

    D1 --> D2[将完整 init 程序转成字节数组]

    D2 --> D3[生成 initcode.h]

    D3 --> D4[首进程直接执行整段 init 逻辑]

    D4 --> D5[绕过再次装载 /init]

这意味着构建流程必须分流：

  • 本地模式下，由短小的 initcode.S 生成 initcode.h；
  • 平台模式下，由完整的 _init 用户程序生成 initcode.h。

随后，proc.c 不再手写 initcode[] 的裸字节，而是统一通过 #include "include/initcode.h" 引入自动生成的内容。

七、评测流程与 /init 的职责变化

Part 1 的评测对象是内核，而不是测试程序本身。riscv64/ 目录中的测试文件已经在标准 Linux 环境下预编译完成，它们的职责是调用约定好的系统调用接口，检查返回值与副作用是否满足预期，并通过 printf 打印标准化输出。由于 printf 最终依赖 write 系统调用，测试结果会沿着“用户程序输出 → 内核 write → QEMU 标准输出”的路径暴露给外部评测脚本，评测端再将整个输出文本与标准答案逐字比较。只要输出完全一致，测试即判定为通过。

因此，虚拟磁盘中通常共存两类用户程序：

  1. xv6-user/ 目录下由本地构建系统编译得到的原生 xv6 用户程序，如 ls、cat、init 等，用于基础用户态功能；

  2. riscv64/ 目录下直接复制到镜像根目录中的预编译测试二进制，如 /getcwd、/write、/getpid、/times、/uname 等，仅用于自动化验收。

为了匹配这一评测模型，/init 不再承担“启动 shell 并长期驻留”的传统职责，而是被改造为测试调度器。它预先保存待测程序名数组，在主循环中依次执行 fork、exec 与 wait，保证每轮只运行一个测试程序，并等待其结束后再进入下一轮。所有测试结束后，/init 调用 shutdown() 触发关机，以避免平台因悬停而超时。


graph TD

    A["/init"] --> B[fork 子进程]

    B --> C[exec 某个测试程序]

    C --> D[测试程序调用系统调用]

    D --> E[内核处理并返回结果]

    E --> F[测试程序 printf 输出]

    F --> G[write 输出到 QEMU 标准输出]

    G --> H[评测脚本捕获文本]

    H --> I[与标准答案逐字比对]

    I --> J[全部完成后 shutdown 退出]

八、系统调用 ABI 对齐

Part 1 中最基础、也最关键的工作，是使测试程序所遵循的系统调用 ABI 与内核中的系统调用分发表保持一致。测试程序在发起系统调用时会将调用号放入 a7；内核的 syscall() 例程则依据该调用号查询 syscalls[] 分发表并跳转到对应处理函数。如果调用号不匹配，即使处理函数本身已经存在，内核仍会在错误的位置查表，最终报出 unknown sys call。

这一步的核心不是“增加功能”，而是让用户态传入的调用号与内核分发表的索引一一对应。

九、shutdown：使测试结束后系统稳定退出

shutdown 并不是 Part 1 的直接测试点之一，但它对于整条评测链至关重要。若测试全部完成后 /init 正常返回，xv6 往往会触发 panic: init exiting；若 /init 挂起不退，则平台可能因超时而判定失败。因此，必须补齐 shutdown 的整条调用链，使测试调度器在完成最后一个测试后能够主动关机。

这条调用链包括五个层面：

  1. 在 user.h 中声明 int shutdown(void);；

  2. 在 usys.pl 中登记 entry("shutdown");，生成用户态存根；

  3. 在系统调用号表中定义 SYS_shutdown 210；

  4. 在 kernel/syscall.c 中声明 extern uint64 sys_shutdown(void);，并将 SYS_shutdown 映射到 sys_shutdown；

  5. 在 kernel/sysproc.c 中实现 sys_shutdown()，内部调用 sbi_shutdown() 完成 QEMU 退出；

  6. 在 /init 的末尾调用 shutdown();。

其完整路径如下：


flowchart TD

    A[user.h 声明 shutdown] --> B[usys.pl 生成用户态入口]

    B --> C["用户程序调用 shutdown()"]

    C --> D[a7 = SYS_shutdown 并 ecall]

    D --> E[syscall.c 按编号分发]

    E --> F[命中 sys_shutdown]

    F --> G["sbi_shutdown()"]

    G --> H[QEMU 退出]

十、getcwd：修正接口语义而不是推翻算法

原项目中 getcwd 往往已经具备基本实现，能够沿当前目录向父目录回溯并拼接出绝对路径。但测试程序要求的并不只是“能够计算出当前路径”，而是要求系统调用接口在参数个数、长度检查和返回值语义上与标准接口一致。

为满足测试要求，getcwd 至少需要满足以下条件：

  1. 读取两个参数：

   - 第 0 个参数为用户缓冲区地址 buf；

   - 第 1 个参数为缓冲区大小 size。

  1. 在内核中使用临时缓冲区构造完整路径；

  2. 若路径长度连同结尾的 \0 超过 size，则返回 0；

  3. 若长度足够，则通过 copyout 将路径复制到用户态；

  4. 成功时返回用户传入的 buf 地址，而不是返回内核中的临时地址或其他数值。

因此，这一项的重点不是重新发明路径回溯算法，而是使调用语义严格符合测试程序的预期。

十一、write 与 getpid：核心在于 ABI 对齐

write 与 getpid 这两个测试点通常并不需要重写实现。原项目一般已经存在 sys_write 与 sys_getpid 的处理函数，它们无法通过测试的主要原因往往不是实现逻辑有误，而是系统调用号与测试程序不一致。

因此，对这两个系统调用需要重点确认的是：

  1. SYS_write = 64、SYS_getpid = 172 是否已正确设置；

  2. syscall.c 中的分发表是否将这两个编号映射到了现有的 sys_write 与 sys_getpid。

一旦 ABI 对齐完成，这两个测试点通常即可直接通过。

4. 为什么必须通过 copyout2() 安全回写用户态

在 times、uname、fstat 等系统调用中，内核需要把结果写回用户提供的缓冲区。这里的关键在于：用户传入的是用户虚拟地址，不是内核可以直接解引用的普通指针。因此，内核不能把这个地址直接强制转换后写入，而必须通过 copyout2() 这类接口完成安全回写。

    B --> B2[不检查写权限]
    B --> B3[可能导致内核异常]

    C --> C1[检查地址有效性]
    C --> C2[检查页表与权限]
    C --> C3[失败时可控返回]

可以概括为：

  • 直接写用户地址：把用户虚拟地址误当成内核普通指针；

  • copyout2() 安全回写：把它当成“用户空间目标位置”，由内核在检查、翻译后受控写入。

在 sys_times 中的意义

sys_times() 需要把 struct tms 返回给用户。如果直接写用户地址，系统调用的正确性就依赖于“这个地址刚好可写”，实现不可靠；只有通过 copyout2()，才能保证在地址非法、越界或权限错误时，系统调用受控失败，而不是把整个内核带崩。

十四、times 与 uname 的实际排障记录

在调试过程中，系统启动日志曾出现如下报错：

  • pid 5 times: unknown sys call 153

  • pid 1 init: unknown sys call 10

  • pid 6 uname: unknown sys call 160

这些错误揭示出当前工程存在一个高风险点：仓库中同时存在两类用户程序来源，一类是偏 xv6 风格的 xv6-user/*，另一类是来自 riscv64/* 的预编译测试产物，后者大量采用 Linux RISC-V 风格的系统调用编号。如果内核号表缺项、分发表缺失或两类程序依赖的编号不一致，就会出现 unknown sys call。

针对这组问题，排查应优先集中在以下文件：

  • kernel/include/sysnum.h

  • kernel/syscall.c

  • kernel/timer.c

  • kernel/sysproc.c

排查重点包括：

  1. syscall() 是否正确从 a7 读取系统调用号；

  2. syscalls[] 的对应下标是否存在函数指针；

  3. SYS_xxx 常量是否已经定义且编号正确；

  4. 处理函数是否真正实现，且在向用户态写回数据时使用了安全的 copyout/copyout2 接口。

最终发现的问题主要有三类：

  1. sysnum.h 一度缺失 SYS_times，导致 times(153) 无法匹配；

  2. 原工程中缺少 SYS_uname 与 sys_uname 的实现，导致 uname(160) 必然失败；

  3. 早期 sys_times 实现错误地将用户地址当作内核地址直接写入，而非通过 copyout2() 安全回写；

  4. dup(10) 报错同样说明号表与分发表之间存在 ABI 失配现象。

修复后的关键改动包括：

  • 在 kernel/include/sysnum.h 中补齐 SYS_times = 153、SYS_uname = 160；

  • 在 kernel/syscall.c 中补齐 sys_times 与 sys_uname 的声明、分发与名字映射；

  • 在 kernel/timer.c 中修复 sys_times()，使用 argaddr 获取用户地址、在锁内读取 ticks，构造 struct tms 后通过 copyout2() 写回，并返回节拍值；

  • 在 kernel/sysproc.c 中新增 sys_uname()，构造 struct utsname 并使用 copyout2() 写回；

  • 新建或补充公共头文件，用于声明 struct utsname 等共享结构。

这组问题说明，系统调用问题首先要检查“编号、分发、实现”三件套。任何一处缺失，都可能表现为 unknown sys call，而不仅仅是某个处理函数实现错误。

十五、initcode.S 在 RV64 下的链接与对齐问题

在将 initcode.S 迁移到 RV64 并重新生成 initcode 的过程中，链接阶段可能出现如下报错：

  • 5 bytes required for alignment to 8-byte boundary, but only 4 present

  • can't relax section: bad value

其本质原因在于：

  1. argv 已从 32 位指针改为 64 位指针，因此应使用 .quad；

  2. 同时又使用了 8 字节对齐（如 .p2align 3）；

  3. 但 initcode.S 的 .text 段极短，字符串和数据紧挨着指令，导致链接器在进行 RISC-V relax 时，为满足 8 字节边界对齐需要插入 5 字节填充，而当前布局中最多只能插入 4 字节，于是对齐失败。

修复方法是：

  • 保留 argv 使用 .quad，因为 RV64 下指针必须为 64 位；

  • 将对齐要求从 .p2align 3 改为 .p2align 2，即从 8 字节对齐降低为 4 字节对齐。

这一修改不会改变 argv 中指针的位宽，但可以降低链接器的填充需求，避免在极短代码段布局下触发 relax 冲突。

十六、构建系统分流与 Makefile 的职责

由于本地测试与平台评测在文件系统镜像和启动方式上存在本质差异，Makefile 不能只维持单一路径，而应显式区分本地构建与平台构建。

从工程要求看，至少需要处理以下事项：

  1. 提供面向平台的一键构建入口，例如 make all；

  2. 在平台构建路径中输出平台所要求的产物名称，如 kernel-qemu 与 sbi-qemu；

  3. 在本地构建路径中生成 fs.img，并将测试样例同步到镜像根目录；

  4. 在本地与平台两种模式下，分别采用不同的 initcode.h 生成策略：

   - 本地模式：由短 initcode.S 生成；

   - 平台模式：由完整 _init 程序生成。

平台模式下，完整 _init 转为 initcode.h 的路径大致为：先编译 init.c 得到 init.o，再与 usys.o、printf.o 等用户态对象链接为完整 _init，之后用 objcopy 提取纯二进制，并通过 od 或脚本转换为十六进制字节列表，最终生成 initcode.h，再由 proc.c 通过 #include 将其编入内核。

十七、实际动手顺序

若按实现顺序组织，Part 1 的工作可归纳为以下步骤：

  1. 修改系统调用号定义，优先对齐 wait、getcwd、write、exit、times、uname、getpid、shutdown、exec 等关键调用；

  2. 在 syscall.c 中补齐所有关键调用的分发表映射；

  3. 重建 initcode，停止手写 proc.c 中的裸字节；

  4. 补齐 shutdown 的整条调用链，使系统能够在测试结束后主动退出；

  5. 修正 getcwd 的参数读取、长度检查与返回值语义；

  6. 对 write 与 getpid 仅做 ABI 对齐确认；

  7. 定义 struct tms 并实现 sys_times()；

  8. 定义 struct utsname 并实现 sys_uname()；

  9. 修改 Makefile，分别处理本地与平台的 initcode.h 生成路径；

  10. 本地执行完整重建与测试，确认五个测试依次通过，并由 shutdown 正常关机。

十八、验证与回归检查

每次修改系统调用相关代码后，都应避免只看源码，而应重新验证整个构建链条。尤其在本实验中，运行结果可能来自旧的 kernel、旧的 fs.img，甚至来自与当前源码不一致的构建产物，因此“源码看起来已经改对了”并不等于“系统实际运行的就是新版本”。

较稳妥的验证流程包括：

  1. 完整重编译内核与用户程序；

  2. 重新打包文件系统镜像；

  3. 重启 QEMU；

  4. 观察启动日志中是否仍出现 unknown sys call；

  5. 重点检查 times、uname 以及 init 初始化阶段的 dup(0) 等调用是否已恢复正常；

  6. 确认测试输出顺序稳定，且系统最终经 shutdown 正常退出。

十九、结论

Part 1 的本质不是单点式地补若干系统调用，而是建立一条能够兼容预编译测试程序的完整用户态启动与系统调用链路。其关键在于：

  1. 正确理解 initcode 在系统自举中的地位，并使其随着 ISA 位宽和系统调用 ABI 的变化自动重建；

  2. 明确区分本地测试与平台评测在 /init 装载方式上的差异，分别采用短跳板与完整内嵌 init 两种方案；

  3. 优先完成系统调用号与分发表的 ABI 对齐；

  4. 对 getcwd、times、uname 等调用在语义、结构布局与用户态回写路径上进行适配；

  5. 通过 shutdown 使测试流程形成完整闭环。

从排障结果看，本次故障的根因集中在系统调用号与分发映射不完整或不一致，表现为 times(153)、uname(160)、dup(10) 等调用在启动阶段即触发 unknown sys call。修复后，只要重新构建内核与镜像，并保证 initcode 与当前 ABI 同步生成，Part 1 所要求的启动、自举与首批系统调用链路即可恢复到可测试状态。
