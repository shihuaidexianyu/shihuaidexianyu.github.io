操作系统概述

Tip

助教导读 这一讲最容易学散，因为课上既讲 ” 操作系统是什么 “，又讲历史、分类、批处理、SPOOLing、多道程序设计，还会穿插很多追问。你可以把这一讲读成一句话：操作系统之所以出现，是为了更有效地管理资源、给用户提供更好的服务，并把难用的裸机扩展成更容易使用的虚拟机器。

课件开头抛出的几个追问

这一讲一开始，其实不是在直接讲定义，而是在提醒你：真正理解操作系统，不能只背概念，还要能把概念和具体机制对应起来。

这一讲里反复会遇到下面这些问题：

  1. UNIX 里文件子系统和块设备之间的 Buffer Cache 是干什么的？
  2. UNIX 这个名字大概率是怎么来的？
  3. 单道批处理和多道批处理到底差在哪？
  4. SPOOLing 的原理是什么？今天还有没有影子？
  5. 传统分时系统今天还有意义吗？
  6. 操作系统和别的软件相比，到底特殊在哪？

你可以把这些问题当成一条复习线索：

  • 凡是谈 ” 为什么需要操作系统 “，都在回答第 6 个问题。
  • 凡是谈 ” 批处理、多道、分时、SPOOLing”，都在回答第 3、4、5 个问题。
  • 凡是谈 “UNIX、文件系统、Buffer Cache”，都在提醒你：操作系统不是抽象口号，它最终一定要落到具体实现上。

核心主线一：操作系统是什么？

1. 一个定义，先抓住三个关键词

操作系统（Operating System, OS）是计算机系统中最核心的系统软件，是由一组程序模块构成的整体。

这个定义不要背死，真正有用的是把它拆成三个关键词：

  • 资源管理者：管理 CPU、内存、磁盘、I/O 设备、时钟、网络接口，以及磁盘上的文件和数据。
  • 服务提供者：给用户和应用程序提供统一、方便、安全的使用接口。
  • 虚机器扩展者：把原本难用、复杂、易出错的裸机，扩展成更通用、更友好的计算平台。

所以，操作系统不是 ” 某个单独的软件 “，而是整个系统秩序的建立者。

2. 操作系统的设计与实现目标

课件里这一部分其实写得很全，不能只记成 ” 提高效率 ” 这么一句话。更准确地说，操作系统的设计目标包括：

  1. 抽象

    • 把复杂硬件细节藏起来。
    • 给用户和程序员提供更高层、更一致的接口。

  2. 模块化

    • 把大系统拆成若干相对独立的部分，便于设计、调试、维护、替换和扩展。

  3. 尽量用高级语言实现

    • 比如大量使用 C，而不是全用汇编。
    • 这样更利于可读性、可维护性和可移植性。

  4. 最小化额外开销

    • 时间开销要小。
    • 空间开销要小。
    • 操作系统不能自己变成性能负担。

  5. 保护与隔离

    • 用户程序不能随便破坏系统。
    • 一个程序出错不能轻易把别人拖死。

  6. 可靠性与安全性

    • 系统必须稳定。
    • 同时要防止非法访问和恶意破坏。

  7. 可移植性

    • 不应该死绑某一台机器。
    • 同样的系统思想最好能迁移到其他平台。

  8. 节能

    • 这一点在移动设备和嵌入式系统里尤其重要。

你会发现，操作系统和普通应用的差别就在这：
普通应用只需要完成自己的功能，操作系统要同时对全系统负责。

核心主线二：从三个角度理解操作系统

课件特别强调，同一个操作系统可以从三个角度来看。这个地方很重要，因为很多同学学到后面会把不同视角混在一起。

1. 资源管理的视角：操作系统是资源的管理者

这是自底向上的看法。

它管理什么？

硬件资源：

  • CPU
  • 内存
  • I/O 设备
  • 磁盘
  • 时钟
  • 网络接口

软件资源：

  • 文件
  • 磁盘上的信息
  • 各种运行时系统对象
它怎么管理？

不是嘴上说 ” 我来管 “，而是真的要维护一套数据结构和算法：

  1. 跟踪记录资源使用状况

    • 哪些资源空闲
    • 哪些资源忙
    • 被谁占用
    • 占用多久
    • 状态是否正常

  2. 分配和回收资源

    • 静态分配策略
    • 动态分配策略

  3. 协调冲突

    • 多个进程都来抢资源怎么办
    • 哪些资源可以共享
    • 哪些资源必须互斥使用
它为什么要这样做？

目标主要有三个：

  • 提高资源利用率
  • 实现资源共享
  • 在共享时做好保护并协调冲突

共享通常依靠两种复用方式实现：

  • 时间复用：同一资源轮流给不同进程使用
  • 空间复用：同一资源在空间上切开给不同进程使用

Example

打印机是最典型的例子。它天然是独占设备，同一时刻只能真正服务一个任务。但操作系统可以通过排队、缓冲和虚拟化，让很多进程 ” 看起来都能用打印机 “。这其实就是资源管理 + 虚拟化一起工作的结果。

2. 进程的视角：操作系统是一个动态运行体

这是从运行过程来观察操作系统。

按照这个角度看，操作系统不是一本静态说明书，而是一组持续运行、彼此协作的执行流：

  • 有的长期驻留后台
  • 有的在事件发生时被唤醒
  • 有的负责调度
  • 有的负责设备管理
  • 有的负责内存管理

课件里的原话可以概括成：
操作系统由一些可同时、独立运行的进程，以及一个协调这些进程的核心组成。

顺手把 ” 进程 ” 这个词也重新钉牢：

  • 进程不是静态程序文本。
  • 进程是程序的一次执行过程。
  • 它是动态的，有诞生、运行、阻塞、结束这一整条生命线。

3. 虚机器的视角：操作系统是分层扩展出来的机器

这是从内部结构来理解操作系统。

可以把它想成：

  • 最底层是真实硬件
  • 上面先建立基础控制能力
  • 再逐层加入进程、内存、设备、文件等抽象
  • 最终形成一个对上层应用更友好的 ” 虚拟机器 ”
flowchart TD
    A[裸机硬件] --> B[基本控制机制]
    B --> C[进程管理]
    B --> D[内存管理]
    B --> E[设备管理]
    D --> F[文件系统与存储抽象]
    C --> G[调度与并发]
    E --> G
    F --> H[应用程序看到的虚拟机器]

这个视角最重要的启发是：
操作系统的很多 ” 高级功能 “，本质上都是层层封装出来的。

核心主线三：操作系统的四大特征

这四个词不是孤立记忆点，它们之间是有因果关系的。

1. 并发（Concurrency）

并发是指系统处理多个同时性活动的能力。

  • 宏观上看，多个程序在同时运行。
  • 微观上看，在单 CPU 情况下，任何时刻通常只有一个程序真正占着 CPU。
  • 它们是在 CPU 上轮流执行的。

这里要顺手区分：

  • 并发：交替推进
  • 并行：真正同时推进

Example

一个厨师来回照看三口锅，这是并发。
三个厨师同时各管一口锅，这是并行。

并发会带来很多后续问题：

  • 活动切换
  • 相互依赖活动之间的同步
  • 资源保护
  • 共享冲突

2. 共享（Sharing）

共享是指多个用户程序和操作系统共同使用系统资源。

常见的两种共享形式：

  • 互斥共享：某一时刻只能一个进程用，比如打印机
  • 同时共享：多个进程可以同时读某个资源，比如共享只读文件

共享本身是好事，因为它提高了资源利用率；但共享一旦出现，就必须引入保护与控制。

3. 虚拟（Virtual）

虚拟就是把一个物理实体映射成多个逻辑实体，或者把复杂的真实对象包装成更好用的抽象对象。

典型例子：

  • 一个物理 CPU 看起来像很多个 ” 虚 CPU”
  • 一块真实内存看起来像每个进程各自独享的地址空间
  • 一堆磁盘块看起来像文件和目录

4. 随机性 / 不确定性

这个词最容易被忽略，但课件很强调。

由于系统里有并发、有共享、有异步事件，所以很多执行顺序不是预先写死的：

  • 谁先抢到 CPU 不一定
  • 谁先拿到锁不一定
  • 设备何时发中断不一定
  • 一个 bug 是否复现也不一定

所以操作系统必须面对一个现实：
系统行为有很强的不确定性。

5. 它们之间的逻辑关系

最推荐你这样串起来记：

  • 因为要让多个活动同时推进，所以出现了并发。
  • 因为多个活动要共同使用资源，所以出现了共享。
  • 因为共享会冲突、物理资源又有限，所以需要虚拟和抽象。
  • 因为并发 + 共享 + 异步事件一起存在，所以系统表现出随机性 / 不确定性。

核心主线四：发展历程与技术启示

操作系统的发展不是凭空长出来的，而是跟着三样东西一起变：

  • 硬件能力
  • 应用需求
  • 软件技术

总目标始终没变：

  • 更充分地利用硬件
  • 给用户提供更好的服务

第 1 阶段：1948-1970，硬件贵，人工便宜

这一阶段的中心矛盾是：机器太贵，不能闲着。

特点：

  • 用户和机器之间几乎没有交互
  • 控制台通常一次只能给一个用户用
  • 早期是装入程序、运行、打印结果
  • 保护机制很弱甚至没有

后来引入了几项特别关键的技术：

  • 数据通道 / 通道技术：让 I/O 和计算能重叠
  • 中断机制：让 CPU 不必傻等 I/O
  • 多道程序设计：一个作业等 I/O 时，CPU 去干别的作业
  • SPOOLing：把慢速 I/O 从计算路径上拆出来

第 2 阶段：1970-1985，硬件便宜，人工贵

这一阶段的中心矛盾变成：不能只让机器高效，还得让人高效。

于是重点从 ” 吞吐量 ” 逐渐转向 ” 交互性 ” 和 ” 响应时间 ”。

特点：

  • 多用户联机终端
  • 分时系统兴起
  • 用户可以在线开发、调试、编辑
  • 为了响应快，可以牺牲一部分 CPU 利用率

但这也引出了一个新问题：

  • 用户一多，系统性能下降
  • 响应时间和抖动变明显

典型历史案例

CTSS：第一个分时系统

CTSS 的思想在 1959 年 MIT 被提出，1961 年由 Fernando Corbató 等人在改装的 IBM 7090/94 上实现。

课件给的几个细节很值得记：

  • 大约支持 32 个交互式用户
  • 总内存 32K
  • 系统占 5K
  • 用户占 27K
  • 用户映像在内存和磁鼓之间切换

这些数字今天看很小，但它说明一件特别重要的事：
分时并不是 ” 资源很充裕时 ” 的产物，而是在资源极其紧张时硬抠出来的。

Atlas：虚拟存储和分页调度的先驱

1962 年 Manchester 大学的 Atlas 计算机投入运行。

它的重要性在于：

  • 它是第一台具有虚拟存储器思想的机器之一
  • 它把分页调度带到了系统设计里

这意味着：操作系统开始不仅在 ” 调作业 “，还在更深层地管理地址空间。

OS/360：一次著名的大失败

IBM 的 OS/360 发布时，带着已知的大量错误。

它之所以常被提到，不是为了嘲笑谁，而是为了提醒你：
操作系统规模一大，复杂度就会爆炸。

这也是后来很多系统工程思想、模块化思想、软件项目管理经验的重要背景。

Multics：理想很大，实现很难

Multics 于 1963 年启动，到 1969 年才发布。

它的名字来自 MULTiplexed Information and Computing Service。

它的很多思想非常超前，比如：

  • 计算服务化
  • 强调共享
  • 分时与多用户
  • 更系统化的保护和组织方式

今天你看 ” 云计算 "" 计算服务平台 ” 时，会发现其中很多理念其实和 Multics 很像。课件里说 ” 技术会轮回 “，说的就是这种现象。

UNIX：更小、更务实的成功路线

UNIX 可以看成是对 Multics 路线的一种 ” 降维实现 ”：

  • 更小
  • 更简洁
  • 更强调工具化组合
  • 更容易传播和移植

至于 UNIX 这个名字，课上提到的是 ” 猜测 “——通常会把它理解为对 Multics 的一种戏谑式命名：
Multics 是 ” 多 “，UNIX 是 ” 单 ” 或 ” 简化 ”。

这个问题本身不是考试大点，但它提醒你：
操作系统的发展里，技术路线和工程风格同样重要。

  • 重点从 ” 单机管理 ” 变成 ” 联网协作 ”
  • 资源共享和通信成为新的主题

  • 人、机、物深度融合
  • 操作系统不再只在 ” 电脑 ” 里，而是扩展到各种设备和环境中

核心主线五：操作系统分类与关键技术

1. 批处理系统（Batch Processing）

批处理系统的基本思路是：
把很多作业先收集起来，形成一批，然后自动、依次处理。

它追求的不是交互，而是：

  • 吞吐量
  • 设备利用率
  • CPU 利用率
批处理的典型工作方式

课件里给了一个很完整的 5 步流程：

  1. 用户把作业交给系统操作员。
  2. 操作员把许多作业组成一批，输入计算机系统。
  3. 操作员启动操作系统。
  4. 系统自动、依次执行每个作业。
  5. 操作员把结果交还给用户。
” 批 ” 到底是什么？

这里的 ” 批 ” 不是一个抽象词，而是：

  • 一次加载的一盘磁带或磁盘
  • 往往由若干个作业组装而成
  • 整批在处理中使用一组相同的系统软件
典型作业结构：FMS JOB

一个作业通常由三部分组成：

  • 用户程序
  • 数据
  • 作业说明书（作业控制语言）

也就是说，系统不是只接收 ” 代码 “，而是接收一整份 ” 请系统怎么帮我跑 ” 的任务包。

2. 单道批处理 Vs 多道批处理

这是课件明确点名要区分的内容。

单道批处理

内存里一次只放一个作业。

流程大概是：

  • 当前作业运行
  • 一旦它要做 I/O
  • CPU 就只能等

所以问题很明显：

  • CPU 可能闲着
  • I/O 设备可能闲着
  • 整体效率不高
多道批处理

内存里同时放多个作业。

当某个作业等待 I/O 时：

  • CPU 可以立刻切去执行别的作业

这样带来的好处是：

  • CPU 和 I/O 可以重叠工作
  • 整体吞吐量大幅提升
  • 用户共享系统成为可能

但它也立刻提出新要求：

  • 内存保护
  • 作业切换
  • 调度机制
  • 更复杂的资源管理

Example

单道批处理像只有一个顾客点餐，厨师在等汤煮开时什么也不干。
多道批处理像厨房里同时有几桌菜，炒锅空了就先炒下一桌，等汤的那桌先挂起。

3. 多道程序设计（Multiprogramming）

多道程序设计是多道批处理背后的核心思想。

它的本质不是 ” 真的同时执行 “，而是：

  • 内存里驻留多个程序
  • 当前程序一旦因为 I/O 等原因不能继续推进
  • CPU 立刻切给另一个可以推进的程序

所以它的核心价值是：

  • 让 CPU 少闲着
  • 让 I/O 设备少闲着
  • 提高整体系统吞吐量

这个思想会一直延续到今天，哪怕系统形式已经变成图形界面、多任务、云平台，本质上还是这套逻辑。

4. 分时系统（Time-Sharing）

分时系统的核心机制是：时间片。

做法是把 CPU 时间切成很多很短的小片，轮流分给各个用户或任务。

用户感受到的是：

  • 自己似乎独占了机器
  • 系统响应很快
  • 可以交互式工作

它追求的重点从 ” 吞吐量优先 ” 变成了 ” 响应时间优先 “。

传统分时系统今天还有没有意义？

有，而且意义很大。

虽然今天我们不再用老式终端接大型机，但很多现代系统都继承了分时思想：

  • 桌面操作系统的多任务切换
  • 服务器上的多用户资源共享
  • 远程登录环境
  • 云平台上的资源调度

你甚至可以说：
今天凡是 ” 大家都觉得自己在同时用一台机器 ” 的地方，本质上都还有分时影子。

5. 通用操作系统：分时 + 批处理结合

课件还提到一种很自然的组合：

  • 前台：频繁交互的作业，优先保证响应
  • 后台：时间要求不强的批作业，放在后面慢慢跑

原则是：
分时优先，批处理在后。

这就是很多通用操作系统的思路来源。

6. 实时操作系统（Real-Time OS）

实时系统最关键的参数不是吞吐量，不是平均速度，而是：
时间。

定义上可以概括为：系统必须在严格规定的时间范围内对外部事件作出反应，并完成处理。

两类典型实时系统

  1. 实时过程控制

    • 工业控制
    • 军事控制
    • 自动化流水线

  2. 实时通信 / 信息处理

    • 电讯交换
    • 银行业务
    • 订票系统
    • 股市行情
目标与特点
  • 严格时限内响应
  • 高可靠性
  • 时间正确性极其重要
硬实时 Vs 软实时
  • 硬实时：绝对不能错过时限
    • 例如安全气囊控制
    • 例如某些工业联锁保护
  • 软实时：偶尔错过时限可以接受，但体验会下降
    • 例如直播画面偶尔卡一下
    • 例如音视频播放偶尔抖一下

7. 个人计算机操作系统

特点很直接：

  • 某一时间主要为单用户服务
  • 追求界面友好、应用丰富、使用方便

重点功能：

  • 网络管理
  • 通信
  • 安全
  • 资源共享
  • 网络应用支持

它追求的目标主要是：

  • 相互通信
  • 资源共享

9. 分布式操作系统

分布式系统是建立在网络基础上的，但它比 ” 联网 ” 更进一步。

它的关键点是：

  • 处理和控制是分散的
  • 功能和任务分布在多台机器上
  • 系统能自动做全局任务分配和负载调度
分布式操作系统的几个关键特征

  1. 是一个统一的操作系统

    • 对用户来说，不希望看到底层一堆机器碎片化暴露出来。

  2. 资源进一步共享

    • 共享范围从单机扩展到整个系统。

  3. 透明性

    • 用户最好感觉不到 ” 分布 ” 这件事。

  4. 自治性

    • 多个主机处于平等地位，没有固定主从关系。

  5. 处理能力增强、速度更快、可靠性更高

10. 嵌入式操作系统

嵌入式系统是指运行在各种设备、装置或系统中的软硬件子系统，它们往往不是传统意义上的 ” 电脑 “，而是一个大系统的一部分。

嵌入式操作系统要完成的事是：

  • 对整个嵌入式环境中的资源做统一协调、调度、控制

典型特点：

  • 常处于反应式环境
  • 对处理时间可能有较严格要求

12. 智能卡操作系统

这是很容易被略过去，但课件专门讲到了的点。

智能卡是一种包含 CPU 芯片的卡片设备。

它的约束极其严苛：

  • 存储很小
  • 能耗很低
  • 功能常常很专用

有些智能卡是面向 Java 的：

  • ROM 里有 JVM 解释器
  • Java 小程序被下载到卡上运行
  • 有些卡还能同时处理多个小程序

一旦多个小程序同时存在，就会出现：

  • 调度
  • 资源管理
  • 保护

这些问题，仍然得由智能卡上的操作系统处理。

它怎么通信？

课件里说得很具体：

  • 读写器与智能卡之间通过命令 - 响应对来通信

  • 再把应答信息加密后送回读写器

这个例子特别适合提醒自己：
只要有资源管理、控制、保护和服务，哪怕系统很小，也可能有操作系统。

重点技术拆解：SPOOLing 技术

1. 为什么需要 SPOOLing？

SPOOLing 的英文是 Simultaneous Peripheral Operation On-Line，通常翻成 ” 假脱机技术 ”。

它是为了解决一个很老但很根本的问题：

  • CPU 很快
  • 外设很慢
  • 如果让 CPU 直接等外设，效率会极差

最典型的慢设备就是：

  • 读卡机
  • 打印机

2. 它的核心思想是什么？

核心思想可以压缩成一句话：
利用磁盘作为大缓冲区，把输入、计算、输出拆成相互独立的任务流。

这样就能做到：

  • 输入设备慢慢输它的
  • CPU 快速算它的
  • 输出设备慢慢打它的

三者尽量并行。

3. 输入井、输出井是什么？

为了让数据在不同环节之间顺畅流动，系统会在高速设备（通常是磁盘）上组织出：

  • 输入井：先把外部输入提前搬进来
  • 输出井：先把程序输出暂存起来

这就像把原本必须 ” 面对面交接 ” 的流程，改成了 ” 先存进仓库，再异步取走 “。

    subgraph 低速设备
        A[输入设备]
        B[输出设备]
    end

    subgraph 高速磁盘缓冲区

        D[(输出井)]
    end

        H[输出进程]
    end

    C --> F --> G
    G --> D --> H --> B

4. SPOOLing 的三大效果

  1. 提高 I/O 效率

    • 进程先和高速磁盘交互，而不是直接等慢设备。

  2. 把独占设备改造成共享设备

    • 比如打印请求都先进输出井排队。

  3. 实现虚拟设备效果

    • 每个进程都感觉自己好像拥有一台打印机。

5. 它和 ” 脱机技术 ” 的关系

课件图里把 ” 脱机技术 ” 和 ” 假脱机技术 ” 放在一起看。你可以这样理解：

  • 脱机：把原来在线串行做的事拆到别处做
  • 假脱机：看起来像在线直接使用设备，实际上中间隔着缓冲与排队

6. 今天还有没有 SPOOLing 的影子？

当然有，而且非常常见。

最直接的例子就是：

  • 打印队列
  • 邮件队列
  • 消息队列
  • 日志缓冲
  • 后台批量上传

它们的共性都是：

  • 前台先快速提交请求
  • 后台再慢慢完成真正的 I/O 或外部操作

所以 SPOOLing 不只是历史名词，它其实是一类很有生命力的系统设计模式。

1. Buffer Cache 可以怎么理解？

严格说，这个问题属于文件系统和块设备之间的实现细节，会在后面文件系统部分学得更完整。这里先抓住直觉就够了：

  • 文件系统按 ” 文件、目录、逻辑块 ” 思考
  • 块设备按 ” 磁盘块、扇区 ” 思考
  • 两者之间需要一个缓冲与协调层

Buffer Cache 的作用就是：

  • 把经常访问的磁盘块先缓存在内存里
  • 减少真实磁盘 I/O
  • 平衡文件子系统和块设备之间的速度差
  • 帮助做读写合并与延迟写回

你可以把它理解成 ” 文件系统和磁盘之间的中转仓库 “。

2. 单道批处理和多道批处理一句话怎么区分？

  • 单道：内存里一次只放一个作业，I/O 一等，CPU 就闲着。
  • 多道：内存里同时放多个作业，一个等 I/O，CPU 就去跑另一个。

它的形式变了，但思想还在：

  • 多任务
  • 时间片调度
  • 多用户共享
  • 快速响应

这些都还是今天系统的骨架之一。

1. 基础运行环境：硬件如何支持操作系统？

操作系统不是飘在空中的，它必须站在硬件提供的基础机制上运行。

课件这一部分的关键点是：
操作系统之所以能 ” 管别人 “，前提是硬件先给了它更高权限。

1.1 CPU 和寄存器：程序运行的最小现场

CPU 可以粗略理解为由这些东西构成：

  • 运算器
  • 控制器
  • 一系列寄存器
  • 高速缓存

课件把寄存器分成两类：

用户可见寄存器

机器语言能直接引用，主要包括：

  • 数据寄存器（通用寄存器）
  • 地址寄存器
  • 条件码寄存器

条件码寄存器里保存的是 CPU 运算结果的各种标记位。

控制和状态寄存器

它们用于控制处理器操作，只有在某些特权级下才能访问或修改，典型包括：

  • PC：程序计数器，下一条要执行的指令位置
  • IR：指令寄存器，当前指令
  • PSW：程序状态字，保存处理器状态信息

Example

你可以把 ” 程序运行现场 ” 想成演员演戏时的舞台位置：

  • PC 像 ” 下一句台词该说到哪 ”
  • 通用寄存器像 ” 手里暂时拿着的道具 ”
  • PSW / EFLAGS 像 ” 当前的灯光和舞台规则状态 ”

一旦中断发生，必须先把这些东西保存好，不然演员回来就接不上戏了。

1.2 保护需求：为什么必须有特权级？

从操作系统四大特征出发，系统里有：

  • 并发
  • 共享

一旦有这两件事，系统就必须有：

  • 保护
  • 控制
  • 隔离

所以硬件必须提供一种基本运行机制，把：

  • 操作系统代码
  • 普通用户程序

区分开来。

1.3 处理器状态（Mode）

现代处理器通常把 CPU 状态划成两种、三种或四种。

最常见的两种
  • 内核态（Kernel Mode / Supervisor Mode）
    • 权限最高
    • 可执行特权指令
    • 运行操作系统核心代码
  • 用户态（User Mode）
    • 权限受限
    • 只能执行非特权指令
    • 运行普通应用程序
特权指令 Vs 非特权指令
  • 特权指令：只能操作系统用，用户程序不能直接用
  • 非特权指令：用户程序也可以执行

1.4 X86 与 RISC-V 的特权模式

X86

X86 支持 4 个特权环：

  • R0
  • R1
  • R2
  • R3

从 R0 到 R3，权限逐级降低。

在多数实际系统里：

  • R0 对应内核态
  • R3 对应用户态
  • R1、R2 很少真正使用
RISC-V

RISC-V 常见的三种特权模式：

  • M 模式（机器模式）
  • S 模式（监管模式）
  • U 模式（用户模式）

1.5 EFLAGS / PSW 在这里的重要性

课件专门给了 X86 的 EFLAGS 图，原因不是要你逐位背，而是让你知道：

  • 处理器状态不是抽象口号
  • 它真的编码在寄存器里
  • 中断允许位、陷阱标志、特权相关信息都在里面

也就是说，” 从用户态到内核态 ” 不是文学描述，而是寄存器状态真的变了。

1.6 CPU 状态如何切换？

用户态 → 内核态

唯一途径：

  • 中断
  • 异常
  • 陷入（trap）机制
内核态 → 用户态

要恢复原程序状态，并通过专门的返回机制回去。

课件同时列了几种不同体系结构的指令例子：

  • int
  • trap
  • syscall
  • sysenter / sysexit
  • ecall

你不用在这一讲把每条指令背成百科，但一定要记住：
从低权限进入高权限，必须依赖体系结构定义好的入口。

2. 核心动力：中断与异常机制

课件有一句话非常值得记：
中断对于操作系统的重要性，就像汽车发动机、飞机引擎。

这是因为操作系统不是靠 ” 主循环一直傻跑 ” 维持秩序，而是靠事件来推动执行。

2.1 什么是中断 / 异常？

可以先用一句最通俗的话定义：

中断 / 异常就是 CPU 对某个事件作出的控制流改变。

当事件发生时：

  • CPU 暂停当前执行流
  • 保存现场
  • 转去执行相应处理程序
  • 处理完后再返回原先位置或转去新的位置

这个机制的几个特点：

  • 是自动处理的
  • 往往带有随机性或异步性
  • 需要保存和恢复现场
  • 本质上改变了处理器控制流

2.2 为什么会引入中断和异常？

中断的引入原因

为了支持 CPU 和设备之间的并行操作。

如果没有中断：

  • CPU 启动 I/O 后只能不停轮询等它

有了中断：

  • 设备完成工作后通知 CPU
  • CPU 中间可以去做别的事
异常的引入原因

为了处理 CPU 在执行指令时自身遇到的问题，或者执行某些特定的软件触发行为。

例如：

  • 除零
  • 算术溢出
  • 非法操作数
  • 地址越界
  • 缺页
  • 执行陷入指令（系统调用）

2.3 中断、陷入、故障、终止的分类

这是非常标准的一组分类。

类别原因同步/异步返回行为例子
中断 Interrupt外部设备或其他硬件信号异步返回到下一条指令键盘、时钟、I/O 完成
陷入 Trap程序有意识安排同步返回到下一条指令系统调用、断点
故障 Fault可恢复错误同步返回当前指令重新执行缺页异常
终止 Abort不可恢复严重错误同步不返回严重硬件错误

顺手记一句最实用的话：

  • 中断看来源：外部来的吗？
  • 异常看发生点：是不是执行当前指令时出的问题？

2.4 ECF：异常控制流

课件说这一部分 ” 很重要 “，不是客气话。

ECF 可以理解成：
程序的控制流并不总是沿着你写出来的顺序线性前进，它会被异常事件改道。

为什么理解 ECF 很重要？

因为它直接帮助你理解：

  • 应用程序如何和操作系统交互
  • 软件异常怎么工作
  • 并发的很多现象为什么会发生

2.5 硬件做什么，软件做什么？

这一点是本讲的关键分工。

硬件负责：中断 / 异常响应

也就是：

  • 发现中断或异常请求
  • 捕获它
  • 保存最基本的上下文
  • 把控制权转给指定处理程序
软件负责：中断 / 异常处理

也就是：

  • 识别具体原因
  • 保存更多寄存器现场
  • 执行对应处理逻辑
  • 恢复现场并返回

你可以把它想成：

  • 硬件负责开门、指路、抬担架
  • 软件负责诊断、治疗、收尾

2.6 硬件中断响应过程

课件给了一个很典型的硬件流程：

  • CPU 在每条指令执行周期末尾扫描中断寄存器
  • 如果没有中断，继续执行下一条指令
  • 如果有中断，就把中断信号按规则编码成中断码
  • 再根据中断向量把控制权引到对应处理程序

所以 ” 响应 ” 不是软件在干，而是硬件先完成第一段流程。

中断向量表就像一本 ” 电话簿 ”：

  • 你给我一个中断号
  • 我查表
  • 我知道该去找谁处理

2.9 从设备中断到返回，完整流程怎么走？

课件的小结可以整成下面这条线：

  2. CPU 执行完当前指令后检测到中断。
  3. CPU 判断中断来源，并向相关设备发送确认。
  4. 处理器切到内核态。
  5. 硬件在系统栈中保存关键上下文，如 PC、PSW。
  6. CPU 根据中断码查中断向量表。
  7. 得到处理程序入口地址，把 PC 设成它。
  8. 软件中断处理程序开始运行。
  9. 软件再保存更多现场。
  10. 软件检查设备状态、搬运数据、更新系统状态。
  11. 处理结束，执行返回指令。
  12. 硬件恢复原上下文，CPU 回到原程序继续。

2.10 为什么 Linux 要分上半部和下半部？

课件在中断响应流程页专门提了这个问题。

直觉上理解就够了：

  • 中断一来，系统要尽快响应
  • 但有些工作非常耗时
  • 如果全都在中断入口里一次做完，就会拖很久，影响别的中断和系统整体响应

所以 Linux 把它拆成：

  • 上半部：尽快做最紧急、最必要的事
  • 下半部：把可以延后的工作推迟处理

这本质上还是一个老原则：
关键路径尽量短。

3. IA32 / X86 对中断的具体支持

这一部分最容易让人头大，因为名词突然变多了。但逻辑其实并不复杂：

中断号只是编号，真正把编号变成 ” 跳到哪段代码去执行 ” 的，是 IA32 定义的一套表和描述符。

3.1 PIC / APIC：中断控制器

硬件外设发出的信号，不能直接变成 CPU 能理解的处理入口。

中断控制器（PIC 或 APIC）的工作是：

  • 接收外设中断信号
  • 把它转换成中断向量
  • 触发 CPU 的中断机制

3.2 实模式 Vs 保护模式

实模式

使用的是中断向量表（Interrupt Vector Table）。

特点：

  • 表项直接存放中断服务程序入口地址
  • 地址通常按 ” 段地址左移 4 位 + 偏移地址 ” 形成
  • 不支持 CPU 运行状态切换
  • 中断处理更像普通过程调用
保护模式

使用的是中断描述符表（IDT, Interrupt Descriptor Table）。

特点：

  • 表项不只是一个裸地址
  • 它使用 ” 门描述符（gate descriptor）” 来描述入口
  • 可以结合特权级、段机制、栈切换做更安全的控制

3.3 门描述符有哪些类型？

课件列了四种：

  1. 任务门（Task Gate）

    • 和 TSS 强相关
    • Linux 一般不使用任务门

  2. 中断门（Interrupt Gate）

    • 给出段选择符和偏移量
    • 通过中断门后，系统会自动禁止中断（清 IF）

  3. 陷阱门（Trap Gate）

    • 跟中断门类似
    • 但通过后不会自动关中断

  4. 调用门（Call Gate）

    • 也是一种跨特权调用机制

3.4 IDT、GDT、LDT、IDTR、GDTR 到底是什么关系？

这里推荐按 ” 查表链路 ” 记：

  1. CPU 已经知道当前是哪个中断向量号 i
  2. 通过 IDTR 找到 IDT
  3. 取出 IDT 中第 i 项，也就是中断描述符
  4. 中断描述符里有一个段选择符和偏移量
  5. 段选择符再去 GDT / LDT 里找对应段描述符
  6. 段描述符给出段基址
  7. 用 ” 段基址 + 偏移量 ” 算出真正入口地址
flowchart LR
    A[中断向量号 i] --> B[IDTR]
    B --> C[IDT 中第 i 项]
    C --> D[段选择符]
    C --> E[偏移量]
    D --> F[GDTR]
    F --> G[GDT/LDT 中的段描述符]
    G --> H[段基址]
    H --> I[入口地址 = 段基址 + 偏移量]

3.5 段选择符是什么？

课件图里强调了：段选择符不是地址本身，它更像一个索引凭证。

它通常编码了：

  • 索引值
  • 选择 GDT 还是 LDT
  • 特权级相关信息

所以 ” 门描述符里写的不是最终地址 ” 这点很关键。

3.6 IA32 中断 / 异常的硬件处理过程

课件把这个过程列得很清楚，建议按步骤背逻辑，不要死背术语。

  1. 确定和中断或异常关联的向量 i
  2. 通过 IDTR 找到 IDT
  3. 取第 i 项中断描述符
  4. 再通过 GDTR 找 GDT
  5. 根据中断描述符中的段选择符，找到对应段描述符
  6. 从段描述符里拿到段基址
  7. 做特权级检查
  8. 如果发生特权级变化，就做栈切换
  9. 硬件压栈保存上下文
  10. 如果这是某些异常，还会额外压入硬件错误码
  11. 如果是中断门，清 IF 位
  12. 根据 ” 段基址 + 偏移量 ” 定位处理程序入口，开始执行

3.7 为什么要做特权级检查？

因为不是任何代码都能随便跳进任何高权限代码。

系统必须确认：

  • 当前调用是否合法
  • 是否允许进入这个门
  • 是否允许发生这次跨级切换

这个检查的存在，本质上是在保证：
进入内核是受控的，不是应用随便一跳就能进去。

3.8 为什么会发生栈切换？

如果当前从用户态进入内核态，还继续使用用户栈，会有两个大问题：

  • 不安全
  • 不可靠

所以一旦特权级变化，CPU 必须切到和新特权级对应的栈。

也就是说：

  • 用户代码用用户栈
  • 内核代码用内核栈

这样内核才能在可信环境里保存和处理上下文。

3.9 什么是硬件错误码？

对于某些异常，CPU 会在进入处理程序前自动压入一个错误码。

它的作用是：

  • 帮助软件进一步判断异常的具体细节

你可以把它理解成：
异常不仅告诉你 ” 出事了 “，还顺手给你塞一张简短诊断单。

4. 系统调用：应用程序进入内核的唯一正规接口

4.1 系统调用是什么？

系统调用是：
用户程序在编程时可以调用的操作系统功能。

课件特别强调两点：

  • 它是操作系统提供给编程人员的唯一接口
  • 它会使 CPU 状态从用户态陷入内核态

系统调用类型很多，比如：

  • 进程控制
  • 进程通信
  • 文件操作
  • 目录操作
  • 设备管理
  • 信息维护

每个操作系统通常都会提供几百个系统调用。

4.2 系统调用、库函数、API、内核函数之间的关系

这个地方一定要理顺，否则后面会一直混。

应用程序

写的是：

  • printf()
  • write()
  • fork()
  • open()
C 函数库 / API

很多时候应用不是直接写裸汇编，而是先调用库函数。库函数负责：

  • 包装参数
  • 触发系统调用
  • 把结果转换成更方便的接口形式
系统调用

系统调用是真正跨过 ” 用户态 / 内核态边界 ” 的入口动作。

内核函数

进入内核后，真正干活的是内核里的处理函数，比如 sys_write()。

所以关系最好记成：

应用程序 → C 库/API → 系统调用入口 → 内核函数

4.4 系统调用和普通函数调用的区别

这是高频混淆点。

普通函数调用
  • 通常不改变特权级
  • 不需要切换到内核态
  • 参数和返回按普通调用约定处理
  • 本质上是同一权限空间里的控制转移
系统调用
  • 会触发特权级切换
  • 依赖体系结构定义的陷入机制
  • 要保存和恢复更多上下文
  • 要经过内核入口、查表分派、返回路径

一句话总结：
函数调用是在屋里走门，系统调用是过安检进机房。

5. 系统调用机制是怎么设计出来的？

课件把系统调用机制设计概括成 4 个关键点：

  1. 选择一条特殊指令

    • 也就是陷入指令 / 访管指令
    • 用来触发用户态到内核态的切换

  2. 系统调用号和参数

    • 每个系统调用预先分配一个编号

  3. 系统调用表

    • 用来把调用号映射到对应的服务例程

  4. 查表分派执行

    • 进入总入口后，根据调用号查表，转到具体内核函数

这其实就是一种很典型的 ” 机制与策略分离 ”：

  • 硬件 + 入口机制负责 ” 怎么进来 ”
  • 调用号 + 查表负责 ” 该调用谁 ”
  • 具体内核函数负责 ” 怎么干活 “

5.1 参数怎么传给内核？

课件列了 3 种常见方法：

  1. 陷入指令自带参数

    • 但指令长度有限，能带的参数很少

  2. 通过通用寄存器传参

    • 很常见
    • 但寄存器个数有限

  3. 在内存中开专用堆栈区传参

    • 适合更多参数

很多系统会组合使用这些方法。

第一步：栈切换

  • 从用户栈切到内核栈

CPU 会从 TSS（任务状态段） 中装入新的 SS:ESP，指向内核栈。

第二步：硬件压栈

用户栈相关信息会被保存下来，用于以后返回，包括：

  • 用户栈 SS:ESP
  • EFLAGS
  • 用户态 CS
  • EIP
第三步：状态位处理
  • 把 EFLAGS 压栈
  • 复位 TF
  • IF 保持不变
第四步：查 IDT
  • 用 128 作为索引查 IDT
  • 找到对应门描述符
  • 把门里的段选择符装入 CS
第五步：定位 system_call() 入口
  • 段描述符里的基地址
  • 加上门描述符里的偏移量
  • 得到 system_call() 第一条指令地址

  • 系统调用总入口
  • 它和 IDT 中第 128 号门直接对应
sys_call_table
  • 系统调用号 → 内核处理函数 的映射表
  • 例如：
    • 3 → sys_read
    • 4 → sys_write
ret_from_sys_call
  • 系统调用执行完之后，不是直接 ” 啪一下回用户程序 ”
  • 而是先转到返回路径例程 ret_from_sys_call
  • 在这里做返回前的收尾检查，再恢复现场回用户态

最推荐你这样串起来记：

flowchart LR
    A[int 0x80] --> B[IDT 第 128 项]
    B --> C[system_call 总入口]
    C --> D[sys_call_table 查表]
    D --> E[具体内核函数 比如 sys_write]
    E --> F[ret_from_sys_call]
    F --> G[恢复现场 返回用户态]

6.10 SAVE_ALL 在干什么？

课件给了 Linux 里的 SAVE_ALL 宏，它会把寄存器按顺序压栈：

  • %es
  • %ds
  • %eax
  • %ebp
  • %edi
  • %esi
  • %edx
  • %ecx
  • %ebx

然后把数据段寄存器切到内核数据段。

它的本质作用就是：

  • 把执行现场尽量完整地保存下来
  • 让后续内核代码可以安全运行

6.11 system_call() 的核心逻辑

课件给出的片段可以概括成：

  1. 先把系统调用号压栈
  2. 执行 SAVE_ALL
  3. 检查 EAX 是否是合法系统调用号
  4. 如果非法：
    • 把返回值设成 -ENOSYS
    • 跳到 ret_from_sys_call
  5. 如果合法：
    • 用 EAX 索引 sys_call_table
    • 调用对应内核函数
  6. 把内核函数返回值写回保存现场中的 EAX
  7. 跳到 ret_from_sys_call

6.12 ret_from_sys_call 在返回前做什么？

这也是实现层非常关键的细节。

课件给的逻辑大概是：

  1. 关中断
  2. 检查 need_resched
    • 如果需要重新调度，就先去调度
  3. 检查 sigpending
    • 如果有待处理信号，就先处理信号
  4. 最后 RESTORE_ALL
    • 弹栈恢复现场
    • 返回用户态

这说明一个很重要的事实：

系统调用完成 ≠ 立刻原样返回。

它还要先确认：

  • 现在是不是应该切换进程

  • 返回前是否还需要做额外清理

7. 为什么后来又有 sysenter/sysexit 和 syscall/sysret？

课件抛了一个追问：

  • Pentium II 300 之后，X86 为什么又提供了 sysenter/sysexit？
  • X86-64 又为什么有 syscall/sysret？

这一讲不用把这些指令的细节背满，先抓住动机就够：

int 0x80 / iret 的路径比较通用，但开销不算小。

后来处理器为了让系统调用更快，引入了更专用、更轻量的快速系统调用指令：

  • 32 位里：sysenter / sysexit
  • 64 位里：syscall / sysret

所以它们并存，不是因为前者错了，而是因为：
系统调用太常用了，值得专门优化入口和返回路径。

8. 系统调用小结：定义、作用与准备条件

8.1 这东西到底是什么？

课件最后的小结里，系统调用的本质被压缩成几句话：

  • 它是用户程序调用操作系统子功能的方式
  • 它是一种特殊的过程调用
  • 它由特殊机器指令实现
  • 它是编程人员访问操作系统功能的唯一正规接口
  • 它让 CPU 从用户态进入内核态

8.2 它能做什么？

通过系统调用，程序可以：

  • 动态申请和释放资源
  • 完成与硬件相关的工作
  • 控制程序执行
  • 请求内核提供文件、进程、设备等服务

8.3 完成一次系统调用，需要哪些准备？

课件把条件分成了静态和动态两类，这是很好的整理方式。

静态准备

这些是系统事先就必须具备的：

  • 封装好的库函数 / API
  • 访管指令与陷入机制
  • 编译器对调用约定的支持
  • 操作系统完成初始化
  • 系统调用编号规则
  • 参数传递约定
  • 系统调用表
动态准备

这是调用发生时系统实际要做的：

  • 陷入内核
  • 进入总入口程序
  • 保存现场（压栈）
  • 查表分派
  • 执行具体服务例程
  • 执行返回路径

9. 中断发生后，操作系统低层到底做了哪些步骤？

课件最后给了一个非常适合考试和口头回答的 8 步总结。

  1. 硬件压栈：保存程序计数器等关键上下文
  2. 硬件从中断向量装入新的程序计数器等
  3. 汇编语言过程保存寄存器值
  4. 汇编语言过程设置新的堆栈
  5. C 语言中断服务程序运行
  6. 调度程序决定下一个运行进程
  7. C 语言过程返回到汇编代码
  8. 汇编过程开始运行新的当前进程

这一串步骤特别重要，因为它把：

  • 硬件
  • 汇编入口
  • C 处理程序
  • 调度
  • 返回

连成了一条完整路径。

你会发现，这正是 ” 中断驱动操作系统 ” 的真实展开版。

10. 操作系统初始化与中断/异常有什么关系？

这个问题课件没有单独长篇展开，但前后信息已经足够拼出来。

初始化阶段至少要完成几件事：

  • 为各种中断 / 异常准备好处理程序
  • 设置中断向量表 / IDT
  • 准备好系统栈、内核入口、段描述符等环境
  • 配置系统调用入口（比如 0x80 → system_call）

所以初始化的本质是：
先把 ” 出事时该怎么接管 ” 的路修好，系统才敢真正跑起来。

11. 内核的进入与退出：把全图串起来

课件最后给了一张 ” 内核的进入与退出 ” 总图，特别适合用来收束这一讲。

你可以这样理解：

进入内核的几条典型路径

  1. 系统调用

    • 应用主动请求服务

  2. 异常

    • 代码执行出错或需要特殊处理

  3. 中断

    • 外设发事件，内核被动响应

内核内部会发生什么

  • 查中断向量表
  • 进入异常服务例程或系统调用实现
  • 调用设备驱动或内核子系统
  • 完成处理、更新状态

退出内核前会做什么

  • 恢复现场
  • 必要时调度
  • 必要时处理信号
  • 最终返回用户态

也就是说，应用程序平时运行在用户态，但随时可能因为：

  • 主动请求服务
  • 代码执行出错
  • 外设事件到来

而进入内核。

这就是 ” 内核进入与退出 ” 的主旋律。

12. 机制与策略分离

这是两讲里都隐约贯穿的设计思想，第二讲把它明确点出来了。

什么是机制？

机制回答的是：
能不能做、靠什么做。

例如：

  • 中断 / 异常硬件响应机制
  • 压栈保存现场
  • 查向量表
  • 通过陷入指令进入内核
  • 系统调用表查表分派

这些东西强调的是：

  • 基础能力
  • 可重复使用的底层框架
  • 相对固定的实现方式

什么是策略？

策略回答的是：
具体怎么做、先做谁、按什么规则做。

例如：

  • 中断到来后先处理谁
  • 调度算法怎么选
  • 什么时候重调度
  • 哪些工作放上半部，哪些放下半部

为什么要分离？

因为底层能力不应该每换一个需求就重造一遍，而上层规则又必须能灵活调整。

一句话说就是：

  • 机制提供能力
  • 策略决定使用方式

这也是操作系统设计里一个非常高频、非常有味道的思想。

本讲小结

这一讲最重要的，不是多记几个缩写，而是把整条逻辑链看清：

  1. 操作系统要想管理整个系统，必须先有硬件提供的特权级支持。
  2. 用户态程序不能直接碰硬件，进入内核的正规方式是中断 / 异常 / 陷入机制。
  3. 中断和异常共同构成 ECF，也就是异常控制流。
  4. 硬件负责响应和初始转移，软件负责真正处理和收尾。
  5. IA32 用 IDT、门描述符、特权级检查和栈切换把 ” 进入内核 ” 这件事变成了一条受控路径。
  6. 系统调用是应用请求内核服务的唯一正规接口。
  7. 在 Linux x86 中，int 0x80 → IDT 第 128 项 → system_call → sys_call_table → 具体内核函数 → ret_from_sys_call → 返回用户态 是最关键的一条实现链。
  8. 中断发生后，真正的流程会跨越硬件、汇编入口、C 处理程序、调度与返回路径。
  9. 机制与策略分离，是理解操作系统设计的一把总钥匙。

操作系统不是 ” 一个一直在前台跑的大程序 “，而是一个依靠特权机制和异常控制流，在关键时刻接管机器、完成处理、再把控制权交还出去的系统。
