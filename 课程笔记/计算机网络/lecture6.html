计算机网络：第六讲 应用层

Summary

这一讲是整个计网里最接地气的一章——应用层。你天天在用的浏览器、邮箱、微信、迅雷下载，底层全是这些东西。核心就三块：Socket 编程（应用怎么调传输层）、各种应用协议（HTTP、FTP、SMTP、SNMP 等怎么交互）、P2P（没中心服务器怎么传文件）。考试重点集中在协议特点对比、端口号、报文结构、各种”易混概念”（比如 FTP 两个连接、HTTP 有无状态、POP3 vs IMAP）。把这些理清楚，这章基本稳了。

一、应用层概述

1.1 应用层的基本位置

应用层直接面向网络应用。网络应用程序运行在端系统上，通过传输层、网络层、数据链路层和物理层所提供的能力完成跨主机的数据交换。互联网中的用户应用通常不运行在网络核心设备上，而只运行在主机中。这种分工的意义在于：网络核心负责通用的数据转发，应用逻辑由端系统承担，从而降低新应用部署的门槛。

flowchart TB
    A[应用进程] --> B[应用层协议]
    B --> C[传输层 TCP/UDP]
    C --> D[网络层 IP]
    D --> E[数据链路层]
    E --> F[物理层]
    F --> G[另一台主机]

应用层协议需要规定以下内容：

  • 消息类别，如请求与响应。
  • 消息语法，即报文由哪些字段构成、字段如何划分。
  • 消息语义，即各字段的含义。
  • 通信规则，即何时发送请求、何时返回响应、双方如何配合。

从开放性看，应用层协议可分为开放协议和专用协议。开放协议通常由 RFC 定义，强调互操作性；专用协议则服务于特定系统或厂商实现。

1.2 应用程序体系结构

应用程序体系结构描述的是应用进程之间如何组织，而不是网络层如何路由。

（1）客户—服务器体系结构

客户—服务器结构中，服务器长期运行，等待请求到达；客户机主动发起通信。服务器通常具有固定 IP 地址或稳定的服务入口，客户机则可以间歇上线、地址动态变化。客户机之间通常不直接通信，而是经由服务器间接交换信息。

flowchart LR
    C1[客户机 1] --> S[服务器集群]
    C2[客户机 2] --> S
    C3[客户机 3] --> S
    S --> C1
    S --> C2
    S --> C3

该结构适合于资源集中管理、统一身份控制和一致性要求较高的场景，如 Web、搜索、电商和邮件服务。

（2）对等体系结构 P2P

P2P 中没有永久中心服务器，主机既可能请求服务，也可能向其他主机提供服务。节点连接具有断续性，地址也可能变化。其关键优点是自扩展性：系统中每增加一个对等方，不仅增加负载，也增加总体服务能力。代价是管理复杂度更高，节点发现、稳定性控制和资源一致性都更困难。

flowchart LR
    P1[Peer 1] <--> P2[Peer 2]
    P2 <--> P3[Peer 3]
    P3 <--> P4[Peer 4]
    P4 <--> P1
    P1 <--> P3

1.3 进程通信与 Socket

应用层中的通信本质上是不同主机上进程之间的通信。同一主机上的进程间通信由操作系统提供 IPC 机制；不同主机上的进程则通过交换报文实现通信。

在通信角色上：

  • 主动发起会话的一方称为客户进程。
  • 被动等待连接建立的一方称为服务器进程。
  • 在 P2P 系统中，同一进程可同时充当客户和服务器。

进程通过 Socket 接入网络。Socket 是应用进程与操作系统协议栈之间的接口。应用开发者控制应用进程及其通过 Socket 发出的数据；底层协议封装、传输和转发由操作系统控制。

Intuition

Socket 翻译成”套接字”确实有点拗口，你就把它想象成墙上的电话插座。你的应用程序（电话机）要打电话，得先插到插座上；插座另一边连着电话线（TCP/UDP），电话线再连到外面的电话网络（IP）。应用进程只管对着电话机说话，线路怎么铺、信号怎么转，操作系统全包了。

flowchart LR
    A[应用进程] --> B[Socket]
    B --> C[TCP/UDP]
    C --> D[IP]
    D --> E[链路与物理介质]

1.4 进程寻址

主机的 IP 地址只能标识主机，不能唯一标识主机中的某个应用进程。进程寻址依赖二元组：

进程标识=(IP 地址, 端口号)

例如，HTTP 服务器通常使用端口 80，邮件服务器常使用端口 25。向某个 Web 服务器发送请求时，必须同时指定目标 IP 地址和目标端口号。

1.5 应用对传输服务的需求

不同应用对下层传输服务的要求不同，主要体现在以下方面：

  • 可靠性：文件传输、电子邮件、Web 页面通常要求无差错交付；实时音视频可容忍部分丢失。
  • 实时性：交互式语音、视频会议和联机游戏要求低时延。
  • 吞吐量：多媒体应用需要最低数据率保障；文本类业务通常是弹性流量。
  • 安全性：包括保密性、完整性与对端认证。

1.6 TCP 与 UDP 面向应用的服务差异

TCP 提供的能力

  • 可靠传输
  • 流量控制
  • 拥塞控制
  • 面向连接

TCP 不直接提供定时保证、最小带宽保证和内建安全性。

UDP 提供的能力

  • 无连接
  • 尽力而为的数据报传输

UDP 不提供可靠性、流量控制、拥塞控制、时延保证和安全性。

之所以仍然需要 UDP，是因为某些应用更关心低时延、低开销与简单交互。例如 DNS 查询通常报文短、请求频繁，若每次查询都建立 TCP 连接，会引入额外时延和服务器负担。

Intuition

选 TCP 还是 UDP，你就问自己：这东西丢一个字节能忍吗？

  • 传文件、发邮件、看网页——丢一个字节都不行，必须 TCP，可靠交付是底线。
  • 打语音、看视频、玩 FPS——卡一下比丢一帧更难受，UDP 快就完了，偶尔丢包大脑会自动”脑补”。

所以 DNS 明明很重要，为什么也用 UDP？因为查询报文就几十个字节，建个 TCP 连接三次握手反而比传输本身还慢。除非报文特别大（比如 DNSSEC），才会切到 TCP。

Trap

考试最容易挖坑的地方：TCP 提供可靠传输，但不保证时延和带宽。很多人误以为 TCP 啥都包，其实它只管”到不到”，不管”多快到”。另外，TCP 的”面向连接”和 HTTP 的”无状态”是两码事，别混了——前者是传输层有没有建立连接，后者是应用层记不记得你。

1.7 应用与传输协议的对应关系

典型互联网应用与底层传输协议之间的关系如下：

  • 电子邮件：SMTP，底层通常为 TCP。
  • 远程终端：Telnet，底层为 TCP。
  • Web：HTTP，底层为 TCP。
  • 文件传输：FTP，底层为 TCP。
  • 流媒体：可使用 TCP 或 UDP。
  • 互联网电话：常结合 UDP 与应用层实时协议。

1.8 应用层安全：SSL 的位置

TCP 与 UDP 本身都不提供加密。若直接将明文送入 Socket，数据在互联网上按明文形式传输。为在应用通信中引入保密性、完整性和对端认证，可在应用层与 TCP 之间引入 SSL 所提供的安全会话能力。

flowchart LR
    A[应用进程] --> B[SSL]
    B --> C[TCP]
    C --> D[互联网]

这里的关键边界是：SSL 不改变 IP 和 TCP 的基本工作方式，而是在应用与 TCP 之间为字节流增加安全保护。

二、Socket 编程

2.1 系统调用与应用编程接口

应用程序不能直接操作协议栈内部资源，而是通过系统调用向操作系统请求服务。系统调用是应用进程控制权与操作系统控制权转换的接口。为了便于程序开发，系统调用接口进一步抽象为应用编程接口 API。

常见网络编程接口包括：

  • Berkeley Socket
  • Windows Socket
  • TLI

其中，Socket 接口最为常用。

2.2 套接字的作用

当应用进程需要使用网络通信时，首先请求操作系统创建套接字。操作系统为该套接字分配通信所需资源，并用套接字描述符进行标识。随后，应用进程通过该描述符执行发送、接收、连接、关闭等操作。

套接字内部通常关联如下信息：

  • 协议族
  • 服务类型，如字节流或数据报
  • 本地 IP 地址
  • 本地端口
  • 远地 IP 地址
  • 远地端口

2.3 基于 TCP 的套接字交互

TCP 是面向连接的，因此应用层编程一般分为三个阶段：连接建立、数据传送、连接释放。

（1）服务器端基本流程

  1. socket：创建套接字。
  2. bind：将本地 IP 地址和端口绑定到套接字。
  3. listen：将套接字置为被动监听状态。
  4. accept：接受到来的连接请求，并创建新的连接套接字。
  5. recv/send：在连接上收发数据。
  6. close：关闭连接套接字。

（2）客户端基本流程

  1. socket：创建套接字。
  2. connect：主动发起连接建立。
  3. send/recv：收发数据。
  4. close：关闭连接。
sequenceDiagram
    participant C as 客户端
    participant S as 服务器监听套接字
    participant CS as 服务器连接套接字
    C->>S: socket
    S->>S: socket + bind + listen
    C->>S: connect
    S->>CS: accept，创建新套接字
    C->>CS: send(request)
    CS->>C: send(response)
    C->>CS: close
    CS->>C: close

（3）并发服务器

并发服务器需要同时处理多个连接。监听套接字只用于接受新连接，accept 返回后，操作系统为该连接创建新的连接套接字，后续数据交换在新套接字上进行。这样，原监听套接字仍可继续接受其他客户请求。

flowchart TB
    A[监听套接字] -->|accept| B[连接套接字 1]
    A -->|accept| C[连接套接字 2]
    A -->|accept| D[连接套接字 3]

这里的工程意义在于：服务器可以把”接入”与”会话处理”分离，提升并发处理能力。

2.4 基于 UDP 的套接字交互

UDP 不需要建立连接。发送方在每个数据报中显式指定对方的 IP 地址和端口号，接收方从到达的数据报中获得发送方地址信息。

UDP 服务器端典型流程

  1. socket
  2. bind
  3. recvfrom
  4. sendto
  5. close

UDP 客户端典型流程

  1. socket
  2. sendto
  3. recvfrom
  4. close
sequenceDiagram
    participant C as UDP 客户端
    participant S as UDP 服务器
    C->>S: sendto(数据报, 目标IP:端口)
    S->>C: sendto(响应, 客户端IP:端口)

UDP 的编程接口更直接，但上层应用必须自行处理丢包、失序、重传或超时等问题。

Trap

TCP 编程里服务器端有两个套接字：一个监听套接字专门用来”揽客”（accept），另一个连接套接字才是真正跟客户端聊的。很多初学者会搞混，以为 accept 之后还在原来的套接字上通信。UDP 则没这烦恼，因为它压根不建立连接，来一个数据报处理一个。

2.5 TCP 与 UDP 套接字编程的边界

二者的本质差异不在于是否能”发送数据”，而在于应用是否依赖连接状态与可靠字节流：

  • 若应用需要可靠、有序、面向连接的会话，应使用 TCP Socket。
  • 若应用需要简单、低开销、报文级交互，且可容忍不可靠性，应使用 UDP Socket。

三、文件传送协议 FTP

3.1 FTP 要解决的问题

文件传输表面上看是将一个主机中的文件复制到另一主机，实质上却要跨越不同操作系统、文件格式、目录结构、命名规则、存取控制机制与命令接口的差异。FTP 的目标就是减少或消除这些异构系统之间的文件处理不兼容性。

3.2 FTP 的基本特点

FTP 使用客户—服务器方式，底层采用 TCP 的可靠传输服务。一个 FTP 服务器通常由主进程和多个从属进程组成：主进程负责接受新请求，从属进程负责处理单个客户请求。

服务器主进程的工作主线为：

  1. 打开熟知端口 21。
  2. 监听客户的连接请求。
  3. 为新会话启动从属进程。
  4. 请求处理完成后从属进程结束，主进程继续等待下一次请求。

3.3 FTP 的双连接机制

FTP 的核心特征是同时使用两条 TCP 连接：

  • 控制连接：用于传送命令和响应，整个会话期间通常保持打开。
  • 数据连接：用于实际传送文件或目录数据，往往按需建立、传送结束后关闭。

Intuition

FTP 的双连接特别像双向车道+应急车道的组合：

  • 控制连接（端口 21） 就是”对讲机频道”，全程开着，你发”我要传这个文件”、“切换目录”这些命令，都走这条线。
  • 数据连接（端口 20 或协商端口） 是”货运通道”，真正有文件要传了才临时打开，传完就关。

这样设计的好处是：就算大文件正在传输，你也能通过对讲机喊”停！“，因为它俩不走同一条路。

flowchart LR
    C1[FTP 客户控制进程] <--控制连接 TCP:21--> S1[FTP 服务器控制进程]
    C2[FTP 客户数据进程] <--数据连接 TCP:20/协商端口--> S2[FTP 服务器数据进程]

控制连接与数据连接分离的原因有三：

  1. 协议结构更清晰，命令控制与数据搬运职责分离。
  2. 数据传输期间控制连接仍可继续工作，例如终止传输。
  3. 两类连接使用不同端口，避免控制信息与数据流混淆。

Trap

FTP 的端口是考试重灾区，记住这组数字：

  • 控制连接：服务器端端口固定是 21，全程保持。
  • 数据连接：服务器端端口通常是 20（主动模式），但也可能是双方协商的某个临时端口（被动模式）。

很多同学只记得”20 和 21”，但搞不清楚谁对应谁。口诀：21 管命令（控制），20 管数据。另外，两个连接都是 TCP，不是 UDP！

3.4 FTP 的工作方式

当客户与服务器建立控制连接后，客户通过该连接发送各种传输命令。服务器在接收到文件传输请求后，再建立相应的数据连接，实际的数据传送由数据传送进程完成。一次数据传送结束后，数据连接关闭，但控制连接可继续保留，以便执行后续命令。

sequenceDiagram
    participant C as FTP 客户
    participant S as FTP 服务器
    C->>S: 建立控制连接（TCP 21）
    C->>S: 发送登录/目录/传输命令
    S->>C: 返回控制响应
    C->>S: 请求文件传输
    S->>C: 建立数据连接
    S->>C: 通过数据连接传送文件
    S->>C: 数据连接关闭

3.5 FTP 与 NFS、TFTP 的边界

FTP 与 NFS

FTP 关注的是”传文件”，即在网络两端复制整个文件。NFS 的思路不同，它允许应用进程打开远地文件，并在特定位置上读写，只在网络上传送被修改的数据片段，因此更像远程文件访问而不是整体文件复制。

FTP 与 TFTP

TFTP 是极简文件传输协议，使用 UDP 而不是 TCP。它只支持简单的读写文件，不支持复杂交互、目录列举和用户身份鉴别，因此常用于实现简单、资源受限的场景。

3.6 TFTP 的停止等待机制

TFTP 的数据传输单位为文件块，每个数据块通常含 512 字节有效数据，块编号从 1 开始。其工作方式类似停止等待协议：

  • 发送方发送一个数据块后等待确认。
  • 若超时未收到确认，则重发该块。
  • 若确认方在超时内未收到下一个块，也可重发确认。
sequenceDiagram
    participant C as TFTP 客户
    participant S as TFTP 服务器
    C->>S: 读/写请求（端口 69）
    S->>C: DATA(块1)
    C->>S: ACK(块1)
    S->>C: DATA(块2)
    C->>S: ACK(块2)

文件结束有两种判定方式：

  • 若最后一个数据块长度小于 512 字节，则该块即为最后一块。
  • 若文件长度恰好是 512 的整数倍，还需再发送一个数据区为空的数据块作为结束标志。

Trap

TFTP 这个”512 字节整数倍”的结束判定特别容易考。记住：正常结束是最后一块小于 512B；如果刚好整除，必须再发一个空数据块（只有头部，数据区长度为0）当”结束哨兵”。这个设计和很多协议的”长度字段直接告知”不太一样，属于 TFTP 的特（keng）色（dian）。

四、远程终端协议 Telnet

4.1 Telnet 的目标

Telnet 提供远程终端服务，使用户能够通过网络登录到远地主机，并像直接使用远地终端一样进行键盘输入和屏幕交互。其底层使用 TCP，以获得可靠传输。

4.2 客户—服务器结构

Telnet 采用客户—服务器方式。用户本地主机运行 Telnet 客户进程，远地主机运行 Telnet 服务器进程。服务器一般由主进程监听请求，再为每个连接建立从属处理进程。

4.3 网络虚拟终端 NVT

Telnet 面临的主要问题是异构主机之间终端表示方式不同。为解决格式差异，Telnet 在通信中引入网络虚拟终端 NVT 作为统一中间表示。

Intuition

NVT 其实就是个翻译官。假设 A 主机用 Windows，B 主机用 Linux，两边键盘编码、换行符、控制字符全不一样。Telnet 不让他们直接对话，而是规定一种”普通话”（NVT 格式）：

  • A 先把本地话翻译成普通话发给 B；
  • B 收到普通话再翻译成本地话。

这样不管有多少种系统，只要各配一个”本地话↔普通话”的翻译器就行，不用两两组合配适配器了。

flowchart LR
    A[本地终端格式] --> B[Telnet 客户端转换]
    B --> C[NVT 格式]
    C --> D[Telnet 服务器转换]
    D --> E[远地系统格式]

工作过程如下：

  • 用户击键由客户端转换为 NVT 格式后发送。
  • 服务器将 NVT 转换为远地主机可理解的格式。
  • 远地主机输出再被转换回 NVT，经网络发回。
  • 本地客户端将 NVT 恢复成本地显示格式。

NVT 的意义在于把”不同主机的终端差异”压缩为”本地格式 ↔ NVT ↔ 远地格式”的两次转换，从而避免为每种主机组合单独设计适配规则。

4.4 Telnet 的边界

Telnet 的核心能力是远程字符终端接入，不负责图形界面展示，也不提供安全加密。随着 PC 与更安全的远程登录方案普及，Telnet 的实际使用已大幅减少，但其作为典型的远程终端协议，仍有助于理解交互式应用如何建立在 TCP 可靠字节流之上。

五、超文本传输协议 HTTP 与万维网

5.1 万维网的基本性质

万维网不是一种独立的物理网络，而是构建在互联网之上的分布式超媒体信息系统。其核心特征是：分布在不同主机上的文档通过链接相互关联，用户可按需访问所需信息。

flowchart LR
    A[页面 A] --> B[页面 B]
    A --> C[页面 C]
    B --> D[页面 D]
    C --> E[页面 E]

超文本是以链接组织的文本信息系统，超媒体则在超文本基础上进一步支持图像、声音、视频等多媒体内容。

5.2 万维网需要解决的四个基本问题

万维网的建立依赖四类基本机制：

  1. 对象标识：使用 URL 唯一定位分布在互联网上的资源。
  2. 交互协议：使用 HTTP 在浏览器与服务器之间传送请求和响应。
  3. 文档表示：使用 HTML 描述页面内容及链接关系。
  4. 信息检索：依靠搜索引擎帮助用户定位信息。

5.3 URL

URL 用于描述资源的位置及访问方法，其一般形式为：

<协议>://<主机>:<端口>/<路径>

其中：

  • <协议> 指明访问该资源所使用的协议，如 HTTP 或 FTP。
  • <主机> 是资源所在主机的域名。
  • <端口> 指服务进程端口；若使用默认端口，可省略。
  • <路径> 指定具体资源在服务器上的位置。

对 HTTP 而言，其默认端口为 80，因此 URL 中常省略端口。若省略路径，则通常表示访问站点主页。

Example

来，拆一个真实的 URL 给你看看：

https://www.example.com:8080/path/to/file.html
  • https → 协议，表示用 HTTP + SSL/TLS 加密传输
  • www.example.com → 主机域名，DNS 会把它解析成 IP
  • 8080 → 端口，不是默认的 80，所以显式写出来
  • /path/to/file.html → 路径，服务器上文件的位置

再比如 ftp://ftp.example.com/pub/README.txt，协议换成了 FTP，默认端口 21 被省略了。

5.4 Web 的基本工作方式

浏览器作为 HTTP 客户端，服务器作为 HTTP 服务器。用户输入 URL 或点击页面中的超链接后，浏览器与服务器建立 TCP 连接，发送 HTTP 请求报文，服务器返回 HTTP 响应报文，随后连接关闭或保持一段时间供后续请求复用。

sequenceDiagram
    participant B as 浏览器
    participant W as Web 服务器
    B->>W: 建立 TCP 连接
    B->>W: HTTP 请求报文
    W->>B: HTTP 响应报文
    W-->>B: 关闭或保持 TCP 连接

5.5 HTTP 的定位与特点

HTTP 是面向事务的应用层协议，其直接任务是完成一次次对象请求与响应交互。HTTP 使用 TCP 作为运输层，以保证报文可靠到达。

HTTP 的主要特点包括：

  • 使用 TCP，获得可靠传输。
  • 协议本身围绕事务展开，请求—响应关系明确。
  • HTTP/1.0 是无状态协议，即服务器默认不保留前一事务的应用状态。

“HTTP 使用了面向连接的 TCP”与”HTTP 自身无状态”并不矛盾：前者描述传输层是否建立连接，后者描述应用层是否保留事务状态。

Intuition

HTTP 的”无状态”三个字特别抽象，你就把它想成一次性交易：

  • 你去便利店买水，付完钱拿了水就走。下次再去，店员根本不认识你，每次都得重新说”我要一瓶水”。
  • 这就是 HTTP/1.0 的原始形态：服务器处理完你的请求，“脑子”就清零了，不记得你刚才来过。

那怎么实现”记住登录状态”呢？靠 Cookie（后面会讲）。Cookie 相当于你每次去便利店，都主动出示一张会员卡，店员一扫就知道你是谁——但店员自己（服务器）还是不记得，全靠你带卡。

Trap

“HTTP 使用面向连接的 TCP” 和 “HTTP 是无状态协议”，这俩完全不矛盾，但考试特别爱把它们放一起混淆你：

  • TCP 面向连接 → 传输层的事，三次握手建连，保证可靠。
  • HTTP 无状态 → 应用层的事，服务器不记前一次请求。

就好比你打电话（TCP 连接通了），但每次说话都当对方失忆（HTTP 无状态）。电话线连着呢，但对方不记事儿。

5.6 请求一个对象的时延构成

对于非持续连接，请求一个对象至少涉及：

  1. 建立 TCP 连接所需的一个往返时间 RTT。
  2. 发送 HTTP 请求并收到响应首字节所需的一个 RTT。
  3. 传输对象本身的时间。

因此，请求对象的总耗时可概括为：

2⋅RTT+对象传输时间

5.7 持续连接

HTTP/1.1 默认支持持续连接。服务器发送完响应后，不立即关闭 TCP 连接，而是在一段时间内保持该连接，以便同一客户继续请求同一服务器上的其他对象。

持续连接的价值在于减少频繁建立 TCP 连接的开销，提高页面中多个对象的传输效率。

（1）非流水线方式

客户只有在收到前一个响应后，才发送下一个请求。该方式已经节省了反复建连的代价，但服务器在对象之间仍可能空闲等待。

（2）流水线方式

客户无需等待前一个响应全部到达，即可连续发送多个请求。服务器可按顺序连续返回多个响应，从而进一步减少空闲时间，提高下载效率。

sequenceDiagram
    participant B as 浏览器
    participant S as 服务器
    B->>S: 请求对象1
    B->>S: 请求对象2
    B->>S: 请求对象3
    S->>B: 响应对象1
    S->>B: 响应对象2
    S->>B: 响应对象3

Trap

非持久连接、持久连接非流水线、持久连接流水线，这三兄弟的区别一定要分清：

方式特点耗时（请求 n 个对象）
非持久连接（HTTP/1.0）每个对象都要新建 TCP 连接约 2n⋅RTT
持久连接 + 非流水线（HTTP/1.1 默认）一个 TCP 连接串行请求约 (n+1)⋅RTT
持久连接 + 流水线一个 TCP 连接，请求连续发约 2⋅RTT 多一点

核心记忆点：HTTP/1.1 默认持久连接，不是非持久！很多同学还以为是 1.0 的行为。

5.8 代理服务器与 Web 缓存

代理服务器又称 Web 高速缓存。它位于浏览器与源点服务器之间，代表浏览器发起请求，并将近期请求和响应缓存在本地。

Intuition

代理服务器/缓存就是个小区快递驿站。你要买东西（访问网页），先问驿站有没有现货（缓存命中）。有的话直接拿走，快得很；没有的话驿站帮你去商家（源服务器）订货，拿到后自己留一份，再给你一份。下次有人要同样的东西，驿站直接给，不用再跑商家了——既省时间又减轻商家压力。

flowchart LR
    B[浏览器] --> C[代理服务器/高速缓存]
    C -->|命中| B
    C -->|未命中| O[源点服务器]
    O --> C
    C --> B

其工作流程为：

  1. 浏览器先向代理服务器发送 HTTP 请求。
  2. 若缓存命中，代理直接返回本地副本。
  3. 若缓存未命中，代理代表浏览器向源点服务器请求资源。
  4. 源点服务器返回对象。
  5. 代理将对象存入本地，并再返回给浏览器。

缓存的主要意义在于：

  • 减少访问源站的时延。
  • 降低瓶颈链路上的流量压力。
  • 减轻源点服务器负担。

5.9 HTTP 报文结构

HTTP 报文分为请求报文和响应报文两类，都是面向文本的结构化 ASCII 字符串。

（1）请求报文

请求报文由三部分构成：

  1. 请求行：方法、URL、版本。
  2. 首部行：若干首部字段。
  3. 实体主体：可选，常用于提交数据。
flowchart TB
    A[请求行: 方法 URL 版本] --> B[首部行]
    B --> C[空行 CRLF]
    C --> D[实体主体 可选]

常见方法包括：

  • GET：读取指定资源。
  • HEAD：读取资源首部。
  • POST：向服务器提交数据。
  • PUT：在指定 URL 下存放文档。
  • DELETE：删除资源。
  • OPTIONS：请求选项信息。
  • TRACE：环回测试。
  • CONNECT：常用于代理场景。

（2）响应报文

响应报文由三部分构成：

  1. 状态行：版本、状态码、短语。
  2. 首部行。

状态码的类别含义如下：

  • 1xx：通知信息。
  • 2xx：成功。
  • 3xx：重定向。
  • 4xx：客户端差错。

5.10 Cookie 与状态保持

HTTP/1.0 是无状态协议，但实际网站常需要识别用户。Cookie 的作用就是在服务器和客户之间传递状态信息。服务器为用户分配唯一标识码，浏览器在后续访问中携带该标识，服务器由此跟踪同一用户在站点中的活动。

Cookie 解决的是”无状态事务协议如何支持有状态会话”这一问题，但它不会改变 HTTP 本身的基本交互模式。

Trap

Cookie 和 Session 是面试和考试的经典坑，记好这张图：

说白了：Cookie 是浏览器揣在兜里的”身份证”，Session 是服务器柜子里存的”档案”。Cookie 可能丢，Session 里的数据外人拿不到，但 Session 太多会撑爆服务器内存。

5.11 Web 文档与页面技术

（1）HTML

HTML 是超文本标记语言，用于描述页面结构与链接关系。它通过标签组织标题、段落、链接等内容，是浏览器还原页面显示的基础。

HTML 的关键职责是定义内容结构，而不是定义显示风格。

（2）XML、XHTML、CSS

  • XML：用于描述和传输数据，强调数据结构而非显示。
  • XHTML：更严格的 HTML 形式，可视为 XML 化的 HTML。
  • CSS：用于描述页面样式与布局，负责”如何显示”。

三者的边界为：HTML/XHTML 负责结构，CSS 负责表现，XML 更偏向数据描述与交换。

（3）静态文档、动态文档与活动文档

  • 静态文档：内容预先存放在服务器上，访问时直接返回。
  • 动态文档：服务器根据请求动态生成内容。
  • 活动文档：服务器返回程序，由客户端执行并生成交互效果。
CGI 与动态文档

CGI 为服务器与外部应用程序之间定义统一接口，使服务器能够把请求数据交给应用程序处理，再把程序输出作为 HTML 返回。

flowchart LR
    B[浏览器] -->|HTTP 请求| W[Web 服务器]
    W --> C[CGI 程序]
    C --> D[数据库/应用逻辑]
    D --> C
    C -->|动态 HTML| W
    W -->|HTTP 响应| B
活动文档与 Java

活动文档把更多处理放到浏览器端。浏览器请求文档后，服务器返回可执行程序副本，程序在客户端运行并生成动态效果。Java Applet 是这一路径的典型代表。其意义在于减少服务器持续计算与更新发送，但前提是客户端具备运行环境。

5.12 搜索引擎与相关网络信息服务

万维网规模极大，单靠链接浏览难以高效定位信息，因此需要搜索引擎。

搜索引擎的两种基本方式

  • 全文检索：搜索系统主动抓取网页、建立索引，用户查询的是索引数据库。
  • 分类目录：网站提交描述信息，经人工分类整理后供用户检索。

此外，还存在面向特定领域或特定人群的垂直搜索引擎。

博客、微博、轻博客和社交网站都属于 Web 应用在内容生产与社会连接方向上的扩展。其共同特征是：普通用户不再只是信息消费者，也成为内容生产者和网络关系的参与者。

六、电子邮件与 SMTP

6.1 电子邮件系统的三个主要构件

电子邮件系统由三类基本构件组成：

  1. 用户代理 UA：即电子邮件客户端，用于撰写、显示、处理与通信。
  2. 邮件服务器：负责发送、接收、缓存邮件，并报告投递状态。
  3. 邮件协议：发送通常使用 SMTP，读取通常使用 POP3 或 IMAP。
flowchart LR
    UA1[发件方用户代理] -->|SMTP| MS1[发送方邮件服务器]
    MS1 -->|SMTP| MS2[接收方邮件服务器]
    UA2[收件方用户代理] -->|POP3/IMAP| MS2

邮件服务器既可能充当服务器，也可能充当客户：向外发送邮件时，它是 SMTP 客户；接收其他服务器送来的邮件时，它又是 SMTP 服务器。

6.2 电子邮件地址与邮件组成

电子邮件地址的一般形式为：

收件人邮箱名@邮箱所在主机域名

电子邮件由两大部分组成：

  • 信封：供邮件传输程序使用，最关键的是收件人地址。
  • 内容：供用户阅读，包括首部和主体。

邮件内容通常由以下部分构成：

  • 首部 header
  • 空行
  • 主体 body

6.3 SMTP 的目标与工作方式

SMTP 是简单邮件传送协议，用于完成：

  • 用户代理到邮件服务器的邮件提交
  • 邮件服务器到邮件服务器的邮件转发

SMTP 使用客户—服务器方式，底层基于 TCP，通过文本形式的命令—响应交互完成邮件传送。

sequenceDiagram
    participant C as SMTP 客户
    participant S as SMTP 服务器
    C->>S: 建立 TCP 连接（端口 25）
    S->>C: 220 Service Ready
    C->>S: HELO 域名
    S->>C: 250 OK
    C->>S: MAIL FROM: 发件人
    S->>C: 250 OK
    C->>S: RCPT TO: 收件人
    S->>C: 250 OK
    C->>S: DATA
    S->>C: 354 Start mail input
    C->>S: 首部 + 空行 + 主体 + <CRLF>.<CRLF>
    S->>C: 250 OK
    C->>S: QUIT
    S->>C: 221 Service closed

SMTP 通信分为三个阶段：

  1. 连接建立
  2. 邮件传送
  3. 连接释放

SMTP 不依赖中间邮件服务器逐段登录接力来完成单次会话中的协议交换，而是在发送方 SMTP 客户与接收方 SMTP 服务器之间直接建立 TCP 会话。

Example

SMTP 的交互过程特别像打电话寄快递：

  1. 拨号建立连接（TCP 25 端口）→ 对方接起说”220 准备好了”
  2. 自报家门（HELO）→ 对方说”250 好的”
  3. 告诉对方”我要从哪发”（MAIL FROM）→ “250 收到”
  4. 告诉对方”发到哪去”（RCPT TO）→ “250 收到”
  5. 喊一声”我开始念内容了”（DATA）→ “354 请开始你的表演”
  6. 念完邮件内容，最后一行单独一个点（.）表示结束 → “250 投递成功”
  7. 挂电话（QUIT）→ “221 再见”

注意那个单独的 . ，这是 SMTP 的结束标记，考试填空题可能会考。

6.4 邮件首部字段

用户可见的邮件内容首部中，常见字段包括：

  • From：发件人地址
  • To：收件人地址
  • Date：发信日期
  • Subject：主题
  • Cc：抄送
  • Reply-To：回复地址

这些字段属于邮件内容的一部分，而 SMTP 交互中的 MAIL FROM、RCPT TO 等属于传输层面的信封信息。二者作用不同，不应混淆。

Trap

注意区分两套”发件人/收件人”：

  • MAIL FROM: / RCPT TO: → 信封上的地址，SMTP 传输时用的，决定邮件实际路由到哪。
  • From: / To: → 邮件内容里的地址，用户看到的，写在信纸上的。

这两套可以不一样！比如群发邮件，信封上 RCPT TO 是每个人的具体地址，但信纸上的 To: 可能写的是”全体用户”。垃圾邮件也常用这招伪造 From: 字段，但信封地址骗不过服务器。

6.5 MIME：SMTP 的扩展而非替代

原始 SMTP 主要面向 ASCII 文本，不适合直接传送图像、音频、视频、PDF 等非 ASCII 内容。MIME 的作用是在不替换 SMTP 的前提下，为邮件内容增加结构化描述和内容编码规则，使多媒体邮件能够通过原有邮件系统传输。

flowchart LR
    A[非 ASCII 或多媒体内容] --> B[MIME 编码与首部说明]
    B --> C[SMTP 传送]
    C --> D[对端解码还原]

MIME 主要包含三部分：

  1. 新增邮件首部字段。
  2. 标准化邮件内容类型。
  3. 定义内容传送编码。

（1）新增首部字段

  • MIME-Version
  • Content-Description
  • Content-Id
  • Content-Transfer-Encoding
  • Content-Type

（2）内容类型 Content-Type

Content-Type 由”类型/子类型”构成，例如：

  • text/plain
  • text/html
  • image/jpeg
  • audio/mpeg
  • video/mp4
  • application/pdf
  • multipart/mixed

（3）内容传送编码

为适应原有邮件系统的传输限制，MIME 定义了多种编码方式，如：

  • 7bit
  • 8bit
  • binary
  • base64
  • quoted-printable

其中，base64 适合二进制内容的 ASCII 化传输；quoted-printable 适合文本中只包含少量非 ASCII 字节的场景。

Note

MIME 不是替代 SMTP 的新协议，而是给 SMTP 打补丁。记住这个层级关系：

  • 底层传输：SMTP（TCP 25 端口）
  • 内容格式：MIME（定义 Content-Type、编码方式等）
  • 读取协议：POP3 / IMAP

常见 MIME 类型记住几个就行：text/plain（纯文本）、text/html（网页邮件）、image/jpeg、application/pdf、multipart/mixed（带附件的混合邮件）。

6.6 POP3 与 IMAP

POP3

POP3 使用客户—服务器方式，基于 TCP，支持用户鉴别。其典型特征是：用户从服务器读取邮件后，服务器上的对应邮件通常被删除，邮件处理状态主要保留在客户端。

IMAP

IMAP 同样基于 TCP，但它是联机协议，强调服务器端邮箱管理能力。其典型特征包括：

  • 先下载首部，支持部分下载。
  • 邮件文件夹可直接在服务器上创建与管理。
  • 支持服务器端搜索。
  • 多终端访问时状态可同步。
flowchart LR
    A[客户端 A] --> B[IMAP 服务器邮箱状态]
    C[客户端 B] --> B
    D[客户端 C] --> B

因此，POP3 更像”把邮件取回本地”，IMAP 更像”在线操作服务器上的邮箱”。

Trap

POP3 和 IMAP 的核心区别，一句话：POP3 是”下载后本地处理”，IMAP 是”服务器上统一管理”。

特性POP3IMAP
默认是否删除服务器邮件是（可配置保留）否
多终端同步差（各终端各自为政）好（状态全在服务器）
下载方式通常整封下载可先下首部， selective 下载
文件夹管理本地服务器端

你现在用手机、电脑同时收邮件还能同步已读状态，全靠 IMAP。如果你用很老的邮件客户端，可能会发现手机上读了、电脑上还显示未读，那八成是 POP3 的锅。

6.7 电子邮件机制的边界

电子邮件系统解决的是异步消息传递与存储转发问题，而不是实时会话通信。SMTP 负责发送，POP3/IMAP 负责读取，MIME 负责多媒体内容表达，三者共同构成现代电子邮件系统的核心应用层机制。

七、简单网络管理协议 SNMP

7.1 网络管理的目标

网络管理不是行政管理，而是通过硬件、软件与管理策略的综合使用，对网络资源进行监视、测试、配置、分析、评价与控制，以满足性能、可靠性和服务质量要求。

网络管理通常包括五大功能：

  • 故障管理
  • 配置管理
  • 计费管理
  • 性能管理
  • 安全管理

Intuition

SNMP 这玩意儿你就把它想象成网络体检系统：

  • 管理站（Manager）是”医院信息中心”，定时给各个设备（被管设备）发体检单。
  • 代理（Agent）是设备上的”健康监测仪”，负责收集本地数据（MIB 里的各种指标）。
  • 管理站问”你 CPU 占用多少？""最近丢包了吗？“——这叫探询（Polling）。
  • 代理发现”我过热了！我断线了！“主动报警——这叫陷阱（Trap）。

日常体检靠探询，突发重病靠 Trap 主动上报，两者配合才能及时发现网络问题。

7.2 网络管理模型

SNMP 所体现的网络管理模型由以下基本角色构成：

  • 管理站：网络管理系统核心，常称为网络运行中心 NOC。
  • 管理程序/管理者 manager：运行在管理站上的管理软件。
  • 代理 agent：运行在被管设备上的管理代理程序。
  • 被管设备：交换机、路由器、主机等网络元素。
  • 被管对象：设备内部可被监视或配置的对象。
flowchart LR
    M[管理站 Manager] <--网管协议--> A1[代理 Agent 1]
    M <--网管协议--> A2[代理 Agent 2]
    M <--网管协议--> A3[代理 Agent 3]
    A1 --> D1[被管设备 1]
    A2 --> D2[被管设备 2]
    A3 --> D3[被管设备 3]

这里的边界十分重要：网管协议本身并不”自动管理网络”，它只是管理程序与代理之间通信的规则集合。

7.3 SNMP 的设计思想

SNMP 的核心指导思想是尽可能简单。其目标是在尽量少影响原设备正常工作的前提下，实现对网络性能、故障和配置的监视与控制。

其基本功能包括：

  • 监视网络性能
  • 检测和分析网络差错
  • 配置网络设备
  • 在异常情况下辅助恢复

7.4 SNMP 的三部分组成

SNMP 网络管理体系由三部分组成：

  1. SNMP 协议本身：规定管理站与代理之间交换的 PDU 格式与操作。
  2. SMI：管理信息结构，定义命名、类型与编码规则。
  3. MIB：管理信息库，定义具体被管对象及其类型。

（1）SMI

SMI 的职责是定义：

  • 如何命名对象
  • 对象类型、范围和长度如何描述
  • 对象值如何编码

它关注的是”管理数据如何表达”，而不是”具体要管理哪些对象”。

（2）MIB

MIB 在被管理实体中建立具体的命名对象集合，并规定对象类型。管理程序通过读取或设置 MIB 中变量的值来了解设备状态或修改设备配置。

flowchart LR
    M[管理程序] <--SNMP--> A[代理]
    A --> B[MIB 变量集合]

7.5 SNMP 的交互方式

SNMP 管理一方面依靠探询，另一方面依靠陷阱报告。

（1）探询 Polling

管理进程周期性地向代理发送查询请求，以维持对网络资源的持续监视。

优点：

  • 机制简单
  • 可控制管理流量规模

缺点：

  • 不够灵活
  • 周期性探询开销较大
  • 可管理设备数量受限

（2）陷阱 Trap

代理在检测到某类事件时，不经请求主动向管理站报告。代理通常会先依据门限进行过滤，只在重要事件发生时发送陷阱。

优点：

  • 能快速报告异常
  • 报文简单、字节数少

SNMP 的有效性正来自这两者结合：日常使用探询维持全局状态感知，特殊事件使用陷阱实现快速告警。

sequenceDiagram
    participant M as 管理站
    participant A as 代理
    M->>A: 周期性 Get/Set/查询
    A->>M: Response
    A-->>M: Trap（异常事件主动上报）

7.6 SNMP 使用 UDP 的原因

SNMP 使用无连接的 UDP，而不是 TCP。其原因在于网络管理要求实现简单、报文开销小、事务独立，管理请求通常较短，适合用请求—响应式数据报交互完成。

端口约定为：

  • 代理侧使用端口 161 接收 get/set 类请求并发送响应。
  • 管理站侧使用端口 162 接收来自代理的 trap 报文。

Trap

SNMP 为什么用 UDP 而不是 TCP？这个问题考试出镜率极高。记住这三点：

  1. 简单：UDP 没有连接建立/释放开销，管理站问一句、代理答一句，完事。
  2. 开销小：UDP 首部只有 8 字节，TCP 20 字节；网络管理报文本身就很短，TCP 的”重成本”不划算。
  3. 事务独立：一次 Get 失败了，下次再发一个就行，不需要维护连接状态。而且 Trap 是代理主动发，本来就是单向通知，UDP 更自然。

但注意：UDP 不可靠，SNMP 自己不会重传，丢包就丢了。网管场景下，偶尔丢一个轮询包问题不大，下次周期再查就是了；Trap 丢了可能有点麻烦，但 SNMP 的设计哲学就是”简单第一”。

Note

SNMP 端口速记：

  • 161 → 代理（Agent）的”门诊接待窗口”，管理站来查询/set 都找它。
  • 162 → 管理站（Manager）的”急诊热线”，专门接代理主动打来的 Trap 报警电话。

别把这两个端口搞反了！

7.7 SNMP 的基本 PDU 类型

SNMP 定义了多种协议数据单元 PDU，常见包括：

  • GetRequest：读取一个或一组变量。
  • GetNextRequest：顺着 MIB 树读取下一个变量。
  • Response：返回查询结果。
  • SetRequest：修改变量值。
  • GetBulkRequest：批量读取大量变量值。
  • InformRequest：管理者之间交换信息。
  • SNMPv2Trap：代理主动报告异常事件。
  • Report：管理者之间报告差错。

7.8 委托代理

若某些网络元素本身不支持 SNMP，而使用专用网管协议，则可在管理站与被管设备之间引入委托代理。委托代理负责协议转换和过滤，使 SNMP 系统仍可纳入这些异构设备。

flowchart LR
    M[SNMP 管理站] --> P[委托代理 Proxy Agent]
    P --> D[专用协议设备]

7.9 SNMP 的边界

SNMP 的核心优势在于简单、轻量、便于跨设备部署；其边界在于：它主要解决”读取状态、修改配置、上报异常”这类网络管理问题，而不是替代具体设备内部的控制平面机制。

八、P2P 应用

8.1 P2P 的基本思想

P2P 应用中，参与通信的主机彼此对等。每个节点既可能是资源请求者，也可能是资源提供者。其关键优势在于系统能力可随节点数量扩张而增强，不再完全依赖中心服务器的上传能力。

Intuition

P2P 最直观的类比就是班级群里的资料共享：

  • 老师（服务器）发了一份复习资料到群里。
  • A 同学下载后，B 同学不仅可以从老师那下，还可以直接从 A 那下。
  • C 同学来了，可以从老师、A、B 任意一个人那里下。
  • 人越多，潜在的上传者越多，下载反而越快——这就是 P2P 的自扩展性。

传统客户—服务器方式呢？所有人都挤到老师那下载，人越多老师越卡，速度越慢。P2P 把这个瓶颈打破了。

8.2 集中式目录模型：Napster

Napster 采用集中式目录服务器。目录服务器并不传送文件本身，而是维护”哪个对等方拥有哪些文件”的索引信息。

其工作过程如下：

  1. 用户向目录服务器查询某文件。
  2. 目录服务器返回拥有该文件的若干对等方地址。
  3. 用户从这些对等方中任选其一发起下载。
  4. 文件在对等方之间直接以 P2P 方式传送。
sequenceDiagram
    participant X as 请求方 X
    participant N as Napster 目录服务器
    participant A as 对等方 A
    X->>N: 谁拥有文件 MP3#？
    N->>X: A、B、C 拥有该文件
    X->>A: 请求下载 MP3#
    A->>X: 传送文件 MP3#

该模型的优点是查找简单；缺点是集中式目录服务器成为性能瓶颈和可靠性单点。

8.3 全分布式查找：Gnutella

Gnutella 去除了集中目录服务器，在非结构化覆盖网络中采用洪泛查询定位内容。某节点需要查找对象时，把查询请求转发给邻居，邻居再继续扩散，直到找到对象或达到传播范围限制。

flowchart LR
    Q[查询节点] --> N1[邻居1]
    Q --> N2[邻居2]
    N1 --> N3[邻居3]
    N2 --> N4[邻居4]
    N3 --> R[命中文件节点]

该方案解决了中心瓶颈问题，但代价是查询流量大、定位效率较低，因此通常需要限制洪泛范围。

8.4 BitTorrent 的块交换机制

BitTorrent 将一个文件划分为多个固定长度的文件块 chunk。参与同一文件分发的全部对等方构成一个洪流 torrent。系统中通常还存在追踪器 tracker，用于帮助节点获得当前可连接的对等方列表。

基本过程为：

  1. 新节点联系追踪器，获取部分对等方列表。
  2. 节点与这些对等方建立 TCP 连接，形成邻居集合。
  3. 各对等方交换自己拥有的文件块信息。
  4. 节点从多个邻居并行下载不同块，同时把已有块上传给其他节点。
flowchart LR
    T[追踪器 Tracker] --> A[对等方 A]
    T --> B[对等方 B]
    T --> C[对等方 C]
    A <--> B
    A <--> C
    B <--> C

flowchart LR
    B[Peer B: 块 1,3] --> A[Peer A]
    C[Peer C: 块 2,4] --> A
    A --> D[Peer D]

BitTorrent 的优势在于：文件分发负载由众多对等方共同承担，服务器不必独自完成全部上传任务。

8.5 P2P 文件分发时间分析

设有一个长度为 F bit 的大文件，需要分发给 N 台主机。服务器上传速率为 us​，第 i 台主机上传速率为 ui​，下载速率为 di​，其中最小下载速率为 dmin​，所有对等方上传速率之和为：

uT​=us​+i=1∑N​ui​

（1）客户—服务器方式的最短时间下限

服务器必须向每个主机各发送一份完整文件，因此至少需要：

us​NF​

同时，最慢的接收者下载一份文件至少需要：

dmin​F​

因此客户—服务器方式下的最短分发时间为：

Tcs​=max(us​NF​,dmin​F​)

（2）P2P 方式的最短时间下限

P2P 中必须满足三个下界：

  1. 服务器至少要把一整份文件注入系统：
us​F​
  1. 最慢接收者仍至少需要：
dmin​F​
  1. 系统总共要向 N 个主机分发 NF bit 数据，而系统总上传能力是 uT​，因此至少需要：
uT​NF​

综合可得：

Tp2p​≥max(us​F​,dmin​F​,uT​NF​)

该结果表明，当对等方数量增加时，P2P 系统的总上传能力也随之增加，因此在大规模分发场景下通常明显优于纯客户—服务器方式。

8.6 P2P 的边界与工程意义

P2P 的根本价值在于利用边缘节点资源分担中心负载，尤其适合大文件分发和大规模内容共享。但其代价包括：

  • 查找与调度更复杂
  • 节点在线状态不稳定
  • 一致性与安全控制更困难
  • 激励与公平问题突出

因此，实际系统常在纯 P2P 与中心化之间折中，例如用追踪器辅助节点发现，用中心服务承担索引、鉴权或协调功能。
