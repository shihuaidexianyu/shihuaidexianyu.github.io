xv6 内存分配机制

0. 核心概念

在 xv6 中，brk 用于调整进程用户地址空间的大小，kalloc 用于分配真实的物理页，而页表负责建立虚拟地址到物理地址之间的映射关系。

从调用链上看，堆增长的大致路径为：

brk → growproc → uvmalloc → mappages → walk → kalloc

堆收缩的大致路径为：

uvmdealloc → vmunmap → walk → kfree

在这一机制中，需要区分三个层次：

  1. 进程逻辑上拥有多大的地址空间
  2. 页表中是否已经建立映射
  3. 底层是否已经分配真实物理页

1. 需要区分的几个对象

1.1 堆顶 sz

proc.sz 表示当前进程用户地址空间的大小，单位是字节。
它描述的是进程当前“逻辑上拥有多少用户空间”，而不是已经分配了多少物理内存。

在 sys_brk 中，通常会读取当前进程的 sz，根据用户传入的新地址计算增量，并调用 growproc 进行扩展或收缩。

1.2 虚拟页

进程访问的是虚拟地址空间。
虚拟地址空间按页划分，每一页对应一个固定大小的地址区间。
CPU 在访问内存时，首先使用虚拟地址，再通过页表将其翻译为物理地址。

1.3 页表项 PTE

页表项用于记录映射关系。
一个页表项通常包含：

  • 目标物理页号
  • 有效位
  • 读写执行权限
  • 用户态可访问标记等

页表项本质上是“地址翻译规则”的具体载体。

1.4 物理页

物理页是物理内存中的基本分配单位，通常大小为 4KB。
kalloc() 从空闲页链表中取出一个物理页，kfree() 则将其归还到空闲页链表。

2. 从 brk 到可访问内存的总体流程

flowchart TD
    A["用户调用 brk(new_addr)"] --> B[sys_brk]
    B --> C{new_addr == 0?}
    C -- 是 --> D[返回当前 p->sz]
    C -- 否 --> E[delta = new_addr - old_sz]
    E --> F["growproc(delta)"]

    F --> G{delta > 0?}
    G -- 是 --> H[uvmalloc]
    G -- 否 --> I[uvmdealloc]

    H --> J[kalloc 分配物理页]
    J --> K[memset 清零]
    K --> L[mappages 建立映射]
    L --> M[walk 找到或创建 PTE]
    M --> N[把 PA 和权限写入 PTE]

    I --> O[vmunmap 解除映射]
    O --> P[walk 找到 PTE]
    P --> Q[kfree 归还物理页]

这个流程体现了 xv6 内存分配的分层结构：

  • brk 负责提出地址空间变更请求
  • growproc 负责决定扩容还是缩容
  • uvmalloc/uvmdealloc 负责处理虚拟地址区间
  • mappages/vmunmap 负责建立或拆除页表映射
  • walk 负责遍历页表
  • kalloc/kfree 负责实际物理页的分配与释放

3. 为什么接口按字节变化，而底层按页分配

进程从接口上看，可以按字节调整堆顶位置；但内核在实际实现中，始终以页为单位进行物理内存管理。

uvmalloc 在处理扩容时，会先将 oldsz 向上对齐到页边界，然后按页循环处理新增区间。
因此，地址空间的逻辑增长可以是 1 字节、64 字节或任意大小，但是否需要真正分配新的物理页，取决于新的堆顶是否跨过新的页边界。

flowchart TD
    A[用户视角: 再申请 64B] --> B[内核视角: 是否跨过新页边界]
    B --> C{跨页了吗?}
    C -- 没跨页 --> D[仅修改 sz]
    C -- 跨页 --> E[kalloc 新物理页]
    E --> F[mappages 建立映射]

示例

假设页大小为 4096B。

  • 当前 sz = 5000
  • 调用 brk(5064)

这次增长只有 64B。
如果这 64B 没有跨越新的页边界，那么可能只需修改 sz；
如果扩展后覆盖到新的虚拟页，则需要分配新的物理页，并建立新的映射。

因此，sz 是字节级边界，而页表和物理分配机制是页级机制。

4. sys_brk：调整堆顶的系统调用入口

sys_brk 的作用主要是接收用户请求，并把它转换为对地址空间的调整操作。

其主要逻辑可概括为：

  1. 读取用户传入的新地址 new_addr
  2. 获取当前进程的旧堆顶 old_sz = p->sz
  3. 如果 new_addr == 0，返回当前堆顶
  4. 否则计算增量 delta = new_addr - old_sz
  5. 调用 growproc(delta)

因此，sys_brk 并不直接完成物理页分配或页表映射，它只是堆增长机制的入口。

sequenceDiagram
    participant U as 用户程序
    participant S as sys_brk
    participant G as growproc

    U->>S: brk(new_addr)
    S->>S: old = p->sz
    alt new_addr == 0
        S-->>U: 返回 old
    else 调整堆
        S->>G: growproc(new_addr - old)
        G-->>S: 成功/失败
        S-->>U: 返回结果
    end

5. growproc：决定扩容还是缩容

growproc 是地址空间调整过程中的分发层。
它根据增量 n 的正负决定调用哪条路径：

  • n > 0：调用 uvmalloc
  • n < 0：调用 uvmdealloc

因此，growproc 的职责不是执行具体的页分配或释放，而是根据地址空间变化方向，把请求转发到相应的实现函数。

6. uvmalloc：为新增虚拟地址区间建立可访问空间

uvmalloc 用于扩展用户地址空间。
其主要过程包括：

  1. 将 oldsz 向上对齐到页边界
  2. 按页遍历新增的虚拟地址区间
  3. 对每一页调用 kalloc() 分配物理页
  4. 对物理页执行清零
  5. 使用 mappages() 建立虚拟页到物理页的映射
flowchart TD
    A["uvmalloc(oldsz, newsz)"] --> B[oldsz 向上页对齐]
    B --> C[遍历每个新增虚拟页 a]
    C --> D[kalloc 获取 1 个物理页 mem]
    D --> E["memset(mem, 0, PGSIZE)"]
    E --> F[mappages 建立映射]
    F --> G[继续下一页]

物理页清零的原因

新分配的物理页通常需要清零，以避免旧数据残留。
这样可以确保新映射到用户空间的页面在初始状态下内容可控，并避免泄露先前使用者的数据。

7. mappages：建立虚拟页到物理页的映射

mappages 的职责是在页表中写入映射关系。
它并不负责分配物理页，而是在已有物理页的前提下，把该物理页映射到指定虚拟地址。

其核心过程为：

  1. 调用 walk(pagetable, va, alloc) 找到目标虚拟地址对应的最终页表项地址
  2. 将物理页号、权限位和有效位写入该页表项
flowchart TD
    A[虚拟地址 va] --> B[walk 找到最终 PTE]
    B --> C[组合物理页号与权限位]
    C --> D[写入 PTE]
    D --> E[映射建立完成]

因此可以明确区分：

  • kalloc 负责提供物理页
  • mappages 负责建立映射关系

二者分别对应“资源分配”和“地址翻译配置”两个不同层面。

8. walk：在三级页表中定位目标页表项

在 Sv39 机制下，页表采用三级结构。
虚拟地址会被拆分为：

  • 第 2 级索引
  • 第 1 级索引
  • 第 0 级索引
  • 页内偏移

walk 的任务是依据这三级索引，从顶层页表逐层向下查找，最终定位到某个虚拟页对应的叶子页表项。

flowchart TD
    VA[虚拟地址 VA]
    VA --> L2[取 L2 索引]
    L2 --> PT2[顶级页表]
    PT2 --> L1[取 L1 索引]
    L1 --> PT1[二级页表]
    PT1 --> L0[取 L0 索引]
    L0 --> PT0[三级页表]
    PT0 --> PTE[最终叶子 PTE]
    PTE --> OFF[页内偏移]

walk 的两种行为

8.1 下一级页表已存在

若当前层页表项有效，说明它已经指向下一级页表。
此时 walk 会沿该指针继续向下遍历。

8.2 下一级页表不存在

若当前层页表项无效，而 alloc = 1，则 walk 会：

  1. 调用 kalloc() 分配新的页表页
  2. 将该页表页清零
  3. 把它的地址写入当前层页表项
  4. 继续向下遍历
flowchart TD
    A["walk(pagetable, va, alloc)"] --> B[从 level=2 开始]
    B --> C{当前层 PTE 有效?}
    C -- 是 --> D[进入下一级页表]
    D --> E{已到叶子层?}
    E -- 否 --> C
    E -- 是 --> F[返回最终 PTE 地址]

    C -- 否 --> G{alloc == 1?}
    G -- 否 --> H[返回 NULL]
    G -- 是 --> I[kalloc 新页表页]
    I --> J[memset 清零]
    J --> K[写入当前层 PTE]
    K --> E

这说明：
在建立某个数据页映射时，系统除了可能分配数据页本身，还可能额外分配新的页表页。

9. kalloc / kfree：物理页分配器

xv6 的物理内存分配器采用空闲页链表实现。
空闲链表中的每个节点代表一个未使用的物理页。

其操作方式如下：

  • kalloc()：从空闲链表中取出一个页
  • kfree()：将一个页重新放回空闲链表
flowchart TD
    A[free list 空闲页链表] --> B[kalloc: 取出表头页]
    B --> C[返回 1 个 4KB 物理页]
    D[待释放的物理页] --> E[kfree]
    E --> F[挂回 free list 表头]

由于分配粒度固定为页，因此该分配器结构简单，适合 xv6 这样的教学系统。

10. 释放路径：uvmdealloc → vmunmap → walk → kfree

当进程缩小堆或释放用户空间时，会沿相反方向执行回收流程：

  1. 确定哪些虚拟页不再需要
  2. 通过页表找到这些虚拟页对应的页表项
  3. 解除映射关系
  4. 将对应物理页归还给分配器
flowchart TD
    A[缩小堆 / 释放内存] --> B[uvmdealloc]
    B --> C[vmunmap]
    C --> D[walk 找到对应 PTE]
    D --> E[清除映射]
    E --> F{需要释放物理页?}
    F -- 是 --> G[kfree]
    F -- 否 --> H[仅解除映射]

这里的顺序很重要：
必须先拆除页表映射，再归还物理页。
否则，页表中仍可能保留指向已释放页的条目，从而导致错误访问。

11. 整体机制的分层理解

flowchart TD
    A[用户层: brk / sbrk 等堆增长请求]
    A --> B[进程层: growproc]
    B --> C[虚拟内存层: uvmalloc / uvmdealloc]
    C --> D[页表层: mappages / vmunmap]
    D --> E[页表遍历层: walk]
    E --> F[物理内存层: kalloc / kfree]
    F --> G[真实物理页]

这一分层说明：

  • 上层接口表达的是“地址空间需求”
  • 中层负责管理虚拟地址区间
  • 页表层负责维护映射关系
  • 最底层才真正操作物理页

12. 示例：brk(cur + 64) 的执行过程

sequenceDiagram
    participant U as 用户程序
    participant SB as sys_brk
    participant GP as growproc
    participant UV as uvmalloc
    participant MP as mappages
    participant WK as walk
    participant KA as kalloc

    U->>SB: brk(cur + 64)
    SB->>SB: old = p->sz
    SB->>GP: growproc(64)
    GP->>UV: uvmalloc(old, old+64)

    alt 未跨越新页边界
        UV-->>GP: 无需分配新物理页
    else 跨入新页
        UV->>KA: kalloc()
        KA-->>UV: 返回新物理页
        UV->>MP: 建立映射
        MP->>WK: walk(..., alloc=1)
        WK-->>MP: 返回目标 PTE
        MP-->>UV: 写入 PTE
    end

    UV-->>GP: 返回 newsz
    GP-->>SB: 更新 p->sz
    SB-->>U: 成功

这个例子说明：

用户请求增加 64B，并不意味着系统一定会分配 64B 的物理内存。
只有在新增地址空间跨越页边界时，内核才需要分配新的物理页并建立映射。

13. 容易混淆的几点

13.1 brk 与 kalloc 不属于同一层

brk 是用户地址空间调整接口。
kalloc 是物理页分配器。
二者之间还经过 growproc、uvmalloc、mappages、walk 等多层处理。

13.2 mappages 不是物理页分配器

mappages 只负责写页表项。
它不从空闲内存池中取页。
物理页的实际来源是 kalloc。

13.3 walk 不仅用于查询

当 alloc = 1 时，walk 在发现中间层页表缺失时，会主动分配新的页表页并补齐页表层级。

13.4 地址空间按字节变化，与底层按页分配并不矛盾

前者是进程接口层面的语义。
后者是内核实现层面的管理粒度。
因此，堆顶调整与物理页分配之间并不是一一对应关系。

14. 总结

xv6 的内存分配机制可以概括为如下过程：

brk 负责提出堆顶调整请求；
growproc 根据增减方向调用 uvmalloc 或 uvmdealloc；
uvmalloc 按页处理新增虚拟地址区间，调用 kalloc 分配物理页并清零，再通过 mappages 建立映射；
mappages 使用 walk 找到最终页表项，将物理页号和权限写入页表；
如果页表层级缺失，walk 会在需要时额外分配页表页；
释放内存时则执行相反过程，先拆除映射，再通过 kfree 回收物理页。

2. 问题分析：从 “unknown sys call 80” 说起

在最初只补全了 sys_mmap 和 sys_munmap 的函数体后，运行测例出现了：

错误处理

5.2 主要流程

  1. 参数校验：长度必须为正；prot 不能包含未定义位；非匿名映射时检查 fd 合法性及文件读写权限。
  2. 地址分配：

   - MAP_FIXED：严格使用传入地址，检查是否和已有 vma 重叠。

   - 非 MAP_FIXED：从 MMAP_START 开始，向后找到第一个足够大的空闲间隙（暴力扫描 vma 数组）。

  1. 页表映射：

   - 按页分配物理内存（kalloc + memset(..., 0, PGSIZE)）。

   - 根据 prot 构造 PTE flags（PTE_R/W/X/U）。

   - 先映射用户页表，再映射内核页表（权限去掉 PTE_U）。

   - 任何一步失败都要回滚已分配的页。

  1. 文件回填（如果是文件映射）：

   - elock(f->ep) 后循环调用 eread(f->ep, 0, va, off, n) 把文件内容读到刚分配的物理页。

   - eunlock(f->ep)。

  1. 记录 VMA：找一个空闲 vma 槽位，填入本次映射的元数据。
  2. 返回虚拟起始地址。

6. sys_munmap 实现细节

6.1 参数与限制

  • addr 必须页对齐。
  • length 必须为正。
  • 查找覆盖 addr 的 vma。
  • 当前实现支持：

  - 完全释放 len == v->len

  - 从头部截断 addr == v->addr

  - 从尾部截断 addr + len == v->addr + v->len

Caution

中间部分 munmap（把一段 vma 分成两段）在这个实现里暂不处理。如果测例有此类需求，需要把单个 vma 拆成两个。

6.2 释放流程

  1. 用 vmunmap 分别解除用户页表和内核页表的映射（用户侧释放物理页）。
  2. 如果是整个 VMA 释放，关闭关联文件并标记 used = 0。
  3. 如果是部分释放，仅调整 vma->addr / vma->len / vma->offset。
