x86-64 汇编、过程与 ABI

本章讨论 x86-64 程序级机器表示，重点包括 AT&T 汇编语法、寄存器与调用约定、常见数据传送与算术指令、条件码与控制转移、过程调用、数据布局、ABI、进程地址空间以及缓冲区溢出防御。

1. AT&T 汇编语法与寄存器约定

AT&T 语法采用

的顺序，即源操作数在前，目的操作数在后。指令后缀常用来表示操作数宽度：b 表示字节，w 表示 16 位字，l 表示 32 位双字，q 表示 64 位四字。立即数前用 $，寄存器前用 %，内存操作数一般写成地址表达式。

x86-64 通用寄存器具有分层别名。例如 %rax、%eax、%ax、%al 分别表示同一寄存器的 64 位、32 位、16 位与低 8 位视图。过程调用中，返回值通常经 %rax 返回。整数或指针类型的前六个参数依次放入 %rdi、%rsi、%rdx、%rcx、%r8、%r9。超出六个的参数通过栈传递。

寄存器还按调用保存责任划分。被调用者保存寄存器包括 %rbx、%rbp、%r12 至 %r15；调用者保存寄存器包括 %r10、%r11，以及其他用于参数和返回值的易失寄存器。若被调用函数需要长期使用被调用者保存寄存器，就必须在进入函数后保存它们，并在返回前恢复。

较小的结构体可能按 ABI 规则拆分后通过寄存器传递；较大的结构体参数或返回值则可能通过内存与调用者栈帧配合完成。第七个及之后的参数位于栈上，且参数压栈的构造顺序需要满足调用约定和栈对齐要求。

2. 数据传送指令

最基本的数据传送指令是 mov。movb、movw、movl、movq 分别表示不同宽度的数据复制。movabsq 用于把 64 位立即数直接送入 64 位寄存器。传送指令本质上只负责复制位模式，不负责推断其算术意义。

扩展类传送指令分为零扩展与符号扩展两类。零扩展指令把源操作数放到目标低位，并在高位补零；符号扩展指令则复制源操作数的符号位到目标高位。例如，movzbl 将一个字节零扩展到 32 位，movsbq 将一个字节符号扩展到 64 位，movslq 将一个 32 位有符号数扩展到 64 位。cltq 的作用是将 %eax 中的 32 位值按有符号数解释并扩展到 %rax。一个重要的特殊规则是：当 movl 以 64 位寄存器为目标时，写入低 32 位后会自动将高 32 位清零。

3. 栈、push/pop 与地址计算

x86-64 的栈向低地址增长，%rsp 始终指向当前栈顶。64 位环境下常用 pushq 与 popq 以 8 字节为单位操作栈。执行 pushq 时，先将 %rsp 减去 8，再把数据写入新的栈顶地址；执行 popq 时，先从栈顶读出数据，再将 %rsp 加回 8。

多字节数据写入内存时会占据一段连续地址区间，地址本身沿从低到高的方向排列，而各字节的高低有效性则由字节序决定。

移位分为左移、逻辑右移与算术右移。左移常记作 sal 或 shl，二者在 x86 中效果相同；shr 为逻辑右移，sar 为算术右移。移位位数可以是立即数，也可以取自 %cl。算术右移保留符号信息，逻辑右移则用零填充高位。

移位与逻辑指令还会影响条件码。课程语境下，移位会使进位标志记录最后一个被移出的位；inc 与 dec 不修改进位标志；xor、or、and 会清除进位与溢出标志。理解这些规则的意义在于：条件码不是独立存储的布尔变量，而是算术与逻辑指令执行后的副产物，后续控制转移会读取它们。

5. 扩展乘法、除法与 %rdx:%rax

一般乘法只保留与目的寄存器同宽的低位结果，而扩展乘法会使用 %rdx:%rax 这对寄存器联合保存更宽的结果。对 64 位乘法而言，完整积是 128 位，其中高 64 位放入 %rdx，低 64 位放入 %rax。有符号与无符号乘法分别由 imulq 与 mulq 完成。

在执行有符号除法前，常需先使用 cqto 将 %rax 中的 64 位被除数符号扩展到 %rdx:%rax，从而得到合法的 128 位被除数。随后 idivq 以某个 64 位操作数为除数，商写入 %rax，余数写入 %rdx。无符号除法 divq 采用相同的寄存器分工，只是解释方式不同。

6. 条件码、比较与布尔结果生成

最核心的条件码包括零标志 ZF、符号标志 SF、进位标志 CF 与溢出标志 OF。ZF 记录结果是否为零，SF 记录结果符号位，CF 对应无符号运算中的进位或借位，OF 对应有符号运算中的符号溢出。CF 与 OF 的区分体现了“同一位模式可以有不同算术解释”的机器级事实。

cmp S1, S2 会计算 S2 - S1 并只更新条件码，不保存差值；test S1, S2 会计算 S2 \& S1 并只更新条件码，常用于测试某些位是否为零。比较与测试之所以重要，是因为控制流通常依赖条件码而不是依赖显式的布尔变量。

一组 cc 后缀描述不同的条件。e 或 z 表示相等或零，依赖 ZF=1；ne 表示不等；l、le、g、ge 用于有符号比较；b、be、a、ae 用于无符号比较。Above/Below 对应无符号语义，Greater/Less 对应有符号语义。这一区分直接源于 CF 与 OF 的不同解释。

setcc 指令根据条件码把一个字节目标设置为 0 或 1。由于目标只能是 8 位寄存器或单字节内存单元，若后续要把它作为更宽整数使用，通常还需要额外进行零扩展。

7. 跳转、条件传送与 switch 跳转表

无条件跳转 jmp 直接修改程序计数器。条件跳转 jcc 先读取条件码，再决定是否跳转。跳转目标既可以是静态标签，也可以通过寄存器或内存间接给出。实际机器码中常使用 PC 相对编码，即以“目标地址与下一条指令地址之差”的形式编码位移，这使得代码在内存中的装载位置更灵活。

cmovcc 是条件传送指令。它不改变控制流，而是在满足条件时把源操作数传送到目标寄存器。其优势在于避免分支预测失败带来的流水线代价，因此在条件计算较简单、没有副作用时，编译器常用它替代短分支。它不能用于单字节目标，而且源操作数往往仍需先被求值，因此并不适合所有分支场景。

当 switch 的分支数较多且各 case 值比较密集时，编译器常将其实现为跳转表。跳转表本质上是一个“索引到代码地址”的数组：先根据 switch 变量的值做边界检查，再用该值索引表项，最后执行一次间接跳转。与一连串 if-else 相比，这种实现把多个条件比较压缩为一次索引和一次跳转，时间复杂度更接近常数。

8. 过程调用、栈帧与返回机制

函数调用通过 call 与 ret 完成。执行 call 时，处理器会先把返回地址压入栈中，再跳转到被调用函数入口；执行 ret 时，处理器从栈顶弹出返回地址，并跳转回该地址继续执行。这里的返回地址是 call 下一条指令的地址。

9. 指针运算、联合体与内存对齐

指针运算不是按“字节数”变化，而是按“元素跨度”变化。若 p 的类型为 T *，则表达式 p + n 的新地址是原地址加上

n×sizeof(T)

这是因为指针语义服务于“指向第几个元素”，而不是“指向第几个字节”。同理，两个同类型指针相减得到的也不是裸地址差，而是它们之间相隔多少个 T 类型元素：

p2​−p1​=sizeof(T)addr(p2​)−addr(p1​)​

联合体的所有成员共享同一段起始地址，因此联合体的大小至少等于其最大成员的大小，并且仍需满足整体对齐要求。联合体适合以不同类型查看同一块存储，或在互斥字段之间节省空间。由于成员共享存储，写入一个成员后再读取另一个成员，得到的往往是对同一位模式的重新解释。

结构体的内存布局受对齐规则控制。每个成员的起始偏移必须是该成员对齐要求的整数倍；整个结构体大小也必须是最大对齐要求的整数倍。因此，结构体中往往会插入填充字节。对齐的直接目的在于让处理器高效且合法地访问数据。

10. ABI 的含义与 API 的区别

ABI 是应用程序二进制接口，它规定编译后的程序、库与操作系统在机器码层面如何协作。它不是源代码书写规范，而是二进制层面的合同。API 面向程序员，规定“有哪些函数、如何调用”；ABI 面向编译器、链接器和运行时系统，规定“调用在机器层面如何实现”。

二者的区别可以从函数调用中看得最清楚。API 只说明可以调用 printf 并传入字符串；ABI 则规定该字符串地址该放入哪个寄存器，调用前栈怎样对齐，返回值放在哪个寄存器，哪些寄存器调用后必须保持不变。正因 ABI 存在，不同编译器生成的目标文件才能在同一平台上互相链接，不同语言编写的模块才能在同一运行时中互操作。

ABI 的核心内容包括调用约定、参数与返回值的传递方式、寄存器保存规则、数据类型大小、结构体布局、二进制文件格式以及系统调用接口等。不同操作系统即使运行在同一指令集上，也可能采用不同 ABI，因此其二进制程序并不能直接互换运行。

11. x86-64 Linux 的典型内存布局

典型的用户进程地址空间可分为代码区、静态数据区、堆、共享库映射区与栈。代码区保存可执行机器指令，通常具有只读和可执行属性；静态数据区保存全局变量、静态变量以及字符串常量；堆用于动态分配，常由 malloc、calloc、new 等接口扩展；栈保存局部变量、返回地址与过程控制信息，并向低地址方向增长。共享库会被映射到进程地址空间中，与主程序共同构成可执行映像。

代码段通常不可写，向只读可执行区域写入数据会触发异常。堆与栈则主要承担可写数据的生命周期管理，只是二者的分配与回收机制不同：堆依赖显式分配释放，栈依赖过程调用自动展开与收缩。

12. 缓冲区溢出与常见防御机制

缓冲区溢出的本质是写入操作越过了对象边界，从而覆盖控制信息、相邻变量或返回地址。它既可能导致程序崩溃，也可能被利用来劫持控制流。现代系统对这类攻击的防御主要包括以下几个方向。

栈保护机制会在栈帧中关键控制信息之前放置金丝雀值。函数返回前先检查该值是否被篡改。若检测失败，程序不会按原路径返回，而是立即中止，从而阻止攻击者通过覆盖返回地址取得控制权。

数据执行保护通过硬件页属性把内存页区分为“可执行”和“不可执行”。栈与堆通常被标记为不可执行，因此即使攻击者把恶意指令写入这些区域，处理器也不会允许直接执行，从而阻断“注入并执行代码”的经典攻击路径。

地址空间布局随机化会在程序每次启动时随机化栈、堆、共享库等区域的基址。攻击者即使知道某段代码的逻辑，也难以预先确定其精确地址，从而提高利用门槛。随机化强度越大，暴力猜测越困难。

最根本的防御仍然是安全编程实践。输入长度检查、边界检查、避免使用无长度限制的危险函数，以及采用 fgets、snprintf、strncpy 等带边界参数的安全替代接口，都是在漏洞形成之前切断攻击面。缓冲区溢出不是单纯的汇编问题，而是源代码、编译器、运行时和硬件共同参与的系统安全问题。
