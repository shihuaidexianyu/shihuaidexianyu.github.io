虚拟内存管理

一、虚拟页式存储管理

1. 先把这件事想明白：虚拟内存到底在解决什么

你可以先把这一章的目标记成一句话：让程序觉得自己拥有一个又大又连续、而且很好用的内存空间。

真实机器上的存储体系其实是分层的：寄存器最快、Cache 很快、主存不算小但也不够大、磁盘容量大但很慢。操作系统要做的，就是尽量把“速度”和“容量”两件事同时做好：让 CPU 感觉内存够快，也让程序感觉内存够大。虚拟内存正是把主存和磁盘协同起来后的那个“看起来很大”的地址空间。程序看到的是虚拟地址，真正访问的是物理地址，中间由硬件和操作系统一起完成转换。

flowchart TD
    CPU[CPU 执行指令] --> VA[虚拟地址]
    VA --> MMU[MMU / 地址转换]
    MMU -->|命中| MEM[物理内存]
    MMU -->|页不在内存| PF[缺页异常]
    PF --> OS[操作系统处理]
    OS --> DISK[磁盘 / 后备存储]
    DISK --> MEM

Intuition

你可以把虚拟内存想成“酒店前台给每位客人发的房号体系”。
每个客人都觉得自己住得很规整、房间号很连续，但后台其实可能把人安置在不同楼层、不同区域，甚至暂时安排到“等候区”。
对客人来说，重要的是“我有这个房号，而且能访问”；对系统来说，重要的是“这个房号现在映射到哪里”。

2. 四个最容易混的词

这几个词考试时很喜欢混着出：

  • 虚拟内存：把主存和磁盘联合起来后，程序所“感觉到”的大内存。
  • 虚拟地址空间：分配给某个进程的整块虚拟地址范围。
  • 虚拟地址：虚拟地址空间里的某一个地址。
  • 虚拟存储技术：程序运行时只把一部分页放进内存，不在内存的部分等需要时再调入。

Trap

“虚拟内存”不是“磁盘空间”本身。
它是主存 + 磁盘 + 地址转换机制 + 缺页处理机制共同形成的一种效果。

3. 请求分页：为什么程序可以一开始几乎不装东西就跑起来

虚拟页式存储管理的核心思想很朴素：

  1. 程序装入时，不必把全部页面都装进内存。
  2. 真正访问到某页时，如果它还不在内存，再动态调入。
  3. 内存不够时，就把暂时不用的页换出去。

这里有两种取页思路：

  • 请求调页：用到再调。
  • 预先调页：提前猜一些将来要用的页先装进去。

Intuition

这本质上是一种资源兑换：
用 CPU 时间和磁盘 I/O，去换看起来更大的内存容量。

4. 调页系统最关心的三件事

这一块你记成三个英文词就很好背：

  • fetch policy：什么时候把页装进来
  • placement policy：装进来后放到哪个页框
  • replacement policy：没位置时，赶谁出去

这三件事里，真正最折磨人的通常是第三件，也就是页面置换。

5. 页表项 PTE：它不是只存“页框号”这么简单

页表项的任务，是把“虚页”翻译成“物理页框”，但它远不止保存一个页框号。课上反复强调的几个字段你一定要记住：

Physical Address=PFN×PageSize+offset
  • PFN / 页框号：这个虚页现在映射到哪个物理页框（相似的概念是VPN）
    • VPN 是虚拟内存里的页号，PFN 是物理内存里的页号，页表负责把前者翻译成后者。
  • Present / Valid：这页是不是当前在物理内存里
  • Accessed / Referenced：最近有没有被访问过
  • Dirty / Modified：装入后有没有被写过
  • Protection：读 / 写 / 执行权限
  • 在 x86 里还会看到 R/W、U/S、PWT、PCD、PS、G 这些控制位，分别对应可写性、用户/内核权限、缓存策略、大页、TLB 全局页等。

Note

PTE 不是单纯的“映射表项”，它同时还是权限检查、替换决策、缓存策略、缺页判定的依据。

Trap

哪些情况会触发 Page Fault，不只是“页不在内存”。
还可能是：页面非法、权限不对、写时复制页被写、请求栈扩展、请求零页等。
也就是说，Page Fault 是地址转换阶段发现“当前 PTE 不允许你这么干”时，由硬件触发给操作系统的异常。

6. 为什么必须用多级页表

如果给每个进程都整一张“平铺到底”的大页表，代价会非常夸张。

在 32 位地址空间、页面 4KB、页表项 4B、用户空间 2GB（231） 的情况下，一个进程就有 219 个页，需要的页表本身就要占 512 页。到 64 位时，平铺页表基本不可接受。

Example

平铺页表大小的一个例子

设：

  • 用户虚拟地址空间大小为 2GB=231B
  • 页面大小为 4KB=212B
  • 每个页表项（PTE）大小为 4B=22B

则：

1. 用户空间一共有多少个虚拟页

212231​=219

所以，用户空间一共被划分为 219个虚拟页。

2. 需要多少个页表项

平铺页表中，每个虚拟页对应一个页表项，因此需要：

219 个 PTE

3. 页表总共占多少字节

219×22=221B=2MB

所以，这张页表本身大小为 2MB。

4. 页表本身要占多少页

由于每页大小仍为 4KB=212B，所以页表需要的页数为：

212221​=29=512

所以，平铺页表本身需要占用 512 个物理页。

直觉理解

这里有两个不同的“页数”：

  • 219：是 被映射的虚拟页数
  • 29：是 存放这些页表项所需的页表页数

原因是：一个页表页大小为 4KB，而一个 PTE 只有 4B，所以一个页表页可以容纳：

22212​=210

即 1024 个页表项。

因此，219 个页表项只需要：

210219​=29

个页表页即可。

一句话总结

对于 2GB 用户空间、4KB 页面、4B 页表项的平铺页表：

219 个虚拟页⇒219 个 PTE⇒2MB 页表⇒512 个页表页

这也说明了：即使在 32 位下，平铺页表的开销也已经不小；到了 64 位，若仍然平铺，页表大小会迅速变得不可接受。

所以多级页表的直觉是：把大数组改造成树。只有真的用到了某一片虚拟地址范围，才为那一层页表分配实际页面。x86-64 常见设计里，48 位虚拟地址会拆成多段索引，逐级下钻，根的位置由 CR3 指向。

flowchart TD
    VA[虚拟地址] --> L4[L4 索引]
    L4 --> L3[L3 索引]
    L3 --> L2[L2 索引]
    L2 --> L1[L1 / PTE]
    L1 --> OFF[页内偏移]
    L1 --> PFN[页框号]
    PFN --> PA[物理地址]
    OFF --> PA

Intuition

多级页表很像图书馆的“楼层 → 书架区 → 书架 → 某本书”。
你不需要一开始就把全馆所有目录都铺开，只给真正用到的区域建目录就行。

7. 反转页表：换个方向思考

普通页表是“从虚拟地址出发”查到物理页框，而且通常每个进程一张。
反转页表则反过来：从物理内存出发，系统维护一张总表，表项记录“哪个进程的哪个虚页”占了这个页框。这样表的规模跟物理内存大小成比例，而不是跟所有进程的虚拟地址空间总量成比例。某些体系结构还会把 (pid, vpn) 作为key哈希后去找对应表项。

Example

普通页表像“每个学生一本选课记录”。
反转页表像“每门课点名册”，你从课程席位反查现在坐的是谁。

8. 地址转换：虚拟地址到底怎么走到物理地址

这一部分建议你记住一句伪代码：

  • 如果该虚页不在内存、非法，或者权限不允许
    → 硬件触发异常，陷入操作系统

  • 否则
    → 从页表拿页框号，与页内偏移拼接成物理地址

MMU 就是干这件事的。操作系统从更高层看待它时，会把存储抽象成进程地址空间。所以你平时看到的代码段、数据段、堆、栈、共享库映射区，本质上都是这个地址空间里的不同区域。

flowchart TD
    VA[虚拟地址 = 虚页号 + 页内偏移] --> PT[查页表]
    PT -->|有效| PFN[页框号]
    PFN --> PA[物理地址 = 页框号 + 页内偏移]
    PT -->|无效 / 非法 / 保护失败| EX[Page Fault]
    EX --> K[进入内核处理]

9. TLB：为什么页表查找不会慢到离谱

如果每次访存都要先访问页表，再去访问目标内存，那内存访问至少翻倍，CPU 会非常痛苦。
所以硬件里会放一个TLB（快表），本质是相联存储器，缓存当前运行进程页表中的一小部分热点表项。命中时，虚拟页号可以直接拿到页框号；未命中时，才走页表查找。

进程切换时，TLB 还会遇到刷新问题。刷新得太狠，性能掉得明显；因此会有 ASID / PCID 这类机制，把不同地址空间的 TLB 项区分开，减少不必要清空。

Intuition

页表像一本大字典，TLB 像你桌上贴的便利签。
常查的几个词先记在便利签上，真的快很多。

10. 缺页异常：这是虚拟内存真正“活起来”的地方

最常见的 Page Fault 就是缺页异常：硬件查页表时发现该页当前不在内存，于是触发异常；操作系统接手后找到磁盘上的后备位置，把页调进来；如果没有空闲页框，还要先选一个牺牲者换出去。若牺牲页是脏页，还得先写回。

flowchart TD
    REF[CPU 访问某虚拟页] --> CHK{页在内存?}
    CHK -->|是| OK[继续执行]
    CHK -->|否| PF[硬件触发缺页异常]
    PF --> OS[内核缺页处理程序]
    OS --> FREE{有空闲页框?}
    FREE -->|有| LOAD[从磁盘调入]
    FREE -->|无| VICTIM[选择牺牲页]
    VICTIM --> DIRTY{牺牲页脏吗?}
    DIRTY -->|是| WB[先写回磁盘]
    DIRTY -->|否| LOAD
    WB --> LOAD
    LOAD --> UPD[更新页表 / TLB]
    UPD --> RET[返回并重启指令]

Trap

缺页异常不是“程序错了”，它很多时候是虚拟内存正常工作的组成部分。
真正该警惕的是频繁缺页，那通常意味着驻留集太小、局部性太差，或者系统开始抖动了。

11. 驻留集：每个进程到底能占多少页框

驻留集就是某时刻某进程实际驻留在物理内存中的页框集合。
分配策略有两种：

  • 固定分配：进程创建时就定好
  • 可变分配：根据缺页率动态增减页框数

驻留集太小，会疯狂缺页；太大，又会挤压别的进程。这里没有绝对最优值，只有跟程序局部性相匹配的“比较合适”的范围。

Note

课上那张“页框数—缺页率”曲线你只要记住一个感觉：
一开始多给几个页框，收益特别大；再往后继续加，边际收益迅速下降。

bladdy现象。

二、页面置换算法

1. 先分清：局部置换 vs 全局置换

发生缺页、需要腾位置时，候选页到底从哪里挑？

  • 局部置换：只在当前缺页进程自己的驻留集中挑
  • 全局置换：在系统全部未锁定页框里挑

固定分配通常配局部置换；可变分配既可能局部，也可能全局。
这背后的区别其实很重要：局部置换比较“各扫门前雪”，全局置换更激进，更追求系统整体吞吐，但进程之间的干扰也更明显。

2. 页框锁定：为什么有些页不能换

不是所有页都能被置换。
正在 I/O 的页、内核关键代码、关键数据结构、I/O 缓冲区这类东西常常会被锁定，避免虚存机制把它们换出导致不可预测延迟。Windows 里还给了 VirtualLock / VirtualUnlock 这样的接口。

Trap

置换算法再聪明，也不能越过“锁定页不能换”这个硬约束。

3. 一组你必须会比较的置换算法

OPT

理想最优算法，换掉“未来最久才会再用”甚至“未来再也不用”的页。它几乎只用于做理论基准，因为现实中你不可能知道未来。

FIFO

最早进来的最先走。实现简单，但可能把特别重要、马上还要用的页也赶走。

Second Chance / Clock

它们是对 FIFO 的修补：
先看访问位 R。如果某页最近用过，就给它一次机会，把它跳过去，甚至顺手把 R 清零。Clock 就是把这个思路做成一个环形缓冲区 + 指针扫描，工程上比精确 LRU 实用得多。

flowchart LR
    P[时钟指针指向某页] --> R{R 位 = 0 ?}
    R -->|是| OUT[选它换出]
    R -->|否| CLR[将 R 清零]
    CLR --> NEXT[指针继续前移]
    NEXT --> P

NRU

利用访问位 R 和修改位 M 把页分四类：

  • 第 0 类：未访问，未修改
  • 第 1 类：未访问，已修改
  • 第 2 类：已访问，未修改
  • 第 3 类：已访问，已修改

优先从编号更小的非空类里挑。它很粗糙，但成本低。

LRU

最近最久未使用。思想特别自然，也很接近 OPT，但精确实现通常代价大：要么打时间戳，要么维护栈。

NFU / Aging

这两个都是“便宜点模拟 LRU”的思路：

  • NFU（最不经常使用算法）：周期性把 R 累加到计数器
  • Aging（老化算法）：计数器先右移，再把当前 R 放到最高位，更能体现“新近访问”比“很久以前访问”更重要

Intuition

如果说 LRU 是问“谁最久没被翻出来看过”，
那 Aging 就是在做“最近一段时间的记忆加权”。

4. 算法优劣怎么背

课上最后给的味道大概是这样：

  • OPT：不可实现，但可作上界
  • FIFO：简单，但会犯傻
  • Second Chance / Clock：现实可用
  • LRU：很强，但实现贵
  • Aging：是很有效的 LRU 近似
  • Working Set / WSClock：思想好，但要平衡实现成本

5. Belady 现象：页框变多，缺页反而变多

这件事第一次看到很反直觉，但真的会发生在 FIFO 里。
也就是说，分给进程更多页框，不保证缺页数一定下降。这个异常现象就叫 Belady 现象。

Trap

“页框更多 ⇒ 缺页更少”不是普遍真理。
对某些算法成立，对某些算法不成立。面试和考试都很爱拿这个下套。

6. 抖动 Thrashing：系统开始“忙着换页，忘了干活”

如果页面在内存和磁盘之间高频来回，系统花在调页上的时间比真正执行程序的时间还多，这就叫 抖动。
抖动常见原因有：驻留集太小、并发进程太多、工作集装不下、程序局部性太差。

7. 页大小、程序写法，也会影响缺页

页面大小不是越大越好，也不是越小越好：

  • 小页：内部碎片小，但页表更长
  • 大页：页表更短、TLB 覆盖更大，但内部碎片更明显，还会增加管理复杂性

程序写法同样很关键。课上的矩阵例子特别经典：二维数组按行存放时，按行遍历局部性很好；按列遍历会把局部性打烂。通常默认每个元素 4B、页面 4KB 时，一行正好约一个页面，所以按行访问缺页远少于按列访问。

Example

这就是为什么写高性能代码时，数据布局和访问顺序经常比“多写几行算法优化”还重要。

8. 工作集：真正该给一个进程多少页框

工作集模型的想法非常漂亮：
在某一段时间里，程序其实只会集中访问一小撮“活跃页面”。如果你给它的页框数大致能装下这些活跃页，它就跑得顺；装不下，就会频繁缺页。

定义是：

W(t,Δ)=进程在过去 Δ 个虚拟时间单位中访问过的页面集合

这里要注意三件事：

  • 它跟访问序列有关
  • 跟当前时刻 t 有关
  • 跟窗口长度 Δ 有关；Δ 越大，工作集一般越大

Trap

工作集和驻留集不是一回事。
工作集是程序运行时自然表现出来的“活跃页集合”；
驻留集是系统当前实际分给它的物理页框集合。

工作集算法的大致做法是：给每页记录最后访问时间，周期性看它是不是落在“当前时间减阈值 T”之前；太久没用的页，就认为已经掉出工作集，可以考虑换出。实际系统里，很多时候会用缺页率上下界来近似这个思想。

9. 清除策略、页缓冲、加载控制

这三块很像“系统级后勤保障”。

清除策略

理想状态是：发生缺页时，系统手里已经有一批干净空闲页框，而不是临时现找。
所以会有一个 paging daemon，平时睡着，空闲页少了就醒来，提前回收并清洗页面。还有双指针时钟这样的实现，让前指针负责把脏页写回，后指针负责真正替换。

页缓冲

被换出的页先别急着彻底忘掉：

  • 干净页进空闲链表
  • 脏页进修改链表，成批写回
  • 如果进程很快又要它，可以低成本重新拿回来

加载控制

内存里同时驻留多少进程，会显著影响整体效率。并发度过高时，系统可能选择挂起一部分进程，把它们占的页换到磁盘上，降低负载。

Intuition

你可以把这三者想成“保洁 + 仓储 + 限流”。
没有它们，虚拟内存机制会一直处在手忙脚乱的救火状态。

三、内存映射文件

1. mmap 的本质：把“文件 I/O”变成“内存访问”

mmap 的思想非常优雅：
不是 read/write 文件，而是把文件的一段直接映射到进程虚拟地址空间的一块区域里。之后程序访问这块区域，就像访问一个大数组。首次访问某页时，系统再按页把内容调进来；解除映射或进程退出时，修改过的页再写回文件。

flowchart LR
    FILE[文件中的一段内容] --> MAP[mmap 建立映射]
    MAP --> VMA[虚拟地址空间中的一段区域]
    VMA --> ACCESS[像访问数组一样访问]
    ACCESS -->|首次触达某页| PF[缺页异常]
    PF --> LOAD[按页读入内存]

Intuition

普通文件 I/O 像“你去仓库搬货”；
内存映射像“仓库直接给你开一扇门，让你伸手就能拿”。

2. mmap 里最该记的参数

函数原型是：

你复习时重点盯住两个东西：

  • prot：这段区域的访问权限，对应 vm_area_struct 里的 vm_prot
  • flags：映射对象的类型，对应 vm_area_struct 里的 vm_flags

其中常见类型：

  • MAP_PRIVATE：私有写时复制对象
  • MAP_SHARED：共享对象
  • MAP_ANON：匿名映射，对应请求零页
  • MAP_PRIVATE | MAP_ANON：常见于 .bss、堆、用户栈等区域

3. Linux 怎么描述一个进程的虚拟空间

这一块建议你直接把图景记住：
task_struct 代表进程；进程虚拟地址空间里会有一串 vm_area_struct，每个 VMA 描述一个连续区域，里面记着 vm_start、vm_end、vm_prot、vm_flags、vm_next 这些信息；再往下是页表，把虚页映射到物理页框；而映射来源可能是 ELF 文件里的 .text、.data、.bss、.rodata 等节。

flowchart LR
    TASK[task_struct / 进程] --> VMA1[vm_area_struct: text]
    TASK --> VMA2[vm_area_struct: data]
    TASK --> VMA3[vm_area_struct: heap/stack/mmap]
    VMA1 --> PT[页表]
    VMA2 --> PT
    VMA3 --> PT
    ELF[ELF 文件各节] --> VMA1
    ELF --> VMA2
    PT --> MEM[物理页框]

Note

这里一定要有“VMA 负责描述区域，页表负责逐页映射”这个层次感。
这两个不是一个东西。

4. 共享映射与私有写时复制

共享对象

比如共享库。多个进程用的是同一份代码，磁盘里一份，内存里也尽量只放一份；各进程页表都指向同一个物理页框。对共享区域的写，会对共享者可见，并反映回对应对象。

私有写时复制对象

比如可执行文件和某些私有映射。开始时多个进程也可以共享同一物理页，但都先标成只读；谁先写，谁触发异常，系统这时才真正复制出一个新页框给它。

Trap

写时复制不等于“一开始就复制”。
如果父子进程都只读，那物理页根本不需要真的复制。

5. 为什么 fork 会很快：因为有 COW

早年 fork() 的笨办法，是把父进程地址空间逐页复制给子进程。
后来 Linux 用 COW（Copy-On-Write） 优化：先复制 mm_struct、vm_area_struct 和页表结构，把相关页都标只读并标记为私有 COW；这样 fork() 返回时，父子进程已经各自拥有“看起来完整独立”的地址空间，但大部分物理页还没复制。后面谁写谁复制。尤其在 fork() 后马上 exec() 的常见模式下，这个优化特别赚。

Example

父子进程像两个人先共用一份复印稿。
只要都只是看，就一直共用；
谁想在纸上写字，系统才临时给他单独复印一份。

6. mmap 和 COW 的关系

这两个机制关系很近：

  • mmap 负责“把文件或匿名对象接到地址空间里”
  • COW 负责“当这个映射被声明为私有、但出现写入时，怎样延迟复制”

所以你会发现，fork() 优化和 MAP_PRIVATE 的核心精神其实是一致的：
先共享，写时再分家。

7. 一个很实用的例子：LMDB

LMDB 就是典型的内存映射型数据库。它把数据集放在映射文件里，很多时候能把文件访问做得非常接近内存访问，适合大规模数据集场景。

四、Windows 虚存管理

4.1 Intel x86 虚拟内存机制

1. 先抓主线：逻辑地址不是直接变物理地址

在 x86 的保护模式语境里，地址转换通常分两步：

  1. 段式转换：逻辑地址 → 线性地址
  2. 页式转换：线性地址 → 物理地址

这里会出现三个寄存器：

  • CR0：控制分页机制是否开启
  • CR2：发生页故障时，硬件把出错的线性地址放这里
  • CR3：保存页目录的物理基地址
flowchart LR
    LA[逻辑地址 = 段选择符 + 段内偏移] --> SEG[段式转换]
    SEG --> LIN[线性地址]
    LIN --> PAGE[页式转换]
    PAGE --> PA[物理地址]

2. 段选择符和段描述符在干嘛

保护模式下，段选择符会去 GDT/LDT 中找到对应的段描述符。描述符里有段基址、界限、类型、特权级等信息。
所以分段机制除了“形成线性地址”，还顺手承担了边界检查和权限保护。而现代系统常用的“平坦内存模型”，本质上就是把段基址设成 0、范围拉很大，让分段在使用上几乎“透明”，重点把内存管理交给分页。

Intuition

分段像先按“街区”找位置，分页像再按“门牌号”找具体房间。
现代系统里，前一道街区划分被故意做得很平，真正细致的管理都压到分页去了。

4.2 Windows 虚拟内存管理

1. Windows 内存管理器在系统里的位置

Windows 的内存管理器位于 Ntoskrnl.exe 中。它不是一个单点函数，而是一组服务程序加上页面错误陷阱处理程序。前者负责虚拟内存的分配、回收和管理，后者负责在硬件发现内存异常时收拾残局。

关键组件你可以记这几个：

  • MmWorkingSetManager：负责工作集修整、老化、触发写回等
  • KeSwapProcessOrStack：负责进程/内核线程栈的换入换出
  • MiModifiedPageWriter：把脏页写回页文件
  • MiMappedPageWriter：把映射文件的脏页写回磁盘
  • MmZeroPageThread：把空闲页清零

Note

这一串名字不一定要求你逐个背 API 风格的细节，但你要知道：
Windows 把“工作集管理、脏页写回、零页准备、进程/栈交换”都明确做成了独立职责。

2. Windows 虚拟内存的大图景

Windows 进程的虚拟空间里，既有 EXE / DLL 的代码和静态数据，也有动态堆、线程栈、环境块、共享页；磁盘上则有页面文件作为后备存储。

在课里的 32 位布局里，可以看到经典的 3GB 用户空间 + 1GB 系统空间（也提到了 /3GB 选项）。用户空间里常见的是应用代码、全局变量、线程栈、DLL、堆、PEB/TEB、共享用户数据页；系统空间里则放内核和执行体、HAL、引导驱动、进程页表、超空间、系统缓存、分页池和未分页池等。

Trap

你要有“用户空间和系统空间不是两种物理内存，而是同一套虚拟地址空间里的不同区域”这个意识。

3. Windows 的地址转换：什么时候硬件自己干，什么时候必须内核上场

当 PDE、PTE 都是有效时，CPU 直接根据它们把虚拟地址转成物理地址，这时操作系统不需要介入。
当 PTE 无效时，格式解释权就交给操作系统了：它需要判断这到底是普通未提交页面、权限错误、COW 写、栈增长、已提交但未映射，还是需求零页。

Intuition

有效 PTE 像“绿灯，硬件自己通行”；
无效 PTE 像“红灯，但红灯背后可能有很多原因”，于是必须请内核交警出来判断。

4. Windows 缺页异常：控制流怎么走

Windows 下，页故障异常号是 0xE。当 CPU 在地址转换时发现页表项无效，会把出错地址写到 CR2，然后根据中断描述符表找到相应处理入口，转到 ntoskrnl!KiTrap0E 一类的缺页处理路径。

这一点很值得你记住：
Page Fault 的“发现”是硬件做的，Page Fault 的“解释与修复”是操作系统做的。

flowchart TD
    ACC[指令访问虚拟地址] --> MMU[硬件查 PDE / PTE]
    MMU -->|有效| OK[完成访问]
    MMU -->|无效| PF[触发 0xE Page Fault]
    PF --> CR2[出错地址写入 CR2]
    CR2 --> IDT[根据 IDT 找处理入口]
    IDT --> HANDLER[KiTrap0E / 缺页处理]
    HANDLER --> FIX[补页 / 扩栈 / COW / 权限检查]

5. Windows 的内存分配：保留和提交不是一回事

Windows 用户空间里，以页为单位的虚拟内存分配主要靠 VirtualAlloc / VirtualFree。这里最容易混的是两个阶段：

  • 保留（reserve）：先占下一段虚拟地址范围
  • 提交（commit）：再把物理存储承诺给这段地址中的某些页

光看页表，其实能看出某页是否已映射，但很难优雅地知道一整段地址空间是否已经被“保留”。于是 Windows 引入了 VAD（Virtual Address Descriptor）：为每个进程维护一组虚拟地址描述符，描述各段已分配虚拟区间的状态，并把它们组织成一棵自平衡二叉树，根挂在 EPROCESS 里。

Trap

“保留了地址空间”不等于“已经有了物理页”。
reserve 更像画地盘，commit 才是正式拨资源。

6. 页目录自映射：Windows 里很有味道的实现细节

Windows 这部分挺有实现感。课上强调了：

  • 页目录的物理地址保存在 KPROCESS 中
  • x86 里 CR3 保存页目录物理基址
  • 页目录还会被映射到固定虚拟地址上，这样内核就能通过虚拟地址直接访问页目录和页表本身
  • 相应还有 MiGetPdeAddress()、MiGetPteAddress() 这样的宏，根据某个虚拟地址快速算出它自己的 PDE / PTE 地址

Note

这一招的意义很工程化：
页表是管理虚拟内存的元数据，而内核最好也能像访问普通内存一样方便地访问这些元数据。

7. Windows 的工作集

Windows 里，工作集是“当前驻留在物理内存中的虚拟页面子集”。既有：

  • 进程工作集
  • 系统工作集

它的变化过程可以这样理解：

  • 进程刚启动时，访问新页面多，工作集会逐步扩张
  • 当局部性区域稳定下来，工作集大小也相对稳定
  • 当程序切换到新的局部性区域，工作集会快速扩张/收缩，进入新的稳定阶段

Intuition

工作集像你考试前桌上摊开的那几本书。
你在学操作系统时，桌上主要是 OS、体系结构、一点点代码；
切到概率论时，桌上那批书会迅速换一套。
桌上的那一小撮“当前真正活跃的资料”，就是工作集。

8. Windows 的物理内存管理：PFN 数据库和页面状态链表

这一块特别值得从“状态机”角度看。

Windows 会维护 页框号数据库（PFN database），并把物理页框按状态组织到多条链表里。课上重点画出来的有：

  • active(valid)：当前在工作集中、有效使用中
  • standby：页已不在工作集，但内容还留在内存里；对应 PTE 往往是 invalid + transition，如果再次访问，往往能形成软缺页
  • modified：页被改过，后续要由写出线程写回
  • free
  • zeroed
  • bad
flowchart LR
    ACTIVE[active / valid] --> TRIM[工作集修整]
    TRIM --> STANDBY[standby]
    ACTIVE --> WRITE[被修改]
    WRITE --> MOD[modified]
    MOD --> WRITER[写出线程]
    WRITER --> STANDBY
    FREE[free] --> ZERO[zeroed]
    ZERO --> DEMAND[需求零缺页]
    STANDBY --> SOFT[软缺页后恢复 active]

Example

standby 页很像“刚从你桌上收进抽屉，但还没扔”的资料。
你要是马上又想看，伸手拿回来就行；
真要等它被别的东西覆盖了，那就只能重新去仓库里找了。

9. Windows 里的 COW 也完全成立

课上最后那张图本质上就是在说：
两个进程可以先共享多页内容，并把它们都标成写时复制；一旦某个进程尝试写其中一页，系统才为它复制一个新的物理页，新页对它私有，对其他共享者不可见。

五、最后收个总线：这一章真正该抓住什么

如果你现在准备睡前过一遍，我建议你把这一章压成下面这条主线：

  1. 虚拟内存 = 地址空间抽象 + 页式映射 + 缺页处理
  2. PTE 不只是映射，还承载权限、访问历史、脏位、缓存策略
  3. 多级页表解决页表太大，TLB 解决地址转换太慢
  4. Page Fault 是硬件发现、操作系统处理
  5. 驻留集、置换算法、工作集，决定系统会不会抖
  6. mmap 和 COW 把“文件”“进程地址空间”“物理页共享”串了起来
  7. Windows 那部分要有实现感：CR2/CR3、KiTrap0E、VAD、PFN 数据库、自映射、工作集。

Intuition

这一章表面上在讲“内存”，其实讲的是一个更大的主题：
操作系统如何把底层资源伪装成一个对程序友好的抽象世界。
CPU 只想要快，程序只想要大，磁盘只会慢吞吞地转。
而虚拟内存，就是这三方矛盾被系统设计巧妙调和后的结果。
