一、目标定义

这个工具的目标不是做“软件使用总时长统计”，而是做一个个人注意力时间线系统，回答下面几个问题：

  • 某一时刻用户的注意力主要放在哪个应用上
  • 如果前台是浏览器，当时具体在哪个域名上
  • 用户是否处于真实使用状态，而不是离开电脑、锁屏或空挂
  • 一天中有哪些连续专注块、有哪些高频切换和打断

所以核心输出应当是：

  • 时间线
  • 总时间分布
  • 专注块与切换分析

二、整体架构

整体分成三层就够了，不需要更复杂。

1. 本地采集服务（Rust）

这是整个系统的核心，常驻后台运行。

职责：

  • 监听 Windows 前台窗口变化
  • 监听用户活跃状态（active / idle / locked）
  • 接收浏览器扩展上报的前台标签页信息
  • 将原始事件合并为连续时间段
  • 存入本地数据库
  • 提供本地 HTTP API 给前端读取

这一层建议用 Rust 实现。它既是“监听器管理者”，也是“轻量后端”。

2. 浏览器扩展

只做一件事：
当浏览器在前台时，上报当前活动标签页的域名等少量信息。

职责：

  • 获取当前活动标签页
  • 提取域名
  • 可选上报标题
  • 将信息发送给本地采集服务

它不负责存储，也不负责分析。

3. 网页前端

使用本地网页完成可视化。

职责：

  • 展示时间线
  • 展示总时间分布
  • 展示专注块、切换次数、主要打断源

前端只读数据，不参与采集。

三、核心设计原则

1. 事件驱动优先，不做低频轮询采集

前台窗口变化不应该 30 秒采一次，因为这样会丢掉最重要的切换信息。
正确方式是：

  • 前台切换：事件驱动
  • 活跃状态变化：事件驱动或短周期检测
  • 写库/汇总：批量执行

也就是说，30 秒可以用于“统计刷新”或“批量落盘”，但不能作为主采样周期。

2. 以“连续时间段”作为主数据，而不是离散点

时间线最适合展示的是 segment，而不是采样点。

例如：

  • 09:10–09:42 VS Code
  • 09:42–09:47 Edge / bilibili.com
  • 09:47–10:03 Obsidian

所以系统内部应该自动把事件合并成连续区间，而不是只存“09:10 在 VS Code、09:10:30 在 VS Code”。

3. 浏览器只记录“前台标签页”的域名信息

你说得对：人的注意力本质上是串行的。
因此浏览器部分没必要追踪所有标签页，只需要记录：

  • 当前前台浏览器窗口里的活动标签页
  • 域名
  • 可选标题

这样简单、隐私风险也更低。

4. 全部数据本地存储，默认隐私优先

第一版建议默认只存：

  • 应用名
  • 窗口标题
  • 浏览器域名
  • 活跃状态

默认不存：

  • 页面正文
  • 完整 URL 参数
  • 输入内容
  • 剪贴板

这样产品定位更清晰，不容易变味。

四、模块设计

1. 采集层

A. Windows 前台窗口监听器

负责捕捉当前活动窗口变化，生成“窗口切换事件”。

记录字段建议包括：

  • 时间戳
  • 进程 ID
  • 可执行文件路径
  • 应用显示名
  • 窗口标题
  • 窗口句柄
  • 是否浏览器

这里要注意，不能只记录进程名，因为对用户不直观。
要做一层“进程/路径 → 应用显示名”的映射，例如：

  • Code.exe → VS Code
  • msedge.exe → Microsoft Edge
  • WeChat.exe → 微信

B. 活跃状态监听器

负责判断用户是否真的在使用设备。

建议状态至少分成：

  • active
  • idle
  • locked

这样你统计的就不是“电脑开着多久”，而是“人真实在用多久”。

C. 浏览器扩展监听器

负责补充浏览器前台标签页的更细信息。

建议第一版只支持：

  • Chrome
  • Edge

字段建议包括：

  • 时间戳
  • 浏览器名
  • 浏览器窗口 ID
  • 标签页 ID
  • 域名
  • 页面标题（可选）

扩展只在这些情况下发送更新：

  • 标签页切换
  • 前台窗口切到浏览器
  • URL 发生变化
  • 定时心跳补充当前状态

2. 后端服务层

建议做成一个非常轻量的本地服务。

职责：

  • 接收采集事件
  • 将事件合并为 segment
  • 维护应用映射
  • 写 SQLite
  • 提供本地 REST API

这一层不需要复杂调度系统，也不需要微服务。
一个本地进程就够了。

技术上可以考虑：

  • Rust
  • Tokio
  • Axum
  • SQLite

3. 存储层

建议直接用 SQLite。
第一版完全够用，而且方便调试和导出。

建议至少有这几张表：

app_registry

存应用映射信息。

字段示例：

  • app_id
  • exe_path
  • process_name
  • display_name
  • icon_path（可选）
  • category（可选）

focus_segments

前台应用连续区间。

字段示例：

  • id
  • start_time
  • end_time
  • app_id
  • window_title
  • hwnd
  • process_id
  • is_browser

browser_segments

浏览器前台标签页连续区间。

字段示例：

  • id
  • start_time
  • end_time
  • browser_app_id
  • browser_window_id
  • tab_id
  • domain
  • page_title

presence_segments

用户状态区间。

字段示例：

  • id
  • start_time
  • end_time
  • state (active / idle / locked)

raw_events（可选）

用于调试。

五、核心数据流

整个系统的主流程可以这样设计：

1. 前台窗口切换

当 Windows 检测到前台窗口变化时：

  • 结束上一条 focus_segment
  • 开启新的 focus_segment
  • 如果新窗口是浏览器，等待浏览器扩展补充域名信息

2. 浏览器标签页变化

当扩展检测到前台标签页变化时：

  • 如果当前前台窗口是浏览器
  • 结束上一条 browser_segment
  • 开启新的 browser_segment

3. 活跃状态变化

当用户从 active 进入 idle 或 locked 时：

  • 结束当前 presence_segment
  • 开启新的状态段
  • 前端统计时只把 active 状态下的前台时间计入“真实使用时间”

六、前端展示方案

前端第一版只做三个页面就够了。

1. 每日时间线

这是最重要的页面。

展示方式：

  • 横轴是时间
  • 第一层显示前台应用 segment
  • 如果某段是浏览器，可以展开显示域名层
  • 用不同颜色区分应用类别
  • 用灰色或斜纹表示 idle / locked

用户能直接看到：

  • 上午做了什么
  • 哪些时间段在刷网页
  • 哪些正事被中断了

2. 时间分布页

统计：

  • 按应用总时长
  • 按域名总时长
  • 按类别总时长

例如：

  • VS Code 2h 10m
  • Edge 3h 05m
  • bilibili.com 48m
  • github.com 37m

3. 专注分析页

建议做几个简单指标：

  • 最长连续专注块
  • 平均专注块时长
  • 每小时切换次数
  • 高频打断应用
  • 浏览器插入正事流程的频率

这一页会比单纯时长统计更有价值。

七、第一版范围控制

为了避免项目膨胀，MVP 建议只做这些：

必做

  • Windows 单机版
  • 前台窗口监听
  • active / idle / locked 状态
  • Chrome / Edge 扩展
  • SQLite 存储
  • 本地网页时间线
  • 总时间分布

暂不做

  • 多设备同步
  • 云端存储
  • 所有后台进程快照
  • AI 自动分类
  • 复杂规则引擎
  • 截图
  • 页面正文解析
  • 全量 URL 追踪

这样项目边界会清楚很多。

八、建议的实现顺序

推荐分四步做。

第一步：先做最小闭环

完成：

  • Windows 前台窗口监听
  • SQLite
  • focus_segments
  • 简单网页时间线

这一步先验证“前台应用时间线”有没有价值。

第二步：加入 presence

完成：

  • active / idle / locked
  • 在时间线上标出不可计时段

这一步让“真实使用时间”成立。

第三步：加入浏览器扩展

完成：

  • Edge / Chrome 当前标签页域名上报
  • browser_segments
  • 浏览器段可展开查看域名

这一步会让你的 timeline 真正和普通软件区分开。

第四步：做分析页

完成：

  • 总时间分布
  • 切换次数
  • 专注块

到这一步就已经是一个完整可用产品了。

九、简要方案总结

可以把你的设计整理成下面这个版本：

这是一个运行在 Windows 上的本地个人注意力时间线系统。
系统由 Rust 本地采集服务、浏览器扩展和网页前端三部分组成。
本地采集服务负责监听前台窗口变化、用户活跃状态变化，并接收浏览器扩展上报的前台标签页域名信息；随后将这些原始事件合并为连续时间段，存入 SQLite，并通过本地 HTTP API 提供给前端。
前端以时间线为核心展示每日注意力流向，并提供总时间分布与专注分析。
第一版只关注“前台应用、前台网页域名、真实活跃状态”三类数据，全部本地存储，强调简单、隐私友好和可解释性。
