操作系统概述

一、操作系统的定义与系统位置

操作系统是计算机系统中的系统软件，是一组彼此协作的程序模块集合。它直接位于硬件与应用之间，负责组织和管理处理器、内存、外设、文件等软硬件资源，控制程序执行过程，向应用与用户提供统一的服务接口，并在此基础上扩展机器的可用能力。

操作系统的基本作用可归结为三个方面。其一是资源管理者，以有效、合理的方式组织和分配有限资源，提高利用率并维持系统秩序。其二是服务提供者，为程序运行、文件访问、设备使用、内存申请、进程创建等活动提供标准化接口。其三是机器能力扩展者，将原始、零散、难以直接使用的硬件能力组织为统一、稳定、可编程的抽象环境。

与一般应用软件相比，操作系统具有更强的全局性、持续性与特权性。它长期驻留并控制系统运行，必须处理来自程序、设备、时钟和外部环境的异步事件；必须在多个执行流之间建立隔离与保护；必须在效率、正确性、安全性与可扩展性之间维持平衡。这些性质使操作系统既是程序系统，又是运行时秩序的建立者。

    U[用户] --> A[应用程序]
    A --> API[系统调用 / API / 运行库]
    API --> K[操作系统内核]
    K --> CPU[处理器]
    K --> MEM[内存]
    K --> DEV[设备]
    K --> DISK[磁盘与文件存储]

1. 虚拟机思想

操作系统并不只是 ” 管理硬件 “，更重要的是把物理资源转换为逻辑资源，将裸机提升为可用的虚拟机。物理处理器被抽象为每个进程都仿佛独占的 ” 虚处理器 “；物理内存被抽象为进程私有的地址空间；块设备被抽象为文件、目录和逻辑数据对象；终端与显示设备被抽象为窗口、会话与虚拟终端。

这种抽象并不意味着资源被无限创造，而是通过时间复用与空间复用，使一个物理实体在逻辑上表现为多个相互隔离、相互独立的服务对象。虚拟化是操作系统提升易用性、实现共享、建立保护、支持并发的共同基础。

2. API 与标准运行环境

虚拟机抽象必须通过接口暴露给应用。操作系统提供系统调用与相应的标准库，应用通过这些接口发出资源请求，而不必直接操作硬件。接口的稳定性使应用能够在硬件变迁中保持可移植性；接口的抽象性使程序能够围绕进程、文件、地址空间、套接字、管道等对象组织逻辑，而不必面对寄存器、控制器和设备时序。

3. 设计与实现目标

操作系统的设计目标通常围绕以下几组要求展开：

  • 抽象性：为复杂硬件提供清晰、稳定的逻辑接口。
  • 模块化：将功能拆分为相对独立的子系统，降低复杂度。
  • 实现可维护性：尽量使用高级语言而非纯汇编，以提高可读性、可移植性与演化能力。
  • 性能：减少额外指令开销、内存开销和磁盘开销。
  • 保护与隔离：在应用之间、应用与内核之间建立边界。
  • 可靠性与安全性：在异常、错误或恶意行为存在时保持系统可控。
  • 节能与移动性：适应从大型系统到小型终端的不同硬件环境。

这些目标之间并不完全一致。更强的抽象可能带来更大开销，更强的保护可能增加路径长度，更高的模块化可能降低极端场景下的性能。操作系统设计的核心难点之一正是这些目标之间的权衡。

二、从不同角度认识操作系统

1. 资源管理观点

从自底向上的角度看，操作系统是资源管理者。它所管理的对象既包括处理器、内存、时钟、磁盘、网络接口等硬件资源，也包括文件、数据、目录等软件资源。资源管理的目标并不是机械地分配资源，而是在共享、利用率、保护、冲突协调与服务质量之间取得平衡。

资源管理通常包括三个基本环节：记录资源状态、执行资源分配、完成资源回收。为了做出分配决策，操作系统必须维护数据结构，用于描述资源是否空闲、归谁占有、使用时长、是否损坏、是否可共享等状态信息。分配策略既可以是静态的，也可以是动态的；现代操作系统更强调按运行时状态进行动态分配。

2. 进程观点

从系统运行的动态角度看，操作系统可以理解为一组彼此独立、又可并发执行的活动与一个协调核心的组合。这里的核心抽象是进程。进程是程序的一次执行过程，是具有动态生命期的活动实体，具有创建、运行、等待、终止等状态变化。进程观点强调的不是 ” 系统提供了哪些功能 “，而是 ” 系统中有哪些活动、这些活动如何被协调、谁在何时获得处理器 “。

3. 虚机器观点

从内部结构角度看，操作系统可以理解为多层抽象的叠加。较低层负责贴近硬件的控制与封装，较高层建立更强、更易用的逻辑对象。每一层向上一层提供一个经过简化和增强的 ” 虚机器 “，最终构成完整的操作系统环境。

flowchart TB
    H[硬件]
    L1[底层控制与中断处理]
    L2[进程 / 内存 / 设备管理]
    L3[文件系统与系统服务]
    L4[系统调用接口]
    L5[应用程序]

    H --> L1 --> L2 --> L3 --> L4 --> L5

这三种观点并不冲突。资源管理强调对象与分配，进程观点强调动态执行与协调，虚机器观点强调结构与抽象层次。三者共同构成理解操作系统的基本框架。

三、操作系统的基本特征

1. 并发

并发是操作系统处理多个同时性活动的能力。多个程序可以在宏观上同时推进，而在单处理器条件下，微观上任一时刻只有一个程序真正占用处理器，系统通过快速切换形成 ” 同时运行 ” 的效果。并发并不等于并行。并行强调多个处理器或多个执行单元在同一时刻真的同时执行；并发强调多个活动的交替推进与协调组织。

并发引入了若干基本问题：执行流之间会发生切换，切换需要保存与恢复运行现场；活动之间可能共享数据，因而需要保护与同步；多个活动对资源的请求顺序不可预测，系统必须处理竞争、阻塞、唤醒和调度。

2. 共享

共享是多个程序、多个用户或多个系统活动共同使用有限资源的现象。共享有两种基本方式。

第一种是互斥共享。某个时刻资源只能被一个活动占用，例如打印机、某些外设寄存器或临界数据结构。第二种是同时共享。多个活动可以在逻辑上同时访问同一资源，例如可重入代码、共享只读数据、磁盘文件的受控并发访问。

共享提高了资源利用率，但也带来了保护、隔离、公平性和最优化之间的冲突。共享越充分，系统越需要精细的访问控制与调度策略。

3. 虚拟

虚拟性表现为一个物理实体映射为多个逻辑实体。处理器通过时间片形成多个虚处理器，内存通过地址空间形成进程私有视图，显示设备通过窗口系统形成多个会话界面。虚拟不是幻觉式的包装，而是一种资源组织方式，它依赖操作系统对底层状态的精确维护与控制。

4. 随机性与不确定性

操作系统必须随时响应不可预测顺序出现的事件。进程推进速度不确定，设备完成时刻不确定，中断到来顺序不确定，用户输入与网络请求也不确定。这种随机性意味着系统在某一瞬间的精确状态往往难以重现，很多并发错误只有在特定时序下才暴露。操作系统因此必须以 ” 面向不确定事件的控制系统 ” 而非 ” 固定顺序的普通程序 ” 来设计。

    C[并发] --> S[共享]
    S --> P[保护]
    C --> SY[同步]
    C --> SCH[调度]
    V[虚拟] --> M[更高利用率]
    R[随机性] --> EH[异步事件处理]
    EH --> SY

四、操作系统的作用再解读

1. 操作系统解决的问题

如果没有操作系统，程序必须自行完成装入、设备控制、内存使用协调、错误处理、资源保护和多程序切换，软件开发会直接暴露在硬件细节和冲突风险之下。操作系统解决的根本问题是：如何在资源有限、事件异步、需求多样的条件下，建立一个可共享、可保护、可扩展、可编程的运行环境。

2. 操作系统为什么必要

操作系统之所以必要，原因并不只在于 ” 方便使用 “，更在于以下三点：

  • 物理硬件的接口过于底层、异构、分散，不适合作为应用的直接编程对象。
  • 多程序、多用户、多设备并存时，资源冲突与安全风险无法靠应用自行解决。
  • 计算机系统的价值不仅在于执行一个程序，而在于支持持续运行、并发服务、长期管理和复杂生态。

3. 操作系统的典型特征

与其他软件相比，操作系统最典型的特征是：

  • 面向全系统资源而非单一任务；
  • 同时承担抽象、控制、分配与保护等多重职责；
  • 必须在特权模式下运行并处理异步中断；
  • 必须长期驻留并维护持续变化的全局状态；
  • 必须为不同层次的软件提供兼容、稳定的接口。

五、发展历程与重要技术

操作系统的发展始终受到三类因素驱动：硬件技术的变化、应用需求的变化、软件技术的进步。其总体目标始终围绕两点：更充分地利用硬件资源，以及向用户提供更好的服务。

1. 第一阶段：批处理时代

在计算机早期，硬件昂贵而人工便宜，系统设计的重点是提高昂贵机器的利用率。最初的方式是控制台独占，一次只有一个用户占用整台机器。随后演进为批处理：用户提交作业，系统操作员收集作业形成一批，操作系统自动顺序执行，再由操作员返回结果。

批处理作业通常由三部分组成：用户程序、输入数据和作业说明书。作业说明书本质上是一种作业控制语言，用来描述装入、编译、连接、运行等控制步骤。典型作业流体现出 ” 命令卡片 + 程序卡片 + 数据卡片 ” 的组织形式。

1.1 单道批处理与多道批处理

单道批处理系统在内存中一次只保留一个用户作业。作业计算时占用处理器，作业等待输入输出时，处理器也常常只能空闲等待。系统吞吐率与资源利用率都较低。

多道批处理系统则将多个作业同时保留在内存中。当某个作业因 I/O 等待而停顿时，处理器可切换到其他就绪作业继续执行，从而实现计算与 I/O 的重叠，提高系统整体效率。这一机制要求系统具备更强的调度能力，并依赖存储保护，以防不同作业互相破坏。

flowchart TB
    subgraph 单道批处理
        A1[作业装入] --> A2[运行]
        A2 --> A3[I/O 等待]
        A3 --> A2
        A2 --> A4[结束]
    end

    subgraph 多道批处理
        B1[作业A运行] --> B2[作业A等待I/O]
        B2 --> B3[作业B运行]
        B3 --> B4[作业C运行]
        B4 --> B5[作业A就绪后再运行]
    end

多道程序设计带来的收益可以用一组典型指标概括：处理器利用率、内存利用率、磁盘利用率和打印机利用率都显著提高，系统总完成时间缩短，吞吐率上升，平均响应时间下降。这说明多道程序设计并不是简单地 ” 让更多作业同时存在 “，而是通过作业搭配与调度，使不同资源的空闲段彼此填补。

1.2 通道、中断与 I/O 重叠

慢速输入输出直接由主机完成时，CPU 会在等待设备期间处于低效状态。为缓解这一矛盾，早期系统引入了数据通道与中断机制：I/O 由专门硬件部件推进，CPU 在 I/O 发起后继续执行其他计算，待设备完成再通过中断通知系统处理结果。通道与中断为多道程序设计提供了重要硬件支撑。

1.3 脱机技术与卫星机

批处理系统的一个关键问题是，卡片机、纸带机、打印机等外设速度远慢于主机。为避免主机被低速 I/O 长时间占用，可以用外围机器先完成面向用户的输入与输出，再将中间结果放入磁带或磁盘，由主机进行集中计算。这样，主机的主要时间用于计算而非等待外设。

flowchart LR
    U[用户提交卡片 / 纸带] --> S1[外围机输入]
    S1 --> M1[磁带或磁盘暂存]
    M1 --> H[主机批处理计算]
    H --> M2[结果暂存]
    M2 --> S2[外围机输出]
    S2 --> P[打印结果]

2. SPOOLing：批处理时代的核心技术

SPOOLing 是 Simultaneous Peripheral Operation On-Line 的缩写，通常称为假脱机技术。它的核心思想是：利用磁盘作为高速缓冲区，将输入、计算、输出组织成彼此独立又可并行推进的任务流，使慢速外设不再直接阻塞主机计算。

系统在磁盘上建立输入井和输出井。输入进程负责把用户作业写入输入井，作业调度从输入井中挑选合适作业装入内存执行，输出进程再把结果从输出井送往打印机等设备。这样，输入、计算、输出三个阶段被解耦，形成流水式处理。

flowchart LR
    INDEV[输入设备] --> INP[输入进程]
    INP --> IW[输入井]
    IW --> JS[作业调度]
    JS --> MEM[内存中的多道作业]
    MEM --> OW[输出井]
    OW --> OUTP[输出进程]
    OUTP --> OUTDEV[输出设备]

SPOOLing 的意义体现在三个方面。

第一，提高 I/O 速度。低速设备的直接访问被转换为对磁盘输入井与输出井的操作，缩小了 CPU 与设备之间的速度鸿沟。

第二，将独占设备改造为共享设备。系统并不把打印机这类独占设备直接分给某个作业，而是在磁盘队列中为各作业建立缓冲区和请求表，多个作业因此可以同时 ” 请求打印 “，设备由守护进程按队列顺序统一服务。

第三，实现虚拟设备。从单个进程视角看，它仿佛独占了打印机、读卡机或其他外设；从系统视角看，设备实际上被排队复用。

以打印为例，应用程序将待打印文档写入打印队列，后台打印进程读取队列并驱动打印机输出。现代系统中的打印后台服务、邮件队列、作业队列、日志缓冲等机制，本质上仍然体现了 SPOOLing 的思想：用更快、更可管理的中间层吸收慢速或异步设备带来的阻塞。

3. 第二阶段：分时系统

随着终端普及，系统目标从 ” 提高机器利用率 ” 扩展为 ” 改善用户交互体验 “。分时系统通过把处理器时间划分为多个时间片，轮流为多个终端用户服务，使多个用户都获得近似独占机器的交互体验。其关键评价指标是响应时间，若终端发出命令的时刻为 tcmd​，系统给出响应的时刻为 tans​，则响应时间可写为 Tresp​=tans​−tcmd​。

分时系统的合理性来自用户行为特征：交互用户大多发出较短命令，命令之间存在思考停顿，因此 CPU 可以在用户等待期间服务其他用户。分时系统牺牲了一部分吞吐效率，换取更低的交互等待时间。

3.1 CTSS 与交互式操作系统

CTSS 是早期具有代表性的分时系统。它证明了 ” 一台机器服务多个联机用户 ” 在工程上是可行的，并形成了终端、时间片、用户会话、内存映像切换等关键概念。此类系统往往需要在主存和辅助存储之间切换用户映像，以在极其有限的内存条件下维持多用户交互。

3.2 Atlas 与虚拟存储

Atlas 的重要历史地位在于把虚拟存储和页面调度推进为可运行系统的一部分。它表明内存不再必须完全等于程序运行所需空间，程序可在逻辑上拥有更大的地址空间，系统则通过分块装入与置换在运行时维持这一抽象。后来的虚拟内存机制正是在这一路径上成熟起来。

3.3 分时与批处理结合

通用操作系统常将分时与批处理结合：前台优先处理需要频繁交互的作业，后台则运行时效性要求不强的大作业。其本质是用不同服务等级满足不同工作负载。

flowchart TB
    T[时间片时钟] --> FQ[前台交互作业队列]
    T --> BQ[后台批处理作业队列]
    FQ --> CPU[CPU]
    BQ --> CPU
    CPU --> RESP[交互响应]
    CPU --> THR[后台吞吐]

3.4 传统分时系统在今天的意义

传统分时系统的外形已经变化，但其核心思想仍然存在于现代系统中：终端会话、Shell、远程登录、服务器多用户共享、前台与后台任务区分、交互优先级、时间片轮转、公平性策略、资源复用和响应时间控制，都属于分时思想的延续。云主机、容器平台和多租户服务虽然规模远大于早期分时机，但仍在处理同一个根本问题：如何让多个使用者共享一组底层资源，并让每个使用者获得足够稳定的响应体验。

4. 重大失败与成功案例

操作系统历史并非线性上升，其间多次出现复杂度失控。

IBM OS/360 的发布伴随大量已知错误，反映出大型软件系统在需求膨胀、组织庞杂、模块耦合严重时的工程危机。Multics 从提出到发布经历长期延迟，说明 ” 功能先进 ” 并不自动转化为 ” 工程可控 “。这些经历推动了对模块化、分层结构、高级语言实现、接口清晰性和复杂度管理的重视。

与之相对，UNIX 是一个极具代表性的成功案例。它由贝尔实验室的开发者在较小规模机器上起步，形成了简洁、统一、可移植的设计风格。UNIX 的名称通常被解释为对 MULTICS 的戏拟式命名，常写作 UNICS，再演化为 UNIX，隐含 ” 较简单的一体化系统 ” 这一意味。

UNIX 的成功说明，操作系统并不必然依赖庞大而封闭的结构；一套清晰的抽象、一组稳定的接口和一套可演化的实现方式，往往比功能堆叠更重要。

5. Buffer Cache：文件子系统与块设备之间的关键层

在 UNIX 一类系统中，文件子系统与块设备之间通常存在 Buffer Cache。它的作用是将 ” 按文件访问数据 ” 的上层需求，转化为 ” 按块读写设备 ” 的下层操作，同时利用主存缓存磁盘块，减少慢速设备访问带来的代价。

Buffer Cache 解决了三个核心问题。

第一，速度鸿沟问题。磁盘远慢于内存，频繁直接访问磁盘会显著拖慢系统。缓存常用块后，重复读取可以直接命中内存。

第二，块设备抽象问题。磁盘设备以固定大小的数据块为基本单位，而文件访问未必与块边界对齐。Buffer Cache 用缓冲块把设备块与文件偏移之间的关系组织起来，使上层文件逻辑不必直接面对设备细节。

第三，写入组织问题。写请求可以先写入缓冲块，再由系统择机回写设备，从而合并写入、减少随机 I/O，并与调度策略协同。

其典型工作方式如下：文件系统根据文件偏移定位到目标磁盘块，先在缓存中查找对应缓冲块；若命中，则直接返回；若未命中，则分配缓冲块并发起块设备 I/O，把数据装入缓存后再交付上层。写操作则先修改缓冲块状态，再按回写策略提交设备。Buffer Cache 因而成为文件子系统、块层与设备驱动之间的重要桥梁。

flowchart LR
    PROC[进程读写文件] --> FS[文件子系统]
    FS --> BC[Buffer Cache]
    BC -->|命中| RET[直接返回数据]
    BC -->|未命中| BLK[块设备层]
    BLK --> DRV[设备驱动]
    DRV --> DISK[磁盘]
    DISK --> DRV --> BLK --> BC --> FS --> PROC

6. 第三阶段：个人计算时代

个人计算机出现后，系统目标从 ” 集中共享昂贵主机 ” 转向 ” 服务单个用户的高可用机器 “。早期 PC 资源有限，系统常接近单任务环境，操作系统更像一组例程库。随着硬件能力增强，存储保护、多道程序设计、图形界面和复杂驱动模型再次出现，PC 操作系统重新演化为大型综合系统。

7. 第四阶段：网络与分布式

网络的发展使不同机器可以方便地共享打印机、文件服务器、Web 服务与通信资源，操作系统因此必须处理网络管理、通信协议、安全控制和远程资源访问等问题。此后，系统进一步向分布式方向发展，不再只是在单机上管理本地资源，而是要在多台机器之间协调处理、共享资源并分担负载。

8. 第五阶段：移动计算

移动终端普及后，操作系统必须在体积小、能耗严、连接变化频繁的硬件环境中运行。移动系统强调电源管理、无线通信、传感器整合、安全沙箱和面向应用生态的统一平台能力。

9. 第六阶段：云计算

云计算操作系统并非单机内核的简单放大，而是面对海量服务器、存储和网络资源的整体管理系统。其职责包括基础硬件与基础软件管理、虚拟计算管理、分布式文件系统、业务与资源调度、安全控制等。它在逻辑上将整个数据中心整合为可按需使用的资源池，为云应用提供统一接口，并对海量任务进行资源编排。

10. 第七阶段：泛在计算与物联网

当计算嵌入到人、机、物交织的环境中，操作系统不再局限于传统计算机，而扩展到传感器、执行器、边缘节点、智能终端和各种联网设备。此时系统更加重视低功耗、实时响应、远程更新、自治协作和场景化服务。

11. 概念的重用与回归

操作系统中的许多思想并不会永久消失，而是随着技术条件变化重新出现。连续文件分配在某些只读介质中仍然合理；硬件保护随着多程序与安全需求不断强化；动态链接从早期大型系统延续至现代软件平台；集中式计算服务在云时代以新的规模和形式回归。这说明操作系统中的许多概念并非简单地 ” 过时 “，而是在新的成本结构和硬件条件下重新获得意义。

六、操作系统的分类

1. 传统分类

1.1 批处理操作系统

批处理操作系统面向成批提交的作业，强调吞吐率和机器利用率。它通常缺乏实时交互能力，适合编译、统计、批量转换等工作负载。其关键技术包括作业控制语言、多道程序设计、中断、通道与 SPOOLing。

1.2 分时操作系统

分时操作系统面向多用户交互，强调响应时间。时间片轮转、公平共享和前后台区分是其典型特征。它牺牲部分纯计算吞吐，换取更好的交互体验。

1.3 实时操作系统

实时操作系统要求系统在严格时间范围内对外部事件作出反应，并以高可靠性协调实时设备和任务。其关键参数不是平均吞吐，而是是否在截止时间前完成处理。

实时系统通常区分为两类。

  • 硬实时系统：某项动作必须在规定时刻或时间区间内完成，违反时限即视为系统失败。典型场景包括飞行控制、工业安全保护、汽车安全气囊触发。
  • 软实时系统：允许偶尔违反时限，但超时会降低服务质量。典型场景包括多媒体播放、行情显示、部分在线通信服务。

1.4 个人计算机操作系统

个人计算机操作系统通常面向单用户环境，强调界面友好、使用方便与丰富的应用生态。随着硬件增强，此类系统也具备多任务、保护、网络和图形子系统等大型操作系统特征。

1.5 网络操作系统

网络操作系统建立在计算机网络和协议标准之上，运行于各类计算机系统之中，提供网络管理、通信、安全和资源共享功能。它的重点在于让不同主机能够彼此通信和共享资源，但每台机器仍明显保留各自的本地系统边界。

1.6 分布式操作系统

分布式操作系统建立在网络基础之上，但目标更进一步：把多台机器组织为一个统一的系统，自动完成全局任务分配、资源共享和负载调度。它具有以下特征：

  • 统一性：对外表现为一个整体系统。
  • 更强资源共享：资源跨机器组织。
  • 透明性：用户无需显式感知资源所在节点。
  • 自治性：各主机地位平等，无固定主从。
  • 更高处理能力与可靠性：通过分布式协作获得扩展性与容错性。

网络操作系统与分布式操作系统的差别在于：前者强调联网机器之间的互联与共享，后者强调系统级的一体化抽象。

1.7 嵌入式操作系统

嵌入式操作系统运行在设备、装置或更大系统的组成部分中，通常面向特定功能、反应式任务或较严格的时间要求。它并不一定运行在传统意义上的 ” 通用计算机 ” 上，而可能存在于路由器、控制器、传感器、家电、穿戴设备和工业部件中。其核心要求通常是小型化、可靠性、定制性和能耗控制。

2. Tanenbaum 式分类

另一种常见分类方法更强调使用环境与体系结构，包括：

  • 大型机操作系统
  • 服务器操作系统
  • 多处理机操作系统
  • 个人计算机操作系统
  • 掌上或移动计算机操作系统
  • 嵌入式操作系统
  • 传感器节点操作系统
  • 实时操作系统
  • 智能卡操作系统

这种分类方式强调 ” 系统运行于何种硬件与应用场景 “，而不仅是 ” 系统采用何种服务模式 “。

七、智能卡操作系统

智能卡是一种带有 CPU 芯片的卡式设备，具有极严格的能耗、存储和运行空间限制。某些智能卡只承担单一功能，例如电子支付；某些智能卡则支持下载与运行多个 Java 小程序，此时卡内 ROM 中通常驻留 JVM 解释器，应用以受控方式装入执行。

在支持多应用的场景下，智能卡同样会出现多道程序、调度、资源管理与保护问题，只是所有机制都发生在极其受限的环境中。智能卡操作系统必须在有限 RAM、EEPROM 和处理能力条件下完成命令解释、权限控制、数据处理和安全保护。

智能卡与读写器之间通常采用 ” 命令—响应 ” 对的通信模式。读写器发送命令，智能卡接收后由操作系统解释命令、完成解密与校验，调用相应程序处理数据，再将应答加密后返回读写器。

flowchart LR
    R[读写器] --> C1[发送命令]
    C1 --> OS[智能卡操作系统]
    OS --> V[命令解释 / 校验 / 解密]
    V --> APP[调用卡内程序或 Applet]
    APP --> RESP[生成响应数据]
    RESP --> ENC[加密与封装]
    ENC --> R

智能卡操作系统表明，操作系统并不只属于大型通用机器。只要存在可编程执行环境、资源约束、保护需求和统一控制逻辑，操作系统思想就会以不同尺度出现。

八、典型系统形态与系统边界

不同类型操作系统虽然目标不同，但都围绕同一组核心问题展开：

  • 谁拥有处理器，持续多久；
  • 谁占用内存，如何隔离；
  • 设备如何被请求、排队与复用；
  • 数据如何组织为逻辑对象；
  • 异步事件如何被接收与处理；
  • 不同用户或任务如何共享又互不破坏。

从这个意义上看，批处理、分时、实时、分布式、嵌入式和云平台并不是互相无关的类别，而是同一组基本机制在不同成本条件、硬件环境和服务目标下的不同组合。

九、本章要点

操作系统是位于硬件与应用之间的系统软件，通过抽象、虚拟化、资源管理和保护机制，把物理机器组织为可共享、可扩展、可编程的虚拟机环境。其基本特征是并发、共享、虚拟和随机性。其发展历程经历了批处理、分时、个人计算、网络与分布式、移动计算、云计算以及泛在计算等阶段。多道程序设计、中断、通道和 SPOOLing 是早期形成而至今仍具启发意义的关键技术。不同分类方法虽然侧重点不同，但都在说明操作系统始终处于 ” 资源约束、服务目标与系统抽象 ” 三者的交汇处。

典型的现代系统实例包括通用桌面与服务器操作系统、移动操作系统、嵌入式系统平台、云计算基础平台，以及麒麟、鸿蒙、OpenEuler、统信等国产系统生态。
