xv6 OS Lab Part 1：系统启动、自举过程与首批系统调用

本部分的核心任务不是编写一组新的用户程序，而是在现有 xv6-k210 内核上建立一套能够兼容预编译测试程序的系统调用接口，使系统可以正确启动、装载测试程序、输出可比对结果，并在测试结束后稳定退出。测试覆盖 getcwd、write、getpid、times 与 uname 五个点，因此这一部分同时涉及启动链路、自举机制、评测流程、系统调用号 ABI 对齐，以及若干系统调用的实现或适配。(Arthals’ ink)

initcode

xv6 在 QEMU 中的启动链路由虚拟硬件环境、引导固件、内核初始化和首个用户进程启动四个层面构成。QEMU 提供 CPU、内存和块设备等虚拟硬件；RustSBI 作为引导固件首先运行，完成硬件初始化并装载内核；内核接管控制权后完成进程管理、内存管理与文件系统挂载；随后内核创建第一个用户进程并执行 initcode，由它发起 exec("/init")，最终把控制权交给 /init 程序。这个链路决定了 Part 1 中所有测试都以 /init 为用户态入口，而不是直接从 shell 启动。(Arthals’ ink)

initcode 的意义在于完成系统自举。首个用户进程出现时，完整的用户态运行时环境尚未建立，因而不可能依赖动态链接器、标准库或常规 ELF 装载路径来“再去启动第一个加载器”。因此，initcode 只能是一段极小、纯粹、可直接复制到用户地址空间中执行的机器码；内核只需把它作为字节数组写入首进程内存，再把 PC 交给它，就能由该进程主动发起 exec("/init")，从而把真正的用户态初始化程序装入进程映像。initcode 不是长期运行的用户程序，而是把“没有用户态环境”过渡到“已有 /init 可运行”的最小桥梁。(Arthals’ ink)

在实现上，initcode 先由 xv6-user/initcode.S 编译为机器码，再以 uchar initcode[] 的形式嵌入内核镜像；userinit() 为 PID 1 分配进程结构后，不从磁盘读取任何程序文件，而是直接把这段字节流复制到新进程地址空间中。调度器开始运行该进程后，它执行的第一组用户态指令就是 initcode 本身；当 exec("/init") 发出以后，控制流重新陷入内核，内核从文件系统中装载 /init，用其覆盖原有的 initcode 映像，随后返回用户态执行 /init 的主逻辑。(Arthals’ ink)

Part 1 对 initcode 的处理之所以成为问题，根本原因在于所基于的 k210 框架较早，其原始 initcode.S 面向 32 位 RISC-V，而当前实验环境是 64 位 RISC-V；同时，initcode.S 中又通过 SYS_exec、SYS_exit 等宏依赖 sysnum.h 中的系统调用号，因此只要系统调用编号发生调整，initcode 就必须重新生成，否则首进程会在最早阶段因调用号不匹配而出现 unknown sys call。这说明 initcode 既受 ISA 位宽影响，也受系统调用 ABI 影响。(Arthals’ ink)

网页材料采用了一种区分本地测试与平台评测的双路径设计。本地测试时，使用理想的短自举片段 initcode.S 生成 initcode.h，再通过 exec("/init") 装载 /init；平台评测时，由于提供的预编译 sdcard.img 中没有可供装载的 /init，且编译阶段也无法改写平台镜像，因此将完整 init 程序的机器码直接硬编码到 initcode.h 中，使首进程一开始就运行整段 /init 逻辑。前者更符合自举的设计本义，后者则是受评测环境约束而产生的工程折中。(Arthals’ ink)

这种折中的直接后果是：本地与平台的 Makefile 目标需要分流。dump 过程在一种模式下把短 initcode.S 转成字节数组，在另一种模式下则把 _init 这个完整用户程序转成字节数组；proc.c 中的 initcode[] 不再手动书写机器码，而是通过 #include "include/initcode.h" 引入自动生成的字节序列。这样，initcode 的维护工作从“修改一串裸字节”转为“修改汇编或用户程序并重新生成头文件”，显著降低了维护成本。(Arthals’ ink)

评测流程

Part 1 的评测对象是内核，不是测试程序本身。riscv64/ 目录中的测试文件已经在标准 Linux 环境下预编译完成，它们的职责是调用约定好的系统调用接口，检查返回值与副作用是否满足预期，并通过 printf 打印标准化输出。由于 printf 最终依赖 write 系统调用，测试结果会沿着“用户程序输出 → 内核 write → QEMU 标准输出”的路径暴露给外部评测脚本，评测端再把整个输出文本与标准答案逐字比较。只要输出完全一致，测试即判定为通过。(Arthals’ ink)

因此，虚拟磁盘中实际共存两类用户程序。第一类是 xv6-user/ 目录下由本地构建系统编译得到的原生 xv6 用户程序，例如 ls、cat、init 等，它们为 shell 环境和基础用户态功能服务。第二类是 riscv64/ 目录下直接复制到 fs.img 根目录中的预编译测试二进制，例如 /getcwd、/write、/getpid 等，它们不参与本地重新编译，存在的唯一目的就是对内核接口进行自动化验收。二者在文件系统中的位置、来源和用途均不相同。(Arthals’ ink)

为了匹配这种评测模型，/init 不再承担“启动 shell 并长期驻留”的传统职责，而是被改造为一个测试调度器。它预先保存待测程序名数组 {"getcwd","write","getpid","times","uname"}，在主循环中依次执行 fork、exec 与 wait，保证每轮只运行一个测试程序，并等待其结束后再进入下一轮。这样可以获得稳定、可预测的输出顺序，并把每个测试点的输出与前后流程严格分隔开来。所有测试结束后，/init 调用 shutdown() 触发关机，避免程序悬停造成平台超时。(Arthals’ ink)

系统调用号对齐

Part 1 的第一项基础工作是让内核的系统调用号与测试程序所遵循的 ABI 保持一致。原因在于 xv6-k210 原始实现虽然已经带有 write、read、fork、exec 等基本系统调用，但这些处理函数对应的编号采用的是项目自身的旧约定，而测试程序是按照标准 Linux 的 RISC-V 系统调用号编译的。以 write 为例，原项目编号为 16，而 Linux 约定编号为 64；如果用户程序把 64 放入 a7 发起 ecall，而内核仍把 16 视为 sys_write 的入口，那么分发表就会在 64 号位置找不到正确处理函数，从而报出 unknown sys call。(Arthals’ ink)

系统调用号对齐的本质，是让“用户态约定的编号”和“内核态分发表的索引”重新对应起来。对 Part 1 来说，真正关键的是与启动和测试相关的一组编号：SYS_getcwd = 17、SYS_write = 64、SYS_exec = 221、SYS_exit = 93、SYS_wait = 3、SYS_getpid = 172，以及新增的 SYS_shutdown = 210。这些编号决定了首进程能否顺利执行 exec("/init")，决定了测试程序能否命中既有的 write、getpid 等处理函数，也决定了测试流程结束后能否正确触发关机。(Arthals’ ink)

系统调用号一旦修改，initcode 也必须同步重建。原因在于 initcode.S 内部并不是直接写死某个数字，而是通过 SYS_exec、SYS_exit 这样的宏生成机器码；如果系统调用号表发生变更而 initcode 没有重生成，那么它携带的仍是旧 ABI 下的编号，首进程在最早的自举阶段就会失败。系统调用号调整因此不是单纯编辑头文件，而是会沿着“头文件定义 → 汇编生成 → 机器码嵌入 → 首进程执行”这一整条链路传播。(Arthals’ ink)

write 与 getpid 之所以在 Part 1 中不需要额外编写内核实现，正是因为原项目已经具备相应处理函数，问题只出在调用号不匹配。完成调用号对齐后，测试程序发出的 Linux 风格系统调用号就会被准确分发到现有的 sys_write 与 sys_getpid 等处理函数上，因此这两个测试点本质上属于 ABI 适配问题，而不是新增功能实现问题。(Arthals’ ink)

shutdown

shutdown 的作用是为测试流程提供一个显式、可控的终止机制。网页材料指出，在某些本地路径下即使不添加它也未必立即出错，但在平台评测中，若系统在所有测试完成后不能可靠退出，就可能因持续运行而引发超时。因此，把“测试全部执行完毕”与“虚拟机终止”之间建立起一个明确的系统调用接口，可以把评测结束从偶然行为变成确定行为。(Arthals’ ink)

shutdown 的用户态可用性依赖于 xv6 自身的系统调用封装链。对于 xv6-user/ 下的源代码程序，仅在 user.h 中声明函数还不够，还必须在 usys.pl 中登记入口，才能自动生成对应的用户态汇编存根。该存根的职责是在用户态把 SYS_shutdown 装入 a7，执行 ecall，并在返回后恢复普通函数调用语义。没有这一步，init.c 虽然可以写出 shutdown() 这样的调用表达式，但链接阶段找不到对应符号。(Arthals’ ink)

与之相对，预编译测试程序不需要在 usys.pl 中再次注册，因为它们在生成二进制时已经把各自的系统调用封装为底层机器码，运行时会直接执行 li a7, <syscall_no>; ecall 这样的序列。usys.pl 面向的是 xv6 自己重新编译的用户态源程序，而不是外部已经编译完成的测试二进制。这个差异解释了为什么 shutdown 需要补全用户态包装，而 getcwd、times、uname 这些由外部测试程序触发的接口本身并不依赖 xv6 的 usys.pl 才能被调用。(Arthals’ ink)

在内核侧，shutdown 需要三层映射关系同时成立。第一层是 sysnum.h 中给出 SYS_shutdown = 210；第二层是 kernel/syscall.c 中把编号 210 映射到 sys_shutdown；第三层是 kernel/sysproc.c 中真正定义 sys_shutdown()，其函数体只调用 sbi_shutdown() 即可。系统调用分发表位于 syscall.c，而具体功能实现根据类别放在不同的 sys*.c 文件中。网页材料将 shutdown、getpid、uname、times 归入 sysproc.c，将 write 与 getcwd 归入 sysfile.c，而 syscall.c 仅负责分发。(Arthals’ ink)

shutdown 的底层语义涉及两次特权级跃迁。第一次跃迁发生在用户程序调用 shutdown() 时，ecall 使 CPU 从 U 模式进入 S 模式，由内核的系统调用分发逻辑处理 SYS_shutdown。第二次跃迁发生在 sys_shutdown() 内部调用 sbi_shutdown() 时，这个 SBI 调用再次通过环境调用从 S 模式进入 M 模式，交由 RustSBI 这样的底层固件执行真正的关机操作。因而，shutdown 表面上是一个普通系统调用，实质上把“用户态测试结束”转换成了“通过内核向固件请求关闭虚拟机”的控制链。(Arthals’ ink)

getcwd

getcwd 的标准语义是把当前工作目录的绝对路径复制到调用者提供的缓冲区 buf 中，缓冲区长度由 size 指定；成功时返回指向 buf 的指针，失败时返回空指针。若路径字符串连同结尾空字符的总长度超过 size，则这一调用应被视为失败。也就是说，getcwd 不只是“把路径写出去”，还必须体现“用户提供的接收空间是否足够”这一边界条件。(man7.org)

原仓库中的 sys_getcwd 已经具有“从目录项链回溯父目录并拼接绝对路径”的基本思路，但接口只接收一个用户地址参数，既不读取 size，也不把返回值设计成“成功时返回用户缓冲区地址”。这种实现对内核内部来说可以写出路径，但对遵循标准调用约定的测试程序而言并不兼容，因为测试程序会传入两个参数，并根据返回值是否等于缓冲区地址来判断成功。(Arthals’ ink)

实验实现的核心是从当前目录项 cwd 沿父指针逐级向上回溯，并在一个内核栈上的临时数组 path[FAT32_MAX_PATH] 中自右向左构造路径字符串。若当前目录已是根目录，则结果直接为 /；否则，每访问一级目录，就把该目录名向左拷入缓冲区，再补一个 /，直到抵达根目录为止。由于构造过程是反向进行的，最终得到的有效字符串位于数组尾部，随后再通过 memmove 把这段结果移动到数组起始位置，以形成连续、从开头开始的 C 字符串。该算法的优点是无需预先知道目录深度，也无需在回溯前做两遍遍历。(Arthals’ ink)

sys_getcwd(void) 之所以函数签名看不到普通 C 风格参数，并不是因为它没有参数，而是因为系统调用参数在进入内核时并不通过当前内核函数栈传递。用户进程与内核拥有彼此独立的虚拟地址空间，当用户态执行 ecall 后，CPU 会把用户寄存器状态保存到当前进程的陷阱帧中。内核实现若想读取第一个参数里的用户地址和第二个参数里的缓冲区大小，就必须通过 argaddr(0, &addr) 与 argint(1, &size) 这样的辅助函数，从陷阱帧中安全提取寄存器内容。系统调用返回给用户的字符串也不能直接把内核指针交出去，而必须借助 copyout2() 把内核缓冲区里的字节复制到用户空间指定地址。(Arthals’ ink)

因此，getcwd 的实现同时体现了三个关键点：其一，系统调用兼容性不仅是“功能存在”，还要求参数数量、返回值语义与边界检查都与调用约定一致；其二，内核中构造路径时使用的是内核私有缓冲区，而不是直接写用户内存；其三，用户态与内核态之间的数据交换必须显式经过参数提取与地址空间拷贝，而不能套用普通函数调用的直觉。(Arthals’ ink)

write / getpid

write 与 getpid 在这一部分没有新增实现代码，原因在于 xv6-k210 已经具备相应的内核处理逻辑。Part 1 中这两个测试点的真实问题不是“内核不会写”或“内核不会返回 PID”，而是预编译测试程序发出的 Linux 风格调用号在旧 ABI 下无法命中原有处理函数。只要系统调用号对齐完成，测试程序就会把 write 与 getpid 正确分发到既有实现上，因此这两个点本质上属于接口兼容性修复。(Arthals’ ink)

times

Linux 语义下的 times(struct tms *buf) 会把当前进程以及已结束子进程的 CPU 时间统计写入 struct tms，其中 tms_utime 表示调用进程的用户态时间，tms_stime 表示调用进程的内核态时间，tms_cutime 与 tms_cstime 则分别表示已等待回收的子进程累计用户态与内核态时间；函数返回值本身表示自某个历史基准点以来流逝的时钟节拍数。这个接口的重点既包括结构体字段布局，也包括时间来源的抽象含义。(man7.org)

网页材料中的实验实现采用了一个简化策略：先在 kernel/include/timer.h 中定义 struct tms，再在 sys_times 中获取全局时钟节拍变量 ticks，把同一个 ticks 值填入四个字段，然后通过用户地址拷贝辅助函数把整个结构体复制回用户空间。其实现依赖 tickslock 对共享时钟状态进行保护，因此读取过程中不会与并发更新发生竞争。对给定测试而言，关键在于返回给用户的结构布局正确、字段含义可解释，并且整体值处于合理范围内。(Arthals’ ink)

这个实现反映出实验语境与完整接口语义之间的差异。完整的 times 接口应区分调用进程自身与其子进程的不同 CPU 时间来源，而简化实现把当前全局节拍统一写入四个字段，实际上是用一个“结构正确、数值稳定”的近似结果满足测试需求。网页材料还提到另一种基于 r_time() 读取硬件周期数的写法；该写法也能通过给定测试，但从接口抽象角度看，使用操作系统维护的 ticks 更贴近 times 所描述的节拍统计语义。(Arthals’ ink)

为了减少样板代码，实验实现把“从第 arg_index 个系统调用参数中取出用户地址，并把一段内核内存复制过去”抽象为 get_and_copyout()。这个辅助函数在内部顺序调用 argaddr() 与 copyout2()，从而把“取用户地址”和“跨地址空间写回”的模式固化为一个通用过程。times 与 uname 这类“在内核栈上临时组装结构，再整体拷回用户态”的系统调用都可以复用这一模式。(Arthals’ ink)

uname

uname(struct utsname *buf) 的标准职责是把当前系统的标识信息写入用户提供的结构体中。标准结构至少包含 sysname、nodename、release、version 与 machine 等字段，在 GNU 扩展下还可以包含 domainname。这些字段分别承担操作系统名、节点名、系统发行版本、版本字符串、硬件架构标识及域名等含义。与 times 相似，uname 的重点不在复杂算法，而在于结构布局、字段语义以及用户态可见的数据回填。(man7.org)

实验实现采取的是固定字符串填充策略：在 sys_uname 内部声明一个包含六个 65 字节字符数组的本地结构体，分别写入诸如 xv6、xv6-node、1.0.0 等系统标识，再通过 get_and_copyout() 一次性复制到用户缓冲区。这种实现没有追求真实机器环境的动态探测，而是提供一组稳定、格式正确、字段齐全的系统身份信息，以满足测试程序对 uname 输出格式和结构完整性的要求。(Arthals’ ink)

uname 被放在 sysproc.c 中而不是 sysfile.c 中，也体现了系统调用按功能类别组织的原则。它返回的并非文件内容或文件系统状态，而是当前操作系统实例的全局身份信息，因而更接近进程与系统管理层面的接口。对 Part 1 而言，这种文件组织方式和 shutdown、times 一样，都是“由 syscall.c 分发，在对应功能域的 sys*.c 中实现”的具体体现。(Arthals’ ink)

测试与结果解释

本地测试流程以清理旧构建并重新执行本地构建为主，网页材料给出的命令序列是先执行 make clean，再执行 make local。在这种路径下，构建系统会按本地模式生成 initcode.h、重建内核与文件系统镜像，并启动 QEMU 运行全部五个测试程序。(Arthals’ ink)

测试输出按固定顺序出现：先后进入 test_getcwd、test_write、test_getpid、test_times 与 test_uname，其中 getcwd 会成功打印根目录路径，write 会把测试字符串写到标准输出，getpid 会给出一个有效的进程号，times 会回显 struct tms 中的四个字段值，uname 则输出系统名、节点名、版本与域名等信息。输出序列的稳定性说明 /init 的调度逻辑、系统调用 ABI、用户态输出链路和关机流程已经形成一个闭环。(Arthals’ ink)

从 Part 1 的整体结构看，真正的主线并不是五个孤立的系统调用测试点，而是“自举 → 装载 /init → 顺序执行预编译测试 → 通过 Linux 风格系统调用号命中内核处理函数 → 将结果写回用户态并输出 → 通过 shutdown 结束虚拟机”这一条完整的系统运行通路。getcwd、times 与 uname 负责体现参数提取、内核态构造结果与用户态回填；write 与 getpid 负责体现 ABI 对齐后对既有实现的复用；shutdown 则把评测结束动作从用户程序一直传递到底层 SBI 固件。Part 1 因而既是首批系统调用实验，也是一次关于“用户态约定如何穿过内核并落到硬件抽象层”的完整演示。(Arthals’ ink)
