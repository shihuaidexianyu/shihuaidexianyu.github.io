操作系统概述

操作系统概述

Apr 26, 202664 min read

核心向导

这节课是操作系统的第一课（Overview），主要回答了三个终极问题：操作系统究竟是什么？它有什么独特的特征？以及它是怎么一步步演化成今天这个样子的？

学习这部分内容，不要死记硬背概念，而是要带入“管理者”的视角去理解——如果你手里只有一堆冰冷的 CPU、内存和磁盘，你要怎么设计一套规章制度，让不同的程序不仅能“同时”运行，还不互相打架，并且让最终用户觉得既快又好用？

这节课的关键词可以先记成一条主线：

操作系统通过抽象和虚拟化，把复杂硬件变成易用接口；通过并发、共享、调度和保护，让多个程序安全高效地共同使用同一套机器资源。

课前导学问题：这一讲到底要会回答什么？

学完本讲之后，至少应该能用自己的话回答下面 6 个问题：

  1. UNIX 操作系统中，文件子系统与块设备之间的 Buffer Cache 有什么作用？它的工作原理是什么？
  2. UNIX 这个名字是怎么来的？
  3. 批处理操作系统为什么分成单道批处理和多道批处理？二者区别是什么？
  4. SPOOLing 技术的工作原理是什么？它今天还在不在？
  5. 传统分时系统在今天还有什么意义？
  6. 操作系统的主要作用是什么？和普通应用软件相比，操作系统有什么典型特征？

这 6 个问题不是孤立的。它们其实对应了这节课的主线：

  • Buffer Cache 体现的是 缓存、抽象、I/O 管理；
  • 单道/多道批处理体现的是 并发与资源利用率；
  • SPOOLing 体现的是 设备虚拟化与慢速 I/O 解耦；
  • 分时系统体现的是 CPU 虚拟化与交互响应；
  • 操作系统作用和特征则是整个课程的总纲。

所以这一讲不是单纯讲历史，而是在告诉我们：操作系统为什么会长成今天这样。

一、操作系统是什么？

传统书本上会说，操作系统（Operating System, OS）是计算机系统中的一个系统软件，是一些程序模块的集合。它的任务是有效、合理地组织和管理软硬件资源，合理组织计算机的工作流程，控制程序执行，并向用户提供各种服务功能。

但在我看来，操作系统本质上就是一个“超级管家”兼“翻译官”。

它对下接管所有复杂的物理硬件，对上给用户和应用程序提供简单易用的接口。如果没有操作系统，我们写一个简单的“在屏幕上打印 Hello World”的程序，可能都需要去查阅显卡的物理寄存器地址，手动组织数据，甚至手动处理设备时序。那就不是写程序了，那是在和硬件肉搏。

因此，操作系统至少有三重身份：

  1. 资源的管理者

    它管理 CPU、内存、磁盘、I/O 设备、网络接口、文件等资源，并尽量让这些资源被有效、合理地使用。

  2. 服务的提供者

    它向用户和应用程序提供系统调用、API、文件接口、进程接口、设备接口等服务，让应用程序不必直接面对复杂硬件。

  3. 机器能力的扩展者

    它把一台裸机包装成更强大、更好用的“虚拟机器”。比如，每个进程都好像有自己的 CPU、自己的内存空间、自己的文件视图。

简单说：

没有操作系统，计算机只是一堆硬件；有了操作系统，计算机才变成一台可被普通程序使用的机器。

二、从三个不同视角理解操作系统

操作系统太复杂了，所以不能只用一个角度看。课件给了三个很重要的视角：资源管理视角、进程视角、虚机器视角。

1. 资源管理的观点：操作系统是资源的管理者

从自底向上的角度看，计算机系统里有两类资源：

  • 硬件资源：CPU、内存、磁盘、I/O 设备、时钟、网络接口等；
  • 软件资源：文件、数据、程序、各种系统信息等。

这些资源都是有限的，但程序和用户都想用它们。于是操作系统要做的事情就很像一个宿舍楼管理员：

  • 哪些资源空闲？
  • 哪些资源正在被谁使用？
  • 一个资源能不能同时被多个进程使用？
  • 如果两个进程都想用打印机，谁先来？
  • 如果一个进程乱写内存，会不会破坏别人的数据？
  • 如果资源不够了，应该让谁等一等？

所以操作系统管理资源，一般要做几类事情：

  1. 跟踪记录资源状态

    比如哪些内存页空闲、哪些磁盘块被占用、哪个进程正在使用哪个设备。

  2. 分配和回收资源

    比如进程创建时分配内存，进程结束时回收资源。

  3. 协调冲突

    比如多个进程同时请求同一个设备时，需要排队、调度或互斥。

  4. 提供保护

    比如一个进程不能随便访问另一个进程的地址空间。

资源复用主要有两种方式：

  • 时间复用：大家轮流用同一个资源，比如多个进程轮流使用 CPU。
  • 空间复用：把资源切成不同部分给不同对象使用，比如内存划分给多个进程。

直觉理解

时间复用像“轮流上台发言”，空间复用像“把教室分成几块区域给不同小组”。

课件里还提到资源分配策略可以有静态分配和动态分配。静态分配比较死板，动态分配更灵活，所以现代操作系统中大量资源管理都倾向于动态分配。

2. 进程的观点：操作系统是动态运行的协调者

从系统运行起来的角度看，操作系统不是静态地躺在磁盘上的一个软件包，而是由一些可以同时、独立运行的进程，以及一个负责协调它们的核心组成。

这里要先分清楚“程序”和“进程”：

  • 程序：静态的代码和数据，就像一张菜谱。
  • 进程：程序的一次执行过程，就像厨师正在照着菜谱做菜。

菜谱本身不会动，但做菜过程是动态的。它有开始、有推进、有暂停、有结束。进程也是这样：它会创建、运行、阻塞、唤醒、终止。

因此，从进程视角看，操作系统最重要的工作之一就是：

协调多个动态执行过程，让它们看起来能同时运行，同时又不会互相破坏。

这也直接引出了后面操作系统课程里非常核心的一堆主题：进程调度、进程同步、互斥、死锁、进程通信等等。

3. 虚机器观点：操作系统是逐层扩展出来的机器

从内部结构看，操作系统可以理解成一层一层堆起来的虚拟机器。

最底层是物理硬件，它只会执行非常原始的机器指令，提供非常粗糙的设备能力。操作系统在这个基础上不断增加抽象：

  • 把物理 CPU 抽象成多个“虚处理器”；
  • 把物理内存抽象成每个进程独立的地址空间；
  • 把磁盘块抽象成目录和文件；
  • 把原始设备抽象成统一的设备接口；
  • 把复杂的硬件细节抽象成系统调用和 API。

最后，用户和应用程序看到的就不是一堆寄存器、总线、磁盘扇区，而是进程、文件、窗口、网络连接这些更容易理解的对象。

关键理解

操作系统最厉害的地方，不是“直接使用硬件”，而是“把硬件变成另一种更好用的东西”。

这就是所谓的虚机器观点。

三、操作系统的设计与实现目标

操作系统不是只要“能跑”就行。它在设计时要同时考虑很多目标，而且这些目标之间经常会互相拉扯。

课件里列了几个很重要的目标：

1. 抽象

操作系统要把复杂硬件包装成简单模型。比如用户操作的是“文件”，而不是“磁盘第几个柱面、第几个磁道、第几个扇区”。

2. 模块化

操作系统太复杂了，必须拆成相对独立的模块，比如进程管理、内存管理、文件系统、设备管理、网络管理。否则整个系统会变成一团浆糊。

3. 使用高级语言

现代操作系统大量使用 C 等高级语言实现，而不是完全用汇编写。这样更容易维护、移植和扩展。当然，少数最底层、最贴近硬件的部分仍然可能需要汇编。

4. 性能

操作系统提供抽象和保护，但不能太慢。它要尽量减少额外的时间开销和空间开销。

比如系统调用很方便，但如果每次系统调用都特别慢，应用程序性能就会崩掉。

5. 保护与隔离

不同应用之间要隔离，应用和操作系统之间也要隔离。

一个普通程序不能随便修改内核，也不能随便读写别的进程内存。否则只要一个程序出错，全系统就可能崩溃。

6. 可靠性

操作系统是整个计算机系统的基础，自己不能动不动就崩。它必须尽量在异常、错误、设备故障、资源不足等情况下保持系统可用。

7. 安全性

保护更多强调“别误伤”，安全性更多强调“防恶意”。在网络化时代，操作系统还要防止恶意应用、病毒、攻击者入侵。

8. 节能与移动性

在移动设备、嵌入式设备和物联网设备中，操作系统还要考虑能耗、体积、硬件能力限制，以及能否运行在越来越小的设备上。

所以操作系统设计本质上是在做权衡：

既要抽象得好用，又不能慢；既要共享资源，又不能互相破坏；既要功能强大，又要可靠、安全、可维护。

四、操作系统的四大基本特征

这四个特征是操作系统的灵魂，特别是“并发”和“共享”，它们几乎是操作系统存在的基础。

直觉理解

想象一个厨房。并发是厨师一会儿切菜、一会儿看锅，两件事交替做；共享是几个厨师合用一个烤箱；虚拟是每个厨师都觉得自己有一个专属厨房；随机性则是你永远不知道下一个进来的点单是炒饭、炖汤，还是突然有人说锅糊了。

1. 并发（Concurrency）

并发指操作系统具有处理多个同时性活动的能力。

宏观上看，多个程序好像在同时执行；但在单核 CPU 下，微观上任意一个时刻其实只有一个程序在执行。所谓“同时”，是因为操作系统让它们在 CPU 上快速轮流切换。

因此：

  • 并发：逻辑上同时推进，可能是交替执行。
  • 并行：物理上真的同时执行，需要多个处理器或多个核心。

举例

你一边煮面，一边洗菜，一边回消息，这叫并发。
三个人分别煮面、洗菜、回消息，这才更像并行。

并发会带来很多问题：

  • 什么时候切换？
  • 切换时保存哪些现场？
  • 多个进程访问同一数据时怎么办？
  • 一个进程出错会不会影响另一个？
  • 多个活动之间怎么同步？

所以后面的进程调度、同步互斥、死锁，本质上都和并发有关。

2. 共享（Sharing）

共享指操作系统与多个用户程序共同使用计算机系统中的有限资源。

资源有限，但大家都想用。于是操作系统要合理分配和管理。

共享有两种典型形式：

互斥共享

某个资源在一段时间内只能被一个进程使用。

典型例子是打印机。一个进程打印文档时，另一个进程不能直接插进来，否则两份文档会混在一起。

同时共享

某些资源可以被多个进程“同时”访问。

比如：

  • 只读的可重入代码；
  • 某些共享库；
  • 磁盘上的文件；
  • 多个进程读取同一份数据。

但注意，“同时共享”并不意味着完全不用管理。只要涉及写操作、状态修改、权限控制，就仍然需要操作系统介入。

共享带来的主要问题是：

  • 资源分配难以达到最优；
  • 资源使用时需要保护；
  • 多个进程之间可能发生冲突。

3. 虚拟（Virtualization）

虚拟指把一个物理实体映射成多个逻辑实体。

这句话听起来抽象，但其实很直观：

  • 只有一个 CPU，但每个进程都觉得自己有一个 CPU；
  • 只有一套物理内存，但每个进程都觉得自己有独立地址空间；
  • 只有一个显示器，但用户可以看到多个窗口；
  • 只有一台打印机，但多个程序都能提交打印请求。

虚拟化的本质是：

用时间复用或空间复用，把有限的物理资源包装成多个逻辑资源。

常见例子：

CPU 虚拟化

通过进程调度和时间片切换，每个进程好像都有自己的“虚处理器”。

内存虚拟化

每个进程看到的是自己的虚拟地址空间，里面有代码段、数据段、堆、栈。它不需要知道真实物理内存怎么分配。

设备虚拟化

比如打印队列、多窗口、虚拟终端，本质上都是让用户感觉自己在独占设备。

虚拟化是操作系统非常核心的思想。可以说，操作系统很多功能都在做一件事：

把不好用的真实资源，变成好用的虚拟资源。

4. 随机性 / 不确定性

随机性指操作系统必须随时响应以不可预测次序发生的事件。

在并发环境下，多个进程的推进速度是不可预知的。一个进程可能正在运行，突然发生 I/O 请求；另一个进程可能等待磁盘，磁盘又突然完成；外部设备可能随时发出中断；用户也可能随时输入命令。

所以操作系统面对的世界不是整整齐齐的，而是充满了“突然”：

  • 突然来了一个中断；
  • 突然有 I/O 完成；
  • 突然某个进程阻塞；
  • 突然某个资源被释放；
  • 突然有多个进程竞争同一个资源。

这也是为什么多线程、多进程程序里的 bug 经常很难复现。因为系统状态和事件顺序可能每次都不一样。

关键理解

操作系统不是在执行一条固定剧本，而是在不断处理随机到来的事件。

五、一个重要例子：Buffer Cache 是干什么的？

课件开头问到 UNIX 操作系统中文件子系统与块设备之间的 Buffer Cache，这个问题虽然在本讲里只是提问，但很重要，因为它能很好地体现操作系统的“缓存、抽象、资源管理”思想。

1. 为什么需要 Buffer Cache？

磁盘这类块设备很慢，而内存很快。

如果每次读文件、写文件都直接访问磁盘，那么系统性能会非常差。尤其是很多文件访问会反复读写相同的磁盘块，如果每次都真的去磁盘上找，就很浪费。

于是 UNIX 在文件子系统和块设备之间加了一层 Buffer Cache。

它可以理解成：

一层专门缓存磁盘块的内存缓冲区。

文件系统不必每次都直接找块设备，而是先问 Buffer Cache：“我要的这个块你这里有没有？”

2. Buffer Cache 的作用

Buffer Cache 主要有几个作用：

减少真实磁盘 I/O

如果某个磁盘块已经在缓存里，就可以直接从内存读取，不用再访问磁盘。

缓和 CPU / 内存 与磁盘之间的速度差异

CPU 和内存很快，磁盘慢得多。Buffer Cache 可以把频繁访问的数据留在内存里，减少等待。

统一文件系统对块设备的访问

文件系统看到的是“块缓存”这一层，而不用每次都直接处理具体设备细节。

支持延迟写回

写文件时，系统可以先把修改写入缓存，并标记为“脏块”，之后再统一写回磁盘。这样可以合并多次写操作，提高效率。

3. Buffer Cache 的基本工作原理

可以分成读和写两种情况。

读操作

当文件系统要读取某个磁盘块时：

  1. 先在 Buffer Cache 里查找这个块；
  2. 如果找到了，叫做 缓存命中，直接从内存返回；
  3. 如果没找到，叫做 缓存未命中，系统再从块设备读取该磁盘块；
  4. 读入后，把这个块放进 Buffer Cache；
  5. 再把数据交给文件系统或用户进程。

所以读操作的核心逻辑是：

写操作

当文件系统要写某个磁盘块时：

  1. 先把数据写到 Buffer Cache 中；
  2. 把对应缓存块标记为 dirty，也就是“脏块”；
  3. 系统不一定马上写回磁盘；
  4. 等到合适时机，比如缓存压力较大、定期同步、文件关闭、系统调用 sync 等，再把脏块写回磁盘。

这种方式叫 延迟写 / 写回。

它的好处是效率高，但也带来一个问题：如果系统突然断电，还没写回磁盘的数据可能丢失。所以文件系统后来还发展出了日志、同步写、写屏障等机制来增强可靠性。

4. Buffer Cache 体现了什么操作系统思想？

Buffer Cache 不只是一个小优化，它体现了操作系统的几个核心思想：

  • 缓存思想：把慢设备的数据放到快设备里；
  • 抽象思想：让文件系统通过统一接口访问块数据；
  • 资源管理思想：缓存空间有限，要决定哪些块保留、哪些块淘汰；
  • 性能与可靠性的权衡：延迟写提高性能，但也带来一致性风险。

所以这题如果考试问，不要只说“提高速度”。更好的答法是：

Buffer Cache 是文件子系统与块设备之间的内存缓存层，用来缓存磁盘块，减少实际块设备访问次数，提高文件 I/O 性能，并通过统一的块缓存机制屏蔽底层设备访问细节。读时先查缓存，命中则直接返回，未命中则从磁盘读入缓存；写时通常先写缓存并标记为脏块，再由系统在合适时机写回磁盘。

六、操作系统的发展历程与启示

技术的发展总是伴随着“硬件与人工成本比率”的变化。 早期是硬件昂贵、人工便宜，所以目标是：让昂贵的机器尽量别闲着。 后来硬件逐渐便宜、人工变贵，于是目标变成：让人的时间更值钱，让人和机器交互更方便。

再后来，计算机进入网络、移动、云计算、物联网时代，操作系统要管理的对象越来越复杂，不再只是一台单机，而可能是一批机器、一个数据中心，甚至是大量联网设备。

一个很有意思的现象是：

技术变化会让某些思想过时，但另一种技术变化又可能让它们复活。

比如早期集中式计算服务在个人计算机时代看起来没那么重要，但到了云计算时代，它又以“数据中心 + 云服务”的形式回来了。

1. 第 1 阶段：大型机与批处理时代

时间大致是 1948—1970。

这个阶段的特点是：

  • 硬件极其昂贵；
  • 人工相对便宜；
  • 计算机资源稀缺；
  • 用户和计算机之间缺乏直接交互。

最开始，一个用户独占控制台。后来为了提高机器利用率，出现了批处理系统。

批处理系统的基本思想是：

用户把作业交给系统操作员，操作员把多个作业组成一批，系统自动连续处理。

这个阶段产生了很多重要技术：

  • 中断；
  • 通道；
  • 多道程序设计；
  • SPOOLing；
  • 存储保护。

它们的共同目标都是：提高硬件利用率，尤其是减少 CPU 等待慢速 I/O 的时间。

2. 第 2 阶段：分时系统时代

时间大致是 1970—1985。 这个阶段硬件逐渐便宜，但人的时间越来越贵。 于是问题变成：

怎么让多个用户能同时、交互式地使用同一台计算机？

分时系统的核心思想是：把 CPU 时间切成很多很短的时间片，轮流给多个用户使用。因为切换很快，用户感觉自己像是在独占机器。

第一个分时系统 CTSS 就是这个方向的代表。课件里提到 CTSS 由 MIT 的 Fernando Corbato 等人在改装的 IBM 7090/94 上开发成功，支持 32 个交互式用户。

分时系统追求的不是单纯吞吐量，而是：

响应时间。

也就是从用户输入命令，到系统给出回答，中间经历的时间。

3. 第 3 阶段：个人计算时代

时间大致从 1981 年开始。

这时硬件已经非常便宜，人工非常昂贵，计算机开始进入个人使用场景。

早期 PC 资源有限，一次常常只能运行一个程序，所以操作系统又变得比较简单，有点像“例程库”。

后来 PC 资源越来越丰富，个人计算机操作系统又重新变复杂：

  • 支持多任务；
  • 支持存储保护；
  • 支持多道程序设计；
  • 支持图形界面；
  • 支持网络；
  • 支持丰富应用软件。

所以这里很能体现课件说的“思想回归”：早期大型机时代出现过的一些机制，在 PC 变强之后又重新出现。

4. 第 4 阶段：网络与分布式时代

网络让不同机器可以很容易地相互共享资源，比如打印机、文件服务器、Web 服务器。

于是操作系统开始面对新问题：

  • 机器之间如何通信？
  • 资源如何共享？
  • 网络环境下如何保证安全？
  • 多台机器如何协同？

网络操作系统强调通信和资源共享。

分布式操作系统更进一步，强调把多台机器组织成一个统一系统，让用户尽量感觉不到资源是分布在不同机器上的。

5. 第 5 阶段：移动计算时代

大致从 1995 年之后，各种移动终端出现：

  • 笔记本；
  • 平板；
  • 手机；
  • 机顶盒；
  • 可穿戴设备。

这些设备小型、移动、便宜，但能力有限，所以操作系统要特别关注：

  • 能耗；
  • 体积；
  • 响应速度；
  • 交互体验；
  • 网络连接；
  • 安全与隐私。

这也是为什么移动操作系统和传统桌面操作系统虽然共享很多基本思想，但设计重点并不完全一样。

6. 第 6 阶段：云计算时代

大致从 2006 年之后，云计算开始成为重要形态。

云计算的目标是提供：

  • 可无限扩展的资源；
  • 可随时获取的资源；
  • 按需使用的资源；
  • 按使用付费的资源。

云计算操作系统不只是管理一台机器，而是管理整个数据中心里的服务器、存储、网络、虚拟机、容器、分布式文件系统、任务调度、安全策略等。

它的作用可以理解成：

把一个数据中心的海量硬件资源，逻辑上整合成一台“超级服务器”。

这样云应用软件就可以通过统一接口申请资源，而不需要关心底层到底是哪台物理机器在运行。

7. 第 7 阶段：泛在计算 / 普适计算 / 物联网

这一阶段强调“人—机—物”的融合。

未来大量设备都会联网，并为大量用户提供个性化服务。这里的操作系统可能不再只是传统意义上的 Windows、Linux、macOS，而是各种嵌入式设备、传感器节点、智能终端背后的基础软件。

这说明操作系统的边界一直在扩展：

从管理一台机器，到管理多台机器，再到管理一个环境里的大量智能设备。

七、失败与成功：OS/360、Multics 与 UNIX

操作系统发展史里有几个很有代表性的案例。

1. OS/360：大型系统复杂性的教训

IBM 的 OS/360 是一个野心很大的系统。课件中提到，它发布时带着已知的 1000 个错误。

这个案例说明：操作系统这种大型软件极其复杂，不是堆人、堆功能就一定能成功。

Frederick P. Brooks 后来写的《人月神话》，很大程度上就是从这类大型软件工程经验中总结出来的。

它给我们的启示是：

操作系统不仅是技术问题，也是软件工程问题。

2. Multics：宏大设计与复杂性的代表

Multics 全称是 MULTiplexed Information and Computing Service。

它从 1963 年开始，到 1969 年才发布，是一个非常宏大的分时系统项目。很多思想很先进，比如动态链接、集中式计算服务、安全机制等，但系统过于复杂，开发周期很长。

虽然 Multics 本身没有像 UNIX 那样广泛流行，但它留下了很多影响深远的思想。

有趣的是，课件里也强调：一些思想可能在当时过早、过重，但在未来会以新形式回归。比如 Multics 式的集中计算服务，在今天可以从云计算中看到影子。

3. UNIX：小而美的成功案例

UNIX 是另一个方向的代表。

课件里讲到，一群计算机迷在贝尔实验室开发出了 UNIX。它的一个趣味起点是：Ken Thompson 想在一台无人使用的 DEC PDP-7 小型计算机上玩星际探险游戏，于是开始做出这个系统。

UNIX 的成功不只是因为技术强，也因为它体现了非常清晰的设计哲学：

  • 简洁；
  • 模块化；
  • 工具组合；
  • 用 C 语言实现；
  • 可移植；
  • 小而优雅。

Ken Thompson 和 Dennis Ritchie 后来获得了 1983 年图灵奖，也获得过美国国家技术奖章。

4. UNIX 这个名字的来历

UNIX 的名字通常被认为和 Multics 有关。

Multics 是一个庞大的分时系统，名字表示 Multiplexed Information and Computing Service。

后来贝尔实验室开发出的系统更小、更简洁。早期名字常被说成 Unics，带有对 Multics 的戏仿意味，后来逐渐写成 Unix / UNIX。

一种常见解释是：

Multics 强调 Multiplexed，而 Unics/Unix 更像是 “Uniplexed Information and Computing Service”。

当然，这个名字来历本身带有一点历史趣味和猜测色彩，不必当成严格技术定义来背。重要的是理解它背后的对比：

Multics 代表宏大复杂的系统愿景，UNIX 代表简洁、实用、小而美的系统风格。

八、批处理操作系统：从单道到多道

批处理操作系统是早期操作系统的重要形态。

它的基本工作方式是：

  1. 用户将作业交给系统操作员；
  2. 操作员把许多用户作业组成一批；
  3. 系统启动后自动依次执行；
  4. 作业完成后，把结果交还给用户。

一个批处理作业通常包括：

  • 用户程序；
  • 数据；
  • 作业说明书，也就是作业控制语言。

课件中举了 FMS JOB 的例子，比如 $JOB、$FORTRAN、$LOAD、$RUN、$END 这类卡片命令。

这说明早期用户并不是坐在电脑前交互，而是把一摞卡片交上去，让系统按顺序处理。

1. 单道批处理

单道批处理，也叫 simple batch processing 或 uniprogramming。

它的特点是：

内存中一次主要只有一个作业运行。

如果这个作业需要 I/O，比如读卡、读磁带、打印输出，那么 CPU 往往只能等待。

这就带来一个严重问题：

CPU 很快，但外设很慢；只要作业等 I/O，CPU 就闲着。

所以单道批处理虽然比人工手动操作进步很多，但资源利用率仍然不高。

2. 多道批处理

多道批处理，也叫 multiprogramming system。

它的核心思想是：

内存中同时放入多个作业。当一个作业等待 I/O 时，CPU 可以切换去执行另一个作业。

这样就可以把不同作业的计算和 I/O 尽量重叠起来。

比如：

  • 作业 A 正在等磁盘；
  • 作业 B 可以用 CPU 计算；
  • 作业 C 可能在等打印机；
  • 等 A 的磁盘 I/O 完成后，再让 A 继续执行。

这就明显提高了资源利用率。

课件里给了一个对比表：在一组计算密集型和 I/O 密集型作业混合的情况下，单道程序设计的处理器利用率只有 22%，总耗时 30 分钟；多道程序设计后，处理器利用率提高到 43%，总耗时缩短到 15 分钟，吞吐量也从每小时 6 个作业提升到每小时 12 个作业。

这组数据想说明的不是具体数字本身，而是这个道理：

多道程序设计通过让计算和 I/O 重叠，大幅提高了系统整体吞吐量和资源利用率。

3. 单道与多道的核心区别

可以整理成一句话：

单道批处理是“一个作业等 I/O，CPU 跟着等”；多道批处理是“一个作业等 I/O，CPU 去跑别的作业”。

更完整地说：

对比点单道批处理多道批处理
内存中的作业数通常一个多个
CPU 等待 I/O 时CPU 容易空闲CPU 可切换到其他作业
资源利用率较低较高
系统复杂度较低较高
需要的机制简单作业控制调度、保护、中断、内存管理
主要目标自动连续处理作业提高吞吐量和资源利用率

注意，多道程序设计并不是白来的。它提高效率的同时，也引入了新的复杂性：

  • 多个作业如何调度？
  • 内存如何分配？
  • 一个作业不能破坏另一个作业，如何保护？
  • I/O 完成后如何通知系统？
  • 资源竞争如何处理？

所以中断、通道、存储保护、调度算法等技术就变得越来越重要。

九、从脱机技术到 SPOOLing 技术

早期批处理系统有一个非常典型的矛盾：

CPU 很快，输入输出设备很慢。

如果慢速输入输出直接由主机完成，那么输入输出时 CPU 就会处于等待状态，昂贵的 CPU 被迫陪着慢设备一起“发呆”。

1. 脱机技术

最早的解决方法之一是使用“卫星机”。

卫星机负责面向用户的输入输出，比如读纸带、读卡片、打印结果；中间结果先暂存在磁带或磁盘上；主机则专心处理计算任务。

这种做法的好处是：主机不必直接被慢速输入输出拖住。

但它还不够优雅，因为它仍然依赖额外机器和比较笨重的流程。

2. SPOOLing 技术：假脱机技术

SPOOLing 全称是：

Simultaneous Peripheral Operation On-Line

意思是“同时的外围设备联机操作”，也叫假脱机技术。

它的核心思想是：

利用磁盘作为缓冲，把输入、计算、输出分别组织成独立的任务流，使 I/O 和计算尽可能并行。

这里最关键的是磁盘上的两个区域：

  • 输入井：暂存输入设备送来的数据和作业；

  • 输出井：暂存程序运行后的输出结果。

3. SPOOLing 的工作原理

可以这样理解：

  1. 输入设备很慢，所以它不直接把数据交给 CPU，而是由输入进程慢慢把数据送入磁盘上的输入井；

  2. 操作系统根据某种调度策略，从输入井中选择作业调入内存；

  3. 作业运行时，CPU 主要和高速磁盘打交道，而不是直接等慢速输入设备；

  4. 作业运行结束后，输出结果先写入磁盘上的输出井；

  5. 输出进程再按照输出设备自己的速度，把输出井中的结果送到打印机等设备。

用流程表示就是：

直觉理解

SPOOLing 就像外卖平台的“订单池”。
用户下单不需要直接堵在厨师面前，订单先进入系统队列；厨师按节奏处理；做完后再交给骑手配送。这样点单、做饭、配送就被解耦了。

4. SPOOLing 的主要作用

课件里总结了三个重点。

提高 I/O 速度

当然，SPOOLing 不能让打印机物理上变快，但它可以让 CPU 不再直接等待慢速外设。

对 CPU 来说，很多 I/O 操作变成了对磁盘输入井、输出井的读写。磁盘速度比传统卡片机、打印机快得多，所以整体效率提高。

将独占设备改造为共享设备

比如打印机本来是独占设备，一次只能服务一个进程。

但有了 SPOOLing，系统并不直接把打印机分配给某个进程，而是在输出井中给每个进程建立输出文件和 I/O 请求表。

多个进程都可以“提交打印任务”，实际打印顺序由打印进程统一安排。

实现虚拟设备功能

从每个进程的角度看，它好像已经拥有了打印机，因为它的输出已经被系统接收了。

但实际上，打印机只有一台，系统只是用磁盘队列制造了一种“每个进程都能使用设备”的假象。

这就是虚拟设备。

5. SPOOLing 在现代操作系统中还应用吗？

当然还在。

最典型的例子就是打印队列。

当多个程序同时请求打印时，系统并不是让它们真的同时占用打印机，而是：

  1. 把待打印文档暂存到磁盘上的打印队列；

  2. 由打印守护进程 daemon 统一管理队列；

  3. 按顺序把打印任务送给打印机。

所以现代打印系统就是 SPOOLing 思想的直接延续。

更广义地看，很多“队列 + 缓冲 + 异步处理”的系统都继承了类似思想，比如：

  • 日志缓冲；

  • 消息队列；

  • 后台任务队列；

  • 网络发送缓冲；

  • 操作系统中的 I/O 请求队列。

当然，严格来说它们不一定都叫 SPOOLing，但思想是一脉相承的：

用高速存储和队列，把慢设备、快 CPU、多个任务之间的速度差和竞争关系解耦。

十、分时操作系统：让多用户“同时”交互

分时操作系统的核心是 时间片。

操作系统把 CPU 时间划分成很短的小片段，每个终端用户轮流获得一个时间片。因为时间片很短，切换很快，所以每个用户都感觉系统在及时响应自己。

1. 分时系统的目标

批处理系统追求的是吞吐量，也就是单位时间内完成多少作业。

分时系统追求的是响应时间。

响应时间指的是：

从终端发出命令，到系统给予回答所经历的时间。

对用户来说，哪怕系统整体吞吐量很高，如果敲一条命令半天没有反应，体验仍然很差。

所以分时系统本质上是把操作系统的目标从“机器效率优先”推进到了“人的交互体验优先”。

2. CTSS 分时系统

CTSS 是早期重要的分时系统。

它的思想是：

  • 每个用户有一个联机终端；

  • 用户调试程序时，往往只发出简短命令；

  • 很多时候用户在思考、输入，CPU 实际上是空闲的；

  • 因此 CPU 可以在多个用户之间快速切换；

  • 同时还可以在后台运行大作业。

这其实非常聪明：它利用了人的操作速度远慢于 CPU 的事实。

直觉理解

人敲一条命令可能要几秒钟，但 CPU 在几秒钟里能做巨量计算。
所以 CPU 完全可以在你思考下一条命令时，去服务别人。

3. 传统分时系统在今天有什么意义？

传统分时系统当然不再是今天操作系统的全部形态，但它的思想仍然非常重要。

现代系统里很多机制都继承了分时思想：

桌面多任务

你可以一边开浏览器，一边写代码，一边播放音乐，一边下载文件。这背后就是 CPU 在多个进程之间快速切换。

多用户服务器

Linux 服务器可以同时有很多用户通过 SSH 登录，每个人都觉得自己在使用系统。

云计算资源共享

云服务器、容器、虚拟机的背后，本质上也是让多个用户、多个任务共享同一批物理资源。

前台优先、后台运行

现代通用操作系统通常把分时系统和批处理系统结合起来：

  • 前台交互任务优先；
  • 后台批处理任务在后；
  • 用户体验优先保证响应时间；
  • 系统空闲时再处理不紧急任务。

所以传统分时系统的意义不只是“历史上曾经有过”，而是：

它奠定了现代多任务、多用户、交互式操作系统的基本思想。

十一、实时操作系统：关键参数是时间

实时操作系统不是简单地“运行速度快”，而是强调：

必须在规定时间内对外部事件作出响应。

它的关键参数是时间。

1. 实时系统的目标

实时操作系统追求两个东西：

  1. 对外部请求在严格时间范围内作出反应；

  2. 高可靠性。

比如工业控制、军事控制、电信交换、银行系统、飞机订票、股市行情等场景，都可能需要实时响应。

2. 硬实时系统

硬实时系统要求某个动作绝对必须在规定时间内完成。

如果错过 deadline，后果可能很严重。

例子：

  • 汽车 ABS 防抱死系统；

  • 飞机控制系统；

  • 导弹控制系统；

  • 工业机械臂控制；

  • 医疗设备中的关键控制模块。

这里的重点不是“平均很快”，而是“最坏情况下也必须准时”。

3. 软实时系统

软实时系统允许偶尔违反时间限制。

错过 deadline 会让体验变差，但不至于造成灾难。

例子：

  • 视频播放偶尔卡顿；

  • 网络语音偶尔延迟；

  • 在线游戏偶尔掉帧；

  • 某些行情显示稍有延迟。

所以实时系统的重点不是“快”，而是“对时间有约束”。

考试时别写错

实时系统不是“最快的系统”，而是“能在规定时间内完成任务的系统”。

十二、其他类型的操作系统

除了批处理、分时、实时，课件还讲了几类常见操作系统。

1. 个人计算机操作系统

个人计算机操作系统在某一时间内主要为单用户服务。

它追求的是：

  • 界面友好；

  • 使用方便；

  • 丰富的应用软件；

  • 良好的交互体验。

比如 Windows、macOS、桌面 Linux 等。

2. 网络操作系统

网络操作系统基于计算机网络，按网络体系结构和协议标准开发。

它的功能包括：

  • 网络管理；

  • 通信；

  • 安全；

  • 资源共享；

  • 各种网络应用。

它追求的是：

相互通信，资源共享。

比如文件服务器、打印服务器、Web 服务器等场景，都离不开网络操作系统的支持。

3. 分布式操作系统

分布式系统的特点是处理和控制分散在多台机器上。

分布式操作系统更进一步，希望把多台计算机组织成一个统一系统，让用户尽量感觉不到背后有多台机器。

它有几个重要特征：

统一性

多个计算机相互协作，共同完成任务。

资源进一步共享

不仅共享单机内部资源，还共享多台机器上的资源。

透明性

用户不需要知道资源具体在哪台机器上。

比如你访问一个文件，可能并不知道它实际存放在哪台服务器。

自治性

分布式系统中的多个主机地位相对平等，没有绝对主从关系。

处理能力、速度、可靠性增强

多台机器协同后，整体处理能力增强；部分节点出问题时，系统也可能继续工作。

4. 嵌入式操作系统

嵌入式系统是各种设备、装置或系统中的一部分，用来完成特定功能。

它不一定长得像传统计算机，但里面可能有处理器、存储器、传感器、控制程序和操作系统。

例子：

  • 路由器；

  • 智能手表；

  • 汽车控制器；

  • 家电控制系统；

  • 工业控制设备；

  • 传感器节点。

嵌入式操作系统运行在嵌入式系统环境中，负责协调、调度、控制系统中的各种资源。

它常常面临几个限制：

  • 资源少；

  • 功耗敏感；

  • 实时性要求高；

  • 功能专用；

  • 可靠性要求高。

5. 智能卡操作系统

智能卡是一种包含 CPU 芯片的卡，比如带芯片的银行卡、身份认证卡等。

智能卡的资源非常有限。课件图中示意的智能卡存储配置包括 ROM、EEPROM 和 RAM，其中 RAM 可能只有几 KB 级别。

智能卡操作系统要在很小的资源空间里完成：

  • 命令解释；

  • 安全校验；

  • 加密解密；

  • 数据处理；

  • 应答生成；

  • 与读写器通信。

它通常通过“命令—响应对”的方式与读写器通信：

  1. 读写器发出操作命令；

  2. 智能卡接收命令；

  3. 操作系统解释命令；

  4. 完成解密与校验；

  5. 调用相应程序处理数据；

  6. 生成应答信息；

  7. 加密后送回读写器。

有些智能卡还面向 Java，ROM 中带有 Java 虚拟机解释器，可以下载并解释执行 Java 小程序。如果多个小程序同时运行，就又会遇到调度、资源管理和保护问题。

这说明哪怕是极小的系统，只要有多任务、资源管理和保护需求，也可能需要操作系统。

十三、操作系统的分类方式

课件里给了两种分类视角。

1. 传统分类

传统上可以分为：

  • 批处理操作系统；

  • 分时系统；

  • 实时操作系统；

  • 个人计算机操作系统；

  • 网络操作系统；

  • 分布式操作系统；

  • 嵌入式操作系统。

其中批处理、分时、实时可以理解为按“工作方式和目标”分类；个人计算机、网络、分布式、嵌入式则更多按“应用场景和系统形态”分类。

2. Tanenbaum 的分类

Tanenbaum 在《现代操作系统》中还给出了另一种分类，更偏向硬件平台和应用场景：

  • 大型机操作系统；

  • 服务器操作系统；

  • 多处理机操作系统；

  • 个人计算机操作系统；

  • 掌上计算机操作系统，也就是移动计算机操作系统；

  • 嵌入式操作系统；

  • 传感器节点操作系统；

  • 实时操作系统；

  • 智能卡操作系统。

这说明操作系统没有一个永远固定的形态。它会随着硬件规模、资源限制、应用需求和用户交互方式的变化而变化。

关键理解

操作系统分类不是为了背名字，而是为了理解：不同场景下，系统最关心的目标不同。

例如：

  • 批处理系统关心吞吐量；

  • 分时系统关心响应时间；

  • 实时系统关心 deadline；

  • 个人操作系统关心易用性；

  • 网络操作系统关心通信和共享；

  • 分布式操作系统关心透明性和协同；

  • 嵌入式系统关心资源限制和可靠性；

  • 云计算系统关心大规模资源管理和弹性调度。

十四、这一讲最重要的技术关键词

课件最后的小结里提到一些关键技术和系统名。这里可以集中整理一下。

1. 多道程序设计

核心是让多个程序同时进入内存，当一个程序等待 I/O 时，CPU 可以执行另一个程序。

关键词：

  • 提高 CPU 利用率；

  • 提高吞吐量；

  • 需要调度；

  • 需要存储保护；

  • 引入并发问题。

2. 中断

中断让 CPU 不必一直轮询等待设备。

设备完成 I/O 后，可以通过中断通知 CPU：“我好了，你可以来处理了。”

这对操作系统非常重要，因为没有中断，多道程序设计和高效 I/O 管理都很难实现。

3. 通道

通道可以理解成专门负责 I/O 操作的处理部件。

它让 CPU 不必亲自处理所有 I/O 细节，从而实现 I/O 和计算的重叠。

4. SPOOLing

核心是用磁盘作为输入井和输出井，把慢速外设与高速 CPU 解耦。

关键词：

  • 输入井；

  • 输出井；

  • 输入进程；

  • 输出进程；

  • 打印队列；

  • 虚拟设备；

  • 独占设备变共享设备。

5. Buffer Cache

核心是用内存缓存磁盘块，减少真实磁盘 I/O。

关键词：

  • 文件子系统；

  • 块设备；

  • 缓存命中；

  • 缓存未命中；

  • 脏块；

  • 延迟写回。

十五、考试可答版总结

如果这一讲要压缩成一页复习骨架，可以这样记。

1. 操作系统的定义

操作系统是计算机系统中的系统软件，是一组程序模块的集合。它负责有效、合理地组织和管理软硬件资源，控制程序执行，并向用户提供服务，使用户能够方便使用计算机，使整个系统高效运行。

2. 操作系统的三大作用

  • 资源管理者；

  • 服务提供者；

  • 机器能力扩展者。

3. 理解操作系统的三个视角

  • 资源管理视角；

  • 进程视角；

  • 虚机器视角。

4. 操作系统的四大特征

  • 并发；

  • 共享；

  • 虚拟；

  • 随机性。

5. 批处理系统的核心

批处理系统把多个作业组成一批，由系统自动连续执行。它追求吞吐量和资源利用率。

单道批处理一次主要运行一个作业，I/O 时 CPU 容易空闲。

多道批处理让多个作业同时进入内存，一个作业等待 I/O 时，CPU 可以执行另一个作业，从而提高资源利用率。

6. SPOOLing 的核心

SPOOLing 利用磁盘作为缓冲，设置输入井和输出井，把输入、计算、输出组织成独立任务流，使 I/O 和计算并行，提高 I/O 效率，并把独占设备改造成共享设备，实现虚拟设备功能。现代打印队列仍然使用这种思想。

7. 分时系统的核心

分时系统把 CPU 时间切成时间片，轮流为多个用户服务。它追求及时响应，其思想仍然存在于现代多任务、多用户、服务器和云计算系统中。

8. 实时系统的核心

实时系统必须在规定时间内响应外部事件。硬实时不能错过 deadline，软实时允许偶尔超时。

9. 操作系统的发展主线

操作系统的发展受到硬件成本、人工成本、应用需求和软件技术变化的共同推动。从批处理到分时，从个人计算到网络分布式，从移动计算到云计算和物联网，本质上都是操作系统管理对象和服务目标不断扩展的过程。

十六、最后的直觉总结

这一讲如果只记一句话，可以记：

操作系统是管理计算机资源、抽象硬件能力、协调并发活动、保护系统安全，并向用户和应用提供服务的系统软件。

如果再展开一点：

它通过并发让多个任务一起推进，通过共享提高资源利用率，通过虚拟化制造更好用的逻辑资源，通过调度和保护避免程序互相打架；而它的发展史，本质上就是在不同硬件条件和应用需求下，不断重新平衡“效率、交互、可靠、安全、易用”的过程。
