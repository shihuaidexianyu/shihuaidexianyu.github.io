异常控制流 (ECF)

控制流与 ECF 的层级

处理器的常规控制流由顺序执行、分支、调用和返回构成，它们只响应程序内部状态的变化。真实系统还必须响应外部设备、计时器、算术错误、缺页和用户中断等系统事件，因此需要异常控制流。异常控制流由低到高可分为异常、进程上下文切换、信号和非本地跳转。

异常的定义与处理流程

异常是由于某个事件引起的控制流突变：处理器暂停当前用户指令流，把控制权转交给内核中的异常处理程序。处理器为不同异常类型维护异常号，异常号作为索引进入异常表，找到对应处理程序入口地址。异常处理结束后，控制要么返回到当前指令，要么返回到下一条指令，要么根本不返回而直接终止程序。

中断由处理器外部事件触发，具有异步性。计时器中断和 I/O 完成中断都属于此类。中断处理完成后，控制流返回下一条指令，因为被打断的当前指令本身并没有出错。

陷阱是由当前指令有意触发的同步异常，最重要的用途是系统调用。syscall 与普通函数调用相似，都会转入另一段代码并在结束后回到下一条指令，但系统调用发生了特权级切换：控制权进入内核态，由操作系统执行受保护操作。

故障是由当前指令引发的同步异常，但具有可恢复性。典型例子是缺页。若访问的虚拟地址合法，只是所需页暂不在物理内存中，则内核可先把页调入，再返回到当前指令重新执行；从用户进程视角看，只是一次访问变慢了。若访问的是非法地址，则故障处理器无法修复，通常会进一步触发程序终止，例如通过 SIGSEGV 报告段错误。

中止表示不可恢复的灾难性异常，例如硬件奇偶校验错误或严重非法执行状态。一旦发生，中止通常直接结束程序，不再返回。

进程的抽象

进程是正在运行的程序实例。进程提供两个关键抽象：逻辑控制流与私有地址空间。逻辑控制流使每个程序看起来都像独占 CPU；私有地址空间使每个程序看起来都像独占主存。这两个抽象分别通过上下文切换和虚拟内存机制实现。

一个典型进程的用户空间通常从低地址到高地址依次包括保留区、.text、.rodata、.data、.bss、堆、内存映射区和栈。堆向高地址增长，栈向低地址增长。更高地址处映射内核空间，所有进程共享这一部分，但用户态代码不能直接访问，只能通过系统调用、异常和中断进入内核态后由内核代为操作。

单核机器上的多进程并发来自时间片轮转；多核机器则允许真正并行。并发是逻辑控制流在时间上重叠，并行则是并发在物理硬件上同时推进的特例。

上下文切换与调度

进程上下文是恢复一个被抢占进程所需的全部状态，包括通用寄存器、程序计数器、用户栈、内核栈、页表、进程表项和打开文件表等。上下文切换的基本步骤是保存当前进程上下文、恢复目标进程上下文、把控制交给新进程。内核并不是独立进程，而是作为某个现有进程的执行片段运行；控制流在用户态和内核态之间穿梭，并通过调度实现不同进程间切换。

进程控制

系统调用出错时通常返回 -1 并设置 errno，因此所有系统级函数都必须检查返回值。getpid() 返回当前进程 PID，getppid() 返回父进程 PID。

进程状态可概括为运行、停止和终止。运行态既包括真正占用 CPU 的状态，也包括处于可运行队列中等待调度的状态。停止态通常由 SIGSTOP、SIGTSTP、SIGTTOU 或 SIGTTIN 触发，直到收到 SIGCONT 才恢复。终止态意味着进程已永久结束。

fork() 用于创建子进程。它最重要的语义是“调用一次，返回两次”：在父进程中返回子进程 PID，在子进程中返回 0。父子进程在 fork() 的下一条语句开始并发执行，它们拥有内容相同但彼此独立的地址空间副本，并共享打开文件的底层文件表项。父子谁先运行不可预测，但在每个单独进程内部，语句先后顺序仍然严格保持。

exit(status) 或从 main 返回会终止当前进程，并把退出状态留给父进程回收。若父进程尚未执行回收操作，已终止子进程会变成僵尸进程，占用少量内核资源。父进程通过 wait() 或 waitpid() 获取子进程终止信息并回收其资源。若父进程先于子进程结束，孤儿进程将被 init 进程领养并最终回收。

execve(filename, argv, envp) 在当前进程上下文中装入并运行一个新程序。它不会创建新进程，而是用新程序替换当前进程的代码、数据和栈，因此成功后通常不返回。典型用法是父进程先 fork()，再由子进程立即 execve() 执行另一程序。

Shell 与作业控制

Shell 的核心工作是读取命令行、解析命令、区分内置命令与外部程序，并在前台或后台运行作业。最基本的实现方式是：若命令不是内置命令，Shell 先 fork() 子进程，再由子进程 execve() 用户程序；若为前台作业，则父进程 waitpid() 等待；若为后台作业，则父进程立即返回提示符。

作业是 Shell 管理进程或进程组的单位。前台作业会占据终端，后台作业则让 Shell 继续接受输入。键盘上的 Ctrl-C 与 Ctrl-Z 并不是发送给单个进程，而是发送给前台进程组。jobs、fg、bg 等命令正是围绕“作业”这一抽象设计的。

简单 Shell 若只等待前台作业而忽略后台作业，就会产生僵尸进程。因此真正可用的 Shell 必须借助信号机制，在后台子进程终止时异步地进行回收。

信号的语义

信号是内核发送给进程的一种小型异步通知，只携带信号编号以及“该信号已经到达”这一事实。每种信号在进程上下文中最多只保留一个待处理实例，因此信号不排队。若某类型信号已经待处理，后续同类型信号会被合并而不是逐个排队。

内核为每个进程维护两个位向量：pending 表示哪些信号待处理，blocked 表示哪些信号被阻塞。只有属于 pending 且不在 blocked 中的信号，才真正有资格被递送。

信号可由内核因系统事件发送，也可由其他进程通过 kill 发送。kill(pid, sig) 中，pid > 0 表示发送给指定进程，pid = 0 表示发送给当前进程组，pid < 0 表示发送给进程组号为 |pid| 的所有成员。

当内核准备把控制权还给某进程时，会检查其未阻塞待处理信号集合。若集合为空，则继续正常执行；若非空，则强制该进程接收其中一个信号，并执行默认动作、忽略动作或用户安装的处理程序。

信号处理程序与可捕获性

进程可以通过 signal(signum, handler) 或更可靠的 sigaction 安装处理程序。处理程序是一个用户态函数，收到信号后被异步调用，返回后通常回到被中断的位置继续执行。SIGKILL 与 SIGSTOP 不能被捕获或忽略；SIGINT、SIGTERM、SIGCHLD 等标准信号通常既可捕获，也可忽略；SIGSEGV、SIGFPE 等严重错误信号虽可捕获，但忽略它们往往会让程序继续运行在错误状态中，风险极高。

处理程序本身是与主程序并发的逻辑流。一个处理程序还可能被其他类型信号打断，从而形成嵌套处理。内核会隐式阻塞当前正在处理的同类型信号，防止无限递归重入。

信号安全与常见陷阱

信号处理最重要的原则是最小化处理程序的行为。处理程序应尽量只做“置位一个标志”“调用一次 waitpid 循环回收子进程”“用 write 输出简短信息”这类简单动作。printf、malloc、exit 等通常都不是异步信号安全函数，不应在处理程序中调用。全局标志宜声明为 volatile sig_atomic_t，以避免编译器优化与非原子访问带来的问题。

SIGCHLD 不能被当作“子进程计数器”。因为信号不排队，多个子进程几乎同时结束时，内核可能只递送一个 SIGCHLD。因此处理程序必须循环调用 waitpid(-1, ..., WNOHANG) 或同类形式，一次性回收所有已终止子进程。

另一个典型陷阱是竞争条件。若父进程先 fork()，再把子进程 PID 写入作业表，而这期间子进程已迅速退出并触发 SIGCHLD，那么处理程序可能在作业尚未登记前就尝试删除该作业。正确做法是在 fork() 之前先阻塞 SIGCHLD，待父进程把子进程加入作业表后再解除阻塞。

显式等待信号

忙等待会浪费 CPU，单独使用 pause() 则存在丢信号窗口：信号可能恰好在条件检查之后、pause() 之前到达，使进程永远睡眠。sigsuspend 通过“原子地替换信号掩码并睡眠”解决这一问题，是等待异步信号事件的标准方式。

非本地跳转

setjmp 与 longjmp 提供一种用户态异常控制流机制。setjmp(j) 保存当前执行环境，包括寄存器、栈指针和程序计数器，并在直接调用时返回 0。随后某个更深层调用中的 longjmp(j, v) 可以恢复该环境，使控制流直接跳回 setjmp 所在位置，并令 setjmp 这一次返回非零值 v。

这种机制打破了常规函数调用与返回的后进先出规则，适合实现错误恢复与“快速逃离深层调用栈”。但它只能跳回一个尚未返回的保存点，不能跳入已经失效的栈帧。与信号配合使用时，通常应改用 sigsetjmp 与 siglongjmp，以便同时保存和恢复信号掩码状态。
