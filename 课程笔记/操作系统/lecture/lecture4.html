处理器调度概述

Summary

这一讲我们要解决的核心问题是：CPU 只有一个（或几个），而想跑的进程有一大堆，到底该让谁先跑？我们会从调度的基本概念出发，聊聊上下文切换的代价、调度算法的设计权衡，再一路遍历 FCFS、SJF、RR、MLFQ、Windows 线程调度、Linux CFS 等经典实现。看完这篇，你应该能回答：什么时候触发调度？各种调度算法各有什么坑？CFS 的 vruntime 到底在算什么？

调度，说白了就是操作系统当“班长”：班里很多同学（进程/线程）都想用投影仪（CPU），班长得定个规矩，决定这一刻投影仪给谁用。这个规矩就是调度算法。

在单处理器系统里，同一时刻真正在跑的进程只有一个，其他都在就绪队列里排队；到了多处理器系统，班长还得额外操心——这个同学该去哪个 CPU 上跑？所以调度问题可以拆成三块：

  • 调度算法：凭啥选它？
  • 调度时机：啥时候做决定？
  • 调度过程：怎么把人换上去？

Intuition

想象你在食堂排队打饭。调度算法就是“谁先打饭”，调度时机就是“什么时候轮到下一个人”，调度过程就是“盘子怎么从上一个同学手里传到下一个同学手里”。

从层次上看，调度通常分三层：

  • 长程调度（作业调度）：决定一个新作业要不要被创建为进程，放进活跃进程集合里。发生频率最低。
  • 中程调度（内存调度）：负责把进程换进换出内存，缓解内存压力。相当于班长先把一些不紧急的同学“请出教室”，腾地方。
  • 短程调度（CPU 调度）：真正决定就绪队列里谁上 CPU。这是发生最频繁的调度，时间尺度往往在毫秒级，所以实现必须又快又稳。

CPU 调度不是随时都发生的，它由一些特定事件触发。典型的触发点有：进程正常结束、异常终止、新进程创建、当前进程等 I/O 阻塞了、被阻塞的进程被唤醒、时间片用完了，或者进程主动 yield 让出 CPU。更一般地说，只要中断、异常或系统调用导致内核接管了控制权，并且在返回用户态之前发现进程状态变了，就会跑调度程序重新选人。

调度的落地离不开进程切换和上下文切换。

  • 进程切换：一个进程下台，另一个进程上台的过程。
  • 上下文切换：更具体一点，就是把 CPU 里当前进程的寄存器状态（程序计数器、PSW、栈指针、通用寄存器等）保存到它的 PCB 里，再把新进程的寄存器状态从 PCB 恢复到 CPU 里。如果换了地址空间，通常还要切换页表和内核栈。

Note

上下文切换是有代价的。直接开销是保存/恢复寄存器、切换地址空间消耗的时间；间接开销更隐蔽——缓存（Cache）、TLB 都会因为换了进程而大量失效。所以调度太频繁虽然响应快，但 CPU 会花很多时间在“换人”上，真正干活的吞吐量反而下降。

处理器调度算法的设计原则

Summary

这节讲调度算法的设计目标和评价指标。没有“放之四海而皆准”的最优算法，只有“最适合当前场景”的算法。我们会看看交互式、批处理、实时系统各自关心什么，以及设计调度器时必须回答的几个结构问题。

不同系统面对的负载不一样，调度的目标自然也不同：

  • 交互式进程：经常等用户输入，目标是响应时间短，体验要均衡。
  • 批处理进程：不要快速响应，目标是吞吐量高、周转时间短、CPU 利用率高。
  • 实时进程：核心是截止时间和可预测性，关键任务不能被低优先级任务卡住。

所以调度算法没有绝对最优，永远是在响应时间、周转时间、吞吐量、公平性、可预测性、资源平衡和能耗之间做折衷。

常见的评价指标：

  • 吞吐量：单位时间完成的进程数。
  • 周转时间：进程从提交到完成经历的总时间。
  • 响应时间：从请求提出到第一次得到响应的时间。
  • 实时性：对实时系统，能不能在截止时间内完成。
  • 能耗：对服务器和移动系统越来越重要。

周转时间的基本定义为

Tturnaround​=Tcompletion​−Tarrival​

其中，Tarrival​ 是作业到达时刻，Tcompletion​ 是作业完成时刻。这个指标特别适合衡量批处理场景里的整体等待代价。

Important

周转时间 = 完成时间 - 到达时间。

设计调度算法时，还要想清楚几个结构问题：

  1. PCB 里要记哪些调度信息？比如静态优先级、动态优先级、剩余时间、已等待时间、时间片余额、处理器亲和性。
  2. 就绪队列怎么组织？单队列？多优先级队列？还是每个 CPU 一个队列？
  3. 要不要用抢占式调度？
  4. 怎么区分 I/O 密集型和 CPU 密集型进程？
  5. 时间片设多大才合适？

优先级可以是静态的，也可以是动态的。

  • 静态优先级：进程创建时定好，运行中不变。
  • 动态优先级：根据等待时间、睡眠行为、I/O 完成情况或饥饿状态动态调整。

Intuition

动态优先级就像班长看某个同学等了很久，临时给他往前排一排，兼顾公平性和响应性。

就绪队列的组织方式直接影响调度复杂度。

  • 如果按优先级分层，可以维护多个就绪队列，高优先级队列先被检查。
  • 如果统一组织，则所有进程在一个公共队列里竞争。

前者便于表达复杂策略，后者实现简单。多队列结构还方便把实时任务、前台任务、后台任务分开处理。

抢占 vs 非抢占：

  • 抢占式调度：允许系统在更高优先级任务就绪时，强制剥夺当前任务的 CPU。有利于交互性和实时性，但实现更复杂，上下文切换开销更大。
  • 非抢占式调度：当前任务不主动阻塞、退出或放弃 CPU，就一直执行到底。实现简单、稳定，但可能导致长任务压制短任务。

Trap

很多同学容易混淆：抢占式调度一定更好吗？不一定！抢占太频繁会导致切换开销爆炸，而且实现难度大。选不选抢占，要看场景需求。

按运行行为可以把进程分成两类：

  • I/O 密集型进程：CPU 突发很短，频繁等待 I/O。如果调度得当，可以让 I/O 设备和 CPU 充分重叠工作。
  • CPU 密集型进程：长时间占用 CPU 做计算，更关心持续处理能力。

现代调度系统通常会对 I/O 密集型或睡眠较多的进程给予一定倾斜，改善系统交互性。

时间片是分时系统的核心参数。

  • 时间片太长：轮转调度退化为先来先服务，短任务响应变差。
  • 时间片太短：上下文切换太频繁，CPU 时间都花在“换人”上了。

Example

假设时间片只有 1ms，但切换一次要 0.5ms。那 CPU 三分之一的时间都在做上下文切换，真正干活的时间只剩三分之二。所以时间片不是越小越好。

时间片的选取要综合考虑切换成本、响应时间要求、就绪进程数、CPU 性能和进程行为模式。

典型处理机调度算法

Summary

这节是考试的重中之重！我们会从批处理系统到交互式系统，逐个拆解 FCFS、SJF、SRTN、HRRN、RR、Virtual RR、MLFQ 等经典算法。每个算法的核心思想、优缺点、适用场景，都要心里有数。

批处理系统中的典型算法

FCFS（先来先服务）

先来先服务算法（FCFS）按进入就绪队列的先后顺序分配 CPU，是最直接的非抢占式策略。它实现简单，形式上也很公平，但有个大坑叫**“护航效应”**（Convoy Effect）：如果一个长作业先到了，后面的短作业即使很快就能跑完，也得眼巴巴地等着。

Intuition

护航效应就像你去奶茶店排队，前面有个人点了 20 杯复杂的定制奶茶，而你只想买一杯纯茶。明明你的单 10 秒就能做完，但因为他在你前面，你得等半小时。

同时，长作业一直占着 CPU 时，其他 I/O 密集型进程难以及时发起 I/O，可能导致 CPU 和 I/O 设备都利用不起来。

来看一个经典例子：设三个作业 P1​,P2​,P3​ 按顺序到达，运行时间分别为 24 s,3 s,3 s。若采用 FCFS，执行顺序为 P1​→P2​→P3​，完成时刻分别为 24,27,30，平均周转时间为

Tˉ=324+27+30​=27

若只改变执行顺序为 P2​→P3​→P1​，完成时刻变为 3,6,30，平均周转时间为

Tˉ=33+6+30​=13

Trap

FCFS 的平均周转时间对到达顺序极其敏感！长作业放前面会显著拉高平均周转时间。考试如果给你几个进程让算 FCFS 的平均周转时间，千万别看漏到达顺序。

SJF（最短作业优先）

最短作业优先（SJF）总是优先执行预计完成时间最短的作业。核心思想是：先让短作业跑完，可以最小化平均周转时间。如果所有作业同时可运行且运行时间已知，SJF 确实能拿到最短的平均周转时间。

但它有两个硬伤：

  1. 需要预测未来：实际中很难准确知道作业还要跑多久。
  2. 长作业饥饿：如果短作业源源不断，长作业可能永远轮不到。

Example

想象医院挂号窗口来了个“只看感冒”的优先通道。如果感冒病人源源不断，那个摔断了腿但流程看起来“很长”的病人可能一直挂不上号。

SRTN（最短剩余时间优先）

最短剩余时间优先（SRTN）是 SJF 的抢占式版本。当新到达进程的剩余时间小于当前运行进程的剩余时间时，系统立即抢占当前进程，改跑新进程。

比如四个进程的到达时间分别为 0,2,4,5，运行时间分别为 7,4,1,4：

  • 非抢占式 SJF 的执行序列：P1​→P3​→P2​→P4​
  • 抢占式 SRTN 的序列：P1​→P2​→P3​→P2​→P4​→P1​

抢占让系统能及时响应更短的新作业，但代价是实现复杂度和切换开销增加。

HRRN（最高响应比优先）

最高响应比优先（HRRN）想在 SJF 的短作业偏好和长作业公平性之间找个平衡点。响应比定义为

R=SS+W​=1+SW​

其中，S 是作业处理时间，W 是等待时间。

Intuition

短作业因为 S 小，响应比容易高；长作业虽然一开始吃亏，但等得越久，W 越大，响应比也会慢慢涨上去，最终一定能轮到。这就好比排队等公交，虽然你排在后面，但站得久了，司机也会让你先上。

HRRN 通常采用非抢占式，因为如果抢占式地频繁重算响应比并排序，系统开销会很大。

交互式系统中的典型算法

RR（时间片轮转）

轮转调度（RR）给每个进程分配固定时间片，时间片用完就通过时钟中断把当前进程送回就绪队列尾部，再调度下一个进程。

它的优点是形式公平，能有效降低短任务和交互任务的响应时间；代价是切换开销较高。时间片太大退化为 FCFS，太小则上下文切换爆炸。

RR 并不总能改善完成时间。若两个进程 A,B 的运行时间都为 100 ms，时间片为 1 ms，忽略切换开销，则 RR 的完成时刻分别约为 199 ms 和 200 ms，平均完成时间为

CˉRR​=2199+200​=199.5 ms

若采用 FCFS，完成时刻为 100 ms 和 200 ms，平均完成时间为

CˉFCFS​=2100+200​=150 ms

Trap

RR 主要改善的是响应公平性，不是总能优化完成时间！考试时如果问 RR 是否一定比 FCFS 好，答案是：看你看重什么指标。

再看一个四进程例子，运行时间分别为 53,8,68,24。课件给出的结果是：时间片为 20 时平均等待时间为 66.25；时间片为 5 时下降到 57.25；而最优顺序的 FCFS 平均等待时间可达 31.25，最差顺序则恶化到 83.5。这说明 RR 更稳定、更公平，而 FCFS 的表现极度依赖作业排列。

Virtual RR（虚拟轮转）

虚拟轮转（Virtual RR）是普通 RR 的改进版，专门照顾 I/O 密集型进程。

基本机制：在普通就绪队列之外再设一个辅助队列。新到达的进程和时间片用完的进程进普通就绪队列；请求 I/O 的进程进 I/O 等待队列；I/O 完成后，该进程不返回普通就绪队列尾部，而是先进入辅助队列。调度时，系统优先从辅助队列里选人，辅助队列空了才从普通就绪队列按 RR 规则选。

flowchart TD
    A["新进程进入系统"] --> B["普通就绪队列<br/>Ready Queue"]
    B --> C{"调度器选择下一个进程"}
    C -->|辅助队列非空| D["辅助队列<br/>Auxiliary Queue"]
    C -->|辅助队列为空| B

    D --> E["CPU 运行"]
    B --> E

    E --> F{"运行结果"}
    F -->|时间片用完| B
    F -->|进程结束| G["Exit"]
    F -->|发起 I/O 请求| H["I/O 等待队列"]

    H -->|I/O 完成| D

这样，那些不断重复“短 CPU 突发 + I/O 等待”的 I/O 密集型进程，在 I/O 完成后能更快重新拿到 CPU，并及时发起下一次 I/O，让处理器和外设保持重叠工作。计算密集型进程则主要在普通就绪队列里轮转。

Example

进程 A 和 B 都需要 1000 ms 纯计算，进程 C 不断重复“CPU 1 ms + I/O 10 ms”。

  • FCFS：A 或 B 一直占着 CPU，C 很难发起新 I/O，磁盘大部分时间空闲。
  • RR 时间片 100 ms：磁盘还是经常闲着。
  • RR 时间片 1 ms：C 能很快重新发起 I/O，磁盘利用率接近 90%，但切换开销剧增。

Virtual RR 的好处就是：不用把时间片切到极端那么小，也能让 I/O 密集型任务获得较好响应。

最高优先级优先调度

最高优先级优先调度总是选当前优先级最高的进程运行。系统进程通常高于用户进程，前台进程高于后台进程，I/O 型进程也常常受到偏好。实现简单，但如果没有动态调整或老化机制，低优先级进程可能长期饥饿。

优先级反转

基于优先级的抢占式系统里，优先级反转是个关键问题，考试常考。

假设有高优先级进程 H、中优先级进程 M、低优先级进程 L：

  1. L 先进入临界区，拿到了某个锁。
  2. L 被 M 抢占（因为 M 优先级比 L 高）。
  3. H 来了，也需要那把锁，于是阻塞等待。
  4. 现在 M 一直跑，L 拿不到 CPU 就无法释放锁，结果最高优先级的 H 反而被卡死了。

Intuition

优先级反转就像：校长（H）急着进会议室，发现门被实习生（L）锁了，但实习生被主管（M）叫去干别的了。主管优先级比实习生高，但比校长低。结果主管一直占用实习生，校长就被晾在门口干瞪眼。

后果可能是系统性能下降，甚至在关键场景下造成系统错误。常见解决方案：优先级继承、优先级天花板协议、在极短临界区内禁止中断。

Mars Pathfinder 案例：火星探路者号就是因为优先级反转导致系统复位的经典事故。

  • 低优先级任务：气象/信息采集线程，先拿到了共享资源上的互斥锁。
  • 高优先级任务：数据总线管理线程，后来也需要这把锁，被阻塞。
  • 中优先级任务：通信等工作线程，不需要那把锁，但优先级比低优先级线程高，不断抢占低优先级线程。

结果低优先级任务没法释放锁，高优先级任务被间接阻塞，最终触发了看门狗复位。看门狗（watchdog timer）要求系统必须在规定时间内“喂狗”，否则就认为系统出问题了，执行恢复动作（通常是重启）。

Note

这个案例在操作系统课上经常被拿出来讲，因为它太经典了——一个理论上已经知道的问题，在真实的航天任务里差点毁了整个项目。修复方法其实很简单：启用优先级继承即可。

MLFQ（多级反馈队列）

多级反馈队列调度（MLFQ）是一种依据进程历史运行行为动态调整优先级的综合方法。系统设置多个优先级递减的就绪队列，高优先级队列时间片较短，低优先级队列时间片较长，各级队列内部通常按轮转方式调度。

flowchart TD
    A["新建进程"] --> Q1["就绪队列 1<br/>最高优先级, 最短时间片"]

    Q1 -->|调度| CPU["CPU 运行"]
    Q2["就绪队列 2"] -->|调度| CPU
    Q3["就绪队列 3<br/>最低优先级, 最长时间片"] -->|调度| CPU

    CPU -->|时间片用完| DOWN1["降到下一优先级队列"]
    DOWN1 --> Q2
    Q2 -->|时间片再次用完| Q3

    CPU -->|I/O 或事件阻塞| WAIT["等待队列"]
    WAIT -->|事件完成| BACK["回到原优先级队列"]
    BACK --> Q1

    H["更高优先级进程就绪"] -->|允许抢占| PREEMPT["当前低优先级进程被抢占"]
    PREEMPT --> Q2

运作规则：

  • 新进程先进入最高优先级队列。
  • 如果进程在当前队列里用完了整个时间片，说明它更像 CPU 密集型，被降入下一优先级队列。
  • 如果进程在时间片耗尽前因 I/O 阻塞，说明它更像交互式/I/O 密集型，事件完成后通常回到原来的优先级队列（不是继续下沉）。
  • 当更高优先级队列中出现新就绪进程时，当前低优先级队列的进程会被抢占。被抢占的进程一般回到其原来的就绪队列。

Intuition

MLFQ 的精髓是：操作系统不静态地给你贴标签，而是“看表现”。你每次都把时间片用完？好，你是计算型的，慢慢往下沉吧。你跑一会儿就去等 I/O？好，你是交互型的，继续留在高层享受短响应时间。

交互式系统中还常见公平共享调度、保证调度和彩票调度等思想。公平共享先把 CPU 份额分配给用户或用户组，再由组内进程共享；保证调度强调每个任务在较长时间窗口内获得接近其应得比例的处理器时间；彩票调度则把处理器份额映射为“彩票”，通过随机抽签实现概率意义上的比例公平。这些方法强调的不是单次选择的确定性最优，而是长期份额控制。

各种典型算法可以从多个维度比较：

  • FCFS 开销小、无饥饿，但对短作业和 I/O 型作业不利；
  • RR 公平、响应快，但时间片过小时吞吐量下降；
  • SJF 和 SRTN 能显著改善平均周转时间，但可能造成长作业饥饿；
  • HRRN 通过等待时间修正改善了公平性；
  • 反馈队列类算法更适合混合负载，尤其对交互式进程更友好，但策略和参数更复杂。

调度设计始终是一种折衷，而不是单一指标最大化。

机制与策略分离、线程调度

Summary

这节讲调度设计中的一个重要原则：机制与策略分离。同时我们会厘清“进程调度”和“线程调度”的区别，这对理解现代操作系统很关键。

调度设计里有一个长期成立的原则：机制与策略分离。

  • 机制：调度赖以运作的内核基础设施，比如时钟中断、就绪队列、优先级位图、上下文切换、处理器亲和性控制、抢占触发点。
  • 策略：回答“谁应当先运行、运行多久、何时提升或降低优先级”。

分离的意义在于：底层机制一旦稳定，系统可以在不重写内核的情况下更换策略；策略变化也不会破坏上下文切换、中断处理等核心机制。这样既好维护，也方便在不同场景下复用同一套基础设施。

在线程调度问题上，关键是内核是否直接感知线程：

  • 如果内核只调度进程，那么系统先选中一个进程，再由用户级运行时决定该进程内哪个线程真正执行。
  • 如果内核直接调度线程，那么不同进程中的线程可以直接竞争 CPU。

后者更利于细粒度抢占和阻塞管理，现代通用操作系统通常采用以线程为基本调度单位的设计。

Windows 线程调度

Summary

这节我们走进 Windows 的调度器内部，看看它怎么用“动态优先级 + 时间配额”来管理线程。Windows 的调度单位是线程而不是进程，这点要记住。

Windows 的基本调度单位是线程，而不是进程。总体机制是：基于动态优先级的抢占式调度 + 时间配额控制。

系统维护多个按优先级划分的就绪队列，总是优先选最高优先级的就绪线程运行；同一优先级上的线程再按时间片轮转分配处理器。多处理器环境下，不同线程可以在不同处理器上并行执行。

Windows 中会触发线程调度的典型事件：

  • 线程优先级发生变化
  • 线程的处理器亲和集合发生变化
  • 线程正常终止或因异常终止
  • 新线程创建
  • 等待线程重新进入就绪态
  • 运行线程进入阻塞态
  • 运行线程因时间配额耗尽或其他原因回到就绪态

只要这些事件改变了可运行线程集合或调度优先次序，调度器就要重新选人。

Windows 把线程优先级划分为 32 个级别：

  • 优先级 0：保留给零页线程。
  • 优先级 1~15：可变优先级范围。
  • 优先级 16~31：实时优先级范围。

实时优先级线程通常不做动态调整；可变优先级线程的当前优先级可以在基本优先级附近上下浮动。基本优先级决定线程的长期调度地位，当前优先级反映调度器在特定阶段的临时修正。

Windows 中的时间配额不是直接用毫秒表示的固定长度，而是以内核配额单位表示的整数值。线程控制块里有 Quantum 和 QuantumReset 字段。一个线程用完当前时间配额后，如果同一优先级上没有其他就绪线程，系统可以重新赋予它一个新的时间配额，允许它继续运行。时间配额的作用不仅是限制单次连续运行时间，也在于调节不同线程实际获得的 CPU 份额。

为提高调度选择效率，Windows 维护就绪位图与空闲位图：

  • 就绪位图：每一位对应一个调度优先级，表示该优先级的就绪队列是否为空。
  • 空闲位图：每一位对应一个处理器，表示该处理器当前是否空闲。

借助这两类位图，调度器可以快速定位最高优先级的非空就绪队列，以及当前可立即投入使用的处理器。

线程离开 CPU 后的处理方式，取决于离开原因：

  • 主动阻塞（等 I/O、事件对象等）：从运行态转入阻塞态，条件满足后再进入就绪队列。
  • 被更高优先级线程抢占：放回原优先级就绪队列。
    • 实时优先级线程：时间配额被重置为完整配额。
    • 可变优先级线程：剩余时间配额通常保留，下次获得 CPU 后继续消耗。

Important

主要的手段是优先级改变和时间配额调整。

Windows 中，会被提升优先级的是可变优先级线程（1~15级），不是实时线程。原因是系统要提高交互响应性和 I/O 完成后的处理效率。

Windows 对可变优先级线程采用“临时优先级提升 + 时间配额调整 + 逐步衰减”的机制：

  • 线程的长期调度地位由基本优先级决定，当前优先级可根据等待与唤醒情况作临时修正。
  • 这种修正只作用于可变优先级范围内的线程，而且不是永久性的；在线程重新获得 CPU 后，提升幅度会随着运行逐步消失，最终回到基本优先级。

I/O 完成后：Windows 会临时提升等待该 I/O 的线程的当前优先级，让它更快重新获得 CPU 处理刚到达的数据。优先级的实际提升幅度由设备驱动在 I/O 完成路径中给出，响应时间要求越高，提升通常越大。为避免长期不公平，系统在唤醒该线程时会将其时间配额减 1。

等待事件对象或信号量结束时：Windows 也会提升其当前优先级，通常以基本优先级为基准提升一级，但不超过 15。线程先在提升后的优先级上运行完当前剩余时间配额，随后每获得一个新时间配额，优先级下降一级，直到恢复为基本优先级。

饥饿处理：平衡集管理器每秒扫描一次就绪队列。若发现某线程排队超过 300 个时钟中断间隔，就把它的优先级提升到 15，并分配一个长度为正常值 4 倍的时间配额。扩展时间配额用完后，该线程的优先级立即恢复为原来的基本优先级。

多处理器与多核调度

Summary

这节把调度问题从单核扩展到多核。多核调度不仅要决定“谁运行”，还要决定“在哪个 CPU 上运行”。缓存亲和性、负载均衡、作业偷取是这里的核心概念。

多处理器调度与单处理器调度相比，增加了一个核心决策：不仅要决定谁运行，还要决定在哪个 CPU 上运行。

对称多处理器系统（SMP）中，每个处理器都运行自己的调度程序，处理器之间可共享负载，但对共享资源的访问必须同步。这样做提高了并行性，也引入了队列竞争、迁移成本和负载均衡问题。

进程在多个 CPU 之间迁移会造成额外开销，特别是缓存失效和 TLB 失效。因此，调度器通常倾向于让一个进程尽量在同一个处理器上反复运行，这就是处理器亲和性的基本思想。但如果过于强调亲和性，又可能导致某些处理器负载过高、另一些处理器空闲，从而降低整体吞吐量。

Intuition

处理器亲和性就像你固定坐教室的某个座位，书本文具都放那，下次来不用重新收拾。但如果全班都死磕自己的座位，而某些座位附近的同学任务特别重，另一些座位却很闲，整体效率反而下降。所以要在“坐老位置”和“帮闲的同学分担”之间找平衡。

从队列组织上看，SMP 进程分配可以分为：

  • 静态分配：进程固定绑定在一个处理器上，每个处理器维护自己的就绪队列。调度开销低，但容易忙闲不均。
  • 动态分配：所有处理器共享公共就绪队列，或允许进程在执行期间迁移到任意空闲处理器上。负载更均衡，但调度和同步开销更大。

负载均衡的核心思想：不要让某些处理器很忙、另一些很闲，而是把线程尽量均匀分配到各个核心上，提升整体吞吐量和利用率。

多核系统中，处理器与缓存层次结构更复杂，调度必须额外关注缓存一致性、缓存亲和性、核间数据共享与负载均衡。

缓存一致性问题：多个核心同时读取并修改共享数据时，例如两个处理器都把共享变量 X 从 0 读入本地缓存后各自加 1，若没有一致性协议，最终主存中的结果可能不是 2。硬件通常通过 MESI 等缓存一致性协议解决。

缓存亲和性要求让特定进程或线程尽量在同一核心上运行，重复利用该核心缓存中的数据，减少访存开销。其与负载均衡的冲突可以用并行渲染任务说明：若把某块数据长期固定在某核心处理，缓存命中率较高；但若各数据块计算量不均，就需要把一部分任务迁移给较空闲的核心，以缩短总完成时间，而迁移又会带来缓存不命中。CDN 场景也存在类似矛盾：将用户请求路由到本地缓存服务器有利于命中率，但高负载时跨区转发又能缓解热点。

作业偷取（Work Stealing）是负载均衡中的典型方法。其思想是：每个处理器先优先执行本地任务队列中的工作；当本地队列为空时，再去其他处理器“偷取”任务。它既保留了本地队列带来的缓存局部性，又能在空闲处理器出现时进行补偿式均衡。代价是偷取本身需要同步，并可能破坏原有数据局部性。

Go 语言运行时的 GMP 调度器是作业偷取与缓存亲和性的典型案例。它尽量把 Goroutine 调度到先前处理过相关数据的处理器上，也尽量把同一 Goroutine 放回原先所在处理器 P 绑定的内核线程 M 上，从而保留缓存局部性；同时，每个 P 维护本地 Goroutine 队列，当本地队列耗尽时，再从其他 P 窃取任务。YARN 则把类似思想推广到分布式资源调度：任务倾向于被放到存有目标数据的节点上，以提高缓存利用率；但 ResourceManager 又会根据各节点负载进行跨节点协调。

实时调度

Summary

实时系统的核心不是“平均快”，而是“在规定时间内必须完成”。这节我们会区分硬实时和软实时，并介绍两种经典实时调度算法：RM 和 EDF。

实时系统关注的核心不是平均意义上的快，而是在规定时间边界内完成指定任务。按照时间约束的严格程度：

  • 硬实时：任务必须在规定时刻或时间区间内完成，违约通常不可接受。
  • 软实时：允许偶尔违反最终时限，但仍希望尽量按时完成。

按照到达模式，实时任务可分为周期性任务、偶发性任务和非周期性任务。实时调度的关键特征是确定性——系统能否明确判断任务是否能在截止时间前完成。

对于由多个周期性事件流构成的系统，若事件 i 以周期 Pi​ 发生，每次事件需要 Ci​ 秒 CPU 时间，则系统负载满足下式时可认为是可调度的：

i=1∑m​Pi​Ci​​≤1

其中，m 是周期性事件流个数，Ci​/Pi​ 表示任务 i 对处理器的平均利用率需求。若总需求超过 1，则处理器平均能力已不足以覆盖全部任务，系统必然无法满足所有截止约束。

Trap

上面这个公式是“必要条件”而不是“充分条件”！也就是说，即使总利用率 ≤ 1，某些调度算法（比如固定优先级的 RM）也可能调度失败。但对 EDF 来说，在单核上这个条件既是必要也是充分的。

速率单调调度（RM）是静态优先级实时调度算法，前提是所有任务都周期性到达且执行时间已知。其基本思想是：周期越短、到达越频繁的任务，优先级越高。因为短周期任务更容易积累时限压力，所以在固定优先级体系下，RM 用周期表达紧迫性。优点是实现简单、可预测性强；局限在于依赖稳定的周期性前提，对变化任务和复杂截止约束的适应性较弱。

最早截止时间优先（EDF）是动态优先级实时调度算法。系统根据当前任务的最后期限和剩余执行需求动态调整优先级，截止时间最早的任务优先级最高。EDF 的优势在于更灵活，通常能在更高处理器利用率下保持可调度，并且响应时间表现较好；其代价则是实现复杂，需要持续维护截止时间信息，并在任务集合动态变化时不断重新排序。

OpenEuler 的多核调度技术

Summary

这节看看国产操作系统 OpenEuler 是怎么做多核调度的。它在 CFS 基础上，针对多核场景做了亲和性和负载均衡的优化，迁移线程的设计很有特色。

OpenEuler 对 CPU 调度的基本理解与通用操作系统一致：在必要时中断当前进程，改由就绪队列中的另一个进程占用 CPU。不同之处在于，多核场景下还必须处理核间共享数据、同步与负载迁移。其普通进程以 CFS 为核心调度策略，强调相对公平；实时进程则联合使用 FIFO、RR 与优先级调度等方法，以满足更严格的时延要求。

OpenEuler 在多核系统上既要考虑缓存亲和性，又要考虑负载均衡：

  • 策略一（单队列）：每个 CPU 简单使用轮转法选择下一个进程，容易导致同一进程在不同 CPU 间来回迁移，破坏缓存亲和性。
  • 策略二（亲和度机制）：让进程尽量停留在同一个 CPU 上，虽然改善了局部性，却可能牺牲部分进程的均衡分布。

为进一步缓解问题，OpenEuler 采用多队列调度。每个 CPU 维护自己的本地队列，例如 Q0​ 由 CPU0 维护，Q1​ 由 CPU1 维护。这样可以增强局部性并减少全局锁竞争。但若某些 CPU 上的进程提前结束，就可能出现一部分队列空闲、另一部分队列仍然积压的负载失衡现象。

OpenEuler 用迁移线程解决这种失衡。每个处理器拥有一个迁移线程 migration/CPUID 和一个停机工作队列。当某 CPU 发现另一 CPU 上有可迁移任务时，会向其停机工作队列投递迁移函数并唤醒对应迁移线程；该迁移线程具有很高优先性，不会被普通进程抢占，因此能迅速完成跨 CPU 任务搬移，实现新的负载平衡。

Intuition

迁移线程的设计很形象：它不是让普通进程自己“搬家”，而是派了一个专门的“搬家公司”（迁移线程），这个搬家公司有特权，不会被路上的其他事打断，所以搬家又快又安全。

Linux 进程调度

Summary

这节是 Linux 调度的演化史，从 2.4 的简单遍历，到 O(1) 的双数组，再到 SD/RSDL 的“楼梯”思想，最后落脚到 CFS 的完全公平模型。这是整讲的高潮，也是面试和考试最常考的部分，建议多看几遍。

Linux 将调度对象分为实时进程与普通进程两大类。实时进程要求极低调度延迟，通常采用 FIFO 或 RR 策略；普通进程包括交互式进程与批处理进程，前者睡眠较多、重视响应，后者位于后台、可容忍更大延迟。现代 Linux 中，普通进程的核心策略是 CFS。

调度算法的演化大体经历了：Linux 2.4 的简单优先级加时间片调度 → Linux 2.6 的 O(1) 调度器 → SD 调度器补丁 → RSDL 调度器补丁 → CFS 调度器 → Linux 6.11 的 EEVDF 调度器。

Linux 2.4 调度器

flowchart TD
    A[进入调度] --> B{是否有实时进程}
    B -- 是 --> R[按实时策略 FIFO / RR 调度]
    B -- 否 --> C[遍历整个 runqueue]
    C --> D[比较普通进程优先级]
    D --> E[优先级主要由 counter 和 nice 决定]
    E --> F[选出最高优先级进程运行]

    F --> G[时钟中断到来]
    G --> H[counter 递减]
    H --> I{当前进程时间片是否耗尽}
    I -- 否 --> F
    I -- 是 --> J{所有 RUNNING 进程的 counter 都耗尽了吗}
    J -- 否 --> A
    J -- 是 --> K[统一重新计算所有进程的 counter]
    K --> L[睡眠较多的进程通常保留更多时间片余额]
    L --> M[下一轮调度中更容易获得较高优先级]
    M --> A

Linux 2.4 采用“运行队列 + 时间片 + 优先级”的设计。每次调度时，系统遍历 runqueue 中所有进程，比较优先级并选出最高者。普通进程创建时被分配时间片，时钟中断到来时当前运行进程的时间片递减；时间片耗尽后，该进程必须等待系统重新分配时间片才能继续运行。只有当所有 RUNNING 进程的时间片都用完时，系统才统一为它们重新计算时间片，从而保证所有进程最终都有机会运行。

在 Linux 2.4 中，普通进程的优先级由 counter 与 nice 共同决定。counter 可理解为当前时间片余额，数值越大优先级越高；nice 是静态负向优先级，取值范围为 −20 到 19，其中 −20 优先级最高。当系统统一重算 counter 时，睡眠中的进程通常保留了更多未用完时间片，因此其新 counter 会相对更大，从而间接提高了交互式进程的优先级。

Trap

Linux 2.4 的最大问题是：每次调度都要遍历整个 runqueue！如果系统里有几百上千个进程，调度复杂度就是 O(n)，高负载下性能很弱。这也是 O(1) 调度器诞生的直接原因。

O(1) 调度器

flowchart TD
    A[进入调度] --> B{是否有实时进程}
    B -- 是 --> R[实时进程优先调度]
    B -- 否 --> C[查看当前 CPU 的 active 数组]
    C --> D[找到最高优先级的非空队列]
    D --> E[取队首进程运行]
    
    E --> F[时钟中断 / 进程继续执行]
    F --> G{时间片是否耗尽}
    G -- 否 --> E
    G -- 是 --> H[重新评估动态优先级]
    H --> I{是否为交互式或实时任务}
    I -- 是 --> J[重置时间片并放回 active]
    I -- 否 --> K[放入 expired]
    
    J --> L{active 是否为空}
    K --> L
    L -- 否 --> A
    L -- 是 --> M[交换 active 与 expired 指针]
    M --> A

为解决 Linux 2.4 的问题，Ingo Molnar 提出了 O(1) 调度器。改进主要在两个方面：一是动态优先级的计算方法，二是 pick next 的选择机制。普通进程的动态优先级由静态优先级、睡眠奖励 bonus 等因素综合决定；平均睡眠时间越长，系统越倾向把该进程识别为交互式进程，并赋予更高优先级。实时进程优先级则不做动态修改，且始终高于普通进程。

O(1) 调度器为每个 CPU 维护两个优先级数组：active 与 expired。当需要选择下一个运行进程时，调度器直接从 active 数组中最高优先级的非空队列取出第一个进程，因此选择开销是常数级。若进程时间片耗尽，调度器会判断其是否属于交互式或实时进程：满足条件者可被重置时间片并重新放回 active；否则进入 expired。当 active 为空时，系统交换 active 与 expired 指针，重新开始下一轮。

Intuition

O(1) 调度器的核心 trick 就是“双数组 + 位图”。active 是“本赛季”，expired 是“下赛季”。本赛季打完的进下赛季，等本赛季所有人都打完了，两个赛季一交换，继续打。这样选下一个进程永远只需要查位图找最高非空优先级，再取队首，复杂度就是 O(1)。

该设计显著提高了选择效率，但优先级调整中大量基于经验的公式较难理解和维护，而且在一些典型测试下仍会对交互式进程产生不佳效果。

SD 与 RSDL 调度器

flowchart TD
    A[进入调度] --> B{是否有实时进程}
    B -- 是 --> R[实时进程按 FIFO / RR 调度]
    B -- 否 --> C[从 active 数组中选最高优先级非空队列]
    C --> D[取队首任务运行]
    
    D --> E[时间片递减]
    E --> F{时间片是否耗尽}
    F -- 否 --> D
    F -- 是 --> G{是否已降到最低优先级}
    
    G -- 否 --> H[下降到下一层更低优先级队列]
    H --> A
    
    G -- 是 --> I[回到接近初始优先级的位置]
    I --> J[重新分配相应倍数时间片]
    J --> A

Con Kolivas 提出的 SD（Staircase Scheduler）抛弃了 O(1) 调度器复杂的动态优先级计算，更强调“完全公平”的思路。它仍为每个优先级维护队列并放在 active 数组中，但当进程用完时间片后，不再进入 expired 数组，而是被降入更低一级的优先级队列，相当于沿着“楼梯”逐步下行。达到最低级后，再次用完时间片的任务会回到其初始优先级之后的某一级，并获得相应倍数的时间片。这样做使高优先级任务最终也会下沉，与低优先级任务竞争，从而避免饥饿。交互式任务若在高台阶处睡眠，唤醒时仍保留相对有利位置，因此能更快获得响应。实时进程仍继续使用 FIFO 或 RR。

flowchart TD
    A[进入调度] --> B{是否有实时进程}
    B -- 是 --> R[实时进程按 FIFO / RR 调度]
    B -- 否 --> C[从 active 中最高优先级组选择任务]
    C --> D[任务运行]

    D --> E{个体配额 Tp 是否用完}
    E -- 是 --> F[minor rotation: 任务下降到下一优先级组]
    E -- 否 --> G{组配额 Tg 是否用完}
    
    G -- 是 --> H[整个优先级组被强制下降]
    G -- 否 --> I{真实 time_slice 是否耗尽}
    
    I -- 是 --> J[放入 expired 中的初始优先级队列]
    I -- 否 --> K[继续在当前 active 中参与调度]

    F --> L{active 是否为空 或 所有任务都降到最低级}
    H --> L
    J --> L
    K --> A

    L -- 否 --> A
    L -- 是 --> M[major rotation: 交换 active 与 expired]
    M --> N[所有任务恢复到初始优先级结构]
    N --> A

**RSDL（Rotating Staircase Deadline Scheduler）**是在 SD 基础上的改进。它重新引入 expired 数组，并为每个优先级分配“组时间配额” Tg​，同一优先级中的每个进程再拥有相同的“个体时间配额” Tp​。

  • 当进程用完自身的 Tp​ 时，会下降到下一优先级组，这一过程称为 minor rotation。
  • 当某高优先级组用完自身的 Tg​ 时，即使组内仍有进程尚未用完 Tp​，整个组也会被强制下降，使低优先级任务能在可预计的未来获得调度机会。
  • 若进程用完自身真实时间片 time_slice，则放入 expired 数组的初始优先级队列中；
  • 当 active 为空或所有进程都降到最低级时，触发 major rotation，交换 active 与 expired，所有进程恢复初始状态。

RSDL 因而比 SD 更强调“等待时间可预测”。

CFS 调度器

**CFS（Completely Fair Scheduler）**是 Linux 普通进程调度的关键里程碑。它吸收了 SD 与 RSDL 的“完全公平”思想，不再跟踪进程的睡眠时间，也不再显式地区分交互式与批处理进程，而是把所有普通进程统一映射到“按权重分配处理器份额”的模型上。

其核心概念是虚拟运行时间 vruntime：每个进程都记录自己“按公平尺度已经运行了多久”，调度器总是选择 vruntime 最小的进程运行。vruntime 与实际运行时间成正比，与进程权重成反比，因此优先级越高、权重越大的进程，在相同实际运行时间下增加的 vruntime 越慢。

设调度周期为 Tsched​，进程 i 的权重为 wi​，系统中所有可运行进程权重之和为 ∑j​wj​，则分配给进程 i 的运行时间为

ti​=Tsched​⋅∑j​wj​wi​​

其中，调度周期表示把所有处于 TASK_RUNNING 状态的进程都调度一遍的大致时间。

CFS 定义虚拟运行时间为

vruntimei​=ti​⋅wi​NICE_0_LOAD​

其中，NICE_0_LOAD 是 nice=0 时的基准权重常量。把上式与时间份额公式联立，可得

vruntimei​=(Tsched​⋅∑j​wj​wi​​)⋅wi​NICE_0_LOAD​=Tsched​⋅∑j​wj​NICE_0_LOAD​

由此可见，从 vruntime 的角度，所有进程在一个调度周期内前进的“虚拟距离”是相同的，这正是“完全公平”的数学表达。

Intuition

vruntime 到底是什么？你可以把它想象成“公平赛道上的里程表”。优先级高的进程权重更大，它真实跑 1 公里，vruntime 只增加 0.5 公里；优先级低的进程真实跑 1 公里，vruntime 增加 2 公里。调度器永远选 vruntime 最小的进程，意思就是：谁在这条公平赛道上“落后”最多，谁就先去跑。所以高优先级进程可以跑得更久才轮到别人，但大家在 vruntime 这条线上最终是齐头并进的。

为高效维护“谁的 vruntime 最小”，CFS 用 vruntime 作为键值构造红黑树。树中最左侧结点就是当前 vruntime 最小、最应获得 CPU 的进程。调度器在选中一个结点运行时，会把它从树中移出；进程重新变回就绪态时再插回树中。通常，一个刚运行过的进程因 vruntime 变大，会落到树的更右侧位置。这样，整棵树中的结点会围绕公平边界不断流动。

CFS 的重要数据结构包括 task_struct、sched_entity、sched_class 与 cfs_rq：

  • task_struct：Linux 中对任务的完整描述，记录状态、内核栈、进程标识、优先级等信息。
  • sched_entity：描述可参与调度的实体，包含权重、红黑树结点等。
  • sched_class：调度类的模块化接口，不同调度类通过实现统一函数集合接入内核调度框架。
  • cfs_rq：某个 CFS 运行队列的表示，其中保存了红黑树等与公平调度直接相关的数据。

min_vruntime 是理解 CFS 的关键变量。它是每个 cfs_rq 上维护的一个下界量，通常不大于该队列内所有就绪进程的最小 vruntime，反映当前队列整体虚拟运行时间的前沿。新进程创建时，并不会把 vruntime 直接设为 0，否则会因“虚拟运行时间太少”而无条件立即压制老进程。更合理的做法，是以当前队列的 min_vruntime 为基准，再加上一个与单轮应得份额相对应的虚拟增量，相当于认为它已经在本轮公平周期中“占过位”。在特定条件下，系统还可能交换父子进程的 vruntime，以实现“子进程优先运行”等策略。

进程唤醒时，CFS 也不会简单保留原值不动。若一个进程睡眠较久，系统会用 min_vruntime 减去一个补偿值来修正其 vruntime，让它醒来后有机会较快重新获得 CPU；但补偿并非无限放大，为防止进程通过频繁短睡眠“刷优先级”，系统最终会在补偿结果与原 vruntime 之间取更大的那个，然后再把该进程插入红黑树，并判断其是否足以触发抢占。

从运行过程看，CFS 的几个典型情景如下：

  • 时钟中断到来：内核先把当前进程这一小段实际运行时间折算进 vruntime，再判断其本轮是否已用完应得时间份额；若超过分配值，则标记为需要调度。
  • 主动调度发生：当前进程先被更新信息并插回红黑树，然后系统选择新的运行实体并将其从树中移出。
  • 创建新进程：先计算其初始 vruntime，再插树并判断是否能够抢占当前进程。
  • 唤醒阻塞进程：先补偿其 vruntime，再插树并根据差距与调度粒度决定是否抢占。

CFS 中还维护 last 与 next 两个指针，用来在主动调度路径中对最近被换下和即将换上的进程做适度偏向；只有当新进程的 vruntime 小于当前进程达到一定粒度阈值时，wakeup_preempt_entity 才会判定可以抢占。

    C["阻塞态"] -->|"唤醒后更新 vruntime 并插入红黑树"| B

CFS 之后，Linux 又继续向 EEVDF 方向演化。课件给出的演化链表明，Linux 6.11 已使用 EEVDF 作为更新的普通调度器方向。其意义在于，Linux 调度系统仍在持续改进“公平性、延迟与交互性”之间的平衡，只是本讲的理论主线仍以 CFS 为中心。

总结

Summary

处理器调度的核心，不是单纯追求“更快”，而是在给定负载类型与系统目标下，决定如何在多个竞争执行实体之间分配有限的处理器资源。

我们这一讲走过的路程很长：

  • 批处理系统重视周转时间与吞吐量，FCFS、SJF、SRTN、HRRN 各有优劣；
  • 交互式系统重视响应与均衡，RR、Virtual RR、MLFQ 是典型代表；
  • 多处理器系统必须同时处理亲和性、迁移成本与负载均衡，作业偷取是重要思想；
  • 实时系统重视截止时间与可预测性，RM 和 EDF 是两大经典算法；
  • Windows 用动态优先级 + 时间配额来调度线程，饥饿处理靠平衡集管理器；
  • Linux 从 2.4 的 O(n) 遍历，到 O(1) 的双数组，再到 SD/RSDL 的楼梯思想，最终演进为 CFS 的 vruntime + 红黑树模型。

这些经典算法、现代操作系统的实现细节，以及多核场景下的亲和与迁移机制，构成了操作系统调度设计的主要知识主线。考试加油！
