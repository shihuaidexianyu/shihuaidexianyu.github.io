操作系统概述

操作系统概述

Apr 23, 202615 min read

核心向导

这节课是操作系统的第一课（Overview），主要回答了三个终极问题：操作系统究竟是什么？它有什么独特的性格（特征）？以及它是怎么一步步演化成今天这个样子的？

学习这部分内容，不要死记硬背概念，而是要带入“管理者”的视角去理解——如果你手里只有一堆冰冷的 CPU、内存和磁盘，你要怎么设计一套规章制度，让不同的程序不仅能“同时”运行，还不互相打架，并且让最终用户觉得既快又好用？

一、 操作系统是什么？

传统书本上会说，操作系统（Operating System, OS）是计算机系统中的一个系统软件，是一些程序模块的集合。它的任务是有效、合理地组织和管理软硬件资源，向用户提供各种服务功能。

但在我看来，操作系统本质上就是一个“超级管家”兼“翻译官”。

它对下接管所有复杂的物理硬件（资源管理器），对上给用户和应用程序提供简单易用的接口（API）。如果没有操作系统，我们写一个简单的“在屏幕上打印 Hello World”的程序，可能都需要去查阅显卡的物理寄存器地址并手动发送电信号。

从三个不同的视角去理解它：

  1. 资源管理的观点（自底向上）

    • 逻辑：计算机里有硬件（CPU、内存、I/O 设备、磁盘、网络接口）和软件（文件、数据）两类资源。操作系统的终极目标是实现资源共享、提高资源利用率。

    • 方法：它通过数据结构和算法来“记账”（谁在用？用了多久？空闲多少？），并采用时间复用（比如大家轮流用 CPU）和空间复用（比如把内存划分给不同程序）来分配资源。通常我们更倾向于动态分配策略，因为更灵活高效。

  2. 进程的观点（动态运行视角）

    • 逻辑：从系统跑起来的样子看，操作系统就是由一些可以同时、独立运行的进程，以及一个负责协调它们的核心构成的。

    • 直觉：你可以把“程序”想象成一张写满步骤的菜谱（静态的），而“进程”就是厨师正在照着菜谱做菜的整个动态过程（有生命的，有诞生和消亡）。

  3. 虚机器观点（内部结构视角）

    • 逻辑：操作系统是由一层一层堆叠起来的。底层是最原始的物理硬件，每一层都在上一层的基础上增加新功能，向上一层提供支持。

    • 结论：最终，操作系统把物理资源（处理器、内存、磁盘）转换成了一个更通用、更强大、更易用的虚拟形式（虚拟机），并通过标准库（APIs）让应用程序去调用。

二、 操作系统的四大基本特征

这四个特征是 OS 的灵魂，特别是“并发”和“共享”，它们是操作系统存在的基础。

直觉理解

想象一个厨房。并发是厨师一会儿切菜、一会儿看锅，两件事交替做；共享是几个厨师合用一个烤箱；虚拟是每个厨师都觉得自己有一个专属的厨房（其实是分时段使用的）；随机则是你永远不知道下一个进来的点单是炒饭还是炖汤。

1. 并发 (Concurrency)

  • 含义：处理多个同时性活动的能力。宏观上看，几个程序好像在“同时”执行；但微观上看（在单核 CPU 下），任何时刻其实只有一个程序在执行，它们只是在 CPU 上轮流飞快地切换。

  • 容易混淆的边界：并发 = 并行 (Parallel)。并发是“交替做多件事”（制造同时发生的幻觉），并行是拥有多个物理核心“真正同时做多件事”。

  • 带来的问题：频繁的活动切换、进程间的隔离保护、以及它们之间的同步（比如谁先用完某个变量）。

2. 共享 (Sharing)

  • 含义：操作系统和多个用户程序共同使用系统中的有限资源。

  • 分类：

    • 互斥共享：一段时间内只允许一个进程使用。比如打印机，张三的文档打印完之前，李四不能插队。

    • 同时共享：宏观上允许多个进程同时访问。比如可重入的只读代码、磁盘文件。

3. 虚拟 (Virtualization)

  • 含义：把一个物理实体映射为若干个逻辑实体。这是提高资源利用率的魔法。

  • 应用场景：

    • CPU 虚拟化：每个进程都以为自己独占了一个“虚处理器”。

    • 内存虚拟化：每个进程都有自己独立的地址空间（代码段、数据段、堆栈）。

    • 显示设备虚拟化：我们在屏幕上看到的多窗口。

4. 随机性 / 不确定性 (Asynchrony)

  • 含义：由于并发环境下的“走走停停”，多个进程的执行速度和推进顺序是完全不可预知的。操作系统必须随时对这些不可预测的事件作出响应。

  • 痛点：这导致在多线程/多进程编程中，复现某个特定时刻的状态（或者抓 bug）变得极其困难。

三、 操作系统的发展历程与启示

技术的发展总是伴随着“硬件与人工成本比率”的变迁。一个很有意思的现象是：技术的变化会导致某些思想过时，但另一种技术的出现又会使它们复活。 例如早期的集中式计算服务（Multics），在今天以“云计算”的形式强势回归。

  • 失败与成功的典型案例：

    • IBM OS/360 & Multics：当年试图打造大而全的系统，结果遭遇了软件工程上的重大灾难（发布时带着几千个 bug，《人月神话》就是这么来的）。

    • UNIX 的诞生：Ken Thompson 和 Dennis Ritchie 因为嫌弃 Multics 太庞大，为了在一台闲置机器上玩星际探险游戏，用最精简优雅的理念写出了 UNIX。这深刻影响了后世所有系统（获得图灵奖）。

四、 操作系统的分类与核心机制

根据应用场景和追求目标的不同，操作系统演化出了不同的形态。

1. 批处理操作系统 (Batch OS)

用户把作业交给操作员，操作员把它们组成“一批”，系统自动依次执行。核心诉求是让昂贵的机器一刻也不要停下来。

  • 单道批处理 (Uniprogramming)：内存里只有一个程序。遇到 I/O 操作（比如读写磁盘）时，CPU 只能傻等。

  • 多道批处理 (Multiprogramming)：内存里装多个程序。当程序 A 遇到 I/O 阻塞时，OS 迅速把 CPU 切给程序 B 去计算。

    • 数据证明威力：在一组包含计算密集型和 I/O 密集型作业的对比中，单道系统 CPU 利用率仅为 22%，耗时 30 分钟；引入多道程序设计后，CPU 利用率飙升至 43%，整体耗时减半至 15 分钟，吞吐量翻倍。

核心机制：SPOOLing 技术 (假脱机技术)

早期批处理最大的瓶颈是：CPU 速度极快，而读卡机/打印机极慢。SPOOLing（Simultaneous Peripheral Operation On-Line）完美解决了这个速度不匹配的矛盾。

工作原理：

它在高速磁盘上开辟了两个缓冲区——“输入井”和“输出井”。

  1. 慢慢吞吞的读卡机把数据读出来，不用直接给 CPU，而是先扔进磁盘的输入井。

  2. CPU 处理时，直接从高速的输入井读取数据，算完后把结果扔进磁盘的输出井。

  3. 慢慢吞吞的打印机再按照自己的节奏，从输出井里取数据去打印。

意义：SPOOLing 实际上是将独占设备改造为了共享设备。几个程序同时请求打印，系统只是在磁盘队列里记了账，对每个程序来说，它都觉得“设备立刻就为我服务了”（虚拟设备功能）。今天我们电脑里的打印机队列，底层依然是这个思想。

2. 分时操作系统 (Time-Sharing OS)

  • 工作机制：把 CPU 的执行时间切成极短的片段（称为时间片，Time slice）。系统以时间片为单位，轮流为每个终端用户服务。因为切换速度超过了人类的感知极限，利用了我们的“错觉”。

  • 追求目标：及时响应（响应时间是指从终端发出命令到系统给予回答的时间）。

  • 演进：现代的“通用操作系统”通常是分时与批处理的结合——优先保证前台交互的响应时间，当 CPU 闲下来时再去处理后台时间性要求不强的批处理作业。

3. 实时操作系统 (Real-Time OS)

关键参数是时间。它必须在严格规定的时间范围内对外部事件作出响应。

  • 硬实时系统：错过截止时间意味着灾难。例如：汽车防抱死刹车系统（ABS）、导弹控制系统、工业机械臂。

  • 软实时系统：允许偶尔违反时间限制，顶多体验变差。例如：在线视频流媒体（卡顿一下不至于出大麻烦）。

4. 其他类型

  • 网络与分布式 OS：分布式更强调“透明性”和“自治性”，多个节点平等协作，用户感觉不到资源是分散在不同机器上的。

  • 云计算 OS：管理后台海量服务器的运营系统，本质上是把整个数据中心的硬件逻辑上整合成“一台超级服务器”，提供按需取用的虚拟资源。

  • 嵌入式操作系统：资源受限，生存在特定设备中。其中最极端的代表是智能卡操作系统（如带芯片的银行卡）。智能卡极其缺乏电能和存储（通常 RAM 只有 3K，ROM 不到 100K），往往内置一个轻量级的 Java 虚拟机解释器来运行安全小程序，通过“命令-响应对”机制与外部读卡器进行异步通信。
