xv6 OS Lab Part 1 实验报告

Note

提交仓库链接时需要以 .git 结尾，例如   https://gitlab.eduxiji.net/xxxxxxxxxxxxxxx/os_2026_spring.git

一、实验目标

Part 1 的目标不是新写一批用户程序，而是在现有 xv6-k210 框架上补齐一条可以被预编译测试程序直接调用的系统调用链路。  

本部分需要重点完成以下内容：

  1. 对齐测试程序所使用的系统调用号。
  2. 保证系统能够正确启动到首个用户进程。
  3. 让 init 依次运行 Part 1 的五个测试点。
  4. 让测试输出通过 write 正常打印到 QEMU 标准输出。
  5. 在全部测试结束后通过 shutdown 主动关机，避免平台超时。

Part 1 对应的五个测试点为：

  • getcwd
  • write
  • getpid
  • times
  • uname

二、整体思路

本实验的核心不是孤立地修几个系统调用，而是打通下面这条完整路径：

QEMU -> RustSBI -> kernel -> userinit -> initcode -> init -> 测试程序 -> shutdown

具体来说，整个实现可以拆成三部分：

  1. 修改系统调用 ABI，使预编译测试程序发起的调用号可以被内核正确识别。
  2. 修正首进程启动链，使本地运行和平台评测都能进入正确的 init 逻辑。
  3. 实现或适配 Part 1 所需的 shutdown、getcwd、times、uname 等系统调用。

三、启动链与 initcode

1. initcode 的作用

系统刚启动时，内核还没有现成的用户进程，因此不能直接“从用户态再启动一个用户程序”。  

解决方法是：由内核先人工构造第一个用户进程，并把一小段用户态机器码直接复制到它的地址空间中，这段代码就是 initcode。

它的职责很简单：

  • 本地模式下，执行最小的 exec("/init") 跳板；
  • 平台模式下，直接把完整的 init 逻辑嵌入内核，避免依赖镜像中必须存在 /init。

2. 本地与平台的两套生成方式

本实验中，Makefile 区分了两种模式。

本地模式

本地使用 bootstrap_initcode：

  • 由 xv6-user/initcode.S 生成 xv6-user/initcode
  • 再把纯二进制转成 kernel/include/initcode.h
  • 由 proc.c 中的 initcode[] 引入

这条路径的特点是：

  • initcode 很小
  • 首进程只负责 exec("/init")
  • /init 由文件系统镜像中的用户程序提供

对应规则位于Makefile 中：

  • bootstrap_initcode
  • $U/initcode
  • $K/include/initcode.h

平台模式

平台评测执行 make all，使用 dump_initcode：

  • 直接把 xv6-user/init.c 编译成完整 _init
  • 再提取为纯二进制
  • 最后转成 kernel/include/initcode.h

这样做的原因是平台镜像与本地镜像组织方式不同，不能完全依赖标准的 exec("/init") 路径，因此将完整 init 逻辑直接嵌入内核更稳妥。

当前 all 目标会执行：

  1. dump_initcode
  2. build
  3. 导出 kernel-qemu
  4. 导出 sbi-qemu

这正是平台要求的提交流程。

四、init 的设计

本实验中的xv6-user/init.c不再负责启动 shell，而是作为一个“测试调度器”。

运行流程如下：

  1. 通过 dev(O_RDWR, CONSOLE, 0) 打开控制台。
  2. 通过两次 dup(0) 建立标准输出和标准错误输出。
  3. 对每个测试程序依次执行：

   - fork()

   - 子进程 exec(tests[i], argv)

   - 父进程 wait()

  1. 全部完成后调用 shutdown()

这样的设计有两个优点：

  • 测试严格串行，输出顺序稳定；
  • 系统在最后主动退出，不会停留在 QEMU 中导致评测超时。

六、shutdown 的实现

shutdown 虽然不属于五个主要测试点之一，但它决定了整个评测是否能正常结束。

如果缺少这条调用链，常见现象有两种：

  • init 返回后触发 panic: init exiting
  • 测试全部执行完成，但系统仍停在 QEMU 中，最终被平台判定为“运行时间过长”

本实验中，shutdown 的调用链如下：

  1. 在 xv6-user/user.h 中声明 shutdown
  2. 在 xv6-user/usys.pl 中生成用户态存根
  3. 在 kernel/include/sysnum.h中定义 SYS_shutdown
  4. 在 kernel/syscall.c中完成分发表映射
  5. 在 kernel/sysproc.c中实现 sys_shutdown
  6. 最终调用 kernel/include/sbi.h中的 sbi_shutdown()

七、getcwd 的实现

getcwd 的重点不是“从零写路径算法”，而是把已有逻辑改到测试接口要求的语义。

当前实现位于 kernel/sysfile.c。

实现步骤如下：

  1. 用 argaddr(0, &addr) 读取用户缓冲区地址。
  2. 用 argint(1, &size) 读取缓冲区大小。
  3. 从 myproc()->cwd 出发沿父目录回溯，拼出绝对路径。
  4. 检查目标字符串长度是否超出用户缓冲区。
  5. 用 copyout2() 将结果写回用户态。
  6. 返回用户传入的缓冲区地址 addr。

这里最关键的是两点：

  • 返回值必须是 buf，而不是简单返回 0
  • 必须做长度检查，避免写回越界

十、uname 的实现

uname 的实现位于 kernel/sysproc.c。

实现流程如下：

  1. 读取用户缓冲区地址。
  2. 构造 struct utsname。
  3. 填入一组稳定字符串。
  4. 用 copyout2() 回写到用户态。

当前填写的内容包括：

  • sysname = "xv6"
  • nodename = "xv6"
  • release = "2026"
  • version = "os-lab"
  • machine = "riscv64"
  • domainname = "localdomain"

这些值本身不要求与真实 Linux 完全一致，但必须满足：

  • 结构完整
  • 字符串稳定
  • 能安全返回到用户空间

十一、为什么要使用 copyout2

getcwd、times、uname 这类系统调用都会把结果写回用户提供的缓冲区。  

这里最大的风险是：用户传入的是用户虚拟地址，内核不能把它当普通指针直接写。

因此本实验统一采用 copyout2()：

  • 地址非法时能安全失败
  • 地址越界时不会直接破坏内核
  • 行为比直接解引用用户指针更可控

这也是系统调用实现中最基础、最重要的边界意识之一。

十二、Makefile 的职责

本实验中 Makefile 的职责不只是编译内核，还要负责把“正确的首进程代码”嵌入内核。

它主要承担四类任务：

  1. 编译内核和用户态程序。
  2. 生成 initcode.h。
  3. 在本地模式和平台模式下切换不同的 initcode 来源。
  4. 为平台导出 kernel-qemu 和 sbi-qemu。

当前采用的规则是：

  • make local

  - 先 bootstrap_initcode

  - 再 run

  • make all

  - 先 dump_initcode

  - 再 build

  - 最后导出平台需要的两个 ELF 文件

这也是本次实验中最关键的工程化修改之一。

1. make build

make build 是最基础的编译命令，它完成两件事：

  1. 生成内核 target/kernel
  2. 编译全部用户程序 xv6-user/_*

其中内核目标依赖于：

  • 内核目标文件 $(OBJS)
  • 链接脚本 linker/qemu.ld
  • kernel/include/initcode.h

因此，build 的含义不是“仅仅把 .c 编译成 .o”，而是：

  • 先确保首进程代码已经被转换成 initcode.h
  • 再把它与内核其他目标文件一起链接成最终内核 ELF

如果只执行 make build，可以得到一个可运行的内核和用户程序集合，但不会自动创建或更新磁盘镜像。

2. make run

make run 会执行：

  1. build
  2. fs
  3. 启动 QEMU

也就是说，它是一条“编译 + 打包镜像 + 运行”的完整本地测试路径。

当前 run 规则中，QEMU 会使用：

  • target/kernel 作为内核
  • bootloader/SBI/sbi-qemu 作为 SBI
  • fs.img 作为块设备镜像

因此，make run 适合在已经确认 initcode.h 正确的前提下直接进行本地验证。

3. make local

make local 是本地模式下最重要的命令，它会先执行：

  1. bootstrap_initcode
  2. run

这里的关键在于 bootstrap_initcode。

它的作用是：

  • 使用 xv6-user/initcode.S 生成一个很小的 xv6-user/initcode
  • 再把这个二进制转成 kernel/include/initcode.h

因此 make local 对应的是：

  • 本地镜像里存在真正的 /init
  • 首进程只需执行一小段跳板代码
  • 由这段跳板代码进一步 exec("/init")

换句话说，make local 代表“本地调试用的标准启动路径”。

4. make all

make all 是平台评测真正会执行的命令，因此它和 make local 的区别必须特别清楚。

它会执行：

  1. dump_initcode
  2. build
  3. 复制 target/kernel 为 kernel-qemu
  4. 复制 bootloader/SBI/sbi-qemu 为 sbi-qemu

这里最重要的是 dump_initcode。

dump_initcode 不再使用短小的 initcode.S，而是：

  1. 直接编译 xv6-user/init.c
  2. 链接生成完整的 _init
  3. 把 _init 提取为纯二进制
  4. 再转成 kernel/include/initcode.h

因此 make all 对应的是：

  • 平台模式
  • 内核首进程直接执行完整的 init 逻辑
  • 不依赖平台镜像中一定存在一个可供标准路径加载的 /init

这正是本实验中“本地能跑、平台也能跑”的关键分流点。

5. make fs

make fs 的职责是生成或更新 fs.img，并把用户程序和测试程序打包进镜像。

它主要完成：

  1. 如果 fs.img 不存在，则创建并格式化一个 FAT32 镜像
  2. 挂载 fs.img
  3. 把 xv6-user/_* 复制到镜像根目录和 /bin
  4. 把 riscv64/ 目录中的预编译测试程序复制到镜像根目录
  5. 卸载镜像

这一步非常重要，因为本地测试时真正运行的测试程序并不是现编现跑，而是先被复制进镜像，再由系统启动后从镜像中加载。

6. make bootstrap_initcode

这个命令只服务于本地模式。

它做的事情非常直接：

也就是把最小的汇编跳板代码转成 kernel/include/initcode.h，供 proc.c 中的 initcode[] 使用。

它的特点是：

  • 生成内容小
  • 逻辑简单
  • 只负责 exec("/init")

7. make dump_initcode

这个命令只服务于平台模式。

它做的事情是：

与 bootstrap_initcode 相比，它不是把一段短汇编嵌入内核，而是把完整的 init 程序嵌入内核。

这也是为什么它只在 make all 中使用，而不在 make local 中使用。

8. make clean

make clean 用于清理编译中间产物和最终产物。

它会删除：

  • *.o、*.d、*.asm、*.sym
  • target/*
  • xv6-user/initcode
  • xv6-user/initcode.out
  • xv6-user/init.S
  • kernel/kernel
  • kernel/include/initcode.h
  • kernel-qemu
  • sbi-qemu
  • xv6-user/usys.S
  • 所有 UPROGS

之所以连 kernel/include/initcode.h 也一起删掉，是因为这个文件不是手写源文件，而是构建过程中自动生成的产物。下次执行 make local 或 make all 时，应重新按对应模式生成。

9. 各命令之间的关系

为了避免混淆，可以把当前 Makefile 中的几个主要命令理解成下面的层次：

  • make build

  - 负责编译

  • make fs

  - 负责打包镜像

  • make run

  - 负责 build + fs + 启动 QEMU

  • make local

  - 负责“本地版 initcode”生成后再 run

  • make all

  - 负责“平台版 initcode”生成后再 build，并导出评测所需 ELF

因此：

  • 想本地调试，优先使用 make local
  • 想准备提交产物，优先使用 make all
  • 想只重新编译，不启动 QEMU，可以使用 make build
  • 想只更新镜像，可以使用 make fs

十三、实验中遇到的问题

本次 Part 1 的问题主要集中在以下几类。

1. 系统调用号不一致

如果 sysnum.h 中的调用号与测试程序不一致，系统一启动就会出现 unknown sys call。  

因此必须优先检查调用号表和系统调用分发表是否同步。

2. initcode 与启动链不匹配

如果本地和平台使用同一套首进程启动方式，往往会出现：

  • 本地能跑
  • 平台超时

原因通常不是某个系统调用本身错误，而是平台启动路径和本地镜像组织方式不同。

3. 测试结束后没有关机

如果 /init 正常执行完五个测试，但没有调用 shutdown()，平台通常会表现为：

因此 shutdown 不是“附加功能”，而是评测闭环的一部分。

十四、最终结果

经过本次修改后，Part 1 的实现形成了如下稳定流程：

  1. Makefile 根据场景生成对应的 initcode.h
  2. 内核通过 userinit() 创建首进程
  3. init 依次执行 getcwd、write、getpid、times、uname
  4. 测试输出通过 write 打印到 QEMU 标准输出
  5. 全部结束后由 shutdown 触发关机

从实现角度看，本实验真正完成的是一条“从内核启动到评测结束”的完整链路，而不是几个孤立函数的堆砌。

十五、总结

通过 Part 1，我主要完成了以下工作：

  1. 理清了首个用户进程的启动机制，理解了 initcode 在系统自举中的作用。
  2. 完成了系统调用号与测试 ABI 的对齐。
  3. 实现了 shutdown，补齐了系统退出闭环。
  4. 适配了 getcwd 的参数与返回值语义。
  5. 实现了 times 与 uname 的基本版本，并统一采用 copyout2() 回写用户态数据。
  6. 修改 Makefile，区分本地和平台两套 initcode 生成路径。

Part 1 看似只涉及五个测试点，但实际上覆盖了：

  • 内核启动
  • 首进程构造
  • 系统调用 ABI
  • 用户态回写
  • 评测流程适配
