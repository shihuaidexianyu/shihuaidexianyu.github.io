操作系统运行环境和运行机制

Summary

这一章讲的是操作系统怎么”罩”住所有程序：用户程序怎么安全地进入内核、内核怎么处理突发事件（中断/异常/陷入）、以及系统调用这条最常用的高速公路是怎么修的。学完这一章，你应该能回答：用户态和内核态切换时硬件做了什么？int 0x80 之后 CPU 去了哪里？为什么中断处理要拆成上半部和下半部？

咱们这章的核心线索就四个字：异常控制流（ECF）。程序并不是永远乖乖地一条指令接一条指令往下跑的——总有些”意外”会把控制流拽走：可能是你按了个键（中断）、可能是程序除零了（异常）、也可能是程序主动喊操作系统帮忙（陷入/系统调用）。操作系统的中枢神经系统，就是建立在这些”异常”之上的。保护、并发、设备管理、调度，全都靠它。

一、操作系统运行环境

Summary

先搭舞台：CPU、寄存器、特权级、中断/异常，这些硬件机制是操作系统能”说了算”的底气。这一节告诉你，用户程序和内核之间那道”防火墙”是怎么砌起来的。

1. 运行环境与异常控制流

操作系统能跑起来，靠的是处理器、内存、I/O 子系统、时钟，以及一系列配套的体系结构机制。我们平时写的应用程序在用户态（低特权级）执行，操作系统则在内核态（高特权级）执行受保护的代码。硬件负责提供模式切换、现场保存、向量分派、特权检查这些基础设施，软件负责解释事件、调度资源、维护抽象。

Intuition

你可以把用户态想象成”租客”，内核态是”房东”。租客不能随意拆承重墙（改页表、关中断），但可以通过正规的报修渠道（系统调用）让房东来帮忙。房东手里有钥匙，能进任何房间。

异常控制流（ECF）描述的就是控制权在不同实体之间跳来跳去的过程。普通控制流就是顺序执行、分支、函数调用与返回；而 ECF 则是被外部事件、故障、软件陷入或调度打断后的跳转。应用程序和操作系统交互，本质上不是普通的函数调用，而是通过受硬件保护的异常控制流进入内核的。

flowchart TD
    U[用户程序顺序执行] -->|函数调用/返回| U
    U -->|系统调用/陷入| K1[内核系统调用入口]
    U -->|故障/异常| K2[异常处理入口]
    D[外部设备/时钟] -->|中断请求| K3[中断处理入口]

    K1 --> S[内核服务例程]
    K2 --> H[故障修复/终止/信号]
    K3 --> I[驱动程序/唤醒/记账]

    S --> R[返回用户态]
    H --> R
    I --> Q{是否需要重新调度}
    Q -->|否| R
    Q -->|是| SCH[调度器选择新进程]
    SCH --> R

ECF 的意义可以归纳为三层：

  • 系统调用：通过受控的陷入路径，把用户程序连到内核服务。
  • 中断：让 CPU 和设备能并行干活，设备完事后再异步通知 CPU。
  • 异常：把硬件错误、非法访问、缺页、断点调试等情况统一纳入内核的管理范围。

Note

如果没有 ECF，操作系统就永远拿不到 CPU 的控制权——用户程序一旦跑起来，除非自己自愿放弃，否则系统什么事都干不了。中断和时钟的存在，才让操作系统有了”强制介入”的能力。

2. 处理器、寄存器与保护

处理器由运算器、控制器、寄存器组和缓存构成。从 CPU 的角度看，一个程序无非就是代码、数据、栈、堆，再加上一堆寄存器状态。寄存器是执行状态最紧凑的表示，也是发生异常或切换时必须优先保存和恢复的对象。

寄存器可以分成两大类：

第一类：用户可见寄存器

  • 数据寄存器：通用计算、参数传递。
  • 地址寄存器：地址计算、栈指针（ESP/RSP）、基址寄存器等。
  • 条件码寄存器：保存运算结果的状态位，比如零标志 ZF、进位标志 CF、符号标志 SF、溢出标志 OF。

第二类：控制与状态寄存器 这些寄存器控制处理器的工作方式，保存全局执行状态。和操作系统关系最密切的包括：

  • PC/EIP/RIP：程序计数器，指向下一条要执行的指令。
  • IR：指令寄存器，保存当前正在执行的指令。
  • PSW/EFLAGS/RFLAGS：程序状态字，包含各种状态位和控制位。
  • IDTR/GDTR/TR：中断描述符表寄存器、全局描述符表寄存器、任务寄存器。
  • CR0–CR4 等控制寄存器：页表基址、地址转换、保护开启等。

在 x86 上，EFLAGS 里有几个位是操作系统必须熟悉的：

  • IF：中断允许位。IF=1 时 CPU 可以响应可屏蔽中断；IF=0 时屏蔽。
  • TF：单步跟踪位，调试器靠它实现单步执行。
  • IOPL：I/O 特权级，限制某些 I/O 指令能在什么特权级执行。
  • CF/ZF/SF/OF：反映算术和逻辑运算结果。

Trap

很多同学会把 IF 和”中断控制器是否屏蔽某个 IRQ”搞混。IF 是 CPU 层面的总开关，关 IF 等于 CPU”装聋”；而 PIC/APIC 里的屏蔽位是针对具体中断源的。两者都要开，中断才能真正送达处理程序。

为什么要搞”保护”？ 因为多个程序共享 CPU、内存和设备，如果没有硬件隔离，一个用户程序就能随意篡改系统状态或偷看别的进程数据。所以处理器必须提供不同的运行模式和指令权限。

3. 处理器模式、特权级与特权指令

特权指令只能在高特权级下执行，比如访问设备、修改页表、开关中断、装载描述符表、切换地址空间这些”破坏性操作”，普通应用程序是碰不得的。

现代处理器通常提供 2~4 级运行模式。操作系统最经典的抽象是用户态和内核态：

  • 内核态：运行操作系统核心代码，权限最高。
  • 用户态：运行普通应用程序，只能执行受限指令集。

x86 用**特权环（Ring）**实现了四级特权：Ring 0 ~ Ring 3，特权从高到低。大多数操作系统只用 Ring 0（内核）和 Ring 3（用户）。RISC-V 则定义了机器模式 M、监管模式 S 和用户模式 U；实际操作系统通常在 S 模式跑内核，在 U 模式跑应用，M 模式留给启动固件和最低层控制。

flowchart LR
    U[用户态 / Ring 3 / U 模式]
    K[内核态 / Ring 0 / S 模式]
    M[机器模式 / M 模式]

    U -->|系统调用、异常、外部中断| K
    K -->|异常返回指令| U
    M -->|启动后降级| K

Intuition

你可以把 Ring 0 想象成”机长驾驶舱”，Ring 3 是”经济舱乘客”。乘客按服务铃（系统调用）可以请机长帮忙，但绝不能自己冲进驾驶舱开飞机。硬件就是那个 enforce 规则的”空警”。

用户态进入内核态的合法路径只有三条：中断、异常、陷入。普通用户程序不能直接跳转到内核代码段来提权。内核态返回用户态，则要靠异常返回指令恢复之前保存的状态，比如 x86 的 iret、RISC-V 的 sret、快速系统调用的 sysexit/sysret 等。

4. 中断、异常与陷入

4.1 基本概念

中断和异常都是处理器对系统事件的反应。共同点是：CPU 先暂停当前执行流，保存必要现场，把控制权交给对应的处理程序，处理完后再恢复原流程或切换到新流程。

但它们的来源和性质不同：

  • 中断：来自处理器外部，比如 I/O 设备、时钟、本地 APIC、处理器间中断等。
  • 异常：来自当前正在执行的指令本身，比如除零、非法操作码、访问越界、缺页、断点、软件陷入。

从时间关系看：

  • 中断是异步的，不和某条具体指令绑定。
  • 异常通常是同步的，由当前指令直接触发。

从返回语义看：

类型典型来源同步/异步返回位置
中断 Interrupt设备、时钟、外部硬件异步下一条指令
陷入 Trap系统调用、断点等有意安排事件同步下一条指令
故障 Fault缺页、可恢复保护错误等同步当前指令重新执行
终止 Abort严重硬件错误、不可恢复故障同步通常不返回

Intuition

中断像是手机响了（外部事件，异步），异常像是走路踩到香蕉皮（当前这一步直接导致的，同步）。陷入则是你主动按了门铃——虽然也是同步事件，但它是”故意”的，目的就是请人开门（进入内核）。

陷入本质上是”软件主动制造的异常”，是用户态请求内核服务的标准入口。系统调用、断点、软件中断都属于这类。它之所以重要，是因为把”进入内核”从一个不受控的跳转变成了受硬件保护、由内核统一分派的标准入口。

4.2 引入原因与工程意义

中断首先是为了支持 CPU 与设备并行工作。CPU 发起 I/O 后，不用傻等着设备做完，可以继续干别的；等设备做完了，再通过中断通知 CPU。这样慢速外设就不会拖垮整个系统。

异常则是为了让处理器遇到非法或特殊情况时，能立即脱离当前控制流进入内核处理。除零、取指非法、特权违例、缺页、系统调用，这些情况都必须被硬件识别并交给操作系统，才能保证系统安全和语义正确。

时钟中断尤为重要。它不仅用于计时，更是抢占式调度、时间片记账、定时器到期、睡眠唤醒的驱动力。没有周期性的时钟中断，操作系统很难强制收回 CPU 控制权，也就谈不上公平的多道程序运行。

Example

想象你在写一个大文件。如果没有中断，CPU 每次发写请求后都得轮询”硬盘写完了吗”，CPU 利用率几乎归零。有了中断，CPU 发完请求就去干别的，硬盘写完后发个中断，CPU 再回来处理下一步。这就是并行带来的吞吐量提升。

4.3 硬件与软件的分工

中断/异常机制是典型的软硬件协同。硬件负责”迅速、准确、不可绕过地转移控制权”；软件负责”解释事件并执行系统策略”。

硬件做的事：

  1. 发现中断或异常源。
  2. 根据中断号或异常类型确定向量。
  3. 必要时完成特权级检查与堆栈切换。
  4. 把关键上下文压入当前或新的内核栈。
  5. 装载处理程序入口地址，开始执行第一条指令。
  6. 对某些门类型自动修改中断允许状态（比如中断门自动清 IF）。

软件做的事：

  1. 保存其余通用寄存器与段寄存器。
  2. 识别事件种类与具体原因。
  3. 执行相应服务：驱动设备、修复故障、实现系统调用、发送信号、更新统计信息等。
  4. 必要时唤醒等待队列或触发调度。
  5. 恢复上下文并执行返回指令。
sequenceDiagram
    participant DEV as 设备/时钟
    participant CPU as 处理器硬件
    participant ENT as 入口汇编
    participant H as 内核处理程序
    participant RET as 返回路径

    DEV->>CPU: 发出中断请求
    CPU->>CPU: 完成当前指令后检测事件
    CPU->>CPU: 查向量/切特权级/必要时换栈
    CPU->>ENT: 跳转到入口地址
    ENT->>ENT: 保存寄存器、建立内核执行环境
    ENT->>H: 调用中断/异常/系统调用处理例程
    H->>RET: 完成服务、可能触发调度
    RET->>CPU: 恢复现场并执行异常返回
    CPU->>CPU: 回到原程序或新选中程序

Note

硬件和软件的分界线并不是绝对的，不同架构有不同约定。比如 x86 的硬件会帮你切栈、压 EFLAGS/CS/EIP；而 RISC-V 的硬件保存的现场更少，很多工作要靠软件的 trampoline 代码来完成。

4.4 中断响应与中断处理

处理器通常在每条指令执行周期结束时检查可屏蔽中断请求。如果没有中断，就继续执行下一条；如果有，硬件通过中断号查向量表找到处理程序。

  • 中断响应：强调的是硬件动作——发现、接收、转交控制权。
  • 中断处理：强调的是软件动作——保存现场、解释原因、操纵设备、恢复执行。

完整流程：

  1. 设备发出中断信号。
  2. 处理器完成当前指令。
  3. 硬件保存最小必要上下文。
  4. 按中断号或异常类型查找向量。
  5. 将处理程序入口装入相关寄存器，开始执行入口代码。
  6. 软件进一步保存寄存器并完成具体处理。
  7. 处理结束后恢复现场，通过返回指令离开内核。

4.5 操作系统初始化与中断/异常

操作系统在启动阶段必须先建立能接管处理器控制权的基础设施，然后才敢安全地开中断。典型的初始化包括：

  • 建立异常与中断处理例程。
  • 设置中断向量表或中断描述符表（IDT）。
  • 初始化系统栈、内核栈以及特权切换所需的 TSS 等结构。
  • 初始化 PIC/APIC 或其他中断控制器。
  • 设置时钟中断与系统调用入口。
  • 在关键处理例程就绪后，才打开中断。

Intuition

这就像是消防队开业：先要把消防站建好、水管接好、消防员培训完，才能打开 119 热线。如果还没准备好就打开中断，来了中断也没人处理，系统直接崩溃。

5. IA-32 体系结构对中断的支持

5.1 实模式与保护模式

在实模式下，x86 使用**中断向量表（IVT）**保存服务程序入口地址。入口地址由段地址和偏移组成：

入口地址=(段地址≪4)+偏移地址

实模式没有成熟的特权切换机制，中断处理更像一种受硬件帮助的过程调用。

在保护模式下，x86 使用中断描述符表（IDT）。表项不是简单地址，而是门描述符。门描述符除了给出偏移和段选择符之外，还携带门类型、描述符特权级 DPL 和存在位等信息，因此能够同时完成”安全检查”和”控制转移”。

Trap

实模式和保护模式下的”中断向量表”名字不同（IVT vs IDT）、结构不同、功能也不同。考试中如果混为一谈，很容易丢分。记住：实模式叫 IVT，保护模式叫 IDT，IDT 里存的是门描述符。

5.2 门描述符、IDT、GDT 与 TSS

IA-32 解析一个中断入口，要查好几个表：

  • IDTR：告诉你 IDT 在哪里。
  • IDT：按向量号索引门描述符。
  • GDTR：告诉你 GDT 在哪里。
  • GDT：按段选择符找到段描述符。
  • TSS：在发生特权级切换时提供新特权级的栈指针。

x86 提供了四类相关门描述符：

  1. 任务门（Task Gate）：与 TSS 相关，Linux 一般不依赖它做普通中断处理。
  2. 中断门（Interrupt Gate）：进入处理程序后自动清 IF，适合外部中断。
  3. 陷阱门（Trap Gate）：不自动清 IF，适合同步异常或系统调用。
  4. 调用门（Call Gate）：用于受控过程调用。

中断描述符中的段选择符并不是直接地址，而是 GDT/LDT 中段描述符的索引。门描述符给出”进入哪个代码段的哪个偏移”；段描述符再给出该代码段的基址与属性。于是处理程序入口地址可抽象表示为：

处理程序入口=段基址+段内偏移 
flowchart LR
    IDTR[IDTR] --> IDT[IDT]
    V[中断向量号 i] --> IDT
    IDT --> G[门描述符<br/>段选择符 + 偏移 + 属性]
    GDTR[GDTR] --> GDT[GDT]
    G --> SEL[段选择符]
    SEL --> GDT
    GDT --> SD[段描述符<br/>段基址 + 段属性]
    G --> OFF[段内偏移]
    SD --> ADDR[处理程序入口地址]
    OFF --> ADDR

Intuition

可以把这想象成”多级快递分拣”：IDT 是”总集散中心”，按快递单号（向量号）找到对应的分拣门；GDT 是”区域仓库”，按门上的地址找到具体的仓库位置；最后把偏移量加上，才能精确送到收件人手里（处理程序入口）。

5.3 IA-32 硬件处理流程

当 IA-32 处理器响应中断或异常时，硬件会执行以下步骤：

  1. 确定与事件关联的向量号。
  2. 通过 IDTR 找到 IDT，并取出对应门描述符。
  3. 由门描述符得到段选择符与偏移。
  4. 由 GDTR 找到 GDT，再根据段选择符取得段描述符与段基址。
  5. 检查特权级是否合法。
  6. 若发生从用户态到内核态的转换，则按照 TSS 提供的栈信息切换到内核栈。
  7. 将返回所需上下文压栈；某些异常还会压入硬件出错码。
  8. 若通过中断门进入，则自动清除 IF。
  9. 根据段基址和偏移定位处理程序入口，开始执行。

堆栈切换是理解系统调用和异常处理的关键。用户栈属于当前用户程序，不能直接承载内核执行时的可信上下文，因此处理器在跨特权级时必须切换到与目标特权级绑定的内核栈。Linux 并不依靠 TSS 完成传统意义上的”硬件任务切换”，但仍利用 TSS 提供 Ring 0 栈指针，用于从用户态陷入内核时的堆栈切换。

Note

为什么 TSS 在 Linux 里只用来提供栈指针，而不是做完整的任务切换？因为硬件任务切换太慢、太僵硬，Linux 选择用软件实现上下文切换（switch_to 或 schedule 里的代码），只保留 TSS 的栈切换功能来满足 x86 保护模式的要求。

6. Linux 中的中断向量与处理

在 Linux/x86 中，部分向量具有固定含义：

  • 0–19：不可屏蔽中断与处理器异常。
  • 32–127：外部中断。
  • 128（0x80）：传统系统调用入口。
  • 239：本地 APIC 时钟中断。
  • 251–253：处理器间中断。
  • 254：本地 APIC 错误中断。
  • 255：本地 APIC 伪中断。

常见处理器异常：

  • 0：除零错误。
  • 1：调试异常。
  • 4：算术溢出。
  • 6：非法操作码。
  • 12：栈异常。
  • 13：一般保护错误。
  • 14：缺页异常。

这些向量构成了”处理器异常语义”和”操作系统异常处理代码”之间的编号契约。缺页异常尤其重要，虚拟内存中的按需调页、写时复制、地址保护，全都依赖它完成。

Example

你在 Linux 里写了一个大数组，第一次访问某页时发现还没分配物理页。CPU 触发缺页异常（向量 14），进入内核。内核检查：这个地址是合法的（在堆扩展范围内），于是分配一页物理内存、填好页表、重新执行触发缺页的指令。对用户程序来说，这一切就像”什么也没发生”，只是稍微顿了一下。

6.1 上半部与下半部

Linux 的中断处理常分为上半部与下半部。根本原因是：中断上下文必须尽快结束，以降低中断关闭时间和响应延迟，不能在中断入口里长时间处理复杂逻辑。

  • 上半部：负责最紧急、最短小的工作，比如确认中断源、读取关键状态、快速搬运少量数据、挂起后续处理。
  • 下半部：负责可延迟的工作，比如大量数据处理、协议栈推进、较长时间的设备后处理。典型形式包括 softirq、tasklet 和 workqueue。

Intuition

上半部就像急诊室分诊：病人一来，护士先量个血压、挂个号、判断要不要立刻进抢救室；下半部才是后续的检查、输液、手术。如果所有事情都在分诊台做，后面来的病人就要在门口排队排到天荒地老。

这种拆分体现了机制与策略的分离：硬件和最小入口机制只保证”及时接住事件”，较复杂的处理策略交给可调度、可延迟的软件层完成。

二、操作系统运行机制：系统调用

Summary

系统调用是用户程序进入内核的”正门”。这一节我们会把系统调用的来龙去脉讲清楚：从你在 C 代码里写 write()，到 CPU 执行 int 0x80，再到内核查表找到 sys_write，最后怎么返回。学完要搞清楚 system_call()、sys_call_table、ret_from_sys_call 这三兄弟的分工。

1. 系统调用的地位

系统调用是用户程序可以请求的操作系统功能，是用户态进入内核态的标准、受控且唯一的服务入口。进程控制、文件操作、目录操作、设备管理、进程通信、内存管理、信息维护等功能，都通过系统调用暴露给应用程序。

系统调用不是普通函数调用的放大版。两者的根本区别：

  1. 跨越特权边界：系统调用要从用户态切到内核态，必须发生特权级切换。
  2. 依赖硬件陷入：通过陷入指令完成受保护的控制转移。
  3. 可能阻塞或调度：系统调用可能睡眠、触发信号或引起调度，返回时间和返回对象都不一定与调用点一一对应。
  4. 服务目标不同：系统调用的目标是内核服务；普通函数调用只是在同一权限域内复用代码。

Intuition

普通函数调用就像在同一家公司里打电话给隔壁同事；系统调用则是你打电话给政府服务窗口（比如税务局），对方有更高的权限，能查到你查不到的信息，但流程也更严格、更慢。

API、库函数和系统调用并不等价。库函数/API 位于用户态，为程序员提供更友好的接口；其中一部分最终会触发系统调用，另一部分可能完全在用户空间实现。内核函数位于内核内部，是系统调用处理过程中被调用的服务例程，但并不直接对用户开放。

flowchart TD
    A[应用程序] --> B[C 库 / API]
    B -->|可能仅用户态处理| B2[纯库函数路径]
    B -->|需要内核服务| C[系统调用封装]
    C --> D[陷入指令]
    D --> E[系统调用总入口]
    E --> F[sys_xxx / 内核服务例程]
    F --> G[驱动程序 / 文件系统 / 调度器 / 内存管理]
    G --> H[返回用户态]

printf() 是理解 API 与系统调用关系的典型例子。printf() 本身是 C 标准库函数，负责格式化输出；真正把字节写入终端或文件描述符的动作，通常会落实为 write() 系统调用。因此：

  • 库函数是”语言层接口”。
  • 系统调用是”内核入口”。
  • 两者位于不同层次。

2. 系统调用机制的组成

一个完整的系统调用机制包含四个基本要素。

2.1 陷入指令

处理器必须提供一条专门用于进入内核的特殊指令。它的作用不是直接完成服务，而是触发一次受控异常，使处理器从用户态转入内核态，把控制权交给系统调用总入口。

不同体系结构的指令：

  • x86 传统路径：int 0x80
  • x86 快速路径：sysenter/sysexit
  • x86-64：syscall/sysret
  • RISC-V：ecall

Note

int 0x80 里的 int 是 “interrupt” 的缩写，但这里触发的是同步异常（陷阱），不是异步外部中断。它只是借用了中断/异常的硬件通路来完成特权切换。

2.2 系统调用号

每个系统调用都事先分配一个编号，作为内核分派服务的索引。例如早期 Linux/x86 中：

  • __NR_exit = 1
  • __NR_fork = 2
  • __NR_read = 3
  • __NR_write = 4

系统调用号的存在使得入口路径与具体服务逻辑解耦。陷入机制只负责”安全进入内核”，而调用号决定”究竟请求哪一项服务”。

2.3 参数传递约定

系统调用不仅要传调用号，还要传参数。常见方案有三种：

  1. 由陷入指令自身携带少量参数（不太常见）。
  2. 用通用寄存器传递参数（最常用）。
  3. 将参数放在用户内存的专用区域或堆栈中，再由内核读取。

x86/Linux 传统 int 0x80 路径采用寄存器传参：eax 保存系统调用号和返回值，ebx/ecx/edx/esi/edi 等寄存器保存参数。寄存器传参速度快，但可容纳参数数量有限；参数较多时往往需要借助内存。

2.4 系统调用表

系统调用表维护”调用号 → 内核处理函数”的映射关系。系统调用总入口根据 eax 中的调用号索引该表，取得真正的处理例程入口。这样，入口逻辑只需要实现一次，具体服务可通过查表分派。

Intuition

系统调用表就像餐厅的点餐单：顾客（用户程序）报一个编号（系统调用号），服务员（总入口）看单子上对应的菜品（sys_xxx），然后通知后厨去做。顾客不需要知道后厨在哪、厨师是谁，只需要报对编号就行。

这正是机制与策略分离在系统调用中的体现：陷入机制、参数约定和查表分派是统一的机制；每个 sys_xxx 的功能语义则是具体策略与服务实现。

考试中常见的一个陷阱是：系统调用的返回值也是通过 eax 返回的。如果系统调用失败，内核通常返回负的错误码（如 -EINVAL），C 库会把它转成 -1 并设置 errno。直接写汇编时要注意，负的错误码在 eax 里是以补码形式存在的。

4. 系统调用的执行过程

系统调用的执行可分为五个阶段：用户态准备、硬件陷入、入口分派、服务执行、返回收尾。

4.1 用户态准备

应用程序通常不直接写陷入指令，而是调用库函数封装。封装例程完成三项工作：

  1. 将系统调用号写入指定寄存器。
  2. 按约定将参数装入寄存器或内存。
  3. 执行陷入指令进入内核。

4.2 硬件陷入

CPU 执行陷入指令后，触发一次同步异常。若发生特权级变化，处理器切换到内核栈，并将用户态 SS:ESP、EFLAGS、CS:EIP 等信息压入新栈，用于之后返回。对于某些异常，还会额外压入硬件出错码。

4.3 总入口分派

系统调用总入口通常由汇编实现。其职责包括：

  1. 保存剩余寄存器现场。
  2. 建立内核数据段等执行环境。
  3. 检查系统调用号是否合法。
  4. 根据系统调用号查系统调用表。
  5. 调用对应 sys_xxx 处理函数。

4.4 服务执行

对应的 sys_xxx 函数在内核态运行，访问受保护资源，完成所请求的操作，例如写文件、创建进程、映射内存或等待事件。

4.5 返回路径

服务例程结束后，并不一定立即回到用户程序。返回路径往往还要处理两类内核事务：

  • 是否需要重新调度。
  • 是否存在待处理信号。

只有在这些收尾工作都满足条件后，内核才恢复保存的寄存器与状态字，执行异常返回指令，重新进入用户态。

flowchart TD
    A[用户态库函数封装] --> B[设置系统调用号与参数]
    B --> C[int 0x80 / syscall / ecall]
    C --> D[硬件切换到内核态并压入最小上下文]
    D --> E[系统调用入口汇编保存更多寄存器]
    E --> F{系统调用号合法?}
    F -->|否| G[返回 -ENOSYS]
    F -->|是| H[按调用号索引 sys_call_table]
    H --> I[执行 sys_xxx 内核服务例程]
    I --> J[ret_from_sys_call]
    G --> J
    J --> K{need_resched?}
    K -->|是| L[调度器]
    K -->|否| M{sigpending?}
    L --> M
    M -->|是| N[处理信号]
    M -->|否| O[恢复现场]
    N --> O
    O --> P[返回用户态]

Intuition

系统调用返回路径就像你考完试走出考场：不是交卷就能立刻回家，可能还要经过”是否需要重新分班（调度）”、“有没有被老师叫去谈话（信号处理）“这两个关卡，最后才能真正回到教室（用户态）。

5. Linux/x86 上的传统系统调用实现

5.1 int 0x80 与 IDT 的关系

在传统 Linux/x86 中，系统调用入口使用 int 0x80。操作系统初始化时，会在 IDT 的第 128 项安装对应门描述符，将其入口指向统一的 system_call() 汇编入口。早期内核中常见的初始化形式是：

这一设计包含三个关键点：

  1. IDT 第 128 项是系统调用统一入口。用户程序执行 int 0x80 时，处理器先通过 IDT 找到门描述符，再跳转到 system_call()。
  2. system_call() 不是具体系统调用本体，而是总入口。真正的 sys_write、sys_read、sys_exit 等函数，是由总入口根据系统调用号进一步查表找到的。
  3. 系统调用表承担分派功能。sys_call_table 中按调用号顺序存放处理函数地址，调用目标可抽象为：
目标地址=sys_call_table[eax]

或在 32 位表项下写为：

目标地址=sys_call_table_base+4×eax

Note

注意这里的 eax 在用户态已经被库函数设置为系统调用号。进入内核后，总入口首先把这个号保存到栈上，然后检查合法性，再作为数组下标去查 sys_call_table。

5.2 为什么使用陷阱门

Linux 在 0x80 处通常设置为 DPL 为 3 的陷阱门。DPL 为 3 使用户态代码有权限通过该门进入内核；门类型选为陷阱门而非中断门，使进入系统调用时不必由硬件自动清 IF。系统调用是同步事件，不同于外部硬件中断；保持中断能力有利于降低系统整体延迟，具体临界区是否关闭中断则由内核软件自行控制。

Trap

如果 0x80 设成中断门，那么每次系统调用进入时硬件都会自动关中断（清 IF）。这对于高频系统调用来说开销很大，而且没必要——系统调用本身不是异步中断，不需要这种保护。所以 Linux 用陷阱门，让软件自己决定是否关中断。

5.3 system_call() 与 sys_call_table

system_call() 的核心逻辑包括：

  1. 将系统调用号压栈。
  2. 执行 SAVE_ALL 保存寄存器。
  3. 检查 eax 是否小于 NR_syscalls。
  4. 若越界，则返回 -ENOSYS。
  5. 否则调用 sys_call_table[eax] 所指向的内核函数。
  6. 将返回值写回栈上的 eax 位置。
  7. 跳转到 ret_from_sys_call 做统一收尾。

SAVE_ALL 的含义，是在硬件已保存最小上下文的基础上，再由软件将 es、ds、eax、ebp、edi、esi、edx、ecx、ebx 等寄存器压栈，并将数据段寄存器切换到内核数据段。于是系统调用处理过程中，内核可以在一个完整、可信的内核栈帧上运行。

flowchart TD
    A[int 0x80] --> B["CPU 根据 IDT[0x80] 进入 system_call"]
    B --> C[硬件已压入 SS ESP EFLAGS CS EIP]
    C --> D[SAVE_ALL 保存通用寄存器与段寄存器]
    D --> E{eax < NR_syscalls?}
    E -->|否| F[返回值设为 -ENOSYS]
    E -->|是| G["call sys_call_table[eax]"]
    G --> H[sys_xxx 执行完成，返回值在 eax]
    F --> I[ret_from_sys_call]
    H --> I

Intuition

system_call() 就像是火车站的安检入口：所有人（所有系统调用）都必须先过这道安检，查身份证（检查调用号合法性）、存行李（SAVE_ALL），然后才能根据车次（调用号）找到对应的站台（sys_call_table）上车（执行 sys_xxx）。

5.4 ret_from_sys_call 的地位

系统调用处理结束后，处理器并不是立即回到原用户指令，而是先执行 ret_from_sys_call。这一返回路径的作用包括：

  1. 检查当前任务的 need_resched 标志。
  2. 若需要，则转入调度器选择新任务。
  3. 检查是否有待处理信号。
  4. 最终恢复全部寄存器并返回用户态。

Example

假设你调用 read() 去读一个空管道。sys_read 发现没数据，就把进程状态改为睡眠，设置 need_resched，然后返回。到了 ret_from_sys_call，发现需要调度，于是调度器选另一个进程运行。等你这个进程再次醒来时，是从 ret_from_sys_call 的恢复路径返回用户态的——此时可能已经过去很久了。

因此，系统调用返回路径本身也是一个重要的内核调度点。进程虽然从用户态通过 int 0x80 进入内核，但返回用户态时，实际获得 CPU 的不一定仍然是原进程；如果发生阻塞、时间片耗尽或更高优先级任务就绪，则返回路径可能先切到调度器，再由调度器决定后续运行对象。

5.5 栈切换与内核栈布局

从用户态执行 int 0x80 时，若发生特权级变化，处理器会从 TSS 中取出 Ring 0 栈指针，完成用户栈到内核栈的切换。硬件先在内核栈中压入返回所需信息，随后入口汇编再继续压入其余寄存器。

flowchart TD
    A[用户栈: 参数与局部变量] -->|int 0x80| B[切换到内核栈]
    B --> C[硬件压栈: SS ESP EFLAGS CS EIP]
    C --> D[某些异常再压入错误码]
    D --> E[SAVE_ALL: ES DS EAX EBP EDI ESI EDX ECX EBX]
    E --> F[system_call / sys_xxx]
    F --> G[RESTORE_ALL + iret]

这种分层压栈结构解释了两个重要事实：

  1. 用户栈与内核栈是不同的抽象对象，前者属于用户地址空间，后者属于内核控制。
  2. 系统调用处理期间使用的是内核栈，即使用户进程的用户栈内容恶意或损坏，内核仍能在可信环境中执行入口代码。

Trap

注意硬件压栈顺序和 SAVE_ALL 压栈顺序可能不一样。恢复时要严格对应，否则寄存器内容会错乱。考试中如果考到栈布局，要分清”硬件自动压的”和”软件手动压的”两部分。

6. 快速系统调用指令

int 0x80/iret 的优点是通用、清晰、完全依赖 IDT 与标准异常机制；缺点是路径较重，涉及门描述符解析、较完整的硬件现场构造和统一异常入口，因此开销较大。

为降低系统调用开销，x86 在较新处理器中提供了 sysenter/sysexit。其特点是：

  • 不是通过普通 IDT 门描述符进入。
  • 入口地址、代码段与栈等信息通常由操作系统预先写入模型专用寄存器（MSR）。
  • 硬件自动完成的状态保存更少，需要软件遵守更严格的调用约定。
  • 适合高频、短路径的系统调用快速进入内核。

x86-64 进一步使用 syscall/sysret 作为 64 位环境下的标准快速系统调用指令。它们与 int 0x80 的根本差异，不在于”能否调用内核”，而在于”进入与返回路径的硬件约定更专门化、负担更轻、性能更高”。代价则是通用性与对称性较弱，需要操作系统在入口、返回地址和寄存器保存上显式配套。

Intuition

int 0x80 像是走火车站的完整安检流程，虽然安全但慢；syscall 像是给常旅客办的快速通道，提前录入了信息，刷卡直接过。代价是你要提前知道快速通道的规矩（MSR 配置、寄存器约定），不能乱走。

7. 系统调用、函数调用与内核函数的边界

系统调用与普通 C 函数调用之间的区别可以从四个边界理解。

7.1 权限边界

普通函数调用不改变特权级，调用者与被调用者处于同一保护域中。系统调用必须跨越用户态与内核态边界，进入受到硬件保护的执行域。

7.2 控制流边界

普通函数调用由 call/ret 等指令完成，控制流仍属于普通过程调用。系统调用通过陷入指令进入异常控制流，由硬件和内核共同维护返回路径。

7.3 资源边界

普通函数调用只能操作调用者本来就有权访问的数据与代码。系统调用进入内核后，可以访问页表、设备、文件系统、调度器等受保护资源，因此必须受到严格入口控制。

7.4 实现边界

系统调用是面向应用程序的稳定服务接口；内核函数是内核内部实现细节。API 和库函数则位于用户态，可能是对系统调用的封装，也可能完全不需要系统调用。于是三者关系可概括为：

  • API/库函数：程序员直接使用的接口层。
  • 系统调用：用户态进入内核的受控接口层。
  • 内核函数：内核内部被调用的实现层。

Example

malloc() 是库函数，它大多数时候只在用户态的堆管理器里分配内存，不需要系统调用；只有当堆不够用时，才会调用 brk() 或 mmap() 向内核要更多内存。brk() 和 mmap() 是系统调用；而内核里真正实现内存映射的函数则是内核函数。

8. 中断发生后内核低层工作的典型步骤

中断发生后，内核低层工作往往遵循如下顺序：

  1. 硬件压入程序计数器和状态字等最小上下文。
  2. 硬件依据中断向量装入新的程序计数器。
  3. 汇编入口代码继续保存寄存器。
  4. 建立或切换到合适的内核栈和内核数据段。
  5. C 语言中断服务程序执行具体逻辑，例如读取设备状态、搬运或缓冲输入数据。
  6. 若当前进程不宜继续运行，则调用调度器。
  7. C 层返回到汇编出口。
  8. 汇编出口恢复新”当前进程”的上下文并返回。

Intuition

这是一条从”硬件 → 汇编 → C → 调度器 → C → 汇编 → 硬件”的完整链条。没有哪个单一函数能独立完成中断处理，它是多层级协作的结果。理解这条链，你就理解了操作系统是怎么”接住”外部事件的。

9. xv6-riscv 的对照理解

xv6-riscv 为理解现代内核的 trap 与系统调用提供了一个更简洁的参照系。

9.1 特权模式与启动阶段转换

RISC-V 定义三种主要特权模式：机器模式 M、监管模式 S 和用户模式 U。xv6 启动时由机器模式开始，随后在早期启动代码中配置委派与中断等状态，通过 mret 进入监管模式运行内核主流程；创建首个用户进程后，再通过 trap 返回路径中的 sret 进入用户模式。

因此，xv6 的基本模式转换链是：

flowchart LR
    M[机器模式 M<br/>上电与早期启动] -->|mret| S[监管模式 S<br/>内核执行]
    S -->|sret| U[用户模式 U<br/>用户进程执行]
    U -->|ecall / 异常 / 中断| S

Note

RISC-V 的 mret 和 sret 分别用于从机器模式和监管模式的 trap 中返回。操作系统启动时巧妙地利用 mret 来”伪造”一次从 M 模式 trap 返回的效果，从而把特权级降到 S 模式。这是一个常见的启动技巧。

9.2 系统调用号、参数与返回值

在 xv6-riscv 中：

  • 系统调用号通过 a7 传递。
  • 参数通常通过 a0–a5 传递。
  • 返回值通过 a0 返回。

这与 x86 eax + 通用寄存器 的方案同属”寄存器传参”，只是具体寄存器约定不同。

9.3 Trap 相关控制寄存器

RISC-V 的 trap 处理高度依赖几个控制寄存器：

  • stvec：保存 trap 向量基址，决定 trap 进入何处。
  • sepc：保存发生 trap 时的程序计数器，用于返回。
  • scause：记录 trap 原因，用于区分系统调用、设备中断、时钟中断或异常。
  • sstatus：保存监管态状态，如中断使能位和先前特权级信息，用于 trap 进入与返回时的状态维护。

Trap

很多同学容易把 sepc 和 stvec 搞混。sepc 是”出问题时 PC 在哪”（类似 x86 的 EIP），stvec 是”出问题后跳到哪去处理”（类似 x86 的 IDT 入口地址）。一个是”案发现场”，一个是”派出所地址”。

9.4 Trap 保存现场与进程切换

xv6 的 trampoline.S 和 kernelvec.S 会在 trap 进入时保存寄存器状态。对用户态 trap，系统会将用户寄存器保存到进程的 trapframe 中；真正发生进程上下文切换时，swtch 主要保存和恢复 ra、sp 与 s0–s11 等需要跨函数保持的寄存器。于是 “trap 保存现场” 和 “进程切换保存上下文” 是相邻但不同的两个层次：

  • trap 保存：关注”从用户态/内核态安全进入内核”。
  • 上下文切换保存：关注”稍后恢复某个内核执行流”。

Intuition

trap 保存像是你进医院时护士登记的个人信息（全身上下查一遍）；进程切换保存则是住院期间换病房时转移的病历档案（只保留必要信息）。前者是为了安全进入医疗系统，后者是为了在不同病房之间流转。

9.5 devintr 与 uartintr

xv6 中，devintr 负责识别设备中断的来源。对于外部设备中断，它通常先从中断控制器处领取中断号，再判断对应设备类型；若是 UART 中断，则调用 uartintr 完成串口收发处理，随后向中断控制器报告完成。对于时钟中断，devintr 则负责触发时钟相关记账与唤醒逻辑。

uartintr 的内部流程通常包括：

  1. 检查串口接收状态。
  2. 从 UART 硬件寄存器取出已到达字符。
  3. 将字符送入控制台输入缓冲或相应上层处理逻辑。
  4. 处理发送完成后的后续输出推进。

devintr 调用 uartintr 的判断条件，本质上是”外部中断号被识别为 UART 所在 IRQ”。这反映了中断框架与具体设备驱动的分层：通用层先识别来源，再把控制权交给设备专用驱动。

三、机制与策略分离

Summary

操作系统设计里有一条黄金法则：先搭好安全的”机制”，再在机制之上灵活调整”策略”。这一节帮你看清，前面讲的中断、系统调用、调度里，哪些是”铁打的机制”，哪些是”流水的策略”。

机制与策略分离是操作系统设计中的基本原则。机制回答”系统具备什么能力、如何安全地实现这种能力”；策略回答”在具体运行时刻，系统决定怎样使用这些能力”。

在本章讨论的对象中，这一原则表现得尤为清楚：

  • 中断/异常机制提供受硬件保护的内核入口；而是否立即调度、如何处理中断后续工作，则属于策略。
  • 系统调用机制提供统一陷入、参数传递与查表分派；而每个 sys_xxx 的具体语义与实现是服务策略。
  • 时钟中断机制周期性夺回控制权；而时间片长度、优先级规则与调度算法属于策略。
  • 页故障机制负责把”访问失败”转化为内核可处理事件；而页面置换和预取属于策略。

Intuition

机制就像汽车的刹车系统——必须有，而且必须安全可靠；策略就像司机决定什么时候踩刹车、踩多狠。你可以换不同的司机（策略），但刹车系统的机械原理（机制）不能乱改。这种分离让操作系统既能保证安全，又能灵活演进。

这种分离使操作系统能够在不改变底层安全框架的前提下，更换调度算法、优化中断后半部、替换文件缓存策略或调整系统调用实现细节，从而兼顾可演化性、可维护性与性能。

四、内核进入与退出的总体路径

Summary

最后我们把所有碎片拼成一张大图：应用程序是怎么进内核的？内核又是怎么回到用户态的？这一节是本章的总览图，建议对照着前面的细节反复看。

从系统整体看，内核的进入路径主要有三种：系统调用、异常和中断。退出路径则统一表现为”恢复上下文并返回用户态”，但在返回前往往要经过调度和信号处理等内核逻辑。

flowchart TD
    APP[应用程序] --> LIB[函数库 / API]
    LIB --> SCI[系统调用接口]
    APP -->|非法访问、除零、缺页| EX[异常]
    DEV[外部设备] -->|中断| INT[中断]

    SCI --> K[内核]
    EX --> K
    INT --> K

    K --> SYSC[系统调用实现]
    K --> EXH[异常服务例程]
    K --> DRV[设备驱动]
    DRV --> DEV

    SYSC --> RET[返回路径]
    EXH --> RET
    DRV --> RET
    RET --> APP

Intuition

这张图就是操作系统的”交通总图”：系统调用是”主动出发”（你自己要去办事），异常是”半路出状况”（车抛锚了），中断是”外部有人找你”（快递到了）。三条路最终都通向内核这座城市，办完事再从统一的出口（返回路径）回家。

这一结构揭示了操作系统运行环境与运行机制的统一性：硬件提供事件与保护边界，内核在边界之内集中处理资源与控制流，应用程序通过受控入口使用系统功能。系统调用解决”主动请求服务”的问题，中断解决”被动接收外部事件”的问题，异常解决”执行流中出现特殊情形”的问题，而三者共同构成了现代操作系统的核心执行框架。
