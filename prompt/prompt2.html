你现在的任务不是总结课件，也不是写讲课提纲，而是基于我提供的 PDF、PPT、PPTX、Markdown（.md）课程材料与现有 Markdown 笔记，整理出一份可直接作为“教材式课程笔记”使用的完整正文。

【任务定位】

  1. 成品应当像课程教材中的章节笔记，而不是课堂讲稿、课件摘要、读书笔记、提纲或材料评论。
  2. 文风应当正式、清晰、克制、稳定，适合作为长期保存与反复复习的参考文本。
  3. 输出内容应直接陈述知识本身，不解释整理过程，不与读者对话，不评论材料来源。
  4. 内容既要保留学科主线，也要落实到定义、机制、条件、公式、步骤、例子、边界与应用意义。

【输入材料处理原则】

  1. 所有输入材料，无论是 PDF、PPT、PPTX 还是 Markdown，均视为同一课程材料体系的一部分。
  2. 若输入中已包含 Markdown 笔记，这些内容应被吸收、校正、展开和整合，而不是机械拼接或原样保留。
  3. 对重复内容应合并整理，保留信息更完整、逻辑更清楚、表述更稳定的版本。
  4. 对互补内容应整合为连贯正文，不得因为来源不同而割裂叙述。
  5. 对冲突内容，应优先采用定义更严格、逻辑更完整、上下文更一致的表述；若差异来自适用条件不同，必须写明前提与边界。
  6. 原材料中的零散记录、课堂口吻、待办、备注、提示语、个人整理痕迹，不得直接进入成品，除非其中包含理解知识所必需的信息。
  7. 原材料中的明显书写错误、LaTeX 错误、渲染错误、符号残缺或 Markdown 结构错误，应在不改变原意的前提下修正为规范写法。

【内容完整性】

  1. 必须覆盖材料中出现的所有重要知识，不能只保留提纲或主干框架。
  2. 必须纳入以下内容：
    • 概念、定义、术语、核心命题
    • 理论框架、前提假设、基本模型、基本机制
    • 分类、比较、层次关系、适用条件与边界
    • 公式、符号、变量含义及其解释
    • 关键流程、推导链条、分析步骤、操作路径或处理过程
    • 图表中承载的关键结构信息
    • 历史背景、典型方法、经典例子、常见应用场景
    • 判断题、辨析题、思考题背后的知识内容
    • 计算示例、推导步骤与结果含义
    • 容易混淆的概念边界与常见误解
  3. 信息量较少的页面或片段也要被有效吸收进整体叙述，不得直接略去。
  4. 不能因为追求简洁而省略变量定义、公式解释、推导步骤、例子背景或结论成立的条件。
  5. 表格、区间、阈值、等级、对应关系、比较维度必须被转化为正文叙述，不得只是抄录。
  6. 同类内容应首次完整说明，后文只保留新增信息、差异、变化与适用条件，避免重复解释。

【结构要求】

  1. 严格保持原材料的章节边界，不得把不同讲、不同文件或不同主题模块的内容合并为同一章。
  2. 重组、合并与精简只能发生在同一章节或同一主题模块内部，不得跨章节重写知识边界。
  3. 每一章内部按知识逻辑展开，形成可连续阅读的完整正文，而不是碎片化罗列。
  4. 输出格式必须为 Markdown，适合直接放入 Obsidian。
  5. 可以使用必要的层级标题，但不能把整份笔记写成只有标题和列表的提纲结构。
  6. 若原材料本身存在“定义—分类—机制—公式—示例—应用—局限”等推进逻辑，应尽量保留其知识展开顺序。

【表达要求】

  1. 不得出现面向读者的提示性话语，例如：
    • “下面来看……”
    • “这里需要注意……”
    • “可以这样理解……”
    • “这一页主要讲……”
  2. 不得出现关于整理过程的说明性话语，例如：
    • “根据材料内容……”
    • “这里将前后内容整合……”
    • “这一部分补充了……”
  3. 不得使用“你”“我们”等课堂讲解式口吻。
  4. 不得写成对材料的评论、摘要或提炼说明，而要直接写成知识正文。
  5. 不得保留“待补充”“TODO”“见上页”“课上提到”“如图所示可知”等依赖原始上下文的残缺表述。
  6. 正文必须写成可独立阅读的教材式叙述，而不是幻灯片式短句堆砌。

【解释深度】

  1. 对抽象概念、关键指标、核心机制或基本模型，必须解释其对象、形成原因、工作逻辑、适用范围以及与相近概念的区别。
  2. 对容易混淆的内容，必须给出明确边界，而不是仅作并列罗列。
  3. 对复杂机制、分析链条或推导过程，可加入简短例子，但例子应自然嵌入正文，作为知识说明的一部分，而不是口语化举例。
  4. 对流程性内容，应明确输入、前提条件、中间步骤、关键转换关系与最终结果。
  5. 对判断题、辨析题式内容，不要保留问答形式，而要改写成完整的陈述性知识，并说明其成立或不成立的条件与理由。
  6. 对计算性内容，不能只给结论，必须保留必要的计算链条、推导步骤与解释环节。

【图表与公式处理要求】

  1. 图中的分布形态、层次关系、位置关系、状态变化、流程路径、结构对应、相关方向、误差来源、分类框架等，必须转化为正文叙述。
  2. 表格中的关键字段、比较维度、数值区间、等级划分、变量关系、对应规则，必须融入正文，不可简单省略。
  3. 对公式和推导，必须尽量保持步骤完整，包括：
    • 已知条件
    • 变量定义
    • 代入过程
    • 运算结果
    • 结果解释
    • 使用前提与适用边界
  4. 如果图表体现的是“不同方法、模型、机制或路径之间的对比”，必须写出比较维度，而不是只分别给出定义。
  5. 所有公式不能只给出表达式，必须说明：
    • 各符号分别代表什么
    • 公式用于解决什么问题
    • 在什么条件下使用
    • 计算结果应如何解释
    • 该结果不能说明什么
  6. 若原材料中的公式写法不规范，应统一改写为标准数学表达，不保留明显错误、残缺或歧义写法。
  7. 概率论、统计学、离散数学等内容中的交、并、补、条件概率、连乘、求和、集合表示等，必须使用规范数学记号，不得保留口语式、逗号式或半自然语言式混写。

【Mermaid 使用要求】

  1. 只有在流程、层次、路径、状态变化或关系结构较复杂时才使用 Mermaid。
  2. Mermaid 图是正文的一部分，用于承载结构信息，不是装饰。
  3. 正文负责解释概念、原因、逻辑、意义、边界与局限；Mermaid 图负责表达结构、流程、关系、状态与路径。二者应互补，不机械重复。
  4. 单幅图只承载一个核心机制；若内容过于复杂，应拆成多幅小图。
  5. 若清晰正文已经足够表达，则不要强行加入 Mermaid。
  6. 所有 Mermaid 图必须使用标准 fenced code block，例如：
graph TD
    A["输入"] --> B["处理"]
    B --> C["输出"]
  1. 一个 Mermaid 代码块只放一张图，不得与普通正文混写。
  2. 图内节点若包含中文、空格、符号或较长短语，优先使用引号包裹，减少渲染歧义。

【数学与 Markdown 硬规则】

以下规则优先级最高。数学公式属于“保真区”，禁止意译、禁止润色、禁止擅自改写结构。

  1. 所有数学公式统一使用 LaTeX 语法。
  2. 行内公式一律使用 $...$。
  3. 独立公式块一律使用 $$...$$，禁止使用 `$$ …
4. 需要独立展示的公式必须单独成段，前后保留空行。 5. 公式块前后不得添加任何 Markdown 结构符号，包括但不限于： - `#` - `##` - `###` - `-` - `*` - `>` 6. 禁止把公式块误写成标题、列表项或引用块的一部分。 7. 若一行以 `#` 开头，则该行必须是标题文本，不能是公式，也不能包裹公式。 8. 不得出现 `# $$ ... $$`、`# $$ ... $$`、`> $$ ... $$` 等错误结构。 9. 集合必须写成 `\{...\}`，不得写成裸 `{...}`。 10. 使用可伸缩定界符时必须成对书写，例如： - `\left\( ... \right\)` - `\left$$ ... \right $$` - `\left\{ ... \right\}` 不得出现 `\left` 与 `\right` 不配对，也不得写成 `\left\{...\right\}`。 11. 交、并、补、条件概率等记号必须写成规范形式，例如： - `A \cap B` - `A \cup B` - `\overline{A}` - `P(A \mid B)` 12. 连乘必须使用 `\cdot`、`\times` 或适当数学间距 `\,`，不得用逗号代替乘号。 13. 若表示单个变量 $x$ 上方一横，一律写作 `\bar{x}`。 14. 禁止将其写成： - `x^_` - `x¯` - `x̄` - `x上横线` - `x头顶横线` 15. 只有当横线覆盖多个字符时，才允许使用 `\overline{...}`。 16. 在 `aligned`、`cases`、`array` 等公式环境中，若需换行并增加垂直间距，必须写成 `\\[4pt]`、`\\[6pt]` 等形式。 17. 禁止把 `\\[4pt]` 错写成 `\[4pt]`。 18. `\\[4pt]` 只能出现在公式环境内部，不能被当作块级公式起始符。 19. 分段函数、大括号表达式、cases 环境中的括号与换行必须完整、规范、可渲染。 20. 公式必须尽量保持原意下的最小改动；除非是在修复明确的 LaTeX 语法错误，否则不得改动公式文本。 21. 公式中原有的等号 `=`、加号 `+`、减号 `-`、上下标、括号、转义符、对齐符 `&` 不得遗漏。 22. 禁止为了语言通顺而改写公式。 23. 禁止在公式块内部自动添加原文没有的逗号、句号、分号、顿号等标点。 24. 若原公式末尾没有标点，则公式块内部末尾不得擅自补 `,` 或 `.`。 【Markdown 稳定性要求】 1. 输出必须是可直接保存、可直接渲染的 Markdown 成品，而不是接近 Markdown 的半结构化文本。 2. 各级标题、列表、代码块、公式块必须闭合完整。 3. 代码块、公式块、Mermaid 块前后应保持合理空行。 4. 不得输出会污染正文的编辑痕迹，例如多余反引号、残留提示词、未删除占位语、未完成句、重复段落或截断内容。 5. 同一概念、同一符号、同一公式对象必须尽量前后一致，不得在不同部分随意切换记号。 【静默自检要求】 在输出最终 Markdown 正文前，必须静默完成一次检查，但不要把检查过程写出来。至少确认以下各项全部满足： 1. 不存在 `# $$`、`> $$`、`# \[`、`> \[` 之类结构污染公式块的情况。 2. 不存在独立公式块前误加 `#`、列表符号或引用符号的情况。 3. 不存在任何原本存在的 `=` 被遗漏。 4. 不存在把 `\bar{x}` 写成非标准形式的情况。 5. 不存在把 `\\[4pt]` 错写成 `\[4pt]` 的情况。 6. 不存在公式块内部混入原文没有的逗号、句号或其他标点的情况。 7. 不存在 `\left` 与 `\right` 不配对的情况。 8. 不存在指数或下标中的异常字符、残缺表达式或多余逗号。 9. 不存在把乘法误写成逗号的情况。 10. 所有集合均写为 `\{...\}`。 11. 所有 Mermaid 代码块均为标准 fenced block，且成对闭合。 12. 不存在未闭合的代码块、公式块或标题结构。 13. 不存在明显的 Markdown 结构错误、LaTeX 渲染错误或语义残缺。 若出现以下任一情况，视为输出失败，必须先自行重写，再输出： - 公式块前出现 `#` - 公式内部原有 `=` 丢失 - `\bar{x}` 被写成非标准形式 - `\\[4pt]` 被错写成 `\[4pt]` - 公式块内部混入原文没有的逗号或句号 【输出要求】 1. 直接输出最终版 Markdown 笔记正文，不要附加说明，不要解释写法，也不要写“以下是整理结果”之类的话。 2. 若材料包含多章内容，每一章必须独立成章，彼此不得混写。 3. 不得省略会影响理解课程核心逻辑的知识环节。 4. 成品目标是“适合复习的教材式笔记”，而不是百科全书式堆叠信息。 5. 在不遗漏关键知识的前提下，优先减少重复、压缩枝节、控制段落密度。 6. 最终输出必须同时满足三点：知识完整、表述稳定、语法可渲染。
