操作系统运行环境和运行机制

操作系统运行环境和运行机制

Apr 23, 202616 min read

核心速览

这部分的重点在于理解应用程序是如何与操作系统交互的。 整个操作系统的底层逻辑可以被概括为一句话：“操作系统是由中断、异常、事件驱动的” 。 应用程序不能随心所欲地操作硬件，它们必须通过系统调用 (System Call) 这扇“唯一合法的大门”进入内核态 。而这扇门的开合，底层依赖的正是硬件提供的异常控制流 (ECF, Exceptional Control Flow) 和中断机制 。

第一章：操作系统的运行环境（硬件基础）

在探讨操作系统如何管理程序之前，我们必须先理解硬件为操作系统提供了哪些“基础设施”。操作系统设计者在写代码时，首先要考虑的就是硬件环境 。

1. 处理器与寄存器

CPU 不仅仅执行运算，它内部的寄存器 (Registers) 是维持系统状态的核心 。 对于操作系统而言，最重要的不是通用数据寄存器，而是控制和状态寄存器 。它们控制着处理器的操作，且通常只能在特权级别下被修改 ：

  • PC (Program Counter)：程序计数器，永远指向下一条要执行的指令 。
  • IR (Instruction Register)：指令寄存器，存放当前正在执行的指令 。
  • PSW (Program Status Word) / EFLAGS：程序状态字。这是操作系统的“晴雨表”，里面保存了 CPU 操作的结果标记（如进位、溢出）以及当前 CPU 的运行权限状态 。

直觉解释：EFLAGS 寄存器

在 x86 架构中，这个状态寄存器叫 EFLAGS 。你可以把它想象成 CPU 的“身份证”。里面特定的比特位（比如 IOPL 位、IF 中断允许标志位）决定了当前正在运行的程序到底是“普通用户”还是“超级管理员” 。

2. 特权级别与运行模式

为了实现并发和共享，操作系统必须将自己与普通用户程序隔离起来，这就催生了保护 (Protection) 需求 。 硬件通过提供不同的运行模式 (Mode) 来实现这种保护：

  • 内核态 (Kernel Mode / Supervisor Mode)：拥有最高权限，可以执行所有指令（包括特权指令），访问所有硬件资源 。
  • 用户态 (User Mode)：权限受限，只能执行非特权指令，不能直接操作硬件 。

不同架构的特权环划分：

  • x86 架构：支持 4 个特权环（R0 ~ R3）。能力从 R0 到 R3 逐级递减。目前主流的 Linux 等操作系统只用了 R0（相当于内核态） 和 R3（相当于用户态） 。
  • RISC-V 架构：划分为机器模式 (M 模式，最高权限)、监管模式 (S 模式，OS运行在此) 和 用户模式 (U 模式) 。

3. 状态转换的唯一途径

程序不能在用户态和内核态之间随意横跳。

  • 从用户态到内核态的唯一途径是：中断 / 异常 / 陷入 (Trap) 机制 。
  • 从内核态返回用户态：操作系统只需修改 PSW（程序状态字）并跳转回用户程序的断点即可 。

第二章：中断与异常机制（操作系统的引擎）

核心速览

如果把计算机系统比作汽车，中断机制就是发动机 。 CPU 在执行指令时，控制流（Control Flow）通常是平滑向前的。但系统总是会遇到突发事件（如键盘敲击、网卡数据到达、程序除以零出错），这时硬件会强制改变控制流，打断当前程序，转去执行操作系统预设的处理程序 。

1. 事件的分类：中断 vs 异常

所有打破正常执行流的事件统称为“中断/异常”，但我们可以根据事件来源将其细分为两大类 ：

类别概念来源同步/异步行为与返回机制
中断 (Interrupt)外中断，由 CPU 外部的硬件信号引发。I/O 设备（如网卡、键盘）、时钟、硬件故障 。异步 (Async)：与当前正在执行的指令无关 。处理完后，总是返回到被中断程序的下一条指令继续执行 。
异常 (Exception)内中断，由 CPU 执行当前指令时本身引发的问题 。软件错误、保护越界、系统调用等 。同步 (Sync)：必然是由刚刚执行的那条指令引发的 。根据类型不同（故障/陷入/终止），可能返回当前指令重新执行，也可能直接终止 。

异常的三个子分类：

  1. 故障 (Fault)：可恢复的错误，比如“缺页异常”(Page Fault)。处理完后，返回到当前引发故障的指令重新尝试执行 。
  2. 陷入 (Trap)：有意识安排的异常，比如断点调试、系统调用 (syscall)。处理完后，返回到下一条指令 。
  3. 终止 (Abort)：不可恢复的致命硬件错误，直接终止程序，不会返回 。

2. 软硬协同：中断处理的生命周期

中断机制之所以精妙，在于它是硬件和软件配合完成的 。

  • 硬件的任务（中断响应）：负责“发现”和“移交”。硬件在每个指令周期的最后扫描中断寄存器。如果发现中断，硬件会自动保存当前断点（PSW 和 PC），并根据中断号查询中断向量表，把 CPU 控制权交给软件 。
  • 软件的任务（中断处理程序）：负责“具体事务”。进入内核态后，操作系统编写的中断处理程序开始运行，保存现场上下文，分析原因，处理具体逻辑，最后执行中断返回指令（如 iret） 。

3. x86 架构下的中断寻址逻辑 (IDT 与 GDT)

在 x86 的保护模式下，中断向量表升级为了 IDT (Interrupt Descriptor Table，中断描述符表) 。查找中断服务程序的流程是一个经典的“两次查表”过程：

graph TD

 B -->|使用 IDTR 寄存器定位| C[找到第 i 个门描述符]
 C --> D[提取: 段选择符 + 段内偏移量]

 E -->|使用 GDTR 和段选择符| F[找到对应的段描述符]
 F --> G[提取: 段基址]
 G --> H[段基址 + 段内偏移量 = 中断处理程序绝对入口地址]

门描述符 (Gate Descriptor) 的种类： IDT 表中的表项被称为“门”，x86 提供了几种常见的门 ：

  1. 中断门 (Interrupt Gate)：穿过此门后，硬件会自动关闭/禁止后续的中断（清空 IF 位），常用于处理硬件中断 。
  2. 陷阱门 (Trap Gate)：与中断门类似，但穿过此门后，硬件不会自动关中断，常用于系统调用或异常 。
  3. 任务门 (Task Gate) & 调用门 (Call Gate) 。

第三章：系统调用 (System Call) 的底层运行机制

核心速览

用户程序打印一句 “Hello World”，并不是直接写到显示器上的。 它调用了 C 标准库的 printf()，printf() 内部最终触发了操作系统的 write() 系统调用。 系统调用是操作系统提供给编程人员的唯一合法接口 。

1. 系统调用 vs C 函数调用

这通常是一个经典的易混淆点 ：

  • C 函数调用 / API：运行在用户态 。API 只是对底层复杂操作的封装（比如 C 库封装了参数校验和格式化）。
  • 系统调用：运行在内核态 。它真正触发了硬件层面的异常机制（陷入 Trap），跨越了特权级边界，执行了操作系统的内核代码 。

直觉解释

应用程序是“客户”，C 函数库/API 是“大堂经理”，而操作系统内核是“仓库管理员”。 客户不能自己去仓库拿东西（用户态无权限操作硬件）。客户只能告诉大堂经理（API），大堂经理整理好需求单后，通过一扇特殊的递交窗口（系统调用），把需求单递给仓库管理员（进入内核态），等管理员处理完再把结果送出来 。

2. 机制与策略的分离

操作系统在设计系统调用时，遵循了机制与策略分离的重要思想 ：

  • 机制 (硬件支持)：选择一条特殊的指令（如 int $0x80、syscall、sysenter）人为引发异常，完成状态切换 。
  • 策略 (软件实现)：为每一个系统调用分配一个功能号，并在内存中维护一张系统调用表 (sys_call_table)，根据传递进来的号码去表中寻找对应的服务例程入口 。

3. 如何跨越边界传递参数？

当程序从用户态陷入内核态时，原来的运行栈（用户栈）被切断了，我们该如何把 write(1, string, 7) 里的参数传给内核 ？ 通常有三种方法 ：

  1. 陷入指令自带参数（受限长度） 。
  2. 通过通用寄存器传递（最常用，但数量受限） 。
  3. 在内存中开辟专用堆栈区传递 。

在 x86 Linux 下，采用了寄存器传参的机制 ：

  • eax 寄存器：永远用来保存系统调用号（进内核前）和返回值（回用户后） 。
  • ebx, ecx, edx, esi, edi：按顺序保存系统调用的具体参数（最多不超过 6 个） 。

4. 庖丁解牛：Linux 响应 int 0x80 的全流程

当我们用汇编执行 int $0x80 时，系统到底发生了什么 ？以下是基于 x86 架构的完整梳理：

第一步：硬件特权级切换与状态保存 由于引发了 Trap，特权级从 R3 变到 R0。硬件会自动强制切换栈，从用户栈切换到内核栈 。 此时，硬件自动将用户程序的断点上下文压入内核栈中：包含原栈指针 SS:ESP、EFLAGS、CS、EIP 。 接着，利用 128 号 (0x80) 在 IDT 中找到陷阱门，定位到操作系统的系统调用总入口程序：system_call() 。

第二步：软件层面的 SAVE_ALL 进入 system_call() 第一件事，就是继续压栈。因为刚才只有硬件压入了少量寄存器，那些存放了参数的通用寄存器还在外面。 内核代码通过 SAVE_ALL 宏定义，将 es, ds, eax, ebp, edi, esi, edx, ecx, ebx 全部依次压入内核栈。这确保了用户程序的上下文被完整冻结 。

第三步：查表分派 (sys_call_table) 检查 eax 中的系统调用号是否合法 。如果合法，使用 eax 作为索引，去 sys_call_table（系统调用表）中调用真正的内核态处理函数，比如 sys_write 。 执行完毕后，将结果写入内核栈中当初保存 eax 的位置，作为返回值 。

第四步：清理与返回 (ret_from_sys_call) 执行返回例程 ret_from_sys_call。在这里，系统会检查是否需要重新调度进程 (need_resched) 或者处理信号 (sigpending) 。 如果一切就绪，执行 RESTORE_ALL 将刚才保存在栈里的通用寄存器弹栈恢复 。 最后执行特殊的硬件返回指令（如 iret），硬件会恢复 EIP、CS 和状态字，CPU 降级回用户态，回到 printf 下面的那行代码继续执行 。

5. 补充视角：XV6 与 RISC-V 下的上下文保存

在教学系统 XV6 (基于 RISC-V 架构) 中，中断处理的核心逻辑与上述 Linux/x86 如出一辙，只不过汇编细节不同 。 当产生陷阱（Trap）时，硬件跳转到 kernelvec.S 和 trampoline.S，在汇编代码中为陷入分配空间，并将整整 32 个寄存器（如 ra 返回地址, sp 栈指针, 以及 a0 到 a7 的传参寄存器, s0 到 s11 的保存寄存器等）存入内存结构 TRAPFRAME 中，然后才调用 C 语言编写的 kerneltrap 进行处理 。这也进一步印证了无论是 x86 还是 RISC-V，保存现场 → 查表处理 → 恢复现场 构成了 ECF 异常控制流不变的核心。
