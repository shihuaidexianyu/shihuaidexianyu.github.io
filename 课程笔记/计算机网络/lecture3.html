数据链路层

前言

1. 数据链路层的地位

数据链路层位于网络层之下、物理层之上，负责在相邻结点之间，经由一条具体链路传送数据。它所面对的不是端到端通信，而是逐跳通信：主机向远端主机发送数据时，网络层负责选择路径，数据链路层则负责将网络层交下来的分组，在当前这一跳上封装成帧并送到下一跳。

当数据从源主机到达目的主机时，会依次经过主机、交换机、路由器等设备。不同设备实现数据链路层的方式不同，但凡参与一跳通信的设备，都必须具备相应的链路层功能。

  • 主机通常实现完整协议栈；
  • 交换机主要工作在链路层；
  • 路由器则在每个接口上实现链路层与物理层，以支持逐跳转发。
flowchart TD
    H1[主机 H1] --> R1[路由器 R1]
    R1 --> R2[路由器 R2]
    R2 --> R3[路由器 R3]
    R3 --> H2[主机 H2]

    subgraph 端到端视角
    H1 -->|应用数据/报文段/IP分组| H2
    end

    subgraph 逐跳视角
    H1 -->|帧| R1
    R1 -->|帧| R2
    R2 -->|帧| R3
    R3 -->|帧| H2
    end

数据链路层的重要性在于：网络层虽然面向全网，但每一跳的实际传输都必须落实到具体链路之上；不同链路可能采用不同的链路层协议，因此同一个 IP 分组在转发过程中，可以反复被拆出、重新封装为新的帧，再沿下一条链路继续传送。

2. 数据链路层面对的两类场景

数据链路层面对的信道大体可分为两类。

  1. 点对点链路：一条链路只连接两个结点，通信关系是一对一的。
  2. 广播与共享链路：多个结点共享同一传输介质，天然支持一对多传播，但需要额外机制协调多个结点对共享信道的使用。

二者的差异并不只体现在拓扑上，更体现在协议任务上。点对点链路的核心问题是成帧、差错控制和流量控制；广播与共享链路的核心问题则是共享介质的访问控制，即多个结点如何竞争、分配或预约共同的信道资源。

flowchart LR
    A[数据链路层信道] --> B[点对点信道]
    A --> C[广播信道]
    B --> B1[一条链路只连接两个结点]
    B --> B2[重点: 成帧、差错控制、流量控制]
    C --> C1[多个结点共享介质]
    C --> C2[重点: 介质访问控制 MAC]

3. 数据链路层的基本任务

数据链路层的核心任务是：在一条具体链路上，以帧为单位完成相邻结点之间的数据交换。围绕这一任务，数据链路层需要解决三类问题：

  1. 一条链路上怎样传输一帧
    包括成帧、透明传输、差错检测、可靠传输和流量控制等机制。

  2. 多个结点怎样共享一条链路
    包括信道划分、随机接入、碰撞检测和碰撞避免等介质访问控制机制。

  3. 多条链路怎样组成一个局域网
    包括交换机自学习转发、生成树协议和VLAN 等组网机制。

第一部分 数据链路层的对象、单位与实现位置

1. 基本术语

  • 结点：主机、路由器、交换机等网络设备。
  • 链路：连接相邻结点之间的一段物理线路或无线信道。
  • 帧：数据链路层传输数据的基本单位，用于承载网络层交下来的分组。

从严格意义上说，链路只是物理通路本身，而数据链路是在链路的基础上，再加上控制数据传输的协议以及相应的软硬件实现。也可以将二者称为物理链路与逻辑链路。

2. 数据链路层的基本功能

数据链路层围绕逐跳传输这一目标，通常承担以下功能：

  1. 成帧与解帧：确定一帧从哪里开始、到哪里结束。
  2. 透明传输：保证任意比特组合的数据都能作为载荷被正确传输，而不会与帧定界符混淆。
  3. 差错检测或纠正：发现甚至定位传输过程中的比特错误。
  4. 流量控制：限制发送方发送速率，使其不超过接收方和链路的承受能力。
  5. 寻址：在局域网等场景中，用链路层地址标识接收方。
  6. 链路管理：某些协议支持建立、维护和释放链路连接。
  7. 区分控制信息与数据信息：通过帧类型字段、控制字段等实现协议控制。

这些功能并非所有协议都同时具备。点对点协议 PPP 仅提供成帧、透明传输和差错检测，不提供编号、确认和流量控制；而 HDLC 则具备更完整的链路控制能力。

3. 端系统中的实现位置

3.1 网卡与链路层实现

在主机中，数据链路层通常主要由网络适配器实现。网络适配器又称网卡或 NIC，它通过系统总线与 CPU 和内存连接，对外接入局域网或其他链路。

网卡通常是硬件、驱动程序与固件的结合体，负责：

  • 将高层交下来的数据封装成帧；
  • 添加目的地址、源地址、校验字段等链路层控制信息；
  • 在发送方向上完成并行到串行的转换；
  • 在接收方向上完成串行到并行的转换；
  • 实施部分差错检测、地址检查、缓存和中断通知。
flowchart TD
    CPU[CPU/内存] --> BUS[系统总线]
    BUS --> NIC[网络适配器 NIC]
    NIC --> PHY[物理层接口]
    PHY --> LINK[链路]
    LINK --> PHY2[对端物理层接口]
    PHY2 --> NIC2[对端 NIC]

3.2 网卡之间的通信过程

从发端看，网卡接收网络层交下来的 IP 数据报，将其封装到帧中，并在必要时加入目的地址、控制字段和帧校验序列。随后，帧被转化为可在物理介质上传输的比特流。

从收端看，网卡从物理介质接收比特流，识别帧边界，进行差错校验和地址检查；若该帧应由本机接收且校验通过，则将帧中的数据部分取出并交给网络层。

这一过程说明，网络层与链路层的边界清晰而紧密：网络层提供要逐跳传输的分组，链路层负责在本跳把分组包装成可实际发送的帧。

3.3 以太网帧的基本结构

以太网发送时，物理层前面通常还会加上前同步码与帧开始定界符，用于位同步与帧起始标识。MAC 帧本体通常包括：

字段长度
目的地址6 字节
源地址6 字节
类型2 字节
数据46～1500 字节
FCS4 字节

其中：

  • 目的地址/源地址用于链路层寻址；
  • 类型字段标识上一层协议，如 IP；
  • 数据字段承载网络层分组；
  • FCS用于差错检测。

3.4 异步传输与同步传输

链路层帧的传输可以依附于不同的物理层传输方式。

  • 异步传输通常以字符为单位，使用起始位、数据位、校验位和停止位，结构简单、成本较低，适于低速传输。
  • 同步传输要求收发双方时钟同步，帧更长，效率更高，适合长距离和高速链路。

对链路层而言，物理层的同步机制是前提；链路层则在此之上处理帧边界、控制字段和错误控制。

第二部分 点对点链路上的帧传输

点对点链路中的核心问题不是“谁来占用信道”，而是“怎样把一帧安全、正确地送到对端”。因此，这一部分的主线依次是成帧、透明传输、差错检测、可靠传输与流量控制，以及相应的协议实例。

1. 成帧

1.1 成帧的必要性

网络层交给链路层的是一个分组，但物理层看到的只是连续比特流。若不在比特流中划定帧的边界，接收方就无法判断某一帧从哪里开始、到哪里结束。因此，链路层必须进行成帧。

成帧就是在数据前后添加首部和尾部，以形成一个完整的帧。首尾字段既承载控制信息，也承担定界作用。链路层还通常规定最大传送单元 MTU，用于限制单帧可承载的数据长度。

flowchart LR
    D[网络层数据] --> H[加入首部]
    H --> T[加入尾部]
    T --> F[形成帧]

1.2 常见成帧方法

常见的帧定界方法包括：

  1. 字节计数法
    在首部中设置长度字段，指出本帧共有多少字节。其优点是简单，但若长度字段出错，接收方会失去帧同步。

  2. 控制字符法
    使用特殊字符标识帧开始和结束，例如用 SOH 表示首部开始，用 EOT 表示帧结束。它适用于面向字符的数据。

  3. 比特定界法
    采用特定的比特模式作为标志字段，例如 HDLC 中的 01111110。这适合面向比特的同步协议。

2. 透明传输

2.1 透明传输问题

如果帧定界符也可能出现在数据字段内部，接收方就可能把载荷误认为帧边界，导致数据被错误截断或拼接。因此，链路层必须保证：无论数据中出现何种比特组合，都能被当作普通数据传送，而不影响帧边界识别。这就是透明传输。

2.2 字节填充

在控制字符法中，常用字节填充解决透明传输问题。其基本思想是：若数据中出现了与帧定界符相同的特殊字符，则在它前面插入一个转义字符 ESC；接收方在恢复数据时，将这些额外插入的 ESC 删除。

flowchart TD
    A[原始数据中出现 SOH/EOT/ESC] --> B[发送端在前面插入 ESC]
    B --> C[链路上传输]
    C --> D[接收端识别 ESC]
    D --> E[删除转义用 ESC]
    E --> F[恢复原始数据]

字节填充适合面向字节的协议。它依赖发送方和接收方共同遵守同一套转义规则。

2.3 比特填充

在面向比特的同步协议中，更常见的方法是比特填充。若帧标志字段为 01111110，则发送方在数据中每出现 5 个连续的 1 后，就自动插入一个 0；接收方检测到 5 个连续 1 后再删除其后的填充 0。这样，数据部分就不会产生与帧标志完全相同的比特模式。

flowchart LR
    A[发送端扫描数据位流] --> B{出现五个连续1?}
    B -- 否 --> A
    B -- 是 --> C[插入一个0]
    C --> A

2.4 透明传输的含义

透明传输并不意味着链路层“什么都不做”，而是意味着：链路层虽然在中间执行了填充、转义、定界等处理，但对上层看来，好像这些处理不存在，原始数据能够不失真地通过链路层。

3. 差错检测与纠错

3.1 比特差错与检错的必要性

在实际传输中，信号会受到噪声、衰减和干扰影响，导致接收比特与发送比特不一致。设信道误码率为 Pb​，帧长为 L 位，则一帧无差错传输的概率为：

P1​=(1−Pb​)L

帧越长，或误码率越高，无差错接收的概率就越低。因此，链路层需要通过冗余编码进行差错检测，必要时还可进行纠错。

3.2 检错码与纠错码

  • 检错码：只能发现错误，不能定位错误位置，如奇偶校验、CRC。
  • 纠错码：既能发现错误，也能指出错误位置并进行纠正，如汉明码。

可用汉明距离刻画码字之间的最小差异。若最小码距为 dmin​，则：

  • 为检测 d 位错误，至少需要 dmin​≥d+1；
  • 为纠正 d 位错误，至少需要 dmin​≥2d+1。

这说明，纠错比检错需要更大的码距，也意味着更高的冗余开销。

3.3 奇偶校验

奇偶校验是在数据后附加一个冗余位，使整个码字中 1 的个数为奇数或偶数。它实现简单，但能力有限：

  • 一维奇偶校验可以检测单比特错误；
  • 二维奇偶校验不仅能检测单比特错误，还能在一定条件下纠正单比特错误。

奇偶校验适合理解冗余检错的基本思想，但在现代高速数据链路中通常不足以胜任主检错机制。

3.4 汉明码

汉明码是一类典型的纠错码。其基本做法是将若干校验位插入到码字中编号为 20,21,22,… 的位置，再根据各数据位对应的位置编号，确定其应参与哪些校验位的计算。接收端根据各校验位的检验结果形成综合症，从而定位出错位置，实现单比特纠错。

汉明码说明：纠错的核心不是简单附加冗余，而是让冗余位对不同数据位形成可区分的约束模式。

也就是 2 的幂次的位置都被纠错码占用了，数据插入到没有纠错码的位置。

3.5 突发错误、交织与校验和

在实际链路中，错误常常不是完全随机分布，而是以突发错的形式出现。交织技术通过重新排列发送次序，把连续的突发错打散为多个码字中的少量随机错，从而提高原有检错或纠错机制的效果。

Note

在信道编码或差错控制里，发送端不会直接把原始数据裸着发出去，而是会按照某种编码规则，把一段原始比特映射成一段更长、带有冗余的信息序列。这个编码后的比特串，就叫做一个码字。

校验和则通过对若干数据单元求和并取反来发现错误。IP、TCP、UDP 等协议都使用这种方法，但链路层更常见的强检错方式仍然是 CRC。

3.6 循环冗余校验 CRC

CRC 是数据链路层中最重要的检错方法之一。其基本思想是把待发送比特串看成一个二进制多项式 D(x)，选择一个生成多项式 G(x)，然后构造冗余位 R，使得拼接后的码字可被 G(x) 整除。

Important

CRC 生成多项式的选择本质上是一个检错能力与实现代价之间的设计问题；实际协议通常预先规定标准多项式，而不是在通信双方之间临时任意选取。

发送端流程为：

  1. 对原始数据后附加 r 个 0；
  2. 用模 2 除法除以生成多项式；
  3. 取余数作为冗余码；
  4. 将余数附加到原始数据后发送。

这里是一个例子：CRC例子。

接收端用同一生成多项式对收到的码字再次做模 2 除法：

  • 余数为 0，则接受；
  • 余数非 0，则判为出错并丢弃。
flowchart TD
    M[原始数据 M] --> Z[后附 r 个 0]
    Z --> D[模2除以生成多项式 G]
    D --> R[得到余数 R]
    R --> C["发送码字 M||R"]
    C --> V[接收端再除以 G]
    V --> J{余数是否为0}
    J -- 是 --> OK[接受]
    J -- 否 --> ERR[丢弃]

CRC 的优点在于：

  • 编码效率高；
  • 对突发错误特别有效；
  • 易于用移位寄存器和异或电路实现；
  • 可检测所有 1 位错误、所有 2 位错误、所有奇数位错误以及长度不超过 CRC 位数的突发错误。

3.7 FCS 与 CRC 的关系

链路层帧尾常带有 FCS（Frame Check Sequence，帧检验序列）。FCS 是附加在帧末尾的冗余码，而 CRC 是生成这种冗余码的一种方法。二者不可等同：CRC 是检错方法，FCS 是冗余结果。

4. 可靠传输与流量控制

4.1 从检错到可靠传输

仅有检错还不够。即使接收方能发现错误帧并将其丢弃，发送方仍然需要知道哪些帧没有成功到达。为此，链路层在某些协议中引入确认、重传和窗口控制机制。

需要区分三个概念：

  1. 无比特差错：接收的帧通过了检错；
  2. 无差错接受：所有被接受的帧都可认为正确；
  3. 无差错传输：发送什么就最终收到什么。

CRC 只能帮助实现无差错接受，不能单独实现无差错传输。若要做到可靠传输，必须配合确认与重传机制，即 ARQ。

4.2 停等协议

停等协议是最基本的可靠传输协议。发送方每发送一帧，就停止继续发送，等待接收方确认；若在规定时间内未收到确认，则重传该帧。

sequenceDiagram
    participant A as 发送方
    participant B as 接收方
    A->>B: DATA 0
    B-->>A: ACK 1
    A->>B: DATA 1
    B-->>A: ACK 0

它解决的问题包括：

  • 数据帧出错或丢失；
  • 确认帧丢失；
  • 延迟到达的重复帧。

为避免重复交付，数据帧必须带有序号；确认帧也必须能表明“期望收到的下一个序号”。在最简单的停等协议中，1 比特序号即可循环使用。

发送方的核心逻辑是：

  1. 发送数据帧并缓存副本；
  2. 启动超时定时器；
  3. 收到正确确认后，删除缓存并发送下一帧；
  4. 若超时，则重传缓存帧。

接收方的核心逻辑是：

  1. 收到损坏帧则丢弃；
  2. 收到正确的新帧，则交付上层并回送确认；
  3. 收到正确但重复的帧，则丢弃数据但重发确认。

4.3 停等协议的利用率

设数据帧发送时间为 tf​，传播时延为 tp​，确认帧发送时间为 ta​，则停等协议的链路利用率可表示为：

U=tf​+RTT+ta​tf​​

若忽略处理时间与确认帧发送时间，定义

α=tf​tp​​

则近似有：

U=1+2α1​

可见，当传播时延大而帧很短时，停等协议的大部分时间都耗费在等待确认上，链路利用率很低。它简单，但不适于长时延高带宽链路。

4.4 连续 ARQ 与滑动窗口

为提高链路利用率，发送方可以在收到确认前连续发送多个帧。这就引出了滑动窗口协议。发送端和接收端分别维护发送窗口与接收窗口。发送窗口限制未确认帧的最大数量，接收窗口规定当前允许接收的序号范围。

flowchart TD
    A[发送窗口] --> B[允许连续发送多个未确认帧]
    B --> C[确认到达后窗口前移]
    C --> D[超时则按协议规则重传]

4.5 回退 N（Go-Back-N）

在连续 ARQ 中，若接收方只接收按序到达的帧，则其接收窗口大小为 1。若某一帧丢失，后续虽已到达但因失序而全部丢弃；发送方超时后，从出错帧开始将窗口内相关帧全部重传。这就是 Go-Back-N。

sequenceDiagram
    participant A as 发送方
    participant B as 接收方
    A->>B: DATA 0
    B-->>A: ACK 1
    A->>B: DATA 1
    B-->>A: ACK 2
    A-xB: DATA 2 丢失
    A->>B: DATA 3
    B-->>A: ACK 2
    A->>B: DATA 4
    B-->>A: ACK 2
    A->>B: DATA 5
    B-->>A: ACK 2
    A->>B: 超时后重传 DATA 2,3,4,5

Go-Back-N 的优点是接收端简单，不需要缓存大量失序帧；缺点是在误码或丢帧较多时，会重复重传已经正确到达的帧。

当帧序号使用 n 比特，且接收窗口为 1 时，发送窗口必须满足：

WT≤2n−1

否则可能导致新旧帧序号重叠而无法区分。

4.6 选择重传（Selective Repeat）

选择重传进一步提高了效率。接收端允许接收窗口内的失序帧，并将其缓存，待缺失帧补齐后再按序交付上层。发送方仅重传被确认丢失的特定帧，而不是从某一点开始整段回退。

它减少了不必要的重传，但代价是：

  • 接收端需要缓存失序帧；
  • 协议状态更复杂；
  • 序号空间与窗口大小必须更谨慎地设计。

为避免新旧帧序号在接收窗口中重叠，若帧序号用 n 比特表示，则选择重传中接收窗口必须满足：

WR≤2n−1

这一约束本质上是为了让“仍可能重传的旧帧”和“未来真正的新帧”在序号空间中不发生可见重叠。

4.7 累计确认与捎带确认

为了减少确认开销，链路层常采用两种优化：

  1. 累计确认：确认序号 i 表示序号小于 i 的帧都已正确收到，并期望下一帧为 i。
  2. 捎带确认：当接收方也有数据要发回时，将确认信息附在数据帧中一并发送。

这两种方法在全双工数据交换中尤其常见，可显著降低纯控制帧所占比例。

5. 点对点链路协议实例

5.1 PPP 协议

5.1.1 PPP 的定位

PPP（Point-to-Point Protocol）是最常见的点对点链路层协议之一，广泛用于用户主机到 ISP 的接入链路等场景。它主要解决点对点链路上的封装、透明传输、参数协商和网络层配置问题。

PPP 的设计目标并不是做“功能最强”的链路层协议，而是在点对点链路上提供足够通用且足够简单的封装和协商能力。

5.1.2 PPP 的设计需求

PPP 需要满足以下要求：

  • 简单；
  • 支持成帧和透明传输；
  • 支持多种网络层协议；
  • 能运行于多种类型链路；
  • 能进行差错检测；
  • 能检测链路状态；
  • 能协商 MTU；
  • 能协商网络层地址；
  • 能协商数据压缩等附加功能。

与此相对，PPP 不要求提供：

  • 纠错；
  • 流量控制；
  • 序号与确认；
  • 多点线路支持；
  • 半双工或单工特化支持。

其背后思想是：在点对点链路误码率已较低时，复杂的链路层可靠机制并不划算；而端到端可靠性本来也不应仅由链路层承担。

5.1.3 PPP 的组成

PPP 由三部分组成：

  1. 封装方法：规定如何把网络层分组装入 PPP 帧；
  2. LCP（Link Control Protocol）：用于建立、配置、测试和释放链路；
  3. NCP（Network Control Protocol）：用于协商具体网络层参数，例如 IP 地址配置。
5.1.4 PPP 帧格式

PPP 帧的典型格式如下：

字段长度作用
Flag1 字节帧定界，取值 0x7E
Address1 字节常置为 0xFF，实际不起寻址作用
Control1 字节常置为 0x03
Protocol2 字节标识信息字段承载的上层协议
Information可变承载 IP 数据报、LCP、NCP 等
FCS2 或 4 字节差错检测
Flag1 字节帧结束标志

其中，Protocol 字段常见取值包括：

  • 0x0021：IP 数据报；
  • 0x8021：网络控制协议数据；
  • 0xC021：链路控制协议数据；
  • 0xC023：鉴别数据。
5.1.5 PPP 的透明传输

PPP 的透明传输方式依链路类型而定。

  1. 异步链路：采用字符填充。

    • 0x7E 转义为 (0x7D, 0x5E)；
    • 0x7D 转义为 (0x7D, 0x5D)；
    • 小于 0x20 的控制字符也需按规则转义。

  2. 同步链路：采用零比特填充。
    当连续出现 5 个 1 时插入一个 0，接收端再删除该填充位。

5.1.6 PPP 为什么不提供可靠传输

PPP 不使用序号和确认，不负责链路层重传。原因有三：

  1. 点对点链路通常质量较好，复杂 ARQ 的收益有限；
  2. PPP 承载的往往是 IP 数据报，链路层的可靠并不等于端到端可靠；
  3. PPP 追求实现简单和良好的互操作性。

因此，PPP 只保证无差错接受，不保证链路层上的可靠交付。

5.1.7 PPP 的工作状态

PPP 的工作通常经历如下阶段：

  1. 链路建立：物理连接建立后，由 LCP 协商链路参数；
  2. 鉴别：若配置了鉴别机制，则执行身份认证；
  3. 网络层配置：由 NCP 协商网络层参数，如分配临时 IP 地址；
  4. 链路打开：进入正常数据传输阶段；
  5. 终止：通信结束后，先释放网络层配置，再释放链路与物理连接。
stateDiagram-v2
    [*] --> 静止
    静止 --> 链路建立: 物理层连接建立
    链路建立 --> 鉴别: LCP 配置协商成功
    链路建立 --> 终止: LCP 协商失败
    鉴别 --> 网络层协议: 鉴别成功或无需鉴别
    鉴别 --> 终止: 鉴别失败
    网络层协议 --> 链路打开: NCP 配置完成
    链路打开 --> 终止: 关闭请求/链路故障
    终止 --> 静止: LCP 终止完成

PPP 的这一结构说明，它虽然立足于数据链路层，但实际还触及了物理连接建立与网络层配置，因此是一种具有跨层协商特征的协议。

5.2 HDLC 协议

5.2.1 HDLC 的定位

HDLC（High-Level Data Link Control）是面向比特的同步链路控制协议。它由 ISO 发展而来，目标是为链路层提供高效率、较完整的控制能力。与 PPP 相比，HDLC 更强调面向比特的同步传输、编号、确认以及链路控制。

5.2.2 HDLC 帧格式

HDLC 帧采用如下结构：

字段长度作用
Flag8 bit帧定界，01111110
Address8 bit地址或命令/响应标识
Control8 bit帧类型、序号与控制信息
Information可变用户数据或控制数据
FCS16 bitCRC 校验
Flag8 bit帧结束

HDLC 使用与标志字段相同的 01111110 作为帧边界，因此必须使用零比特填充来保证透明传输。

5.2.3 HDLC 的三类帧

HDLC 根据控制字段的格式，将帧分为三类。

  1. 信息帧 I
    用于承载用户数据，控制字段中包含发送序号 N(S) 和接收序号 N(R)，支持捎带确认。

  2. 监控帧 S
    不携带用户数据，用于单独确认、流量控制与差错恢复。

  3. 无编号帧 U
    用于链路建立、断开、模式设置、复位等链路管理功能。

flowchart LR
    A[HDLC 帧] --> B[I 帧]
    A --> C[S 帧]
    A --> D[U 帧]
    B --> B1[承载用户数据]
    B --> B2["含 N(S)、N(R)"]
    C --> C1[确认/流量控制/重传控制]
    D --> D1[链路管理]
5.2.4 I 帧

I 帧既承载数据，也能通过 N(R) 实现对已收到帧的累计确认。控制字段中的 P/F 位可用于查询与响应控制。

5.2.5 S 帧

S 帧包括四种典型类型：

  • RR（Receive Ready）：接收方准备好继续接收；
  • RNR（Receive Not Ready）：接收方暂时忙，要求对方暂停发送；
  • REJ（Reject）：否定确认，要求从某帧开始回退重传；
  • SREJ（Selective Reject）：选择性否定确认，仅要求重传指定帧。

这说明 HDLC 同时具备流量控制与差错恢复能力，既可以支持类似 Go-Back-N 的回退重传，也可以在特定变种中支持选择重传。

5.2.6 U 帧

U 帧不携带序号，主要用于链路管理，如：

  • SABM：设置异步平衡模式；
  • DISC：断开链路；
  • UA：确认无编号帧；
  • DM：断开模式；
  • RSET：重置序号状态。
5.2.7 HDLC 的校验与工作方式

HDLC 常采用 CRC-CCITT 作为 FCS 生成方法，生成多项式为：

G(x)=x16+x12+x5+1

HDLC 既可用于非平衡配置，也可用于平衡配置。在非平衡配置中，主站控制从站；在平衡配置中，两端均可独立发起控制与数据交换。

从工程角度看，HDLC 代表了“功能完备型”链路层协议：它不仅解决成帧和透明传输，还将可靠传输、流量控制和链路管理完整纳入协议内部。

第三部分 共享链路上的介质访问控制

点对点链路强调“怎样把一帧送好”，而共享链路强调“谁可以在什么时候发送”。当多个结点共用同一传输资源时，链路层的核心问题转化为介质访问控制。

1. 共享信道与 MAC 问题

1.1 共享信道带来的问题

在广播信道中，多个结点共享同一物理介质。若两个或更多结点同时发送，其信号会互相干扰，导致碰撞。共享介质因此带来一个基本问题：谁可以在什么时候使用信道。

这正是 MAC（Medium Access Control，介质访问控制）子层要解决的问题。

1.2 MAC 的目标与分类

理想的 MAC 协议希望做到：

  • 若只有一个结点发送，则它应能独占全部速率 R；
  • 若有 M 个结点同时发送，则各结点平均应获得 R/M；
  • 协议应尽量分布式、简单、少依赖集中控制。

MAC 协议大体可分为三类：

  1. 信道划分：将信道分割为时隙、频带或码字；
  2. 随机接入：允许竞争，发生碰撞后恢复；
  3. 受控接入/无冲突协议：先预约或轮询，再无碰撞发送。
flowchart LR
    A[MAC 协议] --> B[信道划分]
    A --> C[随机接入]
    A --> D[无冲突/受控接入]
    B --> B1[TDMA]
    B --> B2[FDMA]
    B --> B3[CDMA]
    C --> C1[ALOHA]
    C --> C2[时隙 ALOHA]
    C --> C3[CSMA]
    C --> C4[CSMA/CD]
    C --> C5[CSMA/CA]
    D --> D1[位图预约]
    D --> D2[轮询/令牌]
    D --> D3[有限竞争]

2. 信道划分方法

2.1 TDMA 与 FDMA

  • TDMA：把时间划分为轮与时隙，每个站在固定时隙发送。优点是无碰撞；缺点是空闲站点的时隙会被浪费。
  • FDMA：把频谱划分为多个频段，每个站占用一个固定频段。优点是互不冲突；缺点是频段未用时也会闲置。

对于突发业务，静态划分会降低利用率并增大平均延迟，因此需要动态信道分配机制。

2.2 CDMA

在码分多址中，不同站点使用不同码字区分信号，即使在时间上重叠，也可以通过相关性在接收端加以分离。它同样属于先将共享资源逻辑划分开、从而避免直接碰撞的一类方法。

3. 随机接入方法

3.1 ALOHA 与时隙 ALOHA

纯 ALOHA 中，结点有帧就立即发送。由于发送时刻任意，某帧的易损时间是两个帧长，因此其吞吐量为：

S=Ge−2G

最大值在 G=21​ 时取得：

Smax​=2e1​≈18.4

时隙 ALOHA 将时间离散成等长时隙，结点只能在时隙边界发送。这样易损时间缩短为一个时隙，吞吐量提高为：

S=Ge−G

其最大值为：

Smax​=e1​≈36.8

时隙化通过同步换取了更高利用率，但仍会因碰撞而浪费时隙。

3.2 CSMA

CSMA 在发送前先侦听信道，若空闲则发送，若忙则等待，从而降低碰撞概率。典型变种包括：

  • 1-坚持 CSMA；
  • 非坚持 CSMA；
  • p-坚持 CSMA。

CSMA 改进了 ALOHA，但无法彻底消除碰撞，因为传播时延使得两个远端结点可能都认为信道空闲而几乎同时发送。

4. 受控接入与无冲突协议

当业务负载较高、对实时性有要求时，可采用预约或轮询等无冲突协议：

  • 位图协议：每个站在自己的预约时隙中发送一个比特，表明自己是否有数据待发；
  • 轮询/令牌：由主结点依次邀请发送，或令牌在结点间循环；
  • 有限竞争协议：将站点分组，在不同竞争时隙内限制竞争者数量，以兼顾低负载时的低延迟和高负载时的高利用率。

第四部分 有线局域网中的碰撞检测与以太网

共享链路上的一般 MAC 机制，在有线局域网中最典型的实现就是经典以太网。它将“先侦听、再发送、边发边检、碰撞后退避”这一思想具体化为 CSMA/CD。

1. 局域网与经典以太网

1.1 局域网与以太网

局域网的典型特点是：地理范围有限、常由单一组织拥有、适合资源共享。以太网是最成功的局域网技术之一。历史上它最初采用总线拓扑，多个站共享一根同轴电缆；后续演化出双绞线星形布线，但其经典 MAC 思想来自共享总线环境。

1.2 以太网的基本设计思想

经典以太网采取了两项重要设计：

  1. 无连接、尽力而为的服务
    发送前不建立连接，不对帧编号，不要求接收方回确认。若帧出错则直接丢弃，是否重传交给上层决定。

  2. CSMA/CD
    在共享有线介质上，发送前载波监听，发送中碰撞检测，冲突后退避重传。

2. 碰撞与 CSMA/CD

2.1 为什么会发生碰撞

即便发送前侦听到信道空闲，也不意味着远端没有站在几乎同一时刻开始发送。由于信号传播需要时间，结点 A 尚未来得及听到结点 B 的信号时，两者都可能已开始发送。这种由传播时延造成的不确定性，使得边发边听成为必要条件。

2.2 CSMA/CD 的基本过程

flowchart TD
    A[准备发送] --> B{信道空闲?}
    B -- 否 --> C[持续侦听]
    C --> B
    B -- 是 --> D[开始发送并同时检测碰撞]
    D --> E{检测到碰撞?}
    E -- 否 --> F[继续发送直到完成]
    E -- 是 --> G[立即停止发送]
    G --> H[执行二进制指数退避]
    H --> A

其关键点在于：

  • 载波监听：发送前侦听信道状态；
  • 碰撞检测：发送中检测总线电压异常增大；
  • 立即终止：一旦发现碰撞，停止发送以减少浪费；
  • 随机退避：等待随机时延后重传。

2.3 争用期与最短帧长

设端到端单程传播时延为 τ，则发送方最迟在 2τ 内一定能够知道本次发送是否发生碰撞。这个时间称为争用期或碰撞窗口。

经典 10 Mb/s 以太网规定争用期为 51.2 μs，在此期间可发送 512 bit，即 64 字节。因此：

  • 若一帧在前 64 字节内未碰撞，则后续也不会再发生碰撞；
  • 若帧长小于 64 字节，发送方可能在碰撞信息返回前已发完，无法检测碰撞。

所以以太网规定最短有效帧长为 64 字节。小于该长度的异常帧视为碰撞产生的无效帧。

2.4 二进制指数退避

碰撞后的重传不是立即进行，而是等待随机退避时间。若第 k 次重传前发生碰撞，则从区间

[0,2k−1]

中随机选一个整数 r，实际退避时间为 r 个争用期。通常取

k=min(重传次数,10)

若重传达到 16 次仍失败，则放弃该帧并向高层报告。

这一策略的作用是：碰撞越频繁，竞争者越多，退避窗口就越大，从而降低后续再次碰撞的概率。

2.5 以太网利用率与参数 a

设帧发送时间为 T0​，单程传播时延为 τ，定义

a=T0​τ​

则 a 越小，说明传播时延相对帧发送时间越小，以太网利用率越高。要减小 a，可以：

  • 缩短网络物理范围，减小 τ；
  • 增大帧长，增大 T0​。

这也是经典共享总线以太网对覆盖范围和最短帧长都施加限制的根本原因。

2.6 CSMA/CD 与 ARQ 的边界

CSMA/CD 与点对点链路上的 ARQ 目标不同：

  • CSMA/CD 关注共享介质上的冲突检测与接入控制，发送失败主要理解为碰撞；
  • ARQ 关注确认、重传和可靠传输，发送失败主要理解为差错或丢失。

因此，CSMA/CD 并不等于可靠传输协议。它负责减少碰撞浪费，而真正的端到端可靠性交由高层承担。

3. 从共享总线到扩展以太网

3.1 从总线到星形以太网

随着双绞线和集线器的发展，以太网逐渐从单根总线演化为星形拓扑。10BASE-T 使用双绞线连接主机与集线器，物理上是星形，逻辑上仍是共享介质：集线器只是在物理层把一个端口收到的信号复制到所有端口。

3.2 集线器与交换机

  • 集线器工作在物理层，本质上是多端口中继器。它不缓存帧，不识别 MAC 地址，也不做碰撞检测；与其相连的所有主机仍处于同一个碰撞域。
  • 交换机工作在链路层，能够缓存帧、学习 MAC 地址并按端口有选择地转发。交换机将每条链路分割为独立碰撞域，使多个结点能够同时通信。

3.3 交换带来的变化

交换式以太网的关键收益包括：

  1. 多对主机可并发传输；
  2. 链路彼此独立，不再共享同一碰撞域；
  3. 可支持全双工通信；
  4. 更安全，普通端口无法像集线器那样直接看到所有帧。
flowchart LR
    A[主机 A] --> S[交换机]
    B[主机 B] --> S
    C[主机 C] --> S
    D[主机 D] --> S
    A <-->|独立链路| S
    B <-->|独立链路| S
    C <-->|独立链路| S
    D <-->|独立链路| S

3.4 MAC 地址与以太网帧

现代以太网主要采用 Ethernet V2 帧格式。MAC 地址为 48 位，用于链路层寻址。目的地址可以是：

  • 单播地址；
  • 组播地址；
  • 广播地址。

交换机与网卡都依赖 MAC 地址完成本地网络内的转发与接收判断。

以太网 V2 帧的关键特点包括：

  • 类型字段指示上层协议；
  • 数据字段最少 46 字节、最多 1500 字节；
  • 帧总长最少 64 字节；
  • 采用 FCS 检错；
  • 对无效帧直接丢弃，不负责重传。

3.5 高速以太网

从 10 Mb/s 演化到 100 Mb/s、1 Gb/s 及更高速率后，以太网仍尽量保持帧格式兼容。对半双工千兆以太网，为维持冲突检测所需的最小发送时间，引入了载波延伸和帧突发等机制；而在全双工模式下，由于不再存在共享碰撞域，CSMA/CD 可以被取消。

第五部分 无线局域网中的碰撞避免

无线局域网同样面对共享信道问题，但它不能像有线以太网那样边发边检，因此在原理上从“碰撞检测”转向“碰撞避免”。

1. 无线局域网的特点

无线局域网采用共享无线信道，既有广播信道的竞争问题，又有无线传播的特殊困难：

  • 信号衰减明显；
  • 受其他设备干扰；
  • 存在多径传播；
  • SNR 随位置移动而变化；
  • 很难像有线那样边发边听。

因此，802.11 不采用 CSMA/CD，而采用 CSMA/CA，即载波监听多路访问/碰撞避免。

2. WLAN 的组成方式

802.11 典型组织方式包括：

  1. 基础设施模式
    以接入点 AP 为中心，主机先与 AP 关联，再通过 AP 接入有线网络。

  2. ad hoc 模式
    无固定基础设施，节点之间直接通信；若要多跳，还需要网络层路由机制。

flowchart LR
    subgraph BSS1[基础设施模式]
        H1[无线主机] --> AP1[AP]
        H2[无线主机] --> AP1
        AP1 --> NET[有线网络/Internet]
    end

3. 为什么 WLAN 不采用 CSMA/CD

3.1 无法边发边听

无线环境中，发送节点发出的信号功率远大于自己能接收到的远端信号，难以在发送同时可靠检测碰撞；同时，隐蔽结点问题意味着两个发送方可能彼此听不见，但其信号却会在某个接收点相撞。因此，碰撞检测难以实现，只能改为碰撞避免。

3.2 隐蔽结点与暴露结点

  • 隐蔽结点：两个发送方彼此不可听见，却都能向同一接收方发送，导致接收方处发生碰撞。
  • 暴露结点：某发送方侦听到邻居发送而错误地放弃发送，尽管其发送本不会影响目标接收方，导致信道利用率下降。

隐蔽结点是 802.11 设计 RTS/CTS 机制的重要动机；暴露结点则说明，仅凭本地监听不能完全描述无线信道中的干扰关系。

4. 802.11 的基本 CSMA/CA 过程

当结点要发送数据时：

  1. 持续监听信道；
  2. 若信道空闲持续达到 DIFS，则准备发送；
  3. 若信道忙，则启动随机退避计时器；
  4. 退避计时器仅在信道空闲时递减，信道忙则冻结；
  5. 计时归零后发送数据；
  6. 接收端若正确收到，则在 SIFS 后返回 ACK；
  7. 若发送端未收到 ACK，则推测此次发送失败，扩大退避窗口并重试。
sequenceDiagram
    participant S as 发送方
    participant R as 接收方
    S->>S: 监听信道
    S->>S: 信道空闲达到 DIFS
    S->>R: DATA
    R-->>S: SIFS 后发送 ACK

ACK 在无线链路中十分重要，因为发送端不能通过碰撞检测判断发送是否成功，只能通过“是否收到 ACK”来间接推断。

5. RTS/CTS 与虚拟载波侦听

5.1 RTS/CTS 预约机制

为缓解隐蔽结点问题，802.11 可在发送大数据帧前先进行 RTS/CTS 握手：

  1. 发送方先发 RTS，请求占用信道；
  2. 接收方若准备好接收，则发 CTS；
  3. 周围结点听到 RTS 或 CTS 后，设置 NAV，推迟发送；
  4. 发送方再发数据帧；
  5. 接收方返回 ACK。
sequenceDiagram
    participant A as 发送方
    participant B as 接收方
    participant N as 邻近结点
    A->>B: RTS
    B-->>A: CTS
    Note over N: 根据 RTS/CTS 设置 NAV，延迟接入
    A->>B: DATA
    B-->>A: ACK

RTS/CTS 不能完全消灭碰撞，但它把“可能发生碰撞的对象”从长数据帧缩短成更短的预约帧，从而降低浪费。

5.2 物理与虚拟载波侦听

802.11 同时使用两种“听”信道的方法：

  • 物理载波侦听：检测信号能量、相关比特或 RSS；
  • 虚拟载波侦听：根据 RTS/CTS 等帧中的持续时间字段设置 NAV，逻辑上认为信道已被占用。

此外，802.11 还定义了不同优先级的帧间间隔，如 SIFS、PIFS、DIFS，以保证 ACK、CTS 等短控制帧优先于新的普通数据竞争。

第六部分 交换式局域网的组网机制

前面的内容讨论的是“一条链路怎么传”和“共享信道怎么接入”。当局域网由多条链路和多台交换设备构成时，还需要进一步解决转发、环路控制和广播域划分等组网问题。

1. 交换机的自学习与转发

1.1 交换机为何需要自学习

交换机面临的核心问题是：收到一个以太网帧后，应从哪个端口转发出去。若预先人工配置每个主机在哪个端口，网络规模稍大就会难以维护。因此，以太网交换机普遍采用即插即用的自学习机制。

1.2 交换表

交换机维护一张交换表，典型表项包括：

  • MAC 地址；
  • 对应端口；
  • 时间戳或老化信息。

其基本原则是：

  • 基于源地址学习：从收到帧的源 MAC 地址，知道“该主机位于哪个端口后面”；
  • 基于目的地址转发：查表决定该帧应从哪个端口发出。

1.3 自学习与转发流程

flowchart TD
    A[交换机收到帧] --> B[读取源 MAC 与入端口]
    B --> C[更新/新增交换表项]
    C --> D[读取目的 MAC]
    D --> E{目的 MAC 是否已知?}
    E -- 是 --> F{出端口是否等于入端口?}
    F -- 是 --> G[过滤/丢弃]
    F -- 否 --> H[仅向对应端口转发]
    E -- 否 --> I[除入端口外泛洪]

其含义是：

  • 若已知目的主机所在端口，则定向转发；
  • 若未知，则向除入端口外的所有端口泛洪；
  • 若目的主机就在入端口所在网段，则无需转发。

这一机制保证交换机能在不依赖人工配置的情况下逐步学习整个二层网络的拓扑。

Note

由交换机、网桥、网卡等二层设备连接起来，并且通过 MAC 地址转发帧的那个局部网络。

1.4 多交换机环境

在多交换机互联时，自学习仍然成立。某主机发往未知目的时，帧会逐级泛洪；当对方回应后，中间各交换机就能沿回程路径学习到源主机的位置。经过若干轮通信，交换表逐渐收敛，后续通信转为定向转发。

flowchart TD
    A[主机 A 发帧给 B] --> B1[交换机收到帧]
    B1 --> C1[先记住: A 在这个入端口后面]
    C1 --> D1{知道 B 在哪吗?}
    D1 -- 不知道 --> E1[向除入端口外的端口泛洪]
    E1 --> F1[B 收到后回复]
    F1 --> G1[沿途交换机再次学习: B 在回复帧进入的那个端口后面]
    G1 --> H1[以后 A 和 B 通信时可定向转发]

2. 生成树协议

2.1 为什么需要生成树

为了提高可靠性，交换机之间往往会部署冗余链路。然而冗余链路会带来二层环路，造成：

  • 广播帧无限循环；
  • 未知单播泛洪不断复制；
  • 交换表频繁震荡；
  • 网络资源被耗尽。

因此，二层网络必须在“物理上允许有环”与“逻辑上转发无环”之间取得平衡。

2.2 生成树的基本思想

生成树协议通过交换配置信息，在交换机图中选择一棵覆盖所有交换机的无环树，仅保留树上的转发链路，将其他冗余链路置于阻塞状态。

最小生成树：用尽可能小的总代价，把所有点连起来，但又不形成多余的环。 一般的最小生成树是在图里“选边去环”； 多交换机里的生成树协议是在网络里“阻塞冗余链路去环”。

其核心步骤包括：

  1. 选举根桥；
  2. 每个交换机选择到根桥的最短路径；
  3. 为每个网段确定指定桥；
  4. 将非树边对应端口阻塞。
flowchart LR
    B1[交换机 B1] --- B2[交换机 B2]
    B2 --- B3[交换机 B3]
    B3 --- B4[交换机 B4]
    B4 --- B1
    B2 --- B4

flowchart LR
    B1[根桥 B1] --- B2[交换机 B2]
    B2 --- B3[交换机 B3]
    B3 --- B4[交换机 B4]
    %% 其他冗余边被阻塞，不参与转发

交换机之间通过特定组播地址交换配置信息，依据桥标识和路径代价分布式地构造生成树。这样既保留了物理冗余带来的容错能力，又避免了逻辑转发环路。

3. VLAN

3.1 VLAN 的动机

在没有 VLAN 时，同一个交换基础设施通常构成一个大的二层广播域。这样会带来三个问题：

  1. 广播流量范围过大，ARP、DHCP、未知单播等会扩散到整个 LAN；
  2. 安全与隔离不足，不同部门主机处于同一广播域；
  3. 网络组织缺乏灵活性，物理位置变化会直接影响二层连接关系。

VLAN 通过在同一物理交换网络上划分多个逻辑局域网，解决上述问题。

3.2 基于端口的 VLAN

最常见的是基于端口的 VLAN。交换机管理软件将若干端口划入同一 VLAN，使得这些端口之间互相可达，而不同 VLAN 之间二层广播被隔离。

flowchart LR
    SW[交换机]

    P1[端口1]
    P2[端口2]
    P3[端口3]
    P4[端口4]
    P5[端口5]
    P6[端口6]

    V10[VLAN 10]
    V20[VLAN 20]

    SW --> P1
    SW --> P2
    SW --> P3
    SW --> P4
    SW --> P5
    SW --> P6

    P1 --> V10
    P2 --> V10
    P3 --> V10
    P4 --> V20
    P5 --> V20
    P6 --> V20

VLAN 的作用，就是把“直接通信的本地范围”划分出来；同 VLAN 内直接转发，不同 VLAN 间交给三层设备。

因此，VLAN 是链路层隔离机制，而跨 VLAN 通信属于网络层功能。

3.3 跨交换机的 VLAN 与 802.1Q

当同一 VLAN 跨越多个交换机时，交换机之间的中继链路必须知道“这个帧属于哪个 VLAN”。为此，引入 IEEE 802.1Q 标记。

802.1Q 在以太网帧中增加 4 字节标签，其中包含：

  • 协议标识；
  • 优先级；
  • 12 位 VLAN ID。

交换机端口分为：

  • Access 端口：连接普通主机，不向主机暴露 VLAN 标签；
  • Tag Aware / Trunk 端口：交换机之间传输带标签的帧。

主机侧看不到标签，交换机之间靠标签识别 VLAN。

flowchart TD
    H1[主机 H1] -->|普通帧| A1[ S1 的 Access 口<br/>属于 VLAN 10 ]
    A1 --> B1[ S1 内部认定：这是 VLAN 10 的帧 ]
    B1 --> C1[ S1 的 Trunk 口<br/>加上 802.1Q 标签: VLAN 10 ]
    C1 --> C2[ S2 的 Trunk 口<br/>读到标签: VLAN 10 ]
    C2 --> D1[ S2 只在 VLAN 10 内转发 ]
    D1 --> E1[ S2 的 Access 口<br/>去掉标签 ]
    E1 -->|普通帧| H2[主机 H2]

这意味着：

  • 帧在交换机内部及中继链路上可以保留 VLAN 身份；
  • 到达接入端口时，标签会被剥离；
  • 同一物理基础设施上可承载多个逻辑广播域。

3.4 VLAN 的作用

VLAN 的主要价值包括：

  • 缩小广播域；
  • 提高安全性与隔离性；
  • 按组织或业务逻辑而非物理位置组织网络；
  • 改善带宽利用与管理灵活性。

总结

数据链路层的核心任务是把网络层分组落实为具体链路上的逐跳传输。围绕这一目标，它需要解决三类彼此相关但层次不同的问题。

第一类问题是点对点链路上的帧传输问题。这一部分围绕一条具体链路展开，重点讨论成帧、透明传输、差错检测、流量控制以及必要时的可靠传输。PPP 代表了简洁的点对点封装与协商机制；HDLC 代表了功能更完备的同步链路控制机制。

第二类问题是共享链路上的介质访问控制问题。这一部分围绕多个结点如何共享同一传输介质展开，包括信道划分、随机接入、碰撞检测与碰撞避免。经典以太网通过 CSMA/CD 处理有线共享介质中的冲突；无线局域网通过 CSMA/CA、ACK 与 RTS/CTS 处理无线竞争环境。

第三类问题是交换式局域网中的组网问题。当数据链路层从“共享总线”发展到“交换网络”后，交换机通过自学习转发实现即插即用，通过生成树协议消除二层环路，通过VLAN在同一物理网络上建立多个逻辑广播域。

从工程上看，数据链路层并不是单一协议，而是一组围绕“逐跳传输”“共享介质接入”和“二层组网”展开的机制集合。它既要向下适应具体介质的物理特性，又要向上为网络层提供稳定、清晰而高效的逐跳传输服务。
