lecture5 - 内存管理概述

哈，终于讲到内存管理了！如果说进程调度是 OS 的”大脑”，那内存管理就是它的”仓库管理员”——既要让每个进程都有地方放东西，又要防止它们互相串门、乱翻别人的箱子。这一讲我们先打好基础，把内存管理的演化脉络和基本方案梳理清楚。虚拟内存是下一讲的内容，别急，咱们一步步来。

flowchart TB
    R[寄存器]
    L1[L1 Cache]
    L2[L2 Cache]
    L3[L3 Cache]
    M[内存]
    D1[本地磁盘]
    D2[远程磁盘]

    R --> L1 --> L2 --> L3 --> M --> D1 --> D2

1. 内存管理：不只是”分配空间”那么简单

很多人一开始会误以为内存管理 = 虚拟内存，其实远不止。虚拟内存只是内存管理发展到后期的一个”高级玩法”。早期的内存管理就已经在操心这些事：程序怎么装进内存？地址怎么对应到物理位置？多个进程怎么不撞车？空闲的物理内存怎么管？

你可以把内存管理的演化看成一部”从简单粗暴到精致抽象”的进化史：

flowchart TD
    A[早期物理内存直接使用] --> B[单程序连续装入]
    B --> C[多道程序设计]
    C --> D[分时共享]
    D --> E[地址空间抽象]
    E --> F[虚拟内存]

程序平时躺在磁盘里是个”文件”，只有装入内存后才能变成”进程”跑起来。多道程序和分时共享的出现，让多个程序同时要挤在内存里，这就逼出了内存管理的三大基础职责：

  1. 给进程造一个”地址空间”——让每个进程都以为自己独占一片天地；
  2. 把地址空间映射到物理内存——暗地里决定进程实际住哪；
  3. 保护隔离与权限边界——防止进程 A 偷偷读进程 B 的数据，也防止用户程序乱碰内核区域。

Intuition

想想大学宿舍：每个学生（进程）都有自己的房间号（逻辑地址），但后勤处（MMU）掌握着真实的楼号和房间号（物理地址）。更重要的是，你的钥匙打不开别人的门——这就是内存保护。

局部性原理是这一切的硬件背景。程序任何时候只会频繁访问一小部分代码和数据，所以现代系统把寄存器、Cache、内存、磁盘组织成金字塔形的存储体系，用速度换容量、用成本换性能。内存管理正好处在这个金字塔的中间层，承上启下。

2. 地址空间：进程的”专属地图”

地址空间是进程能看到的那片内存，不是真正的物理内存。它最大的功劳是把”程序用什么地址”和”东西实际放哪”彻底解耦。这样一来，哪怕程序被装到不同的物理位置，它自己完全无感知。

地址空间解决了多道环境下的三个核心矛盾：

  • 逻辑连续 vs 物理离散：程序眼里地址是连续的，但底层可以东一块西一块；
  • 页面常驻 vs 可换出：有些页现在在内存里，有些可能暂时被赶到磁盘上；
  • 一次性全装 vs 分次装入：不需要等整个程序都加载完就能开始跑。

一个典型的进程地址空间长这样（从低地址到高地址）：

flowchart TB
    K[高地址: 内核相关映射]
    T[trampoline / trapframe 或陷入辅助页]
    M[共享库与内存映射区]
    H[堆: 运行期向高地址扩展]
    D[数据段 / BSS]
    C[代码段 / 只读数据]
    S[栈: 通常向低地址扩展]
    U[低地址起点]

    K --> T --> M --> H --> D --> C --> S --> U

各部分的作用：

  • 代码段 / 只读数据：来自可执行文件或共享库，只读；
  • 数据段 / BSS：全局变量、静态变量，已初始化的和未初始化的分开放；
  • 堆：动态内存分配的地盘，malloc / new 从这里要空间，向高地址生长；
  • 栈：函数调用、局部变量、返回地址，向低地址生长；
  • 映射区：装载共享库、映射文件；
  • 高地址辅助页：比如 xv6 里的 trampoline 和 trapframe，负责用户态和内核态的”接力赛”。

Note

在 xv6 里，trampoline 页存的是陷入/返回时的过渡代码，trapframe 页保存用户上下文。你调用系统调用时，CPU 就是借着这两页完成”交接棒”的。

活跃页面、常驻集与工作集

这三个词面试/考试经常考，别搞混了：

  • 活跃页面：最近被访问过、对当前执行直接有用的页面；
  • 常驻集：某一时刻实际留在物理内存中的页面集合；
  • 工作集：一个时间窗口内被访问过的页面集合，用来刻画进程的局部性范围。

Trap

常驻集是物理状态（现在内存里有哪些），工作集是行为统计（最近用了哪些），活跃页面是当前有效性（现在马上要用哪些）。三者不是一个东西！

3. 装载、绑定与地址重定位

程序从源码到运行要经历四个阶段：编译 → 链接 → 装载 → 运行。

flowchart TD
    A[源程序] --> B{编译}
    B --> C[目标文件]
    C --> D{链接}
    D --> E{装载模块}
    E --> F[装入内存]
    F --> G{运行}

绑定就是把程序里的地址和某个实际内存位置”挂钩”。绑定可以发生在不同时期：

  • 编译期：直接生成绝对地址（早年的单道系统这么干）；
  • 链接期：把多个目标文件拼在一起，地址相对模块起点；
  • 装载期：装入内存时一次性修正地址（静态重定位）；
  • 运行期：每次访问内存时动态换算（动态重定位）。

Intuition

编译期绑定就像高考报志愿时直接分配宿舍号；链接期绑定是知道你在哪个楼但不知道具体房间；装载期绑定是入住那天才定房间号；运行期绑定则夸张到每次回宿舍都要现查房间号——但好处是你随时可以被”搬宿舍”而不用改生活习惯。

逻辑地址（也叫虚拟地址）是程序自己用的地址，通常从 0 开始编；物理地址是内存条上真实的地址。地址重定位就是把逻辑地址翻译成物理地址的过程。

静态重定位在装入时一次性完成，实现简单，但程序装好后就不能动了。 动态重定位需要硬件支持，处理器发出的逻辑地址由 MMU 现场换算成物理地址：

PA=R+LA

其中 R 是重定位寄存器，LA 是逻辑地址。

flowchart LR
    CPU[CPU发出逻辑地址 LA] --> MMU[MMU]
    R[重定位寄存器 R] --> MMU
    MMU --> PA[物理地址 PA = R + LA]
    PA --> MEM[物理内存]

动态重定位是现代操作系统的基石——它让进程可以被移动、被换出、被紧凑，而不需要修改程序本身。

4. 地址保护与内存管理目标

重定位解决的是”去哪找”，保护解决的是”能不能找”。多道程序环境下，进程之间必须互相隔离。

最基本的保护机制是基址寄存器 + 界限寄存器：

  • 基址寄存器（base）：进程可访问区域的起点；
  • 界限寄存器（limit）：进程可访问区域的长度。

每次访存时，硬件自动检查：

base≤addr<base+limit

不满足就触发越界异常，陷入操作系统处理。

flowchart TD
    A[访问地址 addr] --> B{addr >= base?}
    C[基址寄存器 base] --> B
    B -->|否| F[地址越界]
    B -->|是| D{addr < base + limit?}
    C --> D
    E[界限寄存器 limit] --> D
    D -->|是| H[允许访问内存]
    D -->|否| F
    F --> G[陷入操作系统]

Note

基址和界限寄存器只能用特权指令修改，所以用户程序没法自己”扩权”。保护边界最终由内核把守。

内存管理的三大目标可以概括为：

  1. 透明性：进程看到的地址空间稳定、独立，和物理布局无关；
  2. 效率：访问延迟低、管理开销小；
  3. 保护：防止越界、非法访问，同时支持可控共享。

5. 物理内存空闲空间管理

前面讲的是”进程看到什么”，现在讲讲”底层怎么管”。物理内存的空闲空间需要数据结构来跟踪，常见的有：

  • 位图：每个分配单元对应一位，0/1 表示空闲/已用；
  • 空闲区表：记录起始地址、长度、状态；
  • 空闲块链表：把空闲块串起来，有隐式链表、显式链表、分离空闲链表等变体。

Note

如果你做过 CS:APP 的 malloc lab，对这些应该很熟悉了！

评价物理内存管理效果，主要看两个指标：

  • 资源利用率：尽量减少碎片；
  • 性能：分配和回收要快，CPU 开销要低。

碎片分为两种：

  • 内碎片：已经分配给某个进程/对象，但内部没用完的浪费空间；
  • 外碎片：总空闲量够大，但被切成太多小块，无法满足大请求。

常见的分配策略：

  • 首次适配（First Fit）：找第一个够大的空闲区，实现简单、平均性能好；
  • 下次适配（Next Fit）：从上次找到的位置继续找；
  • 最佳适配（Best Fit）：找能满足要求的最小空闲区，容易产生大量小碎片；
  • 最差适配（Worst Fit）：从最大空闲区里切，希望保留中等块，但可能迅速破坏大块连续空间。

回收时的关键是合并相邻空闲区。释放一块内存后，它和前后邻居可能有四种关系：只和前相邻、只和后相邻、前后都相邻、前后都不相邻。必须正确合并，否则外碎片会越积越多。

5.1 伙伴系统

伙伴系统是内核物理内存管理中非常经典的一种方案。它的思路很巧妙：把内存按 2 的幂次分层管理。

Intuition

想象你有一摞大小不同的收纳盒：1L、2L、4L、8L……你要装 3L 的东西，就找一个 4L 的盒子。如果手头只有 8L 的，就把它对半剪成两个 4L 的（这俩就是”伙伴”），然后再对半剪成两个 2L 的——不行，2L 装不下 3L，所以停在 4L 那一层，分配一个 4L 盒子。释放时，如果它的”伙伴”也空着，俩人就合并回 8L，再检查上一级能不能继续合并。

请求大小为 s 时，找到满足 2u−1<s≤2u 的最小块 2u 进行分配。如果当前块太大，就不断二分，直到大小合适。释放时，如果伙伴也空闲，就向上合并：

flowchart TD
    A[收到大小请求 s] --> B[找到最小满足的 2^u 块]
    B --> C{块是否过大}
    C -->|是| D[一分为二得到一对伙伴]
    D --> C
    C -->|否| E[分配块]
    E --> F[释放块]
    F --> G{伙伴是否空闲}
    G -->|是| H[向上合并]
    H --> G
    G -->|否| I[插回对应空闲链表]

伙伴系统的优点是分裂和合并规则非常简单，合并判定特别明确；缺点是会按 2 的幂次取整，产生内碎片。Linux 底层的页级物理内存管理就是基于伙伴系统的思想实现的。

5.2 SLAB、SLUB 与 SLOB

伙伴系统适合分配页级或较大的内存块，但内核里经常要频繁分配小对象，比如 inode、dentry、网络缓冲描述符、任务结构体等等。如果每次都找伙伴系统要一页，粒度太粗，浪费严重。

于是SLAB 分配器登场了。它在伙伴系统提供的大块内存之上，再建一层”对象缓存”：

  • 把内存按对象大小分成不同的”缓存池”；
  • 分配时直接从对应缓存池里拿一个空闲对象；
  • 释放时把对象还回缓存池，而不是还给伙伴系统；
  • 对频繁创建/销毁的对象，缓存能大幅减少初始化和碎片开销。
flowchart TB
    A[伙伴系统提供页级块] --> B[SLAB / SLUB / SLOB]
    B --> C[按大小类建立对象缓存]
    C --> D[小对象分配]
    D --> E[对象释放回缓存]

SLAB 的问题在于结构复杂、队列多、元数据开销大，高并发下还容易锁竞争。所以 Linux 后来发展出了：

  • SLUB：简化版 SLAB，数据结构更简单，锁争用更少，在 NUMA 和多核环境下表现更好；
  • SLOB：极简版，追求最小开销，适合嵌入式等小内存场景。

Intuition

伙伴系统是”批发部”，SLAB/SLUB/SLOB 是”零售店”。批发部按大箱（页）卖，零售店把大箱拆开，按小件（对象）卖。SLAB 是连锁大超市（品类全但管理复杂），SLUB 是便利店（精简高效），SLOB 是路边小卖部（极简低成本）。

Trap

SLAB、SLUB、SLOB 的区别不是”谁管理小对象、谁管理大对象”——它们都管理小对象。区别在于管理策略对复杂度、缓存局部性、并发性能的取舍不同。

6. 基本内存管理方案

这一节的加载单位都是整个进程。也就是说，程序是以进程为单位被创建、装入和调度的。下面我们按历史演进顺序，看看六种基本方案。

6.1 单一连续区

最早期的方案，简单粗暴：内存里只装一个用户程序，这个程序独占全部内存，通常装在固定位置。

  • 优点：几乎不需要地址转换，实现极简；
  • 缺点：没有并发能力，内存利用率极低，OS 和用户程序的边界也很模糊。

Note

这种方案现在只存在于教科书和历史博物馆里，但它的存在让我们明白：内存管理的第一课是”怎么让多个程序共处一室”。

6.2 固定分区

把内存预先切成若干固定大小的区域（分区），每个分区装一个进程。分区大小可以相同也可以不同，但运行期间不变。

  • 优点：支持有限并发，实现简单；
  • 缺点：进程小于分区时，多余空间彻底浪费——这就是内碎片。

输入队列的组织方式有两种：给每个分区配独立队列，或者所有进程排一个统一队列再按需分派。

6.3 可变分区

不再预先切块，而是根据进程实际需求，从当前空闲空间中”切”出一块连续区。剩下的部分变成新的空闲区。

  • 优点：短期内比固定分区省空间；
  • 缺点：随着分配和回收反复进行，内存里会出现很多小洞——这就是外碎片。总空闲量可能够，但没有一块连续区域能满足大进程的需求。

紧凑技术（Compaction）就是为解决这个问题而生的：把进程在内存里”搬家”，把小洞挤成一大块连续空闲区。前提是系统支持动态重定位，否则搬家后地址全乱了。

flowchart TD
    A[空闲连续区] --> B[按进程需求切分]
    B --> C[形成已分配区]
    B --> D[形成剩余空闲区]
    D --> E[多次分配与释放]
    E --> F[出现外碎片]
    F --> G[紧凑: 移动进程并合并空闲区]

Trap

内碎片是”分到的碗里吃不完”，外碎片是”碗里总饭量够但碗都太小了”。固定分区主要产生内碎片，可变分区主要产生外碎片。

6.4 页式内存管理

分页是内存管理史上的一次重大突破。它把进程地址空间切成大小相等的 page，把物理内存切成大小相等的 page frame。逻辑上相邻的 page，在物理内存里可以随便落在哪个 page frame 里，不要求连续。

分页彻底摆脱了对”大块连续物理内存”的依赖。

系统用 page table 记录 page number 到 page frame number 的映射，PTE（Page Table Entry）就是表里的每一条记录。处理器拿到逻辑地址后，自动拆成 page number 和 page offset，查页表得到 frame number，再和 offset 拼成物理地址。

每个进程通常有自己的 page table，页表本身存在内存里。在 x86 架构中，当前页表的根地址由 CR3 寄存器指定。

flowchart TD
    LA[逻辑地址] --> SPLIT[拆分为 page number p 与 page offset d]
    SPLIT --> PT[用 p 查 page table]
    PT --> PTE[读取 PTE]
    PTE --> PF[得到 page frame number f]
    PF --> JOIN[拼接 f 与 d]
    JOIN --> PA[物理地址]
    PA --> MEM[物理内存]

分页基本消除了外碎片，但 page 大小固定，进程末尾的最后一个 page 往往用不满，所以会产生内碎片。page size 的选择是典型的折中：

  • page 大：页表小，TLB 命中率高，但内碎片大；
  • page 小：内碎片小，但页表大，管理开销高。

常见 page size 有 4KB、4MB 等。

Intuition

分页就像把一本书拆成同样大小的活页纸，每页纸可以随便塞进抽屉（page frame）的哪个位置。目录（page table）告诉你第几页纸在哪个抽屉里。这样你不需要一个大抽屉装整本书，只要有足够多的小抽屉就行。

6.5 段式内存管理

分段不像分页那样”一刀切”，而是按程序的逻辑结构划分：主程序段、子程序段、数据段、栈段、数组段等等。段长不等，段号有语义。

物理内存里，每个段仍需要分配连续空间。段表记录段号、段首地址和段长。地址转换时，用段号查段表得到段首地址，再加上段内偏移：

flowchart TD
    LA[逻辑地址] --> SPLIT[拆分为 段号 s 与段内偏移 d]
    SPLIT --> ST[查段表]
    ST --> BASE[得到段首地址 base]
    BASE --> ADD[base + d]
    ADD --> PA[物理地址]
  • 优点：更接近程序员的逻辑视图，天然适合模块化共享、保护和独立增长；
  • 缺点：段长可变，仍然存在外碎片和连续分配压力。

Example

编译器给你生成代码时，很自然地会把代码、全局数据、栈分开。分段就是顺应这种自然结构。但操作系统更关心”怎么管方便”，所以分页更受 OS 青睐。x86 的段页式其实是一种妥协：对用户暴露段，对系统内部分页。

6.6 段页式内存管理

段页式想”鱼和熊掌兼得”：对外保留段的逻辑结构，对内用分页解决连续分配问题。

具体做法是：地址空间先按段组织，每个段再分页。段表里存的不是段首物理地址，而是该段的页表起始位置和长度。转换流程分两步：

  1. 逻辑地址先拆成 段号 + 段内地址；
  2. 段内地址再拆成 页号 + 页内偏移；
  3. 查段表找到页表，查页表找到 frame number，最后拼出物理地址。
flowchart TD
    A[逻辑地址] --> B[段号 s + 段内地址]
    B --> C[查段表得到该段页表位置]
    C --> D[段内地址拆成 页号 p + 页内偏移 d]
    D --> E[查该段页表]
    E --> F[得到页框号 f]
    F --> G[拼接 f 与 d]
    G --> H[物理地址]
  • 优点：既有模块化的共享/保护能力，又降低了连续分配压力；
  • 缺点：地址转换链更长，表结构更复杂，需要更多硬件支持和缓存（TLB）优化。

Note

Intel x86-32 就是典型的段页式架构。不过现代 x86-64 实际上弱化了分段，把大部分段的基址设成 0，几乎退化为纯分页。所以你平时学的”页式内存”，才是现代系统的绝对主流。

7. 内存”扩充”技术

物理内存不够怎么办？历史上出现过四种思路：

  1. 紧凑：移动进程合并碎片；
  2. 覆盖：程序内部模块分时复用同一块内存；
  3. 交换：整个进程在内存和磁盘之间来回倒腾；
  4. 虚存：只把需要的部分装进来，现代主流方案（下一讲重点讲）。

7.1 覆盖技术

覆盖技术的核心思想是：程序总大小可以超过物理内存，只要不会同时执行的模块共享同一块”覆盖区”即可。

比如一个程序逻辑上 54K，但常驻区 + 两个覆盖区只占 30K，那它在 30K 内存里也能跑。操作系统在需要时自动把相应模块装入覆盖区。

flowchart TB
    A[常驻区]
    B[覆盖区 0]
    C[覆盖区 1]

    A --> B
    A --> C
  • 优点：用时间换空间，在内存极度匮乏的时代很有用；
  • 缺点：需要程序员手动划分模块和覆盖关系，对用户极不友好，编程复杂度高，模块切换还要从外存读入，开销大。

Note

随着虚拟内存的普及，覆盖技术已经基本退出历史舞台。但它的思想——“程序不必全部常驻内存”——直接启发了后来的按需调页机制。

7.2 交换技术

交换（Swapping）的典型动作是 roll out 和 roll in：内存紧张时，把整个进程换出到外存的交换区（swap area）；需要执行时，再换入内存。

交换区通常是磁盘上专门划出的区域，方便高效块读写。交换的内容主要是进程映像，尤其是栈、堆这些运行期会变化的数据。

flowchart LR
    A[内存紧张] --> B[选择可换出进程]
    B --> C[写入交换区]
    C --> D[释放内存]
    D --> E[换入待运行进程]
    E --> F[建立新映射]
    F --> G[继续调度执行]

交换不是简单的”搬运工”，它和调度器紧密耦合：

  • 什么时候交换？
  • 换谁出去？
  • 换出内容存在磁盘哪？
  • 换入时是否回到原位置？（有动态重定位的话，不必回原位）
  • 进程空间增长怎么办？

处于等待 I/O 的进程通常不适合被换出，因为它和设备操作有外部依赖，换出去可能导致 I/O 结果无处安放。

Trap

交换和虚拟内存里的”页面换出”听起来很像，但交换是以进程为单位整体换入换出，而虚拟内存的页面换出是以页为单位细粒度进行。后者更灵活，是现代系统的标准做法。

进程空间增长也是交换的头疼问题：数据段可能向上扩展，栈可能向下扩展。如果内存布局太紧凑，就没给增长留余地。解决办法要么预先留空间，要么换入时重新选址再重定位。

8. 归纳：内存管理的四层骨架

好了，我们把这一讲的内容串一串。内存管理可以概括为四个层次：

  1. 地址抽象层：把逻辑地址和物理地址分离，造出地址空间；
  2. 底层组织层：管理空闲物理内存，在分配、回收、碎片和性能之间找平衡；
  3. 映射方案层：选择连续区、固定分区、可变分区、分页、分段或段页式；
  4. 内存扩充层：物理内存不够时，通过紧凑、覆盖、交换和虚拟化继续支撑程序运行。

Summary

这一讲是内存管理的”地基工程”。重点要掌握：地址空间的概念、静态/动态重定位的区别、基址/界限保护机制、伙伴系统和 SLAB 家族的工作原理、六种基本内存管理方案的优缺点对比、分页和分段的地址转换流程，以及覆盖和交换这两种”以时间换空间”的内存扩充技术。下一讲我们将在此基础上，深入虚拟内存的奇妙世界——缺页中断、页面置换算法、工作集模型都在等着你！
