进程线程模型

进程线程模型

Apr 23, 202616 min read

核心全局观

现代操作系统的两大基石抽象：

  1. 进程是对 CPU 的抽象：通过分时复用，让每个程序感觉自己独占了一个虚拟的 CPU。

  2. 地址空间是对内存的抽象：通过虚拟内存机制，让每个进程感觉自己拥有连续、独立的物理内存。

后来为了进一步提升并发性能和降低开销，我们又在进程内部剥离出了线程（执行流的抽象），从而实现了资源管理（进程）与调度执行（线程）的分离。

一、 进程基础与状态流转

1. 从顺序执行到多道程序设计

在早期的单道程序环境（顺序环境）中，程序执行是封闭且具有“可再现性”的，也就是结果与执行速度无关。但为了提高系统资源的利用率，我们引入了多道程序设计和并发环境。

在并发环境下，程序执行具有了：

  • 间断性：走走停停。

  • 不可再现性：由于共享资源和执行顺序的不确定性，同样的输入未必有同样的结果。

  • 程序与计算不再一一对应：一个静态的代码文件跑起来，可能产生多个动态的执行实例。

为了准确刻画这种“动态的并发执行过程”，操作系统引入了进程（Process）。

直觉理解：程序 vs 进程

程序是菜谱（静态的指令集合），进程是厨师照着菜谱做菜的过程（动态的执行活动）。如果有多厨师照着同一份菜谱做菜，那就是同一个程序对应的多个进程。

2. 进程的状态模型

进程在其生命周期中并不是一直占有 CPU 的，它会在不同的状态之间流转。

三状态/五状态模型

最经典的是五状态模型，这是理解所有操作系统调度的基础。

stateDiagram-v2
    [*] --> 创建(New): 提交任务
    创建(New) --> 就绪(Ready): 资源到位/入队列
    
    就绪(Ready) --> 运行(Running): 调度器派发(Dispatch)
    运行(Running) --> 就绪(Ready): 时间片用完/被抢占
    
    运行(Running) --> 阻塞(Blocked): 等待I/O或事件
    阻塞(Blocked) --> 就绪(Ready): I/O完成/事件发生
    
    运行(Running) --> 终止(Terminated): 执行完毕/被杀死
    终止(Terminated) --> [*]: 资源回收

  • 运行态 (Running)：正在 CPU 上执行。

  • 就绪态 (Ready)：万事俱备，只欠 CPU。

  • 阻塞态/等待态 (Blocked/Waiting)：就算现在把 CPU 给你，你也跑不下去（比如在等键盘输入、等磁盘读写完成）。

思考：状态转换是必然连锁的吗？

不一定。比如时间片到了，只会发生 运行 -> 就绪，不会导致其他进程的转换；但如果发生 阻塞 -> 就绪（磁盘读完），可能会因为优先级抢占，立刻导致当前运行的进程 运行 -> 就绪。

Linux 的实际状态模型扩展

真实的系统（如 Linux 或 xv6）会更复杂一点。例如 Linux 会细分阻塞态：

  • 浅度睡眠 (TASK_INTERRUPTIBLE)：等待资源，但可以被信号唤醒。

  • 深度睡眠 (TASK_UNINTERRUPTIBLE)：死等资源，不响应信号（通常在等底层硬件 I/O）。

  • 僵尸态 (TASK_ZOMBIE)：进程已经死亡，资源大部分已释放，但“户口”（PCB 里的退出状态）还没被父进程注销收尸。

二、 进程的数据结构与内存视图

操作系统是怎么管理这么多进程的？答案是靠数据结构。

1. 进程控制块 (PCB)

PCB (Process Control Block) 是操作系统感知进程存在的唯一标志。进程与 PCB 是一一对应的。

当进程暂时不运行时，OS 必须把它的“灵魂”（硬件上下文）封存在 PCB 里，下次再加载回来。

在 Linux 中，PCB 叫做 task_struct，它极其庞大，但核心无外乎以下几类信息：

  1. 标识信息：PID（进程号）、UID、父子关系指针。

  2. 状态与控制：当前状态（运行、睡眠等）、优先级、调度策略。

  3. CPU 现场信息 (Context)：程序计数器 (PC)、栈指针 (SP)、通用寄存器的值。

  4. 资源清单：

    • 内存描述符指针（指向地址空间、页表）。

    • 打开的文件描述符表（正在用哪些文件）。

2. 进程地址空间 (内存抽象)

一个反直觉的 C 语言实验

课上提到一个例子：同时运行两次相同的 C 程序（关闭操作系统的 ASLR 随机化机制），打印某个局部变量 myval 的地址和值。

你会发现：进程 A 打印地址是 0x555555558014，进程 B 打印出来的地址也是 0x555555558014！但它们的值互不影响。

为什么？

因为程序打印出来的是虚拟地址。每个进程都有一个“从零开始”的独立幻觉空间。OS 结合硬件（MMU）会在底层把相同的虚拟地址映射到不同的物理内存页上。

一个典型的进程虚拟地址空间布局（由低到高）如下：

  1. 代码段 (Text)：存放编译后的机器指令（只读）。

  2. 数据段 (Data)：已初始化的全局变量。

  3. 堆 (Heap)：程序运行时动态分配的内存（如 malloc），向上增长。

  4. 共享库/内存映射区：动态链接库（.so）加载的地方。

  5. 栈 (Stack)：局部变量、函数调用上下文，向下增长。

  6. 内核空间：最高地址部分（如 0xFFFF...），供 OS 内核使用，用户态代码无权直接访问。

在 Linux 中，内核用 vm_area_struct 这个数据结构来记录上述一块块连续的内存区域（VMA）。

三、 进程控制与写时复制 (COW)

操作系统提供了一系列原语（Primitive，不可中断的原子操作）来控制进程。

1. Unix/Linux 的经典进程控制 API

  • fork()：克隆自己，创建子进程。

  • exec()：灵魂出窍，用一段新的代码和数据覆盖当前进程的内存空间。

  • wait()：阻塞父进程，等待子进程死亡并回收其僵尸。

  • exit()：终止进程。

2. 写时复制 (Copy-On-Write, COW) 机制详解

痛点：早期系统调用 fork() 时，会把父进程的整块物理内存原封不动复制一份给子进程。但这极其浪费！因为往往 fork 之后紧接着就是 exec（执行全新程序），刚才费劲复制的数据瞬间就没用了。

现代操作系统的解决思路：延迟工作，能不干就不干。

graph TD
    A[调用 fork] --> B[分配新的 PCB 和页表给子进程]
    B --> C[并不复制物理内存数据]
    C --> D[将父子进程的页表项都设为'只读']
    D --> E[将内存区域 vm_area_struct 标记为私有 COW]

  • 读取时：父子进程乖乖读取同一块物理内存，相安无事。

  • 写入时（真正发生时）：

    1. 假设父进程试图修改某个变量。

    2. 硬件发现页表权限是“只读”，触发缺页异常 (Page Fault) 陷入内核。

    3. 内核一看 VMA 标记，明白这不是真的越权，而是 COW 机制。

    4. 内核此时才在物理内存中真正复制出新的一页。

    5. 修改对应进程的页表，指向新页，并将权限恢复为“可读写”。

    6. 回到用户态，重新执行写入指令。

COW 的本质

是一种极致的“懒加载”和内存管理策略，利用 MMU 异常机制截获写操作，实现了安全高效的内存共享与按需分配。

四、 为什么引入线程？

1. 痛点分析

传统进程模型有两个大属性绑在一起了：

  1. 资源拥有者：拥有独立的地址空间、文件句柄等。

  2. 调度执行的单位：拥有独立的执行轨迹（PC、寄存器、栈）。

如果我们要在同一个程序里并发做几件事（比如 Web 服务器：一个模块接收网络请求，多个工作模块去磁盘读网页），如果只用单进程，一旦读取磁盘（阻塞系统调用），整个进程就卡死了，无法响应新请求。

如果用多进程，进程间切换开销极大，且共享数据（如网页缓存）非常麻烦（需要配置复杂的 IPC 共享内存）。

2. 线程的定义与内存视图

线程（Thread）就是将“调度执行”的属性从进程中剥离出来。 进程退化为资源的容器，而线程变成了 CPU 调度的基本单位（轻量级进程）。

线程池：厨房里的帮工

进程是一个厨房（拥有锅碗瓢盆等资源），线程是厨房里的厨师。

以前一个厨房只配一个厨师（单线程进程），厨师去拿菜（阻塞）就没人炒菜了。

现在一个厨房配多个厨师（多线程进程），大家共享同样的锅碗瓢盆（地址空间、全局变量），沟通极其方便，一个厨师卡住了，其他厨师还能继续干活。

多线程进程的内存布局特点：

  • 共享的：代码段、数据段、堆、打开的文件、全局变量。

  • 私有的（每个线程独有一份）：

    • 用户栈：保存各自函数调用的局部变量。

    • 内核栈：当线程陷入内核态时使用的栈。

    • 线程控制块 (TCB)：保存各自的寄存器上下文 (PC, SP 等)。

注意：虽然线程栈在逻辑上是私有的，但在虚拟地址空间中它们是并排放在同一个空间里的。只要知道地址，线程 A 物理上是可以读写线程 B 的栈的（但这通常是严重的 Bug）。

五、 线程的实现机制

线程在哪里被管理？根据管理者的不同，有三种经典流派。

1. 用户级线程 (User-Level Thread, ULT)

  • 机制：由用户态的线程库（如早期的 POSIX Pthreads）管理。内核对线程一无所知，内核只知道这里有一个进程。

  • 优点：

    • 切换极快。无需陷入内核态，只需在用户态保存一下寄存器，切换一下栈指针即可。

    • 调度算法可以由开发者根据应用量身定制。

  • 缺点（致命缺陷）：

    • 阻塞问题：因为内核看不见线程，一旦某个线程发起阻塞式的系统调用（如读文件），内核会直接把整个进程挂起。其他原本可以运行的线程也跟着陪葬。

    • 无法利用多核：内核一次只给进程分配一个 CPU，同一进程内的多线程无法在多核上真正并行。

2. 核心级线程 (Kernel-Level Thread, KLT)

  • 机制：内核亲自下场管理线程。操作系统维护进程表和线程表。应用程序通过 API 创建内核级线程（如 Windows 默认机制、现代 Linux 的 NPTL）。

  • 优点：

    • 完美解决阻塞问题。一个线程阻塞，内核会调度同一个进程中的其他就绪线程运行。

    • 可以完美利用多处理器/多核架构。

  • 缺点：

    • 每次线程切换、创建、销毁，都需要陷入内核态，开销比用户级线程大得多。

3. 混合模型与协程 (Coroutines)

  • 混合模型 (Hybrid)：如早期的 Solaris。内核管理一批内核级线程，而在用户空间，线程库将数量庞大的用户级线程多路复用（Multiplex）到这些内核级线程上。结合了灵活性与并发能力，但极其复杂。

  • 协程 (Coroutine) / 纤程 (Fiber)：

    • 本质上是语言或库级别支持的用户级线程的现代化重生。

    • 通过 async/await 等非阻塞 I/O 机制结合事件循环（Event Loop），在一个内核线程上协作式地调度成千上万个协程（如 Go 的 Goroutine，Python 的 asyncio）。

    • 它解决了 ULT 的阻塞问题（因为底层用了非阻塞 I/O 多路复用，如 epoll），同时保持了 ULT 切换极快（微秒/纳秒级）的优势，非常适合高并发的 I/O 密集型场景。
