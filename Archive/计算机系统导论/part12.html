Dynamic Memory Allocation

动态分配器的任务与基本接口

动态内存分配器负责管理进程堆区，使程序能够在运行时申请和释放内存。显式分配器要求程序员显式调用 malloc、free、calloc、realloc 等接口；隐式分配器则由垃圾回收器自动识别不可达对象并回收空间。

malloc(size) 返回至少包含 size 字节有效载荷的块地址，通常还会满足字对齐或双字对齐。free(p) 将此前分配的块归还给分配器。calloc 在分配后将内存清零。realloc 用于调整已有块的大小。分配器在内部常通过 sbrk 或 mmap 向内核申请更多虚拟内存。

堆中的内存不是以单个字节为单位管理，而是以块为单位管理。每个块要么已分配，要么空闲。块内除用户可见的有效载荷外，往往还包含头部、填充以及某些实现中使用的尾部。

内部碎片指已分配块大小大于有效载荷大小。它通常来源于对齐要求、头部尾部开销以及分割策略。外部碎片指空闲总量虽然足够，但不存在单个连续空闲块能满足请求。例如堆中有两个大小均为 200 的空闲块，合计 400，却无法满足一个 300 字节的连续分配请求。

隐式空闲链表

隐式空闲链表将整个堆视为按物理顺序排列的一串块，每个块头部记录块大小以及分配位。由于块大小必须按双字对齐，头部最低若干位原本恒为零，因此可以复用最低位保存分配状态。若头部字为 h，则块大小可由掩码清除低位后得到，分配状态则由最低位读出。

分配器在查找空闲块时，不依赖单独的链表指针，而是根据当前块头部中的大小直接跳到下一个块，因此称为“隐式”。为了简化边界处理，堆首尾常额外设置序言块和结尾块。

常见放置策略包括首次适配、下一次适配和最佳适配。首次适配从头开始找到第一个足够大的空闲块就停止，速度通常较快，但容易在堆前部积累许多小碎片。下一次适配在上次结束处继续搜索，试图减少前部扫描压力。最佳适配寻找最接近请求大小的块，外部碎片往往较少，但搜索代价更高。

当找到的空闲块明显大于请求值时，通常会分割为一个已分配块和一个剩余空闲块。释放块后还必须考虑合并。若相邻空闲块不合并，就会迅速形成外部碎片。为了在常数时间内判断前后邻块是否空闲，常在块尾保存与头部一致的边界标记。这样释放某块时，就能通过前一块尾部和后一块头部判断四种情况：前后都已分配、前空后闲、前闲后空、前后都空，并据此进行局部或双向合并。

显式空闲链表与分离空闲链表

显式空闲链表只在空闲块中维护前驱与后继指针，因此搜索复杂度从“与堆中所有块数线性相关”下降为“与空闲块数线性相关”。当堆很满时，这种差别尤为明显。其代价在于释放和合并时除了更新块头尾，还需要维护链表指针。空闲块的插入策略可以采用后进先出，也可以按地址顺序插入。前者实现简单、时间为常数，但碎片情况可能较差；后者更利于降低碎片，但插入位置搜索较慢。

分离空闲链表进一步按块大小划分类别，例如 16–32、32–64、64–128 等，每一类维护独立链表。分配时先定位合适大小类，再在该类中执行首次适配；若未找到，则转向更大的大小类。这样既提高查找速度，又使首次适配的效果接近全堆范围内的最佳适配，因此现代高性能分配器普遍采用这一思想。

常见内存错误

动态内存管理中最典型的错误包括：对错误地址解引用、读取未初始化内存、分配空间不足导致覆盖相邻对象、返回栈上局部变量地址、对同一块执行多次释放、释放后继续使用，以及分配后遗忘释放造成内存泄漏。这些错误之所以危险，在于它们往往并不立即失败，而是在稍后以数据损坏、崩溃或安全漏洞的形式表现出来。

垃圾回收的基本思想

垃圾回收的目标是自动识别不再可达的块并回收它们。其基本前提是将所有块视为有向图中的节点，块内指向其他块的指针形成边，而寄存器、栈和全局变量中的指针则构成根集合。凡是能够从根集合出发沿指针链到达的块，都属于仍然可达的活动对象；其余不可达块即为垃圾。

标记—清除算法首先从根集合出发递归或迭代标记所有可达块，然后扫描整个堆，将未被标记的块回收到空闲集合中。对于 C 语言这类允许任意位模式出现在内存中的语言，精确判断一个字是否真的是指针并不容易，因此常使用保守垃圾回收：凡是某个字看起来像指向堆中已分配块，就暂时把它当作指针处理。保守方法可能把部分实际上不可达的块误判为可达，因此回收不够彻底，但可以在不改变语言运行时模型的前提下工作。
