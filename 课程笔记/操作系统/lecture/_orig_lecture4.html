处理器调度概述

处理器调度的任务，是在存在多个就绪进程或线程时，决定把哪一个执行实体安排到哪一个处理器上运行。其本质是对处理器资源竞争的控制与协调。

  • 单处理器系统中，同一时刻只有一个进程真正处于运行状态，其余可运行进程只能处于就绪状态并等待被选中；
  • 多处理器系统中，还需要进一步决定执行实体与处理器之间的映射关系。

当系统中没有可运行进程时，内核会安排空闲进程或空闲线程运行，以维持处理器处于受控状态。调度问题可概括为三个方面：

  • 依据什么规则选择执行对象，属于调度算法；
  • 在什么事件点发生选择，属于调度时机；
  • 如何完成从一个执行对象到另一个执行对象的切换，属于调度过程。

从层次上看，处理器调度通常分为长程调度、中程调度和短程调度。

  • 长程调度又称作业调度或宏观调度，发生在创建新进程时，其作用是决定一个新作业是否进入当前活跃进程集合。
  • 中程调度面向内存与外存之间的交换，通过将进程的部分或全部映像换出外存，或把所需部分换入内存，以缓解内存压力并为当前执行进程腾出空间。
  • 短程调度又称微观调度，直接面向处理器分配，从就绪队列中选出一个进程或线程投入运行。由于短程调度发生频率极高，时间尺度通常达到毫秒级，因此实现必须高效。

CPU 调度并非随时发生，而是由一系列特定事件触发。

典型触发点包括：进程正常结束，因异常或错误终止，新进程创建，当前运行进程因等待 I/O 或其他资源而阻塞，被阻塞进程被唤醒重新进入就绪态，时间片耗尽，或者进程主动执行 yield 放弃处理器。

更一般地说，每当中断、异常或系统调用导致内核接管控制流，并在处理完成后返回用户态之前，如果系统中的进程状态发生变化或就绪队列内容被更新，就需要运行调度程序重新选择下一个执行对象。

调度的具体执行离不开进程切换与上下文切换。

  • 进程切换是指一个进程让出处理器，另一个进程获得处理器的过程。
  • 上下文切换则更具体，指把 CPU 当前寄存器中的硬件状态从一个进程切换到另一个进程。

进程运行时，其程序计数器、程序状态寄存器、栈指针、通用寄存器及相关控制寄存器保存在处理器内部；进程不运行时，这些内容需要保存到 PCB 中。

一次完整的切换通常包括：保存旧进程上下文，更新旧进程 PCB 和状态，将其移入适当队列，设置新进程为运行态，并从新进程 PCB 中恢复寄存器状态。若从地址空间角度分析，还常常需要切换页表或页目录，以及切换内核栈。

上下文切换带来直接开销与间接开销。

  • 直接开销是内核执行保存、恢复寄存器以及切换地址空间所消耗的 CPU 时间。
  • 间接开销则更隐蔽，包括缓存失效、缓冲区缓存失效以及 TLB 失效。 因而，调度过于频繁虽然有利于响应性，却可能降低整体吞吐量与处理器有效利用率。

处理器调度算法的设计原则

不同操作系统面对的负载类型不同，调度目标也随之不同。

  • 交互式进程频繁等待用户输入，典型目标是缩短响应时间并保持体验均衡，平均延迟通常希望控制在几十到一百多毫秒量级。
  • 批处理进程不要求快速响应，重点在于提高吞吐量、降低周转时间并提升 CPU 利用率。
  • 实时进程则以截止时间和可预测性为核心，要求关键任务不被低优先级任务拖延。

因而，调度算法不存在脱离场景的“绝对最优”，它总是在响应时间、周转时间、吞吐量、公平性、可预测性、资源平衡和能耗之间折衷。

调度算法的典型评价指标包括吞吐量、周转时间、响应时间、公平性与资源利用率。

  • 吞吐量表示单位时间完成的进程数；
  • 周转时间表示一个进程从提交到完成经历的总时间；
  • 响应时间表示从请求提出到第一次得到响应的时间。
  • 对实时系统，还必须额外考察实时性与是否满足截止约束。
  • 对服务器和移动系统，还可能把能耗作为重要指标。

周转时间的基本定义为

Tturnaround​=Tcompletion​−Tarrival​

其中，Tarrival​ 是作业到达时刻，Tcompletion​ 是作业完成时刻。该量适合衡量批处理场景中的整体等待与执行代价。

Important

周转时间定义为完成时间减去到达时间。

设计调度算法时，还必须处理若干结构性问题。

  • 第一，PCB 中应记录哪些与调度有关的信息，例如静态优先级、动态优先级、剩余时间、已等待时间、时间片余额和处理器亲和性。
  • 第二，就绪队列如何组织，是单队列、多优先级队列，还是每 CPU 一队列。
  • 第三，是否采用抢占式调度。
  • 第四，如何区分 I/O 密集型和 CPU 密集型进程。
  • 第五，时间片多大才合适。

优先级可以是静态的，也可以是动态的。

  • 静态优先级在进程创建时确定，运行过程中不改变；
  • 动态优先级则允许系统根据等待时间、睡眠行为、I/O 完成情况或饥饿状态进行调整。

动态优先级的意义，在于让系统能把当前“更需要处理器”的任务临时前移，从而兼顾响应性与公平性。

就绪队列的组织方式直接影响调度复杂度与行为特征。

  • 若按优先级分层，可以维护多个就绪队列，高优先级队列先被检查；
  • 若按统一顺序组织，则所有进程在一个公共队列中竞争。

前者便于表达优先级策略，后者实现简单。多队列结构还便于把不同类型的任务分开处理，例如实时、前台、后台或不同服务等级。

  • 抢占式调度允许系统在更高优先级任务就绪时，强制剥夺当前任务的 CPU；
    • 抢占式调度更有利于交互性和实时性，但实现更复杂，且会增加上下文切换开销；
  • 非抢占式调度则要求当前任务除非主动阻塞、退出或放弃 CPU，否则一直执行。
    • 非抢占式调度简单、稳定，但可能导致长任务压制短任务。

按运行行为可把进程区分为 I/O 密集型与 CPU 密集型。

  • I/O 密集型进程具有频繁的 I/O 等待和较短的 CPU 突发，若调度得当，可以让 I/O 设备和 CPU 更充分重叠工作。
  • CPU 密集型进程则长时间占用处理器进行计算，更关心吞吐量与持续处理能力。

现代调度系统通常会对 I/O 密集型或睡眠较多的进程给予一定倾斜，以改善系统交互性。

时间片是分时系统中的核心参数。

  • 时间片过长，轮转调度会退化为先来先服务，短任务响应变差；
  • 时间片过短，又会导致大量上下文切换，使 CPU 时间消耗在调度本身而非有效计算上。

时间片的选取需要综合考虑切换成本、响应时间要求、就绪进程数、CPU 性能以及进程行为模式。

典型处理机调度算法

批处理系统中的典型算法

先来先服务算法（FCFS）按进入就绪队列的先后顺序分配 CPU，是最直接的非抢占式策略。它实现简单且形式上公平，但容易产生“护航效应”：若一个长作业先到，后面的短作业即使很快就能完成，也必须长时间等待。同时，长作业持续占用 CPU 时，其他 I/O 密集型进程难以及时发起 I/O，可能导致 CPU 与 I/O 设备都得不到最佳利用。

设三个作业 P1​,P2​,P3​ 按顺序到达，运行时间分别为 24 s,3 s,3 s。若采用 FCFS，则执行顺序为 P1​→P2​→P3​，完成时刻分别为 24,27,30，平均周转时间为

Tˉ=324+27+30​=27

若只改变执行顺序为 P2​→P3​→P1​，完成时刻变为 3,6,30，平均周转时间为

Tˉ=33+6+30​=13

这说明长作业位于队首时，会显著拉高平均周转时间。

最短作业优先（SJF）总是优先执行预计完成时间最短的作业。

其核心思想是优先完成短作业，以最小化平均周转时间。若在所有作业同时可运行且运行时间已知的前提下，SJF 可以取得最短的平均周转时间。但它需要对未来服务时间作出预测，而且连续到来的短作业可能使长作业长期得不到处理器，从而产生饥饿。

最短剩余时间优先（SRTN）是 SJF 的抢占式版本。当一个新到达进程的剩余时间小于当前运行进程的剩余时间时，系统立即抢占当前进程，改为运行新进程。对于到达时间分别为 0,2,4,5，运行时间分别为 7,4,1,4 的四个进程，非抢占式 SJF 的执行序列为 P1​→P3​→P2​→P4​；而抢占式 SRTN 的序列为 P1​→P2​→P3​→P2​→P4​→P1​。

抢占使系统能及时响应更短的新作业，但代价是实现和切换开销增加。

最高响应比优先（HRRN）试图在 SJF 的短作业偏好与长作业公平性之间取得平衡。其响应比定义为

R=SS+W​=1+SW​

其中，S 为作业处理时间，W 为等待时间。短作业因 S 小而更容易得到较高响应比；长作业虽然初期不占优势，但随着等待时间增长，R 会不断增大，最终获得调度机会，因此能缓解 SJF 中的饥饿问题。

HRRN 通常采用非抢占式形式，因为若抢占式地频繁重算响应比并排序，会显著增加系统开销。

交互式系统中的典型算法

轮转调度（RR）为每个进程分配固定时间片，时间片用完后通过时钟中断把当前进程送回就绪队列尾部，再调度下一个进程。

它的优点是形式公平，且能有效降低短任务和交互任务的响应时间；其代价则是切换开销较高。时间片选择过大会退化为 FCFS，过小则会让处理器大量时间耗费于上下文切换。

轮转调度并不总能改善完成时间。若两个进程 A,B 的运行时间都为 100 ms，时间片为 1 ms，且上下文切换开销忽略不计，则 RR 的完成时刻分别约为 199 ms 和 200 ms，平均完成时间为

CˉRR​=2199+200​=199.5 ms

若采用 FCFS，完成时刻为 100 ms 和 200 ms，平均完成时间为

CˉFCFS​=2100+200​=150 ms

这说明 RR 主要改善的是响应公平性，而不是总能优化完成时间。

在另一个四进程示例中，运行时间分别为 53,8,68,24。课件给出的结果表明，不同时间片下 RR 的平均等待时间并不相同：当时间片为 20 时平均等待时间为 66.25，当时间片为 5 时下降到 57.25；而最优顺序的 FCFS 平均等待时间可达到 31.25，最差顺序则会恶化到 83.5。这表明 RR 与 FCFS 各有适用场景：RR 更稳定、更公平，而 FCFS 的表现对作业排列极其敏感。

虚拟轮转（Virtual RR）是在普通时间片轮转基础上，对 I/O 密集型进程进行优先照顾的一种改进。

其基本机制是在普通就绪队列之外再设置一个辅助队列。新到达进程以及时间片用完的进程仍进入普通就绪队列；请求 I/O 的进程进入相应的 I/O 等待队列；当 I/O 完成后，该进程不返回普通就绪队列尾部，而是先进入辅助队列。调度时，系统优先从辅助队列中选择进程运行，只有当辅助队列为空时，才从普通就绪队列中按 RR 规则选取进程。

因此，像不断重复“短 CPU 突发 + I/O 等待”的 I/O 密集型进程，在 I/O 完成后能够更快重新获得 CPU，并及时发起下一次 I/O 请求，从而保持处理器与外设的重叠工作；而计算密集型进程则主要在普通就绪队列中轮转执行。虚拟轮转的目标不是单纯缩短时间片，而是在避免过多上下文切换的同时，减少 I/O 密集型进程因重新排队而产生的额外等待，提高系统的响应性与资源利用率。

    A["新进程进入系统"] --> B["普通就绪队列<br/>Ready Queue"]
    B --> C{"调度器选择下一个进程"}
    C -->|辅助队列非空| D["辅助队列<br/>Auxiliary Queue"]
    C -->|辅助队列为空| B

    D --> E["CPU 运行"]
    B --> E

    E --> F{"运行结果"}
    F -->|时间片用完| B
    F -->|进程结束| G["Exit"]
    F -->|发起 I/O 请求| H["I/O 等待队列"]

    H -->|I/O 完成| D

设进程 A 与 B 都需要 1000 ms 的纯计算，而进程 C 不断重复“使用 CPU 1 ms，再进行 I/O 10 ms”。若采用 FCFS，A 或 B 会长时间阻止 C 发起新的 I/O；若 RR 的时间片为 100 ms，磁盘大部分时间仍会空闲；若时间片缩小到 1 ms，C 可以在 I/O 完成后几乎立刻重新发起下一次 I/O，使磁盘利用率接近 90%，但切换开销会急剧增加。这个例子说明，I/O 密集型任务的良好调度不仅影响响应时间，也影响整机资源重叠利用。

最高优先级优先调度总是选择当前优先级最高的进程运行。系统进程一般高于用户进程，前台进程往往高于后台进程，而 I/O 型进程也常常受到偏好。该算法实现简单，但如果没有动态调整或老化机制，低优先级进程可能发生长期饥饿。

基于优先级的抢占式系统中，优先级反转是一个关键问题。

设高优先级进程 H、中优先级进程 M 与低优先级进程 L。若 L 先进入临界区并持有资源，随后被抢占；H 再尝试进入临界区失败而阻塞；此时若 M 持续占用 CPU，则 L 无法运行并释放资源，进而导致真正最重要的 H 被间接阻塞。

其后果可能是系统性能下降，甚至在关键场景下造成系统错误。常见解决方案包括优先级继承、优先级天花板协议以及在极短临界区内禁止中断。

在 Mars Pathfinder 上，低优先级任务持有锁，高优先级任务需要该锁被阻塞，而中优先级任务不断抢占 CPU，导致低优先级任务无法释放锁，从而间接阻塞了高优先级任务，最终触发系统看门狗复位。看门狗（watchdog timer）是一种定时器机制，要求系统必须在规定时间内“喂狗”（重置计时器），否则就认为系统出了问题，然后执行恢复动作（通常是重启）。

其中

  • 低优先级任务：气象/信息采集相关线程，先拿到了一个共享资源上的互斥锁。
  • 高优先级任务：数据总线管理线程，后来也需要这个锁，于是被阻塞。
  • 中优先级任务：通信等普通工作线程，本来不需要那把锁，但它的优先级高于低优先级线程，所以会不断抢占低优先级线程。

多级反馈队列调度（MLFQ）是一种依据进程历史运行行为动态调整优先级的综合调度方法。系统设置多个优先级递减的就绪队列，高优先级队列时间片较短，低优先级队列时间片较长，各级队列内部通常按轮转方式调度。

新创建进程先进入最高优先级队列。若一个进程在当前队列中用完整个时间片，说明它更偏向 CPU 密集型，于是被降入下一优先级队列；若一个进程在时间片耗尽前因等待 I/O 或其他事件而阻塞，则说明它更偏向交互式或 I/O 密集型，待事件完成后通常回到原来的优先级队列，而不是继续下沉（这里需要讨论在队首还是队尾）。

当系统允许抢占时，只要更高优先级队列中出现新的就绪进程，当前较低优先级队列中的运行进程就会被抢占。被抢占进程一般回到其原来的就绪队列，等待下次调度。至于它回到队首还是队尾、剩余时间片是否保留，不同操作系统实现可以采用不同规则，因此这些属于具体实现参数，而不是 MLFQ 的统一定义。

MLFQ 的核心不是静态地给进程贴上“交互式”或“计算型”的标签，而是根据进程是否频繁用完整个时间片、是否经常主动阻塞等历史行为，动态推断其类型。这样，短 CPU 突发、频繁等待 I/O 的进程会长期停留在较高优先级队列，而长时间持续占用 CPU 的进程会逐步下沉到较低优先级队列。

flowchart TD
    A["新建进程"] --> Q1["就绪队列 1<br/>最高优先级, 最短时间片"]

    Q1 -->|调度| CPU["CPU 运行"]
    Q2["就绪队列 2"] -->|调度| CPU
    Q3["就绪队列 3<br/>最低优先级, 最长时间片"] -->|调度| CPU

    CPU -->|时间片用完| DOWN1["降到下一优先级队列"]
    DOWN1 --> Q2
    Q2 -->|时间片再次用完| Q3

    CPU -->|I/O 或事件阻塞| WAIT["等待队列"]
    WAIT -->|事件完成| BACK["回到原优先级队列"]
    BACK --> Q1

    H["更高优先级进程就绪"] -->|允许抢占| PREEMPT["当前低优先级进程被抢占"]
    PREEMPT --> Q2

交互式系统中还常见公平共享调度、保证调度和彩票调度等思想。公平共享把 CPU 份额首先分配给用户或用户组，再由组内进程共享；保证调度强调每个任务在较长时间窗口内获得接近其应得比例的处理器时间；彩票调度则把处理器份额映射为“彩票”，通过随机抽签实现概率意义上的比例公平。这些方法强调的不是单次选择的确定性最优，而是长期份额控制。

各种典型算法可以从多个维度比较。

  • FCFS 开销小、无饥饿，但对短作业和 I/O 型作业不利；
  • RR 公平、响应快，但时间片过小时吞吐量下降；
  • SJF 和 SRTN 能显著改善平均周转时间，但可能造成长作业饥饿；
  • HRRN 通过等待时间修正改善了公平性；
  • 反馈队列类算法则更适合混合负载，尤其对交互式进程更友好，但策略和参数更复杂。

调度设计因此始终是一种折衷，而非单一指标最大化。

机制与策略分离、线程调度

调度设计中有一个长期成立的原则，即机制与策略分离。机制是调度赖以运作的内核基础设施，例如时钟中断、就绪队列、优先级位图、上下文切换、处理器亲和性控制和抢占触发点；策略则回答“谁应当先运行、运行多久、何时提升或降低优先级”。分离的意义在于：底层机制一旦稳定，系统可以在不重写全部内核路径的情况下更换策略；反之，策略变化也不会破坏上下文切换、中断处理或队列同步等核心机制。这样既利于维护，也便于在桌面、服务器、实时或嵌入式场景下复用同一套调度基础设施。

在线程调度问题上，关键在于内核是否直接感知线程。若内核只调度进程，则系统先选中一个进程，再由用户级运行时决定该进程内哪个线程真正执行；若内核直接调度线程，则不同进程中的线程可以直接竞争 CPU。后者更利于细粒度抢占和阻塞管理，现代通用操作系统因此通常采用以线程为基本调度单位的设计。

Windows 线程调度

Windows 的基本调度单位是线程，而不是进程。其总体机制是基于动态优先级的抢占式调度，并结合时间配额进行控制。系统维护多个按优先级划分的就绪队列，总是优先选择最高优先级的就绪线程运行；同一优先级上的线程再按时间片轮转方式分配处理器。多处理器环境下，不同线程可以在不同处理器上并行执行。

Windows 中会触发线程调度的典型事件：

  • 线程优先级发生变化
  • 线程的处理器亲和集合发生变化
  • 线程正常终止或因异常终止
  • 新线程创建
  • 等待线程重新进入就绪态
  • 运行线程进入阻塞态
  • 运行线程因时间配额耗尽或其他原因重新回到就绪态

只要这些事件改变了可运行线程集合或各线程的调度优先次序，调度器就需要重新选择执行对象。

Windows 将线程优先级划分为 32 个级别。其中，优先级 0 保留给零页线程；优先级 1 到 15 属于可变优先级范围；优先级 16 到 31 属于实时优先级范围。实时优先级线程通常不做动态调整，而可变优先级线程的当前优先级可以在基本优先级附近上下浮动。基本优先级决定线程的长期调度地位，当前优先级则反映调度器在特定运行阶段对该线程作出的临时修正。

Windows 中的时间配额不是直接用毫秒表示的固定时间长度，而是以内核使用的配额单位表示的整数值。线程控制块中记录了与之相关的 Quantum 和 QuantumReset 字段。一个线程用完当前时间配额后，若同一优先级上没有其他就绪线程，系统可以重新赋予它一个新的时间配额，并允许它继续运行。时间配额的作用，不仅在于限制单次连续运行时间，也在于在不直接改变优先级的前提下调节不同线程实际获得的 CPU 份额。

为提高调度选择效率，Windows 维护就绪位图与空闲位图。

  • 就绪位图的每一位对应一个调度优先级，用于表示该优先级的就绪队列是否为空；
  • 空闲位图的每一位对应一个处理器，用于表示该处理器当前是否空闲。

借助这两类位图，调度器可以快速定位最高优先级的非空就绪队列，以及当前可立即投入使用的处理器。

线程离开 CPU 后的处理方式，取决于它离开处理器的原因。若线程因等待 I/O、事件对象或其他资源而主动阻塞，则它从运行态转入阻塞态，待等待条件满足后再重新进入就绪队列。若线程因更高优先级线程变为就绪而被抢占，则它被放回原优先级就绪队列。对于实时优先级线程，发生抢占时其时间配额会被重置为一个完整配额；对于可变优先级线程，发生抢占时其剩余时间配额通常保留，待再次获得处理器后继续消耗。

若线程只是正常用完了自己的时间配额，则处理方式又有所不同。若同一优先级队列中仍有其他就绪线程，系统会选择下一个线程运行，而当前线程回到原队列末尾；若同一优先级上没有其他竞争线程，则系统重新赋予该线程新的时间配额，并允许它继续运行。因而，主动阻塞、被更高优先级线程抢占以及时间配额耗尽，虽然都会导致线程暂时离开 CPU，但它们在状态转移路径、重新入队位置以及时间配额处理方式上并不相同。

Important

主要的手段是优先级改变和时间配额调整。

Windows 中，会被提升优先级的是可变优先级线程（1~15级），不是实时线程。原因是系统要提高交互响应性和 I/O 完成后的处理效率。

Windows 对可变优先级线程采用“临时优先级提升 + 时间配额调整 + 逐步衰减”的机制，以改善响应性并缓解饥饿。线程的长期调度地位由基本优先级决定，而当前优先级则可根据等待与唤醒情况作临时修正。这种修正只作用于可变优先级范围内的线程，且不是永久性的；在线程重新获得处理器后，提升幅度会随着运行逐步消失，最终回到基本优先级。

I/O 操作完成后，Windows 会临时提升等待该 I/O 的线程的当前优先级，使其更快重新获得 CPU 并处理刚到达的数据。优先级的实际提升幅度由设备驱动在 I/O 完成路径中给出，响应时间要求越高，提升通常越大。为避免这种倾向造成长期不公平，系统在唤醒该线程时会将其时间配额减 1。

当线程等待的事件对象或信号量对象结束时，Windows 也会提升其当前优先级。该提升以线程的基本优先级为基准，通常提升一级，但提升后的优先级不会超过 15。线程先在提升后的优先级上运行完当前剩余时间配额，随后每获得一个新的时间配额，优先级下降一级，直到恢复为基本优先级。

对于长期处于就绪状态却始终没有得到运行机会的线程，Windows 通过平衡集管理器处理饥饿问题。平衡集管理器每秒扫描一次就绪队列；若发现某线程排队超过 300 个时钟中断间隔，则把它的优先级提升到 15，并分配一个长度为正常值 4 倍的时间配额。当这一扩展时间配额用完后，该线程的优先级立即恢复为原来的基本优先级。

多处理器与多核调度

多处理器调度与单处理器调度相比，增加了一个核心决策：不仅要决定谁运行，还要决定在哪个 CPU 上运行。

对称多处理器系统（SMP）中，每个处理器都运行自己的调度程序，处理器之间可共享负载，但对共享资源的访问必须同步。这样做提高了并行性，也引入了队列竞争、迁移成本和负载均衡问题。

进程在多个 CPU 之间迁移会造成额外开销，特别是缓存失效和 TLB 失效。因此，调度器通常倾向于让一个进程尽量在同一个处理器上反复运行，这就是处理器亲和性的基本思想。但如果过于强调亲和性，又可能导致某些处理器负载过高、另一些处理器空闲，从而降低整体吞吐量。因此，多处理器调度实际上是在“局部性”与“负载均衡”之间寻找平衡。

从队列组织上看，SMP 进程分配可以分为静态分配与动态分配。静态分配中，进程从开始到结束固定绑定在一个处理器上，每个处理器维护自己的就绪队列，调度开销低，但容易出现忙闲不均。动态分配中，所有处理器共享公共就绪队列，或允许进程在执行期间迁移到任意空闲处理器上，负载更均衡，但调度和同步开销更大。

负载均衡考虑的是：不要让某些处理器很忙、另一些处理器很闲，而是要把线程尽量均匀分配到各个核心上，提升整体吞吐量和处理器利用率。

多核系统中，处理器与缓存层次结构更复杂，调度必须额外关注缓存一致性、缓存亲和性、核间数据共享与负载均衡。缓存一致性问题出现在多个核心同时读取并修改共享数据时，例如两个处理器都把共享变量 X 从 0 读入本地缓存后各自加 1，若没有一致性协议，最终主存中的结果可能不是 2。硬件通常通过 MESI 等缓存一致性协议解决这一问题。

缓存亲和性要求让特定进程或线程尽量在同一核心上运行，以重复利用该核心缓存中的数据，减少访存开销。其与负载均衡的冲突可以用并行渲染任务说明：若把某块数据长期固定在某核心处理，则缓存命中率较高；但若各数据块计算量不均，就需要把一部分任务迁移给较空闲的核心，以缩短总完成时间，而迁移又会带来缓存不命中。CDN 场景也存在类似矛盾：将用户请求路由到本地缓存服务器有利于命中率，但高负载时跨区转发又能缓解热点。

作业偷取（Work Stealing）是负载均衡中的典型方法。其思想是，每个处理器先优先执行本地任务队列中的工作；当本地队列为空时，再去其他处理器“偷取”任务。它既保留了本地队列带来的缓存局部性，又能在空闲处理器出现时进行补偿式均衡。代价是偷取本身需要同步，并可能破坏原有数据局部性。

Go 语言运行时的 GMP 调度器是作业偷取与缓存亲和性的典型案例。它尽量把 Goroutine 调度到先前处理过相关数据的处理器上，也尽量把同一 Goroutine 放回原先所在处理器 P 绑定的内核线程 M 上，从而保留缓存局部性；同时，每个 P 维护本地 Goroutine 队列，当本地队列耗尽时，再从其他 P 窃取任务。YARN 则把类似思想推广到分布式资源调度：任务倾向于被放到存有目标数据的节点上，以提高缓存利用率；但 ResourceManager 又会根据各节点负载进行跨节点协调。

实时调度

实时系统关注的核心不是平均意义上的快，而是在规定时间边界内完成指定任务。按照时间约束的严格程度，实时系统可分为硬实时与软实时。

  • 硬实时任务必须在规定时刻或时间区间内完成，违约通常不可接受；
  • 软实时任务则允许偶尔违反最终时限，但仍希望尽量按时完成。

按照到达模式，又可把实时任务分为周期性任务、偶发性任务和非周期性任务。实时调度的关键特征是确定性，即系统能否明确判断任务是否能在截止时间前完成。

对于由多个周期性事件流构成的系统，若事件 i 以周期 Pi​ 发生，每次事件需要 Ci​ 秒 CPU 时间，则系统负载满足下式时可认为是可调度的：

i=1∑m​Pi​Ci​​≤1

其中，m 是周期性事件流个数，Ci​/Pi​ 表示任务 i 对处理器的平均利用率需求。若总需求超过 1，则处理器平均能力已不足以覆盖全部任务，系统必然无法满足所有截止约束。

速率单调调度（RM）是静态优先级实时调度算法，前提是所有任务都周期性到达且执行时间已知。其基本思想是按任务周期分配优先级：周期越短、到达越频繁的任务，优先级越高。因为短周期任务更容易积累时限压力，因此在固定优先级体系下，RM 用周期表达紧迫性。其优点是实现简单、可预测性强；局限在于它依赖稳定的周期性前提，对变化任务和复杂截止约束的适应性较弱。

最早截止时间优先（EDF）是动态优先级实时调度算法。系统根据当前任务的最后期限和剩余执行需求动态调整优先级，截止时间最早的任务优先级最高。EDF 的优势在于更灵活，通常能在更高处理器利用率下保持可调度，并且响应时间表现较好；其代价则是实现复杂，需要持续维护截止时间信息，并在任务集合动态变化时不断重新排序。

OpenEuler 的多核调度技术

OpenEuler 对 CPU 调度的基本理解与通用操作系统一致：在必要时中断当前进程，改由就绪队列中的另一个进程占用 CPU。不同之处在于，多核场景下还必须处理核间共享数据、同步与负载迁移。其普通进程以 CFS 为核心调度策略，强调相对公平；实时进程则联合使用 FIFO、RR 与优先级调度等方法，以满足更严格的时延要求。

OpenEuler 在多核系统上既要考虑缓存亲和性，又要考虑负载均衡。若采用单队列调度，所有 CPU 都从同一调度队列取任务。策略一中，每个 CPU 简单使用轮转法选择下一个进程，容易导致同一进程在不同 CPU 间来回迁移，破坏缓存亲和性。策略二中，引入亲和度机制，让进程尽量停留在同一个 CPU 上，虽然改善了局部性，却可能牺牲部分进程的均衡分布。

为进一步缓解问题，OpenEuler 采用多队列调度。每个 CPU 维护自己的本地队列，例如 Q0​ 由 CPU0 维护，Q1​ 由 CPU1 维护。这样可以增强局部性并减少全局锁竞争。但若某些 CPU 上的进程提前结束，就可能出现一部分队列空闲、另一部分队列仍然积压的负载失衡现象。

OpenEuler 用迁移线程解决这种失衡。每个处理器拥有一个迁移线程 migration/CPUID 和一个停机工作队列。当某 CPU 发现另一 CPU 上有可迁移任务时，会向其停机工作队列投递迁移函数并唤醒对应迁移线程；该迁移线程具有很高优先性，不会被普通进程抢占，因此能迅速完成跨 CPU 任务搬移，实现新的负载平衡。迁移线程的设计说明，在多核操作系统中，调度不只是“选谁运行”，还包括“如何安全、高效地改变任务所在核心”。

Linux 进程调度

Linux 将调度对象分为实时进程与普通进程两大类。实时进程要求极低调度延迟，通常采用 FIFO 或 RR 策略；普通进程包括交互式进程与批处理进程，前者睡眠较多、重视响应，后者位于后台、可容忍更大延迟。现代 Linux 中，普通进程的核心策略是 CFS。调度算法的演化大体经历了 Linux 2.4 的简单优先级加时间片调度、Linux 2.6 的 O(1) 调度器、SD 调度器补丁、RSDL 调度器补丁、CFS 调度器，并在 Linux 6.11 发展到 EEVDF 调度器。

    I -- 是 --> J{所有 RUNNING 进程的 counter 都耗尽了吗}
    J -- 否 --> A
    J -- 是 --> K[统一重新计算所有进程的 counter]
    K --> L[睡眠较多的进程通常保留更多时间片余额]
    L --> M[下一轮调度中更容易获得较高优先级]
    M --> A

Linux 2.4 采用“运行队列 + 时间片 + 优先级”的设计。在每次调度时，系统遍历 runqueue 中所有进程，比较优先级并选出最高者。普通进程在创建时被分配时间片，时钟中断到来时，当前运行进程的时间片递减；时间片耗尽后，该进程必须等待系统重新分配时间片才能继续运行。只有当所有 RUNNING 进程的时间片都用完时，系统才统一为它们重新计算时间片，从而保证所有进程最终都有机会运行。

在 Linux 2.4 中，普通进程的优先级由 counter 与 nice 共同决定。counter 可理解为当前时间片余额，数值越大，当前优先级越高；nice 是静态负向优先级，取值范围为 −20 到 19，其中 −20 优先级最高。当系统统一重算 counter 时，睡眠中的进程通常保留了更多未用完时间片，因此其新 counter 会相对更大，从而间接提高了交互式进程的优先级。这一机制试图补偿睡眠较多的进程，但总体上可扩展性差、在高负载下性能较弱，对交互式和实时任务的支持也不够理想。

O(1) 调度器

    C --> D[找到最高优先级的非空队列]
    D --> E[取队首进程运行]

    F --> G{时间片是否耗尽}
    G -- 否 --> E
    G -- 是 --> H[重新评估动态优先级]
    H --> I{是否为交互式或实时任务}
    I -- 是 --> J[重置时间片并放回 active]
    I -- 否 --> K[放入 expired]
    
    J --> L{active 是否为空}

    L -- 否 --> A
    L -- 是 --> M[交换 active 与 expired 指针]

为解决 Linux 2.4 的问题，Ingo Molnar 提出了 O(1) 调度器。其改进主要在两个方面：一是动态优先级的计算方法，二是 pick next 的选择机制。普通进程的动态优先级由静态优先级、睡眠奖励 bonus 等因素综合决定；平均睡眠时间越长，系统越倾向把该进程识别为交互式进程，并赋予更高优先级。实时进程优先级则不做动态修改，且始终高于普通进程。

O(1) 调度器为每个 CPU 维护两个优先级数组：active 与 expired。当需要选择下一个运行进程时，调度器直接从 active 数组中最高优先级的非空队列取出第一个进程，因此选择开销是常数级。若进程时间片耗尽，调度器会判断其是否属于交互式或实时进程：满足条件者可被重置时间片并重新放回 active；否则进入 expired。当 active 为空时，系统交换 active 与 expired 指针，重新开始下一轮。该设计显著提高了选择效率，但优先级调整中大量基于经验的公式较难理解和维护，而且在一些典型测试下仍会对交互式进程产生不佳效果。

SD 与 RSDL 调度器

    A[进入调度] --> B{是否有实时进程}
    B -- 是 --> R[实时进程按 FIFO / RR 调度]
    B -- 否 --> C[从 active 数组中选最高优先级非空队列]
    C --> D[取队首任务运行]
    
    D --> E[时间片递减]

    F -- 否 --> D
    F -- 是 --> G{是否已降到最低优先级}
    
    G -- 否 --> H[下降到下一层更低优先级队列]
    H --> A
    
    G -- 是 --> I[回到接近初始优先级的位置]
    I --> J[重新分配相应倍数时间片]
    J --> A

Con Kolivas 提出的 SD（Staircase Scheduler）抛弃了 O(1) 调度器复杂的动态优先级计算，更强调“完全公平”的思路。它仍为每个优先级维护队列并放在 active 数组中，但当进程用完时间片后，不再进入 expired 数组，而是被降入更低一级的优先级队列，相当于沿着“楼梯”逐步下行。达到最低级后，再次用完时间片的任务会回到其初始优先级之后的某一级，并获得相应倍数的时间片。这样做使高优先级任务最终也会下沉，与低优先级任务竞争，从而避免饥饿。交互式任务若在高台阶处睡眠，唤醒时仍保留相对有利位置，因此能更快获得响应。实时进程仍继续使用 FIFO 或 RR。

    A[进入调度] --> B{是否有实时进程}
    B -- 是 --> R[实时进程按 FIFO / RR 调度]
    B -- 否 --> C[从 active 中最高优先级组选择任务]
    C --> D[任务运行]

    I -- 否 --> K[继续在当前 active 中参与调度]

RSDL（Rotating Staircase Deadline Scheduler）是在 SD 基础上的改进。它重新引入 expired 数组，并为每个优先级分配“组时间配额” Tg​，同一优先级中的每个进程再拥有相同的“个体时间配额” Tp​。

  • 当进程用完自身的 Tp​ 时，会下降到下一优先级组，这一过程称为 minor rotation。
  • 当某高优先级组用完自身的 Tg​ 时，即使组内仍有进程尚未用完 Tp​，整个组也会被强制下降，使低优先级任务能在可预计的未来获得调度机会。
  • 若进程用完自身真实时间片 time_slice，则放入 expired 数组的初始优先级队列中；
  • 当 active 为空或所有进程都降到最低级时，触发 major rotation，交换 active 与 expired，所有进程恢复初始状态。RSDL 因而比 SD 更强调“等待时间可预测”。

CFS 调度器

CFS（Completely Fair Scheduler）是 Linux 普通进程调度的关键里程碑。它吸收了 SD 与 RSDL 的“完全公平”思想，不再跟踪进程的睡眠时间，也不再显式地区分交互式与批处理进程，而是把所有普通进程统一映射到“按权重分配处理器份额”的模型上。

其核心概念是虚拟运行时间 vruntime：每个进程都记录自己“按公平尺度已经运行了多久”，调度器总是选择 vruntime 最小的进程运行。vruntime 与实际运行时间成正比，与进程权重成反比，因此优先级越高、权重越大的进程，在相同实际运行时间下增加的 vruntime 越慢。

设调度周期为 Tsched​，进程 i 的权重为 wi​，系统中所有可运行进程权重之和为 ∑j​wj​，则分配给进程 i 的运行时间为

ti​=Tsched​⋅∑j​wj​wi​​

其中，调度周期表示把所有处于 TASK_RUNNING 状态的进程都调度一遍的大致时间。

CFS 定义虚拟运行时间为

vruntimei​=ti​⋅wi​NICE_0_LOAD​

其中，NICE_0_LOAD 是 nice=0 时的基准权重常量。把上式与时间份额公式联立，可得

vruntimei​=(Tsched​⋅∑j​wj​wi​​)⋅wi​NICE_0_LOAD​=Tsched​⋅∑j​wj​NICE_0_LOAD​

由此可见，从 vruntime 的角度，所有进程在一个调度周期内前进的“虚拟距离”是相同的，这正是“完全公平”的数学表达。

为高效维护“谁的 vruntime 最小”，CFS 用 vruntime 作为键值构造红黑树。树中最左侧结点就是当前 vruntime 最小、最应获得 CPU 的进程。调度器在选中一个结点运行时，会把它从树中移出；进程重新变回就绪态时再插回树中。通常，一个刚运行过的进程因 vruntime 变大，会落到树的更右侧位置。这样，整棵树中的结点会围绕公平边界不断流动。

CFS 的重要数据结构包括 task_struct、sched_entity、sched_class 与 cfs_rq。task_struct 是 Linux 中对任务的完整描述，记录状态、内核栈、进程标识、优先级等信息；sched_entity 描述可参与调度的实体，包含权重、红黑树结点等；sched_class 则可理解为调度类的模块化接口，不同调度类通过实现统一函数集合接入内核调度框架；cfs_rq 则是某个 CFS 运行队列的表示，其中保存了红黑树等与公平调度直接相关的数据。

min_vruntime 是理解 CFS 的关键变量。它是每个 cfs_rq 上维护的一个下界量，通常不大于该队列内所有就绪进程的最小 vruntime，反映当前队列整体虚拟运行时间的前沿。新进程创建时，并不会把 vruntime 直接设为 0，否则会因“虚拟运行时间太少”而无条件立即压制老进程。更合理的做法，是以当前队列的 min_vruntime 为基准，再加上一个与单轮应得份额相对应的虚拟增量，相当于认为它已经在本轮公平周期中“占过位”。在特定条件下，系统还可能交换父子进程的 vruntime，以实现“子进程优先运行”等策略。

进程唤醒时，CFS 也不会简单保留原值不动。若一个进程睡眠较久，系统会用 min_vruntime 减去一个补偿值来修正其 vruntime，让它醒来后有机会较快重新获得 CPU；但补偿并非无限放大，为防止进程通过频繁短睡眠“刷优先级”，系统最终会在补偿结果与原 vruntime 之间取更大的那个，然后再把该进程插入红黑树，并判断其是否足以触发抢占。

从运行过程看，CFS 的几个典型情景如下。时钟中断到来时，内核先把当前进程这一小段实际运行时间折算进 vruntime，再判断其本轮是否已用完应得时间份额；若超过分配值，则标记为需要调度。主动调度发生时，当前进程先被更新信息并插回红黑树，然后系统选择新的运行实体并将其从树中移出。创建新进程时，系统先计算其初始 vruntime，再插树并判断是否能够抢占当前进程。唤醒阻塞进程时，则先补偿其 vruntime，再插树并根据差距与调度粒度决定是否抢占。CFS 中还维护 last 与 next 两个指针，用来在主动调度路径中对最近被换下和即将换上的进程做适度偏向；只有当新进程的 vruntime 小于当前进程达到一定粒度阈值时，wakeup_preempt_entity 才会判定可以抢占。

CFS 之后，Linux 又继续向 EEVDF 方向演化。课件给出的演化链表明，Linux 6.11 已使用 EEVDF 作为更新的普通调度器方向。其意义在于，Linux 调度系统仍在持续改进“公平性、延迟与交互性”之间的平衡，只是本讲的理论主线仍以 CFS 为中心。

总结

处理器调度的核心，不是单纯追求“更快”，而是在给定负载类型与系统目标下，决定如何在多个竞争执行实体之间分配有限的处理器资源。批处理重视周转时间与吞吐量，交互式系统重视响应与均衡，实时系统重视截止时间与可预测性，多处理器系统则必须同时处理亲和性、迁移成本与负载均衡。经典算法、Windows 线程调度、Linux 从 O(1) 到 CFS 的演化，以及多核系统中的亲和与迁移机制，构成了操作系统调度设计的主要知识主线。
