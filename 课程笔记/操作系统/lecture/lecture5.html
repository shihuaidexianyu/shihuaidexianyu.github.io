1. 内存管理的范围与演化

内存管理并不等同于虚拟内存管理。虚拟内存只是内存管理发展到较高阶段后的结果之一；更早期的内存管理已经涉及程序装入、地址重定位、存储保护、空闲空间分配与回收等问题。内存管理的演化主线，是从“物理内存直接使用”逐步走向“多道程序共享内存”，再走向“以地址空间为中心的抽象”，最后发展到“以虚拟化方式组织更大、更灵活的存储视图”。

程序只有装入内存才能运行。程序长期以可执行文件形式存放在磁盘中，装入之后才成为进程。多道程序设计使多个程序同时驻留内存，分时共享又进一步强化了并发执行需求。由此带来的直接后果，是程序地址不能再假定为固定物理地址，进程之间也必须彼此隔离，处理器还必须阻止越权操作。内存管理因此承担三类基础责任：为进程建立可用地址空间，将地址空间映射到物理内存，并通过保护机制维持隔离与权限边界。

局部性原理与存储体系构成了内存管理进一步复杂化的硬件背景。程序在任意时刻只会频繁访问一小部分代码和数据，因此现代系统把寄存器、高速缓存、主存和磁盘组织成层次化存储体系，以不同的速度、容量和成本取得折中。内存管理并不是孤立的软件机制，而是位于处理器、缓存、主存与外存之间的关键协调层。

flowchart TD
    A[早期物理内存直接使用] --> B[单程序连续装入]
    B --> C[多道程序设计]
    C --> D[分时共享]
    D --> E[地址空间抽象]
    E --> F[虚拟内存]

2. 地址空间：对内存的抽象

地址空间是进程可见内存的抽象视图，而不是物理内存本身。它把“程序正在使用什么地址”与“这些内容实际放在什么物理位置”分离开来，使程序能够在不知道自身最终物理装入位置的前提下运行。地址空间抽象解决了多道环境中的三个矛盾：

  • 逻辑连续与物理离散的矛盾
  • 页面是否常驻与是否可换出的矛盾
  • 程序内容是否一次性全部装入与能否分次装入的矛盾

一个典型进程地址空间通常包含代码段、数据段、堆、栈，以及共享库和内存映射文件等区域。

  • 代码段与只读数据通常来自可执行文件或共享目标文件；
  • 数据段包含已初始化和未初始化全局数据；
  • 堆在运行期由动态分配器扩展；
  • 栈用于过程调用、局部变量与返回现场；
  • 映射区用于共享库装载和文件映射。
  • 高地址侧往往还保留内核相关映射或陷入辅助页。

地址空间因此既是装载视图，也是运行视图。

活跃页面、常驻集与工作集是描述进程“当前真正需要多少内存”的不同术语。

  • 活跃页面是近期被访问、对继续执行有直接关系的页面；
  • 常驻集是某时刻实际留在内存中的页面集合；
  • 工作集则是一个时间窗口内被访问页面的近似集合，用来刻画局部性范围。

三者并不等价：常驻集偏向物理驻留状态，工作集偏向行为统计，活跃页面偏向当前有效性。它们共同服务于页面保留、换出决策和性能分析。

xv6 的地址空间布局较为简洁：低地址开始依次是 text、data 与 bss、stack、heap 等区域；高地址保留 trampoline 与 trapframe，这两页在用户态和内核态之间的控制转移中起到关键作用。

  • trampoline 负责陷入与返回时的过渡代码映射。
  • trapframe 保存用户上下文，使内核能够在陷入后接管执行并在返回时恢复现场。

    K[高地址: 内核相关映射]
    T[trampoline / trapframe 或陷入辅助页]
    M[共享库与内存映射区]

    D[数据段 / BSS]

    S[栈: 通常向低地址扩展]

3. 装载、绑定与地址重定位

程序执行前通常经历编译、链接、装载和运行四个阶段。

  1. 编译把源程序转化为目标文件；
  2. 链接把多个目标文件与库组合成装载模块；
  3. 装载把装载模块放入内存；
  4. 运行阶段由处理器逐条取指执行。

所谓“绑定”，就是把程序中的地址同某一内存位置发生关联。绑定可能发生在编译期、链接期、装载期或运行期，不同绑定时机决定了系统对重定位的依赖方式。

flowchart TD
    A[源程序] --> B{编译}
    B --> C[目标文件]
    C --> D{链接}
    D --> E{装载模块}
    E --> F[装入内存]
    F --> G{运行}

逻辑地址是程序使用的相对地址或虚拟地址，通常以程序起点为基准编址；物理地址是内存单元的实际地址，可被硬件直接寻址。地址重定位就是把逻辑地址转换成物理地址的过程。只要程序的最终装入位置在运行前不可知，系统就必须提供某种重定位能力。

静态重定位在装载时一次性完成地址修正，一旦装入完成，程序运行期间不再改变地址映射。这种方式实现较简单，可由软件完成，但程序之后很难移动。

动态重定位在执行期间逐条指令访问内存时完成地址映射，需要硬件支持。处理器发出逻辑地址后，由 MMU 将其与重定位信息组合生成物理地址。动态重定位使进程能够在物理内存中被移动，也为交换和更复杂的内存组织方式奠定了基础。

最简单的动态重定位模型使用重定位寄存器。若逻辑地址为 LA，重定位寄存器为 R，则物理地址（physical address，PA）可以写为：

PA=R+LA

这一模型表达了地址空间抽象的核心思想：程序看到的地址从零开始，而系统在底层决定它实际落到哪里。

flowchart LR
    CPU[CPU发出逻辑地址 LA] --> MMU[MMU]
    R[重定位寄存器 R] --> MMU
    MMU --> PA[物理地址 PA = R + LA]
    PA --> MEM[物理内存]

4. 地址保护与内存管理目标

多道程序环境中，地址保护与地址重定位同样基础。进程之间必须隔离，用户态代码不能任意访问其他进程地址，也不能越过系统给定的范围访问内核或未授权区域。

最基本的保护方式是基址寄存器与界限寄存器：基址给出进程可访问区域的起点，界限给出允许的长度。处理器在每次访存时检查地址是否落在合法范围内，越界则陷入操作系统。基址与界限寄存器只能由特权指令装入，因此保护边界最终由内核维护。

内存管理的目标可以概括为三点。

  • 其一是透明性，即为进程提供稳定、独立、与物理布局解耦的地址空间；
  • 其二是效率，即尽量减少访问延迟与管理开销；
  • 其三是保护，即控制越界访问、非法访问与共享权限。

对进程分配地址空间、把内容装入并映射到物理内存、管理共享、降低访问时间，这些都属于同一目标体系中的不同侧面。

flowchart TD
    A[访问地址 addr] --> B{addr >= base?}
    C[基址寄存器 base] --> B
    B -->|否| F[地址越界]
    B -->|是| D{addr < base + limit?}
    C --> D
    E[界限寄存器 limit] --> D
    D -->|是| H[允许访问内存]
    D -->|否| F
    F --> G[陷入操作系统]

5. 物理内存空闲空间管理

物理内存管理关注的不是“一个进程看见什么”，而是“底层空闲页框或空闲区域如何组织、分配、回收”。典型数据结构包括位图、空闲区表与空闲块链表。

  • 位图法将物理内存按固定粒度划分为若干分配单元，每个单元对应位图中的一位。分配单元大小由系统设计决定；在页式内存管理中，通常以页框为单位。
  • 空闲区表和已分配区表记录起始地址、长度和标志；
  • 空闲块链表则把空闲块串联起来，按是否显式记录、是否按大小分类，可进一步区分为隐式链表、显式链表和分离空闲链表（ics重点讲解）。

衡量物理内存管理效果的核心指标有两个：资源利用率与性能。

  • 资源利用率要求尽量减少碎片。
    • 内碎片指已经分配给某个对象或进程、但其内部未被有效利用的空间；
    • 外碎片指虽然总空闲量足够，但被切分成过多小块，无法满足较大分配请求。
  • 性能则要求分配与回收延迟低、CPU 开销可控。

不同分配算法本质上是在搜索成本与碎片模式之间做权衡。

  • 首次适配在空闲区表中寻找第一个足够大的空闲区；
  • 下次适配从上次命中的位置继续查找；
  • 最佳适配搜索全部空闲区，选择刚好够用的最小块；
  • 最差适配则总是从最大的空闲块中切出所需部分。

首次适配实现简单、平均性能较好；最佳适配倾向生成许多小碎片；最差适配希望保留中等大小块，但也可能迅速破坏大块连续空间。

回收时的核心问题是相邻空闲区合并。一个已释放区域与前后区域之间有四种关系：只与前相邻、只与后相邻、前后都相邻、前后都不相邻。合并策略决定外碎片是否会持续增长。空闲区管理因此不是单纯“标记为空”，而是必须维护区间关系。

5.1 伙伴系统

伙伴系统把内存按照 2 的幂次划分为若干层级空闲块表。请求大小为 s 时，系统寻找满足条件的最小块；若当前块过大，就不断二分成两个大小相等的伙伴块，直到得到足够小而又不小于请求的块。若某一级块大小为 2u，则满足

2u−1<s≤2u

时，可直接分配一个大小为 2u 的块。释放时，若该块的伙伴也空闲，则两者合并为更高一级块，如此递归向上。

伙伴系统的优点是分裂与合并规则简单，合并判定明确；代价是按幂次取整会产生内碎片。Linux 底层页级物理内存管理采用的是基于这一思想的方案。

flowchart TD
    A[收到大小请求 s] --> B[找到最小满足的 2^u 块]
    B --> C{块是否过大}
    C -->|是| D[一分为二得到一对伙伴]
    D --> C
    C -->|否| E[分配块]
    E --> F[释放块]
    F --> G{伙伴是否空闲}
    G -->|是| H[向上合并]
    H --> G
    G -->|否| I[插回对应空闲链表]

5.2 SLAB、SLUB 与 SLOB

伙伴系统适合页级或较大块分配，但内核运行时经常需要频繁分配小对象，如 inode、dentry、网络缓冲描述符、协议控制块等。若直接用伙伴系统处理大量小对象，分配粒度过粗，初始化成本和碎片代价都偏高。

SLAB 分配器因此在大块内存之上进一步管理固定大小对象，

  • 将可用内存按照大小划分为不同的区域，每个区域有一个空闲对象列表
  • 需要分配内存时，SLAB在合适的空闲对象列表中寻找一个大小与请求相符的空闲区域进行分配
  • 释放内存时，将该内存块添加回相应的空闲对象列表中
  • 使用 slab cache 提高重复分配与释放的效率。

SLAB 的问题在于结构逐渐复杂，队列多、元数据开销大，在高并发场景下容易出现锁竞争。

SLUB 是对 SLAB 的简化与重构，通过更简单的数据结构降低锁争用，并改善多核与 NUMA 环境下的可伸缩性。SLOB 则追求极简与低开销，更适合小型嵌入式系统。

三者的差别并不在于“是否管理小对象”，而在于管理策略对复杂度、缓存局部性与并发性能的不同取舍。

flowchart TB
    A[伙伴系统提供页级块] --> B[SLAB / SLUB / SLOB]
    B --> C[按大小类建立对象缓存]
    C --> D[小对象分配]
    D --> E[对象释放回缓存]

6. 基本内存管理方案

这里的加载单位为进程。

  • 即程序装入与执行环境建立以整个进程为对象进行。
  • 段页式中，进程地址空间按段组织，物理内存按页分配，但被创建、装入并参与调度的运行实体仍是进程本身。

6.1 单一连续区

单一连续区方案中，一段时间内内存只装入一个用户程序，用户程序独占内存，并通常装入固定位置。

它几乎不需要复杂的数据结构与地址变换，适用于极早期单道系统。其根本缺陷是并发能力弱、内存利用率低，操作系统与用户程序之间的边界也十分粗糙。

6.2 固定分区

固定分区把可分配内存预先切成若干连续区域，每个区域称为分区，每个分区只能装一个进程。分区大小可以相同也可以不同，但在运行期间保持不变。

该方案支持有限并发，实现简单，调度上既可以为各分区维护独立输入队列，也可以维护统一输入队列再按适合分区分派。固定分区的主要问题是内部浪费明显：进程小于分区时，多余部分无法再分配给其他进程，典型表现为内碎片。

6.3 可变分区

可变分区不再预先固定区域大小，而是根据进程需求从当前空闲空间中切出一个连续区。剩余部分重新形成新的空闲区，因此在短期内往往比固定分区更节省空间。

它的根本代价是外碎片：随着装入、释放、再装入交替进行，物理内存中会形成许多彼此分离的小洞，使总空闲量虽足够，却无法为大进程提供足够大的连续区。

紧凑技术就是针对这种问题：通过移动进程，把分散小洞压缩为较大连续空闲区。其可行前提是系统具有重定位能力，否则进程一旦移动，其内部地址引用就会失效。还需要考虑系统开销和移动的时机问题。

flowchart TD
    A[空闲连续区] --> B[按进程需求切分]
    B --> C[形成已分配区]
    B --> D[形成剩余空闲区]
    D --> E[多次分配与释放]
    E --> F[出现外碎片]
    F --> G[紧凑: 移动进程并合并空闲区]

6.4 页式内存管理

页式内存管理将进程地址空间划分为大小相等的 page，将物理内存划分为大小相等的 page frame。系统按 page 进行分配，因此逻辑上相邻的 page 在物理内存中不要求连续落在相邻的 page frame 中。

分页的核心作用，是消除可变分区方案对大块连续物理内存的依赖。

系统通过 page table 记录 page number 到 page frame number 的映射关系，page table entry（PTE） 则是这种映射的基本表项。

处理器获得逻辑地址后，自动将其分解为 page number 和 page offset，再依据 page number 查找 page table，得到对应的 page frame number，最后与 page offset 组合形成物理地址。

一般情况下，每个进程各有一份 page table，page table 本身存放在内存中。在 x86 体系结构下，当前地址空间所使用的 page table 根地址由 CR3 寄存器给出。

分页显著减轻了外碎片问题，但由于 page 大小固定，进程地址空间末尾的最后一个 page 往往不能被完全用满，因此会产生内碎片。page size 的选择体现了典型折中：page 较大时，page table 规模更小，TLB 覆盖范围更大，但内碎片增加；page 较小时，碎片减轻，但 page table 和管理开销上升。典型的 page size 可以是 4KB 或 4MB。

flowchart TD
    LA[逻辑地址] --> SPLIT[拆分为 page number p 与 page offset d]
    SPLIT --> PT[用 p 查 page table]
    PT --> PTE[读取 PTE]
    PTE --> PF[得到 page frame number f]
    PF --> JOIN[拼接 f 与 d]
    JOIN --> PA[物理地址]
    PA --> MEM[物理内存]

6.5 段式内存管理

段式管理按程序的逻辑结构划分地址空间，例如主程序段、子程序段、数据段、栈段、数组段等。段长不等，段名或段号具有语义。物理内存中仍需为每个段分配连续空间，因此底层分配方式更接近可变分区。段表记录段号、段首地址与段长，地址转换时由处理器用段号查表，再将段内偏移加到段首地址上。

段式管理的优点，在于它更接近程序员或编译器的逻辑视图，天然适合按模块共享、保护和独立增长。其缺点在于段长可变，仍然面临外碎片和连续分配压力。页式强调系统管理方便，段式强调程序逻辑自然，两者的优势方向不同。

flowchart TD
    LA[逻辑地址] --> SPLIT[拆分为 段号 s 与段内偏移 d]
    SPLIT --> ST[查段表]
    ST --> BASE[得到段首地址 base]
    BASE --> ADD[base + d]
    ADD --> PA[物理地址]

6.6 段页式内存管理

段页式管理试图同时保留段式的逻辑结构与页式的离散装入能力。对用户或编译器而言，地址空间仍按段组织；对系统而言，每个段又进一步分页。段表不再直接给出段首物理地址，而是给出该段页表的起始位置与长度；页表再完成页号到页框号的映射。于是一个逻辑地址先解释为“段号 + 段内地址”，再把段内地址解释为“页号 + 页内偏移”，最终经过段表与页表两级转换形成物理地址。

这种方案适用于既要求模块化保护与共享，又希望降低连续分配压力的系统。它的代价是地址转换链更长、表结构更复杂，但换来更强的表达能力与更好的物理适应性。

flowchart TD
    A[逻辑地址] --> B[段号 s + 段内地址]
    B --> C[查段表得到该段页表位置]
    C --> D[段内地址拆成 页号 p + 页内偏移 d]
    D --> E[查该段页表]
    E --> F[得到页框号 f]
    F --> G[拼接 f 与 d]
    G --> H[物理地址]

7. 内存“扩充”技术

当物理内存不足时，系统并不只能拒绝装入。历史上存在四类思路：紧凑、覆盖、交换和虚存。紧凑依靠移动进程合并碎片；覆盖依靠程序内部模块互斥装入；交换依靠把进程在内存与外存之间成批移动；虚存则进一步把“不必全部驻留”制度化，形成现代按页调入的主流机制。课件在这一讲中重点展开的是覆盖与交换。

7.1 覆盖技术

覆盖技术用于“程序总大小超过物理内存总量”的场景。其思想不是要求整个程序同时驻留，而是依据调用结构，把不会同时执行的模块安排到同一块覆盖区中，由操作系统在需要时自动装入相应模块。若某程序的总逻辑规模为 54K，但常驻区加两个覆盖区只占 30K，则该程序仍可在 30K 内存中执行。覆盖技术本质上是显式利用程序控制流中的互斥关系，以时间换空间。

覆盖技术的限制也很明显。它要求程序员先划分模块并声明覆盖关系，因此对用户不透明性差，编程复杂度高；模块切换时需要从外存装入新覆盖块，执行时间增加。随着更自动化的虚存机制出现，覆盖技术逐渐退出主流，但其思想仍然揭示了“程序不必整体常驻”的基本方向。

7.2 交换技术

交换技术最早用于小型分时系统，其典型动作是 roll out 与 roll in。系统在内存紧张时把某些进程整体换出到外存交换区，再把其他进程换入内存执行。交换区通常是专门划出的磁盘区域，以便高效进行底层块读写。交换的对象主要是进程的映像，尤其包括运行期会变化的栈与堆等内容。

交换并不是简单搬运，它与调度器紧密耦合。系统需要决定何时交换、交换谁、被换出内容在磁盘何处保存、换入时是否回到原位置，以及进程空间增长如何处理。处于等待 I/O 的进程通常不适合被换出，因为它与设备操作存在外部依赖。被换出的进程重新装入时也不必回到原先位置，只要系统具备动态重定位能力即可。

进程空间增长给交换带来额外困难。数据段可能向上扩展，栈段可能向下扩展，若内存布局过于紧凑，就会压缩未来增长空间。解决这一问题的思路，要么在分配时预留增长余地，要么在换入时重新选择位置并依赖重定位重新建立映射。

flowchart LR
    A[内存紧张] --> B[选择可换出进程]
    B --> C[写入交换区]
    C --> D[释放内存]
    D --> E[换入待运行进程]
    E --> F[建立新映射]
    F --> G[继续调度执行]

8. 归纳

内存管理的主线可以概括为四个层次。第一层是地址抽象，核心问题是把逻辑地址与物理地址分离，形成地址空间。第二层是底层组织，核心问题是如何管理空闲物理内存并在分配、回收、碎片与性能之间取得平衡。第三层是映射方案，核心问题是选择连续区、分区、分页、分段或段页式等不同组织方式。第四层是内存扩充，核心问题是在物理内存不足时，如何通过紧凑、覆盖、交换和进一步的虚拟化继续支撑程序执行。
