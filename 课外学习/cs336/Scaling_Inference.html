Lecture 9

核心问题与方法论意义

Scaling Laws 的意义不在于给出若干经验公式，而在于提供了一套可操作的大模型设计方法：通过小规模实验拟合规律，并据此外推大规模系统的表现。其关键价值在于将高成本试错转化为可预测的实验流程。

在大模型训练场景中，核心问题不再只是 ” 采用何种架构或参数配置 “，而是：模型能力是否随规模呈现稳定、可外推的变化规律。若这一点成立，则大模型工程可以从经验驱动的试错过程转向近似 ” 实验科学 ” 的工作方式。

Scaling Law 的基本形式与变量含义

常见形式为：

L∝N−α

或写为对数形式：

logL=−αlogN+C

其中，(N) 并非固定指代某一变量，而是 ” 被缩放的规模变量 “。在不同语境中，(N) 可以表示：

  • 数据规模（例如训练 token 数）
  • 模型规模（参数量）
  • 计算量（FLOPs）
  • 训练步数（steps）

因此，Scaling Law 的本质不是 ” 某个固定变量与 loss 的关系 “，而是：模型性能与系统规模之间存在稳定的幂律型关系。

为什么是幂律：统计起源与尺度视角

幂律关系并非 Transformer 或 LLM 特有现象。其更基础的来源可追溯至统计估计中的误差收敛规律。以均值估计为例，样本均值的误差会随样本数按多项式速度衰减（典型形式如 (1/n)），这已经构成最基本的 scaling behavior。

从更高层视角看，幂律常意味着系统在一定区间内呈现 ” 无特征尺度（scale-free）” 行为，即不存在明显的单一最优规模点，规模扩展在较大范围内仍能带来稳定收益（即便边际收益逐渐减小）。这也是深度学习系统在实践中能够长期依赖 scale 推进性能的重要原因之一。

LLM 的数据 Scaling 为什么 ” 慢但稳定 ”

课堂中一个重要观察是：LLM 的数据 scaling exponent 通常显著小于经典统计模型中的 (1)。换言之，随着数据量增加，loss 的下降速度较慢，但下降过程稳定且持续。

一种常见解释路径是高维函数逼近视角：当目标函数（或数据分布）对应的有效复杂度很高时，误差收敛速度会变慢，可粗略写作：

Error∼n−1/d

其中 (d) 可理解为某种 ” 有效维度 ” 或问题复杂度刻画。语言分布通常包含多层结构（语法、语义、知识、风格、世界规律等），因此其学习过程天然表现出：

  • 对数据量需求大；
  • 收敛缓慢但长期可持续；
  • 在较大规模区间不易快速饱和。

这一点有直接研究含义：若未来出现更强的归纳偏置（inductive bias）或更高效的表示方式，本质上可能是在降低有效复杂度，从而改变 exponent。相较于常数项优化，这更接近范式级变化。

Data / Model / Compute Scaling 的统一理解

课程分别讨论数据、参数、batch、compute 等因素，但在更统一的视角下，这些关系都在回答同一个问题：在给定资源约束下，如何提升模型实际获得的有效信息量。

Data Scaling

经验上，在合适区间（非极小数据区，也非接近不可约误差区），训练 loss 与数据规模在 log-log 坐标上近似线性，即存在稳定的数据 scaling 区间。这意味着增加数据在该区间内通常是有效且可预测的。

Model Scaling

参数量增加通常对应更强的表示容量与拟合能力，经验关系常写作：

L∝P−β

其中 (P) 为参数量。这一事实在工程上对应一个常见现象：模型性能提升往往首先受 scale 支配，而不是由局部技巧决定。

Compute Scaling

计算量并非单一作用因素，而是在以下三者之间进行分配：

  • 模型规模
  • 数据规模
  • 优化充分程度（训练步数、训练时长）

因此，“compute scaling” 的真正意义是：在固定 FLOPs 预算下，如何分配资源以获得最优训练结果，而不是简单地 ” 增加算力 “。

数据质量、模型质量与 ” 截距 ” 效应

这一部分是 scaling law 在工程实践中最具价值的结论之一。更完整的形式可写为：

L(N)=A⋅N−α+C

其中：

  • (α)：斜率（scaling efficiency）
  • (A)：常数项（决定曲线整体位置，可视为截距相关项）
  • (C)：不可约误差（loss floor）

在 log-log 图上，(α) 决定长期趋势，(A) 决定曲线整体高低。若两种方法具有相近斜率，但其中一种具有更低的 (A)，则其在几乎所有规模区间内都更优。这意味着：

  • 在相同参数量下得到更低 loss；
  • 在相同数据量下得到更低 loss；
  • 在相同计算量下得到更低 loss。

因此，” 好的数据 ” 和 ” 好的模型/训练策略 ” 常表现为降低曲线的常数项（截距），从而在不改变 scaling 趋势的情况下获得等价于大量算力的性能提升。该结论可表述为：

好的数据或模型常常不改变 scaling 斜率，但会降低 scaling 曲线的常数项，从而在相同计算量下实现等价于大量算力的性能提升。

这类改进在工业实践中极具价值，因为 brute-force scaling 成本高昂，而截距下移通常可以通过数据清洗、去重、混合策略、tokenizer 优化、训练稳定性改进、学习率调度等方式实现。

需要区分的是：并非所有改进都仅作用于截距。实践中至少存在三类情况：

  1. 常数项改进（最常见）：曲线整体下移，斜率近似不变；
  2. 有效规模改进：在同等算力下提高 ” 有效 (N)”；
  3. 指数改进（最稀有）：改变 exponent，构成范式级突破。

Joint Scaling Law 与资源最优前沿

单独讨论 data scaling 或 model scaling 在实际训练中是不充分的，因为数据规模、模型规模与计算预算相互耦合。更有用的是联合形式，例如课程中提及的典型表达：

L=n−α+m−β+C

其中 (n) 表示数据规模，(m) 表示模型规模。

该类关系的核心价值在于将 ” 如何训练一个大模型 ” 的模糊问题转化为明确的资源配置问题：在固定预算下，数据与模型应如何配比。实践中若偏离最优配比，容易出现资源浪费：

  • 小模型配过多数据（模型无法充分吸收）；
  • 大模型配过少数据（欠训练）；
  • 训练长度不合理（浪费 compute）。

因此，大模型训练可以被视为寻找某种 ” 资源最优前沿（compute-efficient frontier）” 的过程，竞争重点不仅在总算力规模，也在是否更接近这条前沿。

Chinchilla 的意义：从 ” 更大模型 ” 到 ” 更充分训练 ”

Chinchilla 经常被概括为经验比例关系：

Tokens≈20×Parameters

但其真正意义不在于 “20” 这一数字本身，而在于它对训练策略的纠偏。相较于 GPT-3 时代偏向 ” 更大模型、相对不足数据 ” 的训练方式，Chinchilla 强调在固定训练 compute 下，模型容量与训练经验（数据）应更平衡，否则会出现明显的 undertraining。

这一结论可用更一般的表述概括：

  • 参数量提供容量（capacity）；
  • 数据与训练过程提供经验（experience）；
  • 只有容量而经验不足，不会自然转化为更高性能。

课堂后段还强调了一个实践补充：训练最优不必然等于部署最优。若系统生命周期中 inference compute 占比更大，则更高训练投入（例如更多 token/param）可能是合理的长期优化策略。课程中列举了多代模型 tokens/param 持续上升的现象，可视为这一现实约束下的工程选择。

Pretraining 与 Downstream Scaling 的差异

Scaling laws 在 pretraining loss 上表现得较为稳定，这是其能支持工程外推的前提之一。但该稳定性并不自动延伸到所有下游任务。下游任务通常包含更复杂的分布偏移、能力阈值、评测噪声与任务结构差异，因此 task-level scaling 往往更不规则。

这一点提示后续研究方向不应仅停留在 pretraining loss 的缩放规律，还应系统研究：

  • task scaling laws
  • capability scaling laws
  • reasoning scaling laws

即：不同能力与任务指标是否具有各自可预测的缩放结构，以及它们与 pretraining loss 的关系如何建立。

Scaling 与 Emergence：连续增长与阈值效应

课程内容可与 “emergence” 问题形成一个统一视角。许多表面上的 ” 突然涌现 ” 现象未必意味着底层能力发生离散跳变，也可能是：

  • 底层表示能力随规模连续增强；
  • 评测指标本身存在阈值；
  • 当连续提升越过阈值时，观测结果呈现 ” 突然出现 ”。

因此，emergence 可以在相当程度上被解释为 scaling 的观测效应（至少在部分任务与指标上）。这一视角对机制解释（mechanistic interpretability）具有直接意义：研究问题可从 ” 能力为何突然出现 ” 转向 ” 哪些内部表示/电路随规模连续形成，并在某个阈值后开始显著影响任务表现 “。

总结性框架（统一心智模型）

Lecture 9 的内容可压缩为一个统一框架：

  • 模型性能由容量（参数）、经验（数据）与优化（compute/训练长度）共同决定；
  • scaling law 提供了这些因素与 loss 之间的可预测关系；
  • 工程优化多数体现为降低常数项（截距）；
  • 范式突破更可能体现为 exponent 的改变。

可用非严格表达写作：

Loss≈Scaling Trend(data, params, compute)+Constant Factors+Irreducible Floor

其中：

  • Scaling Trend：决定长期趋势（斜率）；
  • Constant Factors：决定同等规模下的性能位置（截距）；
  • Irreducible Floor：决定最终下限。

该框架比单独记忆某篇论文中的具体指数更具迁移性，因为它可以用于判断新方法的作用类型：是在改变斜率、降低截距，还是提高有效规模。

复习时的判别问题（用于分析新方法）

  1. 该方法的提升更可能来自斜率变化、截距变化，还是有效规模变化？
  2. 提升是否在更大规模下仍然成立，还是会被 scale washing 掩盖？
  3. 当前系统主要瓶颈是数据、模型还是 compute？
  4. 实验是否仍处于 small-scale regime，从而导致结论外推性不足？
  5. 是否混淆了训练最优与部署最优？

Lecture 10

0. 本节课定位

本节课讨论大模型 推理（inference） 的核心问题与优化方法，重点不是训练，而是：

  • 推理 workload 的特性（尤其是自回归生成）
  • 推理中的性能瓶颈（尤其是内存瓶颈）
  • 模型侧、算法侧、系统侧的优化方法
  • 为什么未来可能需要更 ” 推理友好 ” 的架构

1. 本节课核心主线

可以概括为一条主线：

  1. 推理为什么难
  2. 难在哪里（顺序生成 + 内存瓶颈 + 动态请求）
  3. 如何优化（有损 / 无损 / 系统调度）
  4. 为什么未来可能需要改变架构

2. 推理的应用场景（不仅是聊天）

推理不仅用于聊天机器人，还广泛存在于：

  • 在线产品（chatbot、代码补全、搜索增强等）
  • 模型评估（benchmark / instruction following）
  • test-time compute（多次采样、搜索、反思等）
  • 强化学习中的样本生成与打分（RL 训练过程本身也依赖推理）

Insight

很多方法在训练阶段能提升指标，但如果推理成本过高，实际部署价值会显著下降。推理成本往往是持续发生的，而训练成本通常是一次性或阶段性投入。

3. 推理性能指标（必须区分）

3.1 TTFT（Time To First Token）

  • 用户从发出请求到看到第一个 token 的时间
  • 直接影响交互 ” 响应感 “

3.2 Latency（每 Token 延迟）

  • 常用单位：seconds/token
  • 影响连续输出速度（” 字蹦出来快不快 “）

3.3 Throughput（吞吐量）

  • 常用单位：tokens/second
  • 影响系统整体产能与成本效率

3.4 三者关系

  • 三者通常存在 trade-off，不可能在所有场景同时最优
  • 产品侧关注 TTFT 和交互延迟
  • 平台侧关注 throughput 与成本

Insight

推理优化不能只看单一指标。不同业务目标对应不同优化方向：

  • 交互式应用：优先 TTFT、稳定 token latency
  • 批处理/平台服务：优先 throughput、成本/吞吐比

4. 训练 Vs 推理：根本差异

4.1 训练的特点

  • 通常可并行处理完整序列（teacher forcing）
  • 更容易形成大规模矩阵乘法，充分利用 GPU

4.2 推理的特点（自回归）

  • token-by-token 生成
  • 时间维度上必须顺序执行（sequential dependency）
  • 难以像训练那样把硬件 ” 喂饱 “

Insight

这是本节课最核心的第一性原理之一：

推理优化的许多方法，本质上都在绕开或缓解 ” 自回归顺序生成 ” 的限制。

5. 算术强度（Arithmetic Intensity）与推理瓶颈

5.1 定义

算术强度衡量：

  • 每搬运 1 字节数据，能做多少 FLOPs
  • 公式（概念上）：
    [
    \text{Arithmetic Intensity} = \frac{\text{FLOPs}}{\text{Bytes transferred}}
    ]

5.2 为什么它重要

  • 如果算术强度高：更可能 compute-limited
  • 如果算术强度低：更可能 memory-limited

5.3 课程中的关键结论（矩阵乘法示例）

对于常见矩阵乘法分析，在一定假设下可得到近似结论：

  • 算术强度 ~ batch size (B)

这意味着：

  • batch 小（尤其 (B=1)）时，算术强度低 → 更容易 memory-limited
  • 推理 decode 阶段常常接近这种情况（小 batch + 顺序生成）

Insight

这解释了一个常见误解：
“GPU 算力很强，为什么推理还慢？“
因为 decode 阶段很多时候不是算力不够，而是 内存带宽/数据搬运 在卡性能。

6. 推理的两阶段：Prefill Vs Decode（Generation）

这是推理系统设计中的关键划分。

6.1 Prefill（提示词编码阶段）

  • 输入已有 prompt（可并行处理多个 token）
  • 更容易形成较大矩阵计算
  • 更可能接近 compute-limited

6.2 Decode / Generation（生成阶段）

  • 每步通常只生成 1 个 token
  • 时间维度串行
  • batch size 受并发请求限制，不稳定且常偏小
  • 更容易 memory-limited

Insight

很多现代推理系统会专门区分 prefill 和 decode 的调度策略、kernel 优化策略。这不是实现细节，而是因为它们本质上是两类不同的 workload。

7. KV Cache：推理性能的主战场

7.1 为什么需要 KV Cache

在自回归生成中，如果每生成一个 token 都重新计算整个历史前缀，会造成大量重复计算。
KV cache 用于缓存历史 token 的 key/value 表示，避免重复计算。

7.2 推理中的关键代价

推理阶段常见开销包括：

  • 模型参数读取（parameter read）
  • KV cache 读取（随上下文增长）
  • 少量新计算

当上下文变长时，KV cache 的内存占用和访问成本会成为主要瓶颈。

7.3 简化理解

总内存压力可近似看作：

  • 参数内存 + KV cache 内存（且后者随 batch / context 增长）

Insight

这节课的一个重要视角是：
KV cache 不是 ” 实现细节 “，而是决定推理性能上限的核心变量。
因此很多推理优化，本质上是在优化 KV cache 的大小、布局或访问方式。

8. 有损优化（Lossy）一：缩小 KV Cache

本节课重点介绍的思路是：在可接受精度损失下，降低 KV cache 占用，从而提升推理效率。

8.1 GQA（Grouped-Query Attention）

基本思想

  • Query heads 数量保持较多
  • Key/Value heads 数量减少（多个 query 共享一组 KV）

可看作介于以下两者之间：

  • MHA（每个 head 一套 KV）
  • MQA（所有 query 共用一套 KV）

作用

  • 显著减少 KV cache 大小
  • 提升推理吞吐/降低延迟
  • 允许更大 batch fit 进显存

Insight

GQA 的价值在于：它利用了 head 间的冗余结构，以较小精度代价换取显著的推理收益，是 ” 系统压力反向推动架构改造 ” 的典型例子。

8.2 MLA（Multi-head Latent Attention）

基本思想

将 KV 表示压缩到较低维的潜空间（latent space），再用于注意力计算，从而减少缓存大小。

特点

  • 相比 GQA，MLA 更像一种 ” 表示压缩 ” 而不是简单 head 共享
  • 需要处理与位置编码（如 RoPE）的兼容性问题

Insight

MLA 可以理解为把 KV cache 当作一种 ” 需要编码的状态表示 “。这类思路和表示学习、信息压缩、可解释性中的 ” 信息承载子空间 ” 问题有潜在关联。

8.3 CLA（Cross-Layer / 跨层共享 KV 的思路）

基本思想

在层之间共享或部分共享 KV 信息（类比 GQA 在 head 维度共享）。

目标

  • 进一步降低 KV cache 占用
  • 在精度与效率之间寻找更优 Pareto frontier

Insight

这引出一个重要研究问题：不同层是否真的需要独立、完整的记忆？如果层间冗余较高，那么跨层共享可能具有较大空间。

8.4 Local Attention（局部注意力）

基本思想

每个 token 只关注局部窗口内的上下文，不看全部历史。

推理收益

  • 超出窗口的历史可不再保留/访问
  • KV cache 不再线性依赖完整上下文长度（在某些设计下可显著缓解增长）

风险

  • 长距离依赖建模能力下降
  • 常采用 hybrid 方案（局部层 + 稀疏全局层）

Insight

长上下文能力不一定只能通过 ” 无限扩窗口 ” 实现，也可以通过结构设计（混合层、稀疏全局机制）来保留关键全局信息。

9. 有损优化（Lossy）二：不仅优化 Transformer，也可能替换架构

课程提出了更激进的问题：
如果 Transformer + 自回归在推理上天然不友好，是否应考虑更适合推理的架构？

9.1 SSM（State Space Models）方向

代表脉络（课程提及方向）

  • S4
  • Mamba（输入依赖的状态更新）
  • Transformer-SSM 混合模型（如 Jamba 类思路）

推理意义

  • 用固定大小状态（O(1) state）替代随序列增长的 O(T) KV cache（理想化视角）
  • 对长序列推理有潜在优势

Insight

这类架构的价值不仅是 ” 新颖 “，而是可能从根本上改变推理瓶颈类型。
推理优化的终极形式，可能不是把现有 Transformer 服务做得更好，而是让模型本身更适合推理。

9.2 文本 Diffusion / 非自回归生成方向（课程作为探索线索）

吸引力

  • 并行生成多个 token（而不是严格 token-by-token）
  • 通过多步 refinement 修正结果

难点

  • 文本离散性与语言约束使其比图像 diffusion 更难
  • 质量/速度/稳定性 trade-off 更复杂

Insight

这条路线的重要性在于它提出了一个核心问题：

能否从根本上摆脱 ” 单车道串行生成 ” 的限制？

即使最终不是 diffusion 胜出，这个问题本身仍然很重要。

10. 有损优化（Lossy）三：量化（Quantization）

10.1 量化的目标

降低数值精度（例如 fp16/bf16 → int8/int4），从而：

  • 减少模型参数内存
  • 减少 KV cache 内存（若对 KV 也量化）
  • 降低内存带宽压力
  • 提升吞吐 / 降低延迟（尤其在 memory-limited 场景）

10.2 量化类型

  • QAT（Quantization-Aware Training）：训练时考虑量化误差
  • PTQ（Post-Training Quantization）：训练后量化，部署更方便

10.3 实践中的关键问题：Outliers（离群值）

  • 少量激活/权重离群值会显著放大量化误差
  • 一些方法（如 LLM.int8() 类思路）会对 outliers 单独处理（保留高精度通道/路径）

10.4 激活感知量化（Activation-aware）

  • 利用激活统计信息识别 ” 更重要、对误差更敏感 ” 的参数
  • 对关键部分保留更高精度，对其余部分低比特量化

Insight

量化不是纯粹的数值压缩问题，也是在回答：
模型中哪些信息最脆弱、最不能被压缩？
这与模型结构理解、通道重要性分析、可解释性有潜在联系。

11. 有损优化（Lossy）四：剪枝（Pruning）+ 蒸馏修复（Distillation）

11.1 基本流程（典型思路）

  1. 用校准集评估模块重要性（层/头/隐藏维度等）
  2. 删除不重要部分，得到更小模型
  3. 用原模型对剪枝后模型进行蒸馏修复（恢复性能）

11.2 优点

  • 结构层面减少计算与内存开销
  • 可与量化等方法叠加

11.3 风险

  • 剪枝后质量下降明显，需要蒸馏/finetune 修复
  • 不同任务下模块重要性不一定一致

Insight

这体现了一种通用压缩 pipeline 思维：
结构改造 → 能力恢复（蒸馏）
这个框架不限于剪枝，也适用于很多 ” 先压缩后修复 ” 的模型工程实践。

12. 无损优化（Lossless）重点：Speculative Decoding / Sampling

这是本节课最重要的算法优化之一。

12.1 直觉

  • 自回归生成慢（逐 token）
  • 但 ” 验证多个候选 token” 通常可并行完成（尤其借助 prefill-like 计算）

于是可以：

  1. 用小模型（draft model）先生成多个候选 token
  2. 用大模型（target model）并行验证这些候选
  3. 尽可能一次接受多个 token，减少大模型串行生成步数

12.2 核心收益

把原本的 ” 纯串行生成 ” 变成：

  • 串行提出候选
  • 并行验证候选

12.3 为什么它被称为 ” 无损 ”

正确设计的 speculative sampling / decoding 可以保证：

  • 最终采样分布与目标模型一致（exact distribution）
  • 不是启发式近似加速，而是有概率校正机制的严格方法

Insight

Speculative decoding 的思想非常重要：
它不是单纯 ” 猜得更快 “，而是 把慢操作（生成）替换为更多快操作（验证）。
这是一种典型的系统/算法共设计思想，也和经典计算机体系结构中的 speculative execution 有相似精神。

13. 在线推理的动态 Workload 难点

与训练阶段 ” 规则张量 batch” 不同，线上 serving 的请求具有强动态性。

13.1 请求异步到达

  • 请求到达时间不同
  • 等待组成固定 batch 会增加早到请求的延迟（伤害 TTFT）

13.2 前缀共享

  • 多请求可能共享 system prompt / 公共前缀
  • 同一 prompt 的多样本生成也存在共享空间

13.3 长度不一致（ragged sequences）

  • 请求长度差异大
  • padding 会造成计算浪费与显存浪费

Insight

训练代码默认世界是 ” 规则 dense tensor”；线上推理世界往往是 ” 异步 + 不等长 + 突发流量 “。
这解释了为什么 serving 系统设计本身就是一个独立研究/工程方向。

14. Continuous Batching（连续批处理）

14.1 传统批处理的问题

  • 以请求为单位组成 batch
  • 一批开始后通常要等该批完成，才能处理下一批
  • 导致 GPU 空闲、TTFT 变差、资源利用率不足

14.2 Continuous Batching 的核心思想

  • 以 iteration（解码步） 为调度粒度，而不是请求粒度
  • 每个解码 step 都可以动态加入新请求
  • 已完成请求移出 batch，未完成请求继续参与下一步

14.3 效果

  • 提高 GPU 利用率
  • 降低等待时间
  • 更好适应在线动态流量

Insight

Continuous batching 的关键不是 “batch 更大 “，而是 重定义了 batch 的生命周期：
从 request-level batching 转为 iteration-level batching。

15. PagedAttention（vLLM 的代表性思路）

这是将操作系统分页思想引入 KV cache 管理的典型方法。

15.1 传统 KV 分配的问题

若为每个请求预留连续大块 KV 空间（按最大长度预估）：

  • 容易产生内部碎片（分配多、实际用少）
  • 容易产生外部碎片（多段空洞导致难以连续分配）

15.2 PagedAttention 的基本思想

  • 将 KV cache 按块（pages/blocks）管理
  • 使用非连续物理内存布局
  • 通过映射关系组织逻辑上的连续序列

15.3 额外收益：前缀共享与 Copy-on-Write

  • 对共享 system prompt 或相同前缀请求，可以共享前缀 block
  • 当需要分叉生成时再复制（copy-on-write）

Insight

PagedAttention 的意义不仅在于 ” 某个技巧 “，而在于问题抽象的变化：
推理内存管理可以借鉴操作系统的分页与虚拟内存思想。
这说明推理优化很多时候是系统抽象层面的创新，而不是单一 kernel 优化。

16. 本节课的总体方法论

16.1 先判断瓶颈，再选优化

对于大模型推理（尤其 decode）：

  • 常见瓶颈是内存带宽 / KV cache 访问，而非纯算力
  • 因此优先考虑：
    • KV cache 压缩 / 管理优化
    • 参数/KV 量化
    • 动态调度与批处理策略
    • 无损并行验证类方法（speculative decoding）

16.2 将模型优化、算法优化、系统优化放在一个统一框架中

本节课展示的优化可以按层次划分：

  • 模型结构层：GQA / MLA / CLA / local attention / 新架构（SSM）
  • 数值表示层：量化
  • 压缩修复层：剪枝 + 蒸馏
  • 算法层：speculative decoding
  • 系统层：continuous batching / PagedAttention

它们共同优化的目标是：

  • 延迟（TTFT / token latency）
  • 吞吐量（throughput）
  • 精度（quality）
  • 成本（cost）

16.3 推理友好性将成为架构评估的重要维度

未来模型设计不应只关注训练 loss 或 benchmark，还应考虑：

  • 推理阶段的内存占用
  • 顺序依赖强度
  • 动态 serving 兼容性
  • 长上下文成本曲线

Insight

从研究选题视角看，” 推理友好型架构 ” 会越来越重要。
这为模型结构设计、系统优化、可解释性分析之间的交叉研究提供了空间。

17. 复习提纲（适合考试/面试快速回顾）

17.1 必须会解释的概念

  • TTFT / latency / throughput 的区别与 trade-off
  • prefill vs decode 的差异
  • 为什么 decode 常 memory-limited
  • KV cache 为什么是推理主战场
  • GQA / MLA / local attention 的推理动机
  • speculative decoding 为什么能加速、为什么可保持分布正确
  • continuous batching 的调度粒度变化
  • PagedAttention 为什么能缓解碎片与支持前缀共享

17.2 一句话总结本节课

推理优化的核心不是单纯减少 FLOPs，而是在自回归顺序生成和动态服务场景下，尽可能减少内存访问成本、提升并行验证能力与系统利用率。
