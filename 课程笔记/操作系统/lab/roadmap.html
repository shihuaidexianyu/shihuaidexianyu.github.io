roadmap

Mar 20, 202625 min read

这组 Lab 以 xv6 为基础，围绕启动、自举、系统调用、进程控制、虚拟内存、调度、文件系统与并发同步等核心主题，逐步扩展一个教学内核的能力边界。其整体目标并不在于孤立地实现若干系统调用，而在于将一个结构简洁、便于教学分析的 Unix-like 内核，推进为一个能够完成启动、自举、用户程序装载、资源管理、内存优化、命名空间操作与同步控制的完整实验系统。

xv6 的原始设计强调最小实现与结构透明，适合呈现操作系统的基本构成。课程化的九个 Lab 则在此基础上增加了更接近现代操作系统的接口与机制，使实验对象从“能够运行的教学内核”转变为“能够承担更复杂资源管理任务的实验内核”。这一演进过程具有清晰的层次性：先建立系统的运行闭环，再扩展进程与地址空间，再引入调度策略和内存优化，最后将文件系统命名空间与同步机制补充完整。

一、总体结构：从“能运行”到“能管理”

九个 Lab 所构成的知识主线，实际上对应操作系统中的九类基本问题。

第一类问题是启动与自举，即系统如何从虚拟硬件环境进入内核，再从内核进入第一个用户程序。 第二类问题是系统调用接口，即用户态如何通过约定的 ABI 进入内核态并请求内核服务。 第三类问题是进程控制，即进程的创建、等待、退出、让出处理器与时间相关机制。 第四类问题是地址空间管理，即堆增长、内存映射、虚拟地址区域与进程页表之间的关系。 第五类问题是调度，即多个可运行进程之间如何分配 CPU 时间。 第六类问题是虚拟内存优化，即缺页分配、写时复制与页面置换。 第七类问题是文件系统接口，即路径解析、目录访问、文件描述符扩展与挂载机制。 第八类问题是同步，即多个执行流如何在共享资源上维持正确的并发行为。 第九类问题是评测与实验组织本身，即如何让整个内核按照固定流程自动构建、装载、运行测试并结束执行。

这一结构使九个 Lab 形成了一个由浅入深的系统化课程，而不是若干彼此分离的实现任务。

二、Part0：构建链、镜像生成与评测底座

Part0 处于整个路线图的最底层，承担实验环境、构建流程与运行方式的统一工作。其核心内容包括容器环境准备、工具链配置、Makefile 组织方式、内核与用户程序的编译流程，以及文件系统镜像 fs.img 的生成逻辑。

这一阶段的关键意义在于建立实验系统的外部执行闭环。内核并不是单独存在的二进制文件，而是与用户程序、文件系统镜像、测试程序和 QEMU 运行参数共同构成一个完整的实验对象。make fs 的作用在于将用户程序与测试程序一起放入镜像，使之后的 Lab 可以直接在同一个虚拟文件系统中装载这些程序并运行。make run 或其他等价流程则将内核与镜像交给 QEMU 启动，从而构成完整的实验入口。

Part0 所解决的问题不是内核内部机制，而是实验系统的“可运行性”。没有这一层，后续关于进程、系统调用或虚拟内存的所有实现都无法进入可测试状态。

三、Part1：启动链路、自举过程与基础系统调用

Part1 是整个实验主线第一次进入内核本体的部分。其核心内容由两条线索构成：一条是启动链路，另一条是基础系统调用接口。

启动链路的完整过程包括：QEMU 创建虚拟硬件环境，RustSBI 作为引导层运行并装载 xv6 内核，内核完成自身初始化并挂载文件系统，随后创建第一个用户进程，将极小的 initcode 复制到该进程的地址空间中执行，再由 initcode 发起 exec("/init")，装载真正的 /init 程序。此后，控制流由内核正式转入用户态程序。

这一过程中的 initcode 具有严格的自举性质。它不是通常意义上的用户程序，而是一段最小机器码，用来完成从“内核刚创建出的第一个用户进程”到“装载并执行 /init”之间的过渡。其存在的原因在于系统刚完成启动时，尚不存在现成的用户态运行时环境，因此第一个跳板只能由内核直接注入。

与此同时，Part1 还建立了用户态与内核态之间的第一条系统调用通路。课程测试程序依赖特定的系统调用编号与参数约定，因此系统调用 ABI 必须与测试程序集保持一致。该部分所涉及的 getcwd、write、getpid、times、uname 不是孤立的函数补全，而是对“用户程序发起调用—陷入内核—按编号分发—执行内核处理函数—返回用户态”这一完整机制的第一次打通。

flowchart TD
    A[QEMU 启动] --> B[RustSBI]
    B --> C[xv6 内核初始化]
    C --> D[挂载 fs.img]
    D --> E["userinit 创建首个用户进程"]
    E --> F[注入 initcode]
    F --> G["执行 exec('/init')"]
    G --> H["装载 /init"]
    H --> I["/init 启动测试程序"]
    I --> J[系统调用进入内核]

Part1 结束后，系统已经具备最小的启动闭环、最小的用户态执行环境与最小的内核服务接口。后续所有 Lab 都建立在这一闭环之上。

四、Part3：进程控制、父子关系与时间机制

从知识主线上看，进程控制构成了操作系统资源管理的第一根支柱。Part3 围绕 wait、waitpid、clone、fork、execve、getppid、exit、yield、sleep、gettimeofday 等主题展开，目标在于使系统真正能够管理多个执行流的创建、等待、切换与终止。

进程的本质并不仅仅是“正在运行的程序”，而是一个同时包含执行状态、寄存器上下文、页表、文件描述符表、父子关系和退出信息的内核对象。fork 负责创建新的执行流，execve 负责在保留进程身份的前提下用新程序替换地址空间，wait 与 waitpid 负责在父子关系中完成退出状态的回收，yield 负责主动让出处理器，sleep 则将进程置于等待事件的休眠状态。

时间机制在这一阶段也具有重要地位。内核以时钟中断提供基本时间基准，并据此支持 sleep、gettimeofday 等接口。时间不仅是用户程序可见的资源，也是后续调度策略实现的必要基础。

Part3 的完成，使内核从“能跑一个程序”迈向“能管理多个进程”。这一步决定了后续调度与同步机制能否成立，因为没有完善的进程生命周期管理，就不存在真正意义上的多任务系统。

五、Part2：文件接口、堆空间增长与内存映射

Part2 同时覆盖了文件系统接口与地址空间映射两条主线，因而处于整个路线图中的枢纽位置。其主要内容包括 open、openat、brk、mmap、munmap 等。

在文件接口方面，openat 的引入使路径解析不再局限于当前工作目录，而能够相对于一个目录文件描述符进行解释。这一变化扩大了路径操作的表达能力，也为后续 mkdirat、unlinkat 等接口铺平了道路。文件描述符表、全局打开文件表以及目录项、inode 或 FAT 条目之间的连接关系，构成了从“路径字符串”到“内核文件对象”的完整转换链。

在地址空间方面，brk 使进程可以显式调整堆区边界，而 mmap/munmap 则引入了虚拟内存区域管理的概念。地址空间不再只是代码段、数据段、堆和栈的静态组合，而成为由若干虚拟内存区域动态构成的集合。VMA 的出现意味着进程地址空间开始具备“保留某段虚拟地址并赋予其权限、来源和用途”的能力。

    D[目录/文件元数据]

Part2 的完成，使文件访问与地址空间管理都获得了更一般化的接口。文件系统开始呈现出命名空间的一面，地址空间开始呈现出虚拟映射的一面。后续的 COW、缺页分配与页面置换，都依赖这一阶段建立起来的内存区域模型。

六、Part4：调度策略与处理器时间分配

Part4 将注意力从“进程如何存在”转向“进程何时运行”。其核心任务是实现 RR、Priority 和 MLFQ 等多种调度算法，并在统一框架下支持测试与切换。

调度属于操作系统中的策略层。进程控制提供了可运行对象，时钟中断提供了时间边界，调度器则决定在某一时刻将 CPU 分配给哪个进程。RR 通过固定时间片在进程之间循环，强调基本公平性；Priority 引入优先级差异，使处理器倾向于为重要性更高的进程服务；MLFQ 则通过多级反馈队列在不同运行特征之间进行区分，在交互型与计算型任务之间取得折中。

这一阶段的关键在于，调度不再只是“顺序扫描可运行队列”的简单实现，而成为一个可设计、可比较的系统策略问题。调度算法的不同，直接影响响应时间、吞吐量、公平性与饥饿风险等系统行为。

flowchart LR
    A[可运行进程集合] --> B{调度策略}
    B --> C[RR]
    B --> D[Priority]
    B --> E[MLFQ]
    C --> F[获得 CPU]
    D --> F
    E --> F
    F --> G[时间片结束/阻塞/退出]
    G --> A

Part4 完成后，内核首次具备明确的处理器分配政策。其意义在于将前面的进程与时间机制转化为真正的多任务调度系统。

七、Part5：Lazy Allocation 与 Copy-on-Write

Part5 标志着虚拟内存系统从“静态映射”进入“按需管理”的阶段。其核心内容包括 Lazy Allocation 与 Copy-on-Write 两种典型优化策略，并配合若干辅助系统调用，用于观察进程逻辑地址空间大小与物理页占用情况。

Lazy Allocation 的本质，是将物理内存分配从“地址空间声明时刻”推迟到“真正发生访问时刻”。例如 sbrk() 扩大堆区边界时，不必立即为新增虚拟页分配物理页，而只需更新进程的地址空间范围；真正访问某个尚未映射的地址时，处理器触发缺页异常，内核再在异常处理过程中分配物理页并建立映射。这样可以避免大量从未被使用的页占据内存。

Copy-on-Write 的本质，是将页内容复制从 fork() 时刻推迟到“真正发生写入时刻”。父进程和子进程在 fork() 之后共享相同的物理页，这些页被标记为只读；当某一方试图写入时，处理器触发写保护异常，内核再分配新页、复制原内容并修改页表。这样能够显著减少 fork() 的即时开销。

flowchart LR
    A[sbrk/mmap 扩展地址空间] --> B[仅保留虚拟地址范围]
    B --> C[首次访问]
    C --> D[缺页异常]
    D --> E[分配物理页并映射]

    F[fork 共享物理页] --> G[页表设为只读]
    G --> H[写入尝试]
    H --> I[写时缺页]
    I --> J[复制物理页并改写映射]

Part5 的完成，使虚拟内存不再仅仅是“地址到页框的映射表”，而变成一种具有惰性分配与延迟复制能力的资源管理系统。内存资源因此获得更高的利用效率。

八、Part6：页面置换、受害者选择与交换缓冲

Part6 在 Part2 与 Part5 的基础上继续推进，处理内存压力下的页面置换问题。其主要实现为 FIFO 和 LRU 两类置换策略，并将目标范围限定在 mmap 区域，同时采用内存内 swap buffer 作为教学化的替代实现。

页面置换的出现，意味着系统不再假定物理内存始终足够。虚拟地址空间可以大于当前可用物理页数，因此在物理内存紧张时，必须选择某些页暂时移出活跃集合。FIFO 依据进入内存的先后顺序选择受害者，逻辑简单；LRU 则试图逼近“最近最少使用”的局部性原则，以提高实际命中率。

这一阶段将虚拟内存系统的控制粒度进一步细化。Part2 负责建立可映射区域，Part5 负责按需建立物理映射，Part6 则负责在内存不足时重新配置这些映射，使地址空间的可用规模不再被当前物理页数量严格限制。

flowchart TD
    A[虚拟页已建立映射] --> B[物理内存压力上升]
    B --> C{置换策略}
    C --> D[FIFO]
    C --> E[LRU]
    D --> F[选择受害页]
    E --> F
    F --> G[写入 swap buffer]
    G --> H[撤销当前映射]
    H --> I[再次访问该页]
    I --> J[缺页并换回]

Part6 的意义在于补全虚拟内存系统在“资源不足”情形下的行为逻辑，使系统不仅能够分配内存，也能够在竞争条件下回收与重分配内存。

九、Part7：文件系统接口扩展与命名空间操作

Part7 将文件系统从“基本文件访问”推进到“更完整的命名空间操作”。其中包括 dup3、getdents、mkdirat、unlinkat、fstat、mount、umount 等接口。

这一阶段的关键变化在于，文件系统的核心对象不再只是普通文件，而是目录、路径、文件描述符以及文件系统树结构之间的整体关系。dup3 使文件描述符的复制更可控，适合重定向与资源管理；getdents 使目录内容能够作为可遍历对象暴露给用户程序；mkdirat 与 unlinkat 将目录文件描述符引入路径操作，使路径解释具有上下文；fstat 通过文件描述符返回底层对象属性；mount 与 umount 则使多个文件系统实例能够形成树状连接关系。

flowchart TD
    A[路径字符串] --> B[目录文件描述符 dirfd]
    B --> C[路径解析]
    C --> D[目录项]
    D --> E[文件/目录对象]
    E --> F[fstat/getdents]
    E --> G[mkdirat/unlinkat]
    E --> H[mount/umount]

Part7 的完成，使文件系统从“打开并读写文件”的层次，上升到“组织与操作整个命名空间”的层次。文件系统因此更接近 Unix 系统中“以路径树组织资源”的真实形式。

十、Part8：信号量、阻塞唤醒与同步控制

Part8 位于整个路线图的高层，负责将并发执行从“能够发生”推进到“能够正确发生”。其核心是信号量机制及相关系统调用，例如 sem_create、sem_destroy、sem_p、sem_v。

信号量的本质是一种带计数的同步原语，用于在多个执行流之间协调共享资源的访问。P 操作在计数不足时使进程阻塞，V 操作在资源归还时唤醒等待者。这一机制建立在内核锁、睡眠与唤醒、调度切换等已有基础设施之上，因此 Part8 实际上是对前面多个子系统的一次综合运用。

信号量机制的引入，使生产者—消费者、哲学家就餐等经典并发问题可以在用户态以受控方式表达。并发不再只是多个进程同时运行的事实，而成为一个可以通过内核原语精确调节的行为结构。

stateDiagram-v2
    [*] --> Running
    Running --> Waiting : P 操作且资源不足
    Waiting --> Runnable : V 操作唤醒
    Runnable --> Running : 调度器选中
    Running --> [*] : 退出

Part8 结束后，系统具备了处理并发协作所需的基本同步机制。至此，进程、调度、睡眠、唤醒与锁等先前分散存在的机制，形成了面向并发控制的整体闭环。

十一、整套 Lab 的能力演进

九个 Lab 所形成的能力演进，并非简单叠加，而是逐层构成一套完整的内核体系。

Part0 提供实验系统的构建与运行底座。
Part1 建立从启动、自举到基础系统调用的最小闭环。
Part3 建立进程控制、父子关系与时间机制。
Part2 建立文件访问接口与地址空间区域管理。
Part4 在已有进程和时钟机制上引入可设计的调度策略。
Part5 将虚拟内存推进到惰性分配与延迟复制。
Part6 进一步补全内存紧张条件下的页面置换逻辑。
Part7 将文件系统推进到完整命名空间操作层次。
Part8 将并发运行转化为可控的同步行为。

整套 Lab 的最终结果，是一个在教学规模内相对完整的 Unix-like 操作系统实验体系。其核心能力包括：完成启动、自举与用户程序装载；通过系统调用向用户态提供内核服务；管理进程与父子关系；维护并优化进程地址空间；按照不同策略分配 CPU 时间；在物理内存紧张时进行页面回收与置换；通过路径和文件描述符管理文件系统命名空间；通过信号量等原语协调并发执行流的资源访问。
