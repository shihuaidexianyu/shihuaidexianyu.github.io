你现在的任务是基于我提供的 PDF、PPT、PPTX、Markdown（.md）课程材料与现有 Markdown 笔记，整理出一份可直接保存的个人学习笔记。这份笔记首先是给我自己看的，其次才是作为参考资料。

要求如下：

【定位】

  1. 成品应当像一个认真但会思考的学生的课堂笔记，而不是冷冰冰的教材、课件摘要或百科全书。
  2. 文风应当准确、清晰、有个人温度。允许在关键处用“你”“我们”来拉近理解距离，也允许在直觉解释时使用口语化表达。
  3. 内容要直接陈述知识本身，但不要隐藏整理过程和个人理解。如果某个概念我（整理者）觉得容易混淆、很巧妙、或者有点反直觉，应该把这个感受写进去。
  4. 叙述应体现该学科的知识特点：既要说明核心概念、基本原理和整体框架，也要落实到关键机制、结构关系、分析步骤、公式含义、典型实例、适用条件与实际意义。

【输入材料处理原则】

  1. 输入材料可能是 PDF、PPT、PPTX、Markdown（.md）或它们的组合，必须统一纳入整理范围。
  2. 若输入中已经包含 Markdown 笔记，这些内容应视为课程材料的一部分，用于补充、校正、衔接和展开知识，而不是简单原样保留或机械拼接。
  3. 若不同材料之间存在重复定义、相似流程或相同主题的不同表达，应进行合并整理，保留信息更完整、逻辑更清楚、表述更稳定的版本。
  4. 若不同材料之间存在信息互补，应将其整合为连贯正文，不得因为来源不同而割裂表述。
  5. 若不同材料之间存在冲突，应优先采用定义更严格、逻辑更完整、上下文更一致的表述；若冲突涉及条件差异，应将适用前提与边界写清。
  6. 若 Markdown 文件本身已经具有一定结构，可以吸收其章节组织方式，但最终成品仍须服从课程知识逻辑，而不是照搬原始笔记结构。
  7. 若 Markdown 文件中包含零散记录、提纲、提示语、课堂口吻、未展开结论或不完整公式，应改写为完整、可连续阅读的笔记正文。
  8. 若 Markdown 文件中含有与正文无关的纯个人待办、纯情绪吐槽或明显无关的整理痕迹，这些内容不得直接出现在成品中。但如果某个备注确实包含对知识理解不可缺少的信息（例如“这里容易搞混”“注意符号约定变了”），则应将其改写成正文的一部分或放入适当的 callout 中。
  9. 若原材料中的公式、符号、图示文字或 Markdown 语法存在明显书写错误、渲染错误或不完整之处，应在不改变原意的前提下自动修正为可正常渲染、语义明确的规范写法。

【内容完整性】

  1. 必须覆盖材料中出现的所有重要内容，不能只保留主干框架。
  2. 必须纳入以下内容：
    • 概念、定义、术语、核心命题
    • 理论框架、前提假设、基本模型或基本机制
    • 分类、对比、层次关系、适用条件与边界
    • 公式、符号、变量含义及其解释
    • 关键流程、推导链条、分析步骤、操作路径或处理过程
    • 图表中承载的关键信息
    • 历史背景、典型方法、经典案例、常见应用场景
    • 材料中的追问、判断题、辨析题、思考题所对应的知识内容
    • 计算示例、推导步骤与结果含义
    • 容易混淆的概念边界与常见误解
  3. 如果某一页、某一段或某个材料片段信息量较少，也要将其有效吸收进整体叙述，而不是直接略去。
  4. 不能因为追求简洁而省略公式解释、变量定义、步骤展开、例子背景或结论成立的前提。
  5. 对材料中的表格、区间、阈值、等级、公式、比较结论、对应关系，必须转化为清晰正文，而不是简单抄录。
  6. 允许将相邻页面、重复定义、相似流程合并叙述，避免重复解释；同类内容首次完整说明，后续只保留新增信息、差异、变化与适用条件。
  7. 对实现细节、补充图表、练习材料或边界讨论，只保留理解主线所必需的部分；其余内容可压缩整合，但不能遗漏会影响理解的关键知识环节。

【结构要求】

  1. 严格保持原材料的章节边界，不要把不同讲、不同文件或不同主题模块的内容合并成一章。
  2. 如果原本是两章或多章独立内容，输出时必须保持为相互独立的章节。
  3. 每一章内部按照知识逻辑展开，形成连贯正文，而不是零散罗列。
  4. 输出格式为 Markdown，适合直接放入 Obsidian。
  5. 可以使用必要的层级标题，但不要把整份笔记写成只有提纲的列表结构。
  6. 如果材料本身存在“定义—分类—机制—公式—示例—应用—局限”之类的展开路径，应尽量保留其知识推进逻辑，而不是改写成碎片化摘录。
  7. 合并与重组内容时，只能在同一章节或同一主题模块内部进行，不得跨章节重写知识边界。
  8. 分层阅读设计：长文档或复杂推导应提供“快速通道”。可以在每章开头或每个复杂小节顶部使用 Obsidian callout（如 > [!summary]、> [!intuition]）给出人话版摘要，让未来的我能先抓住核心，再决定是否深入细节。

【表达要求】

  1. 允许并鼓励直觉解释。在引入抽象概念或复杂公式时，可以用比喻、类比、或“我们可以把它想象成……”这类句式帮助理解。但比喻应当准确，不能为了通俗而歪曲原意。
  2. 允许使用人称代词。在解释思路时可以使用“我们”“你”，让笔记读起来更像有人在讲解。
  3. 减少纯禁止性的提示语，但以下类型仍应避免：
    • 对整理过程的说明（如“根据材料内容……”“这里把前后内容串起来……”）
    • 依赖原始材料上下文的残缺表述（如“待补充”“TODO”“未完”“课上提到”“见上页”“如图所示可知”）
  4. 不要写成对材料的评论，要把内容直接写成知识正文。
  5. 不要出现引用记号、页码标注、文献式引用、材料页序说明。
  6. 不要把正文写成幻灯片式短句堆砌，必须写成可连续阅读的教材式表述。
  7. 每个重要概念或公式之后，尽量跟上一个具体例子或应用场景，让知识落地。

【解释深度】

  1. 对抽象概念、关键指标、核心机制或基本模型，必须解释其对象、形成原因、工作逻辑、适用范围以及与相近概念的区别。
  2. 对容易混淆的内容，必须给出明确边界，而不是仅作并列罗列。
  3. 对复杂机制、分析链条或推导过程，应在推导前先说明“为什么要这么做”。例子应自然嵌入正文，作为知识说明的一部分。
  4. 对流程性内容，应明确输入、前提条件、中间步骤、关键转换关系与最终结果。
  5. 对判断题、辨析题式内容，不要保留问答形式，而要改写成完整的陈述性知识，并说明其成立或不成立的条件与理由。
  6. 对计算性内容，不能只给结论，必须保留必要的计算链条、推导步骤与解释环节。

【图表与公式处理要求】

  1. 图中的分布形态、层次关系、位置关系、状态变化、流程路径、结构对应、相关方向、误差来源、分类框架等，必须转化为正文叙述。
  2. 表格中的关键字段、比较维度、数值区间、等级划分、变量关系、对应规则，必须融入正文，不可简单省略。
  3. 对公式和推导，必须尽量保持步骤完整，包括：
    • 已知条件
    • 变量定义
    • 代入过程
    • 运算结果
    • 结果解释
    • 使用前提与适用边界
  4. 如果图表体现的是“不同方法、模型、机制或路径之间的对比”，必须写出比较维度，而不是只分别给出定义。
  5. 所有公式不能只给出表达式，必须说明：
    • 各符号分别代表什么
    • 公式用于解决什么问题
    • 在什么条件下使用
    • 计算结果应如何解释
    • 该结果不能说明什么
  6. 若原材料中的公式写法不规范，应统一改写为标准数学表达，不保留明显错误、残缺或歧义写法。
  7. 概率论、统计学、离散数学等内容中的交、并、补、条件概率、连乘、求和、集合表示等，必须使用规范数学记号，不得保留口语式、逗号式或半自然语言式混写。

【mermaid 使用要求】

  1. 只有在流程、层次、路径、状态变化或关系结构较复杂时才使用 mermaid。
  2. mermaid 图是正文的一部分，不是装饰。其作用是承载结构性信息，从而降低正文密度。
  3. mermaid 图必须能直接在 Obsidian 中渲染。
  4. 适合使用 mermaid 的场景包括但不限于：
    • 知识框架或层次结构
    • 机制运行流程
    • 状态变化与转移关系
    • 多个概念、方法或模块之间的关系
    • 分析流程、处理流程、推导路径、决策路径
    • 输入—处理—输出的系统过程
  5. 正文负责解释概念、原因、逻辑、意义、边界与局限；mermaid 图负责表达结构、流程、关系、状态与路径。图与正文应互补，不机械重复。
  6. 单幅图只承载一个核心机制；若内容过于复杂，应拆为多幅小图，而不是绘制一张过大、难以阅读的总图。
  7. 如果仅用清晰正文即可表达，则不要强行加入 mermaid。
  8. 所有 mermaid 图必须使用标准 fenced code block 书写，即： 
    graph TD
...

不得使用非标准包裹方式，不得遗漏代码块结束标记。
9. 一个 mermaid 代码块只放一张图，不得在同一个代码块中混入多张图或普通正文。
10. 图内节点文本若包含空格、中文、符号或较长短语，应尽量使用引号包裹，减少渲染歧义。
11. mermaid 图仅用于表达结构，不得为了“形式丰富”而堆砌无必要图示。

【公式格式要求】

  1. 所有数学公式统一使用 LaTeX 语法书写。
  2. 行内公式一律使用 $...$ 包裹，不使用 \(...\)。
  3. 独立公式块一律使用 $$...$$ 包裹，不使用 \[...\]。
  4. 公式必须符合 Markdown / Obsidian 的直接渲染习惯，避免混用其他数学定界符。
  5. 非公式内容不得误放入公式环境中；变量、符号、表达式需在正文语义中自然出现。
  6. 集合必须写成 \{...\}，不得写成裸 {...}。
  7. 使用可伸缩定界符时，必须成对书写，如 \left( ... \right)、\left[ ... \right]、\left\{ ... \right\}，不得出现 \left 与 \right 不配对，或把花括号误写成 \left\{...\right\}。
  8. 指数与下标中只允许出现合法数学表达式，不得混入多余逗号、残缺符号或无意义字符，如 q^{,n-k}、q^{,k-r} 之类写法必须修正为规范形式。
  9. 连乘关系必须使用 \cdot、\times 或适当的数学间距 \, 表示，不得用逗号代替乘号。
  10. 交、并、补、条件概率、事件同时发生等写法必须统一规范，如 A \cap B、A \cup B、\overline{A}、P(A \mid B)；不得保留会导致歧义或渲染不稳定的临时写法。
  11. 需要独立展示的公式必须单独成段书写，公式前后应留空行，避免与普通段落、列表项或引用块紧贴造成渲染异常。
  12. 不要将 > 引用块与 $$...$$ 独立公式块混用；凡是需要独立公式展示的内容，必须放在普通段落环境中单独成块书写。
  13. 公式块前后不得添加任何 Markdown 标题符号，包括但不限于 #、##、###、-、*、>。
  14. 不得把公式块误写成标题。也就是说，禁止出现 # $$ ... $$、# \[ ... \] 或任何“标题 + 公式块”的错误结构。
  15. 公式必须原样保留，不得丢失公式中的等号 =、加号 +、减号 -、上下标、括号或转义符。
  16. 若一行以 # 开头，则该行必须是标题文本，不能是公式，也不能包裹公式。

【数学记号规则】
17. 涉及“x 上方一横”的写法时，一律使用 LaTeX 形式 \bar{x}。
18. 禁止使用 x^_、x¯、x上横线、x头顶横线 等非规范替代写法。
19. 若表示多个字符上方横线，使用 \overline{...}；若仅表示单个变量 x 的上横线，一律使用 \bar{x}。

【输出前自检】
20. 在输出最终答案前，先自行检查：

  • 是否有任何公式块前面误加了 #
  • 是否有任何公式中的 = 被遗漏
  • 是否把 \bar{x} 错写成了 x^_ 或其他非标准形式
  • 是否有任何面向读者的提示语变成了对整理过程的说明
  • 是否有重要概念只有定义没有例子或直觉解释

【Markdown 稳定性要求】

  1. 输出必须是可直接保存、可直接渲染的 Markdown 成品，而不是“接近 Markdown”的半结构化文本。
  2. 各级标题、列表、代码块、公式块必须闭合完整，不能出现未闭合 fenced code block、未闭合公式块或结构断裂。
  3. 正文中若出现代码块、公式块、mermaid 块，其前后应保持合理空行，以提升 Obsidian 渲染兼容性。
  4. 不得输出会污染正文的编辑痕迹，例如多余反引号、残留的提示词片段、未删除的占位语、未完成句、重复段落或截断内容。
  5. 长文档中同一概念、同一符号、同一公式对象必须尽量保持前后一致，不得在不同部分随意切换记号。
  6. 若某个图、表、公式无法稳定转写，则优先改写为清晰正文，不要输出存在明显渲染风险的残缺结构。

【输出前静默自检要求】

在输出最终 Markdown 正文前，必须自行完成一次静默检查，确保至少满足以下条件，但不要把检查过程写出来：

  • 不存在 > $$ 或引用块包裹独立公式的情况
  • 不存在 \left 与 \right 不配对的情况
  • 不存在指数或下标中的异常字符、残缺表达式或多余逗号
  • 不存在把乘法误写成逗号的情况
  • 所有集合均写为 \{...\}
  • 所有 mermaid 代码块均使用标准 fenced block，且成对闭合
  • 不存在未闭合的代码块、公式块或标题结构
  • 不存在明显的公式渲染错误、Markdown 结构错误或语义残缺
  • 每个核心知识点都有至少一句“人话解释”或例子

【输出要求】

  1. 直接输出最终版 Markdown 笔记正文，不要附加说明，不要解释写法，不要写“以下是整理结果”之类的话。
  2. 输出内容应当是可直接保存的课程笔记成品，而不是提示词说明书。
  3. 若材料中包含多章内容，每一章单独成章，彼此不要混写。
  4. 不要省略任何对理解该课程核心逻辑有帮助的知识环节。
  5. 成品目标是**“适合复习的个人笔记”**——它应该既有知识的完整性，也有阅读的温度。在不遗漏关键知识的前提下，优先减少重复、压缩枝节、控制段落密度。
  6. 最终输出必须兼顾三点：知识完整、表述稳定、语法可渲染。
