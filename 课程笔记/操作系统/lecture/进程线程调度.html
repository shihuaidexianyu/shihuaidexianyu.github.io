进程线程调度

进程线程调度

Apr 23, 202621 min read

课前碎碎念与核心主线

进程/线程调度可以说是操作系统里最核心的“大管家”工作。当我们有一堆任务要跑，但 CPU 只有 1 个（或少数几个核）时，大管家必须决定：什么时候换人？换谁上？怎么平滑地切换？

整个笔记的主线可以归纳为灵魂三问：

  1. When & How：什么时候该切进程？切的时候底层发生了什么（上下文切换）？

  2. What：不同的场景下（批处理、交互式、实时），怎么挑进程才能兼顾“公平”和“效率”？（经典调度算法）

  3. Real World：现实生活里的系统（Linux、Windows、多核处理器）是怎么在各种权衡中做到极致的？

一、 调度的基本概念与层次

操作系统中的调度其实是一个立体的概念，按照时间尺度可以分为三个层次：

  1. 长程调度（Long-term / 作业调度）：决定哪些新创建的进程有资格进入内存的“活跃进程集合”。这就像是大学招生，名额有限，先决定谁能进校门。

  2. 中程调度（Medium-term / 内存调度）：负责在内存和外存之间进行进程的换入换出（Swap）。目的是腾出内存空间，缓解内存压力。

  3. 短程调度（Short-term / 微观调度）：这是本章的绝对核心。 决定就绪队列里的哪个进程能立马获得 CPU 运行。它的时间尺度是毫秒级的，发生极其频繁，所以算法必须极其高效。

状态流转全景图

我们日常所说的“调度”，主要集中在就绪态（Ready）到运行态（Running）的转换。

二、 调度时机与上下文切换 (When & How)

1. 什么时候会发生调度？ (When)

只要会导致当前进程无法继续占用 CPU，或者有更紧急的进程出现时，就会触发调度：

  • 主动放弃：进程执行完毕、执行 yield 主动让出、发生异常终止。

  • 被动阻塞：进程需要等待 I/O 操作或其他资源（如锁），进入阻塞态。

  • 外部中断（核心机制）：时钟中断（时间片用完了），或者 I/O 中断（某个优先级更高的阻塞进程被唤醒了，重新回到就绪态，抢占了当前 CPU）。

2. 上下文切换 (Context Switch)

上下文切换是指一个进程让出处理器，由另一个进程占用的过程。

形象理解上下文切换

就像是两个人共用一张书桌学习。A 同学走的时候，必须把自己的草稿纸、看到哪一页的标记（寄存器状态、程序计数器 PC 等）全都收进自己的书包里（进程控制块 PCB）。B 同学坐下时，再把自己的书本摊开。如果频繁换人，大部分时间都浪费在收拾桌子上了，真正看书的效率极低。

上下文切换是有**代价（Cost）**的：

  1. 直接开销：内核执行切换代码消耗的 CPU 时间（保存和恢复寄存器、切换地址空间/页目录，这些指令通常很昂贵）。

  2. 间接开销（更致命）：高速缓存（Cache）和 TLB（地址变换快表）失效。新进程上来后，Cache 里的数据全都不认识了，必须重新从慢速内存里抓取。

三、 调度算法的设计考量与指标

不同的系统（服务器 vs 个人电脑 vs 手机）追求的目标完全不同。设计算法前，我们要先确立评价指标。

1. 核心评价指标

  • 吞吐量 (Throughput)：单位时间内完成的进程数量。批处理系统最看重这个。

  • 周转时间 (Turnaround Time, TT)：每个进程从提出请求（到达）到运行完成的时间。

Tturnaround​=Tcompletion​−Tarrival​

  • 响应时间 (Response Time, RT)：从提出请求到第一次给出回应的时间。交互式系统（如鼠标点击、键盘打字）最看重这个。

  • CPU 利用率：CPU 做有效工作的时间比例（尽量别让它闲着）。

  • 公平性 (Fairness)：避免某些进程长生不老，某些进程被“饿死”（Starvation）。

2. 调度算法的四个设计支点

我们在看具体算法时，要时刻关注它是怎么在下面四个维度做选择的：

  1. 优先级（静态 vs 动态）：是出生就定好，还是根据你等了多久、表现如何随时调整？

  2. 占用方式（抢占式 vs 非抢占式）：非抢占式是“一旦占有，直到结束或阻塞”；抢占式是“谁官大谁随时踢人”。

  3. 进程偏好（CPU 密集型 vs I/O 密集型）：

    • CPU 型：一直在做数学计算。

    • I/O 型：动不动就要读盘、发网络请求。

    为什么要偏好 I/O 密集型？

    现代操作系统通常更偏好调度 I/O 密集型进程。因为它们只需要极短的 CPU 时间来触发 I/O 指令，一旦触发就会去旁边挂起等设备了。早点让它运行，能让外设和 CPU 最大程度地并行工作。

  4. 时间片（Time Slice / Quantum）大小：太长退化为先来先服务；太短则上下文切换开销爆炸。

四、 经典的调度算法图鉴

我们将算法分为适用于“批处理系统”和“交互式系统”两大阵营。

阵营 A：批处理系统 (关注吞吐和周转)

1. 先来先服务 (FCFS - First Come First Serve)

  • 机制：按就绪先后排队，非抢占。

  • 致命缺点：护航效应 (Convoy Effect)。一个跑一天的长作业排在前面，后面需要跑 1 秒的短作业全得等死。平均周转时间极其糟糕。

2. 短作业优先 (SJF - Shortest Job First) / SRTN (抢占式)

  • 机制：谁需要的执行时间短，谁先上。SJF 是非抢占的；其抢占版叫做最短剩余时间优先 (SRTN)，只要新来的作业比当前作业剩余时间短，直接抢 CPU。

  • 优点：在所有算法中，能得到数学上最短的平均周转时间。

  • 缺点：1) 极度不公平，长作业会饥饿。 2) 需要“预测未来”，系统根本不知道一个代码要跑多久。

3. 最高响应比优先 (HRRN - Highest Response Ratio Next)

  • 机制：这是 FCFS 和 SJF 的折中绝招。每次调度时，计算响应比 R：
R=1+处理时间等待时间​

  • 妙处：

    • 如果是短作业（处理时间极小），R 会很大，类似 SJF，优先调度。

    • 如果是长作业，刚开始 R 很小被冷落，但随着等待时间的累积，R 会稳步上升。等得足够久，长作业也能上位，完美解决了“饥饿”问题。

阵营 B：交互式系统 (关注响应时间与公平)

4. 轮转调度 (RR - Round Robin)

  • 机制：排成一队，每个人固定玩一个时间片（比如 10ms），用完就乖乖去队尾排队。

  • 缺点：对长度相近的进程反而不利。且默认的 RR 对 I/O 密集型不公平（它一去等 I/O 就会主动让出时间片，回来又要重新排队）。

  • 优化 → 虚拟轮转法 (Virtual RR)：给从 I/O 唤醒的进程单独开一个小灶（Auxiliary Queue 辅助队列）。从这个队列出来的进程，能立马抢占 CPU，并只使用它上次没用完的时间片，保证了 I/O 设备的利用率。

5. 优先级调度 (HPF) 与 优先级反转危机

  • 机制：给进程打分，分高的先上。

  • 致命危机：优先级反转 (Priority Inversion)。

    [案例分析：火星探路者号故障]

    假设有高 (H)、中 (M)、低 (L) 三个进程。L 正在运行，并拿到了某个共享资源的锁。此时 H 来了，抢占 L，但 H 也需要那个锁，于是 H 被锁阻塞。此时 M 来了，M 的优先级高于 L，M 抢占 L 开始愉快运行。

    结果：H（火星车通信模块）在苦苦等待 L（气象模块）释放锁，但 L 被中优先级的 M（数据传输模块）剥夺了 CPU 根本没法运行！导致最高优先级的 H 迟迟得不到执行，系统崩溃重启。

    解法：优先级继承 (Priority Inheritance)。当 L 拿着 H 想要的锁时，系统临时把 L 的优先级“提拔”到和 H 一样高，让 L 赶紧跑完释放锁，从而解救 H。

6. 多级反馈队列 (Multilevel Feedback Queue, MFQ)

这是现代操作系统的调度基础，是集大成之作，完美体现了折中权衡 (Tradeoff)。

graph TD
    subgraph "MFQ 机制"
    New(新进程到达) --> Q1[Q1: 最高优先级, 时间片最短]
    Q1 -->|时间片耗尽| Q2[Q2: 中优先级, 时间片中等]
    Q2 -->|时间片耗尽| Q3[Q3: 低优先级, 时间片最长]
    Q1 -.->|"主动让出/IO阻塞"| IO[去等待I/O]
    IO -.->|唤醒| Q1_Return(返回原高优先级队列)
    end

  • 核心思想：

    1. 分类：设置多个队列，优先级越低，给的时间片越长。

    2. 降级试探：所有新来的进程，默认都是最高优先级。如果在短时间片内没跑完，说明它是 CPU 密集型长作业，直接降级打入冷宫（下移一个队列）。

    3. 保护交互：I/O 密集型（比如敲键盘响应）通常在微秒级就主动阻塞了，根本用不完时间片，所以它们会永远留在最高优先级队列，获得极速响应！

  • 意义：系统不需要预测未来，通过观测进程实际耗费时间片的行为，自动把交互式短作业留在顶层，把算力猪（长作业）沉淀到底层。

五、 多核与多处理器调度 (SMP)

当 CPU 从单核走向多核（SMP，对称多处理器）时，调度就不仅是“谁来跑”，还有“在哪跑”的问题。

1. 核心矛盾：缓存亲和性 vs 负载均衡

  • 缓存亲和性 (Cache Affinity)：进程上次在 CPU 0 跑，Cache 里全存着它的数据。如果这次把它调度到 CPU 1，会发生极大的 Cache Miss 开销。所以我们希望进程尽量在同一个核上“从一而终”。

  • 负载均衡 (Load Balancing)：如果死守亲和性，可能出现 CPU 0 排长队，CPU 1 闲的发慌的情况。

2. 解决方案模型

  • 单队列调度 (SQMS)：所有 CPU 共享一个就绪队列。好处是天然负载均衡，坏处是严重破坏了缓存亲和性，且队列锁竞争激烈。

  • 多队列调度 (MQMS)：每个 CPU 维护自己的就绪队列。好处是亲和性极佳，不需要全局锁。坏处是容易负载失衡。

  • 解法 → 工作窃取 (Work Stealing)：平时大家各扫门前雪（优先执行本地队列）。当某个 CPU 实在没活干了，就偷偷去隔壁 CPU 的队列末尾“偷”几个任务过来。Go 语言的 GMP 调度模型就重度依赖了这种思想。

OpenEuler 中的实现

Linux / OpenEuler 采用多队列，且每个处理器都有一个底层的迁移线程 (migration/CPUID)。当检测到负载失衡时，会将进程从高负载的核强制迁移到空闲核。

六、 实时调度 (Real-Time Scheduling)

实时系统的重点不是快，而是确定性 (Determinism)——必须在最后期限 (Deadline) 之前完成。

1. 可调度性条件

假设有 m 个周期性事件，事件 i 的周期是 Pi​，处理它需要 CPU 时间 Ci​。系统不崩溃的前提（可调度条件）是它占用 CPU 的时间总和必须小于 1：

i=1∑m​Pi​Ci​​≤1

2. 两大经典算法

  • 速率单调 (RM, Rate-Monotonic)：静态优先级。周期越短，优先级越高。任务一来优先级就写死了，实现简单。

  • 最早截止时间优先 (EDF, Earliest Deadline First)：动态优先级。谁的最后期限快到了，谁优先级就最高。虽然最优，但实现复杂，需要频繁重排计算。

七、 真实世界的调度器设计

我们来看看操作系统大厂们是如何把上述理论落地到现实的。

1. Windows 的线程调度

  • 调度单位：Windows 调度的基本单位是线程。

  • 机制：采用基于动态优先级的、抢占式多任务调度。

  • 优先级分层：共有 32 个优先级（0-31）。其中 16-31 是固定的实时优先级；1-15 是可变优先级；0 给系统清零空闲页的线程。

  • 时间配额 (Quantum)：时间片在 Windows 里是一个整数配额。用完配额如果没有同级对手，会补充配额继续跑。

  • 避免饥饿与优先级提升机制（核心亮点）：

    1. I/O 操作完成后，立刻提权（响应越高的设备提权越多），同时配额减 1。

    2. 获取锁/信号量被唤醒时，优先级提 1。

    3. 防饥饿扫描：系统有个平衡集管理器（Balance Set Manager），每秒扫描一次。如果发现有就绪线程超过 300 个 ticks 没分到 CPU（被饿晕了），直接一脚踢到优先级 15，并赏赐 4 倍的正常时间配额！等它痛快跑完，再打回原形。

2. Linux 调度的演化史

Linux 的调度器经历了几次史诗级的重构，每一次都是针对上一次的痛点：

演化路径

  1. Linux 2.4 (基于 O(n))：遍历全局队列找最高优先级。可扩展性差。

  2. Linux 2.6 (O(1) 调度器)：发明了双数组架构（Active 和 Expired 数组）。从高到低扫位图找最高优先级， O(1) 时间复杂度。但它花了很大力气用复杂的经验公式去猜谁是交互式进程，代码极其难以维护，且经常猜错。

  3. SD / RSDL (楼梯调度)：抛弃“猜”交互式的思路，回归彻底的公平。

  4. CFS (Completely Fair Scheduler)：当前内核的主流。彻底干掉时间片和动态优先级阵营，引入全新的虚拟运行时间。

CFS：完全公平调度算法的核心内幕

CFS 核心直觉

想象我们在操场跑圈。CFS 的逻辑是：不允许有人跑得太靠前，大家必须齐头并进。

系统给每个人发一个“虚拟跑表”(vruntime)。

  • 优先级低（普通人），现实里跑 1 步，跑表上记 2 步。

  • 优先级高（飞毛腿），现实里跑 1 步，跑表上只记 0.5 步。

    调度器每次都去队伍里挑**当前跑表数字最小（落后最多）**的人，让他跑。这样一来，高优先级的人因为表走得慢，被选中的次数就多，分到的真实时间就多。

  • 核心公式：
vruntime=实际运行时间×进程的权重NICE_0_LOAD​ 
（权重由 nice 值决定，nice 越小优先级越高，权重越大，计算出来的 vruntime 增长就越慢）

  • 数据结构与查找：

    不排队列了，CFS 把所有就绪进程按照 vruntime 作为 Key 排进一棵红黑树 (RB-Tree) 里。

    每次调度，直接摘取红黑树最左下角的节点（vruntime 最小的进程）投入运行。运行一段时间后，它的 vruntime 变大了，再插回树里，可能就排到右边去了。

  • 休眠唤醒补偿：

    如果一个进程去等 I/O 睡了一大觉，它的 vruntime 岂不是一直停留在几个小时前？那它醒来岂不是可以一直霸占 CPU？

    不会。 CFS 会跟踪整个队列的最小基准线 min_vruntime。当休眠进程醒来时，它的 vruntime 会被强制“拔高”到 min_vruntime 减去一小段补偿值的位置。让它既能马上获得调度，又不会无休止地霸占 CPU。
