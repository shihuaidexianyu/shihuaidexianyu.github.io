处理器调度

处理器调度是操作系统对 CPU 这一关键资源进行分配、控制与协调的机制。其基本任务是在一个或多个 CPU 面前，从就绪进程或就绪线程中选择适当的执行对象，并把处理器使用权交给它。若系统中不存在可运行的进程或线程，则由系统空闲进程或空闲线程占用处理器，使处理器维持在受控状态而不是无定义状态。处理器调度所回答的问题可以归结为三个方面：按照什么原则选择执行对象，何时进行调度，以及如何完成从一个执行对象到另一个执行对象的切换。

调度并不是单一层次上的活动，而是贯穿进程进入系统、驻留内存、获得处理器直至退出系统的全过程。长程调度又称作业调度或宏观调度，其任务是在创建新进程时决定该作业是否进入当前活跃进程集合，本质上控制的是系统的并发度。中程调度对应进程在内存与外存之间的交换，从存储资源管理的角度决定哪些进程部分或全部换出、哪些进程重新换入。短程调度又称微观调度，是最频繁、最核心的调度层次，它直接面向处理器资源分配，从就绪队列中选择进程或线程进入运行态；其时间尺度通常为毫秒级，因此实现必须高效。

CPU 调度发生在能够改变当前可运行集合或运行顺序的事件之后。典型时机包括：进程正常结束或因异常终止；新进程创建；运行进程因等待 I/O 或其他资源而阻塞；阻塞进程被唤醒回到就绪态；当前进程时间片耗尽；以及进程主动执行 yield 放弃处理器。事件发生后，硬件首先完成中断、异常或系统调用的响应，控制流进入内核；内核处理完相关事件后，某些进程状态可能已经发生改变，就绪队列也可能被重组，因此在返回用户态前的最后阶段，调度程序会依据既定调度算法重新做出选择。

从调度结果进入真正运行，还必须经过进程切换。进程切换是指一个进程让出 CPU，由另一个进程占用 CPU 的过程；上下文切换则更强调底层硬件状态的保存与恢复。进程运行时，程序计数器、程序状态寄存器、栈指针、通用寄存器以及其他控制寄存器保存在 CPU 中；进程失去处理器后，这些值需要写回 PCB，待该进程再次运行时再从 PCB 恢复。一次完整的上下文切换通常包括两个方面：一是切换地址空间，例如装载新的页表或页目录；二是切换内核栈与寄存器上下文。以“进程 A 下 CPU、进程 B 上 CPU”为例，其基本步骤是：保存 A 的寄存器与状态，更新 A 的 PCB 与队列归属，设置 B 为运行态，再从 B 的 PCB 中恢复寄存器和执行现场。XV6 中的 swtch 例程即通过保存旧上下文指针与装载新上下文指针来完成这一工作。

上下文切换既有直接开销，也有间接开销。直接开销是内核执行切换本身消耗的 CPU 时间，包括保存和恢复寄存器、切换地址空间等。间接开销更隐蔽，却在现代系统中常常更重要：缓存、缓冲区缓存以及 TLB 可能因地址空间或执行流改变而失效，导致随后的真实计算阶段出现更多缓存不命中与地址转换开销。因此，调度算法不能只追求频繁切换带来的表面公平，还必须平衡切换成本。

处理器调度算法的设计原则

不同类型的系统对调度的目标并不相同。交互式进程需要频繁与用户交互，大量时间耗费在等待输入上，因此更关心响应时间与均衡性，典型目标是将平均延迟控制在约 50 ms 到 150 ms 的量级。批处理进程通常在后台运行，不强调即时响应，更关注吞吐量、周转时间与 CPU 利用率。实时进程存在严格时间约束，其核心要求是短响应、可预测性以及满足截止期限，不能被较低优先级任务长期阻塞。调度设计因此不是单一目标优化，而是在不同工作负载与系统场景下进行折衷。

评价调度算法时，最常见的性能指标有吞吐量、周转时间与响应时间。吞吐量是单位时间内完成的进程数；周转时间是一个进程从提交到完成所经历的总时间；响应时间则是从提交到第一次得到系统回应所经历的时间。除此之外，还要考虑公平性、CPU 利用率等非性能指标，以及实时性、能耗等特定场景指标。它们之间往往并不一致：过分追求响应时间可能降低吞吐量，过分追求公平性可能损害短任务体验，过分追求低切换开销又可能退化成交互性很差的长时间独占。

调度算法的实现还依赖若干关键设计点。第一，PCB 中必须保存与调度有关的信息，例如优先级、运行时间统计、等待时间、所属队列和处理器亲和属性。第二，优先级既可以是静态的，也可以是动态的。静态优先级在创建时设定后不变；动态优先级则允许系统依据等待时间、I/O 行为或交互特征进行调整，以缓解饥饿并倾向某类任务。第三，调度既可以是抢占式的，也可以是非抢占式的。抢占式调度允许系统在更高优先级对象到达时强行剥夺当前执行者的 CPU；非抢占式调度则要求当前对象因自身结束、阻塞或主动让出才会失去处理器。第四，必须区分 I/O 密集型与 CPU 密集型行为。前者运行突发短、等待 I/O 多，后者需要长时间计算。现代系统往往对 I/O 密集型任务给予更高倾向，以改善交互体验并提升 I/O 设备利用率。第五，时间片长度的设置极其关键，需综合考虑切换开销、响应要求、就绪进程数目、CPU 性能和典型进程行为。

就绪队列的组织方式直接影响调度代价与策略表达能力。最简单的形式是单一就绪队列，所有可运行对象按到达顺序或按某种排序规则排队；更复杂的形式是按优先级维护多个就绪队列，调度时优先检查高优先级队列。多队列方式能够更自然地表达优先级调度、分级时间片和反馈机制，但也引入了队列间迁移、老化和优先级反转等问题。

典型的处理机调度算法

批处理系统中的调度算法

先来先服务

先来先服务调度（FCFS, First Come First Serve）按照进程进入就绪队列的先后顺序分配 CPU，属于典型的非抢占式算法。它的优点是实现简单、语义直接，并且从排队顺序上看具有表面公平性；缺点则是容易产生“护航效应”，即长进程占用 CPU 时，后续短进程被迫长时间等待，从而显著拉高平均周转时间，同时使 I/O 设备与 CPU 的协同利用变差。

设三个进程 P1​,P2​,P3​ 按顺序到达，运行时间分别为 24 s,3 s,3 s。若采用 FCFS，则执行顺序为 P1​→P2​→P3​，完成时刻分别为 24,27,30，于是三个进程的周转时间分别为 24,27,30，平均周转时间为

324+27+30​=27

若只改变顺序为 P2​→P3​→P1​，总工作量与吞吐量并未改变，但平均周转时间变为

33+6+30​=13

这说明在总服务量相同的情况下，调度顺序会显著影响平均周转时间，而 FCFS 对短作业尤其不利。

最短作业优先与最短剩余时间优先

最短作业优先（SJF, Shortest Job First）总是选择估计完成时间最短的进程优先运行，通常为非抢占式。其核心思想是优先完成短作业，从而降低平均周转时间。最短剩余时间优先（SRTN, Shortest Remaining Time Next）则是 SJF 的抢占式版本：当新到达进程的剩余执行时间小于当前运行进程的剩余时间时，立即发生抢占。

设有四个进程，参数如下：P1​:(0,7)，P2​:(2,4)，P3​:(4,1)，P4​:(5,4)，其中括号内分别表示到达时刻与运行时间。非抢占式 SJF 在时刻 0 只能选择 P1​，因此顺序为 P1​→P3​→P2​→P4​。抢占式 SRTN 则会根据剩余时间实时调整：时刻 0 运行 P1​；时刻 2，P2​ 到达，因其剩余时间 4 小于 P1​ 的剩余时间 5，故切换到 P2​；时刻 4，P3​ 到达，剩余时间最小，立即执行；时刻 5 后在 P2​、P4​ 与 P1​ 中继续选择剩余时间最小者，形成顺序 P1​→P2​→P3​→P2​→P4​→P1​。抢占式情形下，时刻 7 之前各时刻的候选剩余时间依次为：t=0 时 (P1​,7)；t=1 时 (P1​,6)；t=2 时 (P1​,5),(P2​,4)；t=3 时 (P1​,5),(P2​,3)；t=4 时 (P1​,5),(P2​,2),(P3​,1)；t=5 时 (P1​,5),(P2​,2),(P4​,4)；t=6 时 (P1​,5),(P2​,1),(P4​,4)；t=7 时 (P1​,5),(P4​,4)。

SJF 的理论优势在于：当所有进程都已知且可同时运行时，它能实现最小平均周转时间。但该算法存在两个根本问题。其一，它对长作业不公平，若短作业源源不断到达，长作业可能长期得不到 CPU，形成饥饿。其二，它需要预测未来执行时间，而现实中“作业长度”通常只能估计，不能精确获得。SRTN 虽能进一步改善响应时间，却也会带来更高的切换频率与实现复杂度。

最高响应比优先

最高响应比优先（HRRN, Highest Response Ratio Next）试图在 SJF 的低平均周转时间与 FCFS 的非饥饿性之间做折衷。它为每个进程定义响应比

R=作业处理时间作业周转时间​=作业处理时间作业处理时间+作业等待时间​=1+作业处理时间作业等待时间​

每次总是选择响应比最高的进程运行。

该公式的意义在于，短作业由于处理时间小，在相同等待时间下更容易获得较高响应比，因此 HRRN 仍保留了优先照顾短作业的倾向；而长作业随着等待时间增加，响应比会持续上升，最终可能超过后来的短作业，从而获得执行机会，避免长期饥饿。HRRN 因此是一种显式平衡“短作业优待”与“长作业救济”的算法。若将其做成抢占式，则需要频繁重新计算响应比并排序，系统开销会显著增加，因此经典 HRRN 多采用非抢占式实现。

交互式系统中的调度算法

时间片轮转

时间片轮转调度（RR, Round Robin）为每个可运行进程分配一个固定时间片，时钟中断到来或时间片耗尽时，将当前进程移到队尾，再让下一个进程运行。它是典型的抢占式算法，核心目标不是最小平均周转时间，而是让每个进程都能在可预期的时间内获得 CPU，从而改善响应时间，尤其适合交互式工作负载。

时间片的长短决定了 RR 的行为特征。若时间片过长，超过典型交互操作所需的处理时间，则算法会逐渐退化为 FCFS，短任务响应时间变差；若时间片过短，则上下文切换过于频繁，CPU 会把大量时间耗费在切换而非有效计算上。假设时间片为 10 ms，一次上下文切换耗时 0.1 ms，则切换开销约占 1。当时间片继续减小时，这一比例会快速升高。

设四个进程运行时间分别为 P1​=53，P2​=8，P3​=68，P4​=24，若采用时间片 q=20 的 RR，则调度序列为

P1​→P2​→P3​→P4​→P1​→P3​→P4​→P1​→P3​→P3​

由此可得等待时间分别为 72,20,85,88，平均等待时间为

472+20+85+88​=66.25

课件给出的比较表表明，在该例中，RR 的平均等待时间会随时间片变化而变化：q=1 时为 61.25，q=5 时为 57.25，q=8 时为 61.25，q=10 时为 66.25，q=20 时仍为 66.25；而最佳 FCFS 顺序的平均等待时间可低至 31.25，最坏 FCFS 则可高达 83.5。这说明 RR 的优势主要体现在响应公平与交互体验，而不一定体现在平均等待时间最优。

若两个进程 A,B 运行时间均为 100 ms，采用时间片 1 ms 且忽略切换开销，则 RR 下完成时刻分别约为 199 ms 和 200 ms，平均完成时间为 199.5 ms；而 FCFS 下完成时刻为 100 ms 与 200 ms，平均完成时间为 150 ms。因此，RR 并不天然优于 FCFS；当进程规模相近、交互性不强时，频繁轮转甚至会恶化完成时间。

虚拟轮转与 I/O 倾向

在同时存在 CPU 密集型与 I/O 密集型进程的系统中，单纯 RR 可能无法兼顾 CPU 与外设利用率。设进程 A 和 B 均需要 1000 ms 的纯计算，进程 C 的行为是循环执行“使用处理器 1 ms，再进行 I/O 10 ms”。若采用 FCFS 且 A 或 B 先运行，则 C 可能长达约 2000 ms 都无法发出新的磁盘请求，导致磁盘长期空闲。若采用时间片 100 ms 的 RR，序列大致为 C!A!−!−!−!−!−!−!−!B!−!−!−!−!−!−!−!C!A⋯，磁盘在多数时间仍然空闲，每 200 ms 只有约 10 ms 在工作。若将时间片缩短到 1 ms，则 C 能够在上一次 I/O 完成后几乎立刻再次运行并发出下一次请求，磁盘利用率可接近 90%，但代价是进程切换数量急剧增加。SRTN 也能在该场景中取得较好效果，因为 C 的下一次 CPU 突发最短，I/O 完成后会被优先调度。虚拟轮转法及相关改进思想，正是在这类场景下试图对 I/O 密集型进程给予更有效的倾斜。

最高优先级调度与优先级反转

最高优先级优先调度（HPF, Highest Priority First）总是选择当前优先级最高的进程执行。系统进程通常高于用户进程，前台进程通常高于后台进程，而 I/O 型进程也常被赋予更有利的地位。优先级既可以是静态的，也可以在运行过程中动态调整。该算法实现简单，能够强烈体现系统策略，但天然不公平，低优先级进程可能饥饿。

优先级调度在并发资源竞争中还会引出优先级反转问题。设高优先级进程为 H，低优先级进程为 L，中优先级 CPU 密集型进程为 M。若 L 先进入临界区并持有某共享资源，随后被抢占；这时 H 需要该资源却因加锁失败而阻塞；接着 M 获得 CPU 持续运行，则 L 无法继续执行并释放资源，H 也就被间接阻塞在低优先级进程之后。这种现象会造成高优先级任务停滞，严重时可导致系统错误。典型解决方法包括优先级继承、优先级天花板协议以及在极端场合使用中断禁止。

多级队列与多级反馈队列

多级队列调度将就绪进程按某种标准划分到多个队列中，不同队列之间具有不同优先级，也可以采用不同调度策略。其关键设计问题包括：按什么标准分队列，不同队列的排队方式是什么，队列间优先级如何确定，以及各队列内部是否使用同一种调度算法。多级队列适合将前台与后台、系统进程与普通进程、实时任务与普通任务等进行结构化区分。

多级反馈队列（MLFQ, Multilevel Feedback Queue）是在多级队列基础上引入“反馈”机制的综合算法。它由 Corbató 于 1962 年在 CTSS 中提出，后来被 BSD 等系统采用。其基本结构为：设置多个就绪队列，第一级优先级最高；优先级越高，时间片越短，优先级越低，时间片越长；系统优先从高优先级队列中选择进程，各级队列内部通常采用 RR。新创建进程先进入最高优先级队列；若一个进程用完整个时间片而仍未完成，则下降到下一级队列；若因 I/O 或事件阻塞而提前放弃 CPU，则唤醒后通常回到原先级别，从而保留其交互型特征。若系统允许抢占，则高优先级队列中新到达的进程可立即抢占低优先级队列中的当前执行者。MLFQ 的核心思想是：用行为反馈代替事先预测，让短 CPU 突发、频繁阻塞的交互式任务自动停留在高优先级，而把长时间占用 CPU 的计算型任务逐渐下沉。

其他比例分享思想

公平共享调度强调按用户或用户组而不是按单个进程分配处理器份额，适合多用户分时环境；保证调度强调每个可运行实体应得到接近其“应得份额”的处理器时间；彩票调度则通过“彩票张数”表达权重，以随机方式实现近似比例分享。三者都体现了从“谁先来、谁更短、谁优先级高”向“按份额分配”转变的思路。

各类算法的比较

从总体趋势看，FCFS 开销最低、无饥饿，但对短作业和 I/O 密集型进程不利；RR 具有良好公平性和响应性，但时间片过小时吞吐量会下降；SJF 与 SRTN 在平均周转时间和短作业响应方面表现优异，却可能饿死长作业；HRRN 通过等待时间修正实现更平衡的折衷；反馈类算法则试图把交互性、优先级与时间片机制综合在一起，对 I/O 密集型进程通常更有利。不同算法没有统一最优解，其优劣总是依赖工作负载、实现成本与系统目标。

机制与策略分离、线程调度

调度机制与调度策略分离是一项长期有效的操作系统设计原则。机制回答“如何做”，例如中断处理、上下文切换、就绪队列维护、计时与状态保存恢复；策略回答“按什么规则做”，例如 FCFS、RR、优先级调度或比例分享。将两者分离可以使系统在不更换底层切换机制的前提下更换或改进调度算法，也便于在不同负载与平台上复用同一套基础设施。

线程调度又带来新的层次。若内核只调度进程，而进程内部的用户级线程由运行时系统自行选择，则调度顺序表现为“内核先选进程，运行时再选线程”；这意味着不同进程内部线程之间不能直接在内核层面统一竞争 CPU。若线程是内核可见实体，则内核可以直接在全系统范围内选择线程，跨进程线程也能统一参与调度竞争。前者降低了内核管理成本，后者则在阻塞处理、并行执行和系统可控性方面更强。

Windows 线程调度算法设计思想

Windows 的调度单位是线程，而不是进程。其调度器采用基于动态优先级的抢占式调度，并结合时间配额进行控制：就绪线程按优先级进入不同就绪队列，系统总是选择最高优先级的就绪线程运行，同一优先级内部采用时间片轮转；在多处理机系统中，可以同时有多个线程并行运行。会触发线程调度的事件包括：线程优先级改变、处理机亲和集合改变、线程终止、新线程创建、等待线程重新就绪、运行线程阻塞，以及运行线程重新回到就绪态。

Windows 使用 32 个线程优先级，分为三个层次：优先级 0 的零页线程，用于清零空闲物理页；优先级 1 到 15 的可变优先级线程；优先级 16 到 31 的实时优先级线程。实时优先级线程通常不做动态改变，可变优先级线程则具有“基本优先级”和“当前优先级”之分，系统可在一定范围内动态提升或衰减当前优先级，以表达对交互或 I/O 行为的倾向。

Windows 的时间配额不是简单的毫秒数，而是以 quantum unit 表示的离散配额。若一个线程用完时间配额后，同优先级没有其他就绪线程，则系统会为它重新分配一个新配额并继续运行。时间配额的作用并不等价于优先级：提高优先级会更强烈地压制其他线程，而增加前台线程时间配额则更像在不完全剥夺后台线程的前提下，让前台任务获得更多 CPU 份额。

在实现上，Windows 维护就绪位图和空闲位图。就绪位图的每一位对应一个调度优先级，标识该优先级队列中是否存在可运行线程；空闲位图则标识哪些处理器当前空闲。它们使调度器能够迅速定位最高优先级的可运行线程以及可分配的空闲处理器。

Windows 的基本调度情形可归纳为三类。第一，主动切换：当前线程因等待某事件而从运行态转入阻塞态，调度器立即选择新的线程运行。第二，抢占：某个更高优先级线程变为就绪态时，当前线程被剥夺 CPU。被抢占线程会回到相应优先级就绪队列的队首；实时优先级线程被抢占时，时间配额会被重置为完整值，可变优先级线程则保留剩余时间配额。第三，时间配额用完：若当前线程优先级未降低且同优先级还有其他线程等待，则当前线程回到队尾，下一线程运行；若同优先级没有其他线程，就直接重置配额并继续执行；若时间配额用完后当前线程优先级发生下降，则调度器可能转而选择更高优先级线程。

为了体现对某些线程行为的倾向并缓解不公平，Windows 还会在若干情形下提升线程优先级。可变优先级范围内的线程在以下五种情况会被提升：I/O 操作完成；事件或信号量等待结束；前台进程中的线程完成等待；窗口活动唤醒窗口线程；以及线程因长期排队而出现饥饿。I/O 完成后的提升由设备驱动给出建议值，响应要求越高，提升幅度通常越大；为避免过度偏斜，唤醒时还会将该线程时间配额减 1。等待事件或信号量结束后，线程一般提升一级，并在提升后的优先级上先运行剩余配额，随后优先级逐级衰减直至回到基本优先级。对长期得不到运行机会的线程，平衡集管理器每秒扫描一次就绪队列，发现等待超过 300 个时钟中断间隔的线程时，会把其优先级提升到 15，并给予正常值四倍的时间配额；该线程用完整个配额后，优先级立即衰减回原始基本优先级。

当某个处理器上没有可运行线程时，Windows 会调度该处理器对应的空闲线程。多处理机系统中每个处理器都有自己的空闲线程；空闲线程优先级为 0，只在没有其他线程可运行时才执行。其主要功能是在循环中检测并处理待完成的系统工作，例如待处理的中断请求等。空闲线程的存在说明“无任务可运行”并不意味着处理器完全脱离调度体系，而是进入最低优先级的受控执行路径。

多处理器与多核环境下的调度

多处理器调度与单处理器调度的根本区别在于，系统不仅要决定“选哪个进程执行”，还要决定“放到哪个 CPU 上执行”。在对称多处理器系统（SMP）中，每个处理器通常都运行自己的调度程序，处理器之间可以共享负载，但调度器访问共享资源时需要同步。与此同时，进程在不同 CPU 间迁移会带来额外开销，尤其是缓存失效与 TLB 失效，因此调度器必须在负载均衡与局部性之间取得平衡。

SMP 系统中常见两种分配方式。静态进程分配将一个进程从开始到结束固定在某个处理器上运行，每个处理器拥有自己的就绪队列，调度开销低，但容易出现有的处理器很忙、有的处理器空闲的现象。动态进程分配则允许进程在执行过程中被分配给任意空闲处理器，所有处理器共享公共就绪队列或通过迁移机制共享负载，负载更加均衡，但调度代价更高。

在多核处理器中，缓存层次进一步改变了调度问题。核心与主存之间通过多级缓存联系，系统需要同时考虑缓存一致性、缓存亲和性、核间数据共享与负载均衡。缓存一致性问题表现为多个处理器分别缓存同一共享变量时，局部修改可能彼此不可见。例如两个处理器都把变量 X 的旧值 0 缓存在本地，各自执行 X=X+1 后，本地缓存都变为 1，若没有一致性协议协调，最终全局结果可能仍然是 1 而非 2。缓存亲和性则要求进程尽可能继续在此前运行过的处理器核心上执行，从而复用该核心缓存中的热数据，减少缓存不命中。

缓存亲和性与负载均衡之间常常存在张力。若任务总留在原核心上运行，可充分利用缓存，但可能导致各核心负载失衡；若频繁迁移任务以实现全局均衡，则会损失缓存局部性。课件给出的单队列示例中，若多个 CPU 都从公共队列中简单轮转选取任务，进程会在不同 CPU 间频繁转移，违背缓存亲和性；若强行让进程尽可能回到原 CPU，又可能牺牲整体均衡。作业偷取（work stealing）正是为解决这一矛盾提出的一种思路：每个处理器优先处理本地队列任务，仅在本地队列为空时，再去其他处理器队列中“偷取”任务，以尽量兼顾局部性与均衡。

openEuler 的多队列示例进一步展示了多核调度的工程实现。系统为每个 CPU 维护本地队列，初始时可获得较好的局部性；但当部分 CPU 上的任务先结束后，就可能出现某些队列为空、某些队列仍有积压的负载失衡现象。为此，系统为每个处理器设置迁移线程 migration/CPUID，并为其准备停机工作队列。若需要把某一就绪进程从 CPU1 迁移到 CPU0，则 CPU0 会向 CPU1 的停机工作队列投递迁移函数并唤醒 CPU1 上的迁移线程；该线程不会被普通进程抢占，因此能够及时执行迁移操作，从而恢复均衡。

Linux 进程调度算法

Linux 将进程大体分为实时进程与普通进程两类。实时进程对调度延迟要求最高，通常采用 FIFO 或 Round Robin 策略。普通进程中又可区分交互式进程与批处理进程：交互式进程间歇性睡眠、对响应速度敏感；批处理进程则长期在后台运行、能容忍较大响应延迟。课件中普通进程的核心调度策略以 CFS 为主线展开，同时给出了 Linux 调度器从早期简单优先级调度、O(1) 调度器、SD/RSDL 补丁、CFS，再到 Linux 6.11 中 EEVDF 的演化脉络。

CFS 的基本思想

CFS（Completely Fair Scheduler）即完全公平调度器。它吸收了“完全公平”的思想，不再显式跟踪睡眠时间，也不再试图先验地区分交互式进程，而是把问题转换为“按权重比例分配运行时间”。设调度周期为 Tsched​，进程 i 的权重为 wi​，所有可运行进程的总权重为 W=∑j​wj​，则分配给进程 i 的实际运行时间为

$$

t_i

T_{\text{sched}}
\cdot
\frac{w_i}{W}

其中，调度周期表示把所有处于 `TASK_RUNNING` 状态的进程大致都调度一遍所对应的时间尺度。 为了把“不公平”统一映射为一个可比较的量，CFS 又引入了虚拟运行时间 `vruntime`。若进程 $i$ 的实际运行时间为 $t_i$，其权重为 $w_i$，并以 `NICE_0_LOAD` 作为基准权重，则

v_i

t_i
\cdot
\frac{\text{NICE_0_LOAD}}{w_i}

把前一式代入可得

v_i

T_{\text{sched}}
\cdot
\frac{\text{NICE_0_LOAD}}{W}

这一结果表明，在理想公平状态下，所有进程在一个调度周期内增长的 `vruntime` 应当相同。因而，CFS 每次只需选择 `vruntime` 最小的进程运行，就能使所有进程的“虚拟进度”在宏观上同步推进。这里的“相同”指的是虚拟运行时间的推进量相同，而不是实际获得的 CPU 时间相同；权重高的进程会在相同的 `vruntime` 增量下获得更多实际 CPU 时间。 ## CFS 的数据结构 Linux 中所有任务都由 `task_struct` 表示。与调度有关的重要成员包括优先级、调度类以及调度实体 `sched_entity`。`sched_entity` 进一步包含权重信息、红黑树结点等字段，用于把任务插入 CFS 的运行队列。每个运行队列对应一个 `cfs_rq` 结构，其中保存红黑树 `tasks_timeline` 等信息。`sched_class` 则像一条模块链，定义不同调度类应实现的一组接口函数，供内核调度器统一调用。CFS 的核心工程结构因此可以概括为：`task_struct` 负责描述任务，`sched_entity` 负责抽象公平调度对象，`cfs_rq` 负责维护按 `vruntime` 排序的运行实体集合。 CFS 使用红黑树组织所有就绪实体，并以 `vruntime` 作为排序键。调度器总是选择红黑树最左侧结点，即 `vruntime` 最小的实体运行；该实体运行时会暂时从树中移出，待它再次回到就绪态后再插回树中。由于刚运行过的实体其 `vruntime` 通常更大，所以回到树中后往往更靠右；而随着其他实体陆续被调度，整棵树又不断向左推进，从而逼近公平。CFS 还维护 `min_vruntime`，它是每个 `cfs_rq` 上的一个关键变量，通常不大于该队列所有就绪实体的最小 `vruntime`，表示当前队列整体虚拟运行时间的下界。 ## CFS 的运行过程 在时钟中断情形下，CFS 会先更新当前运行进程的统计信息，把这一小段实际执行时间折算为 `vruntime` 增量并累加到原值上；随后判断是否满足抢占条件。若当前进程在这一调度周期中已经消耗了超过其应得份额的时间，则设置重调度标记，在适当时机触发调度；否则继续运行当前进程。 当系统进入一次主动调度时，CFS 先更新当前进程运行信息；若该进程仍处于可运行状态，则重新插入红黑树；然后选择新的运行实体，并将其从红黑树中移出。这里并不是简单机械地选择最左结点。`cfs_rq` 中还维护 `last` 和 `next` 两个指针：`last` 指向最近被调度下去的实体，`next` 指向即将被调度上 CPU 的实体。在某些唤醒或抢占场景下，调度器会优先考虑 `next`，其次考虑 `last`，以实现更细致的调度策略。相关函数 `wakeup_preempt_entity` 的返回值还区分了三种状态：若返回 $-1$，表示新实体 `vruntime` 更大，不能抢占；若返回 $0$，表示虽然更小，但还没小到超过调度粒度，通常也不抢占；若返回 $1$，则表示新实体明显更“落后”，可以抢占当前进程。 在进程创建情形下，新进程的 `vruntime` 初值并不是简单取 $0$。CFS 以当前运行队列的 `min_vruntime` 为基准，再加上“该进程在一个调度周期内应增长的 `vruntime`”，相当于假定新进程已经在本轮公平分配中占有过一次机会，从而避免新进程刚出生就因 `vruntime` 过小而过度抢占。课件还给出了一种特殊处理：若系统设置为“子进程优先运行”，且父子位于同一 CPU、父进程非空、并且父进程 `vruntime` 小于子进程，则允许交换父子 `vruntime`，从而优先让子进程执行。新进程确定好初值后会被插入红黑树，再依据是否满足抢占条件决定是否立即触发调度。 在进程唤醒情形下，CFS 也不会简单保留原 `vruntime` 不变直接入队。系统会对唤醒进程做一定补偿：以 `min_vruntime` 减去一个量作为参考，使睡眠较久的进程在唤醒后能够较快获得 CPU。但若某进程只进行了很短暂的睡眠，则不能因为“睡一会儿”就获得额外运行优势，因此最终会在“补偿后的参考值”和“该进程原有 `vruntime`”之间取较大者，再插入红黑树并判断是否触发抢占。该机制说明 CFS 虽不再显式追踪睡眠时间，却仍会在唤醒路径上通过 `vruntime` 校正兼顾交互体验。 从状态转换角度看，CFS 的工作流可以概括为：新建进程时计算 `vruntime` 并插入红黑树；运行态进程在时钟中断下更新 `vruntime` 与时间片信息，并检测是否需要抢占；调用 `schedule` 时从红黑树中选出最左结点投入运行；阻塞进程被唤醒后重新修正 `vruntime` 并插回红黑树。进程状态转换与 `vruntime` 更新因此紧密耦合，CFS 的公平性并不是在静态队列上一次性决定的，而是在每一次中断、唤醒、创建与重调度中持续维护的。
