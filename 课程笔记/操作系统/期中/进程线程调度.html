进程线程调度

进程线程调度

Apr 25, 202688 min read

核心速览

这一讲真正要理解的，不是“背几个调度算法名字”，而是理解：操作系统如何在一堆想运行的进程/线程之间分配 CPU。

CPU 在某一瞬间只能执行有限数量的控制流。哪怕是多核系统，核心数量也远小于进程/线程数量。因此，操作系统必须像一个“调度室”一样不断做决策：

什么时候要换人？换谁上 CPU？怎么把当前进程安全地换下去、把另一个进程恢复起来？

所以，进程/线程调度可以围绕三个问题来学：

这三个问题刚好对应课件里的主线：调度时机、调度算法、调度过程。

一、为什么操作系统需要调度？

先从最朴素的问题开始：

如果只有一个 CPU，但我们同时打开了浏览器、编辑器、终端、音乐播放器，为什么它们看起来都在运行？

答案不是 CPU 真的同时执行了所有程序，而是操作系统在很短的时间尺度上不断切换执行对象。每个进程或线程轮流获得 CPU，一段时间后让出 CPU，另一个进程再接上。只要切换足够快，用户就会感觉它们“同时”运行。

所以，处理器调度的本质是：

控制和协调进程/线程对 CPU 的竞争。

更具体地说，调度器会按照某种调度算法，从就绪队列中选择一个进程或线程，把 CPU 的使用权交给它。如果没有任何可运行的进程，系统不会让 CPU 完全无所事事，而是会安排一个 idle 进程或 idle 线程运行。它的作用不是做业务，而是在“没人可调度”的时候维持系统的基本循环。

直觉解释：调度器像食堂叫号系统

食堂窗口只有几个，但排队的人很多。叫号系统要决定：谁先来谁先吃？老人小孩是否优先？每个人占窗口多久？如果有人只是取个餐马上走，要不要优先处理？

CPU 调度也是类似的事情。只不过“窗口”变成了 CPU，“顾客”变成了进程/线程，“叫号规则”变成了调度算法。

所以，处理器调度不是一个单纯的技术细节，而是操作系统实现并发、多任务和资源公平分配的基础。

二、调度的三个层次

“调度”这个词其实有不同层次。不同层次关心的资源不一样，时间尺度也不一样。

1. 长程调度：决定谁能进入系统

长程调度也叫作业调度或宏观调度。

它发生在新进程创建或新作业进入系统时，负责决定：

这个作业要不要进入当前活跃进程集合？

可以把它理解为“进校门”的决策。系统资源有限，不可能无限制地把所有作业都塞进内存和就绪队列。长程调度关心的是系统整体负载，决定多道程序的程度。

2. 中程调度：决定谁暂时换出内存

中程调度和内存压力有关。

如果内存不够，操作系统可能把某些进程的部分或全部内容换出到外存；等以后需要运行时，再把它换入内存。这就是交换技术。

所以，中程调度从存储资源管理的角度看问题：

哪些进程暂时不适合留在内存？哪些进程应该被换回来？

它对应的是 Ready Suspended、Blocked Suspended 这类挂起状态。

3. 短程调度：决定谁立刻上 CPU

短程调度也叫微观调度，是本章最核心的内容。

它从处理器资源分配的角度看问题：

就绪队列里有一堆进程/线程，下一刻到底让谁运行？

短程调度的时间尺度通常是毫秒级的，而且使用非常频繁，所以它必须足够高效。一个调度算法如果理论上很漂亮，但每次选择都要耗费太多 CPU 时间，那它自己就会变成系统负担。

我们平时说的 CPU 调度，通常主要指短程调度。

三、调度到底什么时候发生？

调度不是凭空发生的。它往往发生在某个事件改变了进程状态之后，即

内核对中断/异常/系统调用处 理后返回到用户态前最后时刻

比如：

  • 一个进程执行完毕并退出；
  • 一个进程因为错误或异常终止；
  • 一个新进程被创建；
  • 当前运行进程要等待 I/O，于是进入阻塞态；
  • 某个阻塞进程因为 I/O 完成而被唤醒，重新回到就绪态；
  • 当前进程的时间片用完，被放回就绪队列；
  • 当前进程主动执行 yield 放弃 CPU。

这些事件发生后，系统的就绪队列可能发生变化：有人离开了，有人回来了，有人优先级变了，有人时间片用完了。于是内核就需要重新判断：

现在继续运行原来的进程，还是换另一个进程上 CPU？

这个逻辑和中断/异常处理的关系非常紧密：

    A["事件发生<br/>I/O 中断 / 时钟中断 / 系统调用 / 异常"] --> B["硬件机制响应<br/>进入内核"]

    C --> D["进程状态可能发生变化<br/>阻塞 / 唤醒 / 退出 / 就绪"]

    H --> I["上下文切换"]
    I --> J["新进程 / 新线程开始运行"]

直觉解释：调度常常发生在“回用户态之前”

中断、异常、系统调用把 CPU 带进内核。内核处理完事件后，不一定马上回到原来的用户程序。

它会顺手看一眼：有没有更该运行的进程？当前进程的时间片是不是用完了？有没有高优先级任务刚被唤醒？

如果答案是“该换人了”，调度就发生在这个返回用户态之前的最后时刻。

所以，调度不是孤立机制。它和异常控制流、中断、系统调用共享同一条系统主线：事件进入内核，内核处理事件，处理完后决定返回原进程还是切换到另一个进程。

四、调度过程：从选择进程到上下文切换

调度算法只解决“选谁”的问题。但选出来以后，操作系统还必须真正把 CPU 从一个进程交给另一个进程。

这一步叫：

上下文切换（Context Switch）。

1. 什么是上下文？

进程运行时，它的状态不仅在内存里，也在 CPU 里。

比如：

  • 程序计数器 PC：下一条指令在哪里；
  • 程序状态字 PSW：当前 CPU 状态、标志位、中断状态等；
  • 栈指针 SP：当前栈在哪里；
  • 通用寄存器：当前计算中间结果、函数参数、返回值；
  • 控制寄存器：例如页表相关寄存器，影响地址空间。

当进程正在运行时，这些硬件状态保存在 CPU 寄存器中。

当进程不运行时，这些状态必须保存到进程控制块 PCB 或相关内核数据结构中。否则下次恢复这个进程时，它就不知道自己运行到哪里了。

所以，上下文切换就是：

把 CPU 当前硬件状态从进程 A 的上下文，换成进程 B 的上下文。

这套“收拾和恢复现场”的过程，就是上下文切换。

3. 上下文切换为什么有开销？

上下文切换不是免费操作。

它至少有两类开销。

第一类是直接开销。

内核要执行保存和恢复寄存器的代码，还可能切换地址空间，比如切换页表根地址或全局页目录。这些动作本身会消耗 CPU 时间。

第二类是间接开销。

新进程上 CPU 后，CPU Cache、Buffer Cache、TLB 中原来缓存的大量内容可能不适合新进程使用。也就是说，虽然 CPU 已经切到新进程了，但高速缓存还没有“热起来”。接下来一段时间里，Cache miss 和 TLB miss 可能增加，系统性能会下降。

所以，调度不能太频繁。调度太少，交互响应差；调度太多，上下文切换成本又会把 CPU 时间吃掉。

这也解释了为什么时间片大小很重要。

五、调度算法要追求什么？

调度算法不是越“快”越好，也不是越“公平”越好。不同系统的目标不一样。

1. 批处理进程：更关心吞吐量和周转时间

批处理进程通常在后台运行，不直接和用户交互。

例如：

  • 营业额计算；
  • 利息计算；
  • 批量报表生成；
  • 大规模数据处理。

这种系统更关心：

所以，批处理系统常看重吞吐量、周转时间、CPU 利用率。

2. 交互式进程：更关心响应时间

交互式进程经常等待用户输入。

例如：

  • shell；
  • 文本编辑器；
  • 图形界面程序；
  • 浏览器前台页面。

这类进程不一定一直占用 CPU，但用户一旦敲键盘、点鼠标，就希望它很快回应。课件里提到，交互式程序通常希望响应延迟足够低，大致在几十到一百多毫秒的范围内，用户才会觉得系统“跟手”。

所以，交互式系统最关心响应时间和均衡性。

3. 实时进程：更关心最后期限和可预测性

实时进程不是单纯要求“越快越好”，而是要求：

必须在规定时间内完成。

例如：

  • 音视频处理；
  • 工业控制；
  • 机械控制；
  • 汽车装配线中的控制任务。

实时系统最怕的不是平均速度慢一点，而是关键任务偶尔严重迟到。它追求的是确定性、可预测性和最后期限保障。

4. 常见评价指标

调度算法常见指标有：

  • 性能指标
    • 吞吐量 Throughput：
      • 单位时间完成的进程数量。
    • 周转时间 Turnaround Time：
      • 进程从提出请求到运行完成的总时间。
      • T_turnaround = T_completion - T_arrival
    • 响应时间 Response Time：
      • 进程从提出请求到第一次得到回应的时间。
  • 非性能指标
    • CPU 利用率 CPU Utilization：
      • CPU 真正在做有效工作的时间比例。
    • 公平性 Fairness：
      • 避免某些进程长期得不到 CPU。
  • 特定场景考虑的指标
    • 实时性：
      • 任务是否能在最后期限前完成。
    • 能耗：
      • 移动设备、服务器集群中尤其重要。

小结

调度算法的本质是折中。

批处理系统想提高吞吐量和降低周转时间；交互式系统想降低响应时间；实时系统想保证最后期限；多核系统还要考虑缓存亲和性和负载均衡。

所以不存在一个在所有场景中都完美的调度算法。

六、设计调度算法时要看哪些支点？

在进入具体算法前，先抓住几个设计支点。后面所有算法，其实都是在这些支点上做不同取舍。

1. PCB 里要记录调度信息

调度器要做决策，必须知道进程的状态。

因此，PCB 中通常要记录和调度有关的信息，例如：

  • 当前进程状态；
  • 优先级或优先数；
  • 已经运行的时间；
  • 剩余估计运行时间；
  • 等待时间；
  • 所属队列；
  • 时间片或时间配额；
  • 亲和性信息；
  • 调度策略相关字段。

调度算法越复杂，PCB 或调度实体中需要维护的信息也越多。

2. 优先级：静态还是动态？

优先级可以是静态的，也可以是动态的。

静态优先级在进程创建时确定，运行过程中不再改变。它简单、稳定，但可能不够灵活。

动态优先级会在运行过程中变化。例如，一个进程等待太久，可以提高优先级，避免饥饿；一个 CPU 密集型进程长期占用 CPU，可以降低优先级，让交互式进程更快响应。

所以，动态优先级体现的是一种策略：

根据进程的历史行为和当前系统状态，动态调整它获得 CPU 的机会。

3. 就绪队列如何组织？

最简单的方式是一个就绪队列，所有就绪进程排在一起。

更复杂的方式是按优先级维护多个队列：

调度时先看高优先级队列，空了再看低一级队列。同一级内部可以再使用 FCFS、RR 等策略。

多级队列和多级反馈队列就是在这个基础上发展出来的。

4. 抢占式还是非抢占式？

非抢占式调度的规则是：

一个进程一旦被调度运行，除非它自己结束、阻塞或主动让出 CPU，否则就一直运行。

这种方式实现简单，上下文切换少，但响应时间可能很差。一个长任务上 CPU 后，短任务只能干等。

抢占式调度的规则是：

如果出现更应该运行的进程，系统可以强行剥夺当前进程的 CPU。

例如时间片用完、高优先级进程被唤醒，都可能触发抢占。

抢占式调度更适合交互式系统和实时系统，但它也会带来更多上下文切换和同步复杂性。

5. I/O 密集型还是 CPU 密集型？

I/O 密集型进程经常发起 I/O，然后阻塞等待。

CPU 密集型进程则需要长时间连续计算。

这两类进程的行为差异很大：

现代操作系统通常会更偏爱 I/O 密集型进程。原因很简单：它们往往只需要一点 CPU 时间，就能发起 I/O，然后马上阻塞。早点让它们运行，可以让磁盘、网卡等设备尽快忙起来，从而提高整个系统的资源利用率。

6. 时间片应该多长？

时间片，也叫 quantum，是调度器分配给进程的一段连续 CPU 使用时间。

时间片太长，会接近先来先服务。短任务和交互式任务可能要等很久，响应时间变差。

时间片太短，会频繁上下文切换。看起来大家都很公平，但大量 CPU 时间浪费在保存现场、恢复现场、Cache/TLB 重新预热上。

所以，时间片不是越短越公平，也不是越长越高效。它要在响应时间和切换开销之间折中。

七、批处理系统中的经典调度算法

批处理系统主要关心吞吐量、周转时间和 CPU 利用率。

这一类算法往往先假设：任务提交后在后台运行，用户不需要立刻看到响应。

1. FCFS：先来先服务

FCFS 的规则非常朴素：

谁先进入就绪队列，谁先使用 CPU。

它是非抢占式调度。一个进程被选中后，会一直运行，直到结束或阻塞。

优点是公平、简单、开销小。

缺点也很明显：如果一个长进程排在前面，后面一堆短进程都要等它。这种现象常叫“护航效应”。

所以，FCFS 的问题不是“不公平”，而是它太机械地公平，完全不考虑任务长短和交互体验。

2. SJF：短作业优先

SJF 的思路是：

先完成短作业，降低平均周转时间。

它每次选择预计完成时间最短的进程运行。非抢占式 SJF 一旦选中一个短作业，就让它运行到结束或阻塞。

SJF 的优点非常突出：当所有进程同时就绪，并且我们知道它们的运行时间时，SJF 可以得到最短的平均周转时间。

但它有两个大问题。

第一，它不公平。源源不断的短任务可能让长任务一直等下去，造成饥饿。

第二，它需要预测未来。操作系统通常不知道一个进程接下来到底要运行多久，只能根据历史行为估计。

3. SRTN：最短剩余时间优先

SRTN 是 SJF 的抢占式版本。

它不只看“谁最短”，而是看：

谁剩余运行时间最短。

如果一个新进程到达，并且它的剩余运行时间比当前进程更短，当前进程就会被抢占。

SRTN 能进一步改善短任务的响应与周转，但问题也更明显：它需要频繁比较剩余时间，也仍然可能让长任务饥饿。

4. HRRN：最高响应比优先

HRRN 是一个非常漂亮的折中算法。

它给每个作业计算响应比：

R=作业处理时间作业周转时间​=处理时间等待时间+处理时间​=1+处理时间等待时间​

调度时选择响应比最高的进程。

这个公式的直觉很妙：

对于短作业，处理时间小，所以同样等待时间下，响应比更高。它继承了 SJF 对短作业友好的特点。

对于长作业，刚开始响应比可能不高。但只要它等待足够久，等待时间不断增加，响应比也会逐渐升高。这样长作业最终也能获得 CPU，避免无限饥饿。

直觉解释：HRRN 像排队时的“委屈值”

短任务本来就很快，插一下队能让整体效率变高。

但长任务如果等得太久，它的“委屈值”会越来越高。等到一定程度，系统就必须照顾它。

所以 HRRN 不是单纯偏爱短任务，而是在“短任务效率”和“长任务公平”之间折中。

Tip

  • HRRN 通常是非抢占式调度算法。
  • 它每次调度时计算响应比，选择响应比最高的进程运行。 理论上可以做成抢占式版本，但需要频繁重新计算响应比和排序，开销较大，因此经典 HRRN 一般按非抢占式理解。

八、交互式系统中的经典调度算法

交互式系统最关心响应时间和公平性。

用户不在乎后台任务平均周转时间少了几秒，但会非常在乎敲下一个键后，编辑器半天不响应。

1. RR：时间片轮转

RR 的规则是：

每个进程分配一个时间片，用完就回到队尾排队。

例如就绪队列里有 A、B、C：

RR 的核心依赖时钟中断。时间片用完时，时钟中断进入内核，内核发现当前进程该让出 CPU，于是把它放回队尾，调度下一个进程。

RR 的优点是公平，适合交互式系统，短任务能较快得到第一次运行机会。

但它也有问题。

  • 时间片太长，会退化成 FCFS。
  • 时间片太短，上下文切换次数暴增。
    • 课件里的例子很直观：如果进程切换花 0.1ms，时间片是 10ms，那么切换开销大约占 1%；但如果时间片缩到 1ms，开销比例就会明显变大。

另外，RR 对 I/O 密集型进程也不一定天然公平。I/O 密集型进程常常用不完时间片就阻塞，等 I/O 完成回来后又要排队。如果不做补偿，它反而吃亏。

2. Virtual RR：虚拟轮转法

虚拟轮转法就是为了解决 RR 对 I/O 密集型进程不够友好的问题。

普通 RR 中，一个进程因为等待 I/O 主动让出 CPU，回来后通常要重新排队。

虚拟轮转法会给从 I/O 唤醒的进程更优先的机会。它可以设置辅助队列，让这些进程醒来后较快获得 CPU，并且只使用上次没用完的时间片。

这样做的目的不是偏心，而是为了提高整体资源利用率：

I/O 密集型进程只需要一点 CPU 时间，就能发起下一次 I/O，让磁盘或网卡继续忙起来。

如果它一直排队，I/O 设备可能长期空闲，系统整体效率反而下降。

3. 优先级调度

优先级调度的规则是：

选择优先级最高的进程运行。

一般来说：

  • 系统进程优先级高于用户进程；
  • 前台进程优先级高于后台进程；
  • I/O 型进程常常会得到一定偏好；
  • 优先级可以是静态的，也可以动态调整。

优先级调度的优点是表达能力强。操作系统可以明确说：“这个任务更重要，应该先运行。”

但它也有一个危险：低优先级进程可能长期得不到 CPU，出现饥饿。

4. 优先级反转

优先级调度还有一个经典坑：优先级反转。

假设有三个进程：

现在 L 先运行，并拿到一个锁。随后 H 运行，也需要这把锁，但锁在 L 手里，所以 H 阻塞了。

这时 M 就绪。因为 M 的优先级高于 L，M 会抢占 L 运行。

问题来了：H 在等 L 释放锁，但 L 又被 M 压着不能运行。于是最高优先级的 H 反而被中优先级的 M 间接拖住了。

这就是优先级反转：

高优先级任务被低优先级任务持有的资源阻塞，而低优先级任务又被中优先级任务抢占，导致高优先级任务迟迟无法继续执行。

解决办法主要有：

  • 优先级继承：
    • 低优先级进程持有高优先级进程需要的锁时，临时继承高优先级，让它赶紧运行并释放锁。
  • 优先级天花板：
    • 为资源设置优先级上限，进程获取资源时直接提升到某个上限，减少反转风险。
  • 禁止中断或关抢占：
    • 在极短临界区中使用，但不能滥用，否则会影响系统响应性。

直觉解释：优先级继承像“临时通行证”

L 本来级别低，但它手里拿着 H 必须要的钥匙。

系统临时给 L 一张高级通行证，让它赶紧跑完，把钥匙还给 H。等锁释放后，L 再恢复原来的低优先级。

5. 多级队列调度

多级队列调度把进程分到不同队列中，每个队列可以有不同优先级和调度策略。

例如：

  • 前台交互队列：优先级高，使用 RR
  • 后台批处理队列：优先级低，使用 FCFS
  • 系统任务队列：优先级最高

它要回答几个问题：

  • 根据什么分队列？
  • 不同队列优先级怎么定？
  • 队列之间如何调度？
  • 每个队列内部用什么算法？
  • 进程能不能在队列之间移动？

如果进程不能移动，这叫多级队列。它简单，但不够灵活。

如果进程可以根据行为在队列之间移动，就进入了多级反馈队列。

6. 多级反馈队列 MLFQ

多级反馈队列是一个很重要的综合调度算法。

它的核心思想是：

系统不提前预测进程类型，而是通过观察进程行为来分类。

基本规则可以这样理解：

如果你每次拿到 CPU 都用完整个时间片，还不主动阻塞，那系统就判断你像 CPU 密集型任务，于是逐步把你放到低优先级长时间片队列。

如果你经常很快去等 I/O，说明你只需要一点 CPU 来维持交互或发起 I/O，系统就愿意让你保持较高响应速度。

MLFQ 的强大之处在于，它把 FCFS、RR、优先级、动态调整、I/O 偏好都揉在了一起，是典型的折中算法。

九、几种补充调度思想

课件还提到几种交互式系统中的其他调度思想，这里先做概念性了解。

1. 公平共享调度

公平共享调度关心的不是单个进程公平，而是用户、组或项目之间的公平。

比如用户 A 开了 100 个进程，用户 B 只开了 1 个进程。如果按进程平分 CPU，A 会占走绝大多数资源。公平共享调度会把资源先按用户或组分配，再在组内分给具体进程。

它解决的是：

不能让“开更多进程的人”天然获得更多 CPU。

2. 保证调度

保证调度试图给进程一个明确承诺：

如果系统里有 n 个进程，每个进程理论上应该获得大约 1/n 的 CPU 时间。

调度器会记录每个进程实际获得了多少 CPU。如果某个进程低于应得份额，就优先调度它。

它的核心思想是补偿偏差。

3. 彩票调度

彩票调度把 CPU 时间看作抽奖。

每个进程持有若干张彩票。每次调度时随机抽一张，中奖进程获得 CPU。彩票越多，被选中的概率越大。

它的优点是实现简单，能表达比例份额；缺点是短期内有随机波动，不像严格优先级那样确定。

十、多核与多处理器调度

前面讨论单核调度时，我们主要问的是：

就绪队列里有很多进程，下一刻谁来跑？

但进入多核或多处理器系统以后，问题会复杂一层。

因为现在系统里不止一个 CPU 核心，调度器不仅要决定：

谁来跑？

还要决定：

在哪个 CPU 核上跑？

这看起来只是多了一个选择，但实际上会牵出一整套新问题：进程迁移会不会破坏缓存？多个核心同时访问同一份数据会不会出错？各个核心会不会忙闲不均？任务之间的数据共享会不会让跨核调度变得很贵？

所以，多核调度不能只理解成“多个单核调度器并排工作”。它真正的难点在于：

CPU 核心变多以后，调度器不仅要分配计算资源，还要考虑缓存、内存、数据共享和负载均衡之间的关系。

1. 对称多处理器中的进程分配

在对称多处理器系统，也就是 SMP 中，多个处理器在地位上是对等的。理论上，一个进程可以被安排到不同处理器上运行。

这时，进程分配方式大致可以分成两类。

第一类是静态进程分配。

静态分配的意思是：一个进程从开始到结束都被分配到固定的处理器上执行。每个处理器有自己的就绪队列，调度器只需要在本处理器的队列里选择任务。

它的好处是调度开销小，也更容易保持缓存亲和性。因为进程一直在同一个处理器上运行，这个核心的 Cache 里可能已经有它常用的数据。

但缺点也很明显：各个处理器可能忙闲不均。比如 CPU 0 的队列里有一堆任务，CPU 1 却没什么事情做。如果进程不能迁移，系统整体性能就会被拖累。

第二类是动态进程分配。

动态分配的意思是：进程在执行过程中可以被分配到任意空闲处理器上执行。所有处理器可以共享一个公共就绪队列，哪个处理器空了，就从公共队列里取任务。

它的好处是负载更均衡。只要还有可运行任务，就尽量不让处理器闲着。

但它的缺点是调度开销更大。多个处理器访问共享队列需要同步，可能发生锁竞争；而且进程被迁移到另一个核心后，原来核心上的缓存内容就用不上了，可能带来额外 Cache miss。PPT 在“SMP 进程分配”部分正是用静态分配和动态分配对比了这组取舍：静态分配调度开销小但可能忙闲不均，动态分配负载均衡但调度开销大。

2. 多核处理器为什么会带来新问题？

单核系统里，CPU 只有一个执行核心。虽然也有 Cache 和内存层次，但不存在多个核心同时缓存和修改同一份数据的问题。

多核系统就不一样了。

以 PPT 里的 ARMv8 架构示意为例，不同核心可能各自有 L1 Cache，多个核心组成 Cluster，共享更高层的 L2 / L3 Cache，再通过总线访问内存。也就是说，数据可能同时存在于多个层级、多个核心的缓存里。

这四个问题不是彼此独立的。它们之间经常互相拉扯。

比如，为了缓存亲和性，我们希望进程尽量留在原来的核心；但为了负载均衡，我们又可能需要把任务迁移到空闲核心。为了共享数据，不同核心可能需要访问同一份内存；但这又会引发缓存一致性和同步开销。

所以，多核调度的本质不是简单地“多开几个 CPU”，而是在这些目标之间做折中。

这就是缓存一致性问题：

同一份共享数据在多个核心的缓存中存在副本时，系统必须保证这些副本不会互相矛盾。

PPT 中给出的解决方向是 MESI 协议。

Note

MESI 协议是一种经典的缓存一致性协议，它的核心思想是给每个缓存行打上状态标记，用这些标记协调多个 CPU 核心对同一份数据的读写关系，避免各个核心长期拿着彼此矛盾的缓存副本。

  • M（Modified，已修改） 表示该缓存行只在当前核心中有效，并且已经被修改，内存中的值不是最新的；
  • E（Exclusive，独占） 表示该缓存行只被当前核心缓存，但还没有修改，缓存值和内存值一致，之后若要写入可以直接转为 M；
  • S（Shared，共享） 表示该缓存行可能被多个核心同时缓存，大家都可以读，但如果某个核心要写，就必须先让其他核心的副本失效；
  • I（Invalid，无效） 表示该缓存行已经失效，不能再被当前核心使用，若之后还要访问，必须重新从内存或其他核心处获取最新数据。

简单说，MESI 的原则就是：共享时可以一起读，写入时必须独占，旧副本必须失效。

这里要注意：缓存一致性协议本身主要是硬件层面的机制，不是调度器直接实现的算法。

但调度器必须理解它的代价。因为如果一个进程或线程频繁在多个核心之间迁移，或者多个核心频繁读写同一份共享数据，就会导致大量缓存行失效、同步、写回、总线通信，这些都会影响性能。

所以，多核调度不能只问“哪个核心空着”，还要问：

把这个任务迁过去，会不会带来大量缓存失效和一致性维护开销？

4. 核间数据共享

核间数据共享和缓存一致性关系很近，但关注点略有不同。

缓存一致性问的是：

多个核心缓存同一份数据时，如何保证看到的值一致？

核间数据共享更关心的是：

任务之间是否频繁访问同一批数据？如果是，应该怎样调度它们，才能减少跨核通信和缓存失效？

比如，有两个线程经常访问同一个队列、同一个哈希表、同一块图像数据。如果把它们放在相距很远的核心上，它们每次修改共享数据，都可能引起缓存行在核心之间来回失效和迁移。

这种现象有时候会让“看起来 CPU 都很忙”的系统反而变慢。因为 CPU 时间没有完全花在真正计算上，而是花在等待缓存一致性协议、同步锁、内存访问上。

所以，核间数据共享提醒我们：

调度对象不是孤立的。一个线程应该跑在哪个核心上，和它访问的数据、它协作的线程、它共享的缓存层级都有关系。

这也是为什么现代调度器和运行时系统常常会尽量让访问相同数据的一组任务靠近运行，比如放在同一个核心、同一个 NUMA 节点或同一个处理器簇中。

5. 缓存亲和性问题

缓存亲和性指的是：

让特定进程或线程尽可能在同一个处理器核心上运行。

原因很直观。

如果一个进程刚刚在 CPU 0 上运行过，那么 CPU 0 的 L1 / L2 Cache 中可能还保留着它的代码、栈数据、堆数据和页表相关信息。下一次继续让它在 CPU 0 上运行，它就能最大程度利用已经加载好的缓存。

但如果把它迁移到 CPU 1，CPU 1 的 Cache 里未必有这些数据。于是它需要重新从更低层 Cache 或内存里取数据，造成 Cache miss，性能下降。

所以，缓存亲和性的目标是：

尽量让进程继续在之前运行过的核心上运行，从而提高数据访问速度和系统整体性能。

不过，这个目标不能绝对化。

如果某个核心已经排了很多任务，而另一个核心几乎空闲，还坚持“每个进程都不迁移”，那系统整体执行时间可能反而变长。

因此，PPT 特别强调了一个问题：

缓存亲和性调度 VS 负载均衡。

这就是多核调度里最典型的矛盾之一。

6. 缓存亲和性 VS 负载均衡

负载均衡的目标是：

让各个处理器核心尽量都有任务执行，避免有的核心过载、有的核心空闲。

但是，负载均衡往往需要迁移任务；而任务迁移又会破坏缓存亲和性。

PPT 用了两个例子说明这个矛盾。

第一个例子是多核并行计算或图像渲染任务。

假设一张图像被切成几块，每个核心负责处理一块。某个核心处理的数据会被频繁访问，因此它的 Cache 会逐渐存下这部分数据，访问速度变快。

但问题是，每块图像的计算量未必一样。有的区域可能很复杂，有的区域很简单。于是可能出现某个核心还在辛苦计算，另一个核心已经空闲。

为了减少整体完成时间，系统可能会把负载重的核心上的一部分任务迁移到空闲核心上。

这样做提高了负载均衡，但也带来了代价：被迁移的任务到了新核心上，原来的缓存数据用不上了，会产生额外缓存不命中。

第二个例子是 CDN 缓存。

CDN 会根据用户或地区，把热门视频、商品详情等内容缓存到对应地区的服务器上，这样用户访问更快。这个逻辑和缓存亲和性很像：让请求尽量命中“离它近、已经有数据”的缓存。

但如果某个地区服务器负载太高，系统可能会把该地区用户请求路由到其他地区服务器。这可以缓解负载压力，但也可能造成内容缓存不命中，访问变慢。PPT 用这两个例子说明：亲和性和均衡性不是谁绝对正确，而是要根据整体代价折中。

所以可以这样总结：

多核调度器真正难的地方就在这里：

它要判断什么时候值得为了负载均衡牺牲缓存亲和性，什么时候应该为了缓存亲和性忍受一点负载不均。

7. 单队列与多队列

从队列组织上看，多处理器调度可以粗略分成单队列和多队列。

单队列调度是所有处理器共享一个公共就绪队列。

它很接近前面说的动态进程分配。哪个 CPU 空了，就从公共队列中取一个任务运行。

它的好处是负载天然均衡。因为所有处理器都从同一个池子里拿任务，不容易出现某个 CPU 队列空了、另一个 CPU 队列很长的情况。

十一、实时调度

实时系统的关键词不是“平均速度”，而是：

最后期限。

如果一个任务必须在 10ms 内完成，那么平均 1ms 完成没有意义；只要有一次拖到 20ms，就可能是严重错误。

1. 硬实时与软实时

硬实时系统要求某些动作必须在规定时刻或时间范围内完成。

例如工业控制、汽车装配线中的关键控制环节。如果错过截止时间，可能带来严重后果。

软实时系统允许偶尔错过最后期限。

例如音视频播放中，偶尔一帧延迟可能影响体验，但不一定造成灾难。

4. RM：速率单调调度

RM 是静态优先级实时调度算法。

它通常假设任务是周期性的，并且执行时间已知。

它的基本思想是：

周期越短的任务，越需要频繁响应，因此优先级越高。

课件中也强调，它是根据任务周期性和执行时间分配优先级，执行时间较短或周期更紧的任务具有更高优先级。

RM 的优点是实现简单，优先级固定。

缺点是灵活性不如动态优先级算法。

5. EDF：最早截止时间优先

EDF 是动态优先级实时调度算法。

它的规则是：

谁的 deadline 最近，谁先运行。

如果一个任务马上要到最后期限，它就获得最高优先级。

EDF 的优点是调度能力强，能很好地满足最后期限要求。

缺点是实现复杂：系统需要持续跟踪每个任务的 deadline 和剩余执行时间，任务状态变化时还要重新排序和判断，带来时间和空间开销。

小结

普通交互系统追求“快一点响应”，实时系统追求“必须按时响应”。

这两个目标不一样。实时系统的核心不是平均性能，而是确定性和可预测性。

6. I/O 完成后的优先级提升

当一个线程等待 I/O 操作，I/O 完成后被唤醒，Windows 会临时提升它的优先级，让它更快上 CPU 处理数据。

提升幅度和设备响应时间要求有关。响应时间越敏感的设备，提升幅度越大。设备驱动程序完成 I/O 请求时，可以通过内核机制指定提升幅度。

为了避免这种补偿过度造成不公平，I/O 完成唤醒线程时，Windows 还会把该线程的时间配额减 1。

这很好地体现了调度器的折中：

既要让 I/O 线程快点响应，又不能让它借补偿机制无限占便宜。

7. 等待事件和信号量后的提升

如果线程因为等待事件对象或信号量而阻塞，等待结束后，它的优先级也可能提升一级。

原因是这类线程通常不是一直吃 CPU，而是等待同步事件。适当提升可以减少它们相对于 CPU 密集型线程的不利地位。

但提升不是无限的。它以线程基本优先级为基准，提升后不会超过 15。之后线程会在提升后的优先级上运行剩余时间配额，再逐步降低回基本优先级。

8. 饥饿线程的提升

Windows 有一个系统线程叫平衡集管理器（Balance Set Manager）。

它会周期性扫描就绪队列。如果发现某个就绪线程已经等待超过一定时间还没获得 CPU，比如课件中提到超过 300 个时钟中断间隔，就会把它临时提升到优先级 15，并给它一个较长时间配额，比如正常值的 4 倍。

等它用完这段配额后，优先级会衰减回原来的基本优先级。

直觉解释：Windows 的优先级提升像“补偿券”

I/O 线程、等待锁的线程、长期饿着的线程，本来在普通优先级规则下可能吃亏。

Windows 就临时发一张补偿券，让它先跑一会儿。

但补偿券不是永久身份，跑完以后还要回到原来的基本优先级。

9. 空闲线程

如果某个处理器上没有可运行线程，Windows 会调度该处理器对应的空闲线程。

多处理器系统中，每个处理器都可以有自己的空闲线程。

空闲线程的优先级为 0，只在没有其他线程要运行时才运行。它会在循环中检查是否有待处理工作，例如处理中断请求、进行电源管理等。

3. CFS 的核心直觉

CFS，全称 Completely Fair Scheduler，完全公平调度器。

它的核心直觉是：

理想情况下，每个可运行进程都应该像同时在一台理想 CPU 上运行一样，公平地推进。

现实 CPU 不能真的同时跑所有进程，所以 CFS 做了一件近似的事：

它记录每个进程“已经获得了多少公平意义上的 CPU 时间”，然后总是选择最落后的那个进程运行。

这个“公平意义上的运行时间”就是：

虚拟运行时间 vruntime。

4. vruntime：公平跑表

可以把 CFS 想象成操场跑圈。

每个进程都有一个虚拟跑表 vruntime。

普通优先级进程现实中跑 1ms，跑表也许记 1ms。

高优先级进程权重大，现实中跑 1ms，跑表可能只记 0.5ms。

低优先级进程权重小，现实中跑 1ms，跑表可能记 2ms。

于是调度器每次都选择 vruntime 最小的进程，也就是“在公平跑表上最落后的人”。

核心公式可以写成：

vruntime=实际运行时间×进程权重NICE_0_LOAD​

nice 值越小，优先级越高，权重越大，vruntime 增长越慢。增长慢意味着它更容易长期保持在红黑树左侧，也就更容易再次被选中。

5. 红黑树：按照 vruntime 排队

CFS 不使用传统意义上的固定时间片队列。

它把可运行进程按照 vruntime 作为键，放入一棵红黑树。

每次调度时，选择红黑树最左边的节点。

为什么是最左边？

因为红黑树按 vruntime 排序，最左边就是 vruntime 最小的进程，也就是当前最“亏”的进程。

流程大致是：

    B --> C["更新 vruntime"]
    C --> D["重新插入红黑树"]
    D --> E["调度器选择最左节点"]
    E --> F["vruntime 最小者上 CPU"]

直觉解释：CFS 不是看谁排队最久，而是看谁“公平账本”最亏

RR 是轮流来，优先级调度是看等级，CFS 看的是一个长期账本。

谁在公平意义上拿到的 CPU 少，谁就更靠左，更容易被选中。

8. CFS 的思想总结

CFS 的厉害之处在于，它没有简单说“交互式进程优先”或“短任务优先”，而是试图建立一个统一公平模型。

十五、把整章串起来

最后，我们回到开头的问题。

为什么需要调度？

因为 CPU 数量有限，而可运行进程/线程很多。操作系统必须控制和协调它们对 CPU 的竞争。

调度什么时候发生？

调度通常发生在事件导致进程状态变化之后，例如进程退出、阻塞、唤醒、时间片用完、创建新进程、执行 yield、发生异常等。很多时候，它发生在内核处理完中断、异常或系统调用，准备返回用户态之前。

调度器做什么？

调度器从就绪队列或运行队列中选择下一个进程/线程。不同系统可以使用不同策略：FCFS、SJF、SRTN、HRRN、RR、优先级、多级反馈队列、CFS、EDF 等。

上下文切换是什么？

上下文切换是把 CPU 的硬件状态从当前进程切换到另一个进程，包括保存旧进程的 PC、PSW、寄存器、栈和地址空间信息，再恢复新进程的上下文。

调度算法好坏看什么？

要看吞吐量、周转时间、响应时间、CPU 利用率、公平性、实时性、能耗、可预测性等指标。不同系统目标不同，不存在万能算法。

真实系统怎么做？

Windows 使用基于动态优先级的抢占式线程调度，同级线程按时间配额轮转，并通过 I/O 完成、等待结束、饥饿检测等机制临时提升优先级。

Linux 普通进程主要使用 CFS，通过 vruntime 和红黑树实现公平调度；实时进程使用 FIFO、RR 等实时策略；多核场景还要考虑缓存亲和性和负载均衡。

一句话概括这一讲：

调度就是操作系统在事件驱动的内核入口处，根据系统目标选择下一个运行对象，并通过上下文切换把 CPU 从一个进程/线程安全交给另一个进程/线程。
