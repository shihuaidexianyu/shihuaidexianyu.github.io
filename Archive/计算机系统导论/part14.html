Concurrency

并发的含义与逻辑控制流

并发是指多个逻辑控制流在时间上的重叠推进。这里的“同时”首先是逻辑意义上的，而不是物理意义上的。即使在单核处理器上，只要多个执行流能够通过时间片轮换交替前进，系统就已经具有并发表现。若运行平台拥有多个处理核心，那么这些逻辑流还可能进一步转化为真正的物理并行。

并发的主要作用包括隐藏等待时间、提高系统吞吐量、支持多任务交互以及为并行计算提供组织基础。服务器在等待某个客户端 I/O 完成时，可以继续处理其他客户端请求；图形界面程序在等待磁盘或网络操作时，可以保持界面响应；多核程序则可以把彼此独立的计算任务分派到不同执行流上同步推进。

并发不是单纯的性能技巧，而是一种程序组织方式。它带来更高资源利用率和更强交互能力，同时也引入执行顺序不确定、共享状态竞争、同步开销和死锁等问题。因此，并发编程的核心不是“让事情一起做”，而是“在事情一起做时仍然保持语义正确”。

并发服务器的三种基本模型

服务器程序中的并发，通常围绕“一个客户端会话由什么执行实体来承载”这一问题展开。经典答案主要有三种：基于进程的并发、基于 I/O 多路复用的并发，以及基于线程的并发。

    D --> D2["创建较轻量，通信高效"]
    D --> D3["同步与共享控制更复杂"]

这三种模型都能让服务器同时服务多个客户端，但它们对资源隔离、通信方式、调度机制和错误传播范围的取舍不同。基于进程的模型强调隔离，基于 I/O 多路复用的模型强调集中调度，基于线程的模型则强调轻量化执行流与共享内存通信。并发编程的很多设计问题，实际上都是围绕这三种模型的取舍展开的。

基于进程的并发编程

基于进程的并发服务器通常采用“父进程监听，子进程处理会话”的结构。父进程持续调用接受连接操作，一旦有客户端到达，就创建一个子进程，并将该客户端会话交给子进程处理。父进程随后继续回到监听循环中等待后续客户端。

这一模型的突出特点是地址空间隔离。每个子进程拥有独立的用户地址空间，因此一个子进程中的指针错误、缓冲区越界或局部崩溃通常不会直接破坏其他子进程的内存状态。对于稳定性与故障隔离要求较高的系统而言，这是一项重要优势。

但进程模型并不意味着资源完全独立。通过 fork 创建子进程后，父子进程会共享内核中的打开文件对象，只是在各自描述符表中持有对同一对象的不同引用。因此，若父进程与子进程都保留对某个已连接套接字的引用，就会使该连接的引用计数增加。若其中一方不在适当时机关闭自己的副本，内核就无法及时回收连接资源。这说明在并发服务器中，关闭不用的文件描述符不是可选优化，而是保证语义与资源正确性的必要步骤。

进程模型的主要代价是创建与切换开销较大，并且不同进程之间不能直接共享普通用户空间变量。若多个会话需要共享缓存、统计信息或任务队列，就必须借助管道、共享内存或其他进程间通信机制。因此，基于进程的并发模型更适合强调隔离性的场景，而不适合极高并发度下的轻量化共享任务处理。

基于 I/O 多路复用的并发编程

I/O 多路复用是在单个进程中显式管理多个 I/O 源的一种并发方式。其基本思想是：程序把一组文件描述符交给内核，并阻塞等待其中至少一个描述符进入就绪状态；一旦内核返回，程序只处理那些当前不会阻塞的描述符，而不是无条件地轮询全部连接。

“就绪”是 I/O 多路复用中的核心概念。对一个描述符而言，可读并不意味着一定已经积累了很多数据，而是意味着若立刻执行读取操作，该操作不会因为等待数据而阻塞。对于监听套接字，可读通常意味着当前可以立即执行 accept 并获得新的客户端连接；对于已连接套接字，可读可能表示接收缓冲区中已有数据，也可能表示对端已经关闭连接，读取会立即返回 0。

I/O 多路复用的价值在于，它把“等待哪个连接先准备好”这一问题从应用程序手工管理转交给内核完成，再由单个事件循环按需分发处理。因此，一个进程就可以管理大量网络连接，而不必为每个连接创建独立执行实体。这种方式特别适合 I/O 密集而单次处理逻辑较轻的服务器场景。

它的代价在于程序结构通常更复杂。应用程序必须自行维护连接集合、缓冲状态、协议阶段和事件处理顺序。由于所有逻辑最终仍由一个执行流推进，任何一次耗时过长的处理都可能拖延其他连接的响应。因此，这种模型适合将并发视为“事件驱动调度问题”来处理，而不适合把大量阻塞式或 CPU 密集型任务直接混入事件循环。

基于线程的并发编程

线程是同一进程内部的多个执行流。每个线程都有自己的线程 ID、程序计数器、寄存器状态和运行时栈，但共享所属进程的代码段、全局变量、堆对象和打开文件描述符。线程的这种结构决定了它比进程更轻量，同时也使线程之间的通信比进程间通信更加直接。

线程模型的优势在于两点。第一，线程创建和切换通常比进程更便宜，因此更适合大量短生命周期任务。第二，线程共享地址空间，这使得多个执行流可以直接访问共同的数据结构，而不必显式经过进程间通信通道。线程池、共享缓存、并行计算和多会话服务器都常建立在这一模型之上。

但线程共享也意味着更高的同步复杂度。任何全局变量、静态对象、堆结构，甚至被多个线程间接访问的局部对象，都可能成为共享资源。一旦共享对象的访问没有经过适当同步，程序结果就会依赖线程调度顺序，从而产生竞争条件。因此，线程模型虽然执行轻量，却在正确性推理上更为困难。

线程私有状态与共享状态

并发程序分析的第一步，是区分哪些状态属于线程私有，哪些状态属于线程共享。线程私有状态主要包括寄存器值、调用栈、当前函数的局部自动变量和返回地址。这些内容描述的是某个线程自身的执行现场。线程共享状态则包括全局变量、静态变量、堆上分配的对象、共享缓冲区以及进程层面的打开文件描述符等。

这一划分并不是绝对静态的。一个局部变量只要其地址没有逃逸到其他线程，就可以视为线程私有；但若该局部变量的地址被传递给其他线程，那么这个对象就不再具有严格私有性。由于同一进程中的所有线程共享地址空间，其他线程完全可以通过该地址访问它。此时，该对象的生命周期、安全性和同步约束都必须重新分析。

因此，并发中的“共享”不是由对象位于栈、堆还是全局区来机械决定的，而是由其可达性决定的。只要一个对象可能被多个线程访问，它就属于共享对象，应接受并发控制规则的约束。

同步、互斥与临界区

同步是对并发执行顺序施加约束的机制。互斥是同步的一种特殊形式，表示某个时刻只允许一个执行流进入特定代码区域或访问特定资源。凡是涉及共享状态读写并且执行顺序可能影响结果的代码区域，都可称为临界区。

若多个线程无约束地进入同一临界区，程序的最终结果就可能依赖调度器如何安排交错执行。例如对共享计数器执行“读—改—写”操作时，虽然源代码中只写了一条加法语句，但在机器层面往往分解为多个步骤。若其中两个线程同时执行，就可能发生更新丢失。互斥的作用，就是把这一组本应视为整体的步骤包裹成不可被并发打断的原子区间。

同步并不只是为了防止两个线程“同时写”。只要某个操作序列必须整体保持一致，或者某个资源一次只能由一个执行流独占访问，就需要同步。互斥锁、信号量和条件变量等原语，都是为了把这种逻辑上的顺序要求转化为可执行的运行时约束。

读者—写者问题

读者—写者问题描述的是一类典型共享资源访问模式：多个读操作彼此之间通常可以并发进行，但只要有写操作发生，写者就必须独占资源，读者也不能再并发进入。该问题广泛对应于缓存、配置表、索引结构和数据库页等对象的访问控制。

它的核心在于区分“只读共享”和“写时独占”两种权限。多个读者并发读取同一对象通常不会破坏对象内容，也不会彼此干扰，因此可以共享访问；写者则会改变对象状态，因此写者之间不能并发，写者与读者之间也不能并发，否则读者可能读到一半被修改的中间状态，写者之间也可能互相覆盖结果。

读者—写者问题的重要性不只是会写出某种同步代码，而在于它揭示了同步控制可以比“全局加一把总锁”更细致。通过根据访问模式区分权限，系统可以在保证正确性的前提下保留部分并发性。不过，这种设计也会引入公平性问题。若系统长期优先读者，写者可能饥饿；若长期优先写者，读者吞吐率又会下降。因此，读者—写者问题同时也是并发策略设计问题，而不只是互斥实现问题。

线程安全与可重入性

线程安全函数是指多个线程并发调用时仍然能够产生正确结果的函数。其关键不在于“函数体中是否用了线程接口”，而在于函数是否以安全方式处理了共享状态。线程不安全通常有几种典型来源：直接读写未受保护的共享变量、依赖跨调用保存的静态内部状态、返回指向共享静态对象的指针，或在函数内部调用其他线程不安全接口。

可重入性是比线程安全更严格的性质。可重入函数不依赖共享可变状态，因此即使一个调用尚未结束，函数又被再次进入执行，也不会因为内部状态冲突而出错。若函数只依赖传入参数和局部自动变量，不访问共享全局对象或静态可变对象，那么它通常是显式可重入的。若函数通过指针参数操作外部对象，而这些对象本身并不共享或由调用方保证独立，也可视为隐式可重入。

线程安全与可重入性并不等价。一个函数可以通过加锁保护其内部共享状态，从而在多线程并发调用时表现正确，因此它是线程安全的；但它仍然可能依赖锁、阻塞或共享内部对象，因此不满足可重入条件。可重入函数一定线程安全，而线程安全函数不一定可重入。

竞争条件

竞争条件是指程序结果依赖于并发执行的具体交错顺序。只要多个执行流访问同一共享状态，而访问顺序又未被充分约束，就可能出现竞争条件。其危险在于，程序在某些调度顺序下看似正常，在另一些顺序下却产生错误结果，因此问题具有偶发性和难以复现性。

一个典型情形是在创建多个线程的循环中，把循环变量的地址直接传给新线程。循环变量虽然在源代码中是局部变量，但它的地址一旦被多个线程共享，就进入共享语义。主线程继续推进循环并修改该变量时，工作线程实际读取到的值取决于调度时机，因而可能不是创建线程时预期的那个参数值。这个例子说明，竞争条件并不局限于显眼的全局变量；任何逃逸到多线程环境中的对象，都可能成为竞争源。

竞争条件的本质不是“有两个线程都在运行”，而是“对共享状态的访问没有符合语义需要的顺序约束”。因此，解决竞争条件的关键不是一味减少线程，而是识别共享对象、分析访问模式，并在必要处引入同步或消除共享。

死锁

死锁是指一组执行流彼此等待对方所持有的资源，因而所有执行流都无法继续推进的状态。死锁形成后，参与者不会自行恢复，因为每个执行流继续执行所必需的条件都掌握在其他执行流手中，而这些执行流本身也处于同样困境。

死锁常见于多个锁或多个资源的嵌套获取场景。若线程 T1 先获得锁 L1，再等待锁 L2；同时线程 T2 先获得锁 L2，再等待锁 L1，就形成了循环等待。只要二者都不主动释放当前持有的锁，程序就会永久停在该状态。

graph LR
    T1["线程 T1"] -->|持有| L1["锁 L1"]
    T1 -->|等待| L2["锁 L2"]
    T2["线程 T2"] -->|持有| L2
    T2 -->|等待| L1

死锁的本质不是某个线程“运行得很慢”，也不是某个线程“暂时阻塞”，而是系统整体已经失去前进路径。预防死锁的一种重要方法是统一资源获取顺序。若所有线程都按同一全序申请资源，例如总是先申请编号较小的锁，再申请编号较大的锁，那么循环等待就不可能形成。该方法并未减少互斥本身，而是通过消除等待环来防止死锁。

并行性能、加速比与效率

当并发程序运行在多核处理器上时，不同执行流可以真正并行执行，此时并发结构会直接影响程序的并行性能。并行收益通常用加速比与效率来衡量。

若 T1​ 表示程序在单核上的执行时间，Tp​ 表示程序在 p 个处理核心上的执行时间，则加速比定义为

Sp​=Tp​T1​​

加速比描述程序因使用多个核心而获得的总体性能提升。理想情况下，若使用 p 个核心就使运行时间缩短为原来的 p1​，则有 Sp​=p，称为线性加速。

效率定义为

Ep​=pSp​​

效率反映每个核心平均发挥了多少比例的理想能力。效率越接近 1，说明新增核心被充分利用；效率越低，则说明并行带来的额外收益不足。

限制并行扩展性的主要因素有两个。第一是程序中的串行部分。只要某部分工作必须顺序完成，那么无论增加多少核心，这一部分都构成总时间的下界。第二是同步与共享开销。若多个线程频繁争抢锁、等待条件或更新共享状态，新增核心带来的潜在加速会被这些协调成本部分抵消。因此，高性能并发程序通常尽量减少共享、延迟合并结果、扩大线程私有工作量，并把高频更新限制在局部状态中。

并发编程的核心问题

并发编程的本质，不在于简单增加执行实体，而在于在多个逻辑流共享有限资源的条件下，保持程序语义、资源管理和性能行为的正确性。任何一个并发程序，至少都需要回答四个问题：哪些状态是共享的，哪些操作必须串行，哪些资源需要独占，以及在异常、阻塞和退出发生时应如何回收状态。

进程模型强调隔离，多路复用模型强调集中调度，线程模型强调轻量与共享。三者分别对应不同的系统目标和工程约束，没有一种方案能够在所有维度上同时最优。并发编程真正需要掌握的，不是某一种接口模板，而是如何将执行流、共享状态、同步约束、资源依赖和性能边界组织成一个可分析、可验证的整体模型。
