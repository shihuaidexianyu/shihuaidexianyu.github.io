第五讲 传输层

1. 传输层的地位、作用与基本问题

传输层位于应用层之下、网络层之上，是面向通信部分中的最高层，也是直接面向应用进程提供通信服务的最低层。网络层处理的是主机到主机之间的逻辑通信，而传输层处理的是应用进程到应用进程之间的端到端逻辑通信。所谓“端到端”，其真正含义不是两台主机直接相连，而是两端主机中相互通信的应用进程借助网络完成信息交换。路由器在网络核心中只工作在网络层及以下各层，并不实现传输层功能，因此传输层只存在于网络边缘的主机协议栈中。

传输层向上层屏蔽了底层网络核心的复杂细节，使应用进程所看到的是两端传输实体之间的一条逻辑通信信道。该信道的性质由所选用的传输层协议决定：若使用 TCP，则上层看到的是全双工可靠信道；若使用 UDP，则上层看到的是无连接、不可靠的数据报信道。传输层因此不仅承担“发送数据”的职责，还负责在进程标识、复用与分用、差错处理、可靠性控制、流量协调和拥塞适配等方面建立统一机制。

传输层需要解决的核心问题之一，是如何在一台主机上存在多个并发进程、进程又会动态创建和撤销的前提下，仍能准确标识通信终点。操作系统中的进程标识符不适合作为互联网中的统一进程地址，因此 TCP/IP 体系采用协议端口号来标识应用层通信端点。这样，发送方不必知道对方主机上具体是哪一个进程实例在工作，只需知道目标主机所提供服务对应的端口即可。

2. 端口、复用与分用

端口是协议栈层间交互中的软件地址，不是交换机或路由器上的硬件接口。硬件端口是设备连接介质的物理接口，而软件端口是应用进程与传输层实体之间的逻辑交互点。通信时，发送方应用进程的数据先交给本地主机的某个端口，再由传输层封装、交付网络层；到达接收方后，传输层根据目的端口将数据分发给正确的应用进程。

这种机制体现为复用与分用。复用是指多个应用进程都可以通过各自端口共享同一网络层提供的主机到主机通信服务；分用是指接收方传输层根据报文段或用户数据报中的目的端口字段，把收到的数据交付给不同的上层进程。IP 完成的是主机级分用，TCP 和 UDP 完成的是进程级分用。

端口号长度为 16 位，因此理论上可表示的端口编号范围是 0 到 65535。端口号只具有本地意义，同一端口号在不同主机上彼此独立，没有全局绑定关系。真正要唯一确定一端通信终点，必须同时给出 IP 地址和端口号。

端口大体分为三类。数值范围 0 到 1023 的为熟知端口，通常分配给系统级标准服务；1024 到 49151 为登记端口，供已注册应用使用；49152 到 65535 为短暂端口，通常由客户进程临时选用，通信结束后即可释放。典型熟知端口包括：HTTP 使用 80，HTTPS 使用 443，DNS 使用 53，SMTP 使用 25，Telnet 使用 23，FTP 使用 20 和 21，TFTP 使用 69，SNMP 使用 161 和 162，RPC 使用 111。

3. 传输层的两类主要协议：UDP 与 TCP

TCP/IP 体系中的传输层主要包含两种协议：用户数据报协议 UDP 与传输控制协议 TCP。对等传输实体之间传送的数据单位称为传输协议数据单元。TCP 的传输协议数据单元是 TCP 报文段；UDP 的传输协议数据单元是 UDP 用户数据报。

UDP 是无连接协议，提供无连接服务，不保证可靠交付，首部开销小，支持单播、多播和广播，适用于时延敏感、可容忍个别丢失的应用。TCP 是面向连接协议，提供可靠交付、全双工通信和点对点传输，机制复杂，但更适用于网页访问、电子邮件和文件传输等要求正确性和顺序性的场景。典型地，DNS、DHCP、RIP、SNMP 等常基于 UDP；HTTP、SMTP、FTP 等常基于 TCP。

4. UDP：无连接、面向报文的数据报服务

UDP 在 IP 提供的尽最大努力数据报服务之上，只额外增加了两类基本能力：端口级复用与分用，以及差错检测。它不建立连接，不维护连接状态，也不提供可靠传输、流量控制或拥塞控制。由于省去了连接建立、状态维护和复杂控制逻辑，UDP 实现简单、首部短小，发送前时延低，尤其适用于实时业务、短时突发数据、音视频通信以及一次请求一次应答类服务。

UDP 是面向报文的协议。发送方应用层交给 UDP 多长的报文，UDP 就按该边界原样封装后交给 IP 层；接收方 UDP 在去掉首部后，也按报文边界一次交付一个完整报文，而不合并也不拆分。报文过长会使 IP 层发生分片，降低效率并增加差错恢复代价；报文过短又会使首部相对占比过大，降低有效负载效率。因此，UDP 应用需要自行选择合适的报文长度。

4.1 UDP 首部格式

UDP 首部长度固定为 8 字节，共 4 个字段，每个字段 2 字节：

源端口字段用于标识发送方端口；目的端口字段用于标识接收方端口；长度字段给出 UDP 用户数据报总长度，即首部加数据的长度；检验和字段用于对首部和数据做差错检测。接收方收到 UDP 数据报后，依据目的端口字段将数据分用给相应应用进程。

4.2 UDP 检验和与伪首部

UDP 检验和不是只覆盖首部，而是同时覆盖首部和数据。计算时，需要在 UDP 用户数据报前临时加上 12 字节伪首部。伪首部并不实际发送，只用于计算检验和。其内容包括：源 IP 地址、目的 IP 地址、全 0 字段、协议号 17，以及 UDP 长度。然后将伪首部、UDP 首部和数据部分按 16 位一组做反码求和；若数据字节数为奇数，则在末尾补一个全 0 字节后再求和；最后对和取反码，得到检验和。

给定示例中，源地址为 153.19.8.104，目的地址为 171.3.14.11，协议号为 17，UDP 长度为 15 字节，数据区按 16 位字补齐后参与求和，最终得到的反码结果为 0110100100010010，这就是该 UDP 报文的检验和值。该例说明，UDP 差错检测虽简单，但已将端到端地址信息纳入校验范围，从而能发现部分“报文被投递到错误目的地”的异常。

5. TCP 概述：面向连接、可靠、全双工、面向字节流

TCP 运行在无连接且不可靠的 IP 服务之上，但通过一整套控制机制实现了可靠交付。TCP 的基本特征包括：面向连接、点对点、一条连接只有两个端点、提供可靠服务、支持全双工通信，以及面向字节流。

“面向字节流”并不意味着应用层不能按块写入数据，而是指 TCP 不保留应用层写入时的数据块边界。应用程序可以一次写入一个或多个数据块，TCP 把它们统一看作一串连续、无结构的字节流，再根据窗口大小、拥塞状况和最大报文段长度自行决定如何分段。接收方应用程序读到的数据块大小也不一定与发送方写入时一致，但接收到的字节序列必须与发送方发出的字节序列完全相同。

TCP 连接本身是一种逻辑抽象，而不是物理链路。应用进程写入的数据先进入发送缓存，TCP 再将字节流分段，附加首部形成报文段；接收方 TCP 将按序或暂存的字节放入接收缓存，应用进程再从接收缓存中读取。发送报文段的边界由 TCP 决定，而不是由应用层写入动作决定。

6. Socket：TCP 连接端点与编程接口

TCP 连接的端点不是主机本身，不是单独的 IP 地址，也不是单独的端口号，而是套接字。套接字由 IP 地址和端口号拼接而成：

socket=(IP地址:端口号)

一条 TCP 连接由通信两端两个套接字唯一确定：

TCP连接={(IP1​:port1​),(IP2​:port2​)}

因此，同一个 IP 地址可以同时参与多条 TCP 连接，同一个端口号也可以出现在多条不同连接中，只要另一端套接字不同即可。

“socket”一词在不同语境中有多重含义：它可以指套接字编程接口，也可以指某个 API 函数名，还可以指该函数返回的描述符，或操作系统内核中的 Berkeley socket 实现。就网络通信抽象而言，最关键的含义仍然是“网络进程的通信端点”。

在编程模型中，TCP 常见服务原语包括 socket、bind、listen、accept、connect、send、receive 和 close。其中服务器通常先创建套接字并绑定本地地址，再进入监听、接受连接和数据收发循环；客户机通常创建套接字后主动调用 connect 建立连接，再进行发送、接收和关闭。UDP 的编程接口更简单，通常只需 socket、bind、send、receive 和 close，因为它不需要面向连接的建立与接受过程。

7. 可靠传输的基本思想

IP 网络本身只提供尽最大努力的数据报服务，不能保证不出错、不丢失、不重复或不失序。因此，若要实现可靠传输，必须在不可靠的信道之上再叠加可靠传输协议。

理想传输条件有两个：一是传输信道永不出错，二是无论发送方多快发送，接收方总能及时处理。现实网络不具备这两个条件，因此可靠传输协议必须处理差错、丢失、重复、乱序以及收发速率不匹配等问题。

7.1 停止等待协议

停止等待协议的规则极其简单：发送方每发送完一个分组，就停止发送并等待接收方确认；只有收到对前一分组的确认后，才发送下一个分组。若接收方正确收到分组，就发回确认；若分组出错或丢失，则不发送确认。发送方为每个分组设置超时计时器，在超时之前收到确认就转发下一个分组，若超时则重传当前分组。

为了区分新分组与重传分组，数据分组和确认分组都必须编号。若确认丢失，发送方会因超时而重传；接收方收到重复分组后必须将其丢弃，但仍要再次发送确认。若确认只是迟到，则发送方可能先重传，再收到两个确认；对发送方而言，重复确认可直接丢弃。可靠交付正是通过“编号 + 确认 + 超时重传 + 重复丢弃”实现的。

这种机制称为自动重传请求 ARQ。其“自动”含义在于：重传动作由发送方依据超时机制自动执行，接收方不需要显式请求重传。

停止等待协议的主要缺点是信道利用率低。若设发送一个分组所需时间为 TD​，往返时间为 RTT，发送确认所需时间为 TA​，则信道利用率可写为

U=TD​+RTT+TA​TD​​

当 RTT≫TD​ 时，大部分时间都耗在等待确认上，真正用于传送有效数据的比例很低。若出现重传，利用率还会进一步下降。

7.2 流水线传输与连续 ARQ

为提高效率，发送方可以在未收到前一个分组确认时继续发送后续多个分组，这就是流水线传输。流水线传输需要滑动窗口机制协调发送方和接收方允许发送、允许接收的序号范围。

连续 ARQ 协议的基本思想是：发送方一次可以发出多个分组；每收到一个确认，发送窗口就向前滑动；接收方通常采用累积确认；出现丢失时，传统实现常采用回退 N 重传。累积确认意味着接收方无需逐个确认每个分组，而是确认“按序到达的最后一个分组”，从而隐含表示此前所有分组都已正确收到。这样实现简单，也能减少确认报文数量，但代价是发送方无法准确知道哪些后续分组虽失序到达却已被接收方缓存。

回退 N 的含义是：若中间某个分组丢失，接收方只能持续确认丢失点之前最后一个按序分组，发送方在超时后必须把丢失分组及其后续已发送但未确认的多个分组全部重传。线路质量较差时，这种机制会造成较大冗余。

可靠传输协议的设计空间可以从多个维度刻画：确认方式可以是独立确认、累积确认或混合；重传方式可以是回退重传或选择重传；触发机制可以是超时或重复确认；接收方可以只接受按序数据，也可以缓存乱序数据；确认回复既可以逐分组发送，也可以合并发送。TCP 的实际机制正是在这些设计维度上做出的综合选择。

8. TCP 报文段首部

TCP 虽然面向字节流，但其传送的数据单元仍然是报文段。一个 TCP 报文段由首部和数据两部分构成。首部前 20 字节固定，之后还有长度为 4n 字节的可选字段，因此 TCP 首部最小长度为 20 字节。TCP 的几乎全部控制功能都体现在首部字段中。

8.1 固定首部中的主要字段

源端口和目的端口各占 2 字节，用于完成进程级复用与分用。

序号字段占 4 字节。TCP 对连接中传送的每一个字节编号，序号字段的值是本报文段中数据部分第一个字节的序号。若初始序号为 1001，且每个报文段传送 1000 字节数据，则后续报文段序号依次为 1001、2001、3001、4001、5001。

确认号字段占 4 字节，其值是期望收到对方下一个数据字节的序号。确认号不是“确认某个报文段编号”，而是“确认截至某个字节为止的数据都已按序收到”。因此，TCP 的确认本质上是基于字节序号的。

数据偏移字段占 4 位，也称首部长度字段，指出 TCP 数据部分距离报文段起始处的偏移量。该字段的单位是 32 位字，即 4 字节。

保留字段当前应置为 0。

8.2 控制位

URG 为 1 时，表示紧急指针字段有效，报文段中包含需要尽快传送的紧急数据。

ACK 为 1 时，表示确认号字段有效。连接建立后绝大多数报文段都带有 ACK。

PSH 为 1 时，表示接收方应尽快把数据交付应用进程，而不是继续等待缓存填满。

RST 为 1 时，表示连接出现严重异常，需要复位并重新建立连接。

SYN 为 1 时，表示该报文段用于连接建立过程中的同步。

FIN 为 1 时，表示发送方已无数据发送，请求释放连接。

8.3 窗口、检验和、紧急指针与选项

窗口字段占 2 字节，用于告知对方当前接收方还允许接收多少字节，是对方设置发送窗口的重要依据。

检验和字段占 2 字节，覆盖 TCP 首部和数据。在计算时，也要在 TCP 报文段前临时附加 12 字节伪首部。与 UDP 相比，伪首部中的协议号改为 6。

紧急指针字段占 16 位，指出本报文段中紧急数据的字节数。紧急数据放在数据部分最前面，仅当 URG 为 1 时才有意义。

选项字段长度可变。最重要的早期选项是最大报文段长度 MSS。MSS 指的是 TCP 报文段中数据字段的最大长度，不含 TCP 首部本身。若 MSS 取值过小，则首部开销相对偏大；若取值过大，则 IP 层可能需要分片，增加重组和差错恢复代价。因此，MSS 的理想目标通常是在不引起 IP 分片的前提下尽量取大。除 MSS 外，常见选项还包括窗口扩大选项、时间戳选项和选择确认选项。填充字段的作用是把整个首部补齐到 4 字节整数倍。

9. TCP 可靠传输的具体实现

TCP 的可靠传输建立在字节编号、滑动窗口、缓存管理、确认机制和自适应重传定时之上。连接两端都必须维护发送窗口和接收窗口，而且 TCP 的滑动窗口以字节为单位，而不是以报文段为单位。

9.1 发送窗口与接收窗口

发送窗口表示：在尚未收到确认的情况下，发送方还可以连续发送多少字节。接收窗口表示：接收方当前还能接收多少字节。接收方在确认报文中通知自己的窗口值，发送方据此构造本端发送窗口。由于通知和传播存在时滞，发送方看到的窗口不一定始终与接收方真实可用窗口完全一致，因此两端窗口会处于动态变化之中。TCP 标准强烈不赞成发送窗口前沿向后收缩。

用三个指针可描述发送窗口内部状态。设 P1​ 为已发送但尚未确认部分的起始位置，P2​ 为下一个待发送字节的位置，P3​ 为窗口上界，则有

P3​−P1​=发送窗口大小 P2​−P1​=已发送但尚未收到确认的字节数 P3​−P2​=可用窗口

当窗口中的序号都已被发出但还没有新的确认到来时，有效窗口为 0，发送方必须停止发送。

9.2 发送缓存与接收缓存

发送缓存用于暂存两类数据：一类是应用进程写入、等待发送的数据；另一类是已经发送出去、但还未收到确认的数据。接收缓存同样暂存两类数据：一类是按序到达但尚未被应用进程读取的数据；另一类是乱序到达、暂时还不能向上交付的数据。

TCP 并不要求接收方如何处理乱序到达的数据，但常见做法是将其先缓存在接收窗口中，等到缺失字节到达后再按序交付给应用进程。接收方必须支持累积确认，以减少开销。接收方可以立即确认，也可以适度延迟确认，甚至在自己有数据发送时捎带确认；但确认不能推迟过久，否则会导致发送方不必要的重传。

9.3 自适应重传时间

TCP 每发送一个报文段，都会为其设置重传计时器。若计时器到期仍未收到确认，则重传该报文段。重传时间过短会导致大量不必要重传，增大网络负荷；过长又会造成线路空闲和效率下降。因此，TCP 采用自适应算法，根据往返时间 RTT 的测量结果动态计算重传超时值 RTO。

TCP 维护平滑往返时间 RTTS：

RTTSnew​=(1−α)RTTSold​+αRTTsample​

其中 α 的推荐值为 1/8。

TCP 还维护 RTT 偏差的加权平均值 RTTD：

RTTDnew​=(1−β)RTTDold​+β∣RTTS−RTTsample​∣

其中 β 的推荐值为 1/4。

重传超时值按下式给出：

RTO=RTTS+4RTTD

这使得 RTO 不仅跟随平均时延变化，也能对时延波动进行裕量补偿。

若某报文段发生了重传，则随后收到的确认究竟是对应原报文段还是重传报文段，无法可靠判定。Karn 算法因此规定：凡是发生重传的报文段，其 RTT 样本不再用于更新 RTT 和 RTO。修正的 Karn 算法进一步规定：每重传一次，就把 RTO 成倍增大，即

RTOnew​=γRTOold​

其中 γ 的典型取值为 2。待不再发生重传后，再恢复基于新 RTT 样本的正常更新。

9.4 选择确认 SACK

当接收方收到了若干不连续的字节块时，单纯的累积确认无法精确告诉发送方哪些乱序数据已经到达。选择确认 SACK 提供了更细粒度的信息。使用 SACK 时，连接建立阶段需要通过首部选项协商“允许 SACK”；一旦协商成功，确认号字段仍维持原有含义，继续表示按序到达部分的下一个期望字节序号，而附加的 SACK 选项则用来报告已经收到的不连续字节块边界。

若接收方已经按乱序方式收到两个字节块，左边界分别为 L1​=1501、L2​=3501，右边界分别为 R1​=3001、R2​=4501，则分别表示已收到字节区间 [1501,3000] 与 [3501,4500]。由于首部选项总长最多 40 字节，而一个边界要占 4 字节，因此单个 TCP 首部中最多只能报告 4 个字节块边界信息。

10. 流量控制

流量控制解决的是发送方与接收方处理能力不匹配的问题。它的目标是：既不让发送方发送过快而淹没接收方，也不让接收方可用能力长期闲置。TCP 通过接收方窗口 rwnd 与滑动窗口机制实现流量控制。接收方在确认报文中通告当前可用接收窗口，发送方据此调整允许发送的新字节范围。

在一个典型例子中，接收方初始通告 rwnd=400 字节，发送方在每次收到带窗口信息的确认后，动态更新可继续发送的序号区间。当接收方通告 rwnd=0 时，表示接收缓存暂时已满，发送方此时不能再发送新的数据，只能在必要时重传旧数据。若之后接收方缓存恢复并发送非零窗口通知，但该通知在途中丢失，就会出现互相等待的死锁：发送方等待窗口打开，接收方等待新数据到来。

为解决零窗口死锁，TCP 为每条连接设置持续计时器。只要一方收到零窗口通知，就启动该计时器；计时器到期后，发送一个仅携带 1 字节数据的零窗口探测报文段。对方在确认该探测时，会重新给出当前窗口值。若窗口仍为 0，则继续重新计时；若窗口已非 0，则发送方恢复正常发送。

10.1 发送时机与糊涂窗口综合征

发送时机同样影响效率。常见发送触发条件包括：缓存中累积到 MSS 字节时发送；应用进程显式要求推送时发送；发送计时器到期时把当前缓存中的数据组装成不超过 MSS 的报文段发送。

若每次只发送 1 个字节的数据，则一个 TCP 报文段至少需要 20 字节 TCP 首部，再加 20 字节 IP 首部，连同 1 字节数据，总共要传 41 字节，有效数据传输效率只有

411​≈2.44

这种“每次仅发送少量数据导致首部比例异常高”的现象称为糊涂窗口综合征。

发送方侧的糊涂窗口综合征通常出现在应用进程逐字节把数据送入 TCP，而 TCP 又逐字节立即发送。Nagle 算法的基本思想是：先发送第一个小分组，把后续到达的数据缓存起来；在收到对前一小分组的确认后，再把缓存中的数据尽量合并为较大的报文段发送；若累计数据达到发送窗口的一半或达到 MSS，则立即发送。这样可以显著减少“小分组洪泛”。

接收方侧的糊涂窗口综合征出现在接收缓存满后，应用进程又以很慢速度每次只读出极少数据，导致接收方反复通告极小窗口，发送方被迫不断发送 1 字节小报文，形成低效循环。其本质是接收方消费速率过低与窗口通告过细碎共同造成的。

11. 拥塞控制

拥塞控制处理的是网络内部资源不足的问题，而不是端系统收发能力不匹配的问题。流量控制的对象是接收方，拥塞控制的对象是整个网络。TCP 的拥塞控制属于基于反馈的闭环控制：通过监测超时、重复确认、时延和丢包等信号，推断网络拥塞状态，再据此调节发送速率。

TCP 发送方维护拥塞窗口 cwnd。真正允许发送的数据量，不仅受到接收方窗口 rwnd 限制，还受到拥塞窗口限制。因此发送窗口的上限为

发送窗口上限=minrwnd,cwnd

若 rwnd<cwnd，说明接收方处理能力成为瓶颈；若 cwnd<rwnd，则说明网络拥塞状态限制了发送速率。

11.1 慢开始与拥塞避免

TCP 建立连接后，通常先令 cwnd=1 个报文段，并设置慢开始门限 ssthresh。当 cwnd<ssthresh 时，执行慢开始。慢开始并不“慢”，其规律是：每经过一个传输轮次，拥塞窗口加倍；等价地说，每收到一个对新报文段的确认，就把 cwnd 增加 1，从而在一个 RTT 内实现指数式增长。传输轮次强调的是“把当前窗口允许的数据全部发出，并收到相应确认”这一完整过程。

当 cwnd>ssthresh 时，改用拥塞避免算法。此时拥塞窗口不再指数增长，而是每经过一个传输轮次只增加 1，呈线性上升趋势。若 cwnd=ssthresh，则可视实现进入慢开始或拥塞避免。拥塞避免的目的不是完全避免拥塞，而是让窗口增长更为温和，减少再次触发拥塞的概率。

当发送方通过重传定时器超时判断网络拥塞时，无论当前处于慢开始还是拥塞避免阶段，都执行如下调整：

ssthresh = \max\left{\frac{cwnd}{2}, 2\right} cwnd=1

之后重新进入慢开始。这体现了对拥塞的强烈反应：迅速降低注入网络的数据量，让路由器有时间消化积压队列。

11.2 快重传与快恢复

如果接收方收到了失序分组，它不会等待自己有数据可发时再捎带确认，而是立即发送对最后一个按序字节的重复确认。发送方若连续收到 3 个或更多重复 ACK，就可推断某个报文段很可能已丢失，因此无需等待 RTO 到期就立即重传该报文段，这就是快重传。快重传的意义在于更早发现单个分组丢失，避免把一般丢包误判为严重拥塞。

TCP Reno 在收到 3 个重复 ACK 后通常执行快恢复，而不是把 cwnd 直接降回 1。其核心规则为：

ssthresh=2cwnd​ cwnd=ssthresh

然后直接进入拥塞避免阶段，令窗口线性增长。这样做的理由是：连续 3 个重复 ACK 表明仍有数据到达接收方，网络未必已经全面拥塞，更可能只是个别报文段丢失。

慢开始、拥塞避免、超时后退和快恢复共同体现了 TCP 拥塞控制的 AIMD 思想，即“加法增大、乘法减小”。在拥塞避免阶段，窗口按线性规律逐步增大；一旦检测到拥塞信号，则按乘法方式显著减小窗口和门限。

12. TCP 连接管理

TCP 把连接管理视为协议语义的一部分。连接建立前，服务器一侧通常先创建传输控制块 TCB，进入被动打开状态，等待客户连接请求。客户通过主动打开发起连接。

12.1 三次握手

TCP 建立连接采用三报文握手。

第一次，客户向服务器发送连接请求报文段，置 SYN = 1，并选择初始序号 seq = x。

第二次，服务器若同意建立连接，则回复 SYN = 1, ACK = 1, seq = y, ack = x + 1，一方面确认客户的同步请求，另一方面给出自己的初始序号。

第三次，客户收到后再发回确认 ACK = 1, seq = x + 1, ack = y + 1。客户发出第三个报文段后，可认为连接已建立；服务器收到该确认后，也确认连接建立完成。

采用三次握手而不是两次握手，主要是为了防止已经失效的连接请求报文段在网络中滞留后又突然到达服务器，从而导致服务器错误地建立连接。第三次确认使双方都能确认：自己的发送能力与接收能力、对方的发送能力与接收能力，以及双方初始序号同步都已完成。

12.2 四次挥手与半关闭

TCP 连接释放采用四报文握手，因为 TCP 是全双工的，两个方向的关闭需要分别处理。

第一次，主动关闭方发送 FIN = 1, seq = u，表示本方向数据已发送完毕。

第二次，被动关闭方确认该 FIN，发送 ACK = 1, seq = v, ack = u + 1。此时从主动关闭方到被动关闭方这个方向已经关闭，但反方向仍可继续发送数据，连接进入半关闭状态。

第三次，被动关闭方在自己也无数据发送后，再发送 FIN = 1, ACK = 1, seq = w, ack = u + 1。

第四次，主动关闭方回送 ACK = 1, seq = u + 1, ack = w + 1。至此，双方都完成了本方向上的关闭。

12.3 TIME_WAIT 与保活计时器

主动关闭方在发送最后一个 ACK 后，不会立刻进入 CLOSED，而是进入 TIME_WAIT，并等待 2MSL。这样做有两个目的：其一，保证最后一个 ACK 能到达对方；其二，避免旧连接中滞留的失效报文段混入后续新连接。材料中将这一等待时间解释为约 2 分钟。

为防止连接长期空闲而一方已失效却迟迟未被发现，TCP 还可设置保活计时器。典型设置是：若服务器在 2 小时内未收到客户任何信息，则发送探测报文段；若连续发送 10 个探测报文段、每次间隔 75 秒仍无响应，则认为客户故障并终止连接。

12.4 有限状态机

TCP 连接生命周期可由有限状态机描述，主要状态包括 CLOSED、LISTEN、SYN_SENT、SYN_RCVD、ESTABLISHED、FIN_WAIT_1、FIN_WAIT_2、CLOSE_WAIT、LAST_ACK、CLOSING 和 TIME_WAIT。正常客户侧路径通常是 CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED，正常服务器侧路径通常是 CLOSED → LISTEN → SYN_RCVD → ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED。此外，还存在同时打开、同时关闭、收到 RST 等异常转移。

13. 基于 UDP 的 DNS

域名系统 DNS 用于把人类易记的域名映射为网络层使用的 IP 地址。早期互联网曾依赖集中维护的 hosts.txt 文件保存主机名与 IP 对应关系，但随着主机数量增长，集中维护方式难以保证唯一性、可扩展性和及时更新，因此逐步发展为分布式层次化的域名系统。域名的优势在于可读、稳定，并能在服务器迁移或更换时保持名称不变；同一主机还可以配置多个别名，而多个服务器也可以共享同一域名以实现负载分配。

DNS 名字空间采用层次结构。域名可表示为“主机名.三级域名.二级域名.顶级域名”的形式，如 www.pku.edu.cn。顶级域名大体分为两类：国家顶级域名，如 .cn、.us、.uk；通用顶级域名，如 .com、.net、.org、.edu、.gov、.mil 和 .int。

13.1 分布式层次结构

DNS 采用分布式结构而不是集中式结构，原因在于集中式系统存在单点失效、性能瓶颈、地理时延大、数据库规模过大且更新困难等问题。分布式 DNS 主要由四类服务器构成：根域名服务器、顶级域名服务器、权限域名服务器和本地域名服务器。

根域名服务器保存所有顶级域名服务器及其 IP 地址；顶级域名服务器负责 .com、.org、.net、.edu 及国家域名等更高层次区域的解析；权限域名服务器负责某一具体区域的记录维护，是某一域名最终权威答案的来源；本地域名服务器通常由 ISP 提供，主机的 DNS 请求首先发往本地域名服务器，它再代表客户继续向分布式 DNS 体系发出查询。

材料中给出的根服务器个数为 13 个逻辑根服务器，从 a.rootservers.net 到 m.rootservers.net。给出的解释是：早期 UDP 响应长度受 512 字节限制，因此一条根服务器列表响应最多只能容纳 13 个根服务器名称与地址信息。

13.2 递归查询与迭代查询

在递归查询中，被查询的 DNS 服务器需要代替请求方继续完成后续查询，并把最终完整结果返回给请求方。这样，客户负担较轻，但服务器负担较大，因此很多高层 DNS 服务器，尤其是根服务器，通常不支持递归查询。

在迭代查询中，被查询服务器只返回它所知道的下一步线索，例如返回下一级 DNS 服务器的地址，而不是代替客户完成全部解析。请求方或其本地域名服务器再继续向新的服务器发起下一次查询。DNS 的实际工作中，主机到本地 DNS 常表现为递归式请求，而本地 DNS 向根、顶级和权限服务器的过程通常采用迭代查询。

13.3 缓存与可靠性

为减少时延和降低上级服务器负担，DNS 服务器普遍维护高速缓存。缓存中保存最近使用过的名字与映射记录，每条记录都带有有效时间。时间值设置较大可以减少网络开销，但会降低解析结果的实时准确性；时间值较小可以提升准确性，但会增加查询频率。权限域名服务器在响应中会给出记录的有效期，本地 DNS 依据该时间决定缓存保留时长。

DNS 的可靠性可通过主域名服务器与辅助域名服务器配合实现。主服务器负责记录修改，辅助服务器定期从主服务器复制数据。在主服务器故障时，辅助服务器可继续提供解析服务，从而避免单点失效。主机配置中通常可同时列出主、辅 DNS 的 IP 地址。

13.4 DNS 安全问题

DNS 查询本身缺乏对目标 DNS 服务器的强认证，因而容易受到多种攻击。典型问题包括：针对根或顶级 DNS 的大量请求攻击、伪造目标主机地址以诱发反射式响应洪泛、截获并伪造 DNS 响应以实施重定向攻击、中间人通过 ARP 缓存中毒先劫持链路再篡改 DNS 应答，以及本地 DNS 缓存染毒。

缓存染毒的实质，是攻击者伪造来自上级 DNS 的合法响应，并抢在真实响应到达之前把伪造记录写入本地 DNS 缓存。此后，普通用户向本地 DNS 查询某一域名时，就会被错误地映射到攻击者控制的地址。DNS 高速缓存虽然能显著加速解析、降低负载，但也因此成为安全攻击的重要目标。
