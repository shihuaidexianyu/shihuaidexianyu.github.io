第一讲 计算机网络概述

1 计算机网络的作用、定义与基本表示

计算无处不在。计算机是进行数据存储与处理的电子设备，而计算机网络则是在计算设备之间传输数据并支持资源共享的系统。文件传输、电子邮件、即时通信、信息检索、导航定位、电子商务、网络教育等应用，都建立在网络提供的连通能力之上。计算机网络已经成为信息社会的基础设施，其影响不仅体现在信息共享和知识获取，也体现在广告、在线零售、众筹、共享经济等经济形态之中。

从广义上看，” 网络 ” 是把人或物互连而形成的系统。大众熟悉的三类网络包括电信网络、有线电视网络和计算机网络。电信网络主要提供电话、电报与传真等服务，有线电视网络主要提供节目传送服务，计算机网络则提供数据文件及其他数字信息的传送服务。随着数字化和 IP 化的发展，语音、视频与数据逐渐在统一的基础设施上承载，形成 ” 三网融合 ” 的趋势。进一步的发展方向是 ” 人人用网、万物联网 “，即人、设备、传感器、服务系统都成为网络中的活动实体。

网络互连能够产生明显的规模效应。梅特卡夫定律指出，具有 N 个结点的网络，其价值与 N2 成正比。结点数相同时，连接关系越丰富，网络的可达性、资源共享能力和潜在协作价值越高。这一规律并不是严格的物理定律，但揭示了连通性在网络系统中的核心地位。

计算机网络并没有一个绝对统一的精确定义。较为准确的表述是：计算机网络由若干通用的、可编程的硬件通过通信线路互连而成，这些硬件并非只服务于单一用途，而是能够传送多种类型的数据，并支持广泛且持续增长的应用。这里的 ” 可编程硬件 ” 意味着网络中的关键设备通常包含处理器和可执行控制逻辑，因此网络中的结点不仅是传统意义上的计算机，也可以是智能手机、网络摄像头、智能传感器、边缘服务器乃至大型计算机系统。

这一界定包含两个基本点。其一，网络连接的是多种硬件，而不是单一类型主机。其二，网络承载的是多种应用，而不仅限于某种固定业务。现代网络既可以承载文本与文件，也可以承载语音、视频、控制消息、传感数据以及未来尚未出现的新型应用。

计算机网络具有两个最基本的性质。第一是连通性。连通性使得分布在不同地点、通过不同链路接入的用户能够相互交换信息，看起来仿佛彼此直接相连。互联网中的连通具有虚拟性：通信双方往往并不知道底层具体经过了哪些链路和设备，也未必清楚对方的真实地理位置。第二是共享性。共享既包括信息共享，也包括软件共享、硬件共享以及计算资源共享。通过网络访问远程资源，在使用体验上常常表现为 ” 资源就在本地 ”。

在抽象表示上，网络通常用图来描述。图中的结点表示主机或网络设备，边表示链路。随着规模扩大，网络内部结构会变得极为复杂，因此工程实践中经常将复杂网络整体用 ” 云 ” 表示。用云表示网络时，可以把主机画在云中，强调其属于网络内部；也可以把主机画在云外，强调主机是通过云状网络获得服务的端系统。图模型与云模型都服务于同一目的，即在不同抽象层级上刻画结点、链路与连通关系。

2 互联网的定义、演进与组成

2.1 互联网与互连网

互联网是计算机网络的一种实现形式，也是目前最成功、最广泛采用的事实标准。严格区分时，internet 与 Internet 含义不同。小写的 internet 是通用名词，表示由多个网络互连而成的互连网，只要求不同网络之间能够实现通信，并不限定统一协议。大写的 Internet 是专用名词，特指当前全球范围内采用 TCP/IP 协议族、通过路由器将众多网络互连而形成的开放互联网，其历史源头是美国的 ARPANET。

因此，任意把几个网络连接起来并实现互通，只能称为互连网；只有采用 TCP/IP 协议族并融入全球开放互联体系，才构成通常所说的互联网。

2.2 互联网的发展概况

互联网基础结构经历了三个重要阶段。第一阶段是从单个分组交换网络 ARPANET 向多网络互连体系过渡。1983 年 TCP/IP 成为 ARPANET 的标准协议，这一事件通常被视为互联网诞生的标志；1990 年 ARPANET 正式关闭。第二阶段是三级结构互联网，即主干网、地区网和校园网或企业网分层构成全国性互连体系。第三阶段是多层次 ISP 结构的互联网，即由主干 ISP、地区 ISP、本地 ISP 以及互联网交换点 IXP 共同组成层次化互连系统。终端主机接入本地 ISP，本地 ISP 再向上连接地区 ISP 和主干 ISP，不同网络之间可以在 IXP 进行高效交换。

万维网的出现是互联网大规模普及的关键推动因素。20 世纪 90 年代，CERN 提出的 WWW 以统一资源定位、超文本组织和浏览器访问方式，大幅降低了非专业用户使用互联网的门槛，使互联网从专业研究网络转变为社会性基础设施。互联网用户数在 1990 年代之后呈指数增长，网络基础设施、应用生态、操作系统、设备制造商与标准组织之间形成相互促进关系。

2.3 互联网的组成

从工作方式看，互联网由边缘部分与核心部分构成。边缘部分由所有连接到互联网上的主机组成，这些主机又称端系统。端系统规模差异很大，小到手机、家用摄像头、物联网设备，大到高性能服务器和超级计算机。边缘部分是用户直接接触并使用的部分，承担应用执行、数据生成与消费、资源访问等功能。

核心部分由大量网络和连接这些网络的路由器组成。核心部分不直接面向最终用户，而是为边缘部分提供连通性与交换能力。核心网的关键设备是路由器，其核心职责是根据分组首部中的控制信息执行分组转发。

另一种常用视角把互联网画成由家庭网络、企业网络、移动网络、服务节点、缓存节点、数据中心与云基础设施共同构成的整体系统。该视角强调互联网并非只有 ” 主机—路由器—链路 ” 这一狭义结构，还包括内容缓存、边缘服务、移动终端、物联网设备、云服务平台等多种角色。接近用户侧的边缘网络通常提供低于几毫秒到几十毫秒级别的接入时延，而跨运营商或跨地区访问则常出现更高的端到端时延。

2.4 端系统之间的通信方式

” 主机 A 与主机 B 通信 ” 在严格意义上并不是两台机器作为整体相互对话，而是运行在主机 A 上的某个进程与运行在主机 B 上的某个进程进行通信。网络协议栈的抽象对象最终都要落到进程之间的数据交换上。

端系统通信有两种基本组织方式。第一种是客户/服务器方式（Client/Server, C/S）。客户进程是服务请求方，服务器进程是服务提供方。客户在需要服务时主动发起通信，因此必须知道服务器地址；服务器通常长期运行，被动等待来自多个客户的请求，并可并发处理多个会话。连接一旦建立，数据传输可以是双向的，但通信关系的发起逻辑与服务关系仍保持 ” 请求—响应 ” 特征。Web、电子邮件和大多数在线业务都以 C/S 为主要组织方式。

第二种是对等方式（Peer-to-Peer, P2P）。在 P2P 方式下，通信双方不预设严格的服务请求者和服务提供者角色。每个结点都可以同时请求服务并向其他结点提供服务，因而单个主机在不同会话中既可能充当客户，也可能充当服务器。P2P 的优点是能够支持大量用户同时参与并共享资源，典型场景包括文件共享、分布式协作和某些覆盖网络。

2.5 Socket 编程抽象

端系统通信在编程层面通常通过 Socket API 实现。Socket 为应用程序提供客户/服务器式编程抽象，是主流操作系统和编程语言都支持的网络编程接口。其起源可以追溯到 Berkeley Unix。Socket 支持两类基础网络服务：

  1. 字节流（stream），通常对应 TCP 所提供的可靠、有序字节流服务。
  2. 数据报（datagram），通常对应 UDP 所提供的无连接消息服务，不保证可靠送达。

不同应用进程通过端口号区分。端口号是传输层复用与分用的实现基础，它把到达同一主机的数据进一步交付到正确的进程或套接字。

Socket API 的基本原语包括：socket 创建通信端点，bind 将套接字与本地地址和端口关联，listen 进入监听状态，accept 被动接受连接，connect 主动发起连接，send 和 recv 执行数据发送与接收，close 关闭连接并释放资源。客户程序的典型过程是：创建套接字，解析服务器地址，发起连接，发送请求，接收响应，关闭连接。服务器程序的典型过程是：创建套接字，绑定本地端口，进入监听状态，接受连接，接收请求，发送响应，最终关闭会话。阻塞式调用在 connect、accept、recv 等步骤中较为常见。

    A[应用进程] --> B[Socket API]
    B --> C[系统调用接口]
    C --> D[传输层 TCP/UDP]
    D --> E[网络层 IP]
    E --> F[链路层 以太网/WiFi]
    F --> G[设备驱动与网卡]
    G --> H[物理链路]
    H --> I[对端网卡与驱动]
    I --> J[对端协议栈]
    J --> K[对端应用进程]

2.6 接入网

接入网（Access Network, AN）是把用户接入互联网的网络，又称本地接入网或居民接入网。接入网既不属于互联网核心部分，也不属于互联网边缘部分，而是连接端系统与互联网中第一个路由器（边缘路由器）的过渡网络。从覆盖范围看，很多接入网本身属于局域网；从功能上看，它的作用是为用户提供进入互联网的入口。

家庭接入的典型方式包括 DSL 和有线电视网接入。DSL（Digital Subscriber Line）利用现有电话线传输数据，通过分离器把语音与数据分离，语音进入传统电话网，数据进入 DSL 调制解调器，再汇聚到中心局的 DSLAM（DSL Access Multiplexer）。DSL 通过频分方式让语音与数据占用不同频带，并且从家庭到中心局通常采用独享线路，因此不同用户之间的接入段相互隔离。DSL 常表现为非对称速率，上行通常低于 2.5 Mbps，常见小于 1 Mbps；下行通常低于 24 Mbps，常见小于 10 Mbps。

有线电视网接入一般采用 HFC（Hybrid Fiber Coax）结构，即光纤与同轴电缆混合接入。电视信号与数据信号在不同频带上传输，经电缆分离器和 cable modem 接入，再在 cable headend 处通过 CMTS（Cable Modem Termination System）汇聚进入 ISP 网络。与 DSL 不同，HFC 接入中的同轴分布式网络通常由多个家庭共享，因此同一接入段上的用户会争用共享带宽。HFC 也常呈现非对称特性，例如下行可达 30 Mbps，而上行常在 2 Mbps 左右。共享媒介和频分复用是 cable network 的显著实现特征。

家庭网络内部通常由 modem、路由器、防火墙、NAT、以太网接口和无线接入点构成。很多商用家庭网关把这些功能集成在单一盒子中。内部有线接口常为 1 Gbps 以太网，无线侧则通过 WiFi 连接手机、平板和其他无线终端。

企业或校园接入大量采用以太网。主机通过以太网交换机连接到内部网络，内部可能部署邮件服务器、Web 服务器和其他业务主机，再经内部路由器连接到 ISP。企业接入的常见速率包括 10 Mbps、100 Mbps、1 Gbps、10 Gbps 及更高档位。这里的交换机负责局域转发，路由器负责把内部网络与外部互联网连接起来。

无线接入网分为无线局域网与广域无线接入网两类。无线局域网主要是建筑物内部或局部区域内的 WiFi 接入，典型标准包括 IEEE 802.11b/g/n，覆盖范围为数十米，速率可从 11 Mbps、54 Mbps 到数百 Mbps。广域无线接入网由运营商提供，覆盖范围可达几十公里，典型代表是 3G、4G/LTE 以及后续蜂窝系统。无线环境中的多个主机共享接入信道，与路由器或基站相连；在蜂窝系统中，基站承担接入控制和无线链路管理职责。

2.7 互联网核心网与交换方式

互联网核心部分的关键任务是为大量边缘主机提供连通性。核心网中最重要的设备是路由器。路由器实现分组交换，其核心操作是接收分组、暂存分组、查表并转发分组。互联网核心网采用分组交换，而不是电路交换。

2.7.1 电路交换

电路交换最初起源于电话网络。若每两部电话都直接连接，当结点数为 N 时需要 N(N−1)/2 对线路，代价会随结点数平方增长。引入交换机后，每部电话只需接入交换机，交换机再通过交换操作把一条用户线临时接到另一条用户线上，从而建立通信路径。

电路交换是面向连接的。它包括三个阶段：建立连接、数据通信、释放连接。建立连接时，需要为通信双方预留端到端的专用物理通路或等效资源；通信期间，这些资源始终被会话独占；通信结束后，资源再被释放。电话语音业务具有持续占用链路的特点，因而适合电路交换；但计算机数据常呈强突发性，真正发送数据的时间可能只占连接生命周期的很小一部分，导致线路利用率很低。

2.7.2 分组交换与存储转发

分组交换采用存储转发原理。发送端先把较长报文划分为若干较短的数据段，再为每个数据段添加首部形成分组。首部中携带目的地址、源地址和其他控制信息。网络中以分组为基本传输单位，接收端在收到全部数据后再根据需要剥去首部、重组原始报文。

分组交换中的每个分组可以独立选择路径。路由器收到分组后，并不是立即在输入端和输出端之间 ” 直通 ” 转发，而是先把分组放入缓存，查找本地转发表，找到合适输出端口，再送往下一跳。路由选择（routing）负责计算或决定从源到目的的大尺度路径，而转发（forwarding）负责在单个路由器内部依据转发表把分组从输入端口送到正确的输出端口。路由选择偏重控制逻辑和全局路径决策，转发偏重数据平面处理和逐跳执行。

主机与路由器在系统角色上也不同。主机承担用户信息处理和应用执行，能够发出和接收分组；路由器主要承担分组存储转发，通常不直接参与终端应用语义。

flowchart LR
    H1[源主机] --> R1[路由器 A]
    R1 -->|查转发表| R2[路由器 C]
    R2 -->|查转发表| R3[路由器 E]
    R3 --> H2[目的主机]

分组交换的主要优点包括四个方面。其一是高效，链路资源按分组逐段动态占用，不需要在整个会话期间为某个用户独占带宽。其二是灵活，不同分组可根据网络状态走不同路径。其三是迅速，许多场景下无需像电路交换那样先建立专用连接即可发送数据。其四是可靠，可靠性可以通过上层网络协议、冗余路径和分布式路由共同提高。

分组交换也带来代价。由于每跳都要进行缓存和查表操作，分组在结点处会经历处理时延和排队时延；分组首部引入了额外开销；网络拥塞时还可能出现丢包、乱序和时延抖动。报文交换同样采用存储转发，但其传输单元是完整报文而非小分组，因此单次存储和等待时间更长，时延显著高于分组交换，现代互联网几乎不再使用报文交换作为基础技术。

2.7.3 统计复用

分组交换的一个重要思想是统计时分复用（Statistical Time-Division Multiplexing）。用户并不在整个生命周期内持续占用网络，只在真正发送数据时占用带宽。如果大量用户的活跃时间并不同步，同一链路就可以被更多用户共享，从而在总体上提高资源利用率。

例如，一条 100 Mbps 的链路上，每个用户订购 5 Mbps 服务。如果按照专用保留思路，只能支持 20 个用户。若进一步假定每个用户只有 50% 时间处于活跃状态，并且不同用户的活跃状态近似独立，则统计复用条件下支持 30 个用户时，瞬时总需求超过 100 Mbps 的概率仍可能低于 2%。这意味着链路能够以较低拥塞概率承载 1.5 倍于 ” 刚性专用分配 ” 时的用户数量。统计复用的收益来自用户流量的突发性与不同时性，但一旦不利事件发生，用户会体验到时延升高和吞吐下降。

2.7.4 路径探测工具

Traceroute 是探测网络路径的常用工具，Windows 上对应 tracert。其作用是从端系统侧识别到达目标主机路径上的连续跳点（hop），从而观察分组在互联网核心网中的逐跳传播过程。这类工具体现了网络层路径观测的工程需求，与后续的路由和 ICMP 机制密切相关。

3 计算机网络体系结构

3.1 分层思想的形成

两个计算机系统要实现可靠通信，必须在数据格式、时序、异常处理、控制动作等方面高度协调。若把整个通信过程看作一个单体问题，协议会极其复杂，既难以设计，也难以实现和维护。分层思想通过把大问题拆解成若干相对独立的局部问题，使每层只解决一部分功能，再通过层间接口组合成完整系统。

20 世纪 70 年代，IBM 的系统网络体系结构 SNA 率先以分层方式组织复杂通信系统。后来不同厂商也提出了各自的分层体系，但彼此之间缺乏兼容性。为解决异构系统互连问题，ISO 提出了开放系统互连参考模型 OSI/RM，希望在全球范围内给出统一框架。OSI 在理论上体系完整，但在市场上并未取得成功，原因主要包括：缺乏足够强的商业驱动力，协议体系过于复杂、实现代价高、效率偏低，标准制定周期过长，以及部分层次划分存在功能重复。结果是，OSI 成为法律意义上的国际标准，而 TCP/IP 则成为事实上的国际标准。

3.2 协议的含义与组成

网络协议是进行数据交换所必须遵守的规则、标准或约定。协议至少包含三个组成要素：

  • 语法：数据与控制信息的结构和格式。
  • 语义：控制信息的含义、各方应执行的动作、对不同事件的响应。
  • 同步：事件发生顺序、发送与接收时序以及状态推进关系。

协议既可以用文字规范描述，也可以直接体现为程序代码。无论采取哪种形式，都必须对通信过程给出精确而无歧义的解释。

3.3 分层的功能划分

分层的本质是把不同抽象级别的问题分开处理。以文件传送为例，最高层关心文件格式、接收方是否准备好、文件传输语义等应用问题；下一层可以只关心如何把数据可靠送达对方；更下一层再处理与物理网络接口、帧格式、最大帧长等相关的问题。通过逐层下推复杂度，应用不必直接面对所有链路与设备细节。

分层的优点在于：各层相对独立，灵活性较高，结构可分割，便于实现和维护，也有利于标准化。其缺点在于：层间封装与解封装会带来额外开销，某些功能可能在多个层次重复出现，导致效率下降。因此层数过少会使单层过于复杂，层数过多又会增加系统组织与综合难度，合理的层次划分必须在抽象清晰与实现效率之间取得平衡。

各层通常承担若干共性功能，包括差错控制、流量控制、分段与重装、复用与分用、连接建立与释放等。这些功能会在不同层以不同形式出现。例如，传输层的流量控制与链路层的流量控制作用范围和对象并不相同。

3.4 体系结构、实现与服务边界

计算机网络体系结构是各层及其协议的集合，它规定了系统 ” 应当完成什么功能 “；实现则是在既定体系结构下，用硬件和软件 ” 如何完成这些功能 “。体系结构是抽象规范，实现是具体系统。

如果某一层内部算法改变，只要对同层对等实体所遵守的协议不变、向上一层提供的服务接口不变、对下一层所请求的服务方式不变，则这种变化通常只影响该层内部实现，而不会迫使相邻层改变。分层设计的关键价值之一正是通过稳定的服务边界隔离内部实现细节。相反，如果改变破坏了接口语义、服务质量或协议格式，影响就会向上下层传播。

3.5 OSI、TCP/IP 与五层体系

OSI 参考模型有七层：物理层、数据链路层、网络层、传输层、会话层、表示层、应用层。TCP/IP 的经典表达常为四层：网络接口层、网络层、传输层、应用层。教学与分析中常采用折中后的五层体系：物理层、数据链路层、网络层、传输层、应用层。五层模型保留了物理与链路层的区分，同时避免把 OSI 的会话层和表示层单独抽出。

五层体系中，各层解决的问题如下：

  • 物理层：在传输媒体上发送和接收比特流。
  • 数据链路层：在相邻结点之间传输帧，处理帧定界、链路寻址、差错检测等。
  • 网络层：提供主机到主机的分组转发与网络互连能力。
  • 传输层：提供进程到进程的通信服务，处理复用与分用、可靠性、流量控制等。
  • 应用层：面向具体应用提供协议语义与业务逻辑。

在 TCP/IP 体系中，路由器在转发分组时通常只处理到网络层和网络接口层，不参与传输层和应用层语义。主机则完整实现从应用层到链路层的全部协议栈。

现代互联网中的协议栈还呈现出 ” 沙漏形 ” 特征。下方存在多种网络接口技术，如以太网、WiFi、蜂窝、PPP 等；上方存在大量应用协议，如 HTTP、SMTP、DNS、RTP 等；中间的窄腰是 IP 层。所谓 “IP over Everything” 指 IP 可以运行在多种底层网络之上；“Everything over IP” 指多种上层应用都建立在 IP 提供的统一无连接分组服务之上。这种结构解释了互联网为何能在底层和上层同时保持高度多样性，而又通过中间统一的网络层实现全球互连。

3.6 封装、解封装、多路复用与分用

发送端应用进程产生应用数据后，应用层首先加上本层首部形成应用层 PDU。该 PDU 下交给传输层，传输层再加上首部形成传输层报文；网络层继续加上网络层首部形成 IP 数据报；数据链路层再加上首部和尾部形成帧；最后物理层把帧表示为比特流，在传输媒体上发送。接收端执行相反过程：物理层接收比特流，数据链路层去掉帧头帧尾，网络层去掉分组首部，传输层去掉传输首部，应用层再把应用数据交给目标进程。

flowchart TD
    A[应用数据] --> B[应用层 PDU]
    B --> C[传输层报文]
    C --> D[网络层 IP 数据报]
    D --> E[链路层帧]
    E --> F[物理层比特流]
    F --> G[链路层解封装]
    G --> H[网络层解封装]
    H --> I[传输层解封装]
    I --> J[应用层解封装]
    J --> K[目标应用数据]

OSI 中把对等层之间交换的数据单位称为协议数据单元 PDU（Protocol Data Unit）。与之相对，层间通过服务访问点交换的数据单位称为服务数据单元 SDU（Service Data Unit）。一个 PDU 可以由一个或多个 SDU 封装而成，也可能把一个 SDU 拆分为多个 PDU。多路复用是多个上层会话共享同一低层通道的过程，分用则是在接收端把同一低层通道上的不同流量分开交付到不同上层实体。端口号、协议号和链路类型字段等都是实现分用的重要标识。

3.7 实体、协议、服务与 SAP

实体（entity）是能发送或接收信息的硬件或软件进程。协议是控制两个对等实体之间通信的规则集合。服务则是下层通过层间接口向上层提供的能力。协议是水平关系，描述 ” 对等方如何通信 “；服务是垂直关系，描述 ” 本层向上层提供什么能力 “。上层通常只看到服务，而不直接看到下层协议内部细节。

相邻两层交互的逻辑位置称为服务访问点 SAP（Service Access Point）。SAP 是抽象接口，而非某个固定物理元件。上层通过服务原语在 SAP 处请求服务，下层通过协议实现这些服务。Socket API 可以看作某类传输层与应用层之间服务接口的工程化体现。

3.8 协议设计的困难与不可能性边界

协议设计不能假设一切都处于理想状态。丢包、比特错误、重传、失序、结点故障和路径异常都必须纳入设计。著名的 ” 蓝军问题 ” 说明，在不可靠信道上，即使允许确认和再确认，也无法设计出一个使双方在逻辑上达到 100% 共同确信的有限协议。因为任何一方始终无法确定自己最后发出的那一条消息是否被对方成功接收。该结论揭示了分布式系统与网络协议的一个基本边界：可靠传输可以逼近极高概率，但在存在不确定通信失败的条件下，不可能获得绝对意义上的有限次共同知识。

4 计算机网络的性能指标与工程含义

4.1 速率、带宽与吞吐量

速率是计算机网络中最重要的性能指标之一，表示数据传送速率，也称数据率或比特率。单位是 bit/s，也常用 kbit/s、Mbit/s、Gbit/s 表示。速率通常指额定速率或标称速率，而不是实际运行过程中必然达到的平均值。

带宽有两层常用含义。在频域中，带宽是信号占据频谱的宽度，单位是 Hz。在网络工程中，带宽常表示某通道在单位时间内所能通过的最高数据率，单位是 bit/s。二者本质相关：可用频带越宽，通常可支持的最高数据率越高。

吞吐量表示单位时间内通过某个网络、链路或接口的实际数据量。吞吐量是现实测量值，而不是理论上限。它会受到链路带宽、协议开销、端系统处理能力、拥塞状况和路径瓶颈等多方面约束。

4.2 时延的构成

时延是数据从网络一端到另一端所需的时间，可细分为发送时延、传播时延、处理时延和排队时延。

发送时延又称传输时延，表示把数据帧所有比特推入链路所需的时间。若帧长为 L bit，链路发送速率为 R bit/s，则发送时延为：

dtrans​=RL​

传播时延表示信号在信道中传播所需时间。若链路长度为 D 米，传播速度为 V 米每秒，则传播时延为：

dprop​=VD​

处理时延是主机或路由器分析首部、查表、进行差错检验等操作的时间。排队时延是分组在输入队列或输出队列中等待服务的时间，强烈依赖于当时的业务负载。

总时延为：

dtotal​=dtrans​+dprop​+dproc​+dqueue​

时延分析中最容易混淆的是发送速率与传播速度。提高链路带宽会降低发送时延，因为单位时间内可以注入更多比特；但比特在链路中的传播速度由物理介质决定，通常接近光速的某个比例，不会因为带宽提高而显著增加。因此，” 高速链路上比特传播得更快 ” 这一说法是错误的。

在存储转发网络中，分组必须先完整到达路由器，才能被转发到下一跳。若单条链路速率为 (R)，分组长为 (L)，忽略传播时延，则经过两跳链路的端到端发送时延约为 (2L/R)。跳数越多，逐跳存储转发带来的累计发送时延就越明显。

收费站与车队的类比可以直观说明发送时延与传播时延的区别。车辆通过收费站相当于比特依次进入链路，其服务时间对应发送时延；车辆在公路上行驶对应传播时延。即使车辆行驶速度很快，只要车队需要逐辆通过收费站，整个车队到达下一收费站的时间仍由 ” 通过收费站所需总时间 ” 与 ” 行驶时间 ” 共同决定。

4.3 分组交换与电路交换的时延比较

设主机 A 先向主机 B 发送一个 1 Mb 文件，再发送一个 10 Kb 文件，两条链路带宽分别为 2 Mbps 和 1 Mbps，传播时延均为 0.1 s，忽略处理时延与协议开销。

若采用电路交换，建立连接需要 0.3 s，且建立后端到端速率受瓶颈链路限制为 1 Mbps。两个文件的总发送时间为：

1,Mbps1,Mb+10,Kb​=1.01,s

再加上 0.2 s 传播时延和 0.3 s 建连时延，总时间为 1.51 s。

若采用分组交换，并把数据切成 10 Kb 分组，则共需 101 个分组。第一个分组到达 B 的时间为：

0.1+2,Mbps10,Kb​+0.1+1,Mbps10,Kb​=0.215,s

由于后续分组在瓶颈链路上以 1 Mbps 的节奏连续输出，剩余 100 个分组额外需要 1 s，总时间为 1.215 s。该例说明：对于突发数据和可流水化传输的场景，分组交换能够通过逐跳流水和统计复用降低完成时间；在多链路传输中，分组流会被瓶颈链路 ” 整形 “，表现为按最慢链路节奏输出。

4.4 时延带宽积、RTT 与利用率

时延带宽积等于传播时延乘以带宽，可理解为 ” 链路上可同时容纳的比特数量 “，也可称为以比特为单位的链路长度。当一条链路中的 ” 管道 ” 被比特充分填满时，该链路才实现充分利用。这一概念对窗口协议、流水线传输和带宽利用分析都十分重要。

往返时间 RTT（Round-Trip Time）表示从发送方发出数据开始，到发送方收到接收方确认所经历的总时间。RTT 不仅包含来回两程的传播时延，也包含中间结点的发送、处理与排队时间。许多协议的超时估计、吞吐控制和交互体验都直接受 RTT 影响。卫星链路的 RTT 往往较大，因此对交互型业务尤为敏感。

利用率分为信道利用率与网络利用率。信道利用率表示某条信道在多大比例时间内处于传输状态；网络利用率则是全网信道利用率的加权平均。利用率并非越高越好。当利用率趋近于 1 时，排队系统容易进入重载，排队时延会急剧上升。常见排队论近似关系写作：

D=1−UD0​​

其中 D0​ 为网络空闲时的时延，(U) 为利用率。当 U 逼近 1 时，分母趋近于 0，时延迅速发散。这说明高吞吐和低时延之间存在明显工程权衡。

4.5 与服务质量相关的非性能特征

除上述性能指标外，网络系统还具有若干重要非性能特征，包括费用、质量、标准化、可靠性、可扩展性、可升级性以及易管理和可维护性。这些特征往往决定网络是否适合长期部署。

对不同业务而言，服务质量需求也不同。数字语音业务除带宽与平均时延外，还高度依赖时延抖动、分组丢失率与连续可用性；数字视频业务关注持续吞吐、峰值速率、丢帧与抖动控制；金融业务则更看重正确性、低尾时延、可靠性、可追踪性、安全性和高可用性。仅用 ” 带宽大、时延小 ” 无法完整刻画真实网络服务质量。

若一个系统共有 N 层协议，应用层产生 M 字节数据，而每层额外附加 H 字节首部，则纯首部开销占总发送字节数的比例为：

M+NHNH​

该式体现了分层带来的协议开销：当应用负载很小而层数较多时，首部比例会显著增大。

5 计算机网络的分类

计算机网络可以从多个维度分类。

按照作用范围，可分为个人区域网 PAN、局域网 LAN、城域网 MAN、广域网 WAN 以及全球性的互联网。PAN 的范围通常在数米到十米量级，典型例子是蓝牙；LAN 覆盖房间、楼宇或校园，典型例子是 WiFi 和以太网；MAN 覆盖城市范围，典型例子包括有线电视接入网、DSL 和蜂窝网络；WAN 跨越国家与洲际，典型例子包括大型 ISP 骨干网和 VPN；互联网则是全球 ” 网络的网络 “。当中央处理机之间距离极近，例如只有 1 米量级甚至更小，通常更适合称为多处理机系统而非计算机网络。

按照使用者，可分为公用网与专用网。公用网对按规定付费的公众开放；专用网为特定业务目的建设，例如企业专网、工业控制网或特定部门网络。二者都可以承载多种业务，只是服务对象与管理边界不同。

接入网也可视为一种特殊类别。它专门负责把用户连接到互联网入口，是边缘和核心之间的桥梁。许多接入网在局部覆盖范围上属于局域网，但在功能定位上又不同于一般 LAN。

6 计算机网络的发展历史

计算机网络的历史可以概括为从分组交换原理、到异构网络互连、再到全球商业互联网与云时代网络的演进过程。

1961 年到 1972 年是早期分组交换思想形成阶段。Kleinrock 用排队论说明了分组交换的有效性；Baran 讨论了在军用网络中使用分组交换的可行性；1967 年 ARPANET 的构想形成；1969 年第一个 ARPANET 结点投入运行；1972 年 ARPANET 公开演示，NCP 成为早期主机到主机协议，同时电子邮件成为第一个真正意义上的 ” 杀手级应用 “，当时 ARPANET 已有 15 个结点。

1970 年代至 1980 年代是互连思想成熟与多种网络并行发展的阶段。1970 年出现 ALOHAnet 卫星网络；1974 年 Cerf 与 Kahn 提出异构网络互连体系结构，奠定后续 TCP/IP 思想基础；1976 年 Xerox PARC 推出以太网；到 1979 年，ARPANET 已增长到约 200 个结点。

1980 年代到 1990 年代是协议体系确立与网络快速扩张阶段。1982 年 SMTP 定义电子邮件传输协议；1983 年 TCP/IP 开始部署，DNS 被定义为域名到 IP 地址映射机制；Berkeley sockets 也在这一时期成为广泛采用的编程接口；1985 年 FTP 被标准化；1988 年 TCP 拥塞控制机制成为关键网络控制技术。

1990 年代以后，互联网走向商业化和大众化。ARPANET 与 NSFNET 在早期 1990 年代相继退出历史舞台，商业 ISP 成为连接提供者。1993 年 WWW 开始爆发式增长，Mosaic 和后来的 Netscape 使浏览器成为互联网入口。1993 年 BGP 成为自治系统间路由的重要基础。进入 1990 年代后期和 2000 年代，互联网出现即时通信、P2P 文件共享、在线视频等新型应用，网络安全问题迅速凸显，骨干链路速率进入 Gbps 时代。2005 年以后，移动终端、宽带接入、高速无线网络、社交网络、云计算和内容分发网络共同推动互联网形态继续演化。大型内容提供商与云服务商开始构建自己的专用骨干和数据中心网络，使 ” 互联网之上的互联网 ” 成为现实。

我国计算机网络的发展也经历了清晰的时间线。1980 年铁道部开始进行联网实验；1989 年 11 月第一个公用分组交换网 CNPAC 建成运行；1994 年 4 月 20 日，中国以 64 kbit/s 专线正式接入互联网；同年 5 月，中国科学院高能物理研究所建立了我国第一个万维网服务器；同年 9 月 CHINANET 正式启动。此后，我国形成了多个可与互联网互连的全国性公用网络，其中规模较大的包括 CHINANET、UNINET、CMNET、CERNET 与 CSTNET。CERNET 始建于 1994 年，是我国第一个 IPv4 互联网主干网；2004 年 2 月，CNGI 的主干网 CERNET2 试验网正式开通。我国互联网发展统计由 CNNIC 持续发布。

7 计算机网络标准化

标准化的核心目标是互操作性，即来自不同厂商、运行不同实现的设备和软件能够通过一致的接口、字段格式和处理流程协同工作。没有标准化，互联网络只能停留在局部兼容和封闭生态。

网络与互联网相关的主要标准组织包括：

  • ITU：国际电信联盟，分为 ITU-T、ITU-R、ITU-D 三个部门。ITU-T 负责电话、电报、数据接口等技术建议；ITU-R 负责无线频谱协调；ITU-D 面向发展事务。
  • ISO：国际标准化组织，提出了 OSI 参考模型。
  • IEEE：电气与电子工程师协会，负责大量通信标准，典型代表有 IEEE 802.3 以太网和 IEEE 802.11 无线局域网。
  • 3GPP：蜂窝通信标准的重要制定组织，推动 GSM、UMTS、LTE、5G 等移动通信系统演进。
  • IETF：互联网工程任务组，负责互联网协议工程标准，HTTP、DNS 等大量协议文档都以 RFC 形式发布。
  • IRTF：互联网研究任务组，负责更偏研究与前瞻方向的问题。
  • IAB：互联网体系结构委员会，负责互联网体系结构层面的指导。
  • ISOC：互联网协会，为相关技术共同体提供组织支撑。
  • W3C：负责 Web 相关标准，如 HTML 和 CSS。

IETF 的组织结构体现了互联网标准制定的工程特征。IETF 下按领域划分为多个工作组（WG），IRTF 下则有研究组（RG），IAB、IESG、IRSG 等机构分别负责体系结构、工程指导和研究指导。互联网协议标准不是一次性封闭制定，而是在开放讨论、实现试验、互操作验证和持续修订中逐渐成熟。

互联网标准文档主要以 RFC（Request For Comments）形式发布。RFC 不是简单的 ” 评论请求 “，而是互联网技术文档体系的正式载体。互联网草案（Internet Draft）有效期通常只有六个月，还不是 RFC。建议标准（Proposed Standard）进入 RFC 序列；进一步成熟后可成为互联网标准（Internet Standard），并分配 STD 编号。除这两类之外，RFC 还包括历史的、实验的和提供信息的等类别。一个正式标准可能关联多个 RFC 文档，体现出标准演进和修订的累积性。

一些典型 RFC 及其对应协议包括：UDP 的 RFC 768，IP 的 RFC 791，ICMP 的 RFC 792，TCP 的 RFC 793，SMTP 的 RFC 821，ARP 的 RFC 826，FTP 的 RFC 959，DNS 概念与设施的 RFC 1034/1035，BGP-4 的 RFC 1654，PPP 的 RFC 1661，HTTP/1.0 的 RFC 1945，OSPFv2 的 RFC 2328，RTP 的 RFC 3550，URI 通用语法的 RFC 3986。通过这些文档可以看到，互联网的核心协议是在开放文本、可公开检索、可逐步修订的机制下成长起来的。

新需求会不断推动新标准。例如 Multipath TCP（MPTCP）面向多接入环境和多路径传输场景，在终端同时具备 WiFi 和蜂窝接口时，允许同一连接利用多条路径共同传输数据。MPTCP 的目标是提高带宽利用率、增强连接恢复能力，并把数据从拥塞路径迁移到较空闲路径。相关标准文档经历了从威胁分析、体系结构原则、拥塞控制到正式协议扩展的长期演进，体现出互联网标准开放、渐进与工程验证并重的特性。多路径思想不仅适用于移动终端，也广泛适用于数据中心等多链路环境。

网络标准化的边界也在扩展。除传统主机间协议外，面向 AI 和高性能计算的在网计算、在网集合通信协议等，也在新的国际组织和产业联盟中推进。这表明 ” 网络协议 ” 不仅服务于通用互联，也逐渐服务于数据中心、集群计算和智能系统。

8 计算机网络安全

互联网最初建立在相对可信的研究共同体环境之上，早期设计并未充分考虑大规模敌手攻击、恶意软件传播与跨域滥用问题。随着互联网成为全球开放系统，网络安全问题必须在每一层进行考虑。敌手（adversary）可能攻击主机、链路、协议、服务器、路由基础设施或用户行为本身。

8.1 嗅探与明文泄露

在共享以太网或无线网络等广播介质环境中，若网卡工作在混杂模式，就可能读取经过本机但并非发给本机的帧。Wireshark 之类的协议分析工具本身是合法的诊断工具，但同样揭示了嗅探攻击的现实可能性。只要传输内容是明文，用户名、口令和会话数据都可能被窃取。对此最基本的防御措施是端到端加密或链路加密，使攻击者即便截获分组，也难以恢复有效内容。

8.2 地址伪装与身份欺骗

IP spoofing 指攻击者伪造源 IP 地址，使目标主机误以为数据来自某个可信结点。由于传统 IP 转发只依据目的地址，不在路径中逐跳验证源地址真实性，因此源地址伪装在历史上长期存在。伪装可被用于绕过简单过滤规则、隐藏真实身份、参与放大攻击等。其对应防御包括用户认证、会话鉴别、加密认证、访问控制以及更严格的源地址验证机制。

8.3 恶意软件、僵尸网络与主机失陷

恶意软件可直接感染主机。病毒通常依赖某种宿主或传播介质进行自我复制，例如电子邮件附件；蠕虫则更强调通过网络自动复制与传播。恶意软件可以记录击键、窃取访问记录、上传本地信息，并把受害主机纳入僵尸网络（botnet）。僵尸网络由大量受控主机构成，是发动分布式拒绝服务攻击的重要基础设施。

与此相邻但又不同的是一般应用收集用户数据的行为。是否属于恶意，取决于是否获得明确授权、是否遵循最小必要原则、是否透明告知用途、是否存在越权采集和滥用。网络安全不只涉及 ” 技术上能否收集 “，也涉及 ” 权限、目的和边界是否合理 “。

8.4 DoS 与 DDoS

拒绝服务攻击（DoS）通过制造大量无效流量或无意义请求占用目标资源，使正常用户无法获得服务。资源可以是服务器 CPU、内存、连接状态、带宽，也可以是路由器、链路和网络基础设施中的队列。分布式拒绝服务攻击（DDoS）进一步利用大量受损主机同时向目标发起攻击。之所以采用分布式多点攻击，是因为单点攻击的发送频率和带宽受限，更容易被识别和抑制；而多点攻击能够同时从不同位置发起流量，破坏性更强，也更难定位源头。

DDoS 攻击经常结合源地址伪装和反射放大。若某些协议的响应报文显著大于查询报文，攻击者就可以伪造受害者地址向第三方服务器发送小请求，使第三方把大响应打向受害者，从而放大攻击流量。这类机制解释了为什么源地址验证和放大点治理对互联网基础设施安全十分关键。

8.5 真实世界事件与安全观测

网络攻击已经是现代社会冲突的重要组成部分。针对银行、政府机构、外交部门和基础设施的大规模 DDoS 事件说明，网络安全问题不仅影响单个主机和应用，也影响社会运行与国家层面的数字基础设施。数字攻击地图等公开工具通过聚合高强度 DDoS 事件，展示不同国家和地区之间攻击的频繁程度与严重性，反映出互联网已经成为持续对抗环境。

8.6 登录口令泄露的典型链条

用户在网站登录时输入用户名和口令，这一过程可能在多个环节泄露信息。输入环节可能遭遇键盘记录器、恶意浏览器插件或本地主机失陷；传输环节可能遭遇明文嗅探、中间人攻击、错误证书信任或伪造热点；应用环节可能遭遇钓鱼网站和伪造登录页；服务器端若以明文或可逆方式存储口令，也会在数据库泄露时造成大规模风险；用户若在不同网站复用口令，还会导致撞库攻击。对应措施包括：使用 TLS 加密传输、进行服务器身份认证、使用加盐哈希存储口令、启用多因素认证、限制异常登录、隔离本机恶意软件、减少口令复用以及对用户界面进行反钓鱼设计。

8.7 网络控制与访问限制

网络安全与网络控制在某些场景下会结合出现。对某一网站或服务的可达性控制，可以作用于多个层次，包括 DNS 污染或重定向、IP 地址封锁、TCP 连接重置、HTTP 代理拦截、SNI 或域名过滤、TLS 握手干预以及代理与隧道检测等。这类机制说明，” 应用返回 404” 与 ” 网络路径被阻断 ” 是两种不同层次的问题：前者是应用层响应语义，后者是路径控制或访问策略问题。理解可达性故障必须明确发生在哪一层。

9 实现视角：端系统协议栈与 Linux 网络栈

网络课程中的分层模型是抽象描述，但实际网络功能必须落到操作系统内核、设备驱动与硬件控制器中。

在端系统中，应用程序通过系统调用接口和 Socket API 进入内核。内核中的网络子系统负责套接字管理、TCP/UDP 处理、IP 层分组逻辑、以太网封装以及与网卡驱动协作。数据从应用流向网络时，大致经历以下路径：应用进程写入套接字，系统调用接口把数据交给内核；传输层根据套接字类型进行 TCP 或 UDP 处理；网络层完成 IP 寻址与分组；链路层形成帧；设备驱动把数据提交给网卡控制器，由网卡通过端口发送到物理链路。接收路径则反向执行，并最终根据端口号和套接字状态把数据交给正确的应用进程。

Linux 网络栈不是独立存在的，它是整个内核的一部分。内核中同时包含文件系统、虚拟内存、调度器、中断处理、设备模型和驱动框架。网络相关组件至少包括：套接字接口、协议族、TCP/UDP 协议实现、IP 层、链路层接口、网络设备抽象、设备驱动、netfilter/NF_HOOK 等包过滤与处理框架，以及协议栈与用户态之间的系统调用接口。网卡控制器通过 I/O 总线与 CPU 和内存交互，驱动负责配置硬件寄存器、提交发送描述符、接收分组并触发中断或轮询处理。

从观测与调试角度，操作系统还提供多种工具链。tcpdump 和 Wireshark 侧重抓包与协议分析，netstat、ss、ip、nicstat 用于观察连接和接口状态，strace、ltrace、perf、top、pidstat 等工具帮助观察系统调用、性能热点与资源消耗。这些工具共同构成了 ” 实现层可见性 “，使抽象的协议栈能够被测量、诊断和优化。

这一实现视角也解释了分层模型与现实系统的关系。抽象层次并不意味着每层在实现中一定由完全独立的模块承担；在真实内核中，数据路径、缓存、队列、中断、DMA、协议状态和驱动行为相互耦合。但只要服务边界和协议语义保持稳定，复杂实现仍然可以对上层呈现清晰分层结构。这也是体系结构与实现相区分的根本原因。
