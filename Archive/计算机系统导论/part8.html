链接 (Linking)

编译工具链

一个 C 程序从源代码到可执行文件通常经过四个阶段：预处理、编译、汇编和链接。预处理器处理 #include、#define 等文本级指令，得到展开后的源代码；编译器把 C 代码翻译为汇编；汇编器把汇编翻译为可重定位目标文件；链接器把多个目标文件和库组合为最终可执行文件。

这一链条说明，源文件之间并不是直接“拼接”成程序，而是先各自形成带有符号表和重定位信息的目标模块，再由链接器统一解决跨模块引用。

链接器存在的理由

链接器首先服务于模块化。程序可以分拆为多个源文件独立编写和维护，标准函数则可打包为库供众多程序共享。其次，链接器服务于效率。分离编译使单个源文件修改后无需重编全部代码，也允许并行编译；静态库只抽取被实际引用的目标模块，动态库则让多个进程共享一份库代码副本。

符号解析

链接器的第一项核心任务是符号解析，即把每个符号引用与恰好一个符号定义对应起来。这里的符号主要指全局变量、静态变量、全局函数和静态函数；局部自动变量位于栈中，不参与链接级符号解析。

一个目标文件中的符号可分为三类：本模块定义且可被外部引用的全局符号，本模块引用但在外部定义的外部符号，以及本模块内部可见的本地符号。每个目标文件的符号表记录了这些符号的名称、位置、大小和可见性等信息。

强符号与弱符号

跨模块多重定义的处理依赖强弱符号规则。函数和已初始化的全局变量是强符号；未初始化的全局变量是弱符号。链接器遵循三条规则：不允许多个同名强符号；若一个强符号与多个弱符号同名，则选择强符号；若多个弱符号同名，则任取其一。

这一规则会带来类型不一致的隐患。链接器只按名字匹配，不检查类型一致性。若一个模块定义 int x = 7，另一个模块仅声明 double x 且形成弱定义，最终链接成功后，对 x 的访问可能按错误类型解释内存，甚至覆盖相邻对象。避免此类问题的办法是统一在头文件中使用 extern 声明，并尽量减少不必要的全局变量。

重定位

链接器的第二项核心任务是重定位。可重定位目标文件中的节地址和符号地址都只是相对本文件的暂定位置，真正的运行时地址要等链接器把所有输入文件的同类节合并后才能确定。重定位过程包含三步：合并同类节、为合并后的节和符号分配最终地址、修补代码和数据中的引用。

汇编器在遇到尚不知道绝对位置的符号引用时，会生成重定位条目。重定位条目记录了“需要修改的位置”“引用的是谁”“附加常量是多少”以及“修改方式是什么”。对 x86-64 而言，引用全局变量常使用绝对地址重定位，函数调用常使用 PC 相对重定位。对 call 指令来说，机器码中真正填入的不是目标绝对地址，而是“目标地址减去下一条指令地址”的位移。

ELF 目标文件格式

可重定位目标文件、可执行文件和共享对象都采用 ELF 格式。ELF 中最重要的节包括 .text、.rodata、.data、.bss、.symtab、.rel.text 和 .rel.data。.text 存放机器指令，.rodata 存放字符串字面量和只读常量，.data 存放已初始化且非零的全局或静态对象，.bss 存放未初始化或零初始化的全局或静态对象。.symtab 保存符号表，.rel.text 与 .rel.data 分别保存代码段和数据段中的重定位信息。

除真实节外，链接器还使用若干伪节或伪状态。UNDEF 表示本文件引用但未定义的符号，ABS 表示绝对地址符号，COMMON 表示尚未真正安置位置的未初始化全局变量。COMMON 的存在反映了弱符号语义：编译阶段先把决策留给链接器，待最终链接时再决定落入 .bss 还是被强符号覆盖。未初始化的静态变量则不具备跨模块冲突可能，通常直接进入 .bss。

静态库与共享库

静态库本质上是若干 .o 文件的归档。链接器扫描静态库时，会根据当前未解析符号集合按需抽取其中成员，而不是把整个库都拷入可执行文件。静态链接的缺点是可执行文件膨胀、进程间不能共享库代码、修复库缺陷后必须重新链接依赖程序。

共享库在加载时或运行时动态装入。生成共享库通常要求位置无关代码。程序与共享库的链接结果不再是“把库代码复制进去”，而是记录“程序依赖哪些共享对象”。当程序启动时，动态链接器负责装入这些 .so 文件、完成重定位并解析外部符号。多个进程因此可以共享一份库代码映像。

静态链接与动态链接都遵守顺序扫描原则。命令行上依赖者必须位于被依赖者之前，才能在扫描后者时发现尚未解析的符号。若两个库循环依赖，则可能需要重复列出某个库。

PLT、GOT 与延迟绑定

动态链接下，程序通常并不知道共享库函数的最终绝对地址，而是先通过过程链接表 PLT 跳转。PLT 再借助全局偏移表 GOT 查找真正目标地址。若某个 GOT 项尚未解析，PLT 会触发动态链接器完成符号解析并把地址写回 GOT；以后再次调用同一函数时，就能直接经由已填好的 GOT 项跳转。这种机制称为延迟绑定。

库打桩

库打桩是在不改动目标程序主体逻辑的前提下拦截库函数调用。其用途包括调试、监控、安全隔离和性能分析。编译时打桩依赖宏替换，需要源代码；链接时打桩依赖链接器的包装功能，把对原函数的调用改写为对包装函数的调用；加载时打桩最灵活，通过 LD_PRELOAD 把自定义共享库优先装入，从而拦截动态链接程序中的目标函数。
